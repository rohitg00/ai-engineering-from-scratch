## Core concept (the one thing to remember)

An AI engineering environment is a **four-layer dependency stack**: System Foundation (OS, shell, git, GPU drivers) → Package Managers (uv, pnpm, cargo) → Language Runtimes (Python, Node, Rust) → AI/ML Libraries (PyTorch, NumPy, transformers). You install bottom-up because each layer depends on the one below it. Within Layer 3, a **virtual environment** isolates Layer 4 per project so version conflicts between projects stay contained instead of accumulating in a system-wide install. The investment in setting this up once, properly, is what stops every later lesson from becoming a fight against tooling.

## Mental model / analogy

Think of it like a building: foundation (OS + drivers), plumbing (package managers — they move dependencies around), framing (language runtimes — they hold the structure), interior (AI libraries — what you actually use). You don't install the wallpaper before the drywall is up. And a virtual environment is the equivalent of building a separate apartment unit for each project — they share the foundation but have isolated interiors, so renovating one doesn't break another.

## Canonical code (minimum viable example)

The heart of the verification pattern: a single function that runs a named check, prints PASS/FAIL, and returns a bool. The full lesson script is a list of these tuples driven by a single dispatcher.

```python
import sys
import shutil

CHECKS = [
    ("Python 3.11+", lambda: sys.version_info >= (3, 11), f"Python {sys.version}"),
    ("Git", lambda: shutil.which("git") is not None, None),
    ("Node.js", lambda: shutil.which("node") is not None, None),
    ("Rust (cargo)", lambda: shutil.which("cargo") is not None, None),
]

def run_check(name, check_fn, detail=None):
    try:
        if check_fn() is False:
            raise Exception("check returned False")
        suffix = f" ({detail})" if detail else ""
        print(f"  [PASS] {name}{suffix}")
        return True
    except Exception:
        print(f"  [FAIL] {name}")
        return False

passed = sum(run_check(*c) for c in CHECKS)
sys.exit(0 if passed == len(CHECKS) else 1)
```

The design choices worth carrying forward: (1) checks-as-data (a list of tuples) makes the script easy to extend without touching the dispatcher; (2) `__import__("foo")` works as a check for library presence and returns the module if you want to inspect it; (3) optional checks live in a separate list so they don't gate the exit code.

## Key equations / invariants

- **Stack ordering invariant:** `Layer N` cannot function without `Layer N-1`. Specifically: AI libs need a runtime, runtimes need a package manager (or to be a manually-built binary), package managers need a working OS.
- **Venv invariant:** when a venv is active, `which python` resolves to `.venv/bin/python`, not the system Python. If it doesn't, the venv isn't active — that's the diagnostic.
- **Verification contract:** the `verify.py` script is the source of truth for "is this environment ready for the course." Any disagreement between the script and the lesson spec is a bug in one or the other.
- **GPU = optional:** `torch.cuda.is_available()` or `torch.backends.mps.is_available()` being `False` is non-fatal for the course's CPU-friendly lessons. GPU is a force multiplier, not a prerequisite.

## Common pitfalls

- **`sudo pip install`** → pollutes system Python, can break OS-level tools that depend on specific package versions. → Use `uv` with a virtual environment instead. Never `sudo pip`.
- **"My install worked but `import` fails"** → almost always the venv isn't active. The package installed into venv A, but the shell is using venv B (or system Python). → `which python` to confirm; `source .venv/bin/activate` if needed.
- **`python` vs `python3` on macOS** → modern macOS doesn't ship a `python` symlink by default. → Standardize on activating the venv (which gives you `python`), not on writing `python3` everywhere.
- **Installing PyTorch with the wrong CUDA index URL** → pip silently installs a CPU-only wheel even though you have a GPU. → Always specify `--index-url https://download.pytorch.org/whl/cu124` (or the version matching your CUDA install) when GPU is the goal. Verify with `torch.cuda.is_available()` immediately after.
- **Mixing `uv pip install` and `pip install` in the same venv** → mostly harmless but can produce surprising resolution differences. → Pick one (uv) and stick with it inside a given project.
- **Trusting that the verification script covers the full lesson spec** → at the current commit, `verify.py` does not enforce Python 3.11+, does not check for `uv` or `pnpm`, and does not enforce Node 20+. A pass from this script means "some toolchain exists," not "the lesson's canonical toolchain was followed." Action items in the review address this.
