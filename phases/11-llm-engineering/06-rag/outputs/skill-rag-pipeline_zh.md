# RAG 流水线模式

每个 RAG 系统都遵循以下模式：

```
documents -> chunk -> embed -> store
query -> embed -> search(top_k) -> build_prompt -> generate
```

索引每个文档发生一次。查询在每个用户请求时发生。

## 何时使用 RAG

- LLM 需要访问私有或最近的文档
- 微调太贵或更新太慢
- 你需要为答案引用来源
- 知识库频繁变化

## 何时不使用 RAG

- 答案是 LLM 已经具备的一般知识
- 任务是创意性的（写作、头脑风暴）而非事实性的
- 你需要模型采用特定的推理风格（使用微调）

## 实现清单

1. 将文档分块为 256-512 token 的段，50 token 重叠
2. 使用一致的嵌入模型嵌入每个分块
3. 将嵌入存储在向量数据库中，并保留原始文本
4. 查询时，使用相同模型嵌入用户问题
5. 通过余弦相似度检索 top-k（5-10）个最相似分块
6. 构建提示词：系统指令 + 检索上下文 + 用户问题
7. 基于检索到的上下文生成答案
8. 返回答案及来源引用

## 常见错误

- 索引和查询使用不同的嵌入模型（向量不兼容）
- 分块太小（丢失上下文）或太大（稀释相关性）
- 分块之间未包含重叠（在边界处拆分句子）
- 文档变化时忘记重新索引
- 未生成连贯答案就将检索到的分块返回给用户
- 事实性 RAG 查询未设置 temperature=0（更高温度 = 更多幻觉）

## 调试检索

如果正确的分块未被检索到：
1. 打印查询嵌入并验证其非零
2. 手动检查已知相关分块的余弦相似度
3. 尝试改写查询以匹配文档词汇
4. 验证索引和查询时的嵌入模型匹配
5. 检查相关内容是否在分块过程中丢失

## 生产参数

- 分块大小：256-512 tokens
- 重叠：50 tokens（分块大小的 10-20%）
- Top-k：大多数用例 5-10
- 温度：事实性答案为 0
- 嵌入模型：text-embedding-3-small（成本效益）或 text-embedding-3-large（更高准确率）
