# 提示模式选择指南

在构建基于 LLM 的功能时，先选择提示模式再编写提示词。模式决定结构，内容填充细节。

## 模式决策矩阵

| 任务类型 | 主要模式 | 次要模式 | 温度 | 需要少样本？ |
|-----------|----------------|-------------------|-------------|-----------------|
| 数据提取 | 模板填充 | 少样本 | 0.0 | 是（2-3 个示例） |
| 分类 | 少样本 | 护栏 | 0.0 | 是（3-5 个示例） |
| 摘要 | 角色 + 模板 | 受众适配 | 0.3 | 否 |
| 代码生成 | 角色 | 思维链 | 0.0 | 可选 |
| 创意写作 | 角色 | 批判 | 0.7-1.0 | 否 |
| 多步推理 | 思维链 | 分解 | 0.3 | 可选 |
| 问答 | 角色 + 护栏 | 边界 | 0.3 | 否 |
| 提示词生成 | 元提示 | 批判 | 0.7 | 是（1-2 个示例） |
| 内容审核 | 护栏 + 边界 | 少样本 | 0.0 | 是（5+ 个示例） |
| 翻译/适配 | 受众适配 | 少样本 | 0.3 | 是（2-3 个示例） |

## 何时使用每种模式

**角色模式**：作为每个提示词的基线使用。唯一的问题是角色要具体到什么程度。对于通用任务，宽泛的角色即可。对于领域特定任务，角色应指明领域、资历级别和上下文。

**少样本模式**：当输出格式比内容更重要时使用。如果模型需要生成特定的 JSON 形状、CSV 格式或分类标签，示例比指令更有效。经验法则：简单格式 2-3 个示例，复杂或模糊格式 5+ 个示例。

**思维链模式**：用于数学、逻辑、多步分析以及任何需要模型"展示其工作过程"的任务。在推理任务上可将准确率提高 10-40%（Wei 等人，2022）。不要用于简单的事实查找或提取——它会浪费 token。

**模板填充模式**：用于结构化提取，其中每个输出必须具有相同的形状。最适合 temperature=0.0 和对缺失字段显式处理"N/A"。

**批判模式**：当质量比速度更重要时使用。模型生成、批判并改进。大致使 token 成本翻倍，但显著提高了准确性和完整性。最适合高风险的输出（报告、建议、面向公众的内容）。

**护栏模式**：用于任何面向用户的系统。始终包含：范围边界、对超出范围请求的拒绝行为，以及显式的"我不知道"处理。与应用程序端的输入验证结合使用。

**元提示模式**：用于为新任务生成提示词。不要从头开始编写提示词，而是描述任务并让模型编写提示词。然后测试并迭代。节省初始提示词开发时间。

**分解模式**：用于受益于分而治之的复杂问题。模型将问题分解为多个部分，分别解决，然后组合。对具有 3-7 个子任务的问题最有效。

**受众适配模式**：当相同内容需要服务不同受众时使用。显式指定受众——不要依赖模型从上下文中猜测。

**边界模式**：用于生产系统，这些系统绝对不能回答某些类型的问题。比护栏更强，因为它用精确的拒绝消息定义了硬范围。对于合规敏感的领域至关重要。

## 跨模型兼容性

按在 GPT-4o、Claude 3.5 Sonnet、Gemini 1.5 Pro 和 Llama 3 上的一致工作程度排序的模式：

| 模式 | 跨模型一致性 | 说明 |
|---------|------------------------|-------|
| 少样本 | 非常高 | 示例在所有模型间迁移良好 |
| 模板填充 | 非常高 | 显式结构几乎没有发散空间 |
| 思维链 | 高 | 所有主要模型都支持"逐步思考" |
| 角色 | 高 | 到处都有效，但不同模型对角色具体程度的响应不同 |
| 护栏 | 中等 | Claude 最严格地遵循护栏；GPT-4o 在长对话中有时会偏离 |
| 批判 | 中等 | 自我批判的质量因模型而异 |
| 元提示 | 中等 | GPT-4o 和 Claude 产生不同的提示词风格 |
| 边界 | 低-中等 | 拒绝行为各不相同；按模型测试 |

## 常见错误

1. **对所有内容使用思维链**：CoT 增加 token 和延迟。仅在需要推理步骤时使用。
2. **约束过多**：超过 5-7 个约束，模型开始遗漏一些。优先考虑最重要的 3 个。
3. **矛盾的角色 + 约束**："你是一个创意作家" + "永远不要使用隐喻"会让模型困惑。
4. **没有指定温度**：在需要确定性输出时将温度保留为默认值（通常为 1.0）。
5. **跨模型复制粘贴提示词**：始终测试。为 GPT-4o 调优的提示词在 Claude 上可能表现不佳，反之亦然。
6. **忽略系统消息**：将所有内容放在用户消息中，而不是使用系统消息来传递持久规则。
7. **过度依赖负面约束**："不要做 X、Y、Z、A、B、C"不如"只做 W"有效。正面框架给模型一个明确的目标。

## 可靠性目标

| 用例 | 模式组合 | 预期准确率 | Token 成本 |
|----------|-------------------|-------------------|------------|
| 生产提取 | 模板 + 少样本 | 95%+ | 低（500-1K） |
| 面向用户的问答 | 角色 + 护栏 + 边界 | 90%+ | 中等（1-2K） |
| 代码生成 | 角色 + 思维链 | 85%+ | 中等（1-3K） |
| 内容生成 | 角色 + 批判 | 90%+ 质量 | 高（2-4K，双遍） |
| 分类 | 少样本 + 护栏 | 95%+ | 低（300-800） |
| 复杂分析 | 分解 + 思维链 | 85%+ | 高（3-5K） |
