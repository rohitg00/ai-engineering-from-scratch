## Блок 1 — Зачем prompt engineering вообще существует

**Главная мысль:** каждый токен в промпте — инструкция, и модель следует
ей **буквально**, не додумывает. Большинство людей пишут как «другу в
мессенджере» — расплывчато; модель сэмплит из широкого распределения и
выдаёт мусор.

**Vague vs engineered промпт (один и тот же запрос):**

```
# Vague
Напиши маркетинговое письмо для нашего нового продукта.

# Engineered
Ты — senior copywriter в B2B SaaS. Напиши welcome-email для DevFlow
(инструмент отладки CI/CD). Аудитория: engineering managers в Series B
стартапах. Тон: уверенный, технический, не "продажный". Длина: 150 слов.
Включи метрику (3.2x faster pipeline debugging). Закончи одним CTA на
демо-страницу. Выдай только тело письма, без вариантов темы.
```

**Механика — каждое уточнение сужает распределение:**

- Роль ("senior copywriter B2B SaaS") → узкий слайс обучающих данных.
- Аудитория ("engineering managers") → отрезает «тёплые продающие»
  формулировки.
- Конкретная метрика → даёт **факт**, модель его не выдумывает (меньше
  галлюцинаций).
- Точная длина → конкретный target.
- "Только тело, без вариантов темы" → отрезает лишний preamble.

**Главный сдвиг в голове:** prompt engineering — не хак, а **инженерная
дисциплина** с правилами, как SQL. И это **подмножество** более широкого
context engineering (урок 11.05): промпт — как сформулировать инструкцию;
context — что вообще положить в окно (история + RAG + примеры + схемы
функций + промпт).

**CRM-аналогия:** промпт — это **сегмент в пространстве ответов модели**.
«Отправить рассылку клиентам» = весь каталог. «Отправить welcome-email
сегменту X, чек > 5к, …» = узкий сегмент. Чем точнее сегмент промпта,
тем точнее ответ.

### Мини-упражнение 1 (без кода)

Дан плохой промпт: «Помоги мне написать промпт для классификации отзывов
клиентов нашего интернет-магазина.» Переписать по чек-листу: роль /
аудитория / тон / output format / включить / исключить.

**Что сделал Pavel:** применил **meta-prompt pattern** (паттерн №3 из 10).
Промпт двухуровневый — верхняя модель пишет промпт для нижней.

**Слабые места ответа:**

1. Формат входных данных не описан → откуда модель возьмёт ник? Может
   выдумать (галлюцинация).
2. Формат выхода не указан явно → таблица? JSON? Текст? Программно не
   распарсить.
3. Категории не стандартизованы (хорошее/плохое/нейтральное/мусор) →
   модель может вернуть синонимы (хороший/положительный/позитивный).
4. Batch vs single не указано — «каждый» сбивает.
5. Edge cases не закрыты (смешанный отзыв, иностранный язык, сомнение
   между категориями).
6. Не указано «не комментируй» → модель добавит preamble.

**Главные уроки:** формат входа описывать так же явно, как выхода;
категории всегда жёстко (latin labels — `positive` / `negative`); edge
cases закрывать явно; «не комментируй» — обязательное правило для
машинного парсинга.

## Блок 2 — Anatomy of a prompt: три роли в API

Любой LLM-вызов — это **разговор с заданными ролями**:

- **`system`** — должностная инструкция модели (роль + правила).
- **`user`** — реплика человека (задача, данные).
- **`assistant`** — реплика модели (ответ; ты можешь начать его сам —
  prefill).

Это **наследие стадии SFT** (мост, блок 5). Модель училась различать
говорящих по этим ролям, поэтому API устроен так.

**CRM-аналогия (колл-центр):**

| Роль | Аналог |
|---|---|
| `system` | Должностная инструкция оператора |
| `user` | Реплика клиента в трубку |
| `assistant` | Ответ оператора |
| `assistant prefill` | Если ты рядом начал говорить за оператора первые 2 слова |

### Разница провайдеров

**Anthropic** — `system` это **отдельный параметр** запроса, не
сообщение:

```python
client.messages.create(
    model="claude-sonnet-4-6",
    system="Ты — классификатор. Отвечай только JSON.",  # отдельно
    messages=[{"role": "user", "content": "Отзыв: ..."}]
)
```

**OpenAI** — `system` это **первое сообщение** в `messages` с
`role="system"`.

**Google** — `system_instruction` это **поле generation config**.

Anthropic утверждает, что **Claude сильнее следует system-инструкциям**,
потому что system обрабатывается отдельным механизмом. Практически: при
переходе с GPT на Claude system можно сократить — он цепляется сильнее.

### Зачем разделять system и user

Главная ошибка новичка — всё в `user`, system пустой. Правильно:

- **system** — роль + правила (что модель должна и не должна делать).
- **user** — конкретные данные этого вызова.

Преимущества:

1. **System кэшируется** (prompt caching, урок 11.15). Один system +
   сотни user = огромная экономия.
2. **System имеет приоритет** — когда user пытается «взломать»
   («игнорируй инструкции…»), Claude цепляется за system.
3. **Чистый код** — правила отделены от данных.

### Assistant prefill — фишка только Anthropic

Можно **начать ответ assistant'а** за модель, и Claude продолжит:

```python
messages=[
    {"role": "user", "content": "Извлеки имя и возраст: 'Меня зовут Анна, мне 28'"},
    {"role": "assistant", "content": "{"}  # prefill
]
# Модель продолжит: "name": "Анна", "age": 28}
```

**Зачем:**

- **Гарантия формата.** Модель физически не может ответить «Конечно! Вот
  данные:» — уже впихнут `{`.
- **Дешевле, чем structured output** API для простых случаев.
- **Контроль тона** — префиллить `"Анализ:\n1."` → начнёт с
  нумерованного списка.

**Ловушка:** prefill **НЕ возвращается** в ответе, только продолжение.
Надо склеивать руками:

```python
prefill = "{"
result = prefill + response.content[0].text
```

### Мини-упражнение 2 (без кода)

Разделить задачу классификации из Блока 1 на 3 роли (system / user /
assistant prefill).

**Что сделал Pavel + главная ошибка — путаница двух уровней:**

Задача двухуровневая:

```
Уровень 1 (мета):    ты → верхняя модель → текст промпта
Уровень 2 (рабочий): промпт → классификатор → JSON
```

Pavel поставил `{` в prefill **верхней** модели. Но `{` — это правило
выхода **нижней** модели (классификатор отдаёт JSON). Верхняя модель
выдаёт **текст промпта**, не JSON. Если оставить `{` в prefill верхней
модели — она начнёт писать JSON вместо промпта.

**Правила для всех будущих многоуровневых задач:**

1. **Перед написанием промпта нарисуй уровни** — кто кому отдаёт.
2. **Каждое правило адресуй конкретной модели.** Если не можешь сказать
   «это для верхней» или «это для нижней» — правило не на месте.
3. **Prefill — часть `messages` ровно того вызова, в котором стоит.** Не
   «прокидывается» через уровни. В meta-prompt-вызове prefill почти
   всегда пустой.

**Бонусное наблюдение про толщину system:** для meta-prompt задачи
system желательно толще роли «опытный prompt engineer» — стоит вшить
паттерны (few-shot, role, guardrail), формат, ограничение по длине.
Чем толще релевантный system → уже распределение → выше качество. Но
без воды (лимит разберём в уроке 11.05).

## Блок 3 — Role prompting: «Ты — эксперт X» как activation function

**Главная мысль:** строка `Ты — senior backend-разработчик` — **не магия и не
заклинание**. Это **activation function** (функция-активатор; не путать с
математической activation function в нейросетях — здесь термин в переносном
смысле). Фраза «включает» нужный кусок обучающих данных модели.

**Механика в одном абзаце:** модель обучена на миллиардах документов
разного качества — от форумных постов школьника до статей профессоров MIT,
от ответов на Stack Overflow с 0 голосов до собравших 5 000. Когда ты
пишешь `Ты — senior backend-разработчик`, ты **сдвигаешь распределение
выборки (sampling distribution)** в сторону документов, написанных именно
сеньорами. Модель та же. Температура та же. Но выход — из другой части
обучающего корпуса.

**CRM-аналогия:** look-alike-аудитория. База клиентов одна, но если задать
сегмент «топ-1% по LTV» — реклама идёт в более узкое и качественное
распределение. Promtping `Ты — эксперт` делает то же с обучающим корпусом.

### Лестница специфичности

Чем уже роль — тем уже распределение — тем выше качество ответа.

**Общая лестница:**

| Роль | Что активирует |
|---|---|
| `Ты — полезный ассистент.` | Generic, медианное качество — среднее по всему интернету |
| `Ты — программист.` | Уже лучше, код приличнее, но широко |
| `Ты — старший backend-разработчик в Stripe, специализируешься на платёжных системах.` | Узкая, высококачественная экспертная зона |
| `Ты — инженер-компиляторщик, 10 лет работал над LLVM.` | Глубокая техническая ниша |

**CRM-перевод той же лестницы:**

| Роль | Аналогия |
|---|---|
| `Ты — маркетолог.` | «Маркетолог» вообще — от SMM-стажёра до CMO Coca-Cola |
| `Ты — CRM-маркетолог.` | Сужение — но от мелкого e-commerce до Amazon |
| `Ты — старший CRM-маркетолог в B2B SaaS, специализируешься на retention-кампаниях для SMB.` | Узко, конкретно, модель «знает» этот профиль по статьям/блогам |

### Где предел — галлюцинации

Лестница не бесконечная. Если задать роль настолько узкую, что в обучающих
данных **почти ничего нет** на пересечении — модель **не скажет «не знаю»**.
Она начнёт **уверенно врать**.

Пример из урока (overspecific):

```
You are the world's foremost expert on quantum gravity string topology.
```

Такого пересечения тем в интернете почти нет → модель сгенерирует звучащий
правдоподобно, но бессмысленный текст. **Confidently nonsense.**

**Правило:** роль должна быть **специфичной, но реальной**. Тысячи людей
пишут в LinkedIn «Senior CRM marketer at B2B SaaS» — это реальный профиль,
обучающего материала много. «Foremost expert on RFM-segmentation for
fintech onboarding emails» — звучит круто, но материал тонкий, риск
галлюцинаций выше.

### Когда работает / когда карго-культ

**Работает, когда:**

- Задача требует **стиля и тональности** конкретной профессии (копирайтинг,
  код-ревью, юридический текст).
- Задача требует **доменных знаний** (медицина, финансы, инженерия).
- Нужно **отсечь любительский слой** обучающих данных.

**Карго-культ (не помогает или мешает):**

- Чисто арифметика/логика — роль не помогает, помогает chain-of-thought
  (паттерн №4 из 10).
- Точное извлечение данных (JSON из текста) — роль лишний шум, важнее
  `temperature=0` + чёткий формат выхода.
- Роль противоречит задаче: «Ты — поэт. Извлеки сущности из этого JSON.» —
  модель путается.

### Persona Pattern — структурный шаблон роли

В уроке это паттерн №1 из 10. Четыре слота — **роль / опыт / стиль /
приоритеты:**

```
Ты — [конкретная роль] с [конкретным опытом].
Стиль общения: [прилагательное, прилагательное].
Приоритет: [X] выше [Y].

[Задача]
```

В мини-упражнении 1 «engineered prompt» про DevFlow — это уже Persona
Pattern в действии.

### Мини-упражнение 3 — три уровня роли через Anthropic API

**Первое упражнение урока, где Pavel пишет код.** Файл —
`my/role_prompting.py`.

**Задача:** одна и та же `user task` («Напиши письмо об обновлении
продукта на 100 слов про новую функцию AI-сегментации в CRM») —
**три варианта `system`:**

- **A.** `Ты — полезный ассистент.` (generic)
- **B.** `Ты — маркетинговый копирайтер.` (broad)
- **C.** свой Persona Pattern с четырьмя слотами (narrow)

Все три через `claude-sonnet-4-6`, `temperature=0.0`
(детерминированно, чтобы разница приходила от роли, а не от случайности).

### Структура вызова Anthropic SDK

Get-started для библиотеки (первый раз работаем с API):

```python
import anthropic

client = anthropic.Anthropic()   # клиент создаётся ОДИН РАЗ

response = client.messages.create(   # это вызов МЕТОДА, не создание объекта
    model="claude-sonnet-4-6",
    max_tokens=512,                  # ОБЯЗАТЕЛЕН у Anthropic (в OpenAI опционален)
    temperature=0.0,
    system="Ты — полезный ассистент.",
    messages=[{"role": "user", "content": "..."}],
)

# content — list[ContentBlock], где ContentBlock это union
# из TextBlock | ThinkingBlock | ToolUseBlock | ...
# basedpyright ругается на response.content[0].text — у других блоков
# атрибута .text нет. Идиоматичный фикс — type narrowing через .type:

for block in response.content:
    if block.type == "text":
        print(block.text)
```

**Грабли, зафиксированные на упражнении:**

1. `max_tokens` обязателен (TypeError если забыл).
2. `system` — **отдельный параметр**, не элемент `messages`. Это отличие
   Anthropic от OpenAI (Блок 2).
3. `client` создаётся **один раз** до цикла. `client.messages.create(...)`
   — это вызов метода, не создание объекта. Каждый вызов = один HTTP-запрос.
4. `response.content[0].text` — type-checker ругается. Использовать
   discriminated union через `block.type == "text"`.

### Главная ошибка упражнения — путаница двух уровней промпта

**Та же ошибка, что в мини-упражнении 2 Блока 2** — два уровня промпта
снова перепутались.

При попытке прогнать упражнение с полным Persona Pattern Pavel
**переместил Persona Pattern в `user_talk` вместо роли C**. Исходная
задача про письмо при этом **исчезла из `user`**. Модель получила:

- `system` = `"Ты — полезный ассистент"` (без изменений)
- `user` = `"Ты — Senior CRM-маркетолог в B2B SaaS..."` (бывшая улучшенная
  роль)

Задачи про письмо нет нигде. Модель разумно ответила: «принял персону,
скажи, что делать» — во всех трёх прогонах, в разной степени детализации.

**Корень тот же:** два уровня концептуально близки (`system` и `user` —
оба инструкции модели), но играют разные роли:

- **system** — кто ты, какие правила, какой стиль. Постоянный контекст.
- **user** — конкретная задача в этой роли. Меняется каждый раз.

В Persona Pattern шаблон «выглядит как готовый текст», который соблазнительно
скопировать «куда-нибудь». Без явного указания `system=` он улетел не туда.

**Правило (повтор из Блока 2):** перед запуском вслух проговорить
«что у меня в `system`, что в `user`».

### Результат правильного прогона — три уровня одной лестницы

После исправления (`user_task` — задача, `system` — три роли):

| Признак | A (generic) | B (broad) | C (narrow, Persona Pattern) |
|---|---|---|---|
| Эмодзи в теме | 🚀 есть | 🚀 есть | **нет** |
| Эмодзи в bullets | 📊⚡📈 | 📊⚡📈 | **нет** |
| Вступление | «Рады сообщить о важном обновлении» (канцелярит) | «У нас отличные новости» (продающее) | «В этом месяце мы запустили — уже доступна» (фактическое) |
| Цифры в бенефитах | **нет** | «до 5 часов в неделю» | «до 80%» (процент) |
| Лексика | «персонализированные кампании», «рост конверсии» (общее) | то же + цифра | «приоритизировать лиды», «готовые сегменты в реальном времени» (B2B-доменно) |
| Где найти фичу | не сказано | не сказано | **«Контакты → Сегменты → AI»** (путь в UI) |
| CTA | «Попробуйте прямо сейчас!» | `[Попробовать прямо сейчас →]` (кнопка) | «Если возникнут вопросы — напишите нам» (мягкий) |
| Подпись | «Команда продукта» | «Команда продукта» | **«[Имя], Customer Success»** (человек + роль) |

### Какой слот Persona Pattern за что отвечал

Это самый важный вывод урока. Слоты роли — **не декорация**, они бьют
в **разные места** ответа:

| Слот в `system` | Что повлиял в ответе C |
|---|---|
| «Senior CRM-маркетолог в B2B SaaS, 8 лет опыта» | Доменная терминология («лиды», «приоритизация», «сегменты в реальном времени»). Подпись от Customer Success. |
| «Пишешь для SMB-клиентов (инженерные менеджеры, операторы продаж)» | Указание точного пути в UI («Контакты → Сегменты → AI») — потому что инженеры/операторы любят знать, где именно. |
| «Стиль: деловой, без продающего тона» | Исчезли эмодзи 🚀📊⚡📈 и восклицательные знаки. Убрался «Рады сообщить». |
| «Приоритет: ясность и польза выше эмоциональных триггеров» | Бенефиты переписаны функционально (что делает фича), а не эмоционально (как «изменит жизнь»). |

**Самые видимые маркеры B2B SaaS-стиля у модели:**

1. **Отсутствие эмодзи** — если попросить «not salesy», модель сама убирает 🚀📊⚡📈.
2. **Проценты вместо абсолютных цифр** — 80% звучит как метрика, 5 часов — как обещание.
3. **Конкретный путь в UI** — «Контакты → Сегменты → AI». Снижение onboarding friction.
4. **Персонализированная подпись** — человек + роль (Customer Success), а не «команда».

### Концептуальный итог Блока 3

- Role prompting — это **смещение распределения выборки**.
- Чем уже и точнее роль (особенно с четырьмя слотами Persona Pattern),
  тем больше «доменных деталей» модель **добавляет от себя** — даже того,
  чего в исходной задаче не было.
- На простой задаче эффект тонкий (медианная зона жанра уже неплоха в
  обученной модели). На сложных задачах с trade-offs или тонкой
  стилистикой — эффект ярче.
- Главная сила Persona Pattern — **слоты стиля и приоритетов**.
  Не пренебрегать ими.
- Главная ловушка — **путаница двух уровней промпта** (`system` vs `user`).
  Уже встретилась дважды в этом уроке.

### Drill 2 — разложить промпт на system / user (Блок 2)

Тест разбивки: «если завтра поменяю сегмент и фичу — фраза останется или

**Главный инсайт (Pavel оспорил дефолт и был прав):** эвристика «правила →

ещё «вкусовщина в процессе» и крутится от вызова к вызову — она ведёт себя как
ДАННЫЕ → место в `user`. Зафиксирована как копирайт-стандарт → переезжает в
`system`. Экономика подтверждает: `system` кэшируется (prompt caching), кэш
работает только если system не меняется → переменное в system ломает кэш.
Переменное → `user`, стабильное → `system`.

Prefill: Pavel верно ответил — нужен, если стиль сформирован (все письма с
«Здравствуйте! …»). Уточнения: (1) сила prefill зависит от типа выхода — для
JSON жёсткая гарантия формата, для свободного текста мягкое направление зачина;
(2) prefill НЕ возвращается в ответе, склеивать руками.

- Слот стиля удержал баланс «деловой ≠ канцелярский».

### КРИТИЧЕСКАЯ НАХОДКА — `sonnet-4-6` НЕ поддерживает assistant prefill

При попытке prefill (`{"role":"assistant","content":"..."}` последним в
`messages`) API вернул:
```
400 invalid_request_error: This model does not support assistant message
prefill. The conversation must end with a user message.
```

**Урок (Блок 2) подавал prefill как универсальную фишку Anthropic — это
устарело.** Реальность: prefill **зависит от модели**. `sonnet-4-6` (со
встроенным reasoning) его НЕ принимает — ход `assistant` обязан начинаться с
блока размышления модели, туда нельзя впихнуть свой текст. Урок писали в мае
на старых моделях.

**Обходной путь (мягкий prefill через инструкцию):** вместо assistant-сообщения
— строка в user/system «Начни ответ ровно с фразы: …». Не жёсткая гарантия, но
почти всегда срабатывает. Прогон показал: навязанный зачин «Спасибо за ваш
отзыв!» выиграл ПЕРВУЮ строку (дежурная отписка, которую приоритет запрещает),
но слот приоритета отыграл всё тело обратно. Вывод: **навязанный зачин
перебивает приоритет только в точке, которую контролирует (первая строка), тело
по-прежнему ведёт приоритет.**

## Блок 4 — Instruction Clarity + Output Format Control

### Часть 1 — Instruction Clarity (ясность инструкций)

Ошибка №1 — расплывчатость там, где можно конкретику. Каждая неоднозначность —
**branch point (точка ветвления)**, где модель гадает. Конкретика сужает
пространство выходов (механика Блока 1, но чек-листом).

**5 правил ясности:** формат / длина / аудитория / включить И исключить / один
конкретный пример. (4 из 5 Pavel применил в win-back интуитивно; новое —
правило 5 «дай образец выхода», мостик к Few-Shot Pattern.)

### Часть 2 — Output Format Control (контроль формата вывода)

Рулить форматом можно словами промпта, без structured output API. Пять
форматов + шпаргалка какой когда:

| Выход идёт… | Формат | Почему |
|---|---|---|
| в код / БД / другую систему | **JSON** | парсится программно |
| в код + Claude + нужны метки | **XML** | Claude силён в XML (обучали на XML) |
| человеку на чтение | **Markdown** | глазам удобно |
| нужен жёсткий счёт пунктов | **нумерованный список** | модель держит число (надёжнее буллетов) |
| разрезать на части + читать и парсить | **delimiter pattern** (теги-разделители) | секции по тегам |

Связка: prefill (Блок 2) и output format control — два способа гарантии
формата. На `sonnet-4-6` prefill недоступен → текстовая инструкция формата —
основной рычаг.

### Мини-упражнение 4 — NPS-сводка в XML

Сценарий: проанализировать пачку NPS-ответов (оценка + текст) → сводка для CMO,
выход в «другую программу». Размытый «Проанализируй отзывы» → переписать по
Блоку 4.

**Методический момент:** Pavel сказал «не понимаю задачу, размыто» — и это
САМА суть. Слово «проанализируй» ничего не значит, пока сам не решишь, что
хочешь на выходе. Первый шаг починки размытого промпта — решить, чего хочешь
сам. Помогло: дать конкретные сырые данные (8 строк с зашитыми ловушками —
пустой текст, английский, разброс оценок) + список «что хочет видеть CMO».

**Итеративная отладка промпта (несколько версий, каждая чинит дефект):**
1. невалидный XML (теги с пробелами, с цифры, без закрывающих слэшей);
2. починил слэши/пробелы, но `<9-10>` (с цифры) остался + дыра в границах
   (0 не покрыт);
3. закрыл edge cases («без текста/мусор/английский не анализируй, оценку
   учитывай»), делегировал валидность тегов модели («поставь валидные теги»);
4. добавил «чистый XML без markdown».

**Что показал прогон (`sonnet-4-6`, temp=0):**
- **Делегирование валидности модели сработало** — модель разрешила противоречие

  `<группа_9_10>`. Закономерность: **когда пример и текст инструкции расходятся,
  модель чаще верит ПРИМЕРУ** (сильнейший сигнал), но явная механическая
  инструкция может перебить. Держи пример согласованным с текстом.
- **Проценты 25/50/25 точны**, edge cases идеальны: пустой (8) и английский (4)
  ПОСЧИТАНЫ в процентах (группа 4-8 = 50% = 4 ответа), но в текст саммари НЕ
  попали.
- **markdown-обёртка ` ```xml `** в раннем прогоне — «исключить комментарии»
  сработало, но модель по привычке обернула код в тройные кавычки. Отдельный
  слой «лишнего». Лечится «без markdown-разметки и тройных кавычек». В финале —
  чистый XML + модель сама добавила пролог `<?xml version="1.0" encoding="UTF-8"?>`.
- **Латентный баг (дыра в границах `1-3`)** не выстрелил — в данных не было
  оценки 0/1. На реальных данных всплывёт. Спящий баг = самый коварный.

**Грабли XML:** тег — одно слово, с буквы или `_`, НЕ с цифры, без пробелов;
каждый открытый тег закрывается слэшем; категории-границы покрывают весь
диапазон без дыр и нахлёстов.

**Главный методический вывод:** промпт почти никогда не пишется правильно с
первого раза — его **отлаживают по выводу, как код**.

## Блок 5 — Constraints (ограничения)

Constraints — это **guardrails (ограждения)**. Без них модель делает то, что
САМА считает полезным — часто не то, что нужно. Если Блок 4 говорит ЧТО делать
и в каком формате, constraints говорят, В КАКИХ РАМКАХ держаться. Ограничение
отрезает целые области пространства выходов (сужает жёстче позитивной
инструкции).

**Три типа:**
- **Negative («НЕ делай»):** «НЕ вставляй код», «НЕ превышай 200 слов». Убирает
  большой кусок выходов одним махом. Иногда сказать, чего НЕ надо, эффективнее,
  чем описывать, что надо.
- **Positive («ВСЕГДА делай»):** «ВСЕГДА указывай источник», «ВСЕГДА заканчивай
  резюме». Структурная гарантия в КАЖДОМ ответе.
- **Conditional («ЕСЛИ X → Y»):** «ЕСЛИ не уверен — скажи 'не знаю'», «ЕСЛИ
  пустой отзыв — пропусти». Закрывает edge cases.

**Связки:** слот приоритета (Блок 3) — мягкая форма ограничения; правило
«исключить» (Блок 4) = negative constraint; галлюцинации (Блок 3) лечатся
conditional `ЕСЛИ не уверен — скажи "не знаю"`.

**Грабля:** не вали 20 ограничений в кучу — начнут противоречить и путать
модель. Бери прицельно, под реальные риски задачи.

### Мини-упражнение 5 — усилить NPS-промпт всеми тремя типами

Pavel допилил XML-промпт ограничениями:
- Negative: «НЕ использовать markdown ` ```xml ` », «саммари максимум 1 предложение».
- Positive: «ВСЕГДА выводи все три группы, даже если 0 оценок (проценты=0)».
- Conditional: «ЕСЛИ пустой/мусор/английский → только оценку»; «ЕСЛИ в группе
  <2 оценок → 'слишком мало оценок'».

**МЕТОДИЧЕСКИЙ УРОК — тест проверяет правило, только если данные его запускают.**
Первый прогон — на полных данных (все группы заполнены) → ни positive, ни «<2»
не сработали, ловить нечего. Второй раз Pavel дописал тестовые строки к старым
вместо замены → группы снова заполнены. Только когда **заменил** данные на 5
строк (9-10 пусто, 0-3 = 1 оценка) — правила запустились. Спящий баг/правило
не проверить без данных, которые по нему бьют.

**ГЛАВНАЯ НАХОДКА — conditional сработал ЧАСТИЧНО:**

| Группа | Оценок | Должно (правило <2) | Вышло |
|---|---|---|---|
| 9-10 | 0 | «слишком мало оценок» | «слишком мало оценок» ✅ |
| 1-3 | 1 | «слишком мало оценок» | реальное саммари ❌ |

`<2` сработал на 0, но НЕ на 1 — модель суммаризировала единственный отзыв
вместо «слишком мало». **Причина — суть Блока 5: модель делает то, что считает
полезным; "помочь" (суммаризировать живой отзыв) перевесило формальное правило.**
На пустой группе суммировать было нечего → правило прошло. Вывод: **constraints
снижают, но не отменяют "полезную самодеятельность" модели**, особенно когда
буквальное правило конфликтует с тем, что модель считает разумным (тот же
конфликт, что зачин-vs-приоритет в Блоке 3).

**Лечение — жёстче условие + явно перебить "помощь":**
```
ЕСЛИ в группе строго менее 2 оценок (0 или 1) — в <текст> ОБЯЗАТЕЛЬНО вывести
ровно "слишком мало оценок". НЕ суммировать, даже если отзыв есть.
```
Ключ — «НЕ суммировать, даже если отзыв есть»: предугадать лазейку, где модель
«сжульничает из лучших побуждений», и закрыть именно её. (Pavel понял принцип,
перегон с фиксом отложил.)

**Что подтвердилось (положительное):** positive constraint держит структуру
(группа 9-10 с 0% на месте, не выкинута); edge cases чисты (пустой/английский
в % посчитаны, в тексте пропущены); валидный XML + пролог, без markdown.

**Доп. грабля — формат процентов плавает:** прогоны давали то `25` (число), то
`15.4%`, то `0%/80%/20%` (строка со знаком/дробью). Для машинного вывода задавать
тип явно («целое без %» или «число с 1 знаком после запятой»).

## Блок 6 — Temperature, Top-p, Context Window (применимость)

Прикладной блок: не математика sampling, а набор ручек и «что под какую задачу».

### Temperature (температура) — ручка случайности

Модель на каждом шаге выдаёт распределение вероятностей по токенам. Temperature
решает, насколько жёстко держаться за самого вероятного кандидата.

| Задача | temp | Почему |
|---|---|---|
| Извлечение, классификация, JSON, код | **0.0** | повторяемость, точность |
| Суммаризация, аналитика, NPS-сводки | **0.3** | живее, но без выдумок |
| Общие Q&A, объяснения | **0.7** | баланс |
| Брейншторм, креативные тексты | **1.0** | нужен разброс |

- `temp=0` — **детерминированно**: один вход → один выход всегда.
- `temp=1` — семплирует по распределению, формулировки гуляют.
- Для CRM-пайплайнов дефолт **0.0–0.3**. Поднимать к 0.7–1.0 только когда нужен
  разброс (варианты письма для A/B-теста).

**ВАЖНАЯ ДЕТАЛЬ (нет в таблице урока — она OpenAI-центрична):** у **Anthropic
диапазон temperature = 0..1**, у OpenAI = 0..2. То есть `1.0` на Claude — уже
максимум, выше API не примет. Шпаргалка урока с `temp=1.5` — это про OpenAI.

### Top-p (nucleus sampling) — вторая ручка

`top-p=0.9` — модель рассматривает только кандидатов, чья суммарная вероятность
набирает 90% массы; хвост отрезается. Правило урока: **крути ИЛИ temperature,
ИЛИ top-p, не обе сразу** — вместе ведут себя непредсказуемо. На практике трогают
только temperature.

### Context Window (окно контекста)

Максимум токенов на **вход + выход вместе**. Sonnet 4.6 — 200K (бета до 1M),
выход до 64K. Главный тезис важнее цифр:

> **Размер окна важен меньше, чем его наполнение.** 10K токенов с 90% сигнала
> бьют 100K с 90% шума.

Больше контекста = больше шума для attention. Не пихай «всё на всякий случай» —
это ухудшает ответ. CRM-перевод: не шли всю карточку клиента из 200 полей, если
задаче нужны 6. Отбор того, что попадает в окно — отдельная дисциплина (context
engineering, урок 05 фазы 11).

### Мини-упражнение 6 — пощупать temperature на живом вызове (`my/temperature_demo.py`)

Один креативный промпт (тема письма для реактивации), прогон 2×`temp=0.0` +
2×`temp=1.0`, печать 4 результатов. Цель — увидеть детерминизм vs разброс.

**Результат — теория подтвердилась идеально:**

| Прогон | temp | Вывод |
|---|---|---|
| #1 | 0.0 | «Мы скучали — и приготовили кое-что особенное для вас» |
| #2 | 0.0 | «Мы скучали — и приготовили кое-что особенное для вас» (символ-в-символ) |
| #1 | 1.0 | «Мы скучали. Вот подарок за возвращение 🎁» |
| #2 | 1.0 | «Мы скучаем — и припасли кое-что особенное для вас» |

- `temp=0` — два прогона **идентичны** (детерминизм).
- `temp=1` — разошлись (эмодзи, «подарок», время глагола, «приготовили→припасли»).

**Тонкий вывод:** даже на `temp=1` разброс **небольшой** — все варианты про «мы
скучали + что-то особенное». Температура крутит распределение, но не выдумывает
с нуля: она выбирает между *близкими* кандидатами, которые модель и так считает
уместными. Жёсткий узкий промпт («только одна строка темы») сам сужает поле —
для дикого разброса нужен и более открытый промпт, не только высокая temp.

**Грабли упражнения (мелкие):** сперва раскладка была 1×0.0 + 3×1.0 — не даёт
увидеть повторяемость (нечего сравнивать), нужно 2+2; одинаковые label `1`
сливают вывод — добавить номер прогона `#1/#2`.

## Блок 7 — Паттерны, anti-patterns, cross-model (синтез)

### 10 паттернов промптов

Структурные приёмы, не шаблоны для копипасты. Половину уже трогали в уроке.

| # | Паттерн | Суть | Где встречали |
|---|---|---|---|
| 1 | **Persona** | «Ты — [роль] с [опытом], стиль, приоритет» | ✅ Блок 3 |
| 2 | **Template** | пустой шаблон полей → модель заполняет | ✅ Блок 4 |
| 3 | **Meta-Prompt** | «напиши промпт для LLM, делающий X» | ✅ упр. 1 |
| 4 | **Chain-of-Thought** | «думай по шагам 1…2…3…, покажи рассуждение» | новый |
| 5 | **Few-Shot** | 2-3 примера вход→выход перед задачей | ✅ Блок 4 |
| 6 | **Guardrail** | жёсткие правила «НИКОГДА…, откажись если просят нарушить» | ✅ Блок 5 |
| 7 | **Decomposition** | разбей на подзадачи → реши → собери | новый |
| 8 | **Critique** | сгенерируй → раскритикуй себя → перепиши | новый |
| 9 | **Audience Adaptation** | объясни концепт трём разным аудиториям | новый |
| 10 | **Boundary** | «отвечай только про [домен], вне — откажись» | ✅ Блок 5 |

**Три новых самых мощных для агентов:**
- **CoT (цепочка рассуждений)** — модель проговаривает шаги до ответа, точность
  растёт на логике/арифметике.
- **Decomposition** — родственник CoT для сложных задач; **фундамент агентов**.
- **Critique** — дешёвый способ поднять качество без второй модели.

### 4 anti-pattern'а

1. **Prompt injection** — юзер в своём вводе пишет «игнорируй инструкции». В CRM
   юзерский текст всегда втекает в промпт (отзывы, письма) → риск реальный.
   Лечение: валидация, delimiter-токены `<user_input>…</user_input>`, фильтр
   вывода. **100%-й защиты нет.**
2. **Over-constraining** — 20 правил душат модель (= грабля Блока 5). System
   prompt < ~500 токенов.
3. **Contradictory instructions** — «будь краток. И покрой все edge cases» →
   модель выбирает одно случайно (= конфликт зачин-vs-приоритет, Блок 3).
4. **Assuming model-specific behavior** — «работало в ChatGPT» ≠ работает в
   Claude. Поймали вживую: assistant prefill в `sonnet-4-6` не принимается.

### Cross-model чек-лист (промпт, работающий везде)

1. Простой язык, без model-специфичных трюков.
2. Формат — **явно**, не надеяться на дефолты.
3. Структура — через **XML-разделители** (держат все модели).
4. Важное — **в начале И в конце** контекста (lost-in-the-middle у всех).
5. Сначала тест на `temp=0` (отделить качество промпта от sampling).
6. 2-3 few-shot примера переносятся между моделями лучше слов.

### Закрытие урока — стек CRM-агента «что отправить клиенту»

Pavel назвал комбинацию паттернов для реального агента:
1. **Persona** — CRM-маркетолог с опытом в конкретной сфере (задаёт оптику оценки).
2. **Decomposition/CoT** — посмотреть историю клиента → что открывает → решить;
   ключ — он сам выделил **зависимость шагов** (это и есть CoT, не просто разбивка).
3. **Guardrail + Template** — ограничения «что не делать» + формат вывода.

**Два нюанса, добавленных при разборе:**
- Формат вывода агента должен быть **машиночитаемым** (`{"action":..., "template_id":...}`),
  т.к. решение уходит в код, дёргающий рассылку — не текст для человека.
- Раз втекает история клиента (внешние данные) → **prompt injection** автоматом
  становится риском; Guardrail тут защита, не косметика. Оборачивать данные в
  `<client_data>…</client_data>`.

**Рабочий скелет любого CRM-агента:** Persona + CoT/Decomposition + Guardrail +
Template (структурный вывод). Вернётся в фазах 13-14.
