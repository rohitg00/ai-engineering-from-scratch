# 智能体（Agent）框架权衡：LangGraph vs CrewAI vs AutoGen vs Agno

> 每个框架都推销同一个演示（研究智能体生成报告），并隐藏同一个缺陷（状态模式与编排层冲突）。选择其抽象机制与你问题形态匹配的框架；其余的一切都是你会重复编写的胶水代码。

**类型：** 学习
**编程语言：** Python
**先修课程：** 第11阶段·09（函数调用），第11阶段·16（LangGraph）
**预计用时：** ~45分钟

## 问题所在

当你有一个需要不止一次LLM调用的任务时，或许是一个研究工作流（规划、搜索、总结、引用），或许是一个代码审查管道（解析差异、评审、修补、验证），或者是一个执行预订航班、撰写邮件和提交报销的多轮助手。你选择了一个框架。

三天后，你发现框架的抽象机制存在泄漏。CrewAI给了你角色，但当"研究员"需要将一个结构化计划交给"写手"时却难以协作。AutoGen提供了智能体之间的对话，但没有一等公民的状态，因此你的检查点就是一个对话日志的序列化对象。LangGraph提供了状态图，但迫使你在了解智能体将做什么之前就要命名每一个转换。Agno提供了一个单智能体原语，当你尝试扩展到三个并发工作线程时会报错。

解决办法不是"选择最好的框架"，而是将框架的核心抽象与你问题的形态相匹配。本节课就绘制了这样一幅地图。

## 核心概念

![智能体框架矩阵：核心抽象 vs 问题形态](../assets/framework-matrix.svg)

四大框架主导了2026年的格局。它们的核心抽象并不相同。

| 框架 | 核心抽象 | 最适用场景 | 最不适用场景 |
|-----------|------------------|----------|-----------|
| **LangGraph** | `StateGraph` — 类型化状态、节点、条件边、检查点器。 | 具有显式状态和人机交互中断的工作流；需要时间旅行调试的生产级智能体。 | 拓扑未知的松散、角色驱动的头脑风暴场景。 |
| **CrewAI** | `Crew` — 角色（目标、背景故事）、任务、流程（顺序或层级）。 | 具有短线性/层级计划的角色扮演或人格驱动的工作流。 | 任何超出任务历史记录的有状态场景；复杂的分支结构。 |
| **AutoGen** | `ConversableAgent` 对 — 两个或多个智能体轮流发言直到满足退出条件。 | 多智能体*对话*（师生、提议者-批评者、执行者-评审者），思路在对话中涌现。 | 具有已知有向无环图（DAG）的确定性工作流；任何需要跨重启持久化状态的应用。 |
| **Agno** | `Agent` — 单个LLM + 工具 + 记忆，可组合成团队。 | 快速构建的单智能体和轻量级团队；强大的多模态支持和内置存储驱动。 | 具有自定义递减器（Reducer）的深层显式分支图。 |

### "抽象"的真正含义

框架的核心抽象就是你在宣讲架构时在白板上画的东西。

- **LangGraph** → 你画一个图。节点是步骤，边是转换，每个点的状态对象都是类型化的。心智模型是状态机。
- **CrewAI** → 你画一个组织结构图。每个角色有一个职务描述，管理者分配任务。心智模型是一个小型专家团队。
- **AutoGen** → 你画一个Slack私聊。两个智能体互相发送消息；如果需要主持人，第三个智能体加入。心智模型是聊天。
- **Agno** → 你画一个带挂载工具的单个方框。把方框放在一起形成团队。心智模型是"电池全附带的智能体"。

### 状态问题

状态是大多数框架在生产环境中出问题的关键点。

- **LangGraph.** 类型化状态（`TypedDict` 或 Pydantic 模型）、字段级递减器（Reducer）、一等检查点器（SQLite/Postgres/Redis）。恢复、中断和时间旅行都是免费提供的。*（参见第11阶段·16）*
- **CrewAI.** 状态通过 `context` 字段以字符串形式在任务间流动，或通过 `output_pydantic` 进行结构化。开箱即用的持久化团队级存储不存在；如果团队需要在重启后存活，你需要自行添加。
- **AutoGen.** 状态是聊天历史以及任何用户定义的 `context`。对话记录可以持久化；任意的工位流状态则不能，除非你编写适配器。
- **Agno.** 通过 `storage=` 参数内置的存储驱动（SQLite、Postgres、Mongo、Redis、DynamoDB）附加到 `Agent`——对话会话和用户记忆自动持久化。不是完整的图检查点器，而是一个会话存储。

### 分支问题

每一个非平凡的智能体都会分支。谁来决定分支很重要。

- **LangGraph** — 由你通过条件边决定。路由是一个带有命名分支的 Python 函数。分支在编译后的图中是一等公民；检查点器会记录走了哪个分支。
- **CrewAI** — 在层级模式下由管理者决定；在顺序模式下由你在构建时决定。路由隐含在任务列表中；除管理者的提示之外，没有一等公民的"if"。
- **AutoGen** — 智能体通过聊天决定。分支由下一个发言者涌现而成。`GroupChatManager` 选择下一个发言者；你可以手写一个 `speaker_selection_method`，但默认是 LLM 驱动的。
- **Agno** — 智能体通过下一步调用哪个工具来决定。团队有协调者/路由器/协作者模式；超出此范围的分支由开发者负责。

### 可观测性问题

- **LangGraph** — 通过 LangSmith 或任何 OTel 导出器实现 OpenTelemetry。每个节点转换都是一个追踪跨度；检查点同时作为可重放的追踪。LangSmith 是第一方选项；Langfuse/Phoenix 也有适配器。
- **CrewAI** — 自2025年末起提供一等 OpenTelemetry 支持；集成 Langfuse、Phoenix、Opik、AgentOps。
- **AutoGen** — 通过 `autogen-core` 集成 OpenTelemetry；AgentOps 和 Opik 有连接器。追踪粒度是每个智能体消息，而不是每个节点。
- **Agno** — 内置 `monitoring=True` 标志以及 OpenTelemetry 导出器；与 Langfuse 紧密集成以进行会话追踪。

### 成本和延迟

所有四个框架都会增加每次调用的开销（框架逻辑、验证、序列化）。开销大致递增顺序：Agno ≈ LangGraph < CrewAI ≈ AutoGen。差异主要在于框架执行了多少额外的 LLM 路由。CrewAI 的层级管理者花费令牌决定谁下一步行动；AutoGen 的 `GroupChatManager` 也是如此。LangGraph 只在你的 `llm.invoke` 处花费令牌。Agno 的单智能体路径很薄。

当每次运行的成本很重要时，优先选择显式路由（LangGraph 边、AutoGen 的 `speaker_selection_method`）而不是 LLM 选择的路由。

### 互操作性

- **LangGraph** ↔ LangChain 工具、检索器、LLMs。一等 MCP 适配器（工具作为 MCP 服务导入）。
- **CrewAI** ↔ 工具继承自 `BaseTool`；LangChain 工具、LlamaIndex 工具和 MCP 工具都可以适配。通过 `allow_delegation=True` 进行 Crew 到 Crew 的委派。
- **AutoGen** → `FunctionTool` 包装任何 Python 可调用对象；MCP 适配器可用。与 AG2 生态系统紧密耦合以实现智能体-智能体模式。
- **Agno** → `@tool` 装饰器或 `BaseTool` 子类；MCP 适配器；工具可以在智能体和团队之间共享。

## 技能

> 你能用一句话解释为什么某个框架适合某个特定的智能体问题。

预构建检查清单：

1.  **绘制形态。** 这是一个图（类型化状态、命名转换）？一个角色扮演（专家交接工作）？一个聊天（智能体之间交谈直到完成）？还是一个带工具的单个智能体？
2.  **决定谁分支。** 开发者决定分支 → LangGraph。管理者/智能体决定 → CrewAI 层级模式。聊天涌现 → AutoGen。工具调用决定 → Agno。
3.  **检查状态预算。** 你需要从检查点恢复吗？时间旅行？运行中的人机交互中断？如果是，LangGraph 是默认选择；Agno 会话覆盖会话范围的状态。
4.  **检查成本预算。** LLM 选择的路由在每一步都会额外消耗令牌。如果智能体一天运行数千次，优先选择显式路由。
5.  **预算框架开销。** 每个框架都是另一个依赖项。如果任务只是两次 LLM 调用和一个工具，写30行纯 Python 即可；没有框架比没有任何框架更快。

在你能画出图、组织结构图、聊天或智能体方框之前，拒绝使用框架。拒绝选择一个迫使你为了你所需要的东西而与它的状态模型对抗的框架。

## 决策矩阵

| 问题形态 | 首选框架 | 原因 |
|---------------|---------------------|-----|
| 带有类型化状态、人工审批、长时间运行的工作流DAG | LangGraph | 一等状态、检查点器、中断、时间旅行。 |
| 具有不同角色的研究/写作流程 | CrewAI（顺序模式）或 LangGraph 子图 | 在 CrewAI 中按任务分配角色易于表达；当分支变得复杂时用 LangGraph 扩展。 |
| 提议者-批评者或师生对话 | AutoGen | 双智能体聊天是其原生形态。 |
| 单智能体搭配工具、会话、记忆 | Agno | 最薄的设置，内置存储和记忆。 |
| 数千个并行扇出与递减器 | LangGraph + `Send` | 唯一具有一等并行调度原语的框架。 |
| 快速原型，无框架依赖 | 纯 Python + 提供商 SDK | 没有框架就是最快的框架。 |

## 练习题

1.  **简单。** 将同一个任务——"研究 Anthropic 总部，撰写200字简报，注明来源"——分别用 LangGraph（四个节点：规划、搜索、撰写、引用）和 CrewAI（三个角色：研究员、写手、编辑）实现。报告每次运行的令牌成本和代码行数。
2.  **中等。** 在 AutoGen（研究员↔写手对话，编辑通过 `GroupChat` 加入）和 Agno（一个带有 `search_tools` 和 `write_tools` 以及会话存储的单个智能体）中构建相同的任务。对四个实现从（a）每次运行成本，（b）崩溃后恢复能力，（c）在写入步骤前注入人工审批的能力进行排名。
3.  **困难。** 构建一个决策树脚本 `pick_framework.py`，接受一个简短的问题描述（JSON：`{has_typed_state, has_roles, has_dialogue, has_parallel_fanout, needs_resume}`），并返回带有单句解释的建议。在你自行设计的六个案例上验证它。

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|-----------------|-----------------------|
| 编排（Orchestration） | "智能体如何协调" | 决定哪个节点/角色/智能体下一步运行的那一层。 |
| 持久化状态（Durable state） | "重启后恢复" | 在进程死亡后仍然存在的状态，附加到检查点或会话存储。 |
| LLM选择的路由（LLM-selected routing） | "让模型决定" | 一个规划LLM每一步选择下一步；灵活但每次决策消耗令牌。 |
| 显式路由（Explicit routing） | "开发者决定" | 一个Python函数或静态边选择下一步；廉价且可审计。 |
| Crew | "一个CrewAI团队" | 角色+任务+流程（顺序或层级）绑定为一个可运行对象。 |
| GroupChat | "AutoGen的多智能体聊天" | 通过发言者选择器管理的N个智能体之间的受控对话。 |
| Team (Agno) | "多智能体Agno" | 在一组智能体上的路由/协调/协作模式。 |
| StateGraph | "LangGraph的图" | 类型化状态、节点、条件边、检查点器原语。 |
