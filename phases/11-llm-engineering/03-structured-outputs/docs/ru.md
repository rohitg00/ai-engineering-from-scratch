# Урок 03 — Structured Outputs (структурированный вывод)

> LLM возвращает строку. Приложению нужен типизированный JSON. Этот разрыв
> уронил больше продакшенов, чем любые галлюцинации модели.

## Карта урока

Тема — как заставить модель отдавать **не текст, а данные строго заданной
формы** (ключи, типы, ограничения), пригодные для передачи дальше в код.

Для ниши (CRM-агент) это ядро: вывод агента уходит в код рассылки, поэтому
он обязан быть машиночитаемым и предсказуемым. Прямая связка с
портфолио-проектом #6 (Triggered CRM-agent, function calling).

Блоки:

1. **Проблема** — почему «верни JSON» в промпте недостаточно.
2. **Спектр контроля** — 4 уровня надёжности (prompt → JSON mode → schema mode → constrained decoding).
3. **JSON Schema** — язык контракта формы данных.
4. **Pydantic** — описываешь класс на Python, схема генерируется сама.
5. **Function calling / tool use** — главный для агентов интерфейс; у Anthropic это и есть путь к structured output.
6. **Режимы провайдеров** — OpenAI `response_format`, Anthropic tool use, Instructor (ретраи).
7. **Failure modes** — как ломается даже при схеме (галлюцинации значений, enum, вложенность).

Практика (в `my/`): один проход «руками» — мини-валидатор схемы для
интуиции, затем боевой путь через Anthropic tool use.

---

## Блок 1 — Проблема

LLM возвращает строку, приложению нужен типизированный JSON с фиксированными
**ключами** (keys), **типами** и без лишнего. Наивный путь «верни JSON» в
промпте работает ~90%, остальные 10% — markdown-забор, преамбула, битый
синтаксис, поплывшая структура.

Суть: это **не проблема промпта, а проблема декодирования** (decoding) —
модель порождает токены слева направо, и без ограничений на каждом шаге может
выбрать осмысленное слово, которое ломает JSON.

### Мини-упр. 1 (`my/naive_json.py`) — пощупать провал

Наивный промпт «верни JSON с параметрами письма» (поля НЕ заданы), Haiku 4.5,
**temp=0.7**, два прогона. **Учебниковый результат — провал по двум осям сразу:**

- **Parse-level:** JSON завёрнут в ` ```json … ``` ` + после него проза
  «Рекомендации по ответу». `json.loads(весь_ответ)` упадёт. Это **prompt-level**
  беда, от температуры не зависит — не запретили обёртку.
- **Schema-level:** поля в промпте не заданы → **контракта нет совсем**. Прогон 1
  дал 11 полей с русскими ключами (`тип_проблемы`), прогон 2 — 8 полей с
  **английскими** (`complaint_type`). Это не «модель ослушалась», а «ей не сказали».

**Про температуру (исправлено).** Изначально я разбирал вывод, предполагая temp=0,
но в коде стоит **temp=0.7** — сэмплирование включено, поэтому разные прогоны
ожидаемы, никакой экзотики. Тезис «temp=0 не детерминирован» к этому прогону
**не применим**.

**Связка с уроком 01.** Сейчас расшатаны ОБЕ ручки сразу: промпт широкий (нет
полей/типов/языка) + temp=0.7. На уроке 01 (мини-упр. 6) было зеркало — узкий
жёсткий промпт + temp=0 → символ-в-символ. Вывод: форма вывода зависит и от
жёсткости промпта, и от температуры. Промптом гарантию не получить → нужен
механизм-ограничитель (Блок 2).

## Блок 2 — Спектр контроля (4 уровня)

`Промпт (~90%) → JSON mode → Schema mode → Constrained decoding`. Уровни
**кумулятивны** (каждый включает гарантии предыдущего).

1. **Prompt-based** — никакого принуждения, ~90%. Это упр. 1, оно сломалось.
2. **JSON mode** — API гарантирует валидный JSON (лечит parse-level: нет
   markdown-забора/преамбулы). Схему НЕ гарантирует (лишние ключи, типы).
   У OpenAI `response_format:{type:"json_object"}`.
3. **Schema mode** — API принимает JSON Schema и гарантирует совпадение
   (ключи/типы/ограничения). Лечит и parse-, и schema-level. **Минимум**, чтобы
   закрыть обе беды упр. 1. У OpenAI `json_schema`; **у Anthropic — через tool use**
   (нет отдельной кнопки); у Gemini `response_schema`.
4. **Constrained decoding** — на каждой позиции токена обнуляем (token masking)
   вероятности токенов, ведущих к невалидному выводу. 100%. Это **как** часто
   реализован уровень 3 под капотом (Outlines/Guidance/OpenAI). Потолок надёжности.

**Ответ на проверку:** минимум для обеих бед = **schema mode** (он кумулятивно
включает валидный JSON). JSON mode недостаточно (чинит только parse).
Constrained decoding — оверкилл (потолок, не минимум).

## Блок 3 — JSON Schema (язык контракта формы)

Описание формы JSON (какие ключи/типы/ограничения), само пишется как JSON.
Это **не данные, а описание данных** — как настройка полей в CRM. Кирпичики:
`type` (object/array/string/number/boolean/integer), `properties`, `required`,
`enum` (строка только из списка), `minimum`/`maximum`, `minItems`/`maxItems`,
`pattern` (regex), вложенный object, `oneOf`/`anyOf` (одна из форм).

`enum` рубит «семантически близко, но не из списка» (`"раздражённая"` при списке
позитивная/нейтральная/негативная) — отклоняется при валидации, не генерируется
при constrained decoding. Это failure mode «enum confusion» из Блока 7.

## Блок 4 — Pydantic (схема генерируется сама)

JSON Schema руками — нудно/ошибкоопасно. В Python описываешь класс
`BaseModel`, схема рождается из аннотаций типов. `Email.model_json_schema()`
выдаёт ту же схему. Один источник правды: и схема (для модели), и валидация
(на твоей стороне), и автодополнение. OpenAI SDK / Instructor едят класс
напрямую и возвращают готовый провалидированный объект, не строку.

**Правило кода:** идентификаторы и значения Enum — латиницей, комментарии —
по-русски (см. [[feedback-code-latin-comments-russian]]).

### Мини-упр. 2 (`my/pydantic_intro.py`) — Pydantic руками без LLM

Модель `Email` (`subject:str`, `tone:Tone(Enum)`, `is_urgent:bool`,
`tags:list[str]=[]`), печать схемы, прогон корректного и битого словаря.

Что закрепили:
- `Класс(**словарь)` — **распаковка словаря** (dict unpacking) в именованные
  аргументы; ключи обязаны совпадать с именами полей. Альтернатива без `**` и
  без Pylance-ругани — `Класс.model_validate(словарь)` (принимает dict как `Any`).
- `try/except ValidationError as e` — ловля исключения, чтобы скрипт не падал;
  `print(e)` даёт человекочитаемый список всех ошибок сразу.
- **Pylance ≠ рантайм:** жёлтые подчёркивания на `Email(**bad_data)` — это
  статическая проверка типов *до запуска* (warning, не ошибка). Pydantic
  проверяет **значения** на рантайме и кидает `ValidationError`. Два слоя.
- В схеме: `tone`-Enum вынесен в `$defs` + `$ref`; `required` без `tags`
  (необязательное через значение по умолчанию).
- Результат битого словаря: `2 validation errors` — `tone` (enum: не из списка),
  `is_urgent` (bool_parsing: `'очень'` не привести; а `"да"/"true"` Pydantic бы
  принял мягким приведением). Каждая ошибка машиночитаема (`type`/`input_value`).
- Успех: `tone=<Tone.negative: 'negative'>` — это **Enum-объект**, чистая строка
  через `email.tone.value`; `tags=[]` подставлено по умолчанию.
- Это и есть валидационный слой на выходе LLM (→ Instructor авто-ретрай, Блок 6).

## Блок 5 — Function calling / Tool use (ядро урока для ниши)

Вместо «выдай JSON» даёшь модели **инструмент** (tool) с типизированными
параметрами и просишь его вызвать. Модель возвращает **вызов инструмента** со
структурными аргументами. OpenAI: *function calling*; Anthropic: *tool use* —
**единственный нативный путь к schema mode у Anthropic**.

В `messages.create`:
- `tools=[{name, description, input_schema}]` — `input_schema` это JSON Schema
  из Блока 3; `description` подсказывает модели, когда звать.
- `tool_choice`: `{"type":"auto"}` (сама решает) / `{"type":"any"}` (любой из
  инструментов) / `{"type":"tool","name":"..."}` (**принудительно этот** — берём
  для извлечения данных, гарантированный структурный вывод).
- Результат **не** в `.content[0].text`, а в блоке `type=="tool_use"`:
  `block.input` — **готовый dict**, без `json.loads` и без срезания markdown.

**Два рубежа защиты** (паттерн на агентов, фаза 14):
1. `input_schema` в tool use — не дать модели сгенерировать мусор (сторона API).
2. Pydantic `model_validate(block.input)` — не пустить мусор дальше (сторона кода).
В проде держат оба: схема API и Pydantic-модель могут разойтись.

Связь с портфолио #6 (Triggered CRM-agent): tool use = механизм, которым агент
отдаёт машиночитаемое решение в код рассылки. В фазе 14 тот же tool use станет
«руками» агента (не извлекать данные, а вызывать реальные функции).

### Мини-упр. 3 (`my/tool_use_extract.py`) — извлечение через Claude + tool use

То же письмо клиента, что в упр. 1, но через tool use (`extract_email` +
`input_schema` + принудительный `tool_choice`) → `block.input` →
`Email.model_validate(...)` → типизированный объект.

**Результат — контраст с упр. 1:** вместо хаоса пришёл
`{'subject': '...', 'tone': 'negative', 'is_urgent': True}` — ровно 3 нужных
поля, правильных типов, без парсинга, чисто прошло через Pydantic. Переход с
уровня 1 (промпт ~90%) на уровень 3 (schema mode).

Грабли по ходу:
- При принудительном `tool_choice` просить «верни JSON» в промпте НЕ нужно
  (лишнее, сбивает) — отдаёшь просто текст письма.
- Pylance ругается на `tools=tools` (`list[dict]` vs строгий `Iterable[ToolUnionParam]`)
  — снова статика, не рантайм; глушится `# type: ignore` или типом `list[ToolParam]`.
- Результат читать циклом по `resp.content` с проверкой `block.type=="tool_use"`,
  затем `block.input`.
- `print(a, b)` (через запятую, пробелы сами) ≠ `print(f"{a}")` (всё считаемое в `{}`);
  смешение → текст печатается буквально.

## Блок 6 — Режимы провайдеров и Instructor (обзорно)

Один контракт, три API-поверхности: **OpenAI** `response_format=PydanticModel`
(constrained decoding под капотом, возвращает `.parsed`); **Anthropic** tool use +
`input_schema`; **Gemini** `response_schema`. **Instructor** — «Pydantic для LLM»:
оборачивает клиента, замыкает петлю «ValidationError → отправить модели → починить»
(авто-ретрай до `max_retries`), работает с любым провайдером. Прод-вывод для ниши:
на Anthropic скорее всего возьму Instructor (меньше кода, авто-ретрай).

Приятная мелочь: не дублируй схему — `input_schema = Email.model_json_schema()`,
один источник правды.

## Блок 7 — Failure modes (обзорно)

Схема гарантирует **форму**, не **смысл**. Не ловит: **hallucinated values** (форма
ок, значение выдумано — самый опасный для CRM: не та сумма/сегмент); ловит при
constrained decoding/валидации: **enum confusion** (значение вне списка). Прочее:
вложенность 4+, длина массивов (`minItems`/`maxItems`), молчаливый пропуск
optional-полей (лечится `required` + явный `null`). **Главное: schema-валидация ≠
корректность данных** — для важных полей нужен смысловой контроль (LLM-as-judge,
требование цитаты-источника, человек в петле).

### Демо `my/extractor_from_template.py` — применить промпт-шаблон руками

Прогнали то же письмо через **prompt-based** путь (без tool use), довели до объекта
`Email`. Две сильные находки:

1. **Промпт-шаблон ≠ документ о промпте.** Сначала отправили весь .md-файл как
   `user` — модель восприняла его как документацию и начала **писать туториал**
   (роль confusion), а не извлекать. Корень: артефакт мешает жанры (заголовки,
   заметки, примеры). В код идёт ЧИСТАЯ инструкция, не markdown-страница.
2. **Лечение — разделение system/user** (как Drill 2 урока 01): стабильная
   инструкция → `system`, переменные схема+письмо → `user`. Сняло confusion,
   пришла реальная экстракция.
3. **Забор не выключается промптом.** В `system` явно «без markdown-забора»,
   temp=0 — модель всё равно обернула в ` ```json `. Подтверждение урока 01
   (Блок 4): забор — отдельный поведенческий слой. Срезали руками
   (`split("```")[1]` → отрезать `json` → `strip`).

**Цена prompt-based vs tool use на одной задаче:** уровень 1 = 4 шага
(срезать забор → `json.loads` → validate, в `try/except` на `JSONDecodeError` +
`ValidationError`); уровень 3 (упр. 3) = 1 шаг (`model_validate(block.input)`),
забора нет в принципе. Питон-ловушки демо: `model_json_schema` без `()` (метод
вместо dict); `Path(__file__).parent / "..."` (путь относительно скрипта, не cwd);
`.replace()` а не `.format()` (в шаблоне литеральные `{}` из JSON-примеров).

## Упражнение 1 — полиморфные выводы (`oneOf`), часть A

`my/polymorphic_extract.py`. Концепт: входящее сообщение клиента может иметь
РАЗНУЮ форму (жалоба / запрос фичи / вопрос) — вывод должен совпасть ровно с
одной схемой. Собрано по шагам.

Кирпичики Python/Pydantic:
- **`Literal["complaint"]`** — поле может быть только этим значением (бирка-метка).
- **`Union` (`A | B`)** — тип «либо A, либо B» = `oneOf` JSON Schema; присваивается
  в переменную-псевдоним типа (type alias), не класс.
- **`TypeAdapter(Тип)`** — обёртка для валидации по типу-объединению (у псевдонима
  нет `.model_validate`); метод `adapter.validate_python(dict)` возвращает объект
  нужного класса.
- **Discriminated union** — `Annotated[A | B, Field(discriminator="kind")]`. По
  полю-дискриминатору Pydantic сразу выбирает нужный класс, не перебирая все.

Находка (зачем дискриминатор): на битой жалобе (`kind="complaint"` без
`refund_amount`) **обычный** `Union` дал **3 ошибки** (`union[...]`) — настоящую
+ мусор про `FeatureRequest` (примерял на все варианты). С дискриминатором — **1**
чистая ошибка (`tagged-union[...]`), только по `Complaint`. На 5-6 типах это
разница между читаемой ошибкой и простынёй. Это правильный способ полиморфизма;
связь с Блоком 5 — «модель выбирает, какую схему заполнить» (tool use).

### Упражнение 1, часть B (путь 1) — Claude сам маршрутизирует (несколько инструментов)

Тот же файл. Два инструмента `extract_complaint` / `extract_feature_request`,
`input_schema` каждого = `Модель.model_json_schema()` (DRY). `tool_choice={"type":"any"}`
— Claude обязан вызвать ОДИН, но какой — решает сам. **Имя инструмента = бирка типа;**
то, что в части A делал дискриминатор, тут делает модель. Читаем `block.name`
(что выбрал) + `block.input` (аргументы).

Цепочка результата на письме-жалобе:
1. Маршрутизация ✅ — Claude выбрал `extract_complaint`.
2. **Но** `refund_amount` пришёл строкой `'<UNKNOWN>'` — в письме суммы нет, а поле
   было обязательным числом → модель импровизировала. **Вывод: tool use у Anthropic =
   schema mode (~99%), НЕ constrained decoding (100%)** — строку вместо числа он не
   запрещает на уровне токенов (OpenAI с constrained decoding запретил бы).
3. **Второй рубеж** — Pydantic-адаптер поймал: `float_parsing, '<UNKNOWN>' не число`.
   Без него заглушка утекла бы в код рассылки. Ради этого в проде держат оба рубежа.
4. **Дизайн-фикс схемы:** первопричина — обязательное поле без данных в тексте.
   `refund_amount: float | None = None` (тип «число ИЛИ ничего» + дефолт) → модель
   честно вернула `None`, валидация прошла, объект `Complaint` получен.

Питон по ходу: `float | None` — Union с None (необязательное/nullable);
recurring-грабля — в `try` не печатать успех (print только в `except` → тишина).

### Упражнение 1, часть B (путь 2) — Instructor

Копия `my/polymorphic_instructor.py`. `uv pip install instructor`. Клиент
оборачивается: `client = instructor.from_anthropic(anthropic.Anthropic())`.
Вместо всей ручной машинерии (tools + tool_choice + цикл по block + валидация
адаптером + ретрай) — **один вызов** `messages.create(..., response_model=Модель)`.
Instructor сам строит инструмент, валидирует через Pydantic, авто-ретраит,
возвращает готовый объект. ~15 строк → 1 вызов. Поэтому в проде на Anthropic
берут Instructor.

Грабли/находки:
- **Pylance ругается на `response_model`** (`нет перегрузок для create`) — снова
  статика: Instructor патчит клиент в рантайме, Pylance не видит. Не ошибка.
- **Discriminated union + Anthropic в режиме TOOLS (дефолт) ЛОМАЕТСЯ:** Instructor
  заворачивает union в служебную `Response.content` (ждёт объект), а Claude кладёт
  туда JSON-**строку** → `ValidationError: content should be object` → 3 ретрая →
  `InstructorRetryException`. Лечение — **сменить режим**:
  `instructor.from_anthropic(..., mode=instructor.Mode.ANTHROPIC_JSON)`. В JSON-режиме
  модель отдаёт JSON напрямую, дискриминатор `kind` разбирается корректно. На одной
  модели (без union) дефолтный TOOLS-режим работает сразу.
- **Hallucinated value (Блок 7) живьём:** на одном прогоне `refund_amount=2.0` —
  модель схватила «второй раз» из письма как сумму (это число списаний, не сумма).
  Схема-валидно (float), семантически неверно — ни tool use, ни Instructor, ни
  Pydantic не ловят. На другом прогоне честный `None`. Разброс значения от прогона
  к прогону = сигнал «поле ненадёжно из текста». Лечится только смысловым контролем.

**Итог упр. 1:** полиморфизм собран двумя путями — руками (несколько инструментов +
`tool_choice:any`, маршрутизация на стороне модели) и через Instructor (`response_model`
= discriminated union в JSON-режиме). Оба валидны; путь 1 нагляднее, путь 2 короче.

## Упражнение 4 — eval-сьют экстракции

`my/extraction_eval.py`. Набор из 8 писем с эталонными метками (`tone`,
`is_urgent`), размеченными руками; меряем точность по полям (field-level
accuracy). Машинерия: модель `Email` + функция `extract(text)` через Instructor
(`response_model=Email`) → цикл по `dataset`, сравнение `predicted.tone.value`
(Enum→строка) и `predicted.is_urgent` (bool, без `.value`!) с эталоном → счётчики
→ проценты + печать промахов.

Мерили только однозначные поля (`tone` 3 значения, `is_urgent` bool); `subject` —
свободный текст, точным совпадением мерить бессмысленно. Метки специально сделаны
независимыми (есть «нейтрально, но срочно» и «негатив, но не срочно»).

**Результат: 100% / 100%.** Модель не спуталась даже на письмах, где тональность
и срочность тянут в разные стороны — корректно развела их как независимые признаки.
Снова «на Claude потолок» (вывод уроков 01–02): извлечение категориальных полей на
frontier-модели с чёткими метками — решённая задача.

**Методический вывод (главное):** 100% на 8 лёгких кейсах ≠ прод-готовность. Хороший
eval должен ИСКАТЬ, где ломается, а не подтверждать лёгкое. 100% = сигнал «тест
слишком лёгкий». Усиление: сарказм/смешанная тональность, трудные поля (как
`refund_amount`), больший объём, и пополнение кейсами из прод-провалов →
регрессионный сьют (ловит ухудшения при смене модели/промпта). Каркас
переиспользуемый; ценность — в сложности примеров, не в цифре.

## Упражнение 5 — confidence scores (уверенность через согласованность)

`my/confidence_scores.py`. Self-consistency из урока 02, применённый к извлечению:
одно письмо прогоняем N=5 раз с **temp=0.7** (при temp=0 разброса нет), собираем
значения каждого поля в списки, через `Counter(...).most_common(1)[0]` берём
победителя и его долю = уверенность. Если `confidence < 0.8` — флаг «на ревью»
(механизм «человек в петле» из артефакта 2).

**Результат на сарказме** («Ну спасибо, очень помогли 🙄»): **5/5 negative, 5/5 False** —
Claude уверенно прочитал подтекст, флаг не сработал. Снова «на Claude self-consistency
схлопывается» (урок 02): модель консистентна → уверенность высокая. Чтобы увидеть
низкую уверенность, нужен случай, где модель сама колеблется. Машинерия (обёртка
вокруг `extract` → N прогонов → Counter → порог) — переиспользуемая, ценность в ней.
