# Стратегия структурированного вывода (CRM)

Когда строишь LLM-приложение, которому нужны структурированные данные —
применяй этот фреймворк выбора.

## Спектр контроля (4 уровня, кумулятивны)

`Промпт (~90%) → JSON mode → Schema mode → Constrained decoding`

| Подход | Parse | Schema | Усилия | Когда брать |
|---|---|---|---|---|
| Prompt-based | ~90% | ~80% | 1 мин | только прототип / fallback |
| JSON mode | 100% | ~90% | 5 мин | нужен валидный JSON, схему проверяешь сам |
| Schema mode | 100% | ~99% | 15 мин | **прод по умолчанию** |
| Constrained decoding | 100% | 100% | 30 мин | максимум надёжности (OpenAI / Outlines / Guidance) |
| Instructor + retry | 100% | ~99.5% | 10 мин | лучший DX в Python, любой провайдер |

## Какой подход когда

- **Prompt-based** — только прототип или провайдер без schema mode. В прод-пайплайн
  CRM не ставить (markdown-забор, плавающие ключи — см. упр. 1 урока).
- **JSON mode** — гарантирует валидный JSON, но не схему. Годится, если форму
  проверяешь Pydantic'ом на своей стороне.
- **Schema mode** — каждый вывод обязан совпасть со схемой. **Дефолт для прода.**
- **Function calling / tool use** — когда модель выбирает, *какую* функцию звать
  (несколько схем), или интегрируешься с существующей tool-инфраструктурой.
  У Anthropic это **единственный путь к schema mode**.
- **Instructor** — Pydantic-валидация + авто-ретрай поверх любого провайдера.

## Рекомендация для CRM-агента (мой стек)

**Anthropic + tool use + Pydantic + Instructor.**

1. **Контракт** — Pydantic-класс (`Email`, `Decision`, ...). Один источник правды;
   схема для модели = `Model.model_json_schema()`, не дублировать руками.
2. **Принуждение формы** — tool use с `input_schema` и `tool_choice` =
   `{"type":"tool","name":...}` (гарантированный структурный вывод). Это уровень 3.
3. **Второй рубеж** — `Model.model_validate(block.input)` на своей стороне.
   Два рубежа: API не даёт сгенерировать мусор, Pydantic не пускает его дальше
   в код рассылки.
4. **Авто-ретрай** — в проде обернуть в Instructor (`from_anthropic`), он замыкает
   петлю «ValidationError → отправить модели → починить» сам.
5. `temperature=0` для извлечения — нам нужна стабильность, не креатив.

## Дизайн схемы

1. Держи схему плоской. Вложенность глубже 2 уровней множит ошибки.
2. Категориальные поля — через `enum`. Не давай модели выдумывать строку.
3. Неоднозначные поля делай `required` с явным `null`, а не optional — заставляет
   модель принять решение, а не молча выкинуть поле.
4. Добавляй `description` к свойствам — модель читает их как инструкции.
5. Избегай `oneOf`/`anyOf` без нужды — усложняют декодирование.
6. Ставь `minimum`/`maximum` на числах — ловит галлюцинированные экстремумы
   (например, сумма возврата не может быть отрицательной или абсурдно большой).
7. `minItems`/`maxItems` на массивах — против пустых и бесконечных списков.

## Failure modes и фиксы

- **Markdown-забор вокруг JSON** → перейти с prompt-based на JSON/schema mode.
- **Схема-валидно, но фактически неверно** (hallucinated values — самый опасный
  для CRM: не та сумма/сегмент) → схема НЕ ловит; добавить LLM-as-judge или
  требование `quote` (фрагмент-источник), для важных полей — человек в петле.
- **Несогласованные enum** → constrained decoding или нормализация на постобработке.
- **Пропущенные optional-поля** → сделать `required` или дефолт в коде.
- **Медленно** → constrained decoding добавляет 5–15% латентности; упростить схему.
- **Большие массивы разнородных элементов** → чанковать вход, извлекать по чанку,
  потом мержить.

## Главный вывод

**Schema-валидация ≠ корректность данных.** Схема гарантирует форму (ключи, типы,
enum), но не смысл. Для CRM, где вывод уходит в реальную рассылку и деньги,
поверх схемы нужен смысловой контроль важных полей.
