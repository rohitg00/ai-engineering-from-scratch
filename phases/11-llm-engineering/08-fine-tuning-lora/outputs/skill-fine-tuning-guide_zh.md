# 微调决策指南

在微调之前，按顺序尝试这些：

```
1. 提示词工程（分钟，$0）
2. 提示词中的少样本示例（分钟，$0）
3. 用于知识检索的 RAG（天，$10-100/月）
4. 使用 LoRA/QLoRA 微调（天，每次实验 $5-50）
5. 全量微调（周，每次运行 $100-10,000）
```

仅当上一个步骤明显不足时才进入下一步。

## 何时微调

- 模型需要一致的输出风格或格式，提示词无法实现
- 你在蒸馏更大的模型（从 8B 模型获得 GPT-4 质量）
- 延迟很重要，少样本示例添加了太多 token
- 你需要模型可靠地遵循复杂的推理模式
- 你有 1,000+ 高质量的期望输入-输出行为示例

## 何时不微调

- 模型已经能用正确的提示词做你想要的事
- 你需要模型知道事实（改用 RAG）
- 你的训练样本少于 500（可能过拟合）
- 任务频繁变化（重新训练很昂贵）
- 你需要审计哪些数据影响了特定输出（微调是黑盒）

## 方法选择

| GPU 显存 | 7B 模型 | 13B 模型 | 70B 模型 |
|----------|----------|-----------|-----------|
| 16GB (T4) | QLoRA | 不可行 | 不可行 |
| 24GB (3090/4090) | QLoRA 或 LoRA | QLoRA | 不可行 |
| 40GB (A100) | LoRA 或 Full | QLoRA 或 LoRA | QLoRA |
| 80GB (A100/H100) | Full | LoRA 或 Full | QLoRA 或 LoRA |

## LoRA 配置清单

1. 从 r=16，alpha=32 开始（大多数任务的安全默认值）
2. 首先针对 q_proj 和 v_proj（最小可行 LoRA）
3. QLoRA 使用学习率 2e-4，LoRA fp16 使用 5e-5
4. 设置 lora_dropout=0.05
5. 训练 1-3 个 epoch（更多在小数据集上有过拟合风险）
6. 每 100 步在留出集上评估
7. 保存检查点并按评估损失选择最佳

## 常见错误

- 训练太多 epoch（小数据集在 epoch 2-3 后过拟合）
- 使用与全量微调相同的学习率（LoRA 需要更高 LR）
- 忘记设置 pad token（导致 Llama 模型损失为 NaN）
- 未冻结基础模型（违背了 LoRA 的目的）
- 仅在训练数据上评估（始终留出 10-20% 用于评估）
- 跳过提示词工程基线（微调一个提示词已经解决的问题）

## 质量验证

训练后，在 200+ 个留出示例上比较：
1. 最佳提示词的基础模型（基线）
2. 带 LoRA 适配器的基础模型（你的微调模型）
3. 相同提示词的 GPT-4 或 Claude（上限）

如果 LoRA 模型未击败提示词基线，你的训练数据或配置需要改进，而非更多计算。

## 适配器管理

- 为多任务服务保持适配器分离（每个请求切换适配器）
- 将适配器合并到基础权重中以进行单任务部署
- 在 Hugging Face Hub 上存储适配器（10-100MB，易于版本控制和共享）
- 部署前测试合并模型输出与未合并输出匹配
- 使用 TIES-Merging 或 DARE 将多个适配器合并为一个

## 调试训练

如果损失不下降：
1. 检查学习率（LoRA 太低，尝试 2e-4）
2. 验证 LoRA 层是否实际接收梯度
3. 确认基础模型权重已冻结
4. 检查数据格式（tokenizer 必须匹配模型期望的格式）

如果损失下降但评估质量差：
1. 训练数据质量问题（垃圾进，垃圾出）
2. 过拟合（减少 epoch，增加 dropout，添加更多数据）
3. 错误的目标模块（为复杂任务添加 MLP 层）
4. 秩太低（尝试 r=32 或 r=64）
