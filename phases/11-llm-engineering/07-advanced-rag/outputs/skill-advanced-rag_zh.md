# 高级 RAG 模式

基础 RAG：嵌入查询 -> 向量搜索 -> top-k -> 生成。
高级 RAG：嵌入查询 + BM25 -> 融合排名 -> 重排序 -> top-k -> 生成。

```
query -> [vector search (top-50)] -+-> RRF fusion -> reranker (top-5) -> prompt -> LLM
                                   |
query -> [BM25 search (top-50)]  --+
```

## 何时从基础 RAG 升级

- 检索质量低于 70% Recall@5
- 用户报告错误或无关答案
- 语料库超过 100K 分块
- 查询使用与文档不同的词汇
- 多跳问题持续失败

## 实现清单

1. 在向量索引旁添加 BM25 索引
2. 并行运行两种搜索（各取前 50）
3. 使用倒数排名融合（RRF, k=60）合并
4. 用交叉编码器重排序顶部候选
5. 取前 5 个用于最终提示词
6. 在测试集上添加忠实度评估

## 技术选择指南

- **混合搜索**：生产环境始终使用。查询时无额外成本。
- **重排序**：当 Recall@50 好但 Recall@5 差时使用。增加 50-200ms 延迟。
- **HyDE**：当查询模糊或使用与文档不同的词汇时使用。增加一次 LLM 调用。
- **父子分块**：当小分块缺乏上下文但大分块稀释相关性时使用。
- **元数据过滤**：当语料库有明确类别（日期、来源类型、部门）时使用。
- **查询分解**：用于需要从多个文档获取信息的多跳问题。

## 常见错误

- BM25 和向量搜索使用不同的分块集（它们必须搜索相同语料库）
- 重排序的候选池太小（前 10 个太少；使用前 50 个）
- 为每个查询添加 HyDE（仅在词汇不匹配是瓶颈时才有帮助）
- 不评估变更（在添加每种技术前后测量 Recall@k）
- 在测量失败位置之前过度工程化流水线

## 评估工作流

1. 创建 50+ 个带有已知答案分块的测试问题
2. 为每种检索方法测量 Recall@5 和 Recall@10
3. 对于检索成功的查询，测量生成答案的忠实度
4. 随着语料库增长每周跟踪指标
5. 在添加更多技术之前调查个别失败
