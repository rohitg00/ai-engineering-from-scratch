## Блок 1 — Machine Learning в одной картинке (мост из фазы 2)

Машинное обучение — это **не алгоритм, который мы пишем руками**, а
**функция, которую мы подбираем по примерам**. Разница с обычным
программированием:

| Обычный код | ML |
|---|---|
| Программист пишет правила, машина даёт ответ | Программист даёт примеры (вход → правильный ответ), машина выводит правила |

Маркетинговая аналогия: ты не пишешь руками правило «если клиент купил
A и B, рекомендуй C». Ты даёшь машине **миллион историй покупок**, и
она сама находит закономерности.

**Три обязательных ингредиента любой ML-задачи:**

1. **Данные** — пары (вход X, правильный ответ y). Например, X = текст
   отзыва, y = метка «позитив/негатив».
2. **Модель** — функция с настраиваемыми **параметрами** (weights, веса).
   На старте параметры случайные, модель отвечает чушь.
3. **Loss function** (функция потерь) — число, которое говорит «насколько
   ответ модели плох». Чем больше ошибка — тем больше loss.

**Обучение = подкрутка весов так, чтобы loss стал маленьким** на
обучающих данных. Алгоритм подкрутки — **gradient descent** (градиентный
спуск): «в какую сторону подвинуть каждый вес, чтобы loss уменьшился» —
и сделать маленький шаг в эту сторону. Повторить миллионы раз.

**Три ключевых разбиения данных:**

- **Train** (обучение) — на этом модель учится.
- **Validation** (валидация) — на этом подбираем настройки (например,
  learning rate).
- **Test** (тест) — финальная проверка, модель этого никогда не видела.

Зачем разбивать: модель может **запомнить** обучающие примеры
(overfitting, переобучение) и при этом не уметь работать на новых.
Тест ловит это.

**Почему это важно для фазы 11:** когда дойдём до **fine-tuning**
(урок 11.08) и **evaluation** (урок 11.10) — будет та же схема:
train/val/test, loss, метрики. Просто модель будет огромная и
предобученная.

## Блок 2 — Нейросеть как кирпич (мост из фазы 3 — DL Core)

**Нейросеть** — это просто **очень гибкая параметрическая функция**,
собранная из миллионов простых блоков. «Глубокое обучение»
(deep learning) = нейросеть с большим количеством слоёв.

**Один нейрон делает три действия:**

1. Берёт несколько чисел на вход (`x1, x2, …, xn`).
2. Умножает каждое на свой **вес** (`w1, w2, …, wn`), складывает,
   добавляет **bias** (`b`):
   `z = w1·x1 + w2·x2 + … + wn·xn + b`
3. Прогоняет результат через **нелинейность** (activation function,
   например ReLU или sigmoid) — без неё нейросеть схлопывается в
   обычную линейную регрессию и теряет всю мощь.

Узнал формулу? Это **то самое `output = X @ W + b`** из урока 01 фазы 1
— главное уравнение всего deep learning. Нейрон = строка матрицы весов.

**Слой = много нейронов параллельно** (`y = X @ W + b` + активация).
**Сеть = слои стопкой**: выход одного слоя становится входом следующего.
Чем больше слоёв — тем сложнее закономерности модель умеет ловить.

**Forward pass** (прямой проход) — данные идут от входа через все слои
к выходу, считается предсказание и loss.

**Backward pass / backpropagation** (обратное распространение) — главный
трюк DL. Алгоритм считает, **как каждый вес повлиял на loss**, и
сообщает «этот вес нужно чуть уменьшить, этот чуть увеличить».
Технически — это цепное правило производной из матанализа, но идея
простая: «распространи ошибку с выхода обратно к каждому весу».

После backward — **gradient descent делает шаг**: каждый вес сдвигается
на `-learning_rate * градиент`. Повторили миллионы раз — обучили модель.

**Что важно запомнить для фазы 11:**

- **LLM — это огромная нейросеть** (миллиарды нейронов в десятках-
  сотнях слоёв). Принцип тот же — forward, loss, backward, gradient
  descent. Просто в гигантском масштабе.
- **Параметры = веса = «знания» модели**. Когда Anthropic говорит
  «Sonnet 4.6», физически модель — это файл с миллиардами чисел.
  Эти числа и есть всё, что модель «знает».
- **Fine-tuning (урок 11.08)** — это **продолжение того же обучения**
  на твоих данных. Берёшь готовые веса, делаешь ещё несколько шагов
  gradient descent — модель подстраивается под твою задачу.
- **LoRA** (которую мы уже видели в уроке 01 фазы 1) — трюк, чтобы не
  тренировать **все** миллиарды весов, а только маленькие «дельты».
  Будет в уроке 11.08.

**Чего НЕ нужно знать руками** (по нашему плану «applied systematic»):

- Конкретные формулы backpropagation.
- Подкапотную математику оптимизаторов (Adam, SGD, AdamW) —
  достаточно знать, что это разные стратегии «как именно делать шаг
  по градиенту».
- Регуляризацию, dropout, batch normalization — это всё внутренности
  обучения, на этапе пользователя LLM нам неинтересно.

## Блок 3 — Текст для нейросети: токены и эмбеддинги (мост из фазы 5 — NLP)

Нейросеть умеет работать **только с числами**. Текст для неё —
иероглифы. Поэтому между тобой и моделью стоят два преобразователя:
**токенизатор** и **эмбеддинг-слой**.

### Токенизация (tokenization)

**Токен** — это не слово и не буква. Это **подслово** (subword).
Современные модели используют алгоритмы вроде **BPE** (Byte Pair
Encoding) или **SentencePiece**, которые разрезают текст на куски
длиной ~3-5 символов — частые слова целиком, редкие разбиваются.

Примеры (приближённо, у каждой модели свой токенизатор):

- `"hello"` → 1 токен
- `"unbelievable"` → 3 токена: `["un", "believ", "able"]`
- `"Привет"` → 2-3 токена (русский менее «упакован», чем английский)
- `" "` (пробел) — часть соседнего токена

Каждому токену сопоставляется **целое число (token id)** из словаря
модели (~50k-200k вариантов).

**Почему ты обязан про это помнить:**

1. **Контекстное окно меряется в токенах**, не в словах. У Sonnet 4.6
   — 200k токенов. На русском это **меньше слов**, чем кажется (русский
   требует ~1.5-2x больше токенов на ту же мысль, чем английский).
2. **API биллится по токенам** — и input, и output. Длинная системная
   инструкция на русском дороже той же инструкции на английском.
3. **Странности**: модель «не видит» отдельные буквы внутри токена.
   Поэтому LLM плохо считает буквы в слове, плохо делает анаграммы,
   путается в спеллинге — она оперирует не буквами, а кусками.

В уроке 11.01 (prompt engineering) и 11.11 (caching-cost) токены
всплывут постоянно.

### Embeddings (эмбеддинги, векторные представления)

Дальше токен (целое число) надо превратить в **вектор** — потому что
нейросеть умеет работать только с векторами, а не с дискретными id.

**Embedding** — это **вектор из ~1000-4000 чисел**, который представляет
токен (или слово, или предложение) в «семантическом пространстве».
Близкие по смыслу токены имеют **близкие векторы**.

Помнишь cosine similarity из урока 01 фазы 1? Это её главное применение
в проде:

```
similarity("кошка", "кот")     → 0.92  (близко)
similarity("кошка", "автомобиль") → 0.15  (далеко)
```

Эмбеддинги — это **тот самый случай**, где «вектор как точка в
пространстве смыслов» становится буквальным. Модель в процессе
обучения сама выучивает: «слова в похожих контекстах должны иметь
похожие векторы». Никто не размечает синонимы руками.

**Два уровня эмбеддингов, которые встретятся в фазе 11:**

1. **Внутри LLM** — эмбеддинг каждого токена является первым слоем
   модели. Это её внутренняя «кухня», ты к ней не лезешь.
2. **Отдельная embedding-модель** (например, OpenAI
   `text-embedding-3`, Cohere Embed, Voyage) — даёт тебе вектор для
   **целого текста**. На этом построен **RAG** (Retrieval-Augmented
   Generation, уроки 11.06 и 11.07): «найди в базе документ, чей
   вектор ближе всего к вектору вопроса».

**Что важно для фазы 11:**

- **Урок 11.04 (embeddings)** — углубим: какие модели бывают, как их
  использовать, как мерить similarity. Уже знаешь cosine — половина
  дела сделана.
- **Уроки 11.06-07 (RAG)** — это эмбеддинги + векторная база
  (Pinecone, Weaviate, FAISS). Под капотом — поиск ближайших соседей
  по cosine.
- **Эмбеддинги — отдельный продукт от LLM**. Anthropic эмбеддингов
  **не делает**, обычно берут от OpenAI / Cohere / Voyage. В уроке
  11.04 разберём почему.

**Что НЕ нужно знать:** word2vec, GloVe, FastText (исторические
эмбеддинги до transformer-эры). Знать, что они были — достаточно.

## Блок 4 — Transformer и attention (мост из фазы 7)

Самый важный блок моста — **transformer** это та архитектура, на
которой работают **все** современные LLM (GPT, Claude, Gemini, Llama,
Mistral, DeepSeek). Если про DL в целом нам достаточно интуиции «много
слоёв с весами», то про transformer стоит знать **что именно он делает**.

### Проблема, которую решает transformer

До 2017 года тексты обрабатывали **RNN/LSTM** (рекуррентные сети) —
слово за словом, последовательно. Два фатальных минуса:

1. **Не параллелится** — слово N нельзя обработать, пока не обработано
   N-1. На GPU это смерть скорости обучения.
2. **Забывает контекст** — пока дошёл до конца длинного абзаца,
   начало уже «выветрилось».

В 2017 Google выпустил статью **«Attention Is All You Need»** —
придумали архитектуру, где **все токены обрабатываются параллельно**,
а связь между ними устанавливается через механизм **attention**.

### Attention (внимание) в одной фразе

Для каждого токена в последовательности модель **сама решает**, на
какие другие токены **посмотреть, чтобы лучше предсказать этот**.

Пример: `"Маркетолог отправил клиенту письмо, потому что он …"`

Чтобы предсказать следующее слово, модель должна понять — **«он»
относится к маркетологу или к клиенту?** Attention позволяет токену
«он» получить максимальный вес на правильное слово в контексте. Это всё
— линейная алгебра (`Q @ K^T`, потом softmax, потом `@ V`), но идея
чисто концептуальная: **каждый токен смотрит на все остальные и берёт
от них info столько, сколько надо**.

**Self-attention** — это когда токены смотрят друг на друга внутри
одной последовательности. **Multi-head attention** — это когда
параллельно работает несколько «голов внимания», каждая ловит свой
тип связей (грамматику, семантику, ссылки на анафору и т.д.).

### Что такое «слой transformer'а»

Один **transformer block** = self-attention + обычный feed-forward
слой (тот самый из блока 2) + пара технических трюков (residual
connection, layer norm). Современная LLM = **десятки-сотни** таких
блоков стопкой.

Грубо: Sonnet/GPT-4 — это **~80-120 transformer-блоков**, каждый с
миллиардами весов. Поток данных:

```
текст → токены → эмбеддинги → 100 transformer-блоков → логиты → следующий токен
```

### Decoder-only архитектура (важно!)

Оригинальный transformer 2017 года был **encoder-decoder** (для
перевода). Современные LLM используют упрощённую **decoder-only**
архитектуру:

- Модель **только генерирует**, не «кодирует и потом декодирует».
- На каждом шаге она видит **весь предыдущий текст** (промпт + то,
  что уже сгенерировала) и предсказывает **один следующий токен**.
- Потом этот токен добавляется к контексту — и снова, до
  **stop-токена** или лимита длины.

Это называется **autoregressive generation** (авторегрессивная
генерация) — модель генерирует **по одному токену за раз**, использует
свой же выход как вход. Это причина, по которой:

- LLM «думает в одну сторону» — назад поменять уже сгенерированное
  не может (без специальных приёмов).
- Длинный ответ генерируется **дольше**, чем короткий — там физически
  больше шагов forward pass.
- В API ты видишь **streaming** (ответ выводится по словам) — это и
  есть autoregressive в реальном времени.

### Что важно для фазы 11

- **Контекстное окно = сколько токенов transformer может одновременно
  «увидеть» в attention**. У Sonnet 4.6 — 200k. Расширять дорого:
  attention квадратичен по длине (`O(n²)`).
- **Prompt engineering (урок 11.01)** работает потому, что attention
  действительно учитывает **каждое слово** в твоём промпте. Лишнее
  слово — лишний вес в распределении внимания.
- **Context engineering (урок 11.05)** — это **наука о том, что и куда
  положить в контекстное окно**, чтобы attention выловил нужное.
  Порядок имеет значение: модели часто лучше помнят начало и конец
  контекста, а середину — хуже («lost in the middle»).
- **Function calling (урок 11.09)** и **MCP (урок 11.14)** — тоже
  работают через контекст: схема функций кладётся в промпт, и модель
  «обращает внимание» на неё, когда генерирует вызов.

### Чего НЕ нужно знать руками

- Конкретные формулы attention (Q, K, V, scale, softmax) — концепции
  достаточно.
- Positional encoding (как модель понимает порядок токенов) — это
  технический трюк, фаза 7 разбирает.
- Multi-query / grouped-query attention, flash-attention — оптимизации.

**Бери в голову одну вещь:** transformer **видит весь контекст разом**
и **генерирует по одному токену**. Это объясняет 90% поведения LLM.

## Блок 5 — Как из transformer'а получается ChatGPT/Claude (мост из фазы 10)

Архитектура transformer'а — это **пустой движок**. Чтобы получить из
него Claude или GPT, модель проходит **три стадии обучения**, и каждая
делает её принципиально другой. Эту цепочку важно понимать — иначе
непонятно, чем «base model» отличается от «instruct» и почему RLHF —
это вообще отдельная статья расходов.

### Стадия 1 — Pretraining (предобучение)

**Что делают:** берут гигантский корпус текста (терабайты — Common
Crawl, книги, статьи, GitHub, Wikipedia, Reddit и т.д.) и учат
transformer **одной задаче — предсказывать следующий токен**. Никаких
меток, никакой разметки, просто: «вот текст, угадай продолжение».

- **Объём данных:** триллионы токенов.
- **Объём вычислений:** месяцы на тысячах H100/A100 GPU. Стоимость
  одного pretraining'а топовой модели — **десятки-сотни миллионов
  долларов**.
- **Результат:** **base model** (базовая модель). Она «знает мир», но
  **не умеет следовать инструкциям**. Если ей написать «переведи на
  русский: hello», она с большой вероятностью **продолжит как в
  учебнике**: «переведи на русский: hello → привет. Переведи на
  немецкий: hello → hallo». Она не отвечает — она дополняет.

Маркетинговая аналогия: ты прочитал половину интернета, но никогда ни
с кем не разговаривал. У тебя в голове куча фактов и паттернов, но
навыка «диалога» нет.

### Стадия 2 — SFT (Supervised Fine-Tuning, инструктивный файнтюнинг)

**Что делают:** берут base model и **дообучают** её на качественных
парах «инструкция → правильный ответ», написанных людьми (или
другими моделями).

```
[user]: Переведи на русский: hello
[assistant]: Привет
```

Тысячи-сотни тысяч таких диалогов. Технически — тот же gradient
descent, что и в pretraining, просто на меньшем датасете и **с
разметкой формата чата**.

- **Объём данных:** тысячи-сотни тысяч примеров (микроскоп по
  сравнению с pretraining).
- **Результат:** **instruct model**. Она начинает понимать формат
  «вопрос-ответ», следовать инструкциям, отвечать в нужном тоне.

Главное на этой стадии — **формат разговора**. Модель учится: «когда
я вижу `[user]:`, я должна ответить как `[assistant]:`, а не
дописывать следующий `[user]:`-вопрос». Отсюда и берётся структура
**system / user / assistant** в API, которую ты увидишь в уроке 11.01.

### Стадия 3 — RLHF (Reinforcement Learning from Human Feedback)

**Что делают:** берут instruct model и дополнительно **выравнивают
её** под человеческие предпочтения.

Схема:

1. Люди (или модели) **сравнивают пары ответов** модели на одинаковый
   промпт: «ответ A лучше ответа B».
2. На этих сравнениях обучается **reward model** — отдельная нейросеть,
   которая ставит оценку любому ответу.
3. Основная модель дообучается так, чтобы **максимизировать reward**.
   Технически — алгоритмами reinforcement learning (PPO, DPO, и т.д.).

Зачем это нужно отдельно: **SFT учит модель формату**, но не
**качеству и безопасности**. RLHF учит «отвечать так, чтобы человеку
понравилось»:

- Не выдумывать факты, когда не знает (это лечится частично — отсюда
  галлюцинации, блок 6).
- Не отвечать на вредные запросы.
- Быть полезной, точной, не водить за нос.
- Соблюдать стиль (тот самый «характер» Claude или GPT).

У Anthropic есть **Constitutional AI** — вариация RLHF, где сравнения
делает не человек, а сама модель по списку правил («constitution»).
Это их фишка, влияет на то, как Claude отказывается / уточняет /
разговаривает.

### Итоговая цепочка

```
случайные веса
    ↓  pretraining (триллионы токенов, $100M+)
base model — знает мир, не умеет общаться
    ↓  SFT (тысячи диалогов)
instruct model — умеет следовать формату инструкций
    ↓  RLHF / Constitutional AI (сравнения)
production model — Claude Sonnet 4.6, GPT-4, и т.д.
```

### Что важно для фазы 11

- **Когда ты дёргаешь API `claude-sonnet-4-6`** — ты говоришь с
  **третьей стадией**. Все три обучения уже произошли в Anthropic,
  ты ими не управляешь.
- **Fine-tuning (урок 11.08)** — это **четвёртая стадия поверх готовой
  модели**. Берёшь production-веса (или их LoRA-копию) и дообучаешь
  на своих данных. Концептуально — то же самое, что SFT, только
  маленький локальный datasets.
- **Prompt engineering vs fine-tuning** — главный выбор фазы 11.
  Промпт меняешь мгновенно и бесплатно, fine-tune — дорого и надолго.
  В уроках 11.01-11.07 разберём, что можно вытащить без fine-tuning.
- **Instruct/chat-формат** (system / user / assistant) — это
  **наследие стадии SFT**. Поэтому Anthropic API устроен именно так,
  а не как «дай текст, получи продолжение».
- **«Безопасность» Claude (отказы, осторожность)** — это RLHF. Не баг,
  а намеренная настройка. В уроке 11.12 (guardrails) — что с этим
  делать, когда тебе нужно больше / меньше «осторожности».

### Чего НЕ нужно знать

- Конкретные алгоритмы RL (PPO, DPO, GRPO) — концепции достаточно.
- Internal-устройство Constitutional AI и RLAIF.
- Что такое «reward hacking» и «scaling laws» — фоновое знание, для
  прода неважно.

## Блок 6 — Inference: как модель «отвечает» (мост к уроку 11.01)

Последний блок — про **то, что физически происходит**, когда ты
дёргаешь API. Без этого нельзя понять, что такое `temperature`,
`top_p`, почему модели галлюцинируют и почему один и тот же промпт
даёт разные ответы.

### Что выдаёт transformer на самом деле

Transformer **не выдаёт слова**. На каждом шаге генерации он выдаёт
**вектор логитов** — массив чисел длиной = размер словаря токенизатора
(~50k-200k чисел).

```
"Маркетолог отправил клиенту" → [логит_для_токена_1, логит_для_токена_2, ..., логит_для_токена_100000]
```

Каждое число — это «насколько модель считает, что этот токен должен
идти следующим». Чем больше — тем вероятнее.

Дальше логиты прогоняются через **softmax** — функция, которая
превращает массив чисел в **распределение вероятностей** (все числа
от 0 до 1, сумма = 1):

```
[2.3, -1.0, 5.1, ..., 0.7] → [0.04, 0.001, 0.78, ..., 0.02]
                                                   ↑
                                          вероятности
```

Например: после `"Маркетолог отправил клиенту "` модель выдаёт:

- `"письмо"` — 0.45
- `"купон"` — 0.20
- `"уведомление"` — 0.12
- `"подарок"` — 0.05
- ... остальные 99996 токенов — суммарно 0.18

### Sampling — как выбрать токен из распределения

Дальше встаёт вопрос: **какой токен выбрать?** Тут включается
**sampling strategy** (стратегия сэмплинга) — и она напрямую
управляется параметрами в API.

**Greedy sampling (жадный):** всегда брать самый вероятный токен.
**Детерминированно** (одинаковый промпт → одинаковый ответ), но скучно
и часто застревает в повторах.

**Temperature (температура):** число, которое **сглаживает или
заостряет** распределение **до** sampling'а.

- `temperature = 0` — фактически жадный режим, всегда самый вероятный.
  **Детерминированно.** Хорошо для извлечения данных, классификации,
  structured output.
- `temperature = 1` — стандартное распределение, как модель сама его
  выдала.
- `temperature > 1` — распределение «размазывается», редкие токены
  получают шанс. Хорошо для креатива, плохо для точности.
- `temperature = 2`+ — модель начинает нести бред.

Маркетинговая аналогия: температура — это «насколько я готов рискнуть
необычным сообщением». 0 = шаблон рассылки, 1 = персонализированный
текст, 1.5 = креативная кампания.

**Top-k:** оставить только `k` самых вероятных токенов, остальные
занулить. Например, `top_k=50` — выбираем из 50, не из 100k.

**Top-p (nucleus sampling):** оставить наименьший набор токенов, чья
**суммарная вероятность ≥ p**. Например, `top_p=0.9` — берём столько
верхних токенов, чтобы они в сумме давали 90% вероятности. Адаптивный
top-k.

В Anthropic API ты будешь видеть в основном **`temperature`** и
**`top_p`**. На практике в проде обычно крутят только `temperature`.

### Stop conditions — когда модель замолкает

После каждого выбранного токена цикл повторяется: новый токен
добавляется к контексту → forward pass → логиты → sampling → следующий
токен. Цикл прерывается, когда:

1. Модель сама сгенерировала **stop-token** (специальный токен «конец
   сообщения», ставится на стадии SFT).
2. Достигнут лимит **max_tokens** (параметр API — «не больше N токенов
   в ответе»).
3. Сработал явный **stop sequence**, который ты указал (например,
   `"\n\n"`).

### Откуда берутся галлюцинации

**Hallucination (галлюцинация)** — когда модель **уверенно выдумывает
факт**. Имена, цифры, цитаты, статьи. Главное — **звучит
правдоподобно**.

Корень — то, как модель устроена:

1. **Цель обучения — предсказывать правдоподобный следующий токен**,
   не выдавать истину. Модель не знает «я не знаю». В её картине мира
   на любой контекст всегда есть распределение возможных следующих
   токенов, и `softmax` всегда даст какой-то ответ.
2. **Sampling вносит случайность.** Даже если правильный ответ имел
   вероятность 0.6, а неправильный 0.4 — при `temperature > 0`
   модель иногда выберет неправильный.
3. **Узкое горлышко RLHF.** Модель учили «отвечать уверенно и
   полезно». Сказать «не знаю» — менее «полезно» с точки зрения
   reward, поэтому она склонна **выдумать**, чем признаться.
4. **Off-distribution запросы.** Если ты спрашиваешь про то, чего в
   обучающих данных мало (свежие события, узкие домены, твоя
   CRM-система) — модель **интерполирует**, и интерполяция выглядит
   как галлюцинация.

**Что с этим делать (это, собственно, фаза 11):**

- **Prompt engineering (11.01-02)** — внятно сформулировать, что
  модель должна делать и когда сказать «не знаю».
- **Structured outputs (11.03)** — заставить отвечать в строгом JSON,
  чтобы было легко валидировать.
- **RAG (11.06-07)** — дать модели контекст с проверенными фактами и
  заставить отвечать **только из него**.
- **Function calling (11.09)** — пусть модель не «вспоминает» цифру,
  а вызывает функцию, которая её посчитает.
- **Evaluation (11.10)** — мерить долю галлюцинаций и регрессии.
- **Guardrails (11.12)** — отдельный слой проверки перед тем, как
  ответ уйдёт пользователю.

### Stateless API — важный нюанс

LLM API — **stateless**. У модели **нет памяти между запросами**.
«Помнит твой предыдущий вопрос» — иллюзия. На самом деле клиент при
каждом запросе **присылает всю историю диалога заново**. Поэтому:

- **Контекстное окно** — это не только твой текущий вопрос, это
  **весь диалог + system prompt + все RAG-документы + схемы функций**.
  Всё вместе должно влезть в 200k токенов.
- **Каждый запрос платный по полному объёму контекста.** Длинный чат
  дорожает с каждым ходом — пока не подключишь **prompt caching**
  (урок 11.15).
- **Memory** в агентах — это **искусственная штука сверху**. Ты сам
  сохраняешь предыдущие сообщения и подставляешь их в новый запрос.
  Никакой магии.

## Сводка — что унесли в фазу 11

1. **ML** — модель учится по примерам, gradient descent минимизирует
   loss.
2. **Нейросеть** — много слоёв `X @ W + b` + нелинейность. LLM —
   просто **очень большая** нейросеть.
3. **Текст → токены → эмбеддинги.** Контекст меряется в токенах.
   Эмбеддинги — это семантические векторы, основа RAG.
4. **Transformer + attention** — архитектура, которая видит **весь
   контекст разом** и генерирует **по одному токену**.
5. **Pretraining → SFT → RLHF** — три стадии превращения «движка» в
   Claude. Production-модель уже всё это прошла.
6. **Inference** — логиты → softmax → sampling (temperature, top-p) →
   токен → следующий шаг. Галлюцинации заложены в архитектуру,
   лечатся обвязкой вокруг модели.

**Ключевая идея фазы 11:** LLM — это **компонент**, не приложение.
Хорошие AI-системы выигрывают не за счёт более умной модели, а за счёт
правильной **обвязки**: что положить в контекст, как заставить
отвечать структурно, как дать ей инструменты и факты, как мерить
качество. Этим и будем заниматься следующие 17 уроков.
