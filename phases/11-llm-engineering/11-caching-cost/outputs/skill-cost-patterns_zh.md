# LLM 成本优化模式

在构建需要控制成本的 LLM 应用时，应用此决策框架。

## 何时优化

**立即优化当：**
- 月度 LLM 支出超过 $500 或基础设施预算的 10%
- 消费者产品的每次查询成本高于 $0.01
- 你的系统提示词超过 1,000 token 且随每个请求发送
- 超过 30% 的查询是重复或近似重复的
- 你从 100 扩展到 10,000+ 日活用户

**尚未优化当：**
- 你少于 100 DAU 且仍在验证产品-市场契合度
- 月度支出低于 $100 且增长缓慢
- 你仍在迭代提示词设计（缓存会锁定你的提示词）

## 缓存策略选择

### 精确缓存

**在以下情况使用：** temperature=0，相同提示词重复，需要确定性输出。

```python
key = sha256(json.dumps({"model": m, "messages": msgs, "temp": 0}))
```

- 实现：30 分钟
- 命中率：大多数应用 10-25%，FAQ 机器人 40-60%
- 延迟：<1ms（字典查找）
- 风险：如果底层数据变化，响应会过时

**在以下情况跳过：** temperature > 0，每个查询都是唯一的，需要实时数据。

### 语义缓存

**在以下情况使用：** 用户用不同措辞问相同问题，FAQ 密集型产品、客户支持。

- 实现：2-4 小时（嵌入 + 相似度 + 存储）
- 命中率：在精确缓存之上额外 15-35%
- 延迟：10-50ms（嵌入 + ANN 搜索）
- 风险：误报（为相似但不同的问题返回错误的缓存答案）

**阈值指南：**
- 0.98+：非常保守，几乎没有误报，命中率较低
- 0.95：事实性问答的良好平衡
- 0.90：激进，命中率更高但有错误答案风险
- 0.85：仅用于低风险应用（建议、自动补全）

**在以下情况跳过：** 每个查询都有唯一上下文（代码生成），响应必须反映最新数据，查询空间无界。

### 提供商提示词缓存

**在以下情况使用：** 系统提示词 > 1,024 token（OpenAI）或模型特定最小值，相同前缀反复发送。

| 提供商 | 操作 | 节省 |
|----------|--------|---------|
| Anthropic | 在系统消息中添加 `cache_control: {"type": "ephemeral"}` | 缓存前缀 90%（在 25% 写入溢价之后） |
| OpenAI | 无需操作（自动） | 缓存前缀 50% |
| Google | 使用带显式 TTL 的 Context Caching API | 缓存上下文约 75% |

**在以下情况跳过：** 系统提示词每个请求都变化，提示词低于最小长度。

## 模型路由规则

### 基于关键词（简单、快速）

```
simple:  <= 5 个词 OR 匹配 FAQ 关键词 -> gpt-4o-mini ($0.15/$0.60)
medium:  一般查询、摘要        -> claude-sonnet ($3/$15)
complex: "analyze"、"compare"、"debug"     -> gpt-4o ($2.50/$10)
```

- 实现：1 小时
- 准确率：70-80%
- 节省：模型成本的 40-60%

### 基于嵌入（更准确）

每个类别嵌入 50-100 个标注查询。通过最近邻分类新查询。

- 实现：4-8 小时
- 准确率：85-92%
- 节省：模型成本的 50-70%
- 额外成本：分类嵌入约 $0.02/1M token（可忽略）

### 基于 ML（生产级）

在历史查询/模型对上训练一个小型分类器（逻辑回归或小型 BERT）。

- 实现：1-2 周
- 准确率：90-95%
- 节省：模型成本的 60-75%
- 需要：来自生产流量的标注训练数据

## 速率限制配置

### 按层级的令牌桶参数

| 层级 | 桶大小 |  refill 速率 | 最大 RPM | 每日上限 |
|------|-------------|-------------|---------|-----------|
| Free | 50K tokens | 500/秒 | 10 | 50K |
| Pro | 500K tokens | 5K/秒 | 60 | 500K |
| Enterprise | 5M tokens | 50K/秒 | 300 | 5M |

### 实现清单

1. 将桶存储在 Redis 中（非内存中）以支持多实例应用
2. 使用原子操作（MULTI/EXEC）防止竞态条件
3. 拒绝响应返回 `Retry-After` 头
4. 跟踪被拒绝的请求作为指标（>5% 拒绝 = 层级限制过紧）
5. 实现优雅降级：首先拒绝昂贵的模型请求，保持廉价模型访问

## 预算控制

### 三阈值断路器

| 阈值 | 操作 | 可逆 |
|-----------|--------|------------|
| 月度预算的 70% | 记录警告，通过 Slack/PagerDuty 告警团队 | 是（自动） |
| 月度预算的 85% | 将所有流量路由到最便宜的模型 | 是（自动，下个计费周期） |
| 月度预算的 95% | 仅提供缓存响应，拒绝新的 LLM 调用 | 是（手动重置或下个周期） |

### 每用户成本跟踪

跟踪每个用户的累计成本。标记超过中位数 10 倍的用户。常见原因：
- 合法的重度用户（升级他们的层级）
- 提示注入循环（机器人发送自动请求）
- 低效的集成（客户端在每个错误上重试）

### 仪表板关键指标

- **每次查询成本**（P50、P95、P99）——按模型、按功能、按用户层级
- **缓存命中率**——精确 vs 语义，随时间趋势
- **模型分布**——每个模型的流量百分比，每个模型的成本
- **预算消耗率**——当前支出 vs 按当前速率的预计月度支出
- **拒绝率**——按层级限速的请求百分比

## 常见错误

| 错误 | 伤害原因 | 修复 |
|---------|-------------|-----|
| 在 temperature > 0 时缓存 | 非确定性输出，过时缓存给出错误多样性 | 仅缓存 temp=0 调用，或接受缓存响应失去随机性 |
| 语义缓存阈值过低 | 为表面相似的查询返回错误答案 | 从 0.95 开始，仅在测量误报率后降低 |
| 无缓存失效 | 底层数据变化时响应过时 | 设置 TTL（动态数据 1 小时，静态数据 24 小时），数据更新时失效 |
| 将所有流量路由到最便宜的模型 | 质量下降，用户注意到 | 按复杂度路由，测量每层质量，设置最低质量阈值 |
| 无每用户限制 | 一个滥用用户烧掉整个预算 | 始终实现每用户配额，即使很宽松 |
| 忽略输出 token | 输出成本是输入的 2-5 倍每 token | 适当设置 max_tokens，使用 stop 序列，压缩输出 |
| 提示词稳定前缓存 | 缓存填充旧提示词的响应 | 仅在提示词最终确定后启用缓存，提示词变化时刷新缓存 |
