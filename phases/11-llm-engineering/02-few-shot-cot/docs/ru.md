## Блок 1 — Zero-shot vs Few-shot

**Zero-shot** — даёшь задачу и всё, ноль примеров. **Few-shot** — сначала 3-5 пар
«вход → выход», потом своя задача.

**Главная мысль:** примеры — это сжатые инструкции. Формат вывода не описываешь
словами, а *показываешь*. Модель надёжнее копирует паттерн, чем интерпретирует
абстрактное описание.

Цифры (Wei et al. 2022): на простых задачах zero ≈ few (±2%); на сложных
(многошаговая арифметика, символьные) few-shot даёт **+10-25%**.

**Критерий выбора (рабочая формулировка):**
> Чем больше задача про *общечеловеческое знание* — тем меньше нужны примеры.
> Чем больше про *твои внутренние правила и форматы* — тем сильнее помогают.

- Тональность отзыва = общечеловеческое → zero-shot ок.
- Свои категории тикетов (Биллинг/Сбой/Продажи/Отмена) = внутреннее правило → few-shot.

Даже там, где точность почти не растёт, few-shot всё равно полезен для
**стабилизации формата** и **пограничных случаев** (сарказм, неоднозначность).

Оптимум примеров — **3-5**. Меньше 3 — мало сигнала; больше 5 — отдача падает,
токены тратятся. Для классификации с многими метками — по одному примеру на метку.

### Упражнение 1 — `my/few_shot_vs_zero_shot.py`

Классификация CRM-обращений по 4 своим категориям, zero vs few-shot, temp=0.

**Результат — учебниковый:**

| тикет | zero-shot | few-shot |
|---|---|---|
| двойное списание | `**Двойное списание / Ошибочное списание**` | `биллинг` |
| отчёт не грузится | `**Техническая проблема / Ошибка в интерфейсе**` | `технический сбой` |
| стоимость мест | `**Вопрос о стоимости/тарифах**` | `продажи` |
| закрыть аккаунт | `**Закрытие счёта / вывод средств**` | `отмена` |

Выводы:
1. **Zero-shot гуляет** — выдумывает названия, markdown `**…**`, слеши, по два
   варианта. Для пайплайна это мусор (код ждёт ровно одну из 4 строк).
2. **Few-shot прибился** к набору меток и формату.
3. Пограничный тикет («закрыть + забрать деньги»): zero выдал оба смысла,
   few-shot выбрал один (`отмена`) — примеры разрешили неоднозначность.

**Ключевая находка — регистр.** В первом прогоне один пример был `Продажи` с
заглавной — модель **скопировала регистр буква-в-букву** (вывод: `Продажи`, а
остальные строчными). Вывод:
> Few-shot модель копирует формат примеров целиком, включая случайные
> неаккуратности. Пример — не подсказка, а **эталон для воспроизведения**.
> Few-shot настолько аккуратен, насколько аккуратны примеры.

**Связь с уроком 01 (Drill 2):** примеры положены в `system`, а не `user` — и это
правильно: примеры стабильны в пайплайне → `system` (ради prompt caching);
переменный целевой тикет → `user`.

## Блок 2 — Какие примеры выбирать (похожие > случайные)

Похожие на вход примеры обгоняют случайные на **5-15%** (Liu et al. 2022). Три принципа:

1. **Семантическая близость** — бери примеры, близкие по смыслу к входу.
2. **Разнообразие меток** — покрой все категории (иначе модель «забудет» опцию).
3. **Совпадение сложности** — сложность примеров под сложность задачи.

Принципы 1 и 2 в напряжении: близость тянет набрать похожих (риск — все из одной
категории), разнообразие требует покрыть все метки. Баланс: сначала по примеру на
ключевые метки, оставшиеся слоты — самыми похожими.

**Главное следствие.** Принцип 1 означает, что в проде примеры **не зашиты
намертво**, а подбираются **динамически под каждый вход**: пришёл тикет → нашли в
базе N похожих размеченных → подставили в промпт. Это **мини-RAG**.

**Как искать «похожие» — два способа:**
- **A. По ключевым словам** — просто, но тупо: пропустит синонимы («сняли средства
  после расторжения» не найдётся по словам «деньги/отмена»).
- **B. Векторный поиск** — текст → эмбеддинг → ближайшие по **cosine similarity**
  (та самая из фазы 1). Понимает смысл, не слова. **Это ядро RAG.**

Реализацию НЕ делаем сейчас — нужны эмбеддинги (урок 04) и RAG (урок 06). В этом
уроке — только принцип. «Умный выбор примеров» по факту = динамический few-shot.

**Бумажные упражнения:**
- Подбор 3 примеров под тикет «сняли деньги после отмены» вручную: 1 «в точку по
  смыслу» (якорь на пограничный случай отмена+деньги=биллинг) + 2 «для контраста
  меток». Это решение в проде делает векторный поиск, сейчас — головой.
- Разбор плохого набора (4 примера, все `биллинг`): нарушен принцип 2. Следствие —
  на техническом тикете модель всё равно выдаст `биллинг`, т.к. других меток не
  видела. Та же механика «копирует показанное», но показали однобоко.

## Блок 3 — Chain-of-Thought (CoT)

Few-shot (блоки 1-2) — про *формат*. CoT — про *рассуждение*. Идея: вместо ответа
сразу просишь модель сначала **расписать шаги**, потом ответ.

**Почему реально работает (не магия):** один проход модели (forward pass) =
**фиксированное число шагов вычисления** на токен. Задача в 3 действия не влезает в
один проход — негде произвести шаги. CoT разбивает: модель пишет `48/3=16` (лёгкий
шаг, влезает) → это становится контекстом → пишет следующий шаг → … Каждый шаг
получает свой полный проход. Метафора: дал модели **черновик (scratch paper)**.

**Важно — НЕ путать с MoE.** «Не все нейроны активны, разные эксперты под разные
токены» (Mixture of Experts) — реальная штука, но про *эффективность/ёмкость*, НЕ
про причину CoT. Проверка: модель, жгущая 100% нейронов, всё равно выигрывала бы от
CoT — лимит в *глубине одного прохода*, не в числе активных нейронов. Это разные
механизмы.

| Концепция | Про что | Связь с CoT |
|---|---|---|
| Фиксированная глубина одного прохода | сколько вычисления влезает на 1 токен | **прямая — причина CoT** |
| MoE / sparse activation | какие нейроны включаются ради экономии | отдельная тема |

**Цифры (GSM8K, GPT-4o):** zero-shot 78% → zero-shot CoT 91% → few-shot CoT 95%.

**Две разновидности:**
- **Zero-shot CoT** — добавить фразу-триггер «Давай рассуждать шаг за шагом». Ноль
  примеров (Kojima et al. 2022).
- **Few-shot CoT** — примеры, в которых уже расписано рассуждение. Сильнее, т.к.
  модель видит точный формат рассуждения.

**Когда CoT ВРЕДИТ:** простой факт («столица Франции»), одношаговая классификация
(наш классификатор тикетов!), когда важна скорость/цена (CoT = +50-200 токенов на
запрос). Правило: CoT для задач с **несколькими шагами рассуждения**.

**⚠️ Reasoning-модели** (o-series, R1, Claude extended thinking) уже рассуждают
внутри — добавлять им «think step by step» бесполезно/вредно. Как и prefill из
урока 01: фишка зависит от модели, не универсальна.

### Упражнение 2 — `my/cot_demo.py`

CRM-задача про скидку (многошаговая): plain vs CoT, temp=0.

**Главная находка — режимы дали ПРОТИВОПОЛОЖНЫЕ рекомендации:**

| режим | итог |
|---|---|
| **plain** | `8160 руб. Да` → давать скидку |
| **CoT** | `❌ не стоит` → НЕ давать |

Та же модель, temp=0, те же данные — **противоположный бизнес-совет**.

Почему: plain понял вопрос узко (посчитал прибыль со скидкой → положительна → «Да»).
CoT, разворачивая шаги, сам построил **контрольный сценарий** (без подписки клиент
даёт 19200 vs 8160 со скидкой), увидел что скидка режет маржу зря, и даже
**переформулировал вопрос**: «правильный вопрос не "давать ли скидку", а "нужна ли
подписка"». Вот приз CoT для CRM: plain отвечает на вопрос как задан, CoT замечает,
что вопрос задан неправильно.

**Python-ловушки (пойманы):**
- Цикл `for system in systems` с двумя вызовами `classify` внутри одной итерации
  сравнивал plain×2 и cot×2, а не plain vs cot. Фикс — два явных вызова, цикл не нужен.
- `max_tokens=600` обрезал рассуждение на полуслове → подняли до 1500.
- temp=0: два одинаковых вызова дали символ-в-символ идентичный вывод (детерминизм).

## Блок 4 — Self-Consistency (сэмплируй много, голосуй один раз)

**Проблема:** одна CoT-цепочка может содержать ошибку (модель оступилась на шаге и
уверенно выдала неверный ответ — не заметишь).

**Идея (Wang et al. 2023):** запусти **N независимых рассуждений** при `temp > 0`
(чтобы пути разошлись) и возьми **ответ большинством голосов**. Ошибки разных путей
не совпадают между собой и гасятся, правильный ответ воспроизводится стабильно.
Как опрос экспертов: один ошибётся, большинство сойдётся на верном.

**Зачем temp > 0:** при temp=0 пять прогонов = пять идентичных копий (видели в упр.
2), голосовать не из чего. Нужно разнообразие путей → `temp ≈ 0.7`. Связка с уроком
01: там temp крутили ради креатива, здесь та же ручка работает на **точность**.

**Цифры:** PaLM 540B — GSM8K 56.5% → 74.4% при N=40. На топ-моделях (GPT-5, Claude)
прирост мал (97→98%, потолок). **Блестит на базовой точности 60-85%** — где ошибки
часты, но не систематичны.

**Цена — главный минус:** N путей = N× стоимость и задержка. N=5 ловит почти всё,
N=3 минимум, N>10 без отдачи.

**⚠️ Reasoning-модели** (o-series, R1) сэмплируют внутри — снаружи не надо.

### Упражнение 3 — `my/self_consistency.py` (СДЕЛАНО)

CRM-задача про скидку (та же, что в упр. 2). Маршрут собран целиком:
`one_path()` (CoT, temp>0) × N=5 → `extract_answer()` (regex по маркеру
`РЕКОМЕНДАЦИЯ: да/нет`) → `Counter` → победитель + confidence (доля голосов).
Новые инструменты Python: `collections.Counter` (`.most_common(1)[0]`, распаковка
кортежа) и `re.search` (`|` = «или», `\s*` = «сколько угодно пробелов», группа
`(...)` + `m.group(1)`, флаг `re.IGNORECASE`).

**Заставить модель закончить маркером** (`РЕКОМЕНДАЦИЯ: да/нет` отдельной строкой) —
обязательный приём: без фиксированного формата regex не за что зацепиться.

**ГЛАВНАЯ НАХОДКА — расхождения добиться не удалось, и это сам по себе результат:**

| прогон | задача | temp | итог |
|---|---|---|---|
| 1 | жёсткая арифметика (скидка) | 0.7 | **5/5 нет** |
| 2 | двусмысленная (удержание, перевес минусов) | 0.7 | **5/5 нет** |
| 3 | сбалансированная «команда 50/50» | 1.0 | **5/5 да** |

Даже на сбалансированной формулировке и temp=1.0 (максимум у Anthropic) Haiku даёт
единогласие. **На frontier-Claude self-consistency почти не даёт прибавки** — модель
уже консистентна, пути сэмплинга схлопываются в один ответ, голосовать не за что.
Это эмпирическое подтверждение теоретического «на Claude потолок».

**Выводы для практики и ниши:**
- Self-consistency — техника для **слабых/старых моделей** (GPT-3.5, малые open-source,
  базовые без RLHF), где сэмплинг реально разбрасывает ответы.
- На Claude 4.x платить N× за 5 прогонов ради 100%→100% бессмысленно. Для CRM-агента
  на Claude вкладываемся в **качество промпта**, а не в голосование.
- **Формулировка решает сильнее температуры:** вердикт перевернулся «нет»→«да» от
  перевеса аргументов в промпте, а не от temp. Отсылка к Блоку 1 урока 01.

**Python-замечания по коду (не баги, на будущее):** `extract_answer` лучше возвращать
`None` (а не строку-заглушку), иначе «не найдено» попадёт в `Counter` голосом; перед
голосованием фильтровать `Counter(a for a in answers if a)`; `.lower()` на `group(1)`,
чтобы «да»/«Да» считались одним.

**СЛЕДУЮЩИЙ ШАГ (эта же сессия):** обзорно Tree-of-Thought (дорого, 10-40 вызовов,
для поиска/планирования) и ReAct (рассуждение+действия, фундамент агентов → фаза 14).
Structured prompting и prompt chaining — повтор из урока 01, быстро. Затем закрытие
урока + (опц.) quiz.

## Блок 5 — Tree-of-Thought (ToT, обзорно)

CoT рассуждает одной линией; ошибся на шаге — цепочка испорчена. **ToT** превращает
линию в **дерево**: на каждом шаге модель порождает несколько вариантов продолжения,
*оценивает* их и идёт вглубь по перспективным, тупики отбрасывает с **откатом**
(backtracking). Механика — не один промпт, а цикл: генератор вариантов → оценщик
(1–10) → поиск лучших ветвей.

**Цена — десятки вызовов LLM** (10–40 на задачу) против одного у CoT. Окупается лишь
на задачах с перебором и тупиками: головоломки, планирование с ограничениями, игры
(пример из Yao 2023 — игра «24»). **Для CRM почти не нужен**; на frontier-моделях
похожее закрывают extended thinking или явный prompt chaining. Забрать: **ToT = CoT
+ ветвление + откат + оценка ветвей, ценой десятков вызовов.**

## Блок 6 — ReAct (рассуждение + действия, фундамент агентов)

**Самый важный приём урока для ниши** — скелет любого агента, разворачивается на
полную в фазе 14. Проблема: чистый CoT рассуждает **в вакууме**, не имеет доступа к
фактам системы (остаток на счёте, дата, запрос в БД) → уверенно **галлюцинирует**.

**ReAct** (Reasoning + Acting, Yao 2022) — цикл из трёх шагов:
```
Thought      — что мне сейчас нужно?
Action       — вызвать инструмент (поиск, запрос в БД, API)
Observation  — что вернул инструмент
... снова Thought на основе нового факта, пока ответ не готов
```

CRM-пример («стоит ли слать клиенту win-back»): вместо того чтобы *придумать*, когда
он покупал, модель **сходила и узнала** через `get_last_order()`, потом проверила
подписку — и решает на реальных фактах. CoT придумал бы, ReAct узнаёт.

**Почему фундамент агентов:** агент = ReAct-цикл + набор инструментов (tools) +
остановка по готовности. Function calling (вызов функций) — механизм шага *Action*.
Вся фаза 14 (звезда портфолио — multi-agent orchestration) = ReAct во всю мощь.
Здесь — знакомство с термином, механику соберём руками в фазе 14.

## Блок 7 — Structured prompting + Prompt chaining (повтор из урока 01)

- **Structured prompting** — машиночитаемый вывод (JSON/XML/маркер). Применили в этом
  уроке: строка `РЕКОМЕНДАЦИЯ: да/нет` — это он; без неё `extract_answer` не зацепился бы.
- **Prompt chaining** — разбить задачу на последовательность вызовов, выход одного →
  вход следующего (паттерн Decomposition из урока 01). Мост к пайплайнам и ReAct-циклу.
