# 生产 LLM 检查清单

在发布 LLM 应用时，按顺序完成此检查清单。每个部分都有带具体阈值的通过/失败标准。

## 1. 安全（发布阻塞）

这里的每项必须在任何部署之前通过。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 环境变量中的 API 密钥 | 代码库中零硬编码密钥 | `grep -r "sk-" --include="*.py"` 无返回 |
| 输入护栏激活 | 提示注入模式被阻止 | 发送 "Ignore all previous instructions" -- 返回被阻止的响应 |
| PII 编辑 | SSN、信用卡、邮箱模式被捕获 | 发送 "My SSN is 123-45-6789" -- PII 在 LLM 调用前被编辑 |
| 输出过滤 | 危险内容被阻止 | 模型无法返回 `DROP TABLE`、`rm -rf`、`exec()` 模式 |
| 速率限制 | 每用户请求上限强制执行 | 同一用户在 10 秒内 100 个请求 -- 最后 50+ 被拒绝 |
| 所有端点认证 | 无未认证的 LLM 访问 | `curl /v1/chat` 无 token 返回 401 |
| CORS 限制 | 仅允许生产域名 | `Origin: evil.com` 请求被拒绝 |
| 最大输入 token | 超限请求被拒绝 | 发送 50K token 输入 -- 返回 413 或截断 |

## 2. 可靠性（第一周生存）

这些防止你的第一次 on-call 事故。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 带退避的重试 | 5xx 上 3 次重试，指数延迟 | 杀死 LLM mock 中间请求 -- 日志中可见重试 |
| 降级模型链 | 链中 2+ 模型 | 主模型不可用 -- 响应仍从降级返回 |
| 请求超时 | 所有外部调用最多 30 秒 | 慢 LLM mock (60s) -- 请求在 30 秒超时 |
| 优雅降级 | 缓存/RAG 故障不会使服务崩溃 | 停止缓存 -- 请求仍成功（更慢、更贵） |
| 健康检查端点 | 返回依赖状态 | `GET /health` 返回 `{"status": "healthy", "cache": ..., "llm": ...}` |
| 流式工作 | 首 token 在 500ms 内 | 测量首 token 时间，始终 < 500ms |
| 错误消息安全 | 内部错误从不泄露给用户 | 强制 500 -- 用户看到通用错误，非堆栈跟踪 |

## 3. 成本控制（第一个月经济学）

这些防止 $50K 的意外账单。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 每次请求成本跟踪 | 每个请求记录 token 数 + 美元成本 | 请求日志有 `input_tokens`、`output_tokens`、`cost_usd` 字段 |
| 语义缓存激活 | 重复模式上 > 20% 命中率 | 1000 个测试请求后缓存统计显示命中率 |
| 缓存 TTL 配置 | 条目过期（默认：1 小时） | 插入条目 -- TTL 后不再返回 |
| 每用户成本跟踪 | 按 user_id 聚合成本 | 仪表板/API 显示按成本排序的前 10 用户 |
| 成本告警 | 日预算 80% 时告警 | 设置 $10 日预算，发送 $8.50 请求 -- 告警触发 |
| 按成本模型路由 | 低复杂度查询使用更便宜模型 | 简单问题路由到 gpt-4o-mini，复杂到 gpt-4o |
| 最大输出 token 设置 | 响应按模板上限 | max_output_tokens=512 的模板 -- 响应永不超过它 |

**成本估算公式：**
```
月度 LLM 成本 = DAU x 每用户查询数 x 30 x (1 - 缓存命中率) x (平均输入 token x 输入价格 + 平均输出 token x 输出价格) / 1,000,000
```

**按规模基准阈值：**

| DAU | 目标成本/请求 | 月度预算 |
|-----|-------------------|----------------|
| 1K | < $0.005 | < $750 |
| 10K | < $0.003 | < $4,500 |
| 100K | < $0.001 | < $15,000 |

## 4. 可观测性（生产环境调试）

你无法修复你看不见的东西。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 结构化 JSON 日志 | 每个请求产生 JSON 日志行 | 日志包含：request_id、user_id、model、tokens、latency_ms、cost |
| 请求追踪 | 端到端追踪及组件计时 | 单个请求显示：护栏 (5ms) + 缓存 (2ms) + llm (3200ms) + 评估 (1ms) |
| 延迟跟踪 | 测量 P50、P95、P99 | 1000 个请求后：P50 < 2s，P99 < 10s |
| 错误率监控 | 错误被计数和分类 | 仪表板显示：0.5% API 错误、0.1% 护栏阻止、0.01% 超时 |
| 缓存指标 | 命中率、未命中率、条目数可见 | `GET /v1/cache/stats` 返回当前数字 |
| A/B 测试指标 | 每变体质量指标记录 | 每个请求记录 prompt_template + 版本用于比较 |
| 评估记录 | 每个请求记录质量信号 | 存储响应长度、延迟、模型、模板版本用于离线分析 |

## 5. 提示词管理

提示词是代码。像对待代码一样对待它们。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 版本化模板 | 每个模板有名称 + 版本字符串 | 模板变更创建新版本，旧版本保留 |
| A/B 测试支持 | 按确定性用户哈希分流 | 同一用户在实验内始终看到相同变体 |
| 回滚能力 | < 1 分钟回滚到上一版本 | 变更实验配置 -- 流量立即转移 |
| 模板验证 | 渲染前验证变量 | 模板中缺失变量引发清晰错误，非 KeyError |
| 系统提示词分离 | 系统和用户消息在单独字段中 | 系统提示词不拼接进用户消息 |

## 6. 扩展就绪

发布时不需要。10 倍时需要。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 异步 LLM 调用 | API 调用不阻塞线程 | 50 个并发请求 -- 服务器 CPU 保持 < 30% |
| 连接池 | HTTP 连接复用 | 网络跟踪显示到 LLM 提供商的持久连接 |
| 水平扩展 | 无状态服务器设计 | 负载均衡器后 2 个实例 -- 所有请求成功 |
| 队列支持 | 非实时任务进入队列 | 摘要请求返回 job_id，结果通过轮询可用 |
| 负载测试 | 100 并发用户，< 5% 错误率 | `wrk` 或 `locust` 测试在目标并发下通过 |

## 新项目的实现顺序

1. **第 1 天：** API 服务器 + 提示词模板 + 带重试的单一 LLM 调用
2. **第 2 天：** 输入护栏 + 输出护栏 + 错误处理
3. **第 3 天：** 语义缓存 + 每次请求成本跟踪
4. **第 4 天：** 流式 (SSE) + 健康检查端点
5. **第 5 天：** 结构化日志 + 请求追踪 + 评估记录
6. **第 2 周：** A/B 测试 + 提示词版本控制 + 回滚
7. **第 3 周：** 降级模型链 + 优雅降级
8. **第 4 周：** 负载测试 + 异步优化 + 水平扩展

## 快速诊断

如果生产环境出问题，按此顺序检查：

1. **用户抱怨错误？** 检查健康端点，然后日志中的错误率，然后 LLM 提供商状态页面
2. **响应慢？** 检查 P99 延迟，然后缓存命中率，然后追踪中的 LLM 响应时间
3. **成本飙升？** 检查每次请求成本趋势，然后缓存命中率，然后按成本排序的顶部用户，然后查找增加 token 数的提示词模板变更
4. **质量下降？** 检查是否部署了新提示词版本，检查 RAG 检索准确率是否变化，检查模型提供商是否更改了默认模型版本
5. **安全事件？** 检查护栏阻止率（突然下降 = 护栏禁用），检查请求日志中的异常模式，立即轮换 API 密钥
