# 上下文工程

在构建 LLM 应用时，应用此框架来设计上下文组装流水线。

## 核心原则

1. **上下文是稀缺的。** 128K 窗口听起来很大，但很快就满了。为每个组件明确预算。
2. **注意力是不均匀的。** 模型更关注开头和结尾。将关键信息放在那里。中间是死角。
3. **动态优于静态。** 不同查询需要不同上下文。按查询组装，而非启动时一次性组装。
4. **少即是多。** 精心策划的 10K 上下文优于倾倒的 100K 上下文。信噪比比总信息量更重要。
5. **测量一切。** 你无法优化你不测量的东西。每次请求都计算每个组件的 token。

## 上下文预算指南

| 组件 | 典型范围 | 优先级 | 压缩策略 |
|-----------|-------------|----------|---------------------|
| 系统提示词 | 200-1,000 tokens | 固定，高 | 写得紧凑，去除冗余 |
| 工具定义 | 500-3,000 tokens | 动态，中 | 按查询意图裁剪 |
| 检索上下文 | 1,000-5,000 tokens | 动态，高 | 重排序 + 阈值 + 去重 |
| 对话历史 | 500-5,000 tokens | 动态，中 | 摘要旧轮次 |
| 少样本示例 | 500-2,000 tokens | 动态，高 | 按任务相似度选择 |
| 用户查询 | 50-500 tokens | 固定，最高 | N/A |
| 生成预留 | 2,000-8,000 tokens | 固定 | 按预期输出长度调整 |

## 何时使用每种记忆类型

**短期（对话历史）：** 当前会话。通过摘要管理。压缩超过 5-10 轮交换的旧轮次。保留最后 3-4 轮原文。

**长期（事实数据库）：** 跨会话持久化的偏好和项目事实。在会话开始时检索。示例："用户偏好 Python"、"项目使用 PostgreSQL"、"团队遵循基于主干的开发"。存储在 CLAUDE.md、数据库或结构化记忆系统中。

**情景（过去交互）：** 与当前任务相关的特定过去对话。存储为嵌入，按相似度检索。"上周我们调试了一个类似的认证问题"就是情景记忆。

## 工具选择策略

不要在每个请求中都包含所有工具。这会浪费 token 并让模型困惑。

1. 分类查询意图（代码、邮件、日历、研究、数据）
2. 将意图映射到工具类别
3. 仅包含匹配的工具
4. 如果意图模糊，包含前 2 个类别的工具
5. 始终包含一个"通用"工具（如网页搜索）作为后备

预期节省：在意图明确的查询上节省 60-80% 的工具定义 token。

## 检索最佳实践

- **检索后重排序。** 向量相似度是一个粗略的过滤器。重排序器（交叉编码器或基于 LLM 的）显著提高精确率。
- **设置相关性阈值。** 不要包含余弦相似度低于 0.3 的分块。它们增加噪音。
- **去重。** 如果两个分块共享 80%+ 内容，仅保留分数较高的那个。
- **应用 lost-in-the-middle 排序。** 将最相关的分块放在开头和结尾。
- **限制总检索 token。** 3-5 个高度相关的分块胜过 15 个 mediocre 的分块。

## 历史管理

- 保留最后 3-4 轮原文（模型需要近期上下文）
- 将旧轮次摘要为摘要（"我们讨论了 X，决定了 Y，并在 Z 上受阻"）
- 删除不添加信息的系统生成轮次（无用户可见内容的工具调用）
- 当历史超过可用预算的 30% 时触发压缩

## 红旗

- 系统提示词超过 2,000 tokens：可能包含了应该是动态的信息
- 每个请求都包含所有工具：实现基于意图的选择
- 检索无相关性过滤：你在向窗口倾倒噪音
- 历史无界增长：未实现摘要
- 无生成预留：模型截断其响应
- 相同信息出现在 3 个地方（系统提示词、检索文档、历史）：去重
- 上下文利用率超过 60%：你留给模型"思考"的空间太少
