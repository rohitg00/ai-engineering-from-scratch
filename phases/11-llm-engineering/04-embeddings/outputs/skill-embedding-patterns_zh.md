# 嵌入模式

每个嵌入工作流都遵循以下约定：

```
text -> embed(text) -> vector (float array)
similarity(vector_a, vector_b) -> score (float)
```

嵌入模型和相似度度量是唯一重要的两个决策。其他都是管道工程。

## 何时使用嵌入

- 跨文档的语义搜索（找含义，而非关键词）
- 相似项目聚类（支持工单、产品评论、bug 报告）
- 最近邻分类（通过与标注示例的相似度为新项目打标签）
- 推荐系统（找到与用户喜欢的项目相似的项目）
- 去重（使用相似度阈值找到近似重复内容）

## 何时不使用嵌入

- 精确关键词匹配（使用全文搜索）
- 结构化查询（使用 SQL、过滤器）
- 小数据集，手动标注更快（<100 项）
- 可解释性比准确率更重要的任务（嵌入是不透明的）

## 模型选择

根据你的约束选择：

- **需要 API，最佳性价比**：OpenAI text-embedding-3-small (1536d, $0.02/1M tokens)
- **需要最高准确率**：Voyage-3 (1024d, $0.06/1M tokens, 最高 MTEB)
- **需要本地/私有**：BGE-M3 (1024d, free, 多语言, 推荐 GPU)
- **需要快速本地原型**：all-MiniLM-L6-v2 (384d, free, 可在 CPU 上运行)
- **需要多语言**：Cohere embed-v3 (1024d) 或 BGE-M3（两者都强大多语言）

规则：切勿在索引和查询之间混合使用不同的嵌入模型。来自不同模型的向量存在于不兼容的空间中。

## 分块规则

1. 目标每块 256-512 tokens，50 token 重叠
2. 尽可能避免在句子中间拆分
3. 为每块包含元数据（源文件、章节标题、位置）
4. 对于结构化文档（Markdown、HTML），首先在标题边界处拆分
5. 通过搜索已知答案并检查检索质量来测试分块质量

## 相似度度量选择

- **余弦相似度**：默认选择，处理变长文本，已归一化
- **点积**：当向量已经是单位归一化时使用（OpenAI 模型就是），稍快
- **欧氏距离**：用于聚类，当绝对位置重要时

当向量归一化时，三者给出相同的排序。选择仅对非归一化向量重要。

## 存储优化

三个可叠加的压缩级别：

1. **Matryoshka 截断**：减少维度（1536 -> 256 = 节省 6 倍，3-5% 准确率损失）
2. **Float16 量化**：每维度存储减半（节省 2 倍，<1% 准确率损失）
3. **二值量化**：每维度 1 位（节省 32 倍，5-10% 准确率损失，配合重排序使用）

生产模式：在全语料库上进行二值搜索，用 float32 向量对前 1000 个重排序。

## 检索-然后-重排序

两阶段流水线以获得最佳准确率：

1. 双编码器检索前 100 个候选（快速，使用预计算嵌入）
2. 交叉编码器重排序到前 10 个（慢速，处理每个查询-文档对）

这在精确率指标上比单阶段检索高出 10-15%。当准确率比延迟更重要时使用。

## 常见错误

- 索引和查询使用不同的嵌入模型
- 嵌入整个文档而非分块（嵌入变成所有内容的平均值）
- 余弦相似度前未归一化向量（大多数模型预归一化，但需验证）
- 忽略分块重叠（边界处拆分的句子丢失上下文）
- 仅存储向量而不存储原始文本（检索时两者都需要）
- 模型更改时未重新嵌入（旧向量不兼容）
- 仅基于准确率选择维度（存储和延迟与维度成线性关系）

## 调试嵌入

如果搜索结果不佳：

1. 验证查询嵌入是否非零（空或空白输入会产生零向量）
2. 手动检查已知相关文档的相似度分数
3. 尝试改写查询以匹配文档词汇
4. 检查分块边界，确保相关内容未跨块拆分
5. 跨度量比较 top-k 结果（余弦、点积、欧氏）以发现归一化问题
6. 使用简单匹配查询测试（从文档中复制一个句子）以确认流水线正常工作

## 生产参数

- 分块大小：256-512 tokens
- 分块重叠：50 tokens（分块大小的 10-20%）
- Top-k 检索：直接使用 5-10，重排序 50-100
- 相似度阈值：余弦 0.7+（低于此值，结果通常不相关）
- 批量嵌入：每次 API 调用处理 100-500 个文本以提高吞吐量
- 索引重建：模型更改或文档显著更新时重新嵌入
