# 22 · 异步与 Hogwild! 推理

> 「推测解码（speculative decoding）」（阶段 10 · 15）在单个序列内部并行化 token。多智能体框架则跨越整个序列进行并行，但强制要求显式协调（投票、子任务拆分）。「Hogwild! 推理（Hogwild! Inference）」（Rodionov 等人，arXiv:2504.06261）做的是另一件事：让同一个 LLM 的 N 个实例并行运行，共享同一份「键值缓存（key-value cache）」。每个 worker 都能即时看到所有其他 worker 生成的 token。现代推理模型——QwQ、DeepSeek-R1——无需任何微调，就能通过这份共享缓存实现自我协调。该方法尚处实验阶段，但它开辟了一条全新的推理并行轴，与推测解码相互正交。本课用标准库 Python 实现一个双 worker 的 Hogwild! 模拟器，并解释为何这种共享缓存协作会从模型既有的推理能力中自发涌现。

**类型：** 构建
**语言：** Python（标准库）
**前置：** 阶段 10 · 12（推理优化）、阶段 10 · 15（推测解码）
**时长：** 约 60 分钟

## 学习目标

- 描述三种常见的并行 LLM 拓扑（投票、子任务、Hogwild!），并说出每种各自针对哪类问题。
- 阐述 Hogwild! 的核心设置：多个 worker、一份共享 KV 缓存、通过自我提示（self-prompting）涌现的协调。
- 把 Hogwild! 的墙钟（wall-time）加速比表达为 worker 数量 `N`、任务级并行度 `p` 和协调开销 `c` 的函数并计算之。
- 在一个玩具问题上实现双 worker 的 Hogwild! 模拟器，观察自发涌现的任务分工。

## 问题所在

现代 LLM 通过产出长链推理来求解难题——5000 个 token 的逐步逻辑很常见，深度数学问题上则会出现几万个 token。在 70B 模型上以 35 token/秒的解码速度，50k 个 token 需要 24 分钟。这样的模型谈不上交互式。

推测解码（阶段 10 · 15）通过在单序列内部并行化为你带来 3-5 倍加速。再往上，自回归解码的串行依赖就是硬天花板。每个新 token 都依赖于此前所有 token。

显而易见的问题：我们能否跨序列并行？跑同一模型的多份副本去解同一个问题，让它们协作、分工？

既有工作：「投票集成（voting ensemble）」（跑 N 个模型，取多数答案）、「思维树（tree-of-thought）」（分叉推理路径再重组）以及多智能体框架（给每个智能体分配一个子任务，用一个协调者统筹）。这些在特定任务领域都有帮助。它们也都引入了显式的协调机制——投票规则、分支剪枝逻辑、智能体间消息协议。

Hogwild! 推理走的是另一条路。N 个 worker 共享单一 KV 缓存。每个 worker 都能立即看到所有其他 worker 生成的 token，就好像它们是自己的上下文一样。这些 worker——无需任何训练或微调——会自行摸索出如何分工。现代推理模型（QwQ、DeepSeek-R1、Claude 系列的推理模式）能读取共享缓存并说出类似这样的话：「我看到 worker 2 已经处理了基础情形，所以我来做归纳步骤。」

加速比取决于工作负载，截至 2026 年 4 月仍属实验性。但这一思路值得了解，因为它开辟了推理并行的一条新轴。

## 概念

### 设置

初始化 N 个 worker 进程，全部运行同一个 LLM。不再为每个 worker 维护各自的 KV 缓存，而是维护一份共享缓存。当 worker `i` 生成 token `t_j` 时，该 token 被写入共享缓存的下一个位置。当 worker `k` 执行下一步时，它读取缓存的当前状态（其中包含到目前为止所有 N 个 worker 已生成的一切）。

在每一步执行时，worker 们竞相写入 token。不存在每个 worker 各自的位置索引——缓存就是一条单一的、不断增长的序列。顺序由写入到达时间决定。

### 协调为何会涌现

worker 们共享一段提示词。通常类似于「你是协同解决该问题的 N 个实例之一。每个实例都读取共享内存，并能看到其他实例写了什么。避免重复劳动。」这段提示词加上共享缓存就足够了。推理模型读取缓存，注意到问题的哪些部分已经被尝试过，然后（通常但并非总是）转向尚未探索的部分。

Hogwild! 论文（Rodionov 等人，2025）报告了诸如以下观察：

- worker 们制定计划，并通过缓存把计划传达给其他 worker。
- worker 们会注意到其他 worker 推理中的错误并指出来。
- 当某个计划失败时，worker 们会调整并提出替代方案。
- 当被提示检查是否重复时，worker 们能检测出重复并转向别处。

这一切都不需要微调。这种涌现行为来自模型已经具备的推理能力。

### 命名

这篇论文的名字借鉴了「Hogwild! SGD」（Recht 等人，2011），一种异步更新的优化器。类比在于：SGD 的异步 worker 都写入一个共享的参数向量；Hogwild! 推理的 worker 都写入一份共享的 KV 缓存。两者都依赖经验上的收敛，而非同步保证。

### RoPE 让这件事变得可行

「旋转位置编码（Rotary Position Embeddings，RoPE）」（Su 等人，2021）通过对 Q、K 向量施加旋转来编码位置信息。由于位置是旋转而非固化的偏移量，token 的位置可以移动而无需重算 KV 缓存条目。当 worker `i` 在位置 `p` 写入共享缓存时，读取该位置的其他 worker 可以直接使用缓存条目——无需重新旋转。

在采用学习位置或绝对位置的模型中，Hogwild! 在每次并发写入时都需要缓存失效。RoPE 让缓存得以保持稳定。

### 墙钟时间数学

设 `T_serial` 为单个 worker 独自求解问题所需的时间。设 `p` 为任务级可并行比例。设 `c` 为每一步的协调开销（读取被扩展后的缓存、决定写什么）。

单 worker 时间：`T_serial`。
N worker 的 Hogwild! 时间，若协调免费：`T_serial * ((1 - p) + p / N)`。经典阿姆达尔定律。
计入协调开销：`T_serial * ((1 - p) + p / N) + c * steps_per_worker`。

要让一个 worker 真正有产出，`c` 相对于每步解码时间必须很小。在产出 5k+ token 的推理模型上，worker 们可以承担数百个 token 的协调开销，仍然占优。在短对话任务上，协调占主导，Hogwild! 反而比串行更差。

### 具体示例

推理问题：10k 个 token 的思维链。假设该问题有 `p = 0.7` 的可并行内容（不同证明策略、不同情形分析），且每个 worker 有 `c = 200` 个 token 的协调开销。在 `N = 4` 个 worker 下：

- 串行时间：10000 个解码步。
- Hogwild! 时间：10000 * (0.3 + 0.7 / 4) + 200 * 4 = 10000 * 0.475 + 800 = 5550 个解码步。
- 加速比：10000 / 5550 = 1.8 倍。

这一加速比并不大。但在更长的推理问题上（50k 个 token），协调开销被摊薄，加速比可推到 2.5-3 倍。Hogwild! 相当于推理领域的线程级并行——就像在一门让你能自然写出多线程代码的语言里那样。

### 何时该用 Hogwild!

- 长推理问题（数千个 token），且任务能跨独立子目标并行化。
- 经过逐步思考训练的推理模型。非推理模型自我协调得不好。
- 单节点部署，且有足够的 VRAM 容纳共享缓存外加 N 个 worker 进程。缓存是共享的，但每个 worker 有自己的激活内存。

### 何时不该用

- 短交互式对话。协调开销占主导。
- 无法并行化的任务（单条线性证明、单次编译）。N=1 就是上限。
- 非推理模型。不会涌现协调。
- 多节点部署。共享缓存需要极快的跨 worker 同步。节点内尚可，跨节点则是延迟灾难。

### 实验性状态

截至 2026 年 4 月，Hogwild! 是一种带有开源 PyTorch 实现的研究方法，尚未进入生产采用。有三个拦路虎：

1. 跨并发进程管理共享 KV 缓存是一项不平凡的工程。
2. 涌现的协调依赖于任务；基准测试仍在构建中。
3. 与推测解码已经能带来的加速相比，Hogwild! 的加速比并不大，两者可以结合，但结合后的工程又是另一层复杂度。

值得了解。值得实验。但还不值得押上一款产品。

## 动手构建

`code/main.py` 实现了一个玩具版 Hogwild! 模拟器：

- 两个 worker 进程，每个都是确定性的「LLM」，按已知概率产出若干 token 类别之一（work-token、observe-token、coordinate-token）。
- 一份共享缓存（就是一个 token 列表），两个 worker 都读写它。
- 一条简单的协调逻辑：当某个 worker 发现另一个 worker 已经在某个类别上产出了足够多的 work token 时，它就改选另一个类别。

模拟器在固定的步数预算内运行，并报告：

- 产出的 work-token 总数。
- 总墙钟时间（worker 步数）。
- 相对单个 worker 的有效加速比。
- 哪个 worker 写了哪个 token 的轨迹记录。

### 第 1 步：共享缓存

一个两个 worker 都向其追加的列表。在真实实现中用简单的加锁（Python `threading.Lock`）；这里我们用一个计数器来模拟。

### 第 2 步：worker 循环

每个 worker 在每一步：

- 读取当前的共享缓存。
- 根据其中已有的内容决定写入哪个类别的 token。
- 写入一个 token。

### 第 3 步：协调启发式

如果类别 X 在缓存中已经有 K 个 token，而 worker 本打算写的就是类别 X，那么 worker 就切换到类别 Y。这是对推理模型那种「注意到这块已被覆盖，转而做别的事」行为的玩具级替身。

### 第 4 步：测量加速比

用 N=1 个 worker 和 N=2 个 worker 各运行一次模拟器，使用相同的总步数预算。统计产出的 work-token 数。由于协调驱动的任务分工，N=2 应当产出大约多 1.5-1.8 倍的 work-token。

### 第 5 步：给协调施压

降低协调启发式的灵敏度。再运行一次。观察到：没有良好的协调，N=2 会重复产出相同的 token，加速比跌到 1 以下。这与论文的观察一致：只有当 worker 们具备自我协调的推理能力时，这套把戏才奏效。

## 上手运用

截至 2026 年 4 月，Hogwild! 在生产中的集成仍属研究级别。来自 Yandex/HSE/IST 的参考实现基于 PyTorch，面向 DeepSeek-R1 和 QwQ 模型的单节点多进程设置。

务实的采用路径：

1. 剖析你的推理任务工作负载。测量其中属于探索性（多种策略、情形分析、搜索）的 token 占比与线性 token 占比。
2. 若探索性占主导，跑一次双 worker 的 Hogwild! 实验。测量墙钟时间的改善。
3. 若改善低于 1.3 倍，说明你处于协调主导的区间。退回单 worker。
4. 若改善高于 1.5 倍，推到 N=4 再测一次。收益递减通常在 N=4-8 附近显现。

与推测解码结合：每个 Hogwild! worker 都能独立使用推测解码。两种加速（大致上）相乘——3 倍的推测解码与 1.8 倍的 Hogwild! 结合，相对朴素的单 worker 解码可达到约 5.4 倍的有效加速。

## 关键术语

| 术语 | 人们怎么说 | 它实际指什么 |
|------|----------------|------------------------|
| Hogwild! | 「并行 worker，共享缓存」 | 同一个 LLM 的 N 个实例并发运行，共用一份 KV 缓存；通过自我提示涌现协调 |
| 共享 KV 缓存 | 「协调媒介」 | 所有 worker 都读写的一个不断增长的 KV 缓冲区；让 token 在 worker 间即时可见 |
| 涌现协调 | 「无需训练」 | 具备推理能力的 LLM 能读取共享缓存并分工，无需任何微调或显式协议 |
| 协调开销（c） | 「用于定位状态的 token」 | 每个 worker 读取扩展缓存并决定要做什么的开销；相对总解码时间必须保持很小 |
| 可并行比例（p） | 「能并行跑的部分」 | 任务级并行度：总工作中并非本质串行的那一部分 |
| RoPE 使 Hogwild! 成为可能 | 「旋转位置是平移不变的」 | 因为位置是旋转，写入共享缓存不需要重算先前的 token |
| 投票集成 | 「跑 N 个，取多数」 | 最简单的并行推理拓扑；对分类有用，对长篇推理用处较小 |
| 思维树 | 「分支与剪枝」 | 探索多条分支并剪枝的推理策略；带有显式协调逻辑 |
| 多智能体框架 | 「分配子任务」 | 每个智能体获得一个角色；一个协调者负责编排；协议开销沉重 |
