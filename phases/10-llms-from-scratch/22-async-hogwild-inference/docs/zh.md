# 异步和Hogwild!推理

> 推测解码（Speculative decoding，Phase 10 · 15）在单个序列内并行化token。多智能体框架（Multi-agent frameworks）跨整个序列并行化，但强制要求显式协调（投票、子任务分割）。Hogwild!推理（Rodionov et al., arXiv:2504.06261）则采用不同的方法：并行运行N个相同的LLM实例，使用一个共享的键值缓存（SHARED key-value cache）。每个工作进程（worker）能即时看到其他工作进程生成的token。现代推理模型——QwQ、DeepSeek-R1——能够通过该共享缓存进行自我协调，无需任何微调。这种方法是实验性的，但它开辟了一个全新的推理并行化维度，与推测解码正交。本课程在stdlib Python中实现了一个双工作进程Hogwild!模拟器，并解释了共享缓存协作如何从现有模型的推理能力中涌现。

**类型：** 构建
**语言：** Python (stdlib)
**先修知识：** Phase 10 · 12 (推理优化), Phase 10 · 15 (推测解码)
**时间：** 约60分钟

## 学习目标

- 描述三种常见的并行LLM拓扑结构（投票、子任务、Hogwild!），并指出每种结构针对的问题。
- 阐述Hogwild!的核心设置：多个工作进程、一个共享KV缓存、通过自我提示（self-prompting）涌现的协调机制。
- 计算Hogwild!的墙钟时间（wall-time）加速比，作为工作进程数量`N`、任务级并行度`p`和协调开销`c`的函数。
- 在一个玩具问题上实现双工作进程Hogwild!模拟器，并观察涌现的任务划分。

## 问题

现代LLM通过生成长链推理来解决难题——5000个token的逐步逻辑推理很常见，在深度数学问题中会出现数万个token。在70B模型上以35 tokens/sec的速度解码，50k tokens需要24分钟。交互式使用模型是不可能的。

推测解码（Phase 10 · 15）通过在单个序列内并行化获得3-5倍的加速。超过这一点，自回归解码（autoregressive decoding）的顺序依赖性就是硬性上限。每个新token依赖于之前的所有token。

显而易见的问题是：我们能否跨序列并行化？在同一问题上运行多个相同的模型副本，让它们协作，让它们分工？

先前的工作：投票集成（voting ensembles，运行N个模型，选择多数答案）、思维树（tree-of-thought，分支推理路径并重组）和多智能体框架（multi-agent frameworks，为每个智能体分配子任务，使用协调器）。这些都在特定任务领域有所帮助。它们也都引入了显式的协调机制——投票规则、分支修剪逻辑、智能体间消息传递协议。

Hogwild!推理采用不同的方法。N个工作进程共享一个单一的KV缓存。每个工作进程立即看到其他工作进程生成的token，就像它们是自己上下文的一部分一样。工作进程——无需任何训练或微调——自行决定如何分工。现代推理模型（QwQ、DeepSeek-R1、Claude系列推理模式）可以读取共享缓存并说出类似"我看到工作进程2已经处理了基础情况，所以我将处理归纳步骤"这样的话。

截至2026年4月，加速比取决于工作负载且处于实验阶段。但这个想法值得了解，因为它开辟了推理并行化的一个新维度。

## 概念

### 设置

初始化N个工作进程，都运行相同的LLM。不使用每个工作进程独立的KV缓存，而是维护一个共享缓存。当工作进程`i`生成token `t_j`时，该token被写入共享缓存的下一个位置。当工作进程`k`进行下一步时，它读取缓存的当前状态（包含所有N个工作进程到目前为止生成的所有内容）。

在每一步，工作进程竞速写入token。没有每个工作进程的位置索引——缓存是一个单一增长的序列。顺序由写入到达时间决定。

### 为什么协调会涌现

工作进程共享一个提示（prompt）。通常是类似"You are one of N instances working together on this problem. Each instance reads the shared memory and can see what other instances have written. Avoid redundant work."（你是N个共同解决此问题的实例之一。每个实例读取共享内存并可以看到其他实例写入的内容。避免冗余工作。）这样的内容。提示加上共享缓存就足够了。推理模型读取缓存，注意到问题的哪些部分已经被尝试过，并且（经常但不总是）转向未探索的部分。

Hogwild!论文（Rodionov et al., 2025）报告了以下观察：
- 工作进程制定计划并通过缓存与其他工作进程沟通。
- 工作进程注意到其他工作进程推理中的错误并指出。
- 当计划失败时，工作进程进行调整并提出替代方案。
- 当被提示检查冗余时，工作进程检测到冗余并转向其他工作。

这些都不需要微调。涌现的行为来自模型已有的推理能力。

### 命名

论文的名称借鉴了Hogwild! SGD（Recht et al., 2011），这是一种异步更新优化器。类比：SGD的异步工作进程都写入共享参数向量；Hogwild!推理的工作进程都写入共享KV缓存。两者都依赖于经验收敛而非同步保证。

### RoPE使这一点可行

旋转位置编码（Rotary Position Embeddings，RoPE，Su et al. 2021）通过在Q和K向量中的旋转来编码位置信息。因为位置是旋转而非内置的偏移量，token的位置可以移动而无需重新计算KV缓存条目。当工作进程`i`在位置`p`写入共享缓存时，其他读取该位置的工作进程可以直接使用缓存的条目——无需重新旋转。

在学习的位置或绝对位置模型中，Hogwild!每次并发写入都需要缓存失效。RoPE使缓存保持稳定。

### 墙钟时间计算

让`T_serial`为一个工作进程单独解决问题所需的时间。让`p`为任务级可并行化的比例。让`c`为每步协调开销（读取扩展的缓存，决定写入什么）。

单工作进程时间：`T_serial`。
如果协调免费，N个工作进程的Hogwild!时间：`T_serial * ((1 - p) + p / N)`。经典的Amdahl定律。
有协调开销时：`T_serial * ((1 - p) + p / N) + c * steps_per_worker`。

为了使工作进程有生产效率，`c`必须相对于每步解码时间较小。在生成5k+ tokens的推理模型上，工作进程可以承受数百个token的协调开销并仍然保持领先。在短聊天任务中，协调占主导，Hogwild!比串行更差。

### 具体示例

推理问题：10k tokens的链式思维。假设问题有`p = 0.7`的可并行化内容（不同的证明策略、不同的案例分析）和每个工作进程`c = 200` tokens的协调开销。使用`N = 4`个工作进程：
- 串行时间：10000个解码步骤。
- Hogwild!时间：10000 * (0.3 + 0.7 / 4) + 200 * 4 = 10000 * 0.475 + 800 = 5550个解码步骤。
- 加速比：10000 / 5550 = 1.8倍。

这是适度的。但在更长的推理问题（50k tokens）上，协调开销被摊销，加速比提升到2.5-3倍。Hogwild!是推理中线程级并行（thread-level parallelism）的等价物，就像一种让你自然编写多线程代码的语言。

### 何时使用Hogwild!

- 长推理问题（数千tokens），任务可以跨独立子目标并行化。
- 经过训练进行逐步思考的推理模型。非推理模型不能很好地自我协调。
- 单节点部署，有足够的VRAM来容纳共享缓存和N个工作进程。缓存是共享的，但每个工作进程有自己的激活内存。

### 何时不使用

- 短交互式聊天。协调开销占主导。
- 不能并行化的任务（单一线性证明、单一编译）。N=1是最大值。
- 非推理模型。没有协调涌现。
- 多节点部署。共享缓存需要非常快的跨工作进程同步。节点内没问题；跨节点是延迟灾难。

### 实验状态

截至2026年4月，Hogwild!是一种具有开源PyTorch实现的研究方法。尚未在生产中采用。有三个障碍：
1. 跨并发进程的共享KV缓存管理是非平凡的工程。
2. 涌现的协调取决于任务；基准测试仍在构建中。
3. 与推测解码已经提供的加速相比，这些加速是适度的，两者可以结合，但结合的工程是另一层。

值得了解。值得实验。但还不值得为产品下注。

## 构建它

`code/main.py`实现了一个玩具Hogwild!模拟器：

- 两个工作进程，每个是一个确定性"LLM"，以已知概率生成几种token类别之一（工作token、观察token、协调token）。
- 一个共享缓存（只是一个token列表），两个工作进程都读取和写入。
- 一个简单的协调逻辑：当一个工作进程看到另一个已经在某个类别中产生了足够多的工作token时，它会选择另一个类别。

模拟器运行固定的步预算并报告：

- 产生的总工作token数。
- 总墙钟时间（工作进程步数）。
- 相比单个工作进程的有效加速比。
- 哪个工作进程写了哪个token的跟踪。

### 步骤1：共享缓存

一个两个工作进程都追加到的列表。在实际实现中使用简单的锁定（Python `threading.Lock`）；我们用一个计数器来模拟。

### 步骤2：工作进程循环

每个工作进程，在每一步：

- 读取当前共享缓存。
- 根据已有的内容决定写入哪种类别的token。
- 写入一个token。

### 步骤3：协调启发式

如果类别X在缓存中已经有K个token，且工作进程的预期类别是X，则工作进程切换到类别Y。这是推理模型行为的玩具替代品："注意到这已经被覆盖了，做点别的事情"。

### 步骤4：测量加速比

使用N=1个工作进程和N=2个工作进程运行模拟器，使用相同的总步预算。计算产生的work-token数。N=2应该产生大约1.5-1.8倍更多的work-token，因为这是由协调驱动的任务划分。

### 步骤5：测试协调

降低协调启发式的敏感度。再次运行。观察到没有良好的协调时，N=2会冗余地产生相同的token，加速比降至1以下。这与论文的观察一致：只有当工作进程具有自我协调的推理能力时，这个技巧才有效。

## 使用它

截至2026年4月，生产中的Hogwild!集成是研究级别的。来自Yandex/HSE/IST的参考实现基于PyTorch，针对DeepSeek-R1和QwQ模型上的单节点多进程设置。

实用的采用路径：

1. 分析你的推理任务工作负载。测量探索性（多种策略、案例分析、搜索）与线性token的比例。
2. 如果探索性占主导，运行双工作进程Hogwild!实验。测量墙钟时间改进。
3. 如果改进低于1.3倍，你就处于协调占主导的区域。恢复为单工作进程。
4. 如果改进超过1.5倍，推进到N=4并再次测量。收益递减通常在N=4-8时出现。

与推测解码结合：每个Hogwild!工作进程可以独立使用spec decode。两种加速比相乘（大致），将3倍的spec decode和1.8倍的Hogwild!带到比朴素单工作进程解码有效的5.4倍。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| Hogwild! | "并行工作进程，共享缓存" | N个相同的LLM实例并发运行，使用一个共享的KV缓存；通过自我提示涌现的协调机制 |
| 共享KV缓存 | "协调媒介" | 一个所有工作进程都读取和写入的单一增长的KV缓冲区；实现工作进程间的即时token可见性 |
| 涌现的协调 | "无需训练" | 具有推理能力的LLM可以读取共享缓存并划分工作，无需任何微调或显式协议 |
| 协调开销 (c) | "用于定向的token" | 每个工作进程读取扩展缓存和决定做什么的成本；必须相对于总解码时间保持较小 |
| 可并行化比例 (p) | "可以并行运行的部分" | 任务级并行性：总工作中非内在顺序的部分比例 |
| RoPE使Hogwild!可行 | "旋转位置是移位不变的" | 因为位置是旋转，写入共享缓存不需要重新计算之前的token |
| 投票集成 | "运行N个，选择多数" | 最简单的并行推理拓扑；对分类有用，对长篇推理帮助较少 |
| 思维树 | "分支和修剪" | 探索多个分支并修剪的推理策略；显式协调逻辑 |
| 多智能体框架 | "分配子任务" | 每个智能体获得一个角色；一个协调器编排；沉重的协议开销 |
