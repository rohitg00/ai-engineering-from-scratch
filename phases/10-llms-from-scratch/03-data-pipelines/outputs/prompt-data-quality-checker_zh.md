# LLM 预训练数据质量检查器

构建或审计 LLM 预训练数据流水线时，使用此框架在问题到达模型之前捕获它们。

## 流水线输出中的危险信号

**去重移除率低于 20%。** Common Crawl 通常包含 30-40% 的重复内容。如果你的去重步骤移除率低于 20%，说明 MinHash 参数过于保守或阈值过高。检查：shingle 大小 k、哈希函数数量、LSH  band 数量、Jaccard 阈值。

**压缩率低于 2.0 字符/token。** 这意味着你的分词器拆分过于激进。要么用更多合并重新训练，增加词表大小，或检查预分词是否不必要地碎片化文本。

**压缩率高于 6.0 字符/token。** 你的分词器学习了非常领域特定的合并，可能无法泛化。对于领域特定模型这没问题，但对于通用模型是警告信号。

**序列利用率低于 90%。** 填充太多。要么你的文档很短（过滤它们或增加最小文档长度），要么你的序列打包效率低下（从朴素填充切换到多文档打包）。

**词表利用率低于 50%。** 超过一半的词表在此语料库上未被使用。要么词表对你的领域来说太大，要么分词器是在非常不同的数据上训练的。

## 质量过滤器校准

在流水线的每个阶段对 1,000 个文档的随机样本运行这些检查：

1. **清洗后阅读 20 个随机文档。** 它们是否包含残留 HTML、JavaScript、导航文本或样板内容？如果是，你的 HTML 剥离不完整。

2. **阅读 20 个通过质量过滤器的随机文档。** 它们中是否有垃圾邮件、关键词列表或机器生成的内容？如果是，收紧过滤器阈值。

3. **阅读 20 个未通过质量过滤器的随机文档。** 它们中是否有真正优质的内容？如果是，你的过滤器过于激进。放宽阈值或为特定模式添加例外。

4. **阅读 20 个去重后的随机近似重复对。** 它们是否真的相似？如果不是，降低 Jaccard 阈值或增加哈希函数数量。

## 数据混合比例

没有通用公式。从这些基线开始，根据评估结果调整：

| 类别 | Llama 3 比例 | 起始点 |
|----------|--------------|----------------|
| 网页文本 | 50% | 50% |
| 代码 | 25% | 15-25% |
| 书籍/学术 | 13% | 10-15% |
| 数学 | 8% | 5-10% |
| 多语言网页 | 4% | 5-10% |

如果模型应该在编程方面很强，增加代码比例。如果推理能力很重要，增加数学比例。如果需要减少噪声，降低网页比例。更改比例后始终在基准测试上评估。

## 规模估算

对于给定的目标 token 数量：

- 1T token 来自网页：预期约 3-5TB 原始文本，清洗和去重后约 1.5-2TB
- 分词速度 (Rust)：每核约 100M token/秒
- 分词速度 (Python)：每核约 1-10M token/秒
- MinHash 去重，128 个哈希，16 个 band：每核约 10K 文档/秒
- 序列打包：I/O 瓶颈，对于 10GB 以上的语料库使用内存映射文件

对于 15T token（Llama 3 规模），计划约 30-50TB 原始输入数据，在 64 核机器上进行 1-2 周的预处理，以及 100TB+ 的磁盘空间用于中间文件。

## 训练前检查清单

1. 总 token 数量与你的计算预算匹配（使用 Chinchilla 缩放定律或 Llama 3 的过训练比例作为指导）
2. 去重移除了 30-40% 的网页数据
3. 质量过滤器移除了剩余数据的 10-20%
4. 英文的压缩率为 3-5 字符/token
5. 序列利用率高于 95%
6. 随机抽查显示每个流水线阶段都有干净、连贯的文本
7. 数据混合比例已在小规模训练运行中验证
8. PII 移除已在样本上验证
9. 所有二进制格式（打包序列、token ID 数组）通过往返编码/解码测试
10. 流水线可复现：相同输入在固定随机种子下产生相同输出
