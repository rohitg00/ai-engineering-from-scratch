# DualPipe 并行（DualPipe Parallelism）

> DeepSeek-V3 在 2,048 块 H800 GPU 上训练，MoE 专家分散在节点间。跨节点专家全对全通信（all-to-all comm）每消耗 1 GPU 小时的计算，就需要 1 GPU 小时的通信。GPU 有一半时间处于空闲状态。DualPipe（DeepSeek，2024 年 12 月）是一种双向流水线（bidirectional pipeline），它将前向和反向计算与它们触发的全对全通信重叠。气泡（Bubble）减少，吞吐量提升，并且一旦专家并行（Expert Parallelism）已经将专家分散到各 rank 上，保留两份模型参数副本（即名称中的“dual”）的成本就很低。本课程是一次“学习”类型的演练，讲解 DualPipe 到底做了什么，以及为什么 Sea AI Lab 的 DualPipeV 改进版以略微更紧的气泡为代价，去掉了 2 倍参数成本。

**类型：** 学习
**语言：** Python（标准库，调度模拟器）
**先修知识：** 阶段 10 · 05（分布式训练，FSDP，DeepSpeed），阶段 10 · 14（开源模型架构与 MoE）
**时间：** 约 60 分钟

## 学习目标

- 列举 DualPipe 前向-反向块（forward-backward chunk）的四个组成部分，以及为什么每个部分都有自己的重叠窗口。
- 解释大规模下的流水线气泡问题，以及实际中与营销中“无气泡”的含义。
- 为 8 个流水线并行（PP）rank 和 16 个微批次（micro-batch）手动追踪一个 DualPipe 调度，并确认前向流和反向流互相填补对方的空闲时隙。
- 说明 DualPipeV（Sea AI Lab，2025）所做的权衡：当专家并行不激活时，以略微增大的气泡为代价，去掉 2 倍参数复制。

## 问题

在 2k 块 H800 GPU 上训练一个 671B MoE 模型会遇到三个叠加的瓶颈：

1. **内存压力。** 每块 GPU 持有模型的一部分。序列长度为 8k、61 层、128 头的激活内存是巨大的。
2. **流水线气泡。** 传统的流水线并行（GPipe，1F1B）在 GPU 等待其阶段（stage）的输入或梯度时使其空闲。在 8 个阶段的情况下，即使采用 1F1B 调度，大约 12% 的 GPU 时间可能是气泡。
3. **跨节点全对全。** 带专家并行的 MoE 将专家分散到各节点。每次前向传递会触发一个全对全来将 token 分发给它们的专家，再触发另一个来合并。在 2k 块 GPU 上，这很容易变成 1:1 的计算与通信比。

这些问题各有单独的解决方案：梯度检查点（gradient checkpointing）解决内存，零气泡（Zero Bubble）（Sea AI Lab，2023）解决流水线气泡，专家并行通信内核（expert-parallel comm kernels）解决全对全。DualPipe 所做的是让它们协同工作。该调度将计算和通信重叠在一个前向-反向块内部，同时从流水线的两端注入微批次，并利用生成的调度将全对全隐藏在计算窗口内。

报告结果：几乎消除了流水线气泡，在 DeepSeek-V3 的 14.8T token 训练运行中实现了超过 95% 的 GPU 利用率。

## 概念

### 流水线并行复习

将 N 层模型分割到 P 个设备上。设备 `i` 持有层 `i * N/P .. (i+1) * N/P - 1`。一个微批次依次前向流过设备 0 到 P-1，然后反向从 P-1 流回 0。每个设备只有在前一个设备发送其输出后才能开始其前向阶段，只有在下游设备发送上游梯度后才能开始反向。

GPipe（Huang 等人，2019）一次调度一个微批次，这会浪费大部分 GPU 时间。1F1B（Narayanan 等人，2021）将多个微批次的前向和反向传递交错进行。零气泡（Qi 等人，2023）将反向传递拆分为两部分——输入梯度反向（B）和权重梯度反向（W）——并调度它们来填充气泡。经过零气泡之后，流水线几乎是紧凑的。

DualPipe 是下一步。它在此基础上增加了两个想法：

### 想法 1：块的分解

每个前向块被拆分为四个组成部分：

- **注意力（Attention）。** Q/K/V 投影、注意力、输出投影。
- **全对全分发（All-to-all dispatch）。** 将 token 发送到其专家的跨节点通信。
- **MLP。** MoE 专家计算。
- **全对全合并（All-to-all combine）。** 将专家输出带回的跨节点通信。

反向块添加了每个部分的梯度版本。DualPipe 对它们进行调度，使得全对全分发与下一个块的注意力计算并行发生，全对全合并与后续块的 MLP 计算并行发生。

### 想法 2：双向调度

大多数流水线调度从阶段 0 注入微批次，流向阶段 P-1。DualPipe 从**两端**注入微批次。阶段 0 看到从那里出发的前向微批次；阶段 P-1 也看到从那里出发的前向微批次。两个流在中间相遇。

为了实现这一点，设备 `i` 必须同时持有流水线早期阶段 `i` 和流水线晚期阶段 `P - 1 - i`。这就是 DualPipe 的“dual”部分：每个设备保留两份它需要服务的模型层副本（每个方向一份）。在 DeepSeek-V3 的规模下，这是 2 倍参数复制成本。这是可以承受的，因为专家并行已经将 MoE 专家分散得如此之薄，以至于复制非专家层两次是小事一桩。

关键是，一个方向的前向流和另一个方向的反向流正好在单向调度中本会出现气泡的地方重叠。气泡消失了。

### 手动追踪的调度

考虑 P = 4 个 rank，8 个微批次，分为 4 个前向 / 4 个反向。时间从左到右移动；行是设备 rank。

```
           Time →
rank 0:  F1 F2 F3 F4  F5R F6R F7R F8R  B1 B2 B3 B4  ...
rank 1:     F1 F2 F3  F4/F5R F6R F7R   B1 B2 ...
rank 2:        F1 F2  F3/F5R F4/F6R    B1 ...
rank 3:           F1  F2/F5R F3/F6R    ...
```

解读“F4/F5R”符号：rank 1 在同一时间槽中运行微批次 4 的前向（从左到右经过流水线）以及微批次 5 的前向（从右到左）。这就是“双向”在操作上的含义。

在 rank 2 处，交叉流更早重叠，在 rank 0 和 P-1 处则最晚重叠。在调度的稳定中期阶段，每个 rank 运行X方向的前向与Y方向的反向重叠。计算一直忙碌。前向传递的全对全分发隐藏在反向计算内部。全对全合并隐藏在前向计算内部。气泡被挤压掉了。

### 气泡核算

标准 1F1B 流水线气泡（每个 rank 浪费的时间）：

```
bubble_1F1B = (P - 1) * forward_chunk_time
```

零气泡的改进将其降低，但未降为零。在稳定阶段，如果微批次数量是流水线深度的 2 倍的倍数，DualPipe 的气泡为零。在稳定阶段之外（预热和冷却），存在一些气泡，但它不会随微批次数量增长——这是论文强调的一个关键特性。

在营销术语中：“无气泡”。在技术术语中：气泡不随微批次数量增长。Sea AI Lab 的后续分析（DualPipeV / Cut-in-half）显示，只有在专家并行不是瓶颈时才能实现完全零气泡；在 EP 驱动的全对全情况下，总是存在一些调度折中。

### DualPipeV —— 改进版

Sea AI Lab（2025）观察到，当 EP 通信重叠不是重点时，2 倍参数复制是浪费的。他们的 DualPipeV 调度将双向注入折叠成一个“V 形”调度，在单个参数副本上运行。气泡比 DualPipe 稍大，但内存节省显著。DeepSeek 在其开源 DualPipe 实现中将 DualPipeV 作为 EP-off 模式采用。

权衡对比：

| 特性 | DualPipe | DualPipeV | 1F1B | 零气泡（Zero Bubble） |
|---------|---------|-----------|------|------------|
| 每个设备的参数副本数 | 2 | 1 | 1 | 1 |
| 气泡与微批次关系 | 常数 | 小幅增长 | 增长 | 增长 |
| 计算-通信重叠 | 完全 | 部分 | 最小 | 部分 |
| 适用场景 | EP 重负载 MoE | 密集模型或 EP 轻负载 | 基线 | 任意流水线 |

### 对 14.8T token 训练的意义

DeepSeek-V3 的预训练在 2,048 块 H800 GPU 上消耗了约 2.8M GPU 小时，处理了 14.8T token。如果采用朴素的 1F1B，他们将损失 12-15% 的流水线气泡时间——约 340-420K GPU 小时，足以训练一个完整的 70B 模型。DualPipe 恢复了其中的大部分。如果没有内部日志，很难直接量化贡献，但论文声称整个训练的平均 GPU 利用率超过 95%。

对于较小的训练（低于 1k GPU），DualPipe 是杀鸡用牛刀——流水线气泡相对于总成本较小，并且密集模型训练很少遇到全对全瓶颈。对于数千 GPU 规模的前沿 MoE 训练，这实际上是必需的。

### 在技术栈中的位置

- 与 **FSDP**（阶段 10 · 05）互补。FSDP 将模型参数分片到各 rank 上；DualPipe 调度跨 rank 的计算。它们可以结合使用。
- 与 **ZeRO-3** 梯度分片兼容。双副本复制所需的记账工作需要与 ZeRO 的分片梯度协作。
- 需要为特定集群拓扑调整的**自定义全对全内核**。DeepSeek 的开源内核是参考实现。

## 使用

`code/main.py` 是一个流水线调度模拟器。它接受 `(P, n_micro_batches, schedule)` 参数，并打印 1F1B、零气泡、DualPipe 和 DualPipeV 的稳定阶段利用率。这是一个教学工具——数字与论文中的定性主张一致，并不代表生产环境的实测加速。

模拟器的价值：使用不同的 P 和微批次数量运行它，观察 1F1B 的气泡占比如何增长，而 DualPipe 则不会。

实际训练运行中的集成考虑：

- 选择一个能均匀分割微批次数量的流水线并行深度。
- 确保你的专家并行网格支持双向全对全。DeepSeek 的内核是参考。
- 第一次预计要花费一周的调试时间来研究调度本身。记账工作很繁琐。
- 监控每个 rank 的 GPU 利用率，而不仅仅是汇总数据。DualPipe 的好处来自于收紧慢速节点。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------------|------------------------|
| 流水线气泡（Pipeline bubble） | “每个 rank 的空闲时间” | 因为流水线阶段在等待其输入或梯度而浪费的 GPU 周期 |
| 1F1B | “默认流水线调度” | 一次前向 / 一次反向交错调度；DualPipe 击败的基线 |
| 零气泡（Zero Bubble） | “Sea AI Lab 2023” | 将反向拆分为 B（输入梯度）和 W（权重梯度）；几乎完全收紧流水线 |
| DualPipe | “DeepSeek-V3 调度” | 双向流水线 + 计算-通信重叠；气泡不随微批次数量增长 |
| DualPipeV | “Cut-in-half” | V 形改进版，以略大气泡为代价去掉 2 倍参数复制 |
| 块（Chunk） | “流水线工作的单元” | 一个微批次经过一个流水线阶段的前向或反向传递 |
| 全对全分发（All-to-all dispatch） | “将 token 发送给专家” | 将 token 路由到其分配的 MoE 专家的跨节点通信 |
| 全对全合并（All-to-all combine） | “将专家输出带回” | MLP 后收集专家输出的跨节点通信 |
| 专家并行（Expert Parallelism，EP） | “专家分布在 GPU 上” | 将 MoE 专家分片到各 rank 上，使得不同 GPU 持有不同专家 |
| 流水线并行（Pipeline Parallelism，PP） | “层分布在 GPU 上” | 将模型层分片到各 rank 上；DualPipe 调度的维度 |
| 气泡占比（Bubble fraction） | “浪费的 GPU 时间” | （气泡时间 / 总时间）；DualPipe 趋向于零的占比 |
