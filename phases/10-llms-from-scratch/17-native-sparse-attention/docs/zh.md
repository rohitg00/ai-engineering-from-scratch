# Native Sparse Attention（原生稀疏注意力，DeepSeek NSA）

> 在64k tokens下，注意力机制占据了解码延迟的70-80%。每个开放模型实验室都有计划解决这个问题。DeepSeek的NSA（ACL 2025最佳论文）是其中脱颖而出的一种：三个并行的注意力分支——压缩的粗粒度tokens、选择性保留的细粒度tokens以及用于局部上下文的滑动窗口——通过一个学习得到的门控（gate）进行组合。它是对齐硬件的（内核友好）、原生可训练的（适用于预训练，而非推理时强加），并且在64k解码上比FlashAttention运行得更快，同时匹配或超越全注意力的质量。本课程将端到端构建这三个分支，并展示为什么这种稀疏性是端到端可微的。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** 阶段7·12（KV cache，flash-attention），阶段7·15（注意力变体），阶段10·16（微分注意力）
**时间：** ~60分钟

## 学习目标

- 陈述NSA的三个注意力分支以及每个分支捕获的内容。
- 解释为什么NSA是“原生可训练的”，而之前的稀疏注意力方法仅用于推理。
- 以压缩块大小和选择top-k为函数，计算NSA与全注意力在64k上下文下的注意力计算节省量。
- 在短合成序列上使用标准库Python实现三分支组合，并验证门控权重的行为。

## 问题

在序列长度N下，全注意力每层需要`O(N^2)`时间和`O(N)` KV cache。在64k tokens下，计算和内存带宽的数字是灾难性的。来自NSA论文的测量理论估计：注意力在64k下占总解码延迟的70-80%。所有下游指标——TTFT、tokens/秒、每百万tokens成本——都由注意力成本主导。

稀疏注意力是明显的答案。先前的尝试分为两类。固定模式稀疏性（滑动窗口、跨步、块局部）会丢弃信息，并且在长距离召回任务上失败。推理时稀疏性（KV cache剪枝、H2O、StreamingLLM）应用于在密集注意力上预训练的模型，只能恢复潜在加速的一小部分，因为模型从未被要求通过稀疏模式路由信息。

原生稀疏注意力（Yuan等人，DeepSeek + 北大 + 华盛顿大学，ACL 2025最佳论文，arXiv:2502.11089）同时做到了这两点：一种在预训练期间学习的稀疏模式，实现为一种对齐内核的算法，在推理时实际提供计算节省。两年后，NSA或其直接后代将成为每个前沿长上下文模型的默认注意力机制。

## 概念

### 三个并行分支

对于每个查询，NSA对KV cache的三种不同视图运行三次注意力：

1. **压缩分支（Compressed branch）。** Tokens被分组为大小为`l`（通常为32或64）的块。每个块通过一个小型学习得到的MLP压缩成一个摘要token。查询在这些压缩tokens上执行注意力，获得整个序列的粗粒度视图。

2. **选择分支（Selected branch）。** 利用压缩分支的注意力分数，识别出与当前查询最相关的top-k个块。读取这些块中的细粒度（未压缩）tokens，查询对所有它们执行注意力。将压缩分支的注意力视为选择的路由信号。

3. **滑动窗口分支（Sliding-window branch）。** 查询对最近的`W`个tokens（通常为512）执行注意力，以获取局部上下文。该分支捕获结构密集的短程模式（句法、局部共指），其他两个分支可能遗漏这些。

三个分支的输出通过一个学习得到的逐位置门控（per-position gate）进行组合：

```
out = g_cmp * out_cmp + g_sel * out_sel + g_win * out_win
```

`g_cmp, g_sel, g_win` 是来自查询的小型MLP的门控权重。它们不必和为1——它们可以独立地对分支进行加权。

### 为什么这是“原生可训练的”

选择步骤（top-k块）是离散的。离散操作会破坏梯度流。先前的稀疏注意力工作要么跳过了通过选择的反向传播（限制训练），要么使用了在推理时不提供真正稀疏性的连续松弛方法。

NSA绕过了这一点：压缩分支的注意力是整个序列上一个可微的粗粒度注意力。top-k操作只是重用来自压缩分支的最高注意力分数，以选择加载哪些细粒度块。梯度通过压缩分支的分数（它影响压缩输出和选择逻辑）流动，并且所选块对最终输出的贡献也是可微的。不可微的`top_k`操作在前向计算图中是一个无操作——它只控制从内存中加载哪些块。

这就是为什么NSA可以在端到端预训练中使用。模型学会了通过三个分支联合路由信息，产生一个在推理时实际提供承诺加速的稀疏模式。

### 对齐硬件的内核

NSA的内核是为现代GPU内存层次结构设计的。内核通过GQA组（外循环）加载查询，为每组获取相应的稀疏KV块（内循环），并在SRAM上运行注意力。由于每个查询组看到相同的选定块（选择是按查询组进行的，而不是按查询头），KV加载在整个组中分摊。算术强度保持较高。

论文报告称，Triton内核在64k解码上比FlashAttention快9倍，加速比随序列长度增长。前向和反向内核都提供。

### 计算预算

设`N`为序列长度，`l`为压缩块大小，`k`为top-k选择数量，`w`为滑动窗口大小，`b`为选定块大小（通常等于`l`）。

- 压缩分支：每个查询`O(N/l)`个键，总计`O(N * N / l)`。
- 选择分支：每个查询`O(k * b)`个键，总计`O(N * k * b)`。
- 滑动分支：每个查询`O(w)`个键，总计`O(N * w)`。

总计：`O(N * (N/l + k*b + w))`。

当`N = 64k, l = 64, k = 16, b = 64, w = 512`时：每个查询的成本为`1000 + 1024 + 512 = 2536`个键。全注意力为`64000`个键。计算减少25倍。

当`N = 128k, l = 64, k = 16, b = 64, w = 512`时：每个查询的成本为`2000 + 1024 + 512 = 3536`个键。全注意力为`128000`个键。减少36倍。收益随序列长度增长，这正是关键所在。

### 对比如何

| 方法 | 可微 | 实际推理加速 | 长程召回 |
|--------|---------------|----------------------|-------------------|
| 仅滑动窗口 | 是 | 是 | 失败 |
| 跨步/块稀疏 | 是 | 是 | 部分 |
| KV剪枝（H2O, StreamingLLM） | 不适用（推理时） | 是 | 部分 |
| MoBA（Moonshot） | 部分 | 是 | 良好 |
| NSA | 是（原生） | 是（64k下9倍） | 匹配全注意力 |

MoBA（Moonshot, arXiv:2502.13189）是同时发表的，采取了类似的“三个优于一个”的方法，将MoE原则应用于注意力块。NSA和MoBA是2026年长上下文预训练中需要了解的两种架构。

## 构建它

`code/main.py` 在一个短合成序列上实现了三个分支，并展示了：

- 压缩MLP（为教学清晰起见，使用简单的均值池化基线；真正的NSA使用学习得到的MLP）。
- 由压缩分支分数驱动的top-k块选择。
- 对最后`w`个tokens的滑动窗口注意力。
- 门控组合。
- 与全注意力比较的计算计数打印。

### 步骤1：将tokens压缩成块

```python
def compress(K, l):
    n = len(K)
    n_blocks = (n + l - 1) // l
    out = []
    for b in range(n_blocks):
        start, end = b * l, min((b + 1) * l, n)
        block = K[start:end]
        # 对块中每个维度取均值作为摘要
        summary = [sum(row[d] for row in block) / len(block) for d in range(len(K[0]))]
        out.append(summary)
    return out
```

### 步骤2：压缩分支注意力

对查询与压缩后的键执行softmax注意力。压缩分支的分数同时作为top-k选择的信号。

### 步骤3：top-k块选择

选出分数最高的`k`个压缩块的索引。加载这些块的原始未压缩tokens，并对其执行注意力。

### 步骤4：滑动窗口注意力

取最后`w`个tokens，对其执行标准注意力。

### 步骤5：门控+组合

一个基于查询的小型MLP产生三个门控权重。最终输出是三个分支输出的加权和。

### 步骤6：计算计数

打印每个查询在每个分支中关注的键的数量以及总数。与`N`（全注意力）比较。在1024 token的合成数据上，使用`l = 32, k = 4, w = 128`，NSA每个查询关注`32 + 128 + 128 = 288`个键，而全注意力为1024——减少了3.5倍。

## 使用它

NSA正在DeepSeek自己的长上下文预训练流程中部署。截至2026年4月，在公共推理栈中的集成状态：

- **DeepSeek内部**：原生支持，已发布的权重使用NSA或其后续DSA（Deepseek Sparse Attention）。
- **vLLM**：正在开发中对DeepSeek-V3.x权重的实验性NSA支持。
- **SGLang**：已发布NSA基准测试；生产路径跟随vLLM。
- **llama.cpp / CPU**：不支持；内核分解的开销在CPU吞吐量下不值得。

何时使用NSA：

- 针对64k及以上的长上下文且预算充足的预训练或继续训练运行。
- 推理DeepSeek自己的长上下文检查点。权重是NSA原生的。

何时不使用：

- 服务于已有的密集注意力预训练模型。如果没有继续训练，无法改造为NSA。
- 上下文低于16k。三分支的开销超过了节省。
- 批次为1的交互式聊天。对延迟敏感的解码有益，但仅在长上下文时。

## 关键术语

| 术语 | 人们所说的 | 实际含义 |
|------|----------------|------------------------|
| 压缩分支 | “粗视图” | 在块平均后的键上执行注意力，每个查询提供O(N/l)个键的全局上下文 |
| 选择分支 | “Top-k块” | 对压缩分支分数最高的`k`个块执行细粒度注意力 |
| 滑动窗口 | “局部上下文” | 对最后`W`个tokens执行注意力，用于短程模式 |
| 原生可训练性 | “预训练时就开启稀疏性” | 稀疏模式在预训练期间学习得到，而非推理时强加 |
| 压缩块大小l | “粗视图的组大小” | 多少tokens合并成一个摘要；通常32-64 |
| Top-k | “保留的块数” | 读取多少个压缩块的未压缩tokens；通常16 |
| 滑动窗口W | “局部注意力半径” | 通常512；更短则损害局部连贯性，更长则浪费计算 |
| 分支门控 | “如何混合三种结果” | 逐位置MLP输出，对三个分支的贡献加权 |
| 硬件对齐 | “内核友好的稀疏性” | 选择的稀疏模式使得实际GPU内核能达到理论加速 |
| DSA | “NSA的后续” | Deepseek Sparse Attention，DeepSeek家族中紧接NSA的架构 |
