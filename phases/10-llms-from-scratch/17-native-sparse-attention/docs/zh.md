# 17 · 原生稀疏注意力（DeepSeek NSA）

> 在 64k token 长度下，注意力会吃掉 70-80% 的解码延迟。每个开放模型实验室都有自己的解决方案。DeepSeek 的 NSA（ACL 2025 最佳论文）是真正立住脚的那个：三条并行的注意力分支——压缩后的粗粒度 token、被选择性保留的细粒度 token、以及用于捕捉局部上下文的滑动窗口——通过一个可学习的门控（gate）组合起来。它是「硬件对齐（hardware-aligned）」的（对 kernel 友好）、「原生可训练（natively trainable）」的（在预训练阶段就生效，而非推理时硬加上去），并且在 64k 解码上比 FlashAttention 更快，同时质量上追平甚至超过全注意力（full attention）。本课会端到端地构建这三条分支，并说明为什么这种稀疏性是端到端可微的。

**类型：** 构建
**语言：** Python（标准库）
**前置：** 阶段 7 · 12（KV 缓存、flash-attention）、阶段 7 · 15（注意力变体）、阶段 10 · 16（差分注意力）
**时长：** 约 60 分钟

## 学习目标

- 说出 NSA 的三条注意力分支，以及每条分支分别捕捉什么。
- 解释为什么 NSA 是「原生可训练」的，而此前的稀疏注意力方法只能在推理时使用。
- 在 64k 上下文下，以压缩块大小和选择 top-k 为变量，计算 NSA 相对于全注意力的注意力计算量节省。
- 在一段简短的合成序列上，用标准库 Python 实现三分支组合，并验证门控权重的行为是否符合预期。

## 问题所在

序列长度为 N 的全注意力，每层耗费 `O(N^2)` 的时间和 `O(N)` 的 KV 缓存。在 64k token 下，计算量和内存带宽的数字是灾难性的。NSA 论文给出的理论估计为：在 64k 下，注意力占据了总解码延迟的 70-80%。所有下游指标——TTFT、tokens/sec、每百万 token 成本——都被注意力开销所主导。

稀疏注意力是显而易见的答案。此前的尝试分为两类。固定模式稀疏（sliding-window、strided、block-local）会丢弃信息，在长程召回（long-range recall）任务上失效。推理时稀疏（KV 缓存剪枝、H2O、StreamingLLM）是施加在一个基于稠密注意力预训练好的模型之上的，只能恢复潜在加速的一小部分，因为该模型从未被要求让信息按稀疏模式路由。

原生稀疏注意力（Native Sparse Attention，Yuan 等人，DeepSeek + PKU + UW，ACL 2025 最佳论文，arXiv:2502.11089）两者兼得：一种模型在预训练期间学到的稀疏模式，实现为一个 kernel 对齐的算法，在推理时真正兑现计算量节省。两年之内，NSA 或其直系后继者会成为每个前沿长上下文模型的默认注意力。

## 核心概念

### 三条并行分支

对每个查询（query），NSA 会针对 KV 缓存的三种不同视图运行三次注意力：

1. **压缩分支（Compressed branch）。** token 被分组为大小为 `l`（通常是 32 或 64）的块。每个块通过一个小型可学习 MLP 压缩成单个摘要 token。查询在这些压缩 token 上做注意力，获得整个序列的粗粒度视图。

2. **选择分支（Selected branch）。** 利用压缩分支得到的注意力分数，找出与当前查询最相关的 top-k 个块。从这些块中读取细粒度（未压缩）的 token，查询在它们全部上做注意力。可以把压缩分支的注意力理解为选择的路由信号。

3. **滑动窗口分支（Sliding-window branch）。** 查询对最近的 `W` 个（通常 512）token 做注意力，以获取局部上下文。这条分支捕捉那些另外两条可能错过的、结构性强的短程模式（句法、局部指代）。

三条分支的输出通过一个可学习的逐位置门控组合：

```
out = g_cmp * out_cmp + g_sel * out_sel + g_win * out_win
```

`g_cmp, g_sel, g_win` 是来自查询上一个小型 MLP 的门控权重。它们不必加起来等于 1——可以独立地为各分支加权。

### 为什么说它是「原生可训练」的

选择步骤（top-k 块）是离散的。离散操作会切断梯度流动。此前的稀疏注意力工作要么跳过了对选择步骤的反向传播（限制了训练），要么使用了连续松弛（continuous relaxation），但那在推理时无法带来真正的稀疏性。

NSA 巧妙地绕开了这个问题：压缩分支的注意力本身就是对整个序列的可微的粗粒度注意力。top-k 操作只是复用压缩分支的最高注意力分数，来决定要加载哪些细粒度块。梯度流经压缩分支的分数（这些分数同时影响压缩输出和选择逻辑），而被选中块对最终输出的贡献也是可微的。不可微的 `top_k` 操作在前向计算图上是一个空操作（no-op）——它只控制从内存中加载哪些块。

这就是 NSA 能够端到端用于预训练的原因。模型学会联合地让信息在三条分支间路由，产生一种稀疏模式，并在推理时真正兑现所承诺的加速。

### 硬件对齐的 kernel

NSA 的 kernel 是为现代 GPU 内存层级设计的。kernel 按 GQA 分组加载查询（外层循环），按组取出对应的稀疏 KV 块（内层循环），并在 SRAM 上运行注意力。由于每个查询组看到的是同一组被选中的块（选择是按查询组、而非按查询头进行的），KV 加载在组内被摊销。算术强度（arithmetic intensity）保持很高。

论文报告称，在 64k 解码上，Triton kernel 比 FlashAttention 快 9 倍，且加速比随序列长度增长。前向和反向 kernel 都已提供。

### 计算预算

设 `N` 为序列长度，`l` 为压缩块大小，`k` 为 top-k 选择数量，`w` 为滑动窗口，`b` 为被选中块的大小（通常等于 `l`）。

- 压缩分支：每个查询 `O(N/l)` 个键，总计 `O(N * N / l)`。
- 选择分支：每个查询 `O(k * b)` 个键，总计 `O(N * k * b)`。
- 滑动分支：每个查询 `O(w)` 个键，总计 `O(N * w)`。

总计：`O(N * (N/l + k*b + w))`。

取 `N = 64k, l = 64, k = 16, b = 64, w = 512`：每查询成本为 `1000 + 1024 + 512 = 2536 keys`。全注意力为 `64000 keys`。计算量减少 25 倍。

取 `N = 128k, l = 64, k = 16, b = 64, w = 512`：每查询成本为 `2000 + 1024 + 512 = 3536 keys`。全注意力为 `128000 keys`。减少 36 倍。收益随序列长度增长——而这正是整件事的意义所在。

### 它与其他方法相比如何

| 方法 | 可微 | 真实推理加速 | 长程召回 |
|--------|---------------|----------------------|-------------------|
| 仅滑动窗口 | 是 | 是 | 失效 |
| Strided / block-sparse | 是 | 是 | 部分 |
| KV 剪枝（H2O、StreamingLLM） | 不适用（推理时） | 是 | 部分 |
| MoBA（Moonshot） | 部分 | 是 | 良好 |
| NSA | 是（原生） | 是（64k 下 9 倍） | 追平全注意力 |

MoBA（Moonshot，arXiv:2502.13189）是同期发表的工作，采用了类似的「三个胜过一个」思路，把 MoE 原则应用到注意力块上。NSA 和 MoBA 是 2026 年长上下文预训练必须了解的两种架构。

## 动手构建

`code/main.py` 在一段简短的合成序列上实现了这三条分支，并展示：

- 压缩 MLP（为了教学清晰，这里用一个简单的均值池化（mean-pool）基线；真正的 NSA 使用可学习的 MLP）。
- 由压缩分支分数驱动的 top-k 块选择。
- 对最后 `w` 个 token 的滑动窗口注意力。
- 门控组合。
- 一份与全注意力对比的计算量统计打印。

### 第 1 步：把 token 压缩成块

```python
def compress(K, l):
    n = len(K)
    n_blocks = (n + l - 1) // l
    out = []
    for b in range(n_blocks):
        start, end = b * l, min((b + 1) * l, n)
        block = K[start:end]
        summary = [sum(row[d] for row in block) / len(block) for d in range(len(K[0]))]
        out.append(summary)
    return out
```

### 第 2 步：压缩分支注意力

对查询与压缩后的键运行 softmax 注意力。压缩分支的分数同时充当 top-k 选择的信号。

### 第 3 步：top-k 块选择

选出分数最高的 `k` 个压缩块的索引。从这些块中加载原始未压缩的 token 并对它们运行注意力。

### 第 4 步：滑动窗口注意力

取最后 `w` 个 token，对它们运行标准注意力。

### 第 5 步：门控 + 组合

查询上的一个小型 MLP 产生三个门控权重。最终输出是三条分支输出的加权和。

### 第 6 步：计算量统计

打印每个查询在每条分支上所关注的键数量以及总数。与 `N`（全注意力）作对比。在一段 1024-token 的合成序列上、取 `l = 32, k = 4, w = 128`，NSA 每查询看到 `32 + 128 + 128 = 288` 个键，而全注意力为 1024 个——少 3.5 倍。

## 实际应用

NSA 已经落地于 DeepSeek 自己的长上下文预训练管线。截至 2026 年 4 月，在公开推理栈中的集成状态：

- **DeepSeek 内部**：原生支持，已发布的权重使用 NSA 或其后继者 DSA（Deepseek Sparse Attention）。
- **vLLM**：针对 DeepSeek-V3.x 权重的实验性 NSA 支持正在开发中。
- **SGLang**：已发布 NSA 基准测试；生产路径跟随 vLLM。
- **llama.cpp / CPU**：不支持；在 CPU 吞吐下，kernel 分解的开销不值得。

何时该选用 NSA：

- 面向 64k 及以上上下文、且具备可观计算预算的预训练或持续训练（continued-training）任务。
- 推理 DeepSeek 自己的长上下文检查点。这些权重是 NSA 原生的。

何时不该选用：

- 服务一个已有的稠密注意力预训练模型。不经过持续训练，你无法把 NSA 改装上去。
- 上下文低于 16k。三分支的开销会盖过其节省。
- batch-1 的交互式聊天。延迟敏感的解码能受益，但只在长上下文下才成立。

## 关键术语

| 术语 | 人们怎么说 | 它实际是什么 |
|------|----------------|------------------------|
| 压缩分支（Compressed branch） | 「粗粒度视图」 | 对块平均后的键做注意力，以每查询 O(N/l) 个键提供全局上下文 |
| 选择分支（Selected branch） | 「top-k 块」 | 对压缩分支分数最高的 `k` 个块做细粒度注意力 |
| 滑动窗口（Sliding window） | 「局部上下文」 | 对最后 `W` 个 token 做注意力以捕捉短程模式 |
| 原生可训练性（Native trainability） | 「带着稀疏性做预训练」 | 稀疏模式在预训练期间学到，而非推理时硬加 |
| 压缩块大小 l | 「粗粒度视图的分组大小」 | 多少个 token 被合并成一个摘要；通常 32-64 |
| Top-k | 「要保留的块」 | 哪些压缩块的未压缩 token 会被读取的数量；通常 16 |
| 滑动窗口 W | 「局部注意力半径」 | 通常 512；过短伤害局部连贯性，过长浪费计算 |
| 分支门控（Branch gate） | 「如何混合这三者」 | 逐位置的 MLP 输出，为三条分支的贡献加权 |
| 硬件对齐（Hardware alignment） | 「对 kernel 友好的稀疏性」 | 选择的稀疏模式使得实际 GPU kernel 能达到理论加速 |
| DSA | 「NSA 的后继者」 | Deepseek Sparse Attention，DeepSeek 谱系中接续 NSA 的架构 |
