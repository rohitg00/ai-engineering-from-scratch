# 15 · 投机解码与 EAGLE-3

> 第 7 阶段 · 第 16 课已经证明了数学原理：Leviathan 拒绝规则能精确地保持验证器（verifier）的分布不变。本课从训练栈的视角审视 2026 年生产环境中的投机解码（speculative decoding）。EAGLE-3 把草稿模型（draft model）从一个廉价的近似，变成了专门设计、在验证器自身隐藏状态上训练出来的微型网络，再加上一个训练期测试循环，把它的训练分布与推理分布对齐。结果是：端到端加速 3 倍到 6.5 倍，对话场景下逐 token 接受率超过 0.9，且没有任何分布上的取舍。2026 年每一个生产推理栈都默认搭载它。

## 学习目标

- 用一句话陈述 Leviathan 定理，并证明投机循环产出的样本与验证器同分布。
- 走完两年的演进路线，从原始投机解码（Leviathan 2023）到 EAGLE、EAGLE-2、EAGLE-3，并指出每一步精确去除了哪一个限制。
- 从接受率 `α` 和草稿/验证器成本比 `c` 计算期望加速比，并为每种工况选出最优草稿长度 `N`。
- 从零实现完整的投机循环：草稿、验证、从残差分布拒绝采样、拒绝时回滚 KV 缓存、全部接受时发出额外 token（bonus token）。

## 问题所在

在 70B 模型上的自回归解码，在一块 H100 上大概只能跑到每秒 35 个 token。GPU 远未饱和。瓶颈是内存带宽：每个 token 都要从 HBM 加载 70B 的权重，做一步运算，产出一个浮点数。计算单元大部分时间在空转。

投机解码把这个问题转化成一个你真正能解决的吞吐量问题。一个廉价的草稿用 `N` 次小型前向传播提出 `N` 个 token。验证器只在前缀加上全部 `N` 个草稿上跑一次。如果验证器在位置 `i` 处的分布与草稿一致（在我们将要精确定义的统计意义上），就接受；如果不一致，就拒绝，并从残差分布（residual distribution）中采样一个修正值。一次大模型前向最多能产出 `N+1` 个被接受的 token，而不只是一个。

真正关键的定理是 Leviathan、Kalman、Matias（ICML 2023）：输出分布与直接从验证器采样所得到的分布完全相同。不是近似相同，是完全相同。这正是投机解码能在生产环境被接受的全部原因——它是纯粹的延迟优化，没有任何质量上的取舍。

第 7 阶段 · 第 16 课给你的是数学。本课给你的是训练栈。一个好的草稿能带来比廉价草稿多 2 倍的加速。EAGLE、EAGLE-2 和 EAGLE-3（Li 等人，2024–2025）把「草稿 = 同一模型的缩小版」变成了一门精确的工程学科。2026 年的生产推理服务器默认采用 EAGLE-3。

## 核心概念

### 不变量：Leviathan 拒绝采样

设 `p(t)` 是草稿在给定某个前缀下对下一 token 的分布，`q(t)` 是验证器的分布。采样一个草稿 token `d ~ p`。以概率 `min(1, q(d) / p(d))` 接受。拒绝时，从残差分布 `(q - p)_+ / ||(q - p)_+||_1` 中采样。所得样本服从 `q` 分布。无论 `p` 有多糟糕，这都成立——`p` 越差，你拒绝得越频繁，但输出始终精确。

用一次在 `prefix + d_1 + ... + d_N` 上的验证器前向，把 `N` 次这样的调用首尾相接地堆叠起来。验证器同时返回 `q_1, q_2, ..., q_{N+1}`。从左到右遍历。在位置 `j` 处发生的第一次拒绝处，从 `residual(q_j, p_j)` 采样并停止。如果全部接受，则从 `q_{N+1}` 中采样一个额外 token。

### 什么决定加速比

设 `α` 为每个草稿 token 的期望接受率。设 `c = cost(draft) / cost(verifier)` 为成本比。每次验证器前向期望接受的 token 数为：

```
E[accepted] = (1 - α^(N+1)) / (1 - α)
```

每个被接受 token 的期望总墙钟时间为 `(N * c + 1) / E[accepted]`。对 `N` 求最小值就得到了甜点区。对 `α = 0.8, c = 0.05`：最优 `N` 在 5–7 左右，加速比为 3.2 倍。对 `α = 0.95, c = 0.02`：最优 `N` 在 8–10 左右，加速比逼近 5 倍。

最大的单一杠杆是 `α`。在固定 `N = 5` 时，从 `α = 0.6`（原始草稿）提升到 `α = 0.9`（EAGLE-3），会把每次验证器前向的期望接受 token 数从 2.2 提升到 4.1。同一个验证器带来近 2 倍的吞吐量。

### 两年的演进

**原始投机解码（Leviathan，2023）。** 草稿模型是来自同一家族、独立训练出来的较小 LLM。容易搭建，`α ≈ 0.6`，加速比最多 2 倍左右。

**EAGLE-1（Li 等人，2024）。** 草稿是一个微型 transformer——通常一两层——它以验证器最后一层的隐藏状态（hidden state）作为输入，直接预测下一 token。因为草稿看到了验证器的特征表示，它的分布与验证器的分布接近得多。`α` 攀升到 0.7–0.8。

**EAGLE-2（Li 等人，2024）。** 增加了动态草稿树（dynamic draft tree）：不再提出单一的 `N` 个 token 序列，而是提出一棵候选小树，用验证器在一次前向中（树注意力，tree attention）为每个候选打分，并沿着最高概率路径行进。草稿长度变成逐步自适应。每个被接受路径 token 的 `α` 攀升到 0.85 以上。

**EAGLE-3（Li 等人，2025，NeurIPS）。** 又有两处改动。第一，彻底丢弃特征预测损失——EAGLE-1/2 训练草稿去匹配验证器的隐藏状态，这限制了数据量能带来多大帮助。EAGLE-3 直接在 token 预测上训练。第二，训练期测试（training-time test, TTT）：在草稿训练期间，把草稿自己此前的预测作为输入反馈回来，跨多步进行，方式与它在推理时的运作完全一致。这把训练分布与测试分布对齐，并阻止误差累积。实测加速比：对话场景最高 6.5 倍，在 H100 上的 SGLang 中以批量 64 运行时吞吐量提升 38%。

### KV 缓存回滚

验证会在一次前向中把验证器的 KV 缓存扩展 `N` 个条目。如果在位置 `j` 处发生拒绝，那么超过位置 `j-1` 的缓存内容现在就错了。两种常见实现：写入暂存缓冲区，接受时再提交（vLLM、TensorRT-LLM）；或保留一个物理 KV 缓存外加一个逻辑长度，拒绝时截断。无论哪种方式，回滚的开销都是每层每头若干字节，相对于前向传播的开销可以忽略不计。

对于 EAGLE-2 的树搜索，验证器以一个遵循树拓扑的非因果掩码（non-causal mask）运行注意力。工程上很琐碎，但计算本身只是一次带自定义掩码的标准 flash-attention 调用。

### 2026 年的草稿架构

| 策略 | 草稿类型 | `α` | 加速比 | 训练成本 |
|----------|-----------|-----|---------|---------------|
| Vanilla | 独立的小 LLM | 0.55-0.70 | 1.8-2.3× | 无（复用已有小模型） |
| Medusa | 验证器上额外的 LM 头 | 0.65-0.75 | 2-3× | 约 1B SFT token |
| EAGLE-1 | 在隐藏状态上的 1 层 transformer | 0.70-0.80 | 2.5-3× | 约 60B token |
| EAGLE-2 | EAGLE-1 + 动态草稿树 | 0.80-0.88 | 3-4× | 约 60B token |
| EAGLE-3 | 多层特征融合 + TTT | 0.88-0.92 | 3.5-6.5× | 约 60-200B token |
| Lookahead | 无草稿（Jacobi 迭代） | 不适用 | 1.3-1.6× | 无 |

2026 年的生产环境：vLLM 和 SGLang 在可用时默认采用 EAGLE-3，否则采用 EAGLE-2。TensorRT-LLM 对 Meta 和 NVIDIA 的公开模型有最快的 Medusa 路径。llama.cpp 为 CPU 部署提供原始草稿方案。

## 动手构建

参见 `code/main.py`。这是完整的 Leviathan 投机循环，包含全部组件：N 个草稿、验证器并行前向、逐位置拒绝、残差采样、额外 token、KV 回滚，以及对输出分布与直接从 `q` 采样是否匹配的实证验证。

### 第 1 步：拒绝规则

```python
def accept(q_prob, p_prob, u):
    if p_prob <= 0:
        return True
    return u < min(1.0, q_prob / p_prob)
```

### 第 2 步：残差分布

```python
def residual(q, p):
    raw = [max(0.0, qi - pi) for qi, pi in zip(q, p)]
    s = sum(raw)
    if s == 0:
        return list(q)
    return [r / s for r in raw]
```

### 第 3 步：一个完整的投机步

`spec_step` 函数从 `p` 草拟 `N` 个 token，然后在一次并行的 `q` 求值中验证它们全部。对每个草稿 token 应用拒绝规则，并在第一次拒绝处从残差中采样修正值。如果全部接受，则从 `q_{N+1}` 发出一个额外 token。

### 第 4 步：KV 回滚记账

模拟器为每个 worker 跟踪一个逻辑 `kv_length`。接受 `k` 个草稿时，`kv_length += k`。在位置 `j` 处发生拒绝时，缓存已经写到了 `j` 之后，但逻辑长度被设为 `prefix_length + j + 1`——即修正 token 之后的一位。后续读取会截断到逻辑长度。

### 第 5 步：Leviathan 校验

运行 50,000 个投机步。统计被接受 token 的经验分布。与从 `q` 直接采样的 50,000 个样本比较。卡方统计量应当远低于临界值。该定理在实践中通过。

### 第 6 步：加速比 vs. α

通过以不同幅度把 `p` 扰动偏离 `q` 来扫描草稿质量。测量 `α`，然后把每次验证器调用的期望 token 数作为 `α` 和 `N` 的函数画出来。代码会打印一张表，展示 EAGLE-3 级别的草稿质量（`α ≈ 0.9`）如何解锁每次验证器调用 4–5 个 token。

## 实战运用

带 EAGLE-3 的生产级 `vllm serve`：

```bash
vllm serve meta-llama/Llama-3.3-70B-Instruct \
  --speculative-config '{
    "model": "yuhuili/EAGLE3-LLaMA3.3-Instruct-70B",
    "num_speculative_tokens": 5,
    "method": "eagle3"
  }'
```

在 H100 上以批量 64 运行带 EAGLE-3 的 SGLang：根据 EAGLE-3 论文，吞吐量大约比批量 64 的原始解码高 1.38 倍。

何时应该动用投机解码：

- 任何 p50 延迟比峰值吞吐量更重要的交互式对话工作负载。
- 代码生成与结构化输出（JSON、SQL）。因为目标分布高度可预测，`α` 在 0.9 以上。
- 长文生成（数千 token）。摊销后的加速持续兑现收益。

何时不应该用：

- 非常小的模型（< 3B）。草稿并不比验证器便宜多少。
- 微型的批量为 1 的 CPU 部署。草稿模型的内存开销可能不值得。
- `α` 会崩溃的超高温度创意采样。

## 关键术语

| 术语 | 人们怎么说 | 它实际的含义 |
|------|----------------|------------------------|
| Leviathan 规则 | 「min(1, q 除以 p)」 | 以概率 `min(1, q(d)/p(d))` 进行的伯努利接受/拒绝；当拒绝时从残差采样，可精确保持验证器分布 |
| 残差分布 | 「(q 减 p) 取正后归一化」 | `(q - p)_+`，在零处截断并重新归一化——拒绝时正确的采样分布 |
| 接受率 α | 「草稿对的频率」 | 拒绝规则下每 token 的期望伯努利成功概率；主宰全部加速比数学 |
| EAGLE-1 | 「隐藏状态草稿」 | 以验证器最后一层隐藏状态为条件的微型 transformer 草稿（Li 等人，2024） |
| EAGLE-2 | 「动态草稿树」 | EAGLE-1 加上一棵候选续写树，在一次验证器前向中用树注意力打分 |
| EAGLE-3 | 「训练期测试」 | 丢弃特征预测损失，在训练期间把草稿自身输出喂给它，直接在 token 预测上训练 |
| 训练期测试（TTT） | 「暴露偏差修复」 | 在训练期间自回归地运行草稿，使训练与测试的输入分布匹配——计划采样的直接对应物 |
| KV 回滚 | 「撤销被拒绝的草稿」 | 拒绝后把验证器 KV 缓存重置到被接受前缀长度的记账 |
| 额外 token（bonus token） | 「免费的那个」 | 当 `N` 个草稿全部接受时，从 `q_{N+1}` 多采样一个，不增加额外的验证器成本 |
| 树注意力 | 「一次性验证多个候选」 | 带遵循草稿树拓扑的非因果掩码的注意力；在一次前向中为树中每个节点计算 `q_i` |
