# 推测性解码（Speculative Decoding）与EAGLE-3

> 第7阶段·第16课已用数学证明了：Leviathan拒绝规则精确保持了验证器（Verifier）的分布。本课从训练堆栈（training-stack）角度审视2026年的生产级推测性解码。EAGLE-3将草稿模型（Draft Model）从一个廉价近似转化为一个基于验证器自身隐藏状态（Hidden States）训练、专为小型网络而构建的模型，并增加了一个训练时测试循环（Training-Time Test Loop），使其训练分布与推理分布对齐。结果：端到端加速3倍至6.5倍，对话场景下每个token的接受率（Accepted Per-Token Rate）超过0.9，且无分布折中。2026年每个生产推理堆栈默认内置此技术。

**类型：** 构建

**先决条件：** 第7阶段·第16课（推测性解码数学），第10阶段·第12课（推理优化）

## 学习目标

- 用一句话陈述Leviathan定理，并证明推测循环产生的样本与验证器采样同分布。
- 梳理从普通推测性解码（Vanilla Spec-Decoding，Leviathan 2023）到EAGLE、EAGLE-2、EAGLE-3的两年度演进，并指出每一步消除的特定限制。
- 根据接受率 `α` 和草稿对验证器成本比 `c` 计算预期加速，并为每种场景选择最优草稿长度 `N`。
- 从头实现完整的推测循环：草稿、验证、从残差分布中拒绝采样、在拒绝时回滚KV缓存（KV Cache）、在完全接受时生成奖励令牌（Bonus Token）。

## 问题

在H100上，70B模型的自回归解码（Autoregressive Decoding）速度大约为每秒35个token。GPU远未达到饱和。内存带宽是天花板：每个token从HBM加载70B权重，执行一步算术，输出一个浮点数。计算单元大部分时间处于空闲状态。

推测性解码将这个问题转化为一个可实际解决的吞吐量问题。一个廉价的草稿通过 `N` 次小前向传播（Forward Passes）提出 `N` 个token。验证器对前缀加上所有 `N` 个草稿运行一次。如果验证器在位置 `i` 处的分布与草稿达成一致（在我们将精确化的统计意义上），则接受；否则拒绝，并从残差分布（Residual Distribution）中采样一个修正。一次大模型前向传播最多可输出 `N+1` 个接受token，而非一个。

关键定理来自Leviathan、Kalman、Matias（ICML 2023）：输出分布与直接从验证器采样所得分布相同。不是近似，而是完全相同。这是推测性解码在生产中被接受的全部原因——它是一种纯粹的延迟优化，无质量折中。

第7阶段·第16课已为你提供了数学基础。本课则提供训练堆栈。一个优秀的草稿比廉价草稿能带来2倍以上的加速。EAGLE、EAGLE-2和EAGLE-3（Li等人，2024–2025）将“草稿 = 同一模型的较小版本”转化为一门精确的工程学科。2026年生产推理服务器默认使用EAGLE-3。

## 概念

### 不变性：Leviathan拒绝采样

设 `p(t)` 为给定前缀下草稿对下一个token的分布，`q(t)` 为验证器的分布。采样一个草稿token `d ~ p`。以概率 `min(1, q(d) / p(d))` 接受。若拒绝，则从残差分布 `(q - p)_+ / ||(q - p)_+||_1` 中采样。所得样本服从分布 `q`。无论 `p` 有多差，该结论都成立——草稿越差，拒绝越频繁，但输出仍然精确。

将 `N` 个此类调用串联起来，使用一次验证器前向传播来处理 `prefix + d_1 + ... + d_N`。验证器同时返回 `q_1, q_2, ..., q_{N+1}`。从左到右遍历。在第一个拒绝位置 `j` 处，从 `residual(q_j, p_j)` 采样并停止。若全部接受，则从 `q_{N+1}` 采样一个奖励令牌。

### 什么决定了加速

设 `α` 为每个草稿token的预期接受率。令 `c = cost(草稿) / cost(验证器)` 为成本比。每次验证器前向传播预期的接受token数为：

```
E[接受数] = (1 - α^(N+1)) / (1 - α)
```

每个接受token的预期总壁钟时间为 `(N * c + 1) / E[接受数]`。对 `N` 最小化该值即得最佳点。对于 `α = 0.8, c = 0.05`：最优 `N` 约为5–7，加速比为3.2倍。对于 `α = 0.95, c = 0.02`：最优 `N` 约为8–10，加速比可达5倍。

最大的杠杆是 `α`。在固定 `N = 5` 的情况下，将 `α` 从0.6（普通草稿）提升到0.9（EAGLE-3），每次验证器前向传播的预期接受token数从2.2增至4.1。在相同验证器下，吞吐量几乎翻倍。

### 两年演进

**普通推测性（Leviathan, 2023）。** 草稿模型是一个独立训练的、同一家族中较小的LLM。易于搭建，`α ≈ 0.6`，加速比最好约2倍。

**EAGLE-1（Li等人，2024）。** 草稿是一个微型Transformer——通常为一到两层——以验证器的最后一层隐藏状态为输入，直接预测下一个token。由于草稿能看到验证器的特征表示，其分布更接近验证器。`α` 升至0.7–0.8。

**EAGLE-2（Li等人，2024）。** 增加了动态草稿树：不提出单一序列的 `N` 个token，而是提出一个小型候选树，通过一次验证器前向传播（树注意力，Tree Attention）对每个候选评分，并选择概率最高的路径。草稿长度变为每步自适应。每个接受路径token的 `α` 提升至0.85以上。

**EAGLE-3（Li等人，2025，NeurIPS）。** 两项额外改进。第一，完全移除特征预测损失——EAGLE-1/2训练草稿匹配验证器隐藏状态，这限制了数据帮助的上限。EAGLE-3直接在token预测上训练。第二，训练时测试（Training-Time Test，TTT）：在草稿训练期间，将草稿自身之前的预测结果作为输入反馈给后续步骤，方式与其在推理时相同。这使训练分布与测试分布对齐，并阻止误差累积。实测加速：对话场景最高6.5倍，SGLang在H100上batch 64时吞吐量提升38%。

### KV缓存回滚（KV Cache Rollback）

验证通过一次前向传播将验证器的KV缓存扩展 `N` 个条目。如果在位置 `j` 发生拒绝，则缓存中位置 `j-1` 之后的内容现已无效。两种常见实现：写入暂存缓冲区并在接受时提交（vLLM、TensorRT-LLM），或者保留物理KV缓存加上逻辑长度并在拒绝时截断。无论哪种方式，回滚成本是每层每个头的字节数，相比前向传播成本可忽略不计。

对于EAGLE-2树搜索，验证器使用非因果注意力掩码（Non-Causal Mask），该掩码尊重树拓扑。工程实现较为繁琐，但计算本质上是带有自定义掩码的标准Flash Attention调用。

### 2026年的草稿架构

| 策略 | 草稿类型 | `α` | 加速比 | 训练成本 |
|----------|-----------|-----|---------|---------------|
| 普通（Vanilla） | 独立的小LLM | 0.55-0.70 | 1.8-2.3× | 无（复用现有小模型） |
| Medusa | 验证器上的额外LM头 | 0.65-0.75 | 2-3× | 约10亿SFT token |
| EAGLE-1 | 基于隐藏状态的1层Transformer | 0.70-0.80 | 2.5-3× | 约600亿token |
| EAGLE-2 | EAGLE-1 + 动态草稿树 | 0.80-0.88 | 3-4× | 约600亿token |
| EAGLE-3 | 多层特征融合 + TTT | 0.88-0.92 | 3.5-6.5× | 约600-2000亿token |
| 前瞻（Lookahead） | 无草稿（Jacobi迭代） | 不适用 | 1.3-1.6× | 无 |

在2026年生产中：vLLM和SGLang默认使用EAGLE-3（若可用），否则使用EAGLE-2。TensorRT-LLM为Meta和NVIDIA公开模型提供最快的Medusa路径。llama.cpp在CPU部署中提供普通草稿。

## 构建

参见 `code/main.py`。这是完整的Leviathan推测循环，包含所有组件：N个草稿、验证器并行前向传播、逐位置拒绝、残差采样、奖励令牌、KV回滚，以及经验性验证输出分布与直接从 `q` 采样匹配。

### 第一步：拒绝规则

```python
def accept(q_prob, p_prob, u):
    if p_prob <= 0:
        return True
    return u < min(1.0, q_prob / p_prob)
```

### 第二步：残差分布

```python
def residual(q, p):
    raw = [max(0.0, qi - pi) for qi, pi in zip(q, p)]
    s = sum(raw)
    if s == 0:
        return list(q)
    return [r / s for r in raw]
```

### 第三步：一个完整的推测步骤

`spec_step` 函数从 `p` 草稿生成 `N` 个token，然后通过一次并行 `q` 评估验证所有这些token。对每个草稿token应用拒绝规则，并在第一次拒绝时从残差中采样修正。若全部接受，则从 `q_{N+1}` 生成一个奖励令牌。

### 第四步：KV回滚记账

模拟器为每个工作线程（Worker）跟踪一个逻辑 `kv_length`。接受 `k` 个草稿后，`kv_length += k`。若在位置 `j` 拒绝，缓存虽已写入 `j` 之后的内容，但逻辑长度设置为 `prefix_length + j + 1` —— 即修正token之后的位置。后续读取截断到逻辑长度。

### 第五步：Leviathan检验

运行50,000步推测步骤。统计接受token的经验分布。与50,000次直接从 `q` 的采样进行比较。卡方统计量应远低于临界值。定理在实践中成立。

### 第六步：加速比 vs α

通过以不同幅度扰动 `p` 使之偏离 `q` 来扫描草稿质量。测量 `α`，然后绘制每次验证器调用的预期token数随 `α` 和 `N` 变化的图。代码输出一张表格，显示EAGLE-3类草稿质量（`α ≈ 0.9`）如何带来每次验证器调用4–5个token。

## 使用

生产级 `vllm serve` 与EAGLE-3：

```bash
vllm serve meta-llama/Llama-3.3-70B-Instruct \
  --speculative-config '{
    "model": "yuhuili/EAGLE3-LLaMA3.3-Instruct-70B",
    "num_speculative_tokens": 5,
    "method": "eagle3"
  }'
```

SGLang在H100上以batch 64使用EAGLE-3：根据EAGLE-3论文，吞吐量约为batch-64普通解码的1.38倍。

何时使用推测性解码：

- 任何交互式聊天工作负载，其中p50延迟比峰值吞吐量更重要。
- 代码生成和结构化输出（JSON，SQL）。由于目标分布高度可预测，`α` 超过0.9。
- 长文本生成（数千token）。摊销后的加速持续有效。

何时不使用：

- 非常小的模型（< 3B）。草稿相比验证器并不便宜太多。
- 极小的batch-1 CPU部署。草稿模型的内存开销可能不值得。
- 极高温度的创造性采样，此时 `α` 会崩溃。

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|----------------|------------------------|
| Leviathan规则 | "min(1, q over p)" | 以概率 `min(1, q(d)/p(d))` 进行伯努利接受/拒绝，当拒绝时从残差中采样，精确保持验证器分布 |
| 残差分布 | "(q 减 p) 正部分，归一化" | `(q - p)_+` 截断到零并重新归一化——拒绝时应采样的正确分布 |
| 接受率 α | "草稿正确的频率" | 在拒绝规则下每个token预期的伯努利成功概率；支配所有加速数学 |
| EAGLE-1 | "隐藏状态草稿" | 以验证器最后一层隐藏状态为条件的微型Transformer草稿（Li等人，2024） |
| EAGLE-2 | "动态草稿树" | EAGLE-1 加上一个候选延续树，通过一次验证器前向传播中的树注意力评分 |
| EAGLE-3 | "训练时测试" | 移除特征预测损失，在直接token预测上训练，训练时草稿输入其自身输出 |
| 训练时测试 (TTT) | "曝光误差修复" | 在训练期间以自回归方式运行草稿，使训练和测试输入分布匹配——计划采样的直接类比 |
| KV回滚 | "撤销被拒绝的草稿" | 在拒绝后将验证器KV缓存重置为已接受前缀长度的记账操作 |
| 奖励令牌 | "免费的那个" | 当所有 `N` 个草稿都被接受时，从 `q_{N+1}` 采样一个额外token，无需额外验证器成本 |
| 树注意力 | "一次验证多个候选" | 使用非因果注意力掩码，尊重草稿树的拓扑；在一次前向传播中计算树中每个节点的 `q_i` |
