# 百万 Token 上下文的长视频理解

> 1 小时 4K 视频在 24 FPS 下，经过 patch 和嵌入，产生约 6000 万个 token。2 小时播客剧集转录是 30,000 个 token。一部完整的蓝光故事片，即使经过积极池化压缩，也是数十万个 token。Google 的 Gemini 1.5（2024 年 3 月）以 1000 万 token 上下文开启了这一时代，在长达一小时的视频上实现可靠的草垛寻针召回。LWM（Liu 等人，2024 年 2 月）展示了环形注意力的扩展路径。LongVILA 和 Video-XL 进一步扩展了摄取。VideoAgent 用代理检索替换了原始上下文。每种方法在计算、召回和工程复杂度上是不同的权衡。本课并排解读它们。

**类型：** Build
**语言：** Python（stdlib，草垛寻针模拟器 + 代理检索路由器）
**前置知识：** Phase 12 · 17（视频时间 token）
**时间：** ~180 分钟

## 学习目标

- 在不同 FPS 和池化下计算长形式视频的总视觉 token 数量。
- 解释三条扩展路径：暴力上下文（Gemini 1.5）、环形注意力（LWM）、token 压缩（LongVILA / Video-XL）。
- 在精度和延迟上比较原始上下文视频 VLM 与代理检索视频 VLM（VideoAgent）。
- 为 30 分钟视频设计草垛寻针测试，并在特定分钟测量召回。

## 问题所在

Qwen2.5-VL 大小的 patch 在 384 原生分辨率下的单帧约 729 个 token。3x3 池化下每帧 81 个 token。1 FPS 下 30 分钟片段 = 1800 帧 = 145,800 个 token。2025 年开放 VLM 可行，紧张。2 FPS 下，291,600 个 token——只有最大的上下文能容纳。

1 FPS 下 2 小时电影是 583k 个 token。超出大多数 2026 年开放模型；需要 Gemini 2.5 Pro 或更积极地池化。

三条扩展路径出现了。

## 核心概念

### 路径 1：暴力上下文（Gemini 1.5、Claude Opus）

用硬件解决问题。将上下文扩展到数百万 token，在一次前向传播中处理所有内容。

Gemini 1.5 Pro 以 1M token 推出；Gemini 1.5 Ultra 到 10M；2026 年的 Gemini 2.5 Pro 可靠地处理数小时视频。论文（arXiv:2403.05530）记录了在约 950 万 token 内草垛寻针召回率达 99.7%。

工程：具有内存层次结构（局部 + 全局 + 稀疏）的自定义注意力实现，加上用于长上下文效率的 MoE 专家路由。未完整公布细节。非开源。

### 路径 2：环形注意力（LWM、LongVILA）

环形注意力将长序列分布在设备组成的"环"中，每个设备持有一个块。跨完整序列的注意力通过每个设备以环模式将其块发送到下一个设备、计算部分注意力并聚合来实现。

LWM（Liu 等人，2024）以这种方式训练了 100 万 token 上下文模型。训练计算随上下文线性扩展，而非二次——注意力的二次打击被分摊到环的设备上。

LongVILA（arXiv:2408.10188）将这一模式适配到 VLM。1400 帧视频在 192 token/帧 = 268k 上下文，通过 8 路并行环形注意力训练。

### 路径 3：Token 压缩（Video-XL、LongVA）

比暴力上下文更便宜：在 LLM 看到序列之前积极压缩。

Video-XL（arXiv:2409.14485）使用视觉摘要 token：N 帧的每个片段产生一个关注 N 的单一"摘要" token。推理时，LLM 每片段看到一个摘要 token，大幅缩小上下文。

LongVA 通过"长上下文迁移"技术将 LLM 上下文从 200k 扩展到 200 万。在长上下文文本上训练，通过共享表示迁移到长上下文视频。

Token 压缩在特定时间戳的召回与可扩展性之间权衡。模型大致知道发生了什么，但有时错过确切帧。

### 路径 4：代理检索（VideoAgent）

不要将完整视频喂给 LLM。相反，将视频视为数据库，使用 LLM 查询它。

VideoAgent（arXiv:2403.10517）：

1. LLM 阅读问题。
2. LLM 向检索工具请求相关片段（"给我展示有猫的片段"）。
3. 工具返回匹配的片段时间戳。
4. LLM 通过 VLM 阅读这些片段。
5. LLM 组合答案或提出后续查询。

这是应用于长视频的 LLM-as-agent 模式。推理更便宜（只编码相关片段），工程更难（检索质量成为瓶颈）。

### 草垛寻针基准

标准长上下文测试：在视频的随机点插入独特的视觉或文本标记，然后提出需要召回它的查询。

指标：跨视频长度和标记位置的 Recall@k。

Gemini 2.5 Pro 在长达 90 分钟视频上召回率 >99%。开放 72B 模型（Qwen2.5-VL-72B、InternVL3-78B）在 30 分钟时约 85-90%，超过 60 分钟下降。

VideoAgent 在 2+ 小时上可以匹配或击败原始上下文模型，因为如果工具好，检索能命中针。

### 选择哪条路径

对于 15 分钟片段的前沿精度：开放 72B + 原生上下文通常有效。选择 Qwen2.5-VL-72B。

对于 30 分钟到 1 小时内容：开放选 LongVILA 或 Video-XL；封闭选 Gemini 2.5 Pro。质量门槛很重要——前沿走向封闭。

对于 2+ 小时内容：VideoAgent 或类似检索模式。或者，摘要到更小的块并喂入分层摘要。

### 2026 年生产模式

实践中，生产长视频流水线是混合的：

1. 在整个视频上运行动态-FPS 采样 + 积极池化（获得 100k token 全局表示）。
2. 传递给 72B VLM 进行全局摘要。
3. 如果用户提出详细问题，使用摘要作为索引运行代理检索。

这结合了暴力上下文用于全局理解和检索用于局部细节。

## 使用它

`code/main.py`：

- 计算从 1 分钟到 3 小时视频在不同 FPS + 池化下的 token 预算。
- 模拟草垛寻针运行：在随机时间戳注入标记，提出问题，评分召回。
- 包括代理检索路由器模拟器，选择特定片段喂给下游 VLM。

运行预算表，感受规模差距。

## 交付它

本课产出 `outputs/skill-long-video-strategy-planner.md`。给定视频持续时间和查询复杂度，它在暴力上下文、压缩和代理检索之间选择，并计算延迟 + 质量预期。

## 练习

1. 1 FPS 下 45 分钟讲座，每帧 81 个 token。总 token？适合哪些模型的上下文？

2. 设计草垛寻针测试：你在哪分钟注入标记，确切查询格式是什么？

3. 在 1 小时视频上比较暴力上下文 Qwen2.5-VL-72B（80k 上下文）与 VideoAgent（Claude 3.5 + 检索）。哪个在召回上获胜？哪个在延迟上获胜？

4. 环形注意力的内存成本随序列长度线性扩展，随设备数量线性扩展。解释为什么，以及如果去掉环旋转阶段什么会失败。

5. 阅读 Gemini 1.5 第 5 节关于草垛寻针。论文在 1M vs 10M token 边界上发现了什么关于召回？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------------|------------------------|
| 暴力上下文 | "只是更多 token" | 将 LLM 上下文扩展到数百万 token；一次传播处理所有内容 |
| 环形注意力 | "LWM 风格并行" | 分布式注意力模式，每个设备持有一个块并旋转 |
| Token 压缩 | "摘要 token" | 在 LLM 之前通过学习的压缩器减少每片段 token |
| 草垛寻针 | "NIH 测试" | 在随机点插入独特标记，测试时要求模型召回它 |
| 代理检索 | "LLM 作为查询规划器" | LLM 向检索工具请求相关片段，通过 VLM 阅读它们，组合答案 |
| VideoAgent | "视频的检索模式" | 经典代理检索设计：问题 -> 工具 -> 片段 -> 答案 |
