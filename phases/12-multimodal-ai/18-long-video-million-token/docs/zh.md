# 18 · 百万 Token 上下文下的长视频理解

> 一段 1 小时、4K、24 FPS 的视频，经过分块（patch）与嵌入后，会产生大约 6000 万个 token。一集 2 小时的播客转写成文字是 3 万个 token。一部完整的蓝光电影，即便用激进的池化压缩，也仍是数十万个 token。Google 的 Gemini 1.5（2024 年 3 月）以 1000 万 token 的上下文开启了这个时代，能在长达一小时的视频上做可靠的「大海捞针（needle-in-a-haystack）」式召回。LWM（Liu 等人，2024 年 2 月）展示了「环形注意力（ring attention）」的扩展路径。LongVILA 与 Video-XL 进一步扩大了摄入规模。VideoAgent 则用「智能体式检索（agentic retrieval）」替代了原始上下文。每种方案在算力、召回率与工程复杂度上做出了不同的取舍。本课将它们并排对比。

**类型：** 构建（Build）
**语言：** Python（标准库，大海捞针模拟器 + 智能体式检索路由器）
**前置：** 第 12 阶段 · 第 17 课（视频时序 token）
**时长：** 约 180 分钟

## 学习目标

- 计算长视频在不同 FPS 与池化设置下的视觉 token 总量。
- 解释三条扩展路径：暴力上下文（Gemini 1.5）、环形注意力（LWM）、token 压缩（LongVILA / Video-XL）。
- 在准确率与延迟两个维度上，对比原始上下文式视频 VLM 与智能体式检索视频 VLM（VideoAgent）。
- 为一段 30 分钟视频设计大海捞针测试，并测量在特定分钟处的召回率。

## 问题所在

在 384 原生分辨率下，Qwen2.5-VL 尺寸的单帧分块约为 729 个 token。采用 3x3 池化后，每帧约 81 个 token。一段 30 分钟、1 FPS 的片段 = 1800 帧 = 145,800 个 token。2025 年的开源 VLM 勉强能处理。若提到 2 FPS，则为 291,600 个 token——只有最大的上下文才装得下。

一部 2 小时、1 FPS 的电影是 58.3 万个 token。超出了 2026 年大多数开源模型的能力；需要 Gemini 2.5 Pro，或者更激进地池化。

由此出现了三条扩展路径。

## 核心概念

### 路径一：暴力上下文（Gemini 1.5、Claude Opus）

用硬件硬刚问题。把上下文扩展到数百万 token，在一次前向传播中处理全部内容。

Gemini 1.5 Pro 发布时支持 100 万 token；Gemini 1.5 Ultra 达到 1000 万；2026 年的 Gemini 2.5 Pro 已能可靠处理数小时的视频。论文（arXiv:2403.05530）记录了在最高约 950 万 token 时，大海捞针召回率仍达 99.7%。

工程实现：一套带有内存层级（局部 + 全局 + 稀疏）的自定义注意力实现，外加用于长上下文效率的 MoE 专家路由。细节未完整公开，亦非开源。

### 路径二：环形注意力（LWM、LongVILA）

环形注意力把长序列分布到多个设备上，排成一个「环（ring）」，每个设备持有一个分块。要在完整序列上做注意力，每个设备按环形模式把自己的分块发送给下一个设备，计算部分注意力，再聚合。

LWM（Liu 等人，2024）正是用这种方式训练出了一个 100 万 token 上下文的模型。训练算力随上下文长度线性增长，而非二次方增长——注意力的二次方开销被分摊到了环上的各个设备。

LongVILA（arXiv:2408.10188）把这一模式适配到了 VLM。1400 帧视频、每帧 192 个 token = 26.8 万上下文，在 8 路并行的环形注意力下训练完成。

### 路径三：token 压缩（Video-XL、LongVA）

比暴力上下文更省：在 LLM 看到序列之前就激进地压缩。

Video-XL（arXiv:2409.14485）使用「视觉摘要 token（visual summary token）」：每个含 N 帧的片段产出单个「摘要」token，该 token 对这 N 帧做注意力。推理时，LLM 每个片段只看到一个摘要 token，从而大幅缩减上下文。

LongVA 用「长上下文迁移（long context transfer）」技术把 LLM 上下文从 20 万扩展到 200 万。先在长上下文文本上训练，再通过共享表征迁移到长上下文视频。

token 压缩以牺牲特定时间戳处的召回率换取可扩展性。模型大致知道发生了什么，但有时会错过确切的帧。

### 路径四：智能体式检索（VideoAgent）

不要把完整视频喂给 LLM。而是把视频当作一个数据库，用 LLM 去查询它。

VideoAgent（arXiv:2403.10517）：

1. LLM 读取问题。
2. LLM 向检索工具索取相关片段（「给我看有猫的片段」）。
3. 工具返回匹配片段的时间戳。
4. LLM 通过 VLM 阅读这些片段。
5. LLM 组织出答案，或提出后续查询。

这是将「LLM 即智能体（LLM-as-agent）」模式应用于长视频。推理更便宜（只编码相关片段），但工程更难（检索质量成为瓶颈）。

### 大海捞针基准测试

标准的长上下文测试：在视频的随机位置插入一个独特的视觉或文本标记，然后提出一个需要召回该标记的查询。

指标：横跨视频长度与标记位置的 Recall@k。

Gemini 2.5 Pro 在最长 90 分钟的视频上召回率超过 99%。开源 72B 模型（Qwen2.5-VL-72B、InternVL3-78B）在 30 分钟处召回率约为 85-90%，超过 60 分钟后开始下降。

VideoAgent 在 2 小时以上时可以追平甚至击败原始上下文模型，因为只要检索工具足够好，它就能命中「针」。

### 该选哪条路径

15 分钟片段、追求前沿准确率：开源 72B + 原生上下文通常够用。选 Qwen2.5-VL-72B。

30 分钟到 1 小时内容：开源选 LongVILA 或 Video-XL；闭源选 Gemini 2.5 Pro。质量标准很重要——追求前沿就走闭源。

2 小时以上内容：VideoAgent 或类似的检索模式。或者，先摘要成更小的块，再喂入分层摘要。

### 2026 年的生产实践模式

实践中，生产级长视频管线是混合式的：

1. 对整段视频做动态 FPS 采样 + 激进池化（得到约 10 万 token 的全局表征）。
2. 传给一个 72B VLM 生成全局摘要。
3. 若用户提出细节问题，则以该摘要为索引，运行智能体式检索。

这把用于全局理解的暴力上下文与用于局部细节的检索结合了起来。

## 动手实践

`code/main.py`：

- 计算从 1 分钟到 3 小时、不同 FPS + 池化设置下视频的 token 预算。
- 模拟一次大海捞针运行：在随机时间戳注入标记，提出问题，给召回率打分。
- 包含一个智能体式检索路由器模拟器，它会挑选特定片段喂给下游 VLM。

## 练习

1. 一节 45 分钟的讲座，1 FPS，每帧 81 个 token。总 token 数是多少？能装进哪些模型的上下文？

2. 设计一个大海捞针测试：你在第几分钟注入标记？确切的查询格式是什么？

3. 在一段 1 小时视频上，对比暴力上下文的 Qwen2.5-VL-72B（8 万上下文）与 VideoAgent（Claude 3.5 + 检索）。谁在召回率上胜出？谁在延迟上胜出？

4. 环形注意力的内存成本随序列长度线性增长、随设备数量线性增长。解释原因，以及如果去掉环形轮转（ring-rotation）阶段会出什么问题。

5. 阅读 Gemini 1.5 论文第 5 节关于大海捞针的内容。论文对于 100 万 vs 1000 万 token 边界处的召回率有何发现？

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|-----------------|------------------------|
| 暴力上下文（Brute context） | 「就是更多 token」 | 把 LLM 上下文扩展到数百万 token；一次处理全部内容 |
| 环形注意力（Ring attention） | 「LWM 式并行」 | 一种分布式注意力模式，每个设备持有一个分块并进行轮转 |
| token 压缩（Token compression） | 「摘要 token」 | 在 LLM 之前，用一个学习得到的压缩器减少每个片段的 token 数 |
| 大海捞针（Needle-in-haystack） | 「NIH 测试」 | 在随机位置插入一个独特标记，测试时要求模型召回它 |
| 智能体式检索（Agentic retrieval） | 「LLM 即查询规划器」 | LLM 向检索工具索取相关片段，通过 VLM 阅读，再组织出答案 |
| VideoAgent | 「视频的检索模式」 | 经典的智能体式检索设计：问题 -> 工具 -> 片段 -> 答案 |
