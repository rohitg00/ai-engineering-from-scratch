# 01 · 视觉 Transformer 与图块-Token 原语

> 在任何多模态处理开始之前，图像必须先变成 Transformer 能"吃下"的 Token 序列。2020 年的 ViT 论文给出的答案是：16x16 像素的图块、一个线性投影、再加一个位置嵌入。五年过去，2026 年的每一个前沿模型（原生支持 2576px 的 Claude Opus 4.7、Gemini 3.1 Pro、Qwen3.5-Omni）仍然是这样起步的——编码器从 ViT 换到 DINOv2 再换到 SigLIP 2，加入了寄存器 Token，位置方案变成了 2D-RoPE，但这个原语始终未变。本课会从头到尾通读图块-Token 流水线，并用纯标准库 Python 把它实现出来，让第 12 阶段后续课程对"视觉 Token"有一个具体的心智模型。

**类型：** 学习
**语言：** Python（标准库，图块分词器 + 几何计算器）
**前置：** 第 7 阶段（Transformer）、第 4 阶段（计算机视觉）
**时长：** 约 120 分钟

## 学习目标

- 把一张 HxWx3 的图像转换成带有正确位置编码的图块 Token 序列。
- 给定（图块大小、分辨率、隐藏维度、深度），计算某个 ViT 的序列长度、参数量和 FLOPs。
- 说出让 ViT 从 2020 年的研究走向 2026 年生产环境的三项升级：自监督预训练（DINO / MAE）、寄存器 Token（register tokens），以及原生分辨率打包（native-resolution packing）。
- 为下游任务在 CLS 池化、均值池化和寄存器 Token 之间做出选择。

## 问题所在

Transformer 处理的是向量序列。文本本身就是序列（字节或 Token）。而图像是一个带三个颜色通道的二维像素网格——并不是序列。如果你把每个像素都展平，一张 224x224 的 RGB 图像会变成 150,528 个 Token，而在这种长度上做自注意力根本行不通（计算量与序列长度成二次方关系）。

2020 年之前的做法是在前端硬接一个 CNN 特征提取器：ResNet 产生一个 7x7 的特征图，每个位置是 2048 维向量，把这 49 个 Token 喂给 Transformer。这能跑通，但继承了 CNN 的偏置（平移等变性、局部感受野），也丧失了 Transformer 对规模扩展的"胃口"。

Dosovitskiy 等人（2020）抛出了一个直白的问题：如果我们干脆跳过 CNN 呢？把图像切成固定大小的图块（比如 16x16 像素），对每个图块做线性投影得到一个向量，加上位置嵌入，再把整个序列喂给一个普通的 Transformer。在当时这是离经叛道的——没有卷积的视觉。但只要数据足够多（先是 JFT-300M，后来是 LAION），它在 ImageNet 上就超过了 ResNet，而且还在持续进步。

到 2026 年，ViT 原语已是毋庸置疑的基石。每个开放权重视觉语言模型（VLM）的视觉塔都是它的某个后代（DINOv2、SigLIP 2、CLIP、EVA、InternViT）。问题不再是"我们该不该用图块？"，而是"用多大的图块、用什么分辨率调度、用什么预训练目标、用什么位置编码。"

## 核心概念

### 图块即 Token

给定一张形状为 `(H, W, 3)` 的图像 `x` 和一个图块大小 `P`，你把图像切成一个 `(H/P) x (W/P)` 的非重叠图块网格。每个图块是一个 `P x P x 3` 的像素立方体。把每个立方体展平成一个 `3 P^2` 维的向量。再施加一个共享的线性投影 `W_E`（形状为 `(3 P^2, D)`），把每个图块映射到模型的隐藏维度 `D`。

以 ViT-B/16 的标准配置为例：
- 分辨率 224，图块大小 16 → 网格 14x14 → 196 个图块 Token。
- 每个图块是 `16 x 16 x 3 = 768` 个像素值，投影到 `D = 768`。
- 再加一个可学习的 `[CLS]` Token → 序列长度变为 197。

图块投影在数学上等价于一个核大小为 `P`、步长为 `P`、输出通道为 `D` 的二维卷积。生产代码实际上就是这么实现的——`nn.Conv2d(3, D, kernel_size=P, stride=P)`。"线性投影"是概念上的说法；"卷积核"是高效的实现。

### 位置嵌入

图块本身没有固有顺序——Transformer 把它们看作一个"袋子"。早期的 ViT 加入了一个可学习的一维位置嵌入（每个位置一个 768 维向量，共 197 个）。这能用，但会把模型绑死在训练分辨率上：推理时如果你改变了网格大小，就得对位置表做插值。

现代视觉骨干网络使用 2D-RoPE（Qwen2-VL 的 M-RoPE、SigLIP 2 的默认方案）或分解式二维位置。2D-RoPE 根据图块的（行、列）索引来旋转查询向量和键向量，于是模型从旋转角度推断出相对的二维位置。不需要位置表。模型在推理时能处理任意网格尺寸。

### CLS Token、池化输出与寄存器 Token

图像级别的表示是什么？有三种方案并存：

1. `[CLS]` Token。在图块序列前面预置一个可学习的向量。经过所有 Transformer 块之后，CLS Token 的隐藏状态就是图像表示。继承自 BERT。最初的 ViT 和 CLIP 都用它。
2. 均值池化（Mean pool）。对图块 Token 的输出隐藏状态取平均。SigLIP、DINOv2 以及大多数现代 VLM 都用它。
3. 寄存器 Token（Register tokens）。Darcet 等人（2023）观察到，没有显式"汇点 Token"的 ViT 在训练中会发展出高范数的"伪影"图块，它们会劫持自注意力。加入 4–16 个可学习的寄存器 Token 能吸收这部分负载，从而提升稠密预测（分割、深度估计）的质量。DINOv2 和 SigLIP 2 都自带寄存器。

这一选择对下游任务很重要。对分类任务来说，CLS 足矣。而对于把图块 Token 喂进大语言模型（LLM）的 VLM，你会完全跳过池化——每个图块都成为一个 LLM 输入 Token。寄存器 Token 在交给 LLM 之前会被丢弃（它们是脚手架，不是内容）。

### 预训练：监督、对比、掩码、自蒸馏

2020 年的 ViT 是在 JFT-300M 上用监督分类预训练的。很快就被以下方法取代：

- CLIP（2021）：在 4 亿图文对上做对比学习。见第 12.02 课。
- MAE（2021，He 等人）：掩盖 75% 的图块，重建像素。自监督，可在纯图像上工作。
- DINO（2021）/ DINOv2（2023）：师生结构的自蒸馏，无需标签，无需图文描述。2023 年的 DINOv2 ViT-g/14 是最强的纯视觉骨干网络，也是"稠密特征"用例的默认选择。
- SigLIP / SigLIP 2（2023、2025）：用 sigmoid 损失的 CLIP，配合 NaFlex 支持原生宽高比。它是 2026 年开放 VLM 中占主导地位的视觉塔（Qwen、Idefics2、LLaVA-OneVision）。

你对预训练的选择决定了骨干网络擅长什么：CLIP/SigLIP 用于与文本的语义匹配，DINOv2 用于稠密视觉特征，MAE 作为下游微调的起点。

### 缩放定律

ViT 的缩放研究（Zhai 等人，2022）确立了一点：ViT 的质量在模型规模、数据规模和算力上服从可预测的定律。在固定算力下：
- 更大的模型 + 更多的数据 → 更好的质量。
- 图块大小是在序列长度与保真度之间权衡的一个杠杆。图块 14（DINOv2/SigLIP SO400m 的典型选择）比图块 16 每张图产生更多 Token；对 OCR 和稠密任务更好，但速度更慢。
- 分辨率是另一个大杠杆。从 224 提升到 384 再到 512 几乎总是有帮助的，代价是 FLOPs 呈二次方增长。

ViT-g/14（10 亿参数，图块 14，分辨率 224 → 256 个 Token）和 SigLIP SO400m/14（4 亿参数，图块 14）是 2026 年开放 VLM 的两个主力编码器。

### 一个 ViT 的参数量

完整的计算见 `code/main.py`。以 224 分辨率下的 ViT-B/16 为例：

```
patch_embed = 3 * 16 * 16 * 768 + 768  =  591k
cls + pos    = 768 + 197 * 768          =  152k
block        = 4 * 768^2 (QKVO) + 2 * 4 * 768^2 (MLP) + 2 * 2*768 (LN)
             = 12 * 768^2 + 3k          =  7.1M
12 blocks    = 85M
final LN    = 1.5k
total       ≈ 86M
```

在加载检查点之前，用这种方式给每个 ViT 估个大概。骨干网络的大小决定了任何下游 VLM 的 VRAM 下限。

### 2026 年的生产配置

2026 年大多数开放 VLM 搭载的编码器是 SigLIP 2 SO400m/14，运行在原生分辨率下（NaFlex）。它具有：
- 4 亿参数。
- 图块大小 14，默认分辨率 384 → 每张图 729 个图块 Token。
- 图像级别任务用均值池化；VQA 任务则把全部 729 个图块都送入 LLM。
- 4 个寄存器 Token，在交给 LLM 之前丢弃。
- 带图像级缩放的 2D-RoPE，以支持原生宽高比。

这套配置里的每一个决策，都能追溯到一篇你可以亲自去读的论文。

## 动手用

`code/main.py` 是一个图块分词器兼几何计算器。它接收（图像 H、W，图块 P，隐藏维度 D，深度 L），并报告：

- 分块后的网格形状和序列长度。
- 一张合成的 8x8 像素玩具图像的 Token 序列（走一遍展平 + 投影的流程）。
- 按图块嵌入、位置嵌入、Transformer 块和头部分解的参数量。
- 目标分辨率下每次前向传播的 FLOPs。
- 一张横向对比表，涵盖 ViT-B/16 @ 224、ViT-L/14 @ 336、DINOv2 ViT-g/14 @ 224、SigLIP SO400m/14 @ 384。

跑一遍。把参数量与已发表的数字对上。调整图块大小和分辨率，感受一下 Token 数量的代价。

## 练习

1. 计算 Qwen2.5-VL 在原生 1280x720 输入、图块大小 14 时的图块-Token 序列长度。它与一个仅用 CLS 的表示相比如何？

2. 一帧 1080p（1920x1080）画面在图块 14 下产生多少 Token？在 30 FPS、时长 5 分钟的视频中，总共有多少视觉 Token？哪种方式给你省下的成本最多：池化、抽帧采样，还是 Token 合并？

3. 用纯 Python 实现对图块 Token 的均值池化。验证：对一个 DINOv2 输出的 196 个 Token 做均值池化，结果是否与你向模型 `forward` 索取池化嵌入时返回的结果一致。

4. 阅读《Vision Transformers Need Registers》（arXiv:2309.16588）的第 3 节。用两句话描述寄存器吸收的是什么伪影，以及它为何对下游稠密预测重要。

5. 修改 `code/main.py` 以支持图块打包（patch-n'-pack）：给定一组不同分辨率的图像，产生单个打包序列以及块对角的注意力掩码。等你学到第 12.06 课时，对照验证。

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|----------------|------------------------|
| Patch（图块） | "16x16 的像素方块" | 输入图像中一块固定大小、非重叠的区域；会变成一个 Token |
| Patch embedding（图块嵌入） | "线性投影" | 一个共享的、学习得到的矩阵（或步长为 P 的 Conv2d），把展平的图块像素映射成 D 维向量 |
| CLS token（CLS Token） | "类别 Token" | 预置的可学习向量，其最终隐藏状态代表整张图像；在 2026 年是可选项 |
| Register token（寄存器 Token） | "汇点 Token" | 额外的可学习 Token，用于吸收 ViT 在预训练中产生的高范数注意力伪影 |
| Position embedding（位置嵌入） | "位置信息" | 让序列具有顺序感知能力的逐位置向量或旋转；2D-RoPE 是现代默认方案 |
| Grid（网格） | "图块网格" | 给定分辨率和图块大小下，图块组成的 (H/P) x (W/P) 二维数组 |
| NaFlex | "原生灵活分辨率" | SigLIP 2 的特性：单个模型无需重训即可服务多种宽高比和分辨率 |
| Backbone（骨干网络） | "视觉塔" | 预训练的图像编码器，其图块-Token 输出在 VLM 中喂给 LLM |
| Pooling（池化） | "图像级摘要" | 把图块 Token 汇成一个向量的策略：CLS、均值、注意力池化或基于寄存器 |
| Patch 14 vs 16（图块 14 对 16） | "更细对更粗的网格" | 图块 14 每张图产生更多 Token，对 OCR 保真度更好但更慢；图块 16 是经典默认值 |
