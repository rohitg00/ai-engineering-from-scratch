# ColPali 与原生视觉文档检索增强生成（Vision-Native Document RAG）

> 传统的检索增强生成（RAG）将 PDF 解析为文本，然后分块、嵌入块、存储向量。每一步都会丢失信息：OCR 丢失了图表数据、分块破坏了表格行、文本嵌入忽略了图形。ColPali（Faysse 等人，2024年7月）提出了一个更简单的问题：为什么非要提取文本？直接通过 PaliGemma 嵌入页面图像，使用 ColBERT 风格的后期交互（Late Interaction）进行检索，保留文档携带的所有布局、图形、字体和格式信息。已发表的基准测试结果：在视觉丰富的文档上，端到端准确率比文本 RAG 高出 20-40%。ColQwen2、ColSmol 和 VisRAG 扩展了这一模式。本课程将阅读原生视觉 RAG 论文，并构建一个类似 ColPali 的小型索引器。

**类型：** 构建
**语言：** Python（标准库，多向量索引器 + MaxSim 评分器）
**前置条件：** 阶段 11（LLM 工程 — RAG 基础），阶段 12 · 05（LLaVA）
**时间：** 约 180 分钟

## 学习目标

- 解释双编码器检索（每个文档一个向量）与后期交互检索（每个文档多个向量）之间的区别。
- 描述 ColBERT 的 MaxSim 操作，以及 ColPali 如何将其从文本令牌推广到图像块。
- 构建一个类似 ColPali 的小型索引器：页面 → 图像块嵌入 → 查询词嵌入上的 MaxSim → 前 k 个页面。
- 在发票/财务报告用例上比较 ColPali + Qwen2.5-VL 生成器与文本 RAG + GPT-4。

## 问题所在

在 PDF 上使用文本 RAG 会丢弃文档的大部分内容。财务报告中的第三季度收入增长通常以图表形式呈现；医学报告的发现位于带注释的图像中；法律合同的签名块是一个布局事实，而非文本事实。

文本 RAG 流程：

1. PDF → 通过 OCR / pdftotext 转换为文本。
2. 文本 → 300-500 令牌块。
3. 块 → 双编码器嵌入（一个向量）。
4. 用户查询 → 嵌入 → 余弦相似度 → 前 k 个块。
5. 块 + 查询 → LLM。

五个有损步骤。图表未被捕获。表格跨块断裂。多列布局被压平。图形注释消失。

ColPali 的解决方案：跳过 OCR，直接嵌入页面图像。使用 ColBERT 风格的后期交互进行检索，使模型在查询时能够关注细粒度块。

## 概念

### ColBERT（2020）

ColBERT（Khattab 和 Zaharia，arXiv:2004.12832）是一种文本检索方法。它不是为每个文档生成一个向量，而是为每个令牌生成一个向量。在查询时：

- 查询令牌获得自己的嵌入（N_q 个向量）。
- 文档令牌获得嵌入（N_d 个向量，通常缓存）。
- 得分 = 对查询令牌求和，取文档令牌上的最大余弦相似度：Σ_i max_j cos(q_i, d_j)。

这就是 MaxSim 操作。每个查询令牌“挑选”其最佳匹配的文档令牌。最终得分是这些值的总和。

优点：召回率高，处理令牌级语义。缺点：每个文档需要 N_d 个向量，存储成本高。

### ColPali

ColPali（Faysse 等人，arXiv:2407.01449）将 ColBERT 模式应用于图像。

- 每个页面由 PaliGemma（ViT + 语言）编码为图像块嵌入：每个页面 N_p 个向量。
- 每个用户查询（文本）被编码为查询令牌嵌入：N_q 个向量。
- 得分 = Σ_i max_j cos(q_i, p_j)，即查询文本令牌与页面图像块之间的 MaxSim。
- 按总得分检索前 k 个页面。

在文档摄入时：用 PaliGemma 嵌入每个页面，存储所有图像块嵌入。在查询时：嵌入查询令牌，计算与所有存储的页面嵌入之间的 MaxSim，返回前 k 个页面。

优点：在视觉丰富的文档上，端到端比文本 RAG 好 20-40%。每个图像块向量捕获局部布局和内容。

缺点：每个页面 N_p 个图像块 × 4 字节浮点数 × D 维向量 = 存储空间迅速增长。可通过 PQ / OPQ 量化缓解。

### ColQwen2 和 ColSmol

ColQwen2（illuin-tech，2024-2025）将 PaliGemma 替换为 Qwen2-VL。更好的基础编码器，更好的检索性能。

ColSmol 是面向本地/边缘使用的小规模变体。一个约 1B 参数的 ColSmol 检索器可以在消费级 GPU 上运行。

### VisRAG

VisRAG（Yu 等人，arXiv:2410.10594）是另一个变体：它不是对图像块进行 MaxSim，而是使用视觉语言模型（VLM）将每个页面池化为单个向量，然后进行双编码器检索。索引更快 + 存储更小，但召回率较弱。

质量与成本的权衡：ColPali 有利于质量，VisRAG 有利于规模。

### M3DocRAG

M3DocRAG（Cho 等人，arXiv:2411.04952）将多模态检索扩展到多页面多文档推理。跨文档检索页面，为 VLM 组成多页面上下文。

### ViDoRe — 基准测试

ColPali 的配套基准测试。视觉文档检索评估（Visual Document Retrieval Evaluation）。任务包括财务报告、科学论文、行政文件、医疗记录、手册。指标：nDCG@5。

ColPali-v1 在 ViDoRe 上得分约 80% nDCG@5；相同文档上的文本 RAG 得分约 50-60%。

### 端到端 RAG 流程

对于原生视觉 RAG：

1. 摄入：PDF → 页面图像 → PaliGemma 编码 → 存储所有图像块嵌入。
2. 查询：用户文本 → 查询令牌嵌入 → 对所有已索引页面进行 MaxSim → 前 k 个页面。
3. 生成：前 k 个页面图像 + 查询 → VLM（Qwen2.5-VL 或 Claude）→ 答案。

没有任何地方使用 OCR。图形、图表、字体、布局全部流入答案。

### 存储计算

一份 50 页的财务报告，每页 729 个图像块，128 维嵌入：

- ColPali：50 * 729 * 128 * 4 字节 = 约 18 MB 原始大小，PQ 后约 4 MB。
- 文本 RAG：50 个块 * 768 维 * 4 字节 = 约 150 kB。

ColPali 每个文档的存储量大约是文本 RAG 的 30 倍。大规模下，OPQ / PQ 可将其降至约 5-10 倍，通常可以接受。

### 文本 RAG 仍然胜出的场景

- 没有布局信号的纯文本文档（维基文章、聊天记录）。文本 RAG 更简单且存储成本更低。
- 数百万页的档案，其中存储成本占主导地位。
- 严格的监管要求，除了检索外还要有可提取的 OCR 文本。

对于 2026 年的其他所有场景——财务报告、科学论文、法律合同、医疗记录、UX 文档——原生视觉 RAG 胜出。

## 使用它

`code/main.py`：

- 玩具级图像块编码器：将一个“页面”（小特征向量网格）映射为图像块嵌入数组。
- MaxSim 评分器：计算查询令牌嵌入集与页面图像块集之间的 ColBERT 风格得分。
- 索引 5 个玩具页面，运行 3 个查询，返回带有得分的 top-k。

## 练习

1. 一份 200 页的年报，每页 729 个图像块，128 维嵌入，4 字节浮点数。计算原始存储和 PQ 压缩后（8 倍）的存储量。

2. MaxSim 是 Σ_i max_j cos(q_i, p_j)。这个总和捕捉了哪些简单的平均相似度无法捕捉的信息？

3. ColPali 将页面索引为图像块集。如果我们改为像 ColBERT 那样在单词级别进行索引，会发生什么变化？权衡是什么？

4. 为一个 100 万页的语料库设计端到端流程，延迟预算为每查询 500 毫秒。选择 ColQwen2 / VisRAG 并说明理由。

5. 阅读 M3DocRAG（arXiv:2411.04952）。描述多页面注意力模式，以及它与单页面 ColPali 检索的不同之处。

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|-----------|----------|
| 后期交互（Late interaction） | "ColBERT 风格" | 使用每令牌或每图像块嵌入 + MaxSim 进行检索，而不是单个文档向量 |
| MaxSim | "图像块上的最大值" | 对于每个查询令牌，选择相似度最高的文档令牌；然后对查询求和 |
| 双编码器（Bi-encoder） | "单向量" | 每个文档一个向量；更快但失去粒度 |
| 多向量（Multi-vector） | "每文档多向量" | 每个文档/页面存储 N_p 个向量；存储成本增加但召回率提高 |
| 图像块嵌入（Patch embedding） | "页面特征" | 来自 VLM 编码器的每个图像块的一个向量，每页面缓存 |
| ViDoRe | "视觉文档基准" | ColPali 的视觉文档检索基准套件 |
| 乘积量化（PQ quantization） | "积量化" | 在保持向量相似性的同时将存储压缩约 8 倍的压缩技术 |
