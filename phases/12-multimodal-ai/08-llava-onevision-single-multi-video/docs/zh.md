# 08 · LLaVA-OneVision：单图、多图、视频统一于一个模型

> 在 LLaVA-OneVision（Li 等人，2024 年 8 月）出现之前，开源视觉语言模型（open-VLM）领域存在彼此割裂的几条技术脉络：用于单图的 LLaVA-1.5、Mantis 和 VILA 这类多图模型、Video-LLaVA 和 Video-LLaMA 这类视频模型。每个模型都能在各自的基准（benchmark）上夺冠，却在其他场景上失败。LLaVA-OneVision 提出：一套统一的课程（curriculum）就能训练出在三类场景上全面领先的单一模型，而且其涌现出的任务迁移（task-transfer）效应——单图技能迁移到视频、多图推理能力迁移到单图——会胜过各专精模型之和。这套配方看似简单：一个在各场景间保持恒定的视觉 token 预算（visual-token budget），外加一套显式课程，从单图推进到 OneVision（多图）再到视频。本课将剖析这个预算、这套课程，以及由此产生的涌现行为。

**类型：** 实践构建
**语言：** Python（标准库，token 预算求解器 + 课程规划器）
**前置：** 阶段 12 · 05（LLaVA）、阶段 12 · 06（任意分辨率）
**时长：** 约 180 分钟

## 学习目标

- 设计一个在单图、多图与视频输入间保持恒定的视觉 token 预算。
- 编排一套训练课程，将技能从单图迁移到视频且不产生灾难性遗忘（catastrophic forgetting）。
- 解释在相同参数量下，只要课程设计得当，单一模型为何能击败各专精模型。
- 说出 LLaVA-OneVision 报告的三种涌现能力：多摄像头推理、标记集提示（set-of-mark prompting）、iPhone 截图智能体。

## 问题所在

图像、多图与视频对模型施加的压力各不相同。

单图需要高分辨率 token（AnyRes，约 2880 个视觉 token）来捕捉 OCR 和精细细节。每样本预算：一张图，2880 个 token。

多图需要若干张中等分辨率的图像（每张约 576 个 token），从而让跨图像的推理能塞进上下文。每样本预算：4-8 张图，每张 576，共 2300-4600 个 token。

视频需要大量低分辨率的帧（池化后每帧约 196 个 token）来捕捉时间动态。每样本预算：8-32 帧，每帧 196，共 1600-6200 个 token。

如果你训练多个独立模型，那就为每个挑一个预算。如果你训练一个统一模型，就需要这个预算能在各场景间合理伸缩，而不撑爆上下文。

在 OneVision 之前，默认做法是「只训练一个场景，忽略其他」。Video-LLaVA 通过额外的训练阶段把视频能力硬加到一个图像模型上。LLaVA-NeXT 用切片（tiling）加入了多图支持。没有一个能干净利落地同时处理三类场景。

## 核心概念

### OneVision token 预算

LLaVA-OneVision 选定了一个约为每样本 3000-4000 个 token 的统一视觉 token 预算，但在各场景间以不同方式分配：

- 单图：AnyRes-9（3x3 切片 + 缩略图），每个切片在 384 分辨率下含 729 个 patch，经过激进的 2x2 双线性池化（bilinear pooling）→ 每切片 182 个。合计：9 * 182 + 182 = 1820 个 token。或者用 AnyRes-4，每切片 729 = 2916 + 729。
- 多图：每张图像在中等分辨率（384，不切片）下含 729 个 token，不做池化。6 张图的预算 → 4374 个 token。
- 视频：32 帧，384 分辨率，经过激进的 3x3 双线性池化 → 每帧 81 个 token。合计：32 * 81 = 2592 个 token。

这种分配方式使 token 总数大致保持恒定。LLM 永远不会看到一个撑爆其上下文的批次。编码器在不同场景下产生不同的几何结构，但 LLM 消耗的是同一份预算。

### 三阶段课程

LLaVA-OneVision 分三个阶段训练：

1. 单图 SFT（SI 阶段）。所有数据都是「单图 + 文本」。在高分辨率 AnyRes 输入上训练。这一步教会感知、OCR 与细粒度理解。使用 LLaVA-NeXT 数据外加 OneVision 专用的单图数据。
2. OneVision SFT（OV 阶段）。混合单图 + 多图 + 视频（均匀采样的帧）。在统一 token 预算上训练。这一步教会模型处理异构的批次形状。不重置权重——从 SI 阶段继续。
3. 任务迁移（TT 阶段）。继续在一个目标任务混合上训练，通常根据产品需求偏重多图或视频。这是面向部署的可选微调。

关键点：课程顺序很重要。先训练视频或先训练多图，会比先训练单图产生更差的图像性能，即便用相同的数据。论文对此做了显式的消融实验（ablate）。

### 为什么课程有效

单图训练构建了感知基础。patch token 携带细粒度的视觉特征；LLM 学会把它们与文本整合。多图与视频引入了结构性挑战（哪张图是哪张、什么先发生），这些挑战在缺乏强感知基础时很难学会。

如果你从头把所有场景一起训练，模型会对感知欠拟合（每个批次的单图数据有限）而对结构过拟合（大量多图/视频数据）。结果是：一个能遵循跨图推理模式、但视觉上很肤浅的模型。

课程式排序让你先从 SI 阶段获得感知强度，再从 OV 阶段获得组合性/时序推理能力，二者皆不丢失。

### 涌现的跨场景技能

LLaVA-OneVision 论文报告了三种涌现能力：

1. 多摄像头推理。分别在多图 + 视频上训练；在推理时被要求对一个多摄像头的驾驶场景进行推理。尽管训练中从未见过这个确切格式，模型仍正确地整合了各路视图。
2. 标记集提示（set-of-mark prompting）。用户在图像中用编号标记标注若干对象；模型推理「标记 3 相对于标记 7 在做什么」。它既没在标记上训练过，也没在标注上训练过；这种能力是从空间定位（spatial grounding）+ 多图指代的组合中学来的。
3. iPhone 截图智能体。用户提供一张 iPhone 屏幕截图并要求规划下一步点击。模型在 UI 截图、用户工作流视频以及多图前后对比对上训练过，进而泛化到这个智能体用例。

这些都不是被训练过的任务；它们从课程的组合性结构中涌现出来。

### 视觉 token 池化

token 预算需要池化。OneVision 在二维 patch 网格上使用双线性插值（bilinear interpolation）：24x24 = 576 个 patch 变成 12x12 = 144（2 倍因子）或 8x8 = 64（3 倍因子）。池化是在 patch 网格空间中完成的，而非 token 空间，以保留局部性。

每个场景的池化因子选择本身就是一个超参数。池化越少 = token 越多 = 表征越丰富。池化越多 = token 越少 = 能塞进更多帧/图像。

### LLaVA-OneVision-1.5

2025 年的后续工作（LLaVA-OneVision-1.5，arXiv 2509.23661）在训练数据、模型权重与代码上「完全开放」。它在部分基准上追平了与专有模型的差距，并使这套配方平民化。课程相同，数据更多，基座 LLM 更好。架构无变化。

### 与 Qwen2.5-VL 的对比

Qwen2.5-VL（第 12.09 课）做了不同的选择。它使用 M-RoPE 与动态 FPS（dynamic FPS），而非固定池化。它的预算随输入伸缩——一段 1 分钟视频比一段 5 秒视频使用更多 token。LLaVA-OneVision 则固定预算、伸缩池化。两者都可行；它们是在可配置性与可预测性之间做权衡。

## 动手用起来

`code/main.py` 是一个面向 OneVision 风格 VLM 的课程与预算规划器。给定每样本的 token 预算和一个目标场景混合（比如 40% 单图、30% 多图、30% 视频），它会：

- 为每个场景分配分辨率、池化因子与帧数。
- 检查每个场景是否都能塞进共享预算之内。
- 报告预期 token 数量、LLM 的 FLOPs，以及哪些场景的 token 配给不足。
- 打印一份逐阶段的训练计划表。

用它来规划一次 OneVision 微调，或对一次 VLM 部署的每请求成本做合理性核查。

## 交付物

本课产出 `outputs/skill-onevision-budget-planner.md`。给定一个目标任务分布和每样本预算，它会输出 AnyRes 因子、每帧池化、视频帧数以及课程阶段权重。无论你何时训练或微调一个统一场景的 VLM，都可使用它。

## 练习

1. 你的产品支持 80% 单图、10% 多图（2-4 张）、10% 视频（8-16 帧）。设计这份 token 预算。从不做重度多图所节省下来的额外预算，你会放到哪里？

2. 阅读 LLaVA-OneVision 论文第 4.3 节（涌现能力）。提出第四种这套课程很可能解锁、但论文未报告的涌现技能。

3. 交换课程顺序——先训练多图，再单图，最后视频。预测哪些基准会退化，以及原因。

4. 论文报告的视频基准只在每样本 8 帧上训练。这能泛化到推理时的 30 秒视频吗？最先崩掉的是什么——token 预算还是时序推理？

5. 把 24x24 的 patch 双线性池化为 12x12，是每个维度 4 倍的缩减。用标准库 Python 实现这个池化，并验证每个 2x2 块上的均值是否与双线性输出匹配。

## 关键术语

| 术语 | 人们怎么说 | 它实际意味着什么 |
|------|-----------------|------------------------|
| OneVision 场景 | 「单图、多图或视频」 | 统一 VLM 处理的三种输入形状之一；预算在三者间保持恒定 |
| token 预算 | 「每样本多少 token」 | LLM 在每个训练/推理样本上看到的视觉 token 总量，通常 3000-4000 |
| 课程 | 「训练顺序」 | 为涌现迁移而选择的阶段排序（单图 → 多图 → 视频） |
| 双线性池化 | 「token 收缩」 | 对 patch 网格（二维）应用双线性插值，在保留局部性的同时减少 token 数量 |
| 涌现技能 | 「没训练过，照样能用」 | 由课程组合产生、在推理时无需匹配训练数据即出现的能力 |
| AnyRes-k | 「k 切片设置」 | k 个固定分辨率的子切片加一张缩略图，典型 k ∈ {4, 9} |
| 任务迁移 | 「跨场景泛化」 | 在单图上学到、经由共享主干网络应用到视频（反之亦然）的技能 |
