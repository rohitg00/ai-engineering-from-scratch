# 任意分辨率视觉：Patch-n'-Pack 与 NaFlex

> 真实图像不是224x224的方块。收据是9:16，图表是16:9，医学扫描可能是4096x4096，手机截图是9:19.5。2024年之前的VLM答案是——将所有内容调整为固定方块——这丢弃了使OCR、文档理解和高分辨率场景解析工作的信号。NaViT（Google, 2023）展示了可以通过块对角掩码（block-diagonal masking）将可变分辨率图块（patches）打包到单个Transformer批次中。Qwen2-VL的M-RoPE（2024）完全放弃了绝对位置表。LLaVA-NeXT的AnyRes将高分辨率图像平铺为基准图像加子图像。SigLIP 2的NaFlex变体（2025）现在是希望单个检查点服务于所有宽高比的开源VLM的默认编码器。本课程从头实现patch-n'-pack。

**类型：** 构建
**语言：** Python（标准库，图块打包器 + 块对角掩码）
**先修知识：** 阶段12 · 01（ViT图块），阶段12 · 05（LLaVA）
**时间：** ~120分钟

## 学习目标

- 将一批可变分辨率图像的图块打包成一个序列，并构建块对角注意力掩码。
- 针对给定任务在AnyRes平铺（LLaVA-NeXT）、NaFlex（SigLIP 2）和M-RoPE（Qwen2-VL）之间做出选择。
- 在不缩放的情况下计算OCR、图表和摄影的令牌预算。
- 说出正方形缩放的三种失败模式：文本压扁、内容被裁剪、填充令牌浪费。

## 问题

Transformer期望一个序列。一个批次是相同长度序列的堆叠。如果图像是224x224，每次得到196个图块令牌，不需要填充，任务完成。在224上训练，在224上推理，再也不用考虑分辨率。

但现实世界并不配合。文档是竖屏的（8.5x11英寸，约2:3）。图表截图是横屏的（16:9）。收据高而窄（1:3）。医学成像以2048x2048或更大尺寸提供。移动设备屏幕截图是1170x2532（0.46:1）。

2024年之前的三种选择及其失败原因：

1. 缩放到固定正方形（224x224或336x336）。压缩会扭曲文本和面部。下采样破坏图表标签和OCR内容。这是LLaVA-1.5之前的标准做法。
2. 裁剪到固定宽高比。你丢弃了大部分图像，而且选择裁剪位置本身就是一个视觉问题。
3. 填充到最长边。解决形变，但竖屏图像上50%以上的令牌浪费在填充上。所有填充令牌的二次注意力成本。

2024-2025年的答案：让Transformer以图像原始分辨率消费图块，并想办法将一个异构批次打包成一个序列而不浪费计算。

## 概念

### NaViT 与 patch-n'-pack

NaViT（Dehghani et al., 2023）是展示这种方法可大规模工作的论文。其思想是机械式的：

1. 对于批次中的每张图像，以选定的图块尺寸（比如14）计算其原始图块网格。
2. 将每张图像的图块展平为其自身的可变长度序列。
3. 将所有图像的图块连接成一个用于该批次的长序列。
4. 构建一个块对角注意力掩码，使得图像A的图块仅在图像A内进行注意力操作。
5. 携带每个图块的位置信息（2D RoPE或分数位置嵌入）。

一个包含三张图像（336x336, 576个令牌；224x224, 256个令牌；448x336, 768个令牌）的批次变成一个1600令牌的序列，带有一个1600x1600的块对角掩码。没有填充。没有浪费计算。Transformer处理任意宽高比。

NaViT还在训练期间引入了分数图块丢弃（fractional patch dropping）——在批次中随机丢弃50%的图块——这既正则化又加速了训练。SigLIP 2继承了这一点。

### AnyRes（LLaVA-NeXT）

LLaVA-NeXT的AnyRes是实用的替代方案。给定一个高分辨率图像和一个固定编码器（CLIP或SigLIP在336），平铺图像：

1. 从预定义的集合——(1x1), (1x2), (2x1), (1x3), (3x1), (2x2)等——中选择最适合图像宽高比的网格布局。
2. 将完整图像平铺到网格中；每个图块成为一个336x336的裁剪。
3. 同时生成一个缩略图：将整个图像缩放到336x336作为全局上下文令牌。
4. 通过冻结的336编码器编码每一个图块。连接图块令牌 + 缩略图令牌。

对于672x672图像，2x2网格加缩略图：4 * 576 + 576 = 2880个视觉令牌。昂贵但有效——LLM同时看到局部细节和全局上下文。

当编码器冻结且只支持一种分辨率时，AnyRes是首选路线。它会为大型图像爆炸令牌数量（1344x1344图像在4x4网格下为9216 + 576 ≈ 9800个令牌，几乎填满8k LLM上下文）。

### M-RoPE（Qwen2-VL）

Qwen2-VL引入了多模态旋转位置嵌入（Multimodal Rotary Position Embedding）。与NaViT的分数位置或AnyRes的平铺加缩略图不同，每个图块携带一个3D位置（时间、高度、宽度）。查询/键旋转处理任意H、W和时间长度。

M-RoPE无需重新训练即可提供原生动态分辨率。推理时，你输入任意HxW图像，图块嵌入器产生H/14 x W/14个令牌，每个令牌获得其（t=0, r=行, c=列）位置，RoPE以正确的频率旋转注意力，完成。Qwen2.5-VL和Qwen3-VL延续了这一设计。InternVL3的V2PE是同样的思想，但每个模态使用可变编码。

与AnyRes不同，M-RoPE以原始分辨率消耗O(H x W / P^2)个令牌——没有倍增的平铺开销。与NaViT不同，它仍然期望每次前向传播一张图像。跨分辨率批处理仍然需要基于patch-n'-pack。

### NaFlex（SigLIP 2）

NaFlex是SigLIP 2检查点的原生灵活模式。单个模型在推理时服务多个序列长度（256、729、1024个令牌）。内部，它在训练期间使用NaViT风格的patch-n'-pack和每个图块的绝对分数位置。卖点：一个检查点，在推理时根据任务选择令牌预算。

对于语义任务（分类、检索），256个令牌。对于OCR或图表理解，1024个令牌。无需重新训练。

### 打包掩码

块对角掩码是大多数实现会出错的地方。对于一个覆盖图像`i=0..B-1`（长度`n_i`）的总长度为`N_total`的打包序列，形状为`(N_total, N_total)`的掩码`M`：如果两个索引落在同一图像的块内则为1，否则为0。你可以从累计长度列表构建它：

```
offsets = [0, n_0, n_0+n_1, ..., N_total]
M[i, j] = 1 当且仅当存在b使得 offsets[b] <= i < offsets[b+1] 且 offsets[b] <= j < offsets[b+1]
```

在PyTorch中，这可以通过`torch.block_diag`或显式聚集一行代码完成。FlashAttention的可变长度路径（`cu_seqlens`）完全跳过掩码，直接使用累计长度张量在序列内进行注意力操作——对于典型批次，比密集掩码快约10倍。

### 令牌预算

根据任务选择策略：

- OCR / 文档：1024-4096个令牌。SigLIP 2 NaFlex在1024，或AnyRes 3x3加缩略图。
- 图表和UI：在384-448原生分辨率下729-1024个令牌。Qwen2.5-VL动态分辨率带最大像素上限。
- 自然照片：256-576个令牌足够。下游LLM能看到足够信息。在内容密度高的地方为令牌付费。
- 视频：空间池化后每帧64-128个令牌，2-8 FPS。课程12.17涵盖这一点。

2026年的生产规则：选择一个每任务最大像素上限，按原始宽高比编码直至该上限，打包批次，跳过填充。Qwen2.5-VL暴露了`min_pixels`和`max_pixels`用于精确控制这个旋钮。

## 使用它

`code/main.py`实现了对具有整数像素坐标的异构图像批次的patch-n'-pack。它：

- 接收一个（H, W）图像尺寸列表。
- 计算每张图像在图块尺寸14下的图块序列长度。
- 将它们打包成一个总长度为`sum(n_i)`的序列。
- 构建块对角注意力掩码（为清晰起见，使用密集形式）。
- 比较打包成本与正方形缩放和AnyRes平铺的成本。
- 为混合批次（收据、图表、截图、照片）打印令牌预算表。

运行它。得出的数字就是为什么每个2026年开源VLM都使用patch-n'-pack的原因。

## 交付它

本课程生成`outputs/skill-resolution-budget-planner.md`。给定一个混合宽高比的工作负载（OCR、图表、照片、视频帧）和一个总令牌预算，它选择正确的策略（NaFlex、AnyRes、M-RoPE或固定正方形），并输出每个请求的配置。当你在为产品调整VLM尺寸时使用此技能——它可以防止无声的10倍令牌爆炸，这种爆炸会破坏延迟预算。

## 练习

1. 一张收据是600x1500（1:2.5）。在图块尺寸14下，原生分辨率令牌数是多少？缩放到336后的正方形分辨率令牌数是多少？实际上哪种会损失更多OCR精度？

2. 为一个包含四张图像（长度分别为256、576、729、1024）的批次构建块对角掩码。验证注意力矩阵为2585x2585，并且恰好有`256^2 + 576^2 + 729^2 + 1024^2`个非零条目。

3. 对于一张1792x896的图像，图块尺寸14，比较：(a) 缩放到336的正方形然后编码，(b) AnyRes 2x1 + 缩略图，(c) 原生分辨率的M-RoPE。哪一种使用最少的令牌？哪一种保留最多细节？

4. 实现分数图块丢弃：给定一个打包序列，均匀随机丢弃50%的令牌，并相应地更新块对角掩码。测量掩码的稀疏度变化。

5. 阅读Qwen2-VL论文（arXiv:2409.12191）第3.2节。用两句话描述`min_pixels`和`max_pixels`控制什么，以及为什么两个边界都很重要。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| Patch-n'-pack | "NaViT-style packing" | 将来自不同图像的可变长度图块序列连接到一个批次维度 |
| 块对角掩码 | "Packing mask" | 注意力掩码，限制每个图像的图块只关注自身，而不是打包中的邻居 |
| AnyRes | "LLaVA-NeXT tiling" | 将高分辨率图像分割成固定尺寸图块的网格加上全局缩略图；用固定编码器编码每个图块 |
| NaFlex | "SigLIP 2 native-flex" | 单个SigLIP 2检查点，在推理时无需重新训练即可服务256/729/1024令牌预算 |
| M-RoPE | "Multimodal RoPE" | 3D旋转位置编码（时间、行、列），无需位置表即可处理任意H、W、T |
| cu_seqlens | "FlashAttention packing" | FlashAttention可变长度路径使用的累计长度张量，取代密集的块对角掩码 |
| min_pixels / max_pixels | "Resolution bounds" | Qwen2.5-VL的每请求旋钮，用于限制非常小或非常大的输入的令牌数量 |
| 视觉令牌预算 | "How many tokens per image" | 每张图像产生的图块令牌的大致数量；决定了LLM的提示预算和注意力成本 |
