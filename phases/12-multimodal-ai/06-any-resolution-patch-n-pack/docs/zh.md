# 06 · 任意分辨率视觉：Patch-n'-Pack 与 NaFlex

> 真实图像不是 224x224 的正方形。一张收据是 9:16，一张图表是 16:9，一张医学扫描可能是 4096x4096，一张手机截图是 9:19.5。2024 年之前的视觉语言模型（VLM）方案——把一切都缩放成固定正方形——丢掉了让 OCR、文档理解和高分辨率场景解析得以成立的信号。NaViT（Google，2023）证明了你可以把可变分辨率的图块（patch）打包进单个 transformer 批次，再配合块对角（block-diagonal）掩码。Qwen2-VL 的 M-RoPE（2024）干脆彻底丢弃了绝对位置表。LLaVA-NeXT 的 AnyRes 把高分辨率图像切成「基础图 + 子图」的瓦片。SigLIP 2 的 NaFlex 变体（2025）如今已成为那些希望用单一检查点服务所有宽高比的开源 VLM 的默认编码器。本课将端到端地实现 patch-n'-pack。

**类型：** 构建
**语言：** Python（标准库，图块打包器 + 块对角掩码）
**前置：** 第 12 阶段 · 01（ViT 图块），第 12 阶段 · 05（LLaVA）
**时长：** 约 120 分钟

## 学习目标

- 把一批可变分辨率图像的图块打包进一个序列，并构建块对角注意力掩码。
- 针对给定任务，在 AnyRes 瓦片化（LLaVA-NeXT）、NaFlex（SigLIP 2）和 M-RoPE（Qwen2-VL）之间做出选择。
- 在不缩放的前提下，为 OCR、图表和摄影图像计算 token 预算。
- 说出正方形缩放的三种失败模式：文字被挤压、内容被裁切、token 浪费在填充上。

## 问题所在

Transformer 期望一个序列。一个批次是一摞等长的序列。如果你的图像都是 224x224，那么每次都能得到 196 个图块 token，无需填充，搞定。在 224 上训练，在 224 上推理，从此再也不用考虑分辨率。

但现实世界并不配合。文档是竖版（8.5x11 英寸，约 2:3）。图表截图是横版（16:9）。收据又高又窄（1:3）。医学影像出厂就是 2048x2048 或更大。移动设备截图是 1170x2532（0.46:1）。

2024 年之前的三种选择，以及它们各自失败的原因：

1. 缩放到固定正方形（224x224 或 336x336）。挤压会扭曲文字和人脸。降采样会破坏图表标签和 OCR 内容。这是 LLaVA-1.5 之前的标准做法。
2. 裁切到固定宽高比。你会丢掉图像的大部分内容，而且决定裁切位置本身就是一个视觉难题。
3. 沿最长边填充。修复了扭曲问题，但对竖版图像而言会有 50% 以上的 token 浪费在填充上。所有这些填充 token 还带来二次方级的注意力开销。

2024-2025 年的答案是：让 transformer 以图像的原生分辨率吞入图块，并设法把一个异构批次打包进一个序列而不浪费算力。

## 核心概念

### NaViT 与 patch-n'-pack

NaViT（Dehghani 等，2023）是证明这套方法可在规模上奏效的论文。其思路是机械式的：

1. 对批次中的每张图像，按选定的图块尺寸（比如 14）计算其原生图块网格。
2. 把每张图像的图块展平成各自的可变长度序列。
3. 把该批次所有图像的图块拼接成一条长序列。
4. 构建一个块对角注意力掩码，使图像 A 的图块只在图像 A 内部做注意力。
5. 携带每个图块的位置信息（2D RoPE 或分数位置嵌入）。

一个由三张图像组成的批次——336x336（576 token）、224x224（256 token）和 448x336（768 token）——会变成一条 1600 token 的序列，配一个 1600x1600 的块对角掩码。没有填充，没有浪费算力。Transformer 可以处理任意宽高比。

NaViT 还引入了训练期的分数图块丢弃（fractional patch dropping）——在整个批次上随机丢弃 50% 的图块——它既起到正则化作用又加速了训练。SigLIP 2 继承了这一点。

### AnyRes（LLaVA-NeXT）

LLaVA-NeXT 的 AnyRes 是务实的替代方案。给定一张高分辨率图像和一个固定的编码器（CLIP 或 SigLIP，分辨率 336），对图像进行瓦片化：

1. 从一组预定义布局——(1x1)、(1x2)、(2x1)、(1x3)、(3x1)、(2x2) 等——中挑选最贴合图像宽高比的网格布局。
2. 把整张图像切成该网格；每个瓦片成为一个 336x336 的裁剪块。
3. 同时生成一张缩略图：把整张图像缩放到 336x336，作为全局上下文 token。
4. 让每个瓦片都通过冻结的 336 编码器。拼接瓦片 token + 缩略图 token。

对于一张 672x672 的图像，采用 2x2 网格加缩略图：4 * 576 + 576 = 2880 个视觉 token。代价高但有效——大语言模型同时看到了局部细节与全局上下文。

当你的编码器是冻结的、只支持单一分辨率时，AnyRes 是首选路线。它会让大图的 token 数量爆炸（一张 1344x1344 的图像在 4x4 网格下是 9216 + 576 ≈ 9800 个 token，几乎填满一个 8k 的大模型上下文）。

### M-RoPE（Qwen2-VL）

Qwen2-VL 引入了多模态旋转位置嵌入（Multimodal Rotary Position Embedding）。它不像 NaViT 那样用分数位置，也不像 AnyRes 那样用瓦片加缩略图，而是让每个图块携带一个 3D 位置（时间、高度、宽度）。查询/键的旋转处理任意的 H、W 和时间长度。

M-RoPE 无需重新训练即可原生支持动态分辨率。推理时你喂入任意 HxW 的图像，图块嵌入器产出 H/14 x W/14 个 token，每个 token 获得其 (t=0, r=行, c=列) 位置，RoPE 以正确的频率旋转注意力，搞定。Qwen2.5-VL 和 Qwen3-VL 延续了这一做法。InternVL3 的 V2PE 是同样的思路，按模态做可变编码。

与 AnyRes 不同，M-RoPE 在原生分辨率下是 O(H x W / P^2) 个 token——没有瓦片化带来的乘法级开销。与 NaViT 不同，它仍然要求每次前向只处理单张图像。跨分辨率的批处理仍需在其之上叠加 patch-n'-pack。

### NaFlex（SigLIP 2）

NaFlex 是 SigLIP 2 检查点的原生弹性（native-flex）模式。单个模型在推理时服务多种序列长度（256、729、1024 token）。其内部在训练期使用 NaViT 风格的 patch-n'-pack，并为每个图块使用绝对分数位置。卖点在于：一个检查点，推理时根据任务挑选你的 token 预算。

对于语义任务（分类、检索），用 256 token。对于 OCR 或图表理解，用 1024 token。无需重新训练。

### 打包掩码

块对角掩码是大多数实现栽跟头的地方。对于一条总长为 `N_total`、覆盖图像 `i=0..B-1`（各自长度为 `n_i`）的打包序列，形状为 `(N_total, N_total)` 的掩码 `M`，当两个索引落在同一张图像的块内时取 1，否则取 0。你可以从一个累积长度列表来构建它：

```
offsets = [0, n_0, n_0+n_1, ..., N_total]
M[i, j] = 1 iff there exists b where offsets[b] <= i < offsets[b+1] and offsets[b] <= j < offsets[b+1]
```

在 PyTorch 中这只需一行 `torch.block_diag` 或一次显式的 gather。FlashAttention 的可变长度路径（`cu_seqlens`）完全跳过掩码，直接用累积长度张量在各序列内部做注意力——对于典型批次，比稠密掩码快约 10 倍。

### Token 预算

按任务挑选你的策略：

- OCR / 文档：1024-4096 token。SigLIP 2 NaFlex 设为 1024，或 AnyRes 3x3 + 缩略图。
- 图表与 UI：729-1024 token，原生分辨率 384-448。Qwen2.5-VL 动态分辨率，配上最大像素上限。
- 自然照片：256-576 token 就够了。下游大模型看到的信息足够。把 token 花在内容密度高的地方。
- 视频：空间池化后每帧 64-128 token，2-8 FPS。第 12.17 课会讲到这个。

2026 年的生产法则：为每个任务挑选一个最大像素上限，在该上限内以原生宽高比编码，打包批次，跳过填充。Qwen2.5-VL 暴露了 `min_pixels` 和 `max_pixels` 正是为了这个旋钮。

## 上手实践

`code/main.py` 用整数像素坐标为一批异构图像实现了 patch-n'-pack。它会：

- 接收一个 (H, W) 图像尺寸列表。
- 在图块尺寸 14 下计算每张图像的图块序列长度。
- 把它们打包进一条总长为 `sum(n_i)` 的序列。
- 构建块对角注意力掩码（稠密形式，便于说明）。
- 比较打包成本与正方形缩放、AnyRes 瓦片化的成本。
- 为一个混合批次（收据、图表、截图、照片）打印一张 token 预算表。

运行它。最终蹦出来的数字，正是每个 2026 年的开源 VLM 都使用 patch-n'-pack 的原因。

## 练习

1. 一张收据是 600x1500（1:2.5）。在图块尺寸 14 下，有多少个原生分辨率 token？正方形缩放到 336 之后又有多少个？实践中哪一种丢失的 OCR 精度更多？

2. 为一个由四张图像组成、长度分别为 256、576、729、1024 的批次构建块对角掩码。验证注意力矩阵是 2585x2585，且恰好有 `256^2 + 576^2 + 729^2 + 1024^2` 个非零项。

3. 对于一张 1792x896 的图像，在图块尺寸 14 下比较：(a) 正方形缩放到 336 再编码，(b) AnyRes 2x1 + 缩略图，(c) M-RoPE 原生分辨率。哪个用的 token 最少？哪个保留的细节最多？

4. 实现分数图块丢弃：给定一条打包序列，均匀随机丢弃 50% 的 token，并相应更新块对角掩码。测量掩码稀疏度的变化。

5. 阅读 Qwen2-VL 论文（arXiv:2409.12191）的 3.2 节。用两句话描述 `min_pixels` 和 `max_pixels` 控制什么，以及为什么这两个边界都重要。

## 关键术语

| 术语 | 人们怎么说 | 它实际是什么意思 |
|------|-----------------|------------------------|
| Patch-n'-pack | 「NaViT 风格的打包」 | 把来自不同图像的可变长度图块序列拼接到一个批次维度上 |
| 块对角掩码（Block-diagonal mask） | 「打包掩码」 | 一种注意力掩码，把每张图像的图块约束为只关注自己，而不关注打包中相邻图像的图块 |
| AnyRes | 「LLaVA-NeXT 瓦片化」 | 把一张高分辨率图像切成固定尺寸瓦片的网格再加一张全局缩略图；用固定编码器对每个瓦片编码 |
| NaFlex | 「SigLIP 2 原生弹性」 | 单个 SigLIP 2 检查点，推理时无需重新训练即可服务 256/729/1024 token 的预算 |
| M-RoPE | 「多模态 RoPE」 | 3D 旋转位置编码（时间、行、列），无需位置表即可处理任意的 H、W、T |
| cu_seqlens | 「FlashAttention 打包」 | FlashAttention 可变长度路径所用的累积长度张量，取代稠密块对角掩码 |
| min_pixels / max_pixels | 「分辨率边界」 | Qwen2.5-VL 的按请求旋钮，对极小或极大输入的 token 数量设上下限 |
| 视觉 token 预算 | 「每张图像多少 token」 | 每张图像产出的图块 token 的粗略计数；它决定大模型的提示词预算和注意力开销 |
