# 多模态 RAG 与跨模态检索

> 视觉原生文档 RAG 是一个切片。生产多模态 RAG 范围更广——跨文本、图像、音频和视频检索，用于旅行规划（"给我找个安静、自然光充足的素食早午餐"）、医疗分诊（"什么伤与这张照片 + 这些笔记匹配"）、电子商务（"与我自拍相似、尺码合适的服装"）和现场服务（"诊断这个引擎声音加上零件照片"）等工作流。三篇 2025 年综述——Abootorabi 等人、Mei 等人、Zhao 等人——将子问题编纂为：跨模态检索、检索融合、生成接地、多模态评估。本课阅读综述并设计生产流水线。

**类型：** Build
**语言：** Python（stdlib，带融合 + 接地生成器的跨模态检索器）
**前置知识：** Phase 12 · 23（ColPali），Phase 11（RAG 基础）
**时间：** ~180 分钟

## 学习目标

- 设计跨模态检索：文本 → 图像、图像 → 文本、音频 → 视频等。
- 比较三种融合策略：分数融合、基于注意力的融合、MoE 融合。
- 解释生成接地：当来源是混合模态时，"引用你的来源"是什么样子。
- 命名 2025 年三篇经典多模态 RAG 综述及其子问题分类法。

## 问题所在

单模态 RAG 是一个已解决的模式：嵌入查询，嵌入块，检索，塞入 LLM。多模态 RAG 需要：

1. 多个检索头（每种模态需要兼容空间中的嵌入）。
2. 跨模态检索结果的融合。
3. 跨模态引用来源的生成接地。
4. 覆盖跨模态信号的评估指标。

2025 年综述都得出相同的分类法。

## 核心概念

### 跨模态检索

给定模态 A 的查询检索模态 B 的文档。三种模式：

1. 共享嵌入空间。CLIP 和 CLAP 在共享空间中产生文本 + 图像 / 文本 + 音频嵌入。跨模态余弦相似度直接有效。限于 CLIP 训练的对。

2. 每模态编码器 + 翻译。文本编码器 + 图像编码器 + 映射空间之间的小型翻译模块。Gupta 等人的 Sen2Sen 和其他 2024 年设计。灵活但增加复杂度。

3. VLM 作为编码器。使用 VLM 的隐藏状态作为检索表示。VLM 支持的任何模态都有效。更高质量，更昂贵。

选择：文本+图像用 CLIP / SigLIP 2；文本+音频用 CLAP；前沿质量跨模态用 VLM 隐藏状态。

### 融合策略

你检索了 10 个结果：5 张图像、3 段文本、2 个音频片段。如何合并？

分数融合（最便宜）。每种模态有自己的检索器，每个返回分数。模态内归一化分数然后求和。简单，通常有效。

基于注意力的融合。连接所有检索项，让小型注意力网络加权它们。需要训练。

MoE 融合。门控网络路由到模态特定专家。不同查询类型不同路由——视觉问题权重图像更高。

生产默认：分数融合，对查询主导模态略有偏置。如果 A/B 显示领域明显获胜，升级到 MoE。

### 生成接地

LLM 应该引用哪个检索项驱动了每个声明。对于多模态：

- 文本来源：标准引用 `[1]`。
- 图像来源：`[img 3]` 带简短标题。
- 音频：`[audio 2 at 0:34]`。

用接地感知数据训练生成器：训练目标中的每个声明都标记来源索引。推理时，模型自然发出引用。

### 2025 年综述

Abootorabi 等人（arXiv:2502.08826，"Ask in Any Modality"）：多模态 RAG 分类法。涵盖检索、融合、生成。最广泛覆盖。

Mei 等人（arXiv:2504.08748，"A Survey of Multimodal RAG"）：专注于子任务基准和失败模式。对评估设计有用。

Zhao 等人（arXiv:2503.18016）：视觉聚焦综述。ColPali 家族工作强。

阅读全部三篇可了解 2025 年春季最新技术。大多数子问题仍然开放。

### MuRAG——基础论文

MuRAG（Chen 等人，2022）是第一篇多模态 RAG。从多模态知识库检索图像 + 文本，生成答案。在 VLM 浪潮之前展示可行性。现代系统（REACT、VisRAG、M3DocRAG）建立在其上。

### 生产旅行规划器示例

查询："给我找个安静、自然光充足的素食早午餐。"

流水线：

1. 分解查询。"安静" → 音频/评论关键词；"素食早午餐" → 菜单项；"自然光" → 图像特征。
2. 每模态检索：
   - 评论文本检索："素食早午餐，安静氛围。"
   - 餐厅照片图像检索："自然光，通风。"
   - 环境声音片段音频检索："低分贝，无音乐。"
3. 融合分数。每家餐厅有综合分数。
4. Top-k 餐厅 → 带所有证据的 VLM 生成器 → 带引用的答案。

这远超文本 RAG。每种模态添加文本单独遗漏的信号。

### 代理多模态 RAG

多跳：如果首次检索未返回高置信度答案，LLM 重新表述并再次检索。Phase 14 的代理 RAG 模式适用。示例：

- 检索初始 top-10 → LLM 问"太吵，过滤 <40 分贝" → 重新检索。
- 检索图像 → LLM 看到一张有菜单 → 检索菜单文本 → 答案。

增加复杂度但处理单次检索无法处理的查询。

### 评估

跨模态评估仍然不成熟。常见代理：

- 每模态 Recall@k。
- 融合 top-k 精度。
- 人工判断端到端满意度。
- 任务特定（完成的预订、购买的商品）。

无标准基准跨越所有模态。大多数论文在领域特定任务上评估。

## 使用它

`code/main.py`：

- 三个模拟检索器（文本、图像、音频）在共享餐厅语料库上操作。
- 以可配置权重组合模态分数的分数融合。
- 发出带引用最终答案的生成器存根。
- 如果置信度低则重新表述查询的简单代理循环。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------------|------------------------|
| 跨模态检索 | "查询一种模态，检索另一种" | 文本查询检索图像；图像查询检索文本；需要共享空间或翻译器 |
| 分数融合 | "组合分数" | 每模态检索分数的加权和；最简单融合 |
| MoE 融合 | "模态路由专家" | 门控网络选择每查询信任哪种模态的分数 |
| 接地生成 | "引用你的来源" | 答案中每个声明标记来源索引 |
| MuRAG | "第一篇多模态 RAG" | 2022 年论文确立了多模态 RAG 模式 |
| 代理多跳 | "重新表述并重试" | 首次通过置信度低时 LLM 重新查询检索器 |
