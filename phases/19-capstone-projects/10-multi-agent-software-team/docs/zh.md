# 顶点项目 10 —— 多智能体软件工程团队

> SWE-AF 的工厂架构、MetaGPT 的基于角色提示、AutoGen 0.4 的强类型 Actor 图、Cognition 的 Devin 以及 Factory 的 Droids 都在 2026 年汇聚成了相同的形态：架构师负责规划，N 个编码器在并行工作树中工作，审查员负责把关，测试员负责验证。并行工作树将挂钟时间转化为吞吐量。共享状态和交接协议成为故障面。顶点项目是构建团队，在 SWE-bench Pro 上进行评估，并报告哪些交接会中断以及频率如何。

**类型：** 顶点项目
**语言：** Python / TypeScript（智能体）、Shell（工作树脚本）
**先决条件：** Phase 11（LLM 工程）、Phase 13（工具）、Phase 14（智能体）、Phase 15（自主系统）、Phase 16（多智能体）、Phase 17（基础设施）
**涉及阶段：** P11 · P13 · P14 · P15 · P16 · P17
**时间：** 40 小时

## 问题

单智能体编码工具在大型任务上会遇到瓶颈。不是因为任何单个智能体弱，而是因为 20 万 token 的上下文无法同时容纳架构计划、四个并行代码库切片、审查员评论和测试输出。多智能体工厂将问题拆分：架构师拥有计划，编码器在并行工作树中拥有实现，审查员把关，测试员验证。SWE-AF 的"工厂"架构、MetaGPT 的角色、AutoGen 的强类型 Actor 图——这三种框架描述的都是相同的形态。

故障面在于交接。架构师制定的计划编码器无法实现。编码器产生冲突的差异。审查员批准了幻觉修复。测试员与仍在编写的编码器竞速。你将构建这样一个团队，在 50 个 SWE-bench Pro 问题上运行它，跟踪每次交接，并发布事后分析。

## 概念

角色是强类型智能体。**架构师**（Claude Opus 4.7）阅读问题，编写计划，并将其分解为具有显式接口的子任务。**编码器**（Claude Sonnet 4.7，N 个并行实例，每个实例位于一个 `git worktree` + Daytona 沙盒中）独立实现子任务。**审查员**（GPT-5.4）阅读合并后的差异，要么批准，要么请求特定更改。**测试员**（Gemini 2.5 Pro）在隔离环境中运行测试套件，并报告通过/失败及产物。

通信通过共享任务板（基于文件或 Redis）进行。每个角色消费其被允许处理的任务。交接是 A2A 协议类型的消息。协调关注点：合并冲突解决（协调员角色或自动三路合并）、共享状态同步（编码器开始后计划冻结；重新计划是单独的事件），以及审查员把关（审查员不能批准自己的更改或自己提议的更改）。

Token 放大是隐藏成本。每个角色边界都会增加摘要提示和交接上下文。一个 40 轮的单智能体运行变成四个角色共 160 轮。评分标准特别权衡与单智能体基线的 token 效率，因为问题不是"多智能体是否有效"，而是"它每美元是否更划算"。

## 技术栈

- 编排：LangGraph，带共享状态 + 每个智能体的子图
- 消息传递：A2A 协议（Google 2025），用于强类型智能体间消息
- 模型：Opus 4.7（架构师）、Sonnet 4.7（编码器）、GPT-5.4（审查员）、Gemini 2.5 Pro（测试员）
- 工作树隔离：每个编码器使用 `git worktree add` + Daytona 沙盒
- 合并协调员：自定义三路合并 + LLM 介导的冲突解决
- 评估：SWE-bench Pro（50 个问题）、SWE-AF 场景、HumanEval++ 用于单元测试
- 可观察性：Langfuse，带角色标记的跨度，每个智能体的 token 核算
- 部署：K8s，每个角色作为单独的 Deployment + 基于积压的 HPA

## 构建它

1. **任务板。** 基于文件的 JSONL，包含强类型消息：`plan_request`、`subtask`、`diff_ready`、`review_needed`、`test_needed`、`approved`、`rejected`、`replan_needed`。智能体订阅标签。

2. **架构师。** 阅读 GitHub 问题，使用要求显式子任务接口（触及的文件、公共函数、测试影响）的计划模板运行 Opus 4.7。发出一个包含子任务 DAG 的 `plan_request`。

3. **编码器。** N 个并行工作者，每个从板上认领一个子任务。每个生成一个新的 `git worktree add` 分支加上一个 Daytona 沙盒。实现子任务。发出 `diff_ready`，包含补丁 + 测试差异。

4. **合并协调员。** 所有编码器完成后，将 N 个分支三路合并到暂存分支。仅当存在文件级重叠时，才使用 LLM 介导的冲突解决。

5. **审查员。** GPT-5.4 阅读合并后的差异。不能批准自己撰写的差异。发出 `approved`（无操作）或 `review_feedback`，包含特定更改请求，路由回相关编码器。

6. **测试员。** Gemini 2.5 Pro 在干净的沙盒中运行测试套件。捕获产物。发出 `test_passed` 或 `test_failed`，包含堆栈跟踪。失败的测试循环回拥有失败子任务的编码器。

7. **交接核算。** 跨越角色边界的每条消息在 Langfuse 中获得一个跨度，包含负载大小和使用的模型。计算每个子任务的 token 放大率（编码器_token + 审查员_token + 测试员_token + 架构师份额 / 编码器_token）。

8. **评估。** 在 50 个 SWE-bench Pro 问题上运行。将 pass@1 和每个已解决问题的美元成本与单智能体基线（单个工作树中的一个 Sonnet 4.7）进行比较。

9. **事后分析。** 对于每个失败的问题，识别中断的交接（计划太模糊、合并冲突、审查员错误批准、测试员抖动）。生成交接失败直方图。

## 交付它

`outputs/skill-multi-agent-team.md` 是可交付成果。给定问题 URL 和并行级别，团队生成一个可合并的 PR，包含每个角色的 token 核算。

| 权重 | 标准 | 测量方式 |
|:-:|---|---|
| 25 | SWE-bench Pro pass@1 | 匹配的 50 问题子集，pass@1 |
| 20 | 并行加速 | 与单智能体基线相比的挂钟时间 |
| 20 | 审查质量 | 注入错误探测上的错误批准率 |
| 20 | Token 效率 | 每个已解决问题的总 token 与单智能体相比 |
| 15 | 协调工程 | 合并冲突解决、交接失败直方图 |
| **100** | | |

## 练习

1. 在运行中期向差异注入一个明显的错误（主体前额外的 `return None`）。测量审查员的错误批准率。调整审查员提示，直到错误批准率低于 5%。

2. 减少到两个编码器（架构师 + 编码器 + 审查员 + 测试员，编码器顺序运行两个子任务）。比较挂钟时间和通过率。

3. 将合并协调员替换为单写入器约束（子任务触及不相交的文件集）。测量架构师上的规划负担。

4. 将审查员从 GPT-5.4 换成 Claude Opus 4.7。测量错误批准率和 token 成本差异。

5. 添加第五个角色：文档编写员（Haiku 4.5）。审查后，它生成变更日志条目。测量文档质量是否证明额外的 token 支出合理。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 并行工作树 | "隔离分支" | `git worktree add` 为每个编码器生成一个新的工作树 |
| 任务板 | "共享消息总线" | 智能体订阅的强类型消息的文件或 Redis 存储 |
| 交接 | "角色边界" | 从一个角色的上下文跨越到另一个角色的任何消息 |
| Token 放大 | "多智能体开销" | 跨角色的总 token / 相同任务的单智能体 token |
| A2A 协议 | "智能体到智能体" | Google 2025 年强类型智能体间消息规范 |
| 合并协调员 | "集成器" | 运行三路合并并介导冲突的组件 |
| 错误批准 | "审查员幻觉" | 审查员批准包含已知错误的差异 |
