# Линейная алгебра — интуиция

> Каждая AI-модель — это просто матричная математика в красивой шляпе.

Эта лекция — введение в линейную алгебру через её главные образы:
вектор, операции над векторами, метрики сходства. От самого простого
определения «список чисел» до того, как одна формула косинусного сходства
лежит в основе рекомендательных систем, семантического поиска и attention
в трансформерах.

Цель — **геометрическая интуиция**, а не строгие доказательства. После
прохождения должна остаться картинка в голове: что́ означает каждая
операция, и почему именно она встречается в AI.

## Зачем это вообще нужно

Откройте любую современную статью по машинному обучению — на первой странице
будут векторы, матрицы, скалярные произведения, преобразования. Без интуиции
в линейной алгебре это просто символы. С интуицией — становится видно, что́
на самом деле делает нейронная сеть: она двигает точки в многомерном
пространстве.

Базовые применения векторных операций в AI:

- **Embeddings** — каждое слово, изображение, документ превращаются в
  длинный вектор чисел. Семантическая близость объектов становится
  геометрической близостью векторов.
- **Attention** в трансформерах — вычисление «насколько каждое слово
  важно для предсказания следующего» через массивы скалярных произведений.
- **RAG (поиск-дополненная генерация)** — поисковый компонент работает
  через скалярные произведения между вектором запроса и векторами
  документов в базе.
- **Градиентный спуск** — каждый шаг обучения это вычитание векторов
  и умножение их на скаляр.
- **Рекомендательные системы, кластеризация клиентов, сегментация** —
  всё через метрики сходства между векторами.

То есть линейная алгебра — это **общий язык** всей современной AI. Без
её понимания дальше двигаться можно, но каждое следующее явление будет
казаться магией.

## Часть I. Вектор

### Что такое вектор

**Вектор — это просто список чисел.** Например, `[3, 2]`. Это всё. Никакой
магии.

Интересное начинается с того, **что эти числа означают**. У вектора есть
два эквивалентных прочтения:

1. **Как точка в пространстве.** Для `[3, 2]` это точка с координатами
   (x=3, y=2) на плоскости.
2. **Как стрелка.** Та же точка, но представленная стрелкой из начала
   координат `(0, 0)` до этой точки. У стрелки есть **направление**
   (куда смотрит) и **длина**.

Оба прочтения описывают один и тот же объект. Геометры чаще говорят
«стрелка», программисты — «точка», физики — «вектор величины». Это вопрос
акцента, не сущности.

```
y
↑
3 |
2 |       • [3, 2]
1 |      ╱
0 +────────→ x
  0  1  2  3
```

В Python вектор естественно представляется списком чисел:

```python
v = [3, 2]
```

В научных вычислениях используют `numpy.array`, в боевом ML — `torch.Tensor`,
но идея та же.

### Размерность вектора

**Размерность вектора = количество чисел в нём.**

```
[3, 2]              → двумерный (2D) вектор
[1, -2, 3, 0, 7]    → пятимерный (5D) вектор
[a₁, a₂, ..., a₇₆₈] → 768-мерный
```

В Python размерность считается просто:

```python
def dimension(v):
    return len(v)
```

Что меняется при росте размерности:

- **2D** — рисуется на плоскости (лист в клеточку), видим глазами.
- **3D** — представляем как куб, тоже видим, чуть труднее.
- **4D и выше** — **нарисовать невозможно.** Человеческий мозг не
  приспособлен. Но математика работает абсолютно так же — все формулы
  обобщаются на любую размерность.

Это **главный сдвиг в голове**: геометрическая интуиция остаётся, даже
когда наглядная картинка пропадает. В современной AI оперируют векторами
размерности 768, 1024, 4096 — никто их не рисует, но **думают** про них
всё ещё как про точки и стрелки в пространстве.

**Откуда такие большие размерности:**

- Word embeddings BERT — 768 чисел на одно слово.
- OpenAI `text-embedding-3-large` — 3072 числа на кусок текста.
- LLaMA, GPT внутри держат активации размерности 4096 и больше.

**Бизнес-аналогия.** В CRM-системе клиент описывается несколькими
признаками: количество заказов, средний чек, NPS, давность последней
покупки, возраст. Если признаков три — клиент это вектор в 3D. Если
семь — в 7D. ML просто доводит эту мысль до предельного состояния:
«опишем объект 768 числами, пусть модель сама разберётся, что они
значат».

## Часть II. Операции над векторами

### Сложение и вычитание

**Главное правило:** сложение и вычитание определены **только для векторов
одинаковой размерности.** Эти операции **не существуют** для векторов
разных размерностей. Никакого «дополним нулями и сложим» — это совсем
другая операция, которая меняет смысл данных.

**Сложение покоординатно:**

```
[a₁, a₂] + [b₁, b₂] = [a₁+b₁, a₂+b₂]
```

Пример: `[3, 2] + [1, 4] = [4, 6]`.

**Геометрический смысл — «хвост к голове».** Берём первую стрелку из (0,0)
до (3,2). Ставим хвост второй стрелки в голову первой. Куда дотянется
вторая стрелка — там и сумма:

```
y
↑
6 |              • [4, 6] ← сумма
5 |            ╱
4 |    [1,4]╱
3 |        ╱
2 |   •  ↗
1 | ╱ [3,2]
0 +────────────→ x
  0 1 2 3 4
```

«Шаг A потом шаг B — куда придёшь».

**Вычитание** — то же самое покоординатно:

```
[a₁, a₂] - [b₁, b₂] = [a₁-b₁, a₂-b₂]
```

Геометрически — стрелка «от B до A». Если стоять в точке B, то вектор
`a - b` показывает направление до A. На этом стоят все алгоритмы,
которые работают с «разностью» — от движения по градиенту до сравнения
профайлов клиентов.

**В коде:**

```python
def add(a, b):
    if len(a) != len(b):
        raise ValueError(f"размерности не совпадают: {len(a)} vs {len(b)}")
    return [a_i + b_i for a_i, b_i in zip(a, b)]


def sub(a, b):
    if len(a) != len(b):
        raise ValueError(f"размерности не совпадают: {len(a)} vs {len(b)}")
    return [a_i - b_i for a_i, b_i in zip(a, b)]
```

Проверка размерностей через `raise ValueError` — обязательна.
Без неё `zip` молча обрежет до короткого списка, и функция вернёт
неполный, но «правдоподобный» результат. Это худший вид бага —
функция «работает», но данные испорчены без шума.

### Умножение на скаляр

**Скаляр — это просто число**, в отличие от вектора. Умножение вектора
на скаляр умножает каждую его координату:

```
k · [a₁, a₂] = [k·a₁, k·a₂]
```

**Геометрически:**

- `k > 0` — растягивание или сжатие стрелки. `k=2` → длина становится в
  2 раза больше, направление сохраняется. `k=0.5` → длина в 2 раза
  меньше.
- `k < 0` — разворот в противоположную сторону + масштабирование.
  `k=-1` → та же стрелка, но «назад».
- `k = 0` — нулевой вектор.

**В коде:**

```python
def scale(v, k):
    return [elem * k for elem in v]
```

Здесь проверка размерностей не нужна — вектор один, число одно.

### Скалярное произведение (dot product)

**Самая важная операция во всей линейной алгебре с точки зрения AI.**
На ней стоят:

- Attention в трансформерах.
- Поиск похожих документов в RAG.
- Косинусное сходство в рекомендательных системах и кластеризации.
- Каждый линейный слой нейронной сети — это серия скалярных
  произведений.

**Формула** (векторы одинаковой размерности):

```
a · b = a₁·b₁ + a₂·b₂ + ... + aₙ·bₙ
```

Перемножаем покоординатно, складываем результаты в **одно число**.

Пример: `[1, 2, 3] · [4, 5, 6] = 1·4 + 2·5 + 3·6 = 32`.

**Очень важно:** в отличие от сложения, результат — **одно число
(скаляр)**, не вектор. Отсюда английское название — *scalar product*.

**В коде:**

```python
def dot(a, b):
    if len(a) != len(b):
        raise ValueError(f"размерности не совпадают: {len(a)} vs {len(b)}")
    return sum(a_i * b_i for a_i, b_i in zip(a, b))
```

Внутри `sum` использован **generator expression** (без квадратных
скобок) — он не материализует промежуточный список в памяти, считает
сумму на лету. Для миллиона чисел это критично, для пяти — без разницы,
но идиома «sum от генератора» — каноническая.

### Геометрический смысл скалярного произведения

**Скалярное произведение измеряет, насколько векторы «смотрят в одну
сторону».** Знак результата говорит:

| Результат | Угол | Смысл |
|---|---|---|
| `a · b > 0` | < 90° | смотрят в одну сторону |
| `a · b = 0` | = 90° | перпендикулярны, независимы |
| `a · b < 0` | > 90° | смотрят в разные стороны |

Точная связь:

```
a · b = |a| · |b| · cos(θ)
```

где `θ` — угол между векторами, `|a|`, `|b|` — их длины. Из этой формулы
видно: результат зависит **и от направления (cos)**, **и от длин**.
Когда оба вектора длинные, любое скалярное произведение даёт большое
число — даже если направления совпадают плохо. Это создаёт проблему при
использовании скалярного произведения как «меры похожести» — см.
Часть IV.

## Часть III. Длина вектора

### Что такое длина

**Длина вектора (англ. magnitude, norm)** — насколько «далеко» он
тянется от начала координат.

```
|v| = √(v₁² + v₂² + ... + vₙ²)
```

Каждое число в квадрат, складываем, берём корень.

В 2D это **теорема Пифагора**: длина вектора — это гипотенуза
прямоугольного треугольника с катетами в виде его координат.

Пример: `|[3, 4]| = √(9 + 16) = √25 = 5`.

**Имена термина** (встретятся все три):

- **magnitude** — обиходный английский.
- **length** — «длина».
- **norm**, **L2 norm**, **Euclidean norm** — математически. «L2» —
  это конкретно «корень из суммы квадратов». «Euclidean» — потому что
  соответствует обычной геометрической длине из геометрии Евклида.

### Связь длины со скалярным произведением

**Важная связка:** длина выражается через скалярное произведение
вектора с самим собой.

```
v · v = v₁·v₁ + v₂·v₂ + ... = v₁² + v₂² + ... = |v|²
```

Значит:

```
|v| = √(v · v)
```

**В коде:**

```python
import math

def magnitude(v):
    return math.sqrt(dot(v, v))
```

Это и есть та реализация, что используется внутри `numpy.linalg.norm`.

Связка важна не только для краткости кода, но и **концептуально**: она
показывает, что длина — это всего лишь специальный случай скалярного
произведения. В будущем эта связь даёт косинусное сходство «почти
бесплатно».

### Поведение длины при масштабировании

**Если вектор умножить на число k, его длина умножается на |k|:**

```
|k · v| = |k| · |v|
```

Вывод формулы:

```
|[k·v₁, k·v₂]|² = (k·v₁)² + (k·v₂)²
                = k²·v₁² + k²·v₂²
                = k²·(v₁² + v₂²)
                = k²·|v|²
|[k·v₁, k·v₂]|  = |k|·|v|
```

Ключевой шаг — вынос `k²` за скобки из-под корня, после чего
`√(k²) = |k|`. Модуль, потому что для отрицательных `k` длина всё равно
положительна.

Это правило важно для следующей темы — масштабной инвариантности
косинусного сходства.

### Применения длины

- **Расстояние между двумя точками** — `|a - b|`. Применяется в KNN,
  кластеризации, обнаружении выбросов.
- **Нормализация вектора** — деление вектора на свою длину `v / |v|`
  даёт вектор длины 1, смотрящий в ту же сторону. Используется, когда
  важно только направление, не масштаб.
- **Метрики сходства** — косинусное сходство, евклидово расстояние.

## Часть IV. Сходство векторов

Самая практическая часть в линейной алгебре для AI и аналитики данных.

### Задача: «найди похожие объекты»

Типичный сценарий — есть база объектов (клиентов, документов, товаров),
каждый представлен вектором признаков. Нужно найти **похожие** на
заданный объект пары или группы.

**Первая мысль** — использовать скалярное произведение. Большое число
→ «похожи». Маленькое → «нет». Простая, прозрачная метрика.

### Проблема скалярного произведения как метрики сходства

Скалярное произведение **зависит от длин обоих векторов**:

```
a · b = |a| · |b| · cos(θ)
```

Если хотя бы один вектор имеет большие координаты — результат большой,
**даже если направление не совпадает**. Это создаёт системную ошибку
для метрики сходства.

**Пример.** Пусть в CRM-базе есть три клиента, описанных вектором
`[количество заказов в год, средний чек]`:

```
A:      [50, 8000]       # крупный клиент
100·A:  [5000, 800000]   # буквально A, увеличенный в 100 раз — то же направление, но длиннее
B:      [10, 3000]       # совсем другой клиент
```

`100·A` геометрически — это **тот же тип клиента** (то же направление в
пространстве), просто числа в 100 раз больше. Эту «масштабированную копию»
ожидается видеть **одинаково похожей** на A, что и A сам на себя.

Скалярные произведения:

```
A · A      =     64 002 500
A · 100·A  =  6 400 250 000      ← в 100 раз больше, чем A на самого себя
A · B      =     24 000 500
```

Если интерпретировать «больше = более похожи», получается:

> «100·A более похож на A, чем сам A на себя.»

Это **абсурд**. A сам на себя похож на 100% — это эталон сходства, никто
не может быть «более похож на A». Скалярное произведение этого правила
**не соблюдает** — оно зависит от длин, и любое масштабирование
исходного вектора умножает результат, имитируя «супер-сходство».

### Косинусное сходство (cosine similarity)

**Решение проблемы:** разделить скалярное произведение на произведение
длин обоих векторов. Это убирает влияние масштаба — остаётся **только
направление**.

```
cos_sim(a, b) = (a · b) / (|a| · |b|)
```

**Откуда название.** Из формулы `a · b = |a| · |b| · cos(θ)`:

```
cos(θ) = (a · b) / (|a| · |b|)
```

То есть результат — **буквально косинус угла между двумя векторами**.

**В коде:**

```python
def cosine_similarity(a, b):
    if magnitude(a) == 0 or magnitude(b) == 0:
        raise ValueError("косинусное сходство не определено для нулевого вектора")
    return dot(a, b) / (magnitude(a) * magnitude(b))
```

Проверка от нулевого вектора нужна — длина нулевого вектора 0, деление
на ноль не определено.

### Свойства косинусного сходства

**Диапазон результата всегда `[-1, +1]`:**

| Значение | Угол | Смысл |
|---|---|---|
| `+1` | 0° | смотрят в одну сторону (полное сходство) |
| `0` | 90° | перпендикулярны (никакой связи) |
| `-1` | 180° | противоположны |

**Масштабная инвариантность (scale invariance) — главное свойство.**

Если умножить любой из векторов на положительное число — косинусное
сходство **не изменится**:

```
cos_sim(k·a, b) = (k·(a·b)) / (|k|·|a|·|b|) = (a·b) / (|a|·|b|) = cos_sim(a, b)
```

Для отрицательного `k` знак результата перевернётся (направление
становится противоположным).

В частности:

```
cos_sim(A, A)       = 1.0    # эталон
cos_sim(A, 100·A)   = 1.0    # та же стрелка по направлению
cos_sim(A, B)       < 1.0    # другое направление
```

То «супер-сходство», которое имитировало скалярное произведение, в
косинусном исчезает. Это и есть **метрика, измеряющая чисто направление,
безотносительно к размеру**.

### Перевод косинусного сходства в угол

Cosine — это **косинус**, и через арккосинус можно перевести в сам угол:

```python
import math
angle_deg = math.degrees(math.acos(cos_value))
```

Перевод в градусы даёт куда более интуитивный масштаб:

| cos | угол | смысл |
|---|---|---|
| 1.0 | 0° | одна стрелка |
| 0.97 | ≈ 13.6° | похожи, но различимы |
| 0.5 | 60° | заметно различные |
| 0 | 90° | независимые |
| -1.0 | 180° | противоположные |

### «Cos = 0.97 это много или мало?» — нет универсального ответа

Это **зависит от контекста** и распределения cosine во всей базе:

| Контекст | «Сильное сходство» начинается с |
|---|---|
| Word embeddings (BERT, OpenAI) | 0.7–0.8 |
| Профайлы клиентов в 2-3 признаках | 0.95+ |
| Высокоразмерные embeddings (768D+) | 0.5+ |

Чем выше размерность пространства, тем **реже** случайные пары близки →
даже средний cosine уже сигнал.

Практический приём:

1. Посчитать cosine между **всеми** парами в базе.
2. Построить гистограмму распределения.
3. Эмпирически выбрать порог (например, верхние 10% значений).

Одно значение cosine само по себе **ни о чём не говорит** — смысл
появляется только в сравнении с распределением.

### Применения косинусного сходства в AI

- **RAG (retrieval-augmented generation)** — запрос и каждый документ
  превращают в векторы. Сравнивают `cos_sim(запрос, документ)` для всей
  базы, возвращают топ-K. Это и есть «семантический поиск».
- **Рекомендательные системы** — товар × товар матрица косинусных
  сходств. «Кто покупал X, покупал и Y».
- **Word embeddings** — `cos_sim(vec("король"), vec("королева"))` будет
  высоким (≈ 0.7–0.9 в типичной модели), `cos_sim(vec("король"),
  vec("банан"))` — около 0.
- **Кластеризация клиентов** — нахождение групп с похожим паттерном
  поведения.

### Нормализация вектора (vector normalization)

**Нормализовать вектор** = разделить его на свою же длину
(`magnitude`). Получается вектор **единичной длины** (unit vector —
единичный вектор), смотрящий в ту же сторону:

```
v_norm = v / magnitude(v)
```

После операции:
- `magnitude(v_norm) == 1.0` (всегда),
- направление сохранилось,
- информация о «масштабе» вектора потерялась.

**Зачем это нужно.** Это **тот же приём, что уже сидит внутри
`cosine_similarity`**:

```
cosine_similarity(a, b) = dot(a, b) / (|a| · |b|)
                       = dot(a / |a|, b / |b|)
                       = dot(normalize(a), normalize(b))
```

Косинусное сходство — это просто скалярное произведение **уже
нормализованных** векторов. Если работаешь с большим набором эмбеддингов
и нужно много раз сравнивать — выгодно **один раз** нормализовать всё и
потом сравнивать через простой `dot` (это в 2 раза быстрее, чем каждый
раз пересчитывать magnitude).

**Грабли:** нулевой вектор. У него длина 0, делить на 0 нельзя — нужен
`raise ValueError`. И сравнивать длину с нулём лучше не через
`mag == 0`, а через `math.isclose(mag, 0)` — иначе вектор-«почти-ноль»
(длина `1e-17`) даст после деления координаты в миллиарды.

**Рабочий код в `my/vector.py`:**

```python
def normalize(v: list[float]) -> list[float]:
    mag = magnitude(v)
    if math.isclose(mag, 0):
        raise ValueError("передан нулевой вектор")
    return scale(v, 1 / mag)
```

Заодно по этому же поводу обновили `cosine_similarity` — длины обоих
векторов считаем **один раз** и сохраняем в переменные:

```python
def cosine_similarity(a: list[float], b: list[float]) -> float:
    mag_a = magnitude(a)
    mag_b = magnitude(b)
    if math.isclose(mag_a, 0) or math.isclose(mag_b, 0):
        raise ValueError("передан нулевой вектор")
    return dot(a, b) / (mag_a * mag_b)
```

⚠️ **Не путать с feature normalization** из следующей Части V. Это
**два разных** приёма, оба зовутся «нормализация».

| | Vector normalization | Feature normalization |
|---|---|---|
| На что делим | весь вектор на одно число (его длину) | каждую координату на **свою** константу (min/max или std) |
| Цель | привести к длине 1 | уравнять шкалы признаков между собой |
| Где применяется | внутри `cosine_similarity`, в эмбеддингах для RAG | **перед** подачей в модель, до расчёта сходства |
| Меняет направление? | нет, только длину | да (это, по сути, новый вектор) |

В CRM-кейсе (когда координаты — это «заказы», «рубли», «NPS») сначала
нужна **feature normalization** (заказы и рубли несравнимы как есть).
**Vector normalization** работает, когда координаты уже сопоставимы —
например, эмбеддинг из transformer'а (768 чисел в одной «шкале»).

## Часть V. Подготовка данных к расчёту сходства

Просто посчитать косинусное сходство на сырых данных — обычно
**недостаточно**. На реальных данных вылезает важная ловушка, связанная
со шкалами признаков. Без её преодоления даже cosine начинает обманывать.

### Проблема разных шкал

У объекта в реальной задаче обычно несколько признаков **разной
природы и разных диапазонов**:

| Признак | Типичный диапазон |
|---|---|
| Количество заказов в год | 1 – 100 |
| Средний чек | 500 – 50 000 |
| NPS | 0 – 10 |
| Возраст | 18 – 80 |
| Дней с последней покупки | 1 – 365 |

Когда математика обрабатывает такого клиента как вектор, **каждый
признак вкладывается в результат пропорционально своему размеру**. Чек в
тысячах × тысячах = миллионы. NPS в десятках × десятках = сотни. Чек
будет весить в **тысячи раз больше** во всех формулах.

**Аналогия с жюри.** Представим пять экспертов, оценивающих клиентов.
Каждому дали свою шкалу:

- Эксперт А: 0–10.
- Эксперт Б: 0–100.
- Эксперт В: 0–10 000.

Если просто **сложить** оценки, кто будет решать? Эксперт В. Его «6 000»
всегда в тысячи раз больше любого «10» эксперта А, даже если для эксперта
А это была отличная оценка. Голоса А и Б **не слышны**.

Это и есть проблема разных шкал признаков. Без предварительной обработки
ML-алгоритм слушает только один признак — самый «громкий по числам».

**Иллюстрация на данных.** Возьмём двух клиентов:

```
A = [50, 8000]      # 50 заказов в год, средний чек 8000
B = [10, 3000]      # 10 заказов в год, средний чек 3000
```

Скалярное произведение:

```
A · B = 50·10 + 8000·3000 = 500 + 24 000 000
                            ↑      ↑
                            шум    весь сигнал
```

Заказы вкладывают 500, чек — 24 миллиона. Соотношение 1 : 48 000. Заказы
**не учитываются** в расчёте сходства.

Косинусное сходство в этом случае ≈ **0.99996** — обманчиво высокое
значение. Подумаешь — «практически идентичные клиенты», тогда как у A в
5 раз больше заказов и почти в 3 раза выше чек. Косинус «слышит» только
чек, для которого пропорции близки.

### Min-Max нормализация

Стандартное решение — **привести каждый признак к единой шкале**.
Простейший способ — Min-Max:

```
x_norm = (x - min) / (max - min)
```

После применения все значения признака лежат в `[0, 1]`. Минимум → 0,
максимум → 1, остальные равномерно между ними.

**Как запомнить формулу через логику:**

- **Числитель `x - min`** = «насколько выше минимума».
- **Знаменатель `max - min`** = «вся длина шкалы».
- **Деление** = «доля шкалы, пройденная от минимума».

Аналогия: «уровень 35 из 80» в видеоигре → прогресс `(35-0)/(80-0) =
43.75%`. То же самое для признака.

**Применим к примеру.** Пусть по всей базе:

```
max заказов в год = 100, min = 0
max средний чек   = 10 000, min = 0
```

```
A_norm = [50/100, 8000/10000] = [0.5, 0.8]
B_norm = [10/100, 3000/10000] = [0.1, 0.3]
```

**Все координаты теперь одного порядка**, между 0 и 1. Ни одна не
доминирует.

Перепроверяем скалярное произведение:

```
A_norm · B_norm = 0.5·0.1 + 0.8·0.3 = 0.05 + 0.24
                  ↑          ↑
                  1 часть    5 частей
```

Соотношение вкладов теперь **1 : 5** вместо **1 : 48 000**. Обе
координаты участвуют. Чек всё ещё немного важнее (числа больше), но это
уже не «диктатура одной координаты».

Косинусное сходство:

```
cos_sim(A_norm, B_norm) ≈ 0.972
```

Что в градусах — около 13.6°. **Реальное** сходство, учитывающее обе
координаты на равных. На фоне до-нормализационных 0.99996 (≈ 0.5°)
видно, что нормализация **многократно увеличила различимость** разных
клиентов.

### Когда нормализация нужна

| Алгоритм | Нужна? |
|---|---|
| KNN, K-means, SVM, нейросети, регрессия | Да |
| Метрики сходства (cosine, евклидово) | Да |
| Деревья решений, Random Forest, gradient boosting | Нет |
| Naive Bayes | Нет |

Грубое правило: если алгоритм работает через **расстояния** или
**градиенты** — нормализация обязательна. Если через **сравнения
«больше / меньше»** — нет (деревьям шкала не важна, они выбирают
точки разбиения).

### Другие виды нормализации

| Метод | Формула | Когда применять |
|---|---|---|
| **Min-Max** | `(x - min) / (max - min)` | Простой случай, шкала известна. Чувствителен к выбросам. |
| **Z-score (стандартизация)** | `(x - mean) / std` | Устойчив к выбросам. Среднее = 0, отклонение = 1. Самый частый в ML. |
| **L2-нормализация вектора** | `v / |v|` | Делает **весь вектор** длины 1. Не путать с предыдущими — это про вектор целиком, а не про отдельные признаки. |

**Min-Max vs L2-нормализация** — это **разные** операции под **разные**
цели:

- **Min-Max / Z-score** — между **признаками** (колонками таблицы). Цель:
  уравнять шкалы признаков, чтобы один признак не доминировал.
- **L2** — внутри **одного вектора** (одной строки таблицы). Цель:
  сделать стрелку длины 1, оставив только направление. Для случаев,
  когда нужно сравнивать **только направления**, не масштабы.

Обе могут применяться последовательно: сначала Min-Max между признаками
для уравнивания шкал, потом L2 для каждой строки.

### Feature weighting — взвешивание признаков под задачу

После нормализации можно пойти ещё дальше — **назначить весам разную
важность под конкретную бизнес-задачу**:

```python
weights = [w_orders, w_check, w_nps, ...]
v_weighted = [v[i] * weights[i] for i in range(len(v))]
cosine_similarity(v_weighted, u_weighted)
```

Это не «правильная» научная нормализация — это **бизнес-настройка
метрики**. Одна и та же база, **разные сегментации** под разные кампании:

| Цель | Что взвесить тяжелее |
|---|---|
| Up-sell на дорогую категорию | средний чек |
| Активация спящих клиентов | дни с последней покупки |
| Программа лояльности | NPS + стаж |

Веса работают как «усилитель координаты» в метрике. Усилили признак —
система сильнее реагирует на отличия по нему.

Формальные методы в ML делают это автоматически: **расстояние
Махаланобиса** учитывает корреляции признаков, **gradient boosting**
выдаёт **feature importance** на основе размеченных данных. Эти
importance можно использовать как веса для метрик сходства. Но
ручное взвешивание остаётся рабочим инструментом из-за прозрачности.

**Главная мысль про сходство:** «похожесть» — это **не объективная
штука**. Это **выбор**, что считать важным. Любая метрика сходства —
это функция твоих весов. Сделал веса равными → одна сегментация.
Поднял вес одного признака → другая. Этим и отличается «просто
похожи» от «похожи для конкретной задачи».

## Часть VI. Подводные камни вычислений

### Арифметика с плавающей точкой

Числа `float` хранятся в **двоичном формате** ограниченной длины (обычно
64 бита). **Большинство десятичных дробей не представимы в нём точно**
— так же, как `1/3` нельзя точно записать в десятичной (бесконечная
дробь 0.333...).

Классическая иллюстрация:

```python
print(0.1 + 0.2)        # 0.30000000000000004
print(0.1 + 0.2 == 0.3) # False
```

Это **не баг конкретного языка**. Это фундаментальная природа стандарта
IEEE 754 для float, который используют все языки (Python, C,
JavaScript, Java, NumPy, PyTorch).

**В векторных вычислениях** это вылезает везде, где есть деление, корни,
тригонометрия. Например, при подсчёте косинусного сходства двух
противоположных векторов теоретически должно получаться ровно −1, но на
практике получится `-0.9999999999999998` — из-за накопления
микро-ошибок в `√` и делении.

**Главное правило: НЕ сравнивать float через `==`.**

```python
# ❌ Никогда
if cos_sim == 1.0: ...

# ✅ Правильно
import math
if math.isclose(cos_sim, 1.0): ...
```

`math.isclose(a, b)` проверяет «достаточно ли близки» с допуском по
умолчанию `1e-9`. В NumPy и PyTorch есть аналоги — `np.allclose`,
`torch.allclose`.

**Где это критично в ML:**

- **Нормализация** — `|v| == 1` → `math.isclose(magnitude(v), 1.0)`.
- **Проверка ортогональности** — `dot == 0` → `abs(dot) < epsilon`.
- **Сходимость градиентного спуска** — `|gradient| < tolerance`.
- **Сравнение тензоров в тестах** — всегда через `allclose`.

**Косметика отображения.** Микро-хвосты в выводе скрываются через
формат:

```python
print(f"{x:.4f}")    # 4 знака после точки
```

Округление **только при отображении**, в памяти число остаётся прежним.

**Распространённая ловушка:** `1/3 ≠ 0.3`. Это **два разных числа**.

```
1/3 ≈ 0.333333...    (бесконечная дробь)
0.3 = 0.3             (конечная)
```

В коде всегда писать `1/3` (или `1/N`), а не «приближение от руки» `0.3`.

## Часть VII. Класс Vector — упаковка функций в объект

До этого момента в `my/vector.py` все операции были **отдельными
функциями**: `add(a, b)`, `dot(a, b)`, `normalize(v)`. Это работает,
но у такого подхода три проблемы:

1. **Тип «вектор» нигде не закреплён.** Список — это список, любая
   функция может его обработать или сломать. Python не знает, что
   `[3, 4]` — это «вектор».
2. **Вызовы громоздкие:** `normalize(add(scale(a, 2), b))` против
   читаемого `(a * 2 + b).normalize()`.
3. **Нет «представления» объекта.** `print([3, 4])` показывает
   список — не понятно, вектор это или что-то ещё.

**Класс** решает все три. Класс — это **шаблон** типа. Если вектор —
это шаблон карточки клиента в CRM, то:
- **атрибуты** (поля карточки) — имя, телефон, LTV;
- **методы** (действия с карточкой) — отправить SMS, посчитать
  средний чек;
- **экземпляр** (instance) — конкретная карточка Анны Ивановой.

Параллельная реализация в `my/vector_class.py` (отдельный файл рядом
с функциональной версией).

### Минимальный синтаксис

```python
class Vector:
    def __init__(self, components: list[float]):
        self.components = components
```

- `class Vector:` — объявление. Имена классов — `PascalCase`
  (функции/переменные — `snake_case`).
- `__init__` — **дандер-метод** (dunder = double underscore, двойное
  подчёркивание). Зовётся автоматически при `Vector([3, 4])`. Имя
  не произвольное — это из конкретного списка «магических» имён,
  которые Python знает сам.
- `self` — сам объект, который сейчас настраиваем. Python подставляет
  его автоматически: при `v = Vector([3, 4])` Python внутри делает
  `Vector.__init__(v, [3, 4])`, где `v` попадает в `self`. Имя `self`
  — соглашение (как `this` в Java); технически можно `me`, но **все
  пишут `self`**.
- `self.components = components` — сохраняем координаты в **атрибут**
  объекта. После этого у `v` появилось поле `v.components`.

### `__repr__` — как объект показывает себя

Без `__repr__` `print(v)` выдаст `<__main__.Vector object at
0x7f...>` — адрес в памяти, бесполезно для человека.

```python
def __repr__(self) -> str:
    return f"Vector({self.components})"
```

Принимает только `self`, возвращает **строку**. Когда Python вызывает
`__repr__`:
- `print(v)`, `f"{v}"`, `repr(v)`;
- `>>> v` в REPL/Jupyter без `print`;
- **внутри контейнеров**: `print([v1, v2])` зовёт `__repr__` каждого
  элемента списка.

**Идиома:** делать вывод похожим на валидный Python-код вида
`ИмяКласса(аргументы)` — `Vector([1.0, 2.0])`. Зачем:
1. Лог-сообщение можно скопировать и вставить в консоль для
   воспроизведения бага.
2. В контейнерах скобки задают **визуальные границы** между объектами:
   `[Vector([1, 2]), Vector([3, 4])]` понятно, `[Vector 1, 2,
   Vector 3, 4]` — нет.
3. Старый идеал: `eval(repr(obj)) == obj` (на практике `eval` не
   используют — опасно, но сама идея остаётся).

### Перегрузка операторов: магия `+`, `-`, `*`

В Python нет встроенного `+` для каждого типа. Когда пишешь `a + b`,
Python превращает это в `a.__add__(b)`. Для `list.__add__` —
конкатенация, для `str.__add__` — склейка, для `int.__add__` —
сложение. Если **ты** определишь `__add__` в своём классе — `+`
заработает и с твоими объектами.

```python
def __add__(self, other) -> "Vector":
    if len(self.components) != len(other.components):
        raise ValueError(
            f"разные размерности векторов: "
            f"{len(self.components)} и {len(other.components)}"
        )
    return Vector(
        [s + o for s, o in zip(self.components, other.components)]
    )
```

Ключевое:
- **Возвращать новый `Vector`**, не список. Тогда работает цепочка:
  `(v + w).magnitude()`. Если бы возвращали список — `.magnitude()`
  у списка не существует.
- **Не мутировать** `self` или `other`. Всегда создаём новый объект.

`__sub__` — точно такой же, но со знаком `-`.

### Forward reference в аннотациях типов

Type hint `-> Vector` внутри самого `Vector` ругается:
`Undefined name 'Vector'`. Причина — на момент парсинга `def __add__`
имя `Vector` ещё **не создано** (класс не закончил определяться).

Три способа починить:

```python
# 1. Строка вместо имени (самое простое, выбрано в уроке)
def __add__(self, other) -> "Vector":

# 2. from __future__ import annotations (все аннотации отложенные)
from __future__ import annotations
def __add__(self, other) -> Vector:

# 3. typing.Self (Python 3.11+) — точнее, работает при наследовании
from typing import Self
def __add__(self, other) -> Self:
```

### Зеркальные операторы: `__rmul__` для случая «число × вектор»

Мы хотим, чтобы работали **оба** порядка:

```python
v * 3      # → v.__mul__(3)        — стандартный путь
3 * v      # → int.__mul__(3, v)   — int не знает Vector
           # → возвращает NotImplemented
           # → Python пробует v.__rmul__(3)
```

Без `__rmul__` второй случай вылетит с `TypeError`. Обычно `__rmul__`
— это **просто переадресация** к `__mul__`, потому что умножение на
скаляр коммутативно:

```python
def __rmul__(self, other):
    return self * other
```

(У некоммутативных операций — например, матричного умножения —
`__rmul__` пришлось бы писать отдельно.)

### `NotImplemented` vs `raise TypeError`

Если `other` — это что-то не наше (строка, словарь и т.п.), Python
ожидает увидеть **возврат специальной константы**, а не исключение:

```python
def __mul__(self, other) -> "Vector":
    if not isinstance(other, (int, float)):
        return NotImplemented           # ← именно так
    return Vector([s * other for s in self.components])
```

Почему `NotImplemented`, а не `raise TypeError`:
- Возврат `NotImplemented` говорит Python'у «не моё дело, пробуй
  другую сторону». Python автоматически попробует `other.__rmul__(self)`.
- Если другая сторона тоже не справилась — Python **сам** поднимет
  `TypeError` с правильным сообщением.
- Если ты делаешь `raise`, Python никогда не пробует обратку. В
  частности, ломаются легитимные случаи вроде `MyMatrix() * v`, где
  у `MyMatrix.__mul__` могла бы быть правильная логика.

**Грабли с `v * "abc"`.** При `NotImplemented` Python пошёл искать
`str.__rmul__`. А у `str` он **есть**: `"ab" * 3 == "ababab"`. Str
интерпретирует `v` как «количество повторов», `v` — не int, и str
поднимает **своё** странное сообщение: `can't multiply sequence by
non-int of type 'Vector'`. Это поведение **корректное**: для прода
проверяют типы более тонко, чтобы такие хвосты не лезли. Для учебной
версии — оставляем как есть.

### Обычные методы: вызов точкой

`dot`, `magnitude`, `normalize`, `cosine_similarity` — **не дандеры**.
Зовутся по имени:

```python
v.dot(w)
v.magnitude()
v.normalize()
v.cosine_similarity(w)
```

Логика **точно та же**, что в функциональной версии в `vector.py`.
Разница только в форме:
- Первый параметр определения — `self` (Python подставит при вызове).
- Доступ к данным — `self.components`, не отдельная переменная.
- Если метод нужен другому методу — зови через тот же `self`:
  `self.dot(self)` внутри `magnitude`.

```python
def dot(self, other) -> float:
    if len(self.components) != len(other.components):
        raise ValueError("разные размерности")
    return sum(s * o for s, o in zip(self.components, other.components))

def magnitude(self) -> float:
    return math.sqrt(self.dot(self))

def normalize(self) -> "Vector":
    mag = self.magnitude()
    if math.isclose(mag, 0):
        raise ValueError("передан нулевой вектор")
    return self * (1 / mag)

def cosine_similarity(self, other) -> float:
    mag_s = self.magnitude()
    mag_o = other.magnitude()
    if math.isclose(mag_s, 0) or math.isclose(mag_o, 0):
        raise ValueError("передан нулевой вектор")
    return self.dot(other) / (mag_s * mag_o)
```

В `normalize` обрати внимание: вместо явного списка-comprehension
вызываем уже определённый `*` (`__mul__`) — это и есть умножение
на скаляр. Если оператор уже работает — нет смысла повторять
реализацию.

### Грабли, которые проявились по пути

1. **`len(self)` не работает** — у Vector нет `__len__`. Чтобы узнать
   размерность, иди в `len(self.components)`.
2. **`zip(self, other)` не работает** — Vector не итерируется (нет
   `__iter__`). Правильно `zip(self.components, other.components)`.
3. **`return [...]` вместо `return Vector([...])`** в дандерах
   возвращает список — ломает цепочки вызовов.
4. **`self.dot(self.components, self.components)`** — лишний аргумент.
   `self` уже подставлен через точку. Правильно `self.dot(self)`.
5. **`def cosine_similarity(other):` без `self`** — это уже не метод.
   В **определении** `self` обязан быть первым параметром; при
   **вызове** — Python подставляет сам.
6. **Каскадная ошибка через `dot`** — опечатка `s + o` вместо
   `s * o` в скалярном произведении ломает не только сам `dot`,
   но и всё, что от него зависит (`magnitude`, `cosine_similarity`,
   `normalize`). Один баг — несколько неверных результатов.
7. **Стиль имени:** в Python принят **американский** вариант
   (`normalize`, `serialize`, `analyze`), не британский. Лучше
   держаться одного варианта внутри проекта.

### Запоминалка про `self`

| Контекст | Что пишем | Почему |
|---|---|---|
| Определение метода | `def cosine_similarity(self, other):` | Python должен знать имя первого параметра |
| Вызов через точку | `v.cosine_similarity(w)` | `v` уже подставится как `self` |

В скобках при **определении** на один параметр **больше**, чем при
**вызове**.

## Часть VIII. Матрицы — введение

### Зачем нужны матрицы после векторов

Вектор — это **один объект** (один клиент: `[возраст, MRR,
дней_с_логина]`). В реальной задаче объектов **много** — тысяча
клиентов, миллион эмбеддингов, сотня документов в RAG-базе. Цикл
`for obj in objects:` по такому списку — медленный (см. урок 05,
упражнение 1: numpy vs list-comprehension, ускорение ~9×). Правило
было «vectorize, don't loop» — работать не с одним объектом, а сразу
со всей пачкой.

Чтобы «пачка объектов» стала **одной сущностью**, над которой работают
без цикла, нужна структура — **матрица**.

### Матрица как пачка векторов

**Матрица (matrix)** — это таблица чисел. Строки (rows) и столбцы
(columns). Самый понятный способ думать о ней — **стопка векторов**.

```
              возраст  MRR   дней_с_логина
клиент_1  →   [ 28,    1500,  2 ]
клиент_2  →   [ 45,    8000, 30 ]
клиент_3  →   [ 33,    3200,  5 ]
```

В виде матрицы:

```
M = [[28, 1500,  2],
     [45, 8000, 30],
     [33, 3200,  5]]
```

Это **матрица 3×3**: 3 строки, 3 столбца. Размерность всегда пишут как
`строки × столбцы` — этот порядок держится во всех формулах.

### Два взгляда на одну матрицу

**По строкам (row-wise):** каждая строка — это один объект-вектор.
Строка 0 = `[28, 1500, 2]` = клиент 1 целиком. **Матрица = стопка
строк-векторов.**

**По столбцам (column-wise):** каждый столбец — это одна фича по всей
базе. Столбец 1 = `[1500, 8000, 3200]` = MRR всех клиентов сразу.
**Матрица = стопка столбцов-векторов.**

Один и тот же набор чисел, два разных способа «нарезать»:

- Хочешь работать с **конкретным клиентом** — берёшь строку.
- Хочешь работать с **одной фичей по всей базе** (нормализовать,
  посчитать среднее, найти выбросы) — берёшь столбец.

Эта двойственность — основа всей работы с матрицами. Дальше всё, что
делается с матрицами, сводится к выбору: вдоль какой оси работаем.

### Главная идея: работа без цикла

Главное, ради чего матрицы вообще существуют:

**Операция над матрицей = операция сразу над всеми строками или всеми
столбцами одной командой.**

- «Поднять MRR всем клиентам на 10%» → одна операция над столбцом.
- «Посчитать сходство каждого клиента с эталоном» → одна операция
  матрица × вектор.
- «Нормализовать каждую фичу» → одна операция по столбцам.

Это и есть **векторизация на уровне всей базы**, а не одного объекта.

### NumPy: минимум для матриц

**Главная сущность** — `ndarray` (N-dimensional array, N-мерный
массив). Один объект для вектора (1D), матрицы (2D) и тензора (3D+).

Конвенция импорта (без вариантов — весь курс и комьюнити):

```python
import numpy as np
```

**Главные атрибуты `ndarray`:**

| Атрибут | Что показывает |
|---|---|
| `.shape` | размерность как кортеж: `(3, 3)` для 3×3, `(5,)` для вектора длины 5 |
| `.ndim` | сколько измерений: 1 для вектора, 2 для матрицы |
| `.dtype` | тип чисел: `int64`, `float64` и т.п. |
| `.size` | общее количество элементов |

**Создание и доступ:**

```python
import numpy as np

M = np.array([[28, 1500,  2],
              [45, 8000, 30],
              [33, 3200,  5]])

M.shape           # (3, 3)
M.ndim            # 2

M[0]              # array([28, 1500, 2]) — строка 0, клиент 1 целиком
M[:, 1]           # array([1500, 8000, 3200]) — столбец 1, MRR всех
M[1, 0]           # 45 — элемент строки 1, столбца 0 (возраст клиента 2)
M[:, 1].mean()    # 4233.33 — среднее MRR по базе
```

Двоеточие `:` в `M[:, 1]` значит «все элементы по этой оси». Эта форма
ещё много раз встретится: `M[:, 0]` — весь нулевой столбец, `M[0, :]`
— вся нулевая строка (то же, что `M[0]`), `M[1:, :]` — все строки
кроме первой.

### Грабли, которые проявились по пути

1. **Индексация с нуля (zero-indexing).** «Второй столбец»
   по-человечески — это `M[:, 1]`, а не `M[:, 2]`. Самая частая ошибка
   в начале, особенно когда мысленно считаешь «первый, второй, третий».
   Признак, что ты ошибся, — цифры «не сходятся»: среднее MRR должно
   быть в тысячах, а получилось 12 (тогда ты случайно усреднил столбец
   «дней с логина»). Один раз увидел свою же ошибку — дальше мозг сам
   проверяет.

2. **`np.array([1, 2, 3])` vs `np.array([[1, 2, 3]])`.** Первое —
   вектор (1D, shape `(3,)`), второе — матрица 1×3 (2D, shape
   `(1, 3)`). Две квадратные скобки — это уже двумерная структура.
   Многие операции (умножение матриц, например) требуют именно 2D и
   упадут на 1D.

3. **Индексация `M[i, j]` через запятую, а не `M[i][j]`.** Питоновская
   форма `M[i][j]` работает, но медленнее и не даёт аккуратно достать
   столбец (`M[:][1]` — это не столбец, а просто `M[1]`, потому что
   `M[:]` — копия всей матрицы). Только `M[:, j]` — корректный способ
   взять столбец.

4. **`np.array([1, 2.5, 3])` — вся матрица станет float.** Один float
   среди int'ов автоматически приводит весь `dtype`. Иногда полезно,
   иногда сюрприз.

### Конвенция имён: `M` для матриц, `v` для векторов

В математической и ML-литературе устоявшаяся конвенция:

- **Матрицы — заглавными:** `M`, `A`, `X` (матрица фичей в sklearn —
  всегда `X`).
- **Векторы — строчными:** `v`, `x`, `w`.

Эта конвенция перевешивает PEP 8 (который требует lowercase для
переменных), потому что когда в одном коде есть и матрица, и вектор,
визуальная разница спасает от перепутывания, что чему передаёшь.
Посмотри любой numpy-туториал или исходники sklearn — там `X` всегда
заглавная.

### Транспонирование (transpose)

**`M.T`** — переворачивает матрицу: строки становятся столбцами,
столбцы — строками. Если матрица имела форму `(m, n)`, после `.T`
будет `(n, m)`. Это **атрибут**, без скобок:

```python
M.T          # не M.T(), просто M.T
```

**Геометрически:** представь, что положил таблицу набок. Что было
вертикально — стало горизонтально.

**Зачем практически:**

- **Сменить ориентацию данных.** В одной библиотеке «строки = фичи,
  столбцы = клиенты», в другой — наоборот. `.T` переводит одно в
  другое за одну операцию.
- **Привести формы к совместимости** для матричного умножения. Чаще
  всего `.T` в ML-коде стоит именно для этого: «у меня форма не та,
  кручу».
- **`M.T @ M`** — частый паттерн в линейной регрессии (метод
  наименьших квадратов) и PCA. Даёт «матрицу скалярных произведений
  всех пар столбцов».

**Грабли:**

- `M.T` — это **view** (вид) на ту же память, не копия. Если изменишь
  элемент в `M.T`, изменится и в `M`. Когда нужна реальная копия —
  `M.T.copy()`.
- Для **1D-вектора** `v.T` ничего не делает: `np.array([1, 2, 3]).T`
  остаётся `(3,)`. Транспонировать имеет смысл только начиная с 2D.

### Поэлементные операции

Сложение, вычитание, умножение на скаляр — **работают точно так же,
как у векторов**, только сразу по всей матрице.

```python
A + B        # element-wise сложение, формы должны совпасть
A - B        # element-wise вычитание (например, дельта между периодами)
3 * A        # каждый элемент × 3
A * 0.95     # каждый элемент × 0.95 (например, скидка 5% на ВСЁ подряд)
```

**Главное правило для `+` и `-`:** формы должны **совпадать**. `(3,3)`
нельзя сложить с `(2,3)` — numpy кинет ошибку. Та же дисциплина
размерностей, что была у векторов, только по двум осям.

**Семантика операций не следует за формой.** Применить скидку 5% ко
всей матрице сразу через `M * 0.95` математически корректно — но в
CRM-кейсе это значит, что и возраст клиента «упал на 5%» (28 → 26.6),
что бессмыслица. **Числа складываются и умножаются легко, а смысл
столбцов математика не различает.** Дисциплина — всегда отслеживать
руками, что такое каждый столбец и что значит операция над ним. Если
скидка должна применяться только к MRR — нужна маска с `1` в
ненужных столбцах: `M * [[1, 0.95, 1], ...]`.

### Поэлементное умножение `A * B` — НЕ матричное умножение

В numpy символ `*` для двух матриц делает **поэлементное умножение**
(Hadamard product, произведение Адамара):

```python
A = [[1, 2],         B = [[10, 20],
     [3, 4]]              [30, 40]]

A * B = [[ 10,  40],         # 1·10, 2·20
        [ 90, 160]]          # 3·30, 4·40
```

Каждая клетка `A[i,j] · B[i,j]`.

**Это не та операция, которая «матрица × матрица» в линейной
алгебре.** Та делается через специальный оператор **`@`** — см.
раздел про матрица × вектор ниже. В numpy символ `*` обманывает
математическую интуицию — будь внимателен.

### Косметика отображения: `np.set_printoptions`

Когда в матрице соседствуют числа разной величины (например, MRR в
тысячах и возраст в десятках), numpy по умолчанию переключается на
**научную нотацию** при печати:

```
[[2.80e+01 1.65e+03 2.00e+00]
 [4.50e+01 8.80e+03 3.00e+01]]
```

`2.80e+01` = 2.80 × 10¹ = 28. `1.65e+03` = 1.65 × 10³ = 1650. Запись
`XeY` значит «X × 10^Y».

Это **косметика, не математика** — данные хранятся как обычно, но
читать вывод неудобно. Лечится одной командой в начале файла:

```python
np.set_printoptions(suppress=True, precision=4)
```

- `suppress=True` — выключить научную нотацию.
- `precision=4` — максимум 4 знака после точки (не округляет данные,
  только укорачивает вывод).

**В большинстве учебных туториалов и проектов эта строка стоит сразу
после `import numpy as np`.** Полезная привычка.

### Матрица × вектор — главная операция ML

**Это центральная операция всего машинного обучения.** На ней
построены линейная регрессия, логистическая регрессия, каждый слой
нейросети, attention в трансформерах, скоринг клиентов в CRM. Понять
её — значит понять, как работает любая линейная модель.

**Что делает в одной фразе:** **каждой строке матрицы сопоставляет
одно число**. Если матрица из N строк — на выходе вектор длины N.

**Мотивация — скоринг клиентов в CRM**

Имеем матрицу дельт по 10 клиентам — что изменилось у каждого по 3
фичам (возраст, MRR, дни-до-логина) между двумя месяцами:

```
M_delta shape = (10, 3)
```

Хотим из каждой строки получить **одно число** — общий скор клиента.
Но фичи **не одинаково важны**: возраст не интересен (вес `0`), MRR
важно прямо (вес `+1`), дни-до-логина важно **обратно** (вес `-50` —
больше дней без логина = хуже).

Веса собираем в вектор:

```python
w = np.array([0, 1, -50])    # shape (3,)
```

Скор клиента — **взвешенная сумма** его дельт. Для клиента с
дельтой `[0, 200, 3]`:

```
score = 0·0 + 200·1 + 3·(-50) = 50
```

Это **скалярное произведение строки на вектор весов** — та же `dot`,
что в `vector.py`. Делаем её для каждой из 10 строк, кладём результаты
в один вектор:

```
scores shape = (10,)
```

Numpy умеет делать все 10 dot product'ов одной операцией через
**специальный оператор `@`** (с Python 3.5):

```python
scores = M_delta @ w
```

Альтернативы — `np.dot(M, v)` и `np.matmul(M, v)`. Современный код
пишут через `@`.

**Механика — почему выход вектор, а не матрица**

Внутри `dot(строка, w)` происходит **два шага**:

1. **Поэлементное умножение:** `[r₁·w₁, r₂·w₂, r₃·w₃]` — вектор из
   3 чисел.
2. **Сумма всех элементов:** `r₁·w₁ + r₂·w₂ + r₃·w₃` — **одно число**.

**Второй шаг — суммирование — и схлопывает 3 числа в 1.** Это коллапс
размерности. Без него на выходе была бы матрица; с ним — вектор.

Применяем к 10 строкам → получаем 10 скаляров → numpy кладёт их в
позиции 0..9 выходного массива → готов вектор длины 10.

**Правило форм**

```
M (m × n)  @  v (n,)  =  result (m,)
        ↑      ↑              ↑
        └──n=n┘               │
                              │
        m строк ──────────────┘
        задаёт длину выхода
```

«**Внутренние совпадают, наружные дают форму.**»

- **Внутренние:** число столбцов M = длина вектора v.
- **Наружные:** число строк M становится длиной выходного вектора.

Если формы не сошлись — numpy кинет:

```
ValueError: shapes (10,3) and (5,) not aligned: 3 (dim 1) != 5 (dim 0)
```

**Самая частая ошибка матричных операций.** Лечится либо
транспонированием `M.T`, либо проверкой, какой вектор передаёшь.

**Как масштабируется**

| База | shape M | shape w | shape scores |
|---|---|---|---|
| 3 клиента, 3 фичи | (3, 3) | (3,) | (3,) |
| 10 клиентов, 3 фичи | (10, 3) | (3,) | (10,) |
| 1 000 клиентов, 3 фичи | (1000, 3) | (3,) | (1000,) |
| 1 000 000 клиентов, 3 фичи | (1000000, 3) | (3,) | (1000000,) |
| 1000 клиентов, **20 фич** | (1000, **20**) | **(20,)** | (1000,) |

- **При росте базы** (строки M ↑): `w` не меняется (фич столько же),
  `scores` растёт.
- **При добавлении фич** (столбцы M ↑): `w` растёт, `scores` не
  меняется (клиентов столько же).

Это фундаментальный разрез **строки = объекты, столбцы = фичи**, на
котором держится весь ML-код. В туториалах и документации часто
увидишь `X.shape == (n_samples, n_features)` — это именно про это.

**`@` vs `*` — два разных оператора**

| | `M @ v` | `M * v` |
|---|---|---|
| Что делает с каждой строкой | dot с v → схлопывает в число | поэлементное умножение → сохраняет длину |
| Форма выхода (для 10 клиентов) | `(10,)` — вектор | `(10, 3)` — матрица |
| Смысл | скоринг, линейная трансформация | масштабирование столбцов |

`@` **схлопывает** одну ось. `*` ничего не схлопывает. **Путать
нельзя** — синтаксически оба корректны, но возвращают **разные
результаты**.

**Связь с ML — что «учится» нейросеть**

Уравнение

```
output = X @ W + b
```

— **базовое уравнение всего deep learning**. X — матрица фичей
`(n_samples, n_features)`, W — матрица весов, b — смещение, output —
предсказание.

- **Линейная регрессия** = один такой шаг.
- **Логистическая регрессия** = тот же шаг + сигмоидная обёртка.
- **Один линейный слой нейросети** (Linear/Dense) = тот же шаг.
- **LLM/GPT** = **много** таких шагов подряд (десятки/сотни слоёв),
  разбавленных нелинейностями между ними.

«Обучение» = поиск таких матриц W₁, W₂, ..., при которых output
близок к реальному ответу. **Миллиарды параметров в современных
LLM — это буквально числа в матрицах весов.**

Когда выбираешь веса `[0, 1, -50]` руками — это упражнение. В
реальном ML те же числа находил бы **алгоритм** по размеченным
данным («вот клиенты с известным churn'ом — найди веса, при которых
скор совпадает с реальностью»). Это и есть «обучение».

### Ловушка: данные на разных шкалах — отголосок Части V

Маленький пример из практики. Сгенерировали случайную дельта-матрицу:

```python
M_delta = np.random.randint(-100, 1500, size=(10, 3))
```

И применили веса `[0, 1, -50]`. Все 10 скоров получились
**отрицательными**, порядка -10000..-50000. Что не так?

`randint(-100, 1500)` раздаёт **одинаковый широкий диапазон всем
столбцам**. Третий столбец (дни-до-логина) получил числа до 1500 —
что нереально (никто не пропадает на 4 года). Умноженный на вес
`-50`, он **доминирует** над остальными вкладами.

**Это не баг модели** — модель работает идеально. **Сломаны входные
данные**: третий столбец имеет неестественный диапазон.

**Связка с Частью V (feature normalization):** именно эту проблему
решает Min-Max или Z-score. Перед скорингом признаки должны быть на
одной шкале, иначе доминирующий столбец «съест» сигнал от остальных.
В реальных проектах либо генерируешь каждый столбец отдельно с
реалистичным диапазоном, либо потом нормализуешь.

### Геометрический смысл — матрица как функция

Возвращаемся к двойной интерпретации матриц.

**Раньше** (в начале Части VIII):
- Матрица = **таблица данных** (строки = клиенты, столбцы = фичи).

**Теперь**:
- Матрица = ещё и **функция трансформации**, действующая на векторы.

```
v_new = M @ v_old
```

Читается как: «матрица M **трансформирует** вектор v». Одна и та же
математическая сущность — **два значения**, в зависимости от
контекста. Когда читаешь чужой код:

- `X` или `data` — обычно «таблица данных».
- `W` или `weights` — обычно «функция (трансформация)».

**Что геометрически делают трансформации**

| Тип | Что происходит | Пример матрицы 2×2 |
|---|---|---|
| **Растяжение / сжатие** (scaling) | Каждая ось растягивается/сжимается | `[[2,0],[0,3]]` — x×2, y×3 |
| **Поворот** (rotation) | Все векторы поворачиваются на угол | `[[0,-1],[1,0]]` — поворот на 90° |
| **Отражение** (reflection) | Зеркалирование относительно оси | `[[1,0],[0,-1]]` — отражение по y |
| **Проекция** (projection) | Схлопывание пространства, теряется измерение | `[[1,0],[0,0]]` — всё на ось x |
| **Тождество** (identity) | Ничего не делает | `[[1,0],[0,1]]` |

Большинство «реальных» матриц в ML — это **комбинации** этих базовых
трансформаций. Любую сложную трансформацию можно разложить на эти —
тема **матричных разложений** (SVD, PCA, QR), которая будет в фазе 2.

**Глубокая нейросеть — серия трансформаций**

```
v₀ → W₁ → v₁ → W₂ → v₂ → W₃ → v₃ → ... → output
```

Каждый слой **двигает точки в пространстве**. Хорошо обученная сеть
выстраивает их так, чтобы похожие объекты собрались рядом, а разные
— далеко. И тогда финальное решение (классификация, генерация
следующего слова, рекомендация) становится тривиальным — буквально
«возьми ближайшие точки».

«Обучение» сети — это **поиск матриц W₁, W₂, ...**, при которых эти
трансформации делают данные удобными для финальной задачи.

**Композиция трансформаций**

`M₁ @ M₂ @ v` означает «применить M₂ к v, потом применить M₁ к
результату». Чтение справа налево. Геометрически — две трансформации
подряд.

Если **две матрицы перемножить заранее** — `M_total = M₁ @ M₂` —
получится **одна матрица**, эквивалентная их композиции. Это и есть
**матрица × матрица** (отдельным блоком не разбирали, но идея уже
в голове).

**CRM-перспектива**

В будущих проектах:
- **Эмбеддинг клиента** (профиль → 768-мерный вектор) — результат
  серии матричных трансформаций входных токенов через слои LLM.
- **Семантический поиск похожих клиентов** — сравнение векторов
  **после** трансформации (через cosine similarity).
- **Триггерный агент** — серия трансформаций контекста разговора в
  действие.

Везде под капотом одна геометрическая идея: **векторы движутся через
пространство по правилам матриц**.

## Часть IX. Линейная независимость, базис, Gram-Schmidt

### Зачем — реальная проблема в CRM-данных

Представь, что собираешь признаки для модели churn-prediction.
Среди фич:

- `yearly_orders` — заказов в год,
- `monthly_orders` — заказов в месяц,
- `avg_orders_per_week` — заказов в неделю.

Логично — кажется, три разных угла на активность клиента. Но:

```
monthly_orders     = yearly_orders / 12
avg_orders_per_week = yearly_orders / 52
```

Все три фичи говорят одно и то же, просто в разных масштабах.
Если подсунуть их в линейную регрессию, модель **сломается** — не
упадёт с ошибкой, а вернёт **нестабильные веса**: один прогон → одни
числа, следующий прогон на тех же данных → совсем другие. Это
**multicollinearity (мультиколлинеарность)** — одна из самых частых
ловушек в реальном ML.

**Линейная алгебра ровно про это** даёт строгий ответ. Три понятия —
**линейная независимость**, **базис**, **ранг** — это математический
язык для трёх вопросов:

1. Какие фичи реально несут новую информацию?
2. Какой минимальный набор фич описывает данные без потери?
3. Можно ли из «грязного» набора сделать «чистый» — где все фичи
   перпендикулярны и масштабированы одинаково?

### Линейная зависимость — определение

**Вектор `v₃` линейно зависим** от `v₁` и `v₂`, если его можно
построить как **линейную комбинацию**:

```
v₃ = a·v₁ + b·v₂           (для каких-то чисел a, b)
```

**Набор векторов линейно независим**, если ни один из них не
выражается через остальные. Каждый несёт **уникальное направление**.

В CRM-примере выше набор `{yearly_orders, monthly_orders,
avg_orders_per_week}` — **линейно зависимый**. В наборе «3 фичи», но
**реально полезной информации — на одну фичу**.

### Критическая граница: линейное vs нелинейное

Линейная комбинация = **только сложение и умножение на константу**.
Никаких произведений переменных, корней, логарифмов.

| Связь | Линейна? |
|---|---|
| `B = 2·A` | да |
| `B = -3·A + 5·C` | да |
| `B = A / 12` (деление на константу) | да (`= (1/12)·A`) |
| `B = A · C` (произведение переменных) | **нет** |
| `B = A²` | **нет** |
| `B = log(A)` | **нет** |
| «`B` коррелирует с `A` на 0.9» | **нет** (это статистика, не алгебра) |

**Правило:** если в формуле только **константы и одна переменная** и
операции `+ − · /` — это линейно. Если рядом с переменной стоит **ещё
одна переменная** (либо переменная под нелинейной функцией) —
нелинейно.

⚠️ **Multicollinearity в широком смысле** (термин из эконометрики)
включает оба случая — и строгую линейную зависимость, и нелинейную
избыточность. **Linear dependence в строгом смысле** — только первый.
Когда читаешь «столбцы матрицы линейно независимы» — это не значит
«фичи никак не связаны». Это значит конкретно: «ни одну фичу нельзя
получить из других через **сумму с коэффициентами**».

### Базис — минимальный набор для построения всего

**Базис пространства** — минимальный набор линейно независимых
векторов, из которых **любой** вектор пространства можно построить
как линейную комбинацию.

**Стандартный базис 3D:**

```
e₁ = [1, 0, 0]
e₂ = [0, 1, 0]
e₃ = [0, 0, 1]
```

Любая точка раскладывается уникально:

```
[3, 2, 5] = 3·e₁ + 2·e₂ + 5·e₃
```

Числа `[3, 2, 5]` — это **координаты** точки в этом базисе. То
самое, что ты с детства называешь «координатами», математически —
это **коэффициенты разложения по базису**.

**Главный факт:** количество векторов в базисе = размерность
пространства.

- 3D → базис из 3 векторов.
- 768D (типичный эмбеддинг BERT) → 768.
- N-мерное → N.

### Ранг матрицы и реальная размерность данных

**Ранг матрицы (rank)** = количество линейно независимых столбцов =
размер базиса = **«реальная размерность» твоих данных**.

Если матрица фичей `X` формы `(1000, 50)`:

- Все 50 фичей независимы → `rank = 50` → данные живут в 50D.
- Среди 50 есть 10 «производных» (через деление на константу) →
  `rank = 40` → данные реально в 40D, втиснутом в 50D «выкройку».

**Следствия в ML:**

| Состояние | Что происходит с моделью |
|---|---|
| `X.rank == n_features` (full rank) | Регрессия решается, веса стабильны |
| `X.rank < n_features` (rank deficient) | `(X.T @ X)` вырождена — формула регрессии падает |
| Почти-rank-deficient (сильная корреляция) | Веса нестабильны, нужна регуляризация (Ridge, Lasso) |

**PCA** (фаза 2) — алгоритм, который ищет «оптимальный базис данных»:
k наиболее значимых независимых осей, в которых данные «реально
живут». Это сжатие размерности — основа очень многих ML-пайплайнов.

### Смена базиса — одна точка, разные координаты

Базисов для одного пространства **бесконечно много**:

```
{[1, 0], [0, 1]}      ← стандартный
{[-1, 0], [0, -1]}    ← повёрнутый на 180°
{[1, 1], [1, -1]}     ← повёрнутый на 45°
{[5, 0], [0, 3]}      ← растянутый
{[2, 1], [1, 3]}      ← произвольный (главное — независимы)
```

**Что меняется при смене базиса:**

- Координаты одной и той же точки **разные** в разных базисах.
- Сам объект (точка/вектор) — **тот же**.

Пример. Точка `[3, 2]` в стандартном базисе. Хотим её координаты в
базисе `{[1, 1], [1, -1]}`:

```
[3, 2] = a·[1, 1] + b·[1, -1] = [a+b, a-b]

=>  a + b = 3,    a - b = 2
=>  a = 2.5,      b = 0.5
```

Та же точка в новом базисе имеет координаты `(2.5, 0.5)`. **Физически
точка не сдвинулась — изменились «линейки», через которые мы её мерим.**

**Зачем смена базиса в ML:**

- **PCA** — находит «оптимальный» базис данных, в котором первые
  координаты несут больше всего информации. Снижение размерности.
- **Эмбеддинги** — внутри слоёв нейросети происходит постоянная
  смена базиса. Каждый слой ищет базис, в котором семантически
  близкие объекты оказались рядом.
- **Attention в трансформерах** — Q/K/V матрицы это проекции в три
  разных «удобных» базиса.

### Ортонормированный базис — золотой стандарт

Среди бесконечного числа возможных базисов есть особо удобные:

1. **Векторы перпендикулярны друг другу** (ортогональны,
   `dot(eᵢ, eⱼ) = 0` при `i ≠ j`).
2. **Каждый вектор имеет длину 1** (нормирован).

Такой базис называется **ортонормированным**.

**Почему он удобен:**

- **Координаты вычисляются просто** — `coord_i = dot(точка, e_i)`.
  Без решения систем уравнений.
- **Длины и углы сохраняются** при работе в нём — никаких искажений.
- **Численная устойчивость** алгоритмов на порядки лучше — нет
  деления на близкие к нулю числа, нет накопления ошибок в `√`.

Стандартный базис `{[1, 0, 0], [0, 1, 0], [0, 0, 1]}` —
ортонормированный по построению.

### Gram-Schmidt — алгоритм построения ортонормированного базиса

**Gram-Schmidt процесс** — это **алгоритм, который из любого набора
линейно независимых векторов делает ортонормированный**.

**Идея в трёх шагах** (для трёх векторов `v₁, v₂, v₃`):

1. Берём `v₁`. Нормализуем (делим на длину). Это первый базисный
   вектор `e₁`.
2. Берём `v₂`. Из него **вычитаем «тень» (проекцию) на `e₁`** —
   получаем компоненту, перпендикулярную `e₁`. Нормализуем — это
   `e₂`.
3. Берём `v₃`. Вычитаем «тени» на `e₁` и `e₂`. Нормализуем — `e₃`.

Каждый шаг «выпрямляет» новый вектор так, чтобы он был перпендикулярен
всем уже построенным. **Алгоритм от руки не пишем** — в numpy есть
`np.linalg.qr(X)` (QR-разложение), которое делает то же самое
численно стабильнее. Под капотом современных реализаций — модификации
Gram-Schmidt'а.

### Где это в реальном ML — связки

- **QR-разложение** (`np.linalg.qr`) — используется в линейной
  регрессии **вместо** «голой» формулы `(X.T @ X)⁻¹ @ X.T @ y` для
  **численной устойчивости**.
- **PCA** — алгоритм опирается на ортогональные базисы (не Gram-Schmidt
  напрямую, а через собственные векторы, но тот же дух).
- **Attention в трансформерах** — Q/K/V проекции в разные удобные
  базисы.
- **LoRA (Low-Rank Adaptation)** — техника file-tuning'а LLM, прямое
  применение идеи **низкого ранга**. Гипотеза LoRA: обновления весов
  `ΔW` при дообучении имеют **низкий ранг** (живут в узком
  подпространстве), поэтому их можно разложить как
  `ΔW ≈ A @ B`, где `A` (`d × r`) и `B` (`r × d`) — тонкие матрицы с
  маленьким `r` (обычно 8, 16, 32). Это сокращает количество
  **тренируемых** параметров в десятки-сотни раз (исходную матрицу
  `W` замораживаем, не трогаем). Подробно — в фазе 11.

### Главные мысли по разделу

1. **Линейная зависимость** — про сумму с коэффициентами, не про
   произведения переменных, не про статистическую корреляцию.
2. **Ранг матрицы фичей** = реальная размерность данных. Часто
   меньше числа столбцов в таблице.
3. **Базисов бесконечно много**. Смена базиса меняет координаты,
   но не сами объекты.
4. **Ортонормированный базис** — самый удобный тип. Gram-Schmidt —
   алгоритм перехода к нему.
5. **В практике ML руками Gram-Schmidt не пишут** — есть
   `np.linalg.qr`, PCA, и т.д. Но **понимать, что внутри** этих
   функций — уровень интуиции, который отделяет «вызываю функцию»
   от «понимаю, что она делает».

## Что дальше

В этой лекции пройдены **векторы, матрицы и основы линейной алгебры
для ML**. Темы, на которые это естественно вырастает дальше:

- **Тензоры** — обобщение векторов и матриц на любое число измерений.
  Базовая структура данных в PyTorch и TensorFlow. Картинка-батч в CV
  — это 4D-тензор `(batch, channels, height, width)`. Будут в фазе 3
  (Deep Learning Core).
- **Автоматическое дифференцирование (autodiff)** — как нейросети
  учатся, вычисляя градиенты через цепочки матричных операций
  (`.backward()` в PyTorch). Фаза 3.
- **Матричные разложения** — SVD, Cholesky, собственные значения.
  Практические техники для PCA, регрессии, LoRA. Фаза 2.

Главное, что нужно унести с собой:

1. **Вектор — это и точка, и стрелка одновременно.**
2. **Сложение и вычитание — покоординатно, скаляр — это растяжение.**
3. **Скалярное произведение измеряет «насколько одно направление», но
   зависит от длин — это его слабость как метрики сходства.**
4. **Косинусное сходство решает эту проблему, нормируя на длины.**
5. **Перед метриками сходства признаки должны быть на одной шкале —
   иначе доминирующий признак съест все остальные.**
6. **Float — приближённый тип. Не сравнивать через `==`.**
7. **Матрица — это и таблица данных, и функция-трансформация.** Один
   и тот же объект с двумя интерпретациями, в зависимости от
   контекста (`X` — таблица, `W` — функция).
8. **Матрица × вектор (`M @ v`) — главная операция ML.** Каждая
   строка матрицы через dot product превращается в одно число; на
   выходе вектор длины = числу строк. На этом построены линейная
   регрессия, каждый слой нейросети, attention в трансформерах.
9. **`@` ≠ `*`.** Первое — матричное умножение со схлопыванием оси.
   Второе — поэлементное умножение, ничего не схлопывает. Путать
   нельзя — синтаксически оба корректны, результаты разные.
10. **Ранг матрицы фичей = реальная размерность данных.** Часто
    меньше числа столбцов в таблице — и тогда линейная регрессия
    ломается на вырожденной `(X.T @ X)`.
11. **Линейная зависимость — это сумма с коэффициентами**, не
    произведения переменных и не статистическая корреляция. Узкий
    математический критерий, не интуитивное «фичи похожи».
12. **Базисов в одном пространстве бесконечно много.** Смена базиса
    меняет координаты, но не сами объекты. PCA ищет «оптимальный»
    базис для конкретных данных; LoRA опирается на низкий ранг
    обновлений весов.
