# Линейная алгебра — интуиция

> Каждая AI-модель — это просто матричная математика в красивой шляпе.

Эта лекция — введение в линейную алгебру через её главные образы:
вектор, операции над векторами, метрики сходства. От самого простого
определения «список чисел» до того, как одна формула косинусного сходства
лежит в основе рекомендательных систем, семантического поиска и attention
в трансформерах.

Цель — **геометрическая интуиция**, а не строгие доказательства. После
прохождения должна остаться картинка в голове: что́ означает каждая
операция, и почему именно она встречается в AI.

## Зачем это вообще нужно

Откройте любую современную статью по машинному обучению — на первой странице
будут векторы, матрицы, скалярные произведения, преобразования. Без интуиции
в линейной алгебре это просто символы. С интуицией — становится видно, что́
на самом деле делает нейронная сеть: она двигает точки в многомерном
пространстве.

Базовые применения векторных операций в AI:

- **Embeddings** — каждое слово, изображение, документ превращаются в
  длинный вектор чисел. Семантическая близость объектов становится
  геометрической близостью векторов.
- **Attention** в трансформерах — вычисление «насколько каждое слово
  важно для предсказания следующего» через массивы скалярных произведений.
- **RAG (поиск-дополненная генерация)** — поисковый компонент работает
  через скалярные произведения между вектором запроса и векторами
  документов в базе.
- **Градиентный спуск** — каждый шаг обучения это вычитание векторов
  и умножение их на скаляр.
- **Рекомендательные системы, кластеризация клиентов, сегментация** —
  всё через метрики сходства между векторами.

То есть линейная алгебра — это **общий язык** всей современной AI. Без
её понимания дальше двигаться можно, но каждое следующее явление будет
казаться магией.

## Часть I. Вектор

### Что такое вектор

**Вектор — это просто список чисел.** Например, `[3, 2]`. Это всё. Никакой
магии.

Интересное начинается с того, **что эти числа означают**. У вектора есть
два эквивалентных прочтения:

1. **Как точка в пространстве.** Для `[3, 2]` это точка с координатами
   (x=3, y=2) на плоскости.
2. **Как стрелка.** Та же точка, но представленная стрелкой из начала
   координат `(0, 0)` до этой точки. У стрелки есть **направление**
   (куда смотрит) и **длина**.

Оба прочтения описывают один и тот же объект. Геометры чаще говорят
«стрелка», программисты — «точка», физики — «вектор величины». Это вопрос
акцента, не сущности.

```
y
↑
3 |
2 |       • [3, 2]
1 |      ╱
0 +────────→ x
  0  1  2  3
```

В Python вектор естественно представляется списком чисел:

```python
v = [3, 2]
```

В научных вычислениях используют `numpy.array`, в боевом ML — `torch.Tensor`,
но идея та же.

### Размерность вектора

**Размерность вектора = количество чисел в нём.**

```
[3, 2]              → двумерный (2D) вектор
[1, -2, 3, 0, 7]    → пятимерный (5D) вектор
[a₁, a₂, ..., a₇₆₈] → 768-мерный
```

В Python размерность считается просто:

```python
def dimension(v):
    return len(v)
```

Что меняется при росте размерности:

- **2D** — рисуется на плоскости (лист в клеточку), видим глазами.
- **3D** — представляем как куб, тоже видим, чуть труднее.
- **4D и выше** — **нарисовать невозможно.** Человеческий мозг не
  приспособлен. Но математика работает абсолютно так же — все формулы
  обобщаются на любую размерность.

Это **главный сдвиг в голове**: геометрическая интуиция остаётся, даже
когда наглядная картинка пропадает. В современной AI оперируют векторами
размерности 768, 1024, 4096 — никто их не рисует, но **думают** про них
всё ещё как про точки и стрелки в пространстве.

**Откуда такие большие размерности:**

- Word embeddings BERT — 768 чисел на одно слово.
- OpenAI `text-embedding-3-large` — 3072 числа на кусок текста.
- LLaMA, GPT внутри держат активации размерности 4096 и больше.

**Бизнес-аналогия.** В CRM-системе клиент описывается несколькими
признаками: количество заказов, средний чек, NPS, давность последней
покупки, возраст. Если признаков три — клиент это вектор в 3D. Если
семь — в 7D. ML просто доводит эту мысль до предельного состояния:
«опишем объект 768 числами, пусть модель сама разберётся, что они
значат».

## Часть II. Операции над векторами

### Сложение и вычитание

**Главное правило:** сложение и вычитание определены **только для векторов
одинаковой размерности.** Эти операции **не существуют** для векторов
разных размерностей. Никакого «дополним нулями и сложим» — это совсем
другая операция, которая меняет смысл данных.

**Сложение покоординатно:**

```
[a₁, a₂] + [b₁, b₂] = [a₁+b₁, a₂+b₂]
```

Пример: `[3, 2] + [1, 4] = [4, 6]`.

**Геометрический смысл — «хвост к голове».** Берём первую стрелку из (0,0)
до (3,2). Ставим хвост второй стрелки в голову первой. Куда дотянется
вторая стрелка — там и сумма:

```
y
↑
6 |              • [4, 6] ← сумма
5 |            ╱
4 |    [1,4]╱
3 |        ╱
2 |   •  ↗
1 | ╱ [3,2]
0 +────────────→ x
  0 1 2 3 4
```

«Шаг A потом шаг B — куда придёшь».

**Вычитание** — то же самое покоординатно:

```
[a₁, a₂] - [b₁, b₂] = [a₁-b₁, a₂-b₂]
```

Геометрически — стрелка «от B до A». Если стоять в точке B, то вектор
`a - b` показывает направление до A. На этом стоят все алгоритмы,
которые работают с «разностью» — от движения по градиенту до сравнения
профайлов клиентов.

**В коде:**

```python
def add(a, b):
    if len(a) != len(b):
        raise ValueError(f"размерности не совпадают: {len(a)} vs {len(b)}")
    return [a_i + b_i for a_i, b_i in zip(a, b)]


def sub(a, b):
    if len(a) != len(b):
        raise ValueError(f"размерности не совпадают: {len(a)} vs {len(b)}")
    return [a_i - b_i for a_i, b_i in zip(a, b)]
```

Проверка размерностей через `raise ValueError` — обязательна.
Без неё `zip` молча обрежет до короткого списка, и функция вернёт
неполный, но «правдоподобный» результат. Это худший вид бага —
функция «работает», но данные испорчены без шума.

### Умножение на скаляр

**Скаляр — это просто число**, в отличие от вектора. Умножение вектора
на скаляр умножает каждую его координату:

```
k · [a₁, a₂] = [k·a₁, k·a₂]
```

**Геометрически:**

- `k > 0` — растягивание или сжатие стрелки. `k=2` → длина становится в
  2 раза больше, направление сохраняется. `k=0.5` → длина в 2 раза
  меньше.
- `k < 0` — разворот в противоположную сторону + масштабирование.
  `k=-1` → та же стрелка, но «назад».
- `k = 0` — нулевой вектор.

**В коде:**

```python
def scale(v, k):
    return [elem * k for elem in v]
```

Здесь проверка размерностей не нужна — вектор один, число одно.

### Скалярное произведение (dot product)

**Самая важная операция во всей линейной алгебре с точки зрения AI.**
На ней стоят:

- Attention в трансформерах.
- Поиск похожих документов в RAG.
- Косинусное сходство в рекомендательных системах и кластеризации.
- Каждый линейный слой нейронной сети — это серия скалярных
  произведений.

**Формула** (векторы одинаковой размерности):

```
a · b = a₁·b₁ + a₂·b₂ + ... + aₙ·bₙ
```

Перемножаем покоординатно, складываем результаты в **одно число**.

Пример: `[1, 2, 3] · [4, 5, 6] = 1·4 + 2·5 + 3·6 = 32`.

**Очень важно:** в отличие от сложения, результат — **одно число
(скаляр)**, не вектор. Отсюда английское название — *scalar product*.

**В коде:**

```python
def dot(a, b):
    if len(a) != len(b):
        raise ValueError(f"размерности не совпадают: {len(a)} vs {len(b)}")
    return sum(a_i * b_i for a_i, b_i in zip(a, b))
```

Внутри `sum` использован **generator expression** (без квадратных
скобок) — он не материализует промежуточный список в памяти, считает
сумму на лету. Для миллиона чисел это критично, для пяти — без разницы,
но идиома «sum от генератора» — каноническая.

### Геометрический смысл скалярного произведения

**Скалярное произведение измеряет, насколько векторы «смотрят в одну
сторону».** Знак результата говорит:

| Результат | Угол | Смысл |
|---|---|---|
| `a · b > 0` | < 90° | смотрят в одну сторону |
| `a · b = 0` | = 90° | перпендикулярны, независимы |
| `a · b < 0` | > 90° | смотрят в разные стороны |

Точная связь:

```
a · b = |a| · |b| · cos(θ)
```

где `θ` — угол между векторами, `|a|`, `|b|` — их длины. Из этой формулы
видно: результат зависит **и от направления (cos)**, **и от длин**.
Когда оба вектора длинные, любое скалярное произведение даёт большое
число — даже если направления совпадают плохо. Это создаёт проблему при
использовании скалярного произведения как «меры похожести» — см.
Часть IV.

## Часть III. Длина вектора

### Что такое длина

**Длина вектора (англ. magnitude, norm)** — насколько «далеко» он
тянется от начала координат.

```
|v| = √(v₁² + v₂² + ... + vₙ²)
```

Каждое число в квадрат, складываем, берём корень.

В 2D это **теорема Пифагора**: длина вектора — это гипотенуза
прямоугольного треугольника с катетами в виде его координат.

Пример: `|[3, 4]| = √(9 + 16) = √25 = 5`.

**Имена термина** (встретятся все три):

- **magnitude** — обиходный английский.
- **length** — «длина».
- **norm**, **L2 norm**, **Euclidean norm** — математически. «L2» —
  это конкретно «корень из суммы квадратов». «Euclidean» — потому что
  соответствует обычной геометрической длине из геометрии Евклида.

### Связь длины со скалярным произведением

**Важная связка:** длина выражается через скалярное произведение
вектора с самим собой.

```
v · v = v₁·v₁ + v₂·v₂ + ... = v₁² + v₂² + ... = |v|²
```

Значит:

```
|v| = √(v · v)
```

**В коде:**

```python
import math

def magnitude(v):
    return math.sqrt(dot(v, v))
```

Это и есть та реализация, что используется внутри `numpy.linalg.norm`.

Связка важна не только для краткости кода, но и **концептуально**: она
показывает, что длина — это всего лишь специальный случай скалярного
произведения. В будущем эта связь даёт косинусное сходство «почти
бесплатно».

### Поведение длины при масштабировании

**Если вектор умножить на число k, его длина умножается на |k|:**

```
|k · v| = |k| · |v|
```

Вывод формулы:

```
|[k·v₁, k·v₂]|² = (k·v₁)² + (k·v₂)²
                = k²·v₁² + k²·v₂²
                = k²·(v₁² + v₂²)
                = k²·|v|²
|[k·v₁, k·v₂]|  = |k|·|v|
```

Ключевой шаг — вынос `k²` за скобки из-под корня, после чего
`√(k²) = |k|`. Модуль, потому что для отрицательных `k` длина всё равно
положительна.

Это правило важно для следующей темы — масштабной инвариантности
косинусного сходства.

### Применения длины

- **Расстояние между двумя точками** — `|a - b|`. Применяется в KNN,
  кластеризации, обнаружении выбросов.
- **Нормализация вектора** — деление вектора на свою длину `v / |v|`
  даёт вектор длины 1, смотрящий в ту же сторону. Используется, когда
  важно только направление, не масштаб.
- **Метрики сходства** — косинусное сходство, евклидово расстояние.

## Часть IV. Сходство векторов

Самая практическая часть в линейной алгебре для AI и аналитики данных.

### Задача: «найди похожие объекты»

Типичный сценарий — есть база объектов (клиентов, документов, товаров),
каждый представлен вектором признаков. Нужно найти **похожие** на
заданный объект пары или группы.

**Первая мысль** — использовать скалярное произведение. Большое число
→ «похожи». Маленькое → «нет». Простая, прозрачная метрика.

### Проблема скалярного произведения как метрики сходства

Скалярное произведение **зависит от длин обоих векторов**:

```
a · b = |a| · |b| · cos(θ)
```

Если хотя бы один вектор имеет большие координаты — результат большой,
**даже если направление не совпадает**. Это создаёт системную ошибку
для метрики сходства.

**Пример.** Пусть в CRM-базе есть три клиента, описанных вектором
`[количество заказов в год, средний чек]`:

```
A:      [50, 8000]       # крупный клиент
100·A:  [5000, 800000]   # буквально A, увеличенный в 100 раз — то же направление, но длиннее
B:      [10, 3000]       # совсем другой клиент
```

`100·A` геометрически — это **тот же тип клиента** (то же направление в
пространстве), просто числа в 100 раз больше. Эту «масштабированную копию»
ожидается видеть **одинаково похожей** на A, что и A сам на себя.

Скалярные произведения:

```
A · A      =     64 002 500
A · 100·A  =  6 400 250 000      ← в 100 раз больше, чем A на самого себя
A · B      =     24 000 500
```

Если интерпретировать «больше = более похожи», получается:

> «100·A более похож на A, чем сам A на себя.»

Это **абсурд**. A сам на себя похож на 100% — это эталон сходства, никто
не может быть «более похож на A». Скалярное произведение этого правила
**не соблюдает** — оно зависит от длин, и любое масштабирование
исходного вектора умножает результат, имитируя «супер-сходство».

### Косинусное сходство (cosine similarity)

**Решение проблемы:** разделить скалярное произведение на произведение
длин обоих векторов. Это убирает влияние масштаба — остаётся **только
направление**.

```
cos_sim(a, b) = (a · b) / (|a| · |b|)
```

**Откуда название.** Из формулы `a · b = |a| · |b| · cos(θ)`:

```
cos(θ) = (a · b) / (|a| · |b|)
```

То есть результат — **буквально косинус угла между двумя векторами**.

**В коде:**

```python
def cosine_similarity(a, b):
    if magnitude(a) == 0 or magnitude(b) == 0:
        raise ValueError("косинусное сходство не определено для нулевого вектора")
    return dot(a, b) / (magnitude(a) * magnitude(b))
```

Проверка от нулевого вектора нужна — длина нулевого вектора 0, деление
на ноль не определено.

### Свойства косинусного сходства

**Диапазон результата всегда `[-1, +1]`:**

| Значение | Угол | Смысл |
|---|---|---|
| `+1` | 0° | смотрят в одну сторону (полное сходство) |
| `0` | 90° | перпендикулярны (никакой связи) |
| `-1` | 180° | противоположны |

**Масштабная инвариантность (scale invariance) — главное свойство.**

Если умножить любой из векторов на положительное число — косинусное
сходство **не изменится**:

```
cos_sim(k·a, b) = (k·(a·b)) / (|k|·|a|·|b|) = (a·b) / (|a|·|b|) = cos_sim(a, b)
```

Для отрицательного `k` знак результата перевернётся (направление
становится противоположным).

В частности:

```
cos_sim(A, A)       = 1.0    # эталон
cos_sim(A, 100·A)   = 1.0    # та же стрелка по направлению
cos_sim(A, B)       < 1.0    # другое направление
```

То «супер-сходство», которое имитировало скалярное произведение, в
косинусном исчезает. Это и есть **метрика, измеряющая чисто направление,
безотносительно к размеру**.

### Перевод косинусного сходства в угол

Cosine — это **косинус**, и через арккосинус можно перевести в сам угол:

```python
import math
angle_deg = math.degrees(math.acos(cos_value))
```

Перевод в градусы даёт куда более интуитивный масштаб:

| cos | угол | смысл |
|---|---|---|
| 1.0 | 0° | одна стрелка |
| 0.97 | ≈ 13.6° | похожи, но различимы |
| 0.5 | 60° | заметно различные |
| 0 | 90° | независимые |
| -1.0 | 180° | противоположные |

### «Cos = 0.97 это много или мало?» — нет универсального ответа

Это **зависит от контекста** и распределения cosine во всей базе:

| Контекст | «Сильное сходство» начинается с |
|---|---|
| Word embeddings (BERT, OpenAI) | 0.7–0.8 |
| Профайлы клиентов в 2-3 признаках | 0.95+ |
| Высокоразмерные embeddings (768D+) | 0.5+ |

Чем выше размерность пространства, тем **реже** случайные пары близки →
даже средний cosine уже сигнал.

Практический приём:

1. Посчитать cosine между **всеми** парами в базе.
2. Построить гистограмму распределения.
3. Эмпирически выбрать порог (например, верхние 10% значений).

Одно значение cosine само по себе **ни о чём не говорит** — смысл
появляется только в сравнении с распределением.

### Применения косинусного сходства в AI

- **RAG (retrieval-augmented generation)** — запрос и каждый документ
  превращают в векторы. Сравнивают `cos_sim(запрос, документ)` для всей
  базы, возвращают топ-K. Это и есть «семантический поиск».
- **Рекомендательные системы** — товар × товар матрица косинусных
  сходств. «Кто покупал X, покупал и Y».
- **Word embeddings** — `cos_sim(vec("король"), vec("королева"))` будет
  высоким (≈ 0.7–0.9 в типичной модели), `cos_sim(vec("король"),
  vec("банан"))` — около 0.
- **Кластеризация клиентов** — нахождение групп с похожим паттерном
  поведения.

## Часть V. Подготовка данных к расчёту сходства

Просто посчитать косинусное сходство на сырых данных — обычно
**недостаточно**. На реальных данных вылезает важная ловушка, связанная
со шкалами признаков. Без её преодоления даже cosine начинает обманывать.

### Проблема разных шкал

У объекта в реальной задаче обычно несколько признаков **разной
природы и разных диапазонов**:

| Признак | Типичный диапазон |
|---|---|
| Количество заказов в год | 1 – 100 |
| Средний чек | 500 – 50 000 |
| NPS | 0 – 10 |
| Возраст | 18 – 80 |
| Дней с последней покупки | 1 – 365 |

Когда математика обрабатывает такого клиента как вектор, **каждый
признак вкладывается в результат пропорционально своему размеру**. Чек в
тысячах × тысячах = миллионы. NPS в десятках × десятках = сотни. Чек
будет весить в **тысячи раз больше** во всех формулах.

**Аналогия с жюри.** Представим пять экспертов, оценивающих клиентов.
Каждому дали свою шкалу:

- Эксперт А: 0–10.
- Эксперт Б: 0–100.
- Эксперт В: 0–10 000.

Если просто **сложить** оценки, кто будет решать? Эксперт В. Его «6 000»
всегда в тысячи раз больше любого «10» эксперта А, даже если для эксперта
А это была отличная оценка. Голоса А и Б **не слышны**.

Это и есть проблема разных шкал признаков. Без предварительной обработки
ML-алгоритм слушает только один признак — самый «громкий по числам».

**Иллюстрация на данных.** Возьмём двух клиентов:

```
A = [50, 8000]      # 50 заказов в год, средний чек 8000
B = [10, 3000]      # 10 заказов в год, средний чек 3000
```

Скалярное произведение:

```
A · B = 50·10 + 8000·3000 = 500 + 24 000 000
                            ↑      ↑
                            шум    весь сигнал
```

Заказы вкладывают 500, чек — 24 миллиона. Соотношение 1 : 48 000. Заказы
**не учитываются** в расчёте сходства.

Косинусное сходство в этом случае ≈ **0.99996** — обманчиво высокое
значение. Подумаешь — «практически идентичные клиенты», тогда как у A в
5 раз больше заказов и почти в 3 раза выше чек. Косинус «слышит» только
чек, для которого пропорции близки.

### Min-Max нормализация

Стандартное решение — **привести каждый признак к единой шкале**.
Простейший способ — Min-Max:

```
x_norm = (x - min) / (max - min)
```

После применения все значения признака лежат в `[0, 1]`. Минимум → 0,
максимум → 1, остальные равномерно между ними.

**Как запомнить формулу через логику:**

- **Числитель `x - min`** = «насколько выше минимума».
- **Знаменатель `max - min`** = «вся длина шкалы».
- **Деление** = «доля шкалы, пройденная от минимума».

Аналогия: «уровень 35 из 80» в видеоигре → прогресс `(35-0)/(80-0) =
43.75%`. То же самое для признака.

**Применим к примеру.** Пусть по всей базе:

```
max заказов в год = 100, min = 0
max средний чек   = 10 000, min = 0
```

```
A_norm = [50/100, 8000/10000] = [0.5, 0.8]
B_norm = [10/100, 3000/10000] = [0.1, 0.3]
```

**Все координаты теперь одного порядка**, между 0 и 1. Ни одна не
доминирует.

Перепроверяем скалярное произведение:

```
A_norm · B_norm = 0.5·0.1 + 0.8·0.3 = 0.05 + 0.24
                  ↑          ↑
                  1 часть    5 частей
```

Соотношение вкладов теперь **1 : 5** вместо **1 : 48 000**. Обе
координаты участвуют. Чек всё ещё немного важнее (числа больше), но это
уже не «диктатура одной координаты».

Косинусное сходство:

```
cos_sim(A_norm, B_norm) ≈ 0.972
```

Что в градусах — около 13.6°. **Реальное** сходство, учитывающее обе
координаты на равных. На фоне до-нормализационных 0.99996 (≈ 0.5°)
видно, что нормализация **многократно увеличила различимость** разных
клиентов.

### Когда нормализация нужна

| Алгоритм | Нужна? |
|---|---|
| KNN, K-means, SVM, нейросети, регрессия | Да |
| Метрики сходства (cosine, евклидово) | Да |
| Деревья решений, Random Forest, gradient boosting | Нет |
| Naive Bayes | Нет |

Грубое правило: если алгоритм работает через **расстояния** или
**градиенты** — нормализация обязательна. Если через **сравнения
«больше / меньше»** — нет (деревьям шкала не важна, они выбирают
точки разбиения).

### Другие виды нормализации

| Метод | Формула | Когда применять |
|---|---|---|
| **Min-Max** | `(x - min) / (max - min)` | Простой случай, шкала известна. Чувствителен к выбросам. |
| **Z-score (стандартизация)** | `(x - mean) / std` | Устойчив к выбросам. Среднее = 0, отклонение = 1. Самый частый в ML. |
| **L2-нормализация вектора** | `v / |v|` | Делает **весь вектор** длины 1. Не путать с предыдущими — это про вектор целиком, а не про отдельные признаки. |

**Min-Max vs L2-нормализация** — это **разные** операции под **разные**
цели:

- **Min-Max / Z-score** — между **признаками** (колонками таблицы). Цель:
  уравнять шкалы признаков, чтобы один признак не доминировал.
- **L2** — внутри **одного вектора** (одной строки таблицы). Цель:
  сделать стрелку длины 1, оставив только направление. Для случаев,
  когда нужно сравнивать **только направления**, не масштабы.

Обе могут применяться последовательно: сначала Min-Max между признаками
для уравнивания шкал, потом L2 для каждой строки.

### Feature weighting — взвешивание признаков под задачу

После нормализации можно пойти ещё дальше — **назначить весам разную
важность под конкретную бизнес-задачу**:

```python
weights = [w_orders, w_check, w_nps, ...]
v_weighted = [v[i] * weights[i] for i in range(len(v))]
cosine_similarity(v_weighted, u_weighted)
```

Это не «правильная» научная нормализация — это **бизнес-настройка
метрики**. Одна и та же база, **разные сегментации** под разные кампании:

| Цель | Что взвесить тяжелее |
|---|---|
| Up-sell на дорогую категорию | средний чек |
| Активация спящих клиентов | дни с последней покупки |
| Программа лояльности | NPS + стаж |

Веса работают как «усилитель координаты» в метрике. Усилили признак —
система сильнее реагирует на отличия по нему.

Формальные методы в ML делают это автоматически: **расстояние
Махаланобиса** учитывает корреляции признаков, **gradient boosting**
выдаёт **feature importance** на основе размеченных данных. Эти
importance можно использовать как веса для метрик сходства. Но
ручное взвешивание остаётся рабочим инструментом из-за прозрачности.

**Главная мысль про сходство:** «похожесть» — это **не объективная
штука**. Это **выбор**, что считать важным. Любая метрика сходства —
это функция твоих весов. Сделал веса равными → одна сегментация.
Поднял вес одного признака → другая. Этим и отличается «просто
похожи» от «похожи для конкретной задачи».

## Часть VI. Подводные камни вычислений

### Арифметика с плавающей точкой

Числа `float` хранятся в **двоичном формате** ограниченной длины (обычно
64 бита). **Большинство десятичных дробей не представимы в нём точно**
— так же, как `1/3` нельзя точно записать в десятичной (бесконечная
дробь 0.333...).

Классическая иллюстрация:

```python
print(0.1 + 0.2)        # 0.30000000000000004
print(0.1 + 0.2 == 0.3) # False
```

Это **не баг конкретного языка**. Это фундаментальная природа стандарта
IEEE 754 для float, который используют все языки (Python, C,
JavaScript, Java, NumPy, PyTorch).

**В векторных вычислениях** это вылезает везде, где есть деление, корни,
тригонометрия. Например, при подсчёте косинусного сходства двух
противоположных векторов теоретически должно получаться ровно −1, но на
практике получится `-0.9999999999999998` — из-за накопления
микро-ошибок в `√` и делении.

**Главное правило: НЕ сравнивать float через `==`.**

```python
# ❌ Никогда
if cos_sim == 1.0: ...

# ✅ Правильно
import math
if math.isclose(cos_sim, 1.0): ...
```

`math.isclose(a, b)` проверяет «достаточно ли близки» с допуском по
умолчанию `1e-9`. В NumPy и PyTorch есть аналоги — `np.allclose`,
`torch.allclose`.

**Где это критично в ML:**

- **Нормализация** — `|v| == 1` → `math.isclose(magnitude(v), 1.0)`.
- **Проверка ортогональности** — `dot == 0` → `abs(dot) < epsilon`.
- **Сходимость градиентного спуска** — `|gradient| < tolerance`.
- **Сравнение тензоров в тестах** — всегда через `allclose`.

**Косметика отображения.** Микро-хвосты в выводе скрываются через
формат:

```python
print(f"{x:.4f}")    # 4 знака после точки
```

Округление **только при отображении**, в памяти число остаётся прежним.

**Распространённая ловушка:** `1/3 ≠ 0.3`. Это **два разных числа**.

```
1/3 ≈ 0.333333...    (бесконечная дробь)
0.3 = 0.3             (конечная)
```

В коде всегда писать `1/3` (или `1/N`), а не «приближение от руки» `0.3`.

## Глоссарий

| Англ. | Рус. | Что это |
|---|---|---|
| vector | вектор | список чисел; точка или стрелка в пространстве |
| dimension | размерность | количество чисел в векторе |
| scalar | скаляр | просто число (не вектор) |
| dot product, scalar product | скалярное произведение | сумма поэлементных произведений; результат — одно число |
| magnitude, norm, length | длина, норма, магнитуда | `√(сумма квадратов координат)` |
| L2 norm, Euclidean norm | евклидова норма | классическая длина |
| cosine similarity | косинусное сходство | мера похожести через косинус угла |
| scale invariance | масштабная инвариантность | свойство «не меняется при умножении на положительное число» |
| feature normalization | нормализация признаков | приведение признаков к одной шкале (Min-Max, Z-score) |
| feature weighting | взвешивание признаков | назначение весов важности признакам |
| embedding | эмбеддинг, векторное представление | превращение объекта в вектор чисел |
| floating-point | арифметика с плавающей точкой | приближённое представление дробей в двоичном виде |
| epsilon (ε) | эпсилон | малая величина для сравнения float с допуском |

## Что дальше

Из освоенных операций над векторами естественно растут несколько
следующих тем:

- **Матрицы** — таблицы чисел, которые **трансформируют** вектор:
  поворачивают, масштабируют, проецируют. Один шаг нейронной сети — это
  матричное умножение. Вся «магия» Linear / Dense слоя — это умножение
  входного вектора на матрицу весов плюс смещение.
- **Линейная независимость, базис, ранг матрицы** — теоретическая
  основа для понимания, какие признаки «работают», а какие избыточны
  (multicollinearity в регрессии).
- **Проекция и Gram-Schmidt процесс** — основа линейной регрессии
  (метод наименьших квадратов), PCA (метода главных компонент), QR-
  разложения и многих численных методов.
- **Тензоры** — обобщение векторов и матриц на любое число измерений.
  Базовая структура данных в PyTorch и TensorFlow.
- **Автоматическое дифференцирование** — как нейросети учатся, вычисляя
  градиенты через цепочки матричных операций.

Каждая из этих тем — естественное продолжение того, что было в этой
лекции. Главное, что нужно унести с собой:

1. **Вектор — это и точка, и стрелка одновременно.**
2. **Сложение и вычитание — покоординатно, скаляр — это растяжение.**
3. **Скалярное произведение измеряет «насколько одно направление», но
   зависит от длин — это его слабость как метрики сходства.**
4. **Косинусное сходство решает эту проблему, нормируя на длины.**
5. **Перед метриками сходства признаки должны быть на одной шкале —
   иначе доминирующий признак съест все остальные.**
6. **Float — приближённый тип. Не сравнивать через `==`.**
