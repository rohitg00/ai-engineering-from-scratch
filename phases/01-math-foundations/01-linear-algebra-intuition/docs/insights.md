## Python-идиомы

### Распаковка списка (unpacking)

```python
x, y = vector  # вместо x = vector[0]; y = vector[1]
```

Слева — кортеж переменных, справа — итерируемая последовательность той же длины.
Если справа три элемента, слева две переменные — `ValueError`. Очень частый
приём в ML-коде: функции часто возвращают несколько значений через `return a, b`,
и снаружи их сразу распаковывают.

### `[0] * N` — список из N одинаковых элементов

```python
zeros = [0] * 768  # 768-мерный нулевой вектор
ones = [1.0] * 4   # [1.0, 1.0, 1.0, 1.0]
```

Используется для инициализации буферов, тензоров, заглушек заданной размерности.

### `zip` идёт по нескольким спискам параллельно

```python
[a_i + b_i for a_i, b_i in zip(a, b)]
```

На каждой итерации отдаёт кортеж `(a_i, b_i)` — пару соседних элементов. Чище,
чем `for i in range(len(a)): a[i] + b[i]`. По умолчанию **обрезает по короткой**
последовательности — это ловушка (см. ниже).

### Generator expression vs list comprehension

```python
# list comprehension — создаёт список в памяти
[x**2 for x in nums]

# generator expression — считает на лету, без хранения
sum(x**2 for x in nums)  # ← скобки от sum, не квадратные
```

Для агрегаций (`sum`, `max`, `min`, `any`, `all`) — generator expression
эффективнее, потому что не материализует промежуточный список. Для миллиона
чисел разница в памяти заметна.

### list vs tuple для вектора

| Тип | Изменяемый? | Семантика для вектора |
|---|---|---|
| `list` | да | удобно для алгоритмов, мутаций |
| `tuple` | нет | семантически чище — координаты не должны меняться |

На практике используют `list` (а в боевом ML — `numpy.array`). Но знать различие
важно: tuple — это «зафиксированная структура», list — «изменяемая
последовательность».

### `from typing import List` устарел

В Python 3.9+ встроенные типы умеют в дженерики напрямую:

```python
# ❌ старо
from typing import List, Dict, Tuple
def f(x: List[int]) -> Dict[str, int]: ...

# ✅ современно
def f(x: list[int]) -> dict[str, int]: ...
```

LSP (basedpyright) ругается жёлтым на `List` как deprecated.

### `print` ≠ `return`

`print` — побочный эффект (выводит на экран), функция при этом возвращает `None`.
`return` — отдаёт значение наружу, чтобы вызывающий код мог его использовать.

```python
# ❌ функция-тупик
def dimension(vec) -> int:
    print(len(vec))

# ✅ переиспользуемая функция
def dimension(vec) -> int:
    return len(vec)

print(f"размерность: {dimension(v)}")  # печать снаружи
```

LSP ловит несоответствие, если есть type hint `-> int`, но внутри нет `return`.

### `if __name__ == "__main__":`

Идиома, превращающая файл одновременно в **скрипт** и в **библиотеку**:

```python
def add(a, b): ...
def sub(a, b): ...

if __name__ == "__main__":
    # выполняется только при прямом запуске `python file.py`
    # при импорте `from file import add` — пропускается
    print(add([1, 2], [3, 4]))
```

`__name__` — специальная переменная Python:
- при прямом запуске = `"__main__"`
- при импорте = имя модуля (например, `"vector"`)

### Импорт между файлами в одной папке

```python
# crm_examples.py
from vector import add, sub, scale, dot, magnitude, cosine_similarity
```

Python ищет `vector.py` в той же директории, что и запускаемый скрипт.
Импортировать **только нужные функции**, не всё подряд — чище код, меньше
шумных зависимостей.

### Опечатки `1` vs `i` в моноширинных шрифтах

```python
return [a_i + b_1 for a_i, b_i in zip(a, b)]
              ↑                      ↑
            b_1                    b_i
```

Цифра «один» и буква «i» **почти идентичны** в большинстве кодовых шрифтов.
LSP-подсветка «Undefined name» спасает.

### f-string форматирование

```python
f"{x:.4f}"      # 4 знака после точки
f"{x:.2f}"      # 2 знака
f"{x:,.0f}"     # с разделителями тысяч, без дробной
f"{x:.2%}"      # как процент
```

Округление **только при печати**, в памяти число остаётся прежним.

## Защитное программирование

### Silent corruption — молчаливая порча данных

Самый опасный вид багов: функция «работает», но при некорректных входах
возвращает мусор без падения. Пример:

```python
def add(a, b):
    return [a_i + b_i for a_i, b_i in zip(a, b)]

add([1, 2, 3], [10, 20])  # → [11, 22], третий элемент потерялся МОЛЧА
```

Решение — **явная проверка + явное падение**:

```python
def add(a, b):
    if len(a) != len(b):
        raise ValueError(f"размерности не совпадают: {len(a)} vs {len(b)}")
    return [a_i + b_i for a_i, b_i in zip(a, b)]
```

Принцип: функция должна **отказаться работать** при некорректных входах, а не
выдать что-то «похожее на правду».

### `raise ValueError` vs `raise TypeError`

| Когда | Тип |
|---|---|
| Тип аргумента правильный, но значение не подходит | `ValueError` |
| Тип аргумента неправильный (передали str вместо list) | `TypeError` |
| Что-то редкое и специфичное | свой подкласс `Exception` |

Стандартное разделение, на нём держится Python-экосистема.

### `zip(a, b, strict=True)` — Python 3.10+

Заставляет `zip` самому проверять одинаковые длины и кидать `ValueError`,
если не совпадают:

```python
list(zip([1, 2, 3], [10, 20], strict=True))
# ValueError: zip() argument 2 is shorter than argument 1
```

Можно использовать вместо ручной проверки **или** как «двойную страховку»
вместе с ней.

### Чтение traceback

Traceback в Python пишется **сверху вниз по порядку вызовов**, поэтому самое
последнее (и самое важное) сообщение находится **внизу**.

```
Traceback (most recent call last):
  File "main.py", line 43, in <module>
    result = add([1.0, 2.0, 3.0], [10.0, 20.0])    ← кто вызвал
  File "vector.py", line 23, in add
    raise ValueError(...)                          ← где упало
ValueError: разные размерности векторов: 3 и 2     ← что упало
```

Читать снизу вверх: тип ошибки и сообщение → строка, где `raise` → откуда вызвано.

### DRY (Don't Repeat Yourself) — но не слишком рано

Дублирующийся код — кандидат на вынос в общую функцию. Но **не сразу**:
правило большого пальца — выноси на **третьем** повторе, не на втором.
Преждевременная абстракция (два разных места завязаны на одну функцию,
а потом расходятся) дороже, чем дублирование.

## Floating-point — арифметика с плавающей точкой

### Главное правило: НЕ сравнивать float через `==`

Числа float хранятся в двоичном виде ограниченной длины, и большинство
десятичных дробей **не представимы точно**:

```python
0.1 + 0.2 == 0.3   # False
0.1 + 0.2          # 0.30000000000000004
```

Правильное сравнение — через допуск:

```python
import math
math.isclose(a, b)                  # допуск ≈ 1e-9 по умолчанию
math.isclose(a, b, rel_tol=1e-6)    # настроить допуск

# или вручную
abs(a - b) < 1e-9
```

В NumPy/PyTorch — `np.allclose`, `torch.allclose`. Те же `rtol` (relative
tolerance) и `atol` (absolute tolerance).

### `1/3 ≠ 0.3`

Это два **разных числа**. `1/3 ≈ 0.333...` (бесконечная дробь), `0.3` — конечная.
В коде всегда писать `1/3` (или `1/N`), а не «приближение от руки» `0.3`.

```python
print(f"{0.3:.20f}")     # 0.29999999999999998890
print(f"{1/3:.20f}")     # 0.33333333333333331483
print(0.3 == 1/3)        # False
```

Хвосты разные, потому что числа разные.

### Когда float-хвост виден, а когда нет

```python
print(0.3)            # 0.3        — Python показывает кратчайшее представление
print(f"{0.3:.20f}")  # 0.299...   — точная двоичная аппроксимация
```

При обычной печати Python ищет **самый короткий** литерал, который при чтении
обратно даст тот же float. Поэтому «0.3» в `print(0.3)` — это не означает, что
в памяти ровно 0.3.

### Когда scale invariance срабатывает «точно»

```python
cosine_similarity(a, scale(a, 100))  # == 1.0 ровно
cosine_similarity(a, b)              # 0.99996... — хвост
```

В первом случае множитель 100 сокращается алгебраически в числителе и
знаменателе, float не успевает накопить ошибку. Во втором — каждая операция
(`√`, `*`, `/`) добавляет микро-ошибку.

## LSP и инструменты разработки

### LSP-предупреждения vs ошибки

| Цвет | Что значит |
|---|---|
| Красный | Ошибка — синтаксис не валидный или серьёзная проблема |
| Жёлтый | Предупреждение — код работает, но стиль/потенциальная проблема |
| Зелёный/Синий | Информация — подсказка |

Жёлтое предупреждение не ломает код. Можно жить с ним, исправлять опционально.

### LSP может ошибаться (false positives)

`basedpyright` запускается из корня проекта и не всегда корректно находит
локальные модули в подпапках. Например, в `crm_examples.py` импорт
`from vector import ...` может подсветиться «Cannot find module `vector`»,
хотя файл `vector.py` лежит рядом. **Это статический анализ**; при запуске
кода Python автоматически добавит директорию скрипта в `sys.path` и импорт
сработает.

Подавить предупреждение можно построчно:

```python
from vector import add  # pyright: ignore[reportMissingImports]
```

### Подавление правила линтера на конкретной строке

Универсальный приём:

```python
# Ruff
some_code  # noqa: B905

# pyright / basedpyright
some_code  # pyright: ignore[reportMissingImports]
```

`noqa` = «no quality assurance» — «знаю, осознанно игнорирую». Желательно рядом
коротко пояснить, почему.

### Авто-форматирование при сохранении

Если настроен Ruff в режиме форматтера (через `conform.nvim` в AstroNvim или
аналогично), при сохранении файла автоматически:

- Расставляются два пустых перевода строки между функциями верхнего уровня
  (PEP 8, правило E302).
- Убираются хвостовые пробелы.
- Импорты сортируются (если включено).

Можно настроить автоматическое удаление неиспользованных импортов
(`unused-imports autofix`).

## Математическая интуиция

### Длина вектора через `dot(v, v)`

```
|v|² = v · v
|v|  = √(v · v)
```

В коде это две строки, переиспользующие `dot`:

```python
def magnitude(v):
    return math.sqrt(dot(v, v))
```

Внутри `numpy.linalg.norm` ровно эта связка.

### Длина масштабированного вектора

```
|k · v| = |k| · |v|
```

Если умножить вектор на число `k`, длина умножается на `|k|` (модуль). Не на
корень из `k`, не на `k²`.

Вывод:
```
|[k·v₁, k·v₂]|² = (kv₁)² + (kv₂)²
                = k²·v₁² + k²·v₂²
                = k²·(v₁² + v₂²)
                = k²·|v|²
|[k·v₁, k·v₂]|  = |k|·|v|
```

### Cosine similarity как нормализованный dot product

```
cos_sim(a, b) = (a · b) / (|a| · |b|)
```

Это просто `dot product` двух единичных векторов в направлении `a` и `b`.
Отсюда:

- Диапазон строго `[-1, +1]`.
- Не зависит от длин исходных векторов.
- Равно косинусу угла между ними.

### Перевод cosine в угол

```python
import math
angle_deg = math.degrees(math.acos(cos_value))
```

Угол — куда более интуитивная единица:

| cos | угол | смысл |
|---|---|---|
| 1.0 | 0° | одна стрелка |
| 0.97 | 13.6° | похожи, но различимы |
| 0 | 90° | независимы |
| -1.0 | 180° | противоположны |

Один и тот же `cos = 0.99996` может скрывать **0.5°** (практически параллельны)
или **обманчивую близость** из-за доминирующей координаты.

### Композиция функций — изнутри наружу

```python
average = scale(add(add(a, b), c), 1/3)
```

Читать **с самого вложенного** к внешнему: сначала `add(a, b)` → потом
`add(того, c)` → потом `scale(итог, 1/3)`. Та же логика, что в формуле
`MSE = mean(square(predictions - targets))`.

Альтернатива — через промежуточные переменные с именами, для читаемости:

```python
total = add(add(a, b), c)
average = scale(total, 1/3)
```

## Метрики сходства

### Скалярное произведение **зависит от длин**

`a · b = |a| · |b| · cos(θ)`

Если хоть один вектор «длинный» (большие числа), результат большой —
**безотносительно направления**. Это значит, что VIP-клиент с большими
числами по dot product «выглядит похожим» вообще на всех. Это **не подходит**
как метрика сходства.

### Доминирование признака с большой шкалой

Если в векторе одна координата на порядки больше остальных (например, чек в
тысячах, а NPS в десятках), все формулы сходства (dot, cosine, евклидово
расстояние) **слышат только её**. Остальные становятся шумом.

```
A = [50, 8000]
B = [10, 3000]

A · B = 50·10 + 8000·3000 = 500 + 24 000 000
                            ↑      ↑
                            шум    весь сигнал

cos(A, B) ≈ 0.99996  ← обманчиво высокое значение
```

Решение — **нормализация признаков** до общей шкалы. После Min-Max:

```
A_norm = [0.5, 0.8]
B_norm = [0.1, 0.3]
cos(A_norm, B_norm) ≈ 0.97  ← честное значение, обе координаты участвуют
```

### Нет универсального порога «cos = ?» = «похожи»

Шкала «похожи / не похожи» зависит от **распределения cosine во всей базе**.

| Контекст | «Сильное сходство» начинается с |
|---|---|
| Embeddings слов (BERT, OpenAI) | 0.7 – 0.8 |
| Профайлы клиентов в 2-3D | 0.95+ |
| Высокоразмерные embeddings (768D+) | 0.5+ |

В реальной задаче нужно построить **гистограмму распределения cosine** между
парами и выбрать порог эмпирически (например, верхние 10%).

### Min-Max нормализация — формула через логику

```
x_norm = (x - min) / (max - min)
```

- **Числитель `x - min`** = «насколько выше минимума».
- **Знаменатель `max - min`** = «вся длина шкалы».
- **Деление** = «доля от шкалы».

Аналогия: «уровень 35 из 80» = `(35-0)/(80-0) = 43.75%` прогресса.

### Min-Max vs L2-нормализация — разные операции

| Тип | Что нормализуется | Цель |
|---|---|---|
| **Min-Max / Z-score** | Каждый **признак** (колонка таблицы) отдельно | Уравнять шкалы признаков между собой |
| **L2-нормализация** | Каждый **вектор** целиком (строка таблицы) | Сделать вектор длины 1, оставив только направление |

Это **разные** операции под **разные** цели. Min-Max — про подготовку фичей.
L2 — про переход «к чистому направлению».

### Когда нормализация нужна, когда — нет

| Алгоритм | Нужна? |
|---|---|
| KNN, K-means, SVM, нейросети, регрессия | Да |
| Метрики сходства (cosine, евклидово) | Да |
| Деревья решений, Random Forest, gradient boosting | Нет |
| Naive Bayes | Нет |

Грубое правило: если алгоритм работает через **расстояния** или
**градиенты** — нормализация обязательна. Если через **сравнения «больше / меньше»** — нет.

### Feature weighting под бизнес-задачу

После нормализации можно умножить каждую координату на свой **вес важности**:

```python
weights = [w_orders, w_check, w_nps, ...]
v_weighted = [v[i] * weights[i] for i in range(len(v))]
cosine_similarity(v_weighted, u_weighted)
```

Маркетинговый смысл: одна метрика, **разные сегментации** под разные кампании.
Up-sell на дорогую категорию → вес чека больше. Win-back → вес давности больше.

Формальные методы в ML (Mahalanobis distance, learned feature importance из
gradient boosting) автоматизируют этот выбор. Ручное взвешивание — рабочий
быстрый инструмент с явным маркетинговым смыслом.

## CRM-аналогии для математических операций

Все операции над векторами имеют **прямой бизнес-смысл**, если воспринимать
клиента как вектор признаков.

| Операция | Бизнес-вопрос |
|---|---|
| `sub(A, B)` | На сколько A отличается от B? Куда нужно дорасти? |
| `add` + `scale(1/N)` | Какой средний (эталонный) клиент в группе? (центроид) |
| `scale(v, k)` | Что если клиент вырастет в k раз? Прогноз. |
| `dot(A, B)` | Базовая «похожесть», но смешивает направление и размер |
| `magnitude(v)` | «Размер» клиента — суммарный показатель активности |
| `cosine_similarity(A, B)` | Похожи ли паттерны поведения, **без учёта** размера |
| Min-Max нормализация перед cosine | Чтобы признаки на разных шкалах учитывались на равных |
| Feature weighting перед cosine | Подкрутить метрику под конкретную маркетинговую цель |

### «Похожи по типу» vs «похожи по размеру» — разные вопросы

Когда маркетолог говорит «найди похожих на VIP-клиента», обычно имеется в виду
**один из трёх вопросов**:

| Вопрос | Метод | Что получаешь |
|---|---|---|
| Кто похож по **типу** поведения | нормализация + cosine | сегментация по паттернам |
| Кто **самый ценный** | магнитуда или сумма по чеку | топ-N по выручке |
| Кто похож по **объёму** | евклидово расстояние без нормализации | соседи по абсолютным числам |

Это разные ML-задачи, разные метрики, разный код. Грамотный аналитик
**уточняет вопрос** до того, как выбрать инструмент.

### Аналогия из жюри для нормализации

Жюри из пяти экспертов, у каждого своя шкала (0–10, 0–100, 0–10 000).
Если просто сложить оценки — победит эксперт с самой большой шкалой,
остальные «не слышны». Это и есть проблема разных шкал признаков.
Решение — привести всех к одной шкале (Min-Max).

### Сегмент vs микросегмент

Иерархическая сегментация работает так:

1. Большие сегменты — по магнитуде или объёмному показателю (VIP, средние,
   спящие).
2. Внутри каждого — микросегменты по cosine с **взвешенными** признаками
   под конкретную кампанию.

Двухступенчатая кластеризация — стандартный приём в маркетинговой аналитике.

## Word arithmetic — операции над эмбеддингами слов

Знаменитый пример из NLP:

```
vec("король") - vec("мужчина") + vec("женщина") ≈ vec("королева")
```

Это **буквальное** сложение и вычитание векторов в 768-мерном пространстве
embedding'ов. Геометрически — «отними мужественность, прибавь
женственность». Возможность таких операций — побочный эффект того, как
обучают embedding-модели: они выкладывают слова в пространство так, что
семантические оси (пол, время, степень и т.п.) **направлены** в нём
последовательно.

## Градиентный спуск — вычитание + умножение на скаляр

Каждый шаг обучения нейросети:

```
weights = weights - learning_rate · gradient
```

Это **те же операции** над векторами: вычитание `sub(weights, ...)` плюс
умножение на скаляр `scale(gradient, learning_rate)`. Никакой особой
«магии» — стандартная линейная алгебра, просто над миллионами координат.
