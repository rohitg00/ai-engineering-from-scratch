# 生产扩展——队列、检查点、持久性

> 将多智能体系统扩展到数千个并发运行需要**持久执行**。LangGraph的运行时在每次超级步骤后写入一个以`thread_id`为键的检查点（默认Postgres）；工作者崩溃会释放租约，另一个工作者恢复。智能体可以无限期休眠等待人类输入。**MegaAgent**（arXiv:2408.09955）运行了一个具有三种状态（空闲/处理/响应）的每智能体生产者-消费者队列和两层协调（组内聊天 + 组间管理聊天）。**Fiber/async**在LLM流式传输上击败每作业线程：线程99%的时间空闲等待token，fiber在I/O上合作式让出。反方观点：Ashpreet Bedi的"扩展智能体软件"主张在负载证明需要之前使用**FastAPI + Postgres + 仅此而已**——简单架构比预期走得更远。本课程构建一个持久检查点日志、一个带状态转换的每智能体工作队列、一个异步与线程演示，并落地务实的"从简单开始"规则。

**类型：** 学习 + 构建
**语言：** Python（标准库，`asyncio`，`sqlite3`）
**前置知识：** 第16阶段 · 09（并行群体网络），第16阶段 · 13（共享内存）
**时间：** 约75分钟

## 问题

原型多智能体系统在一台笔记本电脑上用内存中的事件循环运行三个智能体。你移到生产环境：

- 智能体有时运行数小时（长程研究、人在回路等待）。
- 工作者进程崩溃。重启丢失状态。
- 峰值负载是平均值的10倍；你需要水平扩展。
- 用户按智能体运行付费；你需要收费的精确一次语义。

内存中的事件循环做不到这些。你需要一个底层的持久执行层。2026年的规范选项是：

1. 带检查点的工作流引擎（Temporal、LangGraph运行时）。
2. 带状态存储的消息队列（Postgres + SQS/RabbitMQ）。
3. 演员模型框架（MegaAgent的每智能体生产者-消费者）。
4. 手卷的FastAPI + Postgres（Bedi的论点）。

本课程构建每个的微缩版本。

## 核心概念

### 持久执行，模式

持久执行引擎在每次"步骤"后持久化完整程序状态（超级步骤，用LangGraph的语言）。崩溃时：

```
工作者在步骤中崩溃
  -> 租约超时
  -> 另一个工作者拾取thread_id
  -> 从最后一个检查点恢复
  -> 无重复副作用
```

这工作的要求：

- **可序列化状态。** 所有智能体状态必须可持久化。带有活动数据库连接的函数闭包无法存活。
- **确定性恢复。** 给定相同状态和相同输入，智能体产生相同动作（或将LLM调用推迟到外部确定性预言机）。
- **幂等副作用。** 外部调用（工具调用、支付）必须是幂等的或使用去重键。

LangGraph在每次超级步骤后写入检查点；Temporal在每次活动后写入；Restate使用事件溯源日志。三者实现相同的模式。

### LangGraph的运行时

每个智能体有一个`thread_id`；状态是类型化字典；每次超级步骤向检查点表写入一行。恢复时，运行时从最后一个检查点重放，不是从头开始。智能体可以`interrupt()`等待人类输入；运行时持久化并释放工作者。当输入到达时，任何工作者都可以恢复。

这是2026年4月的参考生产设计。

### MegaAgent的每智能体队列

arXiv:2408.09955描述了一个规模实验：一个集群中的数千个并发智能体。架构：

```
智能体 i：
  状态 ∈ {空闲, 处理, 响应}
  in_queue   <- 寻址到智能体 i 的消息
  out_queue  -> 回复 + 副作用

协调器：
  组内聊天  （同一组中的智能体）
  组间管理聊天  （高级路由）
```

两层协调让组内对话密集发生，而组间保持稀疏——用于在数千个智能体中保持成本线性的模式。

### 异步 vs 每作业线程

LLM调用是I/O绑定的。等待下一个token的线程99%的时间空闲。每个线程成本约1MB RAM；在10,000个并发调用时，仅栈就10GB。

Fiber（Python `asyncio`、Go goroutine、Rust `tokio`）在I/O上合作式让出。相同的10,000个调用舒适地放入进程中。在LLM智能体规模，异步不是优化——它是架构。

例外：CPU绑定的后处理（嵌入、tokenizer技巧）仍然需要线程或进程。将你的I/O层与CPU层分开。

### Bedi的反方观点

"扩展智能体软件"（Ashpreet Bedi，2026）认为大多数团队在测量负载之前就过度工程化了。务实的默认：

- FastAPI + Postgres。
- 每次智能体运行是一行；状态通过乐观并发原地更新。
- 通过`pg_notify`或简单Celery工作者的后台作业。
- 应用代码中的重试策略。

对于在可管理任务上低于约100个并发智能体运行的负载，这通常是你需要的全部。在测量到失败时升级。

规则：当你遇到简单架构无法解决的具体问题时，采用持久执行框架。过早采用会浪费时间在不会回报的仪式上。

### 精确一次语义

对于付费智能体运行，你需要"精确一次有效"（至少一次交付 + 幂等消费者）。工程动作：

- **每次运行的去重键。** 将其包含在每个副作用调用中。
- **发件箱模式。** 副作用先写入表，然后单独进程执行它们。两个步骤都是幂等的。
- **补偿事务。** 当副作用成功但其跟踪写入失败时，安排补偿。

这些是数据库工程模式，不是LLM特定的。LLM税只是LLM调用慢；其他一切都是标准分布式系统。

### 彩虹部署

Anthropic的多智能体研究系统使用"彩虹部署"：多个版本的智能体运行时并发运行，因此长程智能体不必在每次代码部署时被杀死。在流量切片上金丝雀新版本；当它们的智能体完成时淘汰旧版本。

这对长程有状态系统是标准的；2026年的适应是智能体可以存活数小时，因此部署周期必须适应。

### 规范生产检查清单

- 持久状态（检查点、快照或发件箱 + 可重放日志）。
- 幂等副作用。
- LLM调用的异步I/O层。
- 带去重的至少一次交付。
- 有状态工作负载的彩虹/金丝雀部署。
- 可观察性：每智能体追踪、超级步骤审计、重试计数器。

## 构建它

`code/main.py`实现：

- `CheckpointStore` —— 以SQLite为后盾的检查点日志，以线程id为键。每次超级步骤追加一行。
- `run_with_checkpoint(agent, thread_id)` —— 模拟运行中崩溃；第二个工作者从最后一个检查点恢复。
- `AgentQueue` —— 带小型工作队列的每智能体空闲/处理/响应状态机。
- `demo_async_vs_threads()` —— 通过asyncio和线程运行500个并发模拟"LLM调用"；报告挂钟和峰值内存（近似）。

运行：

```
python3 code/main.py
```

预期输出：模拟崩溃后检查点恢复成功；异步版本在<1秒内处理500个并发调用；线程版本需要数秒，每个并发单元使用数量级更多的内存。

## 交付它

规范生产加固：

- **从简单开始（Bedi规则）。** FastAPI + Postgres直到你测量到失败。
- **优化前对所有内容进行检测。** 每次运行延迟直方图、每步时间、重试计数、失败分类。
- **副作用的发件箱模式。** 尤其是支付和外部API调用。
- **彩虹部署。** 部署期间绝不杀死进行中的智能体运行。
- **当你遇到具体问题时采用持久执行引擎（Temporal / LangGraph / Restate）：** 数小时的人回路等待、跨区域协调、复杂的重试/补偿策略。
- **I/O层用异步。** 线程仅用于CPU绑定的后处理。

## 练习

1. 运行`code/main.py`。确认检查点恢复有效；测量异步与线程并发差异。
2. 实现**发件箱**表：每个工具调用先写入发件箱，然后单独的goroutine/任务执行。通过运行工具调用两次验证幂等性。
3. 模拟**彩虹部署**：两个并发运行时版本；将一半新thread_id路由到每个；确认旧版本上的进行中的线程不被中断。
4. 阅读LangGraph运行时文档（见下文链接）。识别运行时中哪些功能在手卷的FastAPI + Postgres版本中复制最耗时。这是采用的理由，还是可以推迟？
5. 阅读MegaAgent（arXiv:2408.09955）第3节。两层协调（组内 + 组间管理聊天）是显式的。草图你将如何将其映射到具有两个队列族的消息队列。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 持久执行 | "持久化程序状态" | 引擎在每次超级步骤后写入状态；崩溃恢复是确定性的。 |
| 超级步骤 | "事务边界" | 检查点之间的工作单元。LangGraph术语。 |
| thread_id | "智能体运行标识符" | 绑定检查点和恢复逻辑的键。 |
| 幂等性 | "安全重试" | 重复副作用产生与一次尝试相同的结果。 |
| 发件箱模式 | "解耦副作用" | 将意图写入表；单独执行器执行并标记完成。 |
| 至少一次交付 | "可能重复" | 消息队列语义；去重键使消费者有效一次。 |
| 彩虹部署 | "重叠版本" | 长程工作负载期间多个运行时版本并发。 |
| 异步fiber | "合作式让出" | 用户模式并发；与线程相比对I/O绑定负载便宜。 |
| 检查点 | "状态快照" | 超级步骤边界处的序列化状态；恢复的关键。 |
