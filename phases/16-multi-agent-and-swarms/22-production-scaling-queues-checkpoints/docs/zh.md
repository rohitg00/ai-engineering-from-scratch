# 生产扩展 — 队列、检查点、持久性

> 将多智能体系统扩展到数千个并发运行需要**持久执行**。LangGraph 的运行时在每个由 `thread_id` 键控的超级步之后写入一个检查点（默认 Postgres）；工作进程崩溃会释放租约，另一个工作进程恢复。智能体可以无限期睡眠等待人类输入。**MegaAgent**（arXiv:2408.09955）运行一个每个智能体的生产者-消费者队列，具有三种状态（空闲 / 处理中 / 响应）和两层协调（组内聊天 + 组间管理聊天）。**Fiber/async** 在 LLM 流传输方面胜过每个作业一个线程：线程 99% 的时间空闲等待令牌，fibers 在 I/O 上协作让出。相反的观点：Ashpreet Bedi 的"扩展智能体软件"主张**FastAPI + Postgres + 没有其他东西**，直到负载证明 otherwise——简单的架构比预期的要走得更远。本课构建了一个持久检查点日志、一个具有状态转换的每个智能体工作队列、一个异步 vs 线程演示，并落地了务实的"从简单开始"规则。

**类型：** 学习 + 实践
**语言：** Python（标准库、`asyncio`、`sqlite3`）
**前置要求：** 第 16 阶段 · 09（并行群体网络）、第 16 阶段 · 13（共享内存）
**时间：** 约 75 分钟

## 问题

一个原型多智能体系统在一台笔记本电脑上通过内存中事件循环与三个智能体一起工作。你转移到生产环境：

- 智能体有时会运行数小时（长时间研究、人在回路等待）。
- 工作进程崩溃。重新启动会丢失状态。
- 峰值负载是平均值的 10 倍；你需要水平扩展。
- 用户为每个智能体运行付费；你需要精确一次语义来进行计费。

内存中事件循环做不到这些。你需要在下面有一个持久执行层。2026 年的规范选项是：

1. 带有检查点的工作流引擎（Temporal、LangGraph 运行时）。
2. 带有状态存储的消息队列（Postgres + SQS/RabbitMQ）。
3. 参与者模型框架（MegaAgent 的每个智能体生产者-消费者）。
4. 手写的 FastAPI + Postgres（Bedi 的论点）。

本课构建每一个的微型版本。

## 概念

### 持久执行，模式

持久执行引擎在每次"步骤"（超级步，用 LangGraph 的语言）之后持久化完整程序状态。崩溃时：

```
工作进程在步骤中途崩溃
  -> 租约超时
  -> 另一个工作进程接管 thread_id
  -> 从最后一个检查点恢复
  -> 没有重复的副作用
```

要使其工作，需要满足以下要求：

- **可序列化状态。** 所有智能体状态必须可持久化。带有实时数据库连接的函数闭包无法存活。
- **确定性恢复。** 给定相同的状态和相同的输入，智能体产生相同的动作（或针对 LLM 调用推迟到外部确定性预言机）。
- **幂等副作用。** 外部调用（工具调用、支付）必须是幂等的或使用重复数据删除密钥。

LangGraph 在每个超级步之后写入一个检查点；Temporal 在每个活动之后写入；Restate 使用事件溯源日志。所有三个都实现了相同的模式。

### LangGraph 的运行时

每个智能体有一个 `thread_id`；状态是一个类型化的字典；每个超级步向检查点表写入一行。恢复时，运行时从上一个检查点重放，而不是从头开始。智能体可以 `interrupt()` 等待人类输入；运行时持久化并释放工作进程。当输入到达时，任何工作进程都可以恢复。

这是 2026 年 4 月的参考生产设计。

### MegaAgent 的每个智能体队列

arXiv:2408.09955 描述了一个规模实验：一个集群中的数千个并发智能体。架构：

```
智能体 i：
  状态 ∈ {空闲, 处理中, 响应}
  in_queue   <- 寻址到智能体 i 的消息
  out_queue  -> 回复 + 副作用

协调者：
  组内聊天（同一组中的智能体）
  组间管理聊天（高级路由）
```

两层协调允许组内对话密集发生，而组间保持稀疏——用于将成本在数千个智能体中保持线性的模式。

### 异步 vs 每个作业一个线程

LLM 调用是 I/O 绑定的。等待下一个令牌的线程 99% 的时间都是空闲的。线程每个花费约 1MB RAM；在 10,000 个并发调用时，仅堆栈就达到 10GB。

Fibers（Python `asyncio`、Go goroutines、Rust `tokio`）在 I/O 上协作让出。相同的 10,000 个调用舒适地适合进程。在 LLM 智能体规模上，异步不是优化——它是架构。

例外：CPU 绑定的后处理（嵌入、分词器技巧）仍然需要线程或进程。将你的 I/O 层与 CPU 层分开。

### Bedi 的相反观点

"扩展智能体软件"（Ashpreet Bedi，2026）认为，大多数团队在测量负载之前就过度设计了。务实的默认设置：

- FastAPI + Postgres。
- 每个智能体运行是一行；状态通过乐观并发就地更新。
- 通过 `pg_notify` 或简单的 Celery 工作进程的后台作业。
- 应用程序代码中的重试策略。

对于在可管理任务上约 100 个并发智能体运行以下的负载，这通常就是你所需要的。当你测量到它失败时再进行升级。

规则：当你遇到简单架构无法解决的具体问题时，采用持久执行框架。过早采用会在不会得到回报的仪式上浪费时间。

### 精确一次语义

对于付费智能体运行，你需要"有效精确一次"（至少一次交付 + 幂等消费者）。工程举措：

- **每次运行都有重复数据删除密钥。** 将其包含在每个副作用调用中。
- **发件箱模式。** 副作用首先写入表，然后一个单独的过程执行它们。两个步骤都是幂等的。
- **补偿事务。** 当副作用成功但其跟踪写入失败时，安排一个补偿。

这些是数据库工程模式，不是 LLM 特定的。LLM 税只是 LLM 调用很慢；其他一切都是标准的分布式系统。

### 彩虹部署

Anthropic 的多智能体研究系统使用"彩虹部署"：只要长时间运行的智能体不必在每次代码部署时都被杀死，智能体运行时的多个版本就可以并发运行。在一部分流量上进行新版本的金丝雀发布；当它们的智能体完成时，淘汰旧版本。

这对于长时间运行的有状态系统是标准的；2026 年的适应是智能体可以存活数小时，因此部署周期必须适应。

### 规范生产检查清单

- 持久状态（检查点、快照或发件箱 + 可重放日志）。
- 幂等副作用。
- 用于 LLM 调用的异步 I/O 层。
- 带有重复数据删除的至少一次交付。
- 用于有状态工作负载的彩虹/金丝雀部署。
- 可观察性：每个智能体的追踪、超级步审计、重试计数器。

## 动手实践

`code/main.py` 实现：

- `CheckpointStore` — 带有线程 id 密钥的 SQLite 支持的检查点日志。每个超级步附加一行。
- `run_with_checkpoint(agent, thread_id)` — 模拟运行中崩溃；第二个工作进程从上一个检查点恢复。
- `AgentQueue` — 具有小型工作队列的每个智能体空闲 / 处理中 / 响应状态机。
- `demo_async_vs_threads()` — 通过 asyncio 和线程运行 500 个并发模拟的"LLM 调用"；报告挂钟和峰值内存（近似值）。

运行：

```
python3 code/main.py
```

预期输出：模拟崩溃后检查点恢复成功；异步版本在 < 1 秒内处理 500 个并发调用；线程版本需要几秒钟，并且每个并发单元使用数量级更多的内存。

## 部署

规范生产加固：

- **从简单开始（Bedi 的规则）。** FastAPI + Postgres，直到你测量到它失败。
- **在优化之前检测所有内容。** 每次运行的延迟直方图、每步时间、重试计数、失败分类。
- **副作用的发件箱模式。** 特别是支付和外部 API 调用。
- **彩虹部署。** 永远不要在处理中的智能体运行期间将其杀死。
- **当你遇到具体问题时采用持久执行引擎（Temporal / LangGraph / Restate）：** 长达一小时的人在回路等待、跨区域协调、复杂的重试/补偿策略。
- **I/O 层的异步。** 线程仅用于 CPU 绑定的后处理。

## 练习

1. 运行 `code/main.py`。确认检查点恢复有效；测量异步 vs 线程并发差异。
2. 实现一个**发件箱**表：每个工具调用首先写入发件箱，然后一个单独的 goroutine/task 执行。通过运行工具调用两次来验证幂等性。
3. 模拟一个**彩虹部署**：两个并发运行时版本；将一半的新 thread_id 路由到每个版本；确认旧版本上的在途线程不会被中断。
4. 阅读 LangGraph 的运行时文档（下面链接）。确定在手写 FastAPI + Postgres 版本中复制运行时哪些功能需要最长的时间。这是采用的原因，还是你可以推迟？
5. 阅读 MegaAgent（arXiv:2408.09955）第 3 节。两层协调（组内 + 组间管理聊天）是显式的。勾勒你将如何将其映射到具有两个队列族的消息队列。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|----------------|------------------------|
| 持久执行（Durable execution） | "持久化程序状态" | 引擎在每个超级步之后写入状态；崩溃恢复是确定性的。 |
| 超级步（Super-step） | "事务边界" | 检查点之间的工作单元。LangGraph 术语。 |
| thread_id | "智能体运行标识符" | 绑定检查点和恢复逻辑的密钥。 |
| 幂等性（Idempotency） | "安全重试" | 重复副作用产生与一次尝试相同的结果。 |
| 发件箱模式（Outbox pattern） | "解耦副作用" | 将意图写入表；单独的执行器执行并标记为完成。 |
| 至少一次交付（At-least-once delivery） | "可能重复" | 消息队列语义；重复数据删除密钥使消费者有效一次。 |
| 彩虹部署（Rainbow deploy） | "重叠版本" | 长时间运行工作负载期间的多个运行时版本。 |
| 异步纤维（Async fiber） | "协作让出" | 用户态并发；对于 I/O 绑定负载而言，与线程相比便宜。 |
| 检查点（Checkpoint） | "状态快照" | 超级步边界的序列化状态；恢复的关键。 |
