给定受益于并行子代理研究的用户查询，生成可连接到任何框架（LangGraph、OpenAI Agents SDK、CrewAI Hierarchical）的监督者模式设计。

生成：

1. **复杂度估计。** 此查询是简单（1 个代理，3-10 个工具调用）、中等（2-4 个工作者）还是复杂（5+ 个工作者）？使用 Anthropic 的规模-努力启发式用一句话证明。
2. **引导系统提示词。** 必须包括：(a) 分解指令，(b) 合成指令，(c) 引导从不读取原始源内容、只读取工作者摘要的显式规则。
3. **工作者系统提示词。** 每个角色一个，每个命名其狭窄范围和引导期望的输出格式。
4. **子问题分解规则。** 引导如何拆分查询？先广泛后狭窄，还是直接分解？什么使子问题不合格（与另一个重叠、太广泛）？
5. **合成模板。** 显式冲突处理规则：如果两个工作者返回矛盾事实，合成必须提出分歧而不是静默选择一个。
6. **模型配对。** 引导用哪个模型（推理层），工作者用哪个（更快/更便宜层）。解释权衡。
7. **可观测性要求。** 最小跟踪点：计划、每个工作者开始/结束、合成输入、合成输出。

硬性拒绝：

- 引导自己做工具使用的任何设计。引导只计划和合成。
- 允许范围漂移的工作者提示词（例如，没有界限的"研究与 X 相关的任何内容"）。
- 隐藏冲突的合成模板。

拒绝规则：

- 如果查询简单（估计总共少于 10 个工具调用），拒绝设计并推荐单代理。引用 Anthropic 15× token 成本发现。
- 如果查询是顺序的（步骤 2 需要步骤 1 的输出），拒绝并推荐管道/链模式。
- 如果用户优化确定性和审计，拒绝监督者并推荐 LangGraph 静态图。

输出：一页设计简报。以复杂度估计和模式匹配裁决（"supervisor fits"）开头。如果系统将持续运行，以 rainbow-deployment 提醒结束。
