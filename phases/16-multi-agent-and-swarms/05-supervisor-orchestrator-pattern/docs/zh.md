# 监督者/编排器-工作者模式

> 一个主导智能体计划和委派；专业工作者在并行上下文中执行并报告回来。这是Anthropic研究系统背后的模式（Claude Opus 4为主导，Sonnet 4为子智能体），在内部研究评估中比单智能体Opus 4高出+90.2%。Anthropic的工程文章报告称，BrowseComp上80%的方差仅由token使用量解释——多智能体获胜主要是因为每个子智能体获得一个全新的上下文窗口。本课程从原语构建监督者模式，并涵盖2026年生产部署的工程经验。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**前置知识：** 第16阶段 · 04（原语模型）
**时间：** 约75分钟

## 问题

研究是单智能体系统失败的典型任务。你问"2023年到2026年多智能体系统发生了什么变化？"一个单智能体顺序读取五篇论文，用它们的文本填满一半的上下文，然后必须一起推理它们。到第五篇时它已经忘记了第一篇。它无法并行化。

监督者模式解决了这个问题：一个主导智能体计划搜索，将每个子问题委派给工作者，并综合结果。每个工作者获得自己的20万token窗口用于一个狭窄的问题。主导者从不看原始论文——只看工作者的摘要。

Anthropic的生产研究系统报告内部研究评估比单Opus 4高出+90.2%。同一篇文章指出，80%的BrowseComp方差仅由*token使用量*解释。每个子智能体的全新上下文是主要机制。

## 核心概念

### 模式

```
                 ┌──────────────┐
                 │   主导者     │  计划、分解、
                 │  (Opus 4)    │  综合
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ 工作者1 │  │ 工作者2 │  │ 工作者3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         全新        全新        全新
         上下文      上下文      上下文
```

主导者从不读取原始材料。工作者在主导者综合之前从不看到彼此的工作。每个箭头都是带有狭窄工件的交接。

### 为什么它获胜

三种机制：

1. **每个子智能体的全新上下文。** 探索"FIPA-ACL传承"的工作者不携带主导者花费计划所用的4万token。它获得一个20万窗口用于一个问题。
2. **通过提示专业化。** 主导者的提示是"分解和综合"，不是"研究"。每个工作者的提示是狭窄的："找出X中发生了什么变化。"专注的提示产生专注的输出。
3. **并行性。** 工作者并发运行。挂钟时间大致是`max(工作者时间) + 计划 + 综合`，不是`sum(工作者时间)`。

### 工程经验（Anthropic 2025）

Anthropic的文章列出了几个生产经验，在2026年仍然相关：

- **根据查询复杂度扩展工作量。** 简单查询：一个智能体，3-10次工具调用。复杂查询：10+智能体。主导者必须估计这个，不是调用者。
- **先广后窄。** 先分解为广泛的子问题，如果答案需要深度，则每个子问题生成更多工作者。
- **彩虹部署。** 智能体是长期运行和有状态的。传统的蓝绿部署不起作用。Anthropic使用彩虹：逐步推出新版本，同时旧版本排空。
- **Token使用占主导。** 多智能体是单智能体的约15倍token。仅在任务价值证明成本合理时才运行它。

### LangGraph的转变

LangGraph最初发布了一个带有高级`create_supervisor`助手的`langgraph-supervisor`库。2025年LangChain将建议转移到直接通过工具调用实现监督者模式，因为工具调用给予更多控制*监督者看到什么*（上下文工程）。该库仍然有效；文档现在推荐工具调用形式。

### 失败模式

- **主导者幻觉计划。** 如果主导者生成不分解真实问题的子问题，工作者在错误目标上进行精确研究。
- **工作者过度探索。** 没有显式范围边界，工作者漂移到分配的子问题之外，污染综合步骤。
- **综合冲突。** 两个工作者返回矛盾的事实。主导者必须要么重新询问（增加一轮），要么明确记录分歧。静默选择一方是最糟糕的失败：用户永远不知道分歧发生了。

### 何时监督者是错误的

- **串行任务。** 如果步骤2字面上需要步骤1的输出，并行性没有收益。使用流水线（CrewAI顺序、LangGraph线性图）。
- **简单查询。** 单智能体处理它们更快更便宜。在生成工作者之前使用主导者的"扩展工作量"检查。
- **严格确定性。** 监督者使用LLM选择的委派。当审计/重放比适应性更重要时，静态图更好。

## 构建它

`code/main.py`使用`threading`实现了三个并行工作者的监督者。主导者将查询分解为子问题，工作者在每个子问题上并发运行，主导者综合结果。没有真正的LLM——工作者是脚本化的，模拟获取和摘要。

关键结构：

- `Lead.plan(query)` 将查询拆分为3个子问题。
- `Worker.run(sub_q)` 返回一个假摘要（在生产中可以是任何使用工具的智能体）。
- `Lead.run(query)` 在线程中启动工作者，加入，并综合结果。

运行：

```
python3 code/main.py
```

输出显示计划、带有开始/结束时间戳的并行工作者跟踪，以及最终综合。你可以看到挂钟时间上的胜利：三个0.3秒的工作者在约0.35秒内运行，不是0.9秒。

## 使用它

`outputs/skill-supervisor-designer.md`接受用户查询并产生监督者模式设计：主导系统提示、工作者角色、子问题分解规则和综合模板。在构建新的研究风格智能体系统之前使用此功能。

## 交付它

部署监督者模式前的检查清单：

- **模型配对。** 主导者使用推理级模型（Opus级、`o3`级）。工作者使用更快、更便宜的模型（Sonnet、`o4-mini`）。
- **工作者超时。** 任何超过2倍中位运行时间的工作者被终止；主导者要么用更窄的范围重新生成，要么在没有它的情况下继续。
- **每个工作者的token上限。** 硬限制（比如预期综合输入的10倍）防止失控工作者吹掉预算。
- **可观察性。** 追踪主导者的计划、每个工作者的工具调用和综合。这是任何事后调试的基础。
- **彩虹推出。** 有状态的长期运行智能体需要渐进版本过渡，不是热交换。

## 练习

1. 运行`code/main.py`，然后修改主导者生成5个工作者而不是3个。观察挂钟效果。在这个演示中，工作者数量达到多少时生成开销超过并行节省？
2. 实现工作者超时：终止任何运行超过0.5秒的工作者，让主导者综合剩余结果。你需要什么可观察性来知道工作者被切断了？
3. 在主导者的综合中添加冲突检测步骤：如果两个工作者返回矛盾的答案，主导者记录分歧而不是选择一方。如何在不调用LLM的情况下检测矛盾？
4. 阅读Anthropic的研究系统工程文章。列出这个玩具演示需要采用的三个实践才能在生产中运行。
5. 比较LangGraph的`create_supervisor`（遗留）与新的工具调用建议。哪个给予你更好的控制主导者看到什么？为什么Anthropic明确只传递子答案而不是原始工作者上下文到综合中？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 监督者 | "主导智能体" | 一个计划、委派和综合的编排器智能体。不自己做工作。 |
| 工作者 | "子智能体" | 由监督者调用的专注智能体，具有狭窄范围和自己的上下文窗口。 |
| 编排器-工作者 | "监督者模式" | 相同的东西，不同的名称。2026年的文献两者都用。 |
| 全新上下文 | "干净窗口" | 工作者的上下文从它的系统提示和分配的问题开始，不是主导者的历史。 |
| 彩虹部署 | "渐进推出" | 长期运行有状态智能体需要版本化的排空和替换，不是蓝绿。 |
| Token主导 | "上下文是变量" | 根据Anthropic，80%的研究评估方差来自使用的总token，不是模型选择。 |
| 扩展工作量 | "将智能体数量与复杂度匹配" | 主导者估计查询难度，相应地生成1个 vs 10+工作者。 |
| 综合冲突 | "工作者不同意" | 两个工作者返回矛盾事实；主导者必须 surfaced 分歧，不是静默选择一方。 |
