# 监督者/编排器-工作者模式

> 一个主导智能体进行规划和委派；专门的工作者们在并行上下文中执行并向其报告。这是 Anthropic 研究系统背后的模式（Claude Opus 4 作为主导，Sonnet 4 作为子智能体），在内部研究评估中比单智能体 Opus 4 高出 +90.2%。Anthropic 的工程帖子报告说，BrowseComp 中 80% 的方差仅由令牌使用量解释——多智能体的胜利很大程度上是因为每个子智能体都获得了一个全新的上下文窗口。本课从原语构建监督者模式，并涵盖来自生产部署的 2026 年工程经验。

**类型：** 学习 + 构建
**语言：** Python（标准库、`threading`）
**前置条件：** 阶段 16.04（原语模型）
**时长：** 约 75 分钟

## 问题背景

研究是单智能体系统失败的典型任务。你问"2023 年和 2026 年之间多智能体系统有什么变化？" 单个智能体顺序读取五篇论文，用它们的文本填满一半上下文，然后必须一起推理所有论文。当它到达第五篇论文时，它已经忘记了第一篇。它无法并行化。

监督者模式解决了这个问题：一个主导智能体规划搜索，将每个子问题委派给一个工作者，并进行综合。每个工作者获得自己 20 万令牌窗口来处理一个狭窄的问题。主导者从不看原始论文 —— 只看工作者的摘要。

Anthropic 的生产研究系统在内部研究评估中比单个 Opus 4 高出 +90.2%。同一篇帖子指出，BrowseComp 中 80% 的方差仅由*令牌使用量*解释。每个子智能体的全新上下文是主要机制。

## 概念讲解

### 模式

```
                 ┌──────────────┐
                 │   主导       │  规划、分解、
                 │  (Opus 4)  │  综合
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ 工作者1 │  │ 工作者2 │  │ 工作者3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         全新        全新       全新
         上下文      上下文     上下文
```

主导者从不读取原始材料。工作者们在主导者综合之前从不看到彼此的工作。每个箭头都是带有狭窄产物的交接。

### 为什么它胜出

三种机制：

1. **每个子智能体的全新上下文。** 探索"FIPA-ACL 遗产"的工作者不携带主导者用于规划的 4 万令牌。它为一个问题获得 20 万窗口。
2. **通过提示词专业化。** 主导者的提示词是"分解和综合"，而不是"研究"。每个工作者的提示词是狭窄的："找出 X 中的变化。" 聚焦的提示词产生聚焦的输出。
3. **并行性。** 工作者们并发运行。挂钟时间大致是 `max(工作者时间) + 规划 + 综合`，而不是 `sum(工作者时间)`。

### 工程经验（Anthropic 2025）

Anthropic 的帖子列出了几个仍然是 2026 年相关的生产经验：

- **将努力缩放到查询复杂度。** 简单查询：一个智能体，3-10 次工具调用。复杂查询：10+ 个智能体。主导者必须估计这一点，而不是调用者。
- **先广后窄。** 首先分解为广泛的子问题，然后如果答案需要深度，为每个子问题生成更多工作者。
- **彩虹部署（Rainbow deployments）。** 智能体是长时间运行且有状态的。传统的蓝绿部署不起作用。Anthropic 使用彩虹：在旧版本排空前逐步推出新版本。
- **令牌使用占主导。** 多智能体大约是单智能体的 15 倍令牌。只有当任务价值证明成本合理时才运行它。

### LangGraph 的转变

LangGraph 最初发布了一个带有高级 `create_supervisor` 辅助函数的 `langgraph-supervisor` 库。2025 年，LangChain 将建议改为通过工具调用直接实现监督者模式，因为工具调用可以更多地控制*监督者看到什么*（上下文工程）。该库仍然有效；文档现在推荐工具调用形式。

### 失败模式

- **主导者幻觉计划。** 如果主导者生成的子问题没有分解真正的问题，工作者就会在错误的目标上进行精确研究。
- **工作者过度探索。** 没有明确的范围边界，工作者会偏离其分配的子问题并污染综合步骤。
- **综合冲突。** 两个工作者返回矛盾的事实。主导者必须要么重新询问（增加一轮），要么明确注意分歧。 Silent 选择一方是最糟糕的失败：用户永远不会知道发生了分歧。

### 何时监督者模式是错误的

- **顺序任务。** 如果步骤 2 确实需要步骤 1 的输出，并行性买不到任何东西。使用流水线（CrewAI Sequential、LangGraph 线性图）。
- **简单查询。** 单智能体更快更便宜地处理它们。在生成工作者之前使用主导者的"缩放努力"检查。
- **严格确定性。** 监督者使用 LLM 选择的委派。当审计/重放比适应性更重要时，静态图更好。

## 构建实现

`code/main.py` 使用 `threading` 实现了一个具有三个并行工作者的监督者。主导者将查询分解为子问题，工作者在每个子问题上并发运行，然后主导者进行综合。没有真实的 LLM——工作者被编写脚本来模拟获取和总结。

关键结构：

- `Lead.plan(query)` 将查询拆分为 3 个子问题。
- `Worker.run(sub_q)` 返回虚假摘要（在生产中可以是任何使用工具的智能体）。
- `Lead.run(query)` 在线程中启动工作者、加入并综合。

运行：

```
python3 code/main.py
```

输出显示计划、带有开始/结束时间戳的并行工作者跟踪以及最终综合。你可以看到挂钟获胜：三个 0.3 秒的工作者在约 0.35 秒内运行，而不是 0.9 秒。

## 实际应用

`outputs/skill-supervisor-designer.md` 接受用户查询并产生监督者模式设计：主导系统提示词、工作者角色、子问题分解规则和综合模板。在构建新的研究风格智能体系统之前使用此功能。

## 部署实现

部署监督者模式前的检查清单：

- **模型配对。** 主导者使用推理层模型（Opus 级别、`o3` 级别）。工作者使用更快、更便宜的模型（Sonnet、`o4-mini`）。
- **工作者超时。** 任何超过 2 倍中位数运行时间的工作者都会被终止；主导者要么以更窄的范围重新生成，要么在没有它的情况下继续。
- **每个工作者令牌上限。** 硬性限制（比如预期综合输入的 10 倍）防止失控的工作者耗尽预算。
- **可观测性。** 追踪主导者的计划、每个工作者的工具调用和综合。这是任何事后调试的基础。
- **彩虹推出。** 有状态的长时间运行智能体需要逐步版本过渡，而不是热切换。

## 练习

1. 运行 `code/main.py`，然后修改主导者以生成 5 个工作者的 3 个。观察挂钟效应。在这个演示中，生成开销超过并行节省的工作者数量是多少？
2. 实现工作者超时：终止运行超过 0.5 秒的任何工作者，并让主导者综合剩余结果。你需要什么可观测性来知道工作者被切断了？
3. 在主导者的综合中添加一个冲突检测步骤：如果两个工作者返回矛盾的答案，主导者注意分歧而不是选择一个。如何在不调用 LLM 的情况下检测矛盾？
4. 阅读 Anthropic 的研究系统工程帖子。列出这个玩具演示需要采用的三项实践才能在生产中运行。
5. 比较 LangGraph 的 `create_supervisor`（遗留）与新的工具调用建议。哪个让你更好地控制监督者看到什么？为什么 Anthropic 明确只将子答案而不是原始工作者上下文传递到综合中？

## 关键术语

| 术语 | 人们说的 | 它实际意味着什么 |
|------|----------------|------------------------|
| Supervisor（监督者） | "主导智能体" | 规划、委派和综合的编排器智能体。自己不做工作。 |
| Worker（工作者） | "子智能体" | 由监督者调用的具有狭窄范围及其自己上下文窗口的聚焦智能体。 |
| Orchestrator-worker（编排器-工作者） | "监督者模式" | 同一件事，不同名称。2026 年文献两者都用。 |
| Fresh context（全新上下文） | "干净窗口" | 工作者的上下文从其系统提示词和分配的问题开始，而不是主导者的历史。 |
| Rainbow deployment（彩虹部署） | "逐步推出" | 长时间运行的有状态智能体需要版本化排出和替换，而不是蓝绿部署。 |
| Token dominance（令牌主导） | "上下文是变量" | 根据 Anthropic，研究评估方差的 80% 来自使用的总令牌，而不是模型选择。 |
| Scale effort（缩放努力） | "将智能体数量与复杂度匹配" | 主导者估计查询难度，相应地生成 1 对 10+ 工作者。 |
| Synthesis conflict（综合冲突） | "工作者不同意" | 两个工作者返回矛盾的事实；主导者必须表面分歧，而不是静默选择一个。 |
