# 05 · 监督者 / 编排者-工作者模式

> 一个主导智能体负责规划与委派；专职工作者在各自并行的上下文中执行任务并回报结果。这正是 Anthropic 研究系统（Research system）背后的模式（Claude Opus 4 作为主导者，Sonnet 4 作为子智能体），在内部研究类评测中相对单智能体 Opus 4 测得 +90.2% 的提升。Anthropic 的工程博客指出，在 BrowseComp 上 80% 的差异仅由 token 用量这一项就能解释——多智能体之所以胜出，很大程度上是因为每个子智能体都获得了一个全新的上下文窗口。本课将从基本原语出发构建监督者模式，并涵盖来自生产部署的 2026 年工程经验。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**前置：** 阶段 16 · 04（原语模型）
**时长：** 约 75 分钟

## 问题

研究（Research）是单智能体系统最典型会失败的任务。你问「多智能体系统在 2023 到 2026 之间发生了哪些变化？」单个智能体依次阅读五篇论文，用论文文本填满了一半上下文，然后还得对所有论文整体进行推理。等它读到第五篇时，已经忘了第一篇。它无法并行处理。

监督者模式解决了这个问题：一个主导智能体规划检索，把每个子问题委派给一个工作者，最后进行综合。每个工作者针对一个狭窄的问题，独享自己的 200k-token 窗口。主导者从不直接看原始论文——它只看工作者的摘要。

Anthropic 的生产级研究系统报告称，相比单个 Opus 4，在内部研究类评测上取得了 +90.2% 的提升。同一篇博客指出，BrowseComp 上 80% 的差异仅由 *token 用量* 这一项就能解释。为每个子智能体提供全新上下文是其中的主要机制。

## 概念

### 该模式

```
                 ┌──────────────┐
                 │   Lead       │  plans, decomposes,
                 │  (Opus 4)    │  synthesizes
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ Worker1 │  │ Worker2 │  │ Worker3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         fresh       fresh        fresh
         context     context      context
```

主导者从不阅读原始材料。在主导者进行综合之前，各工作者也看不到彼此的工作成果。每一条箭头都是一次只传递狭窄产物的交接（handoff）。

### 为什么它能胜出

三个机制：

1. **每个子智能体获得全新上下文。** 一个探索「FIPA-ACL 渊源」的工作者，不会背负主导者在规划阶段消耗的那 40k token。它为一个问题获得一个完整的 200k 窗口。
2. **通过提示词实现专精。** 主导者的提示词是「拆解并综合」，而不是「做研究」。每个工作者的提示词都很狭窄：「找出 X 发生了哪些变化」。聚焦的提示词产出聚焦的结果。
3. **并行性。** 工作者并发运行。墙钟时间（wall-clock time）大致为 `max(worker_times) + plan + synthesis`，而不是 `sum(worker_times)`。

### 工程经验（Anthropic 2025）

Anthropic 的博客列出了若干生产经验，在 2026 年仍然适用：

- **按查询复杂度调整投入。** 简单查询：一个智能体，3-10 次工具调用。复杂查询：10+ 个智能体。这必须由主导者来估算，而不是调用方。
- **先广后窄。** 先把问题拆解成宽泛的子问题，然后如果答案值得深入，再为每个子问题派生更多工作者。
- **彩虹部署（Rainbow deployments）。** 智能体是长时运行且有状态的。传统的蓝绿部署（blue-green）行不通。Anthropic 采用彩虹部署：在旧版本逐步排空（drain）的同时，逐步推出新版本。
- **token 用量占主导。** 多智能体的 token 用量约为单智能体的 15 倍。只有当任务价值能证明成本合理时才运行它。

### LangGraph 的转向

LangGraph 最初提供了一个 `langgraph-supervisor` 库，配有高层级的 `create_supervisor` 辅助函数。2025 年，LangChain 将推荐做法转向直接通过工具调用（tool-calling）来实现监督者模式，因为工具调用能对 *监督者看到什么* 提供更多控制（即上下文工程）。该库仍然可用；如今文档推荐的是工具调用形式。

### 失败模式

- **主导者臆造计划。** 如果主导者生成的子问题并未真正拆解出原始问题，工作者就会对错误的目标做精确研究。
- **工作者过度探索。** 没有明确的范围边界时，工作者会偏离其被分配的子问题，污染综合环节。
- **综合冲突。** 两个工作者返回相互矛盾的事实。主导者要么重新提问（增加一轮），要么明确指出这一分歧。最糟糕的失败是悄无声息地选取其中一方：用户永远不知道曾出现过分歧。

### 何时不该用监督者模式

- **顺序任务。** 如果步骤 2 确实需要步骤 1 的输出，并行无法带来任何收益。请用流水线（CrewAI 的 Sequential、LangGraph 的线性图）。
- **简单查询。** 单智能体处理得更快也更便宜。在派生工作者之前，先用主导者的「按投入扩展」检查来判断。
- **严格的确定性。** 监督者使用由 LLM 选择的委派方式。当审计/重放比适应性更重要时，静态图（static graph）更合适。

## 动手构建

`code/main.py` 使用 `threading` 实现了一个监督三个并行工作者的监督者。主导者把一个查询拆解为多个子问题，工作者针对各子问题并发运行，最后由主导者进行综合。没有用到真实的 LLM——工作者是脚本化的，用以模拟「抓取并摘要」的过程。

关键结构：

- `Lead.plan(query)` 把一个查询拆分为 3 个子问题。
- `Worker.run(sub_q)` 返回一段假的摘要（在生产中可以是任意使用工具的智能体）。
- `Lead.run(query)` 在线程中启动工作者、join 等待，并进行综合。

运行：

```
python3 code/main.py
```

输出会显示计划、带有起止时间戳的并行工作者轨迹，以及最终的综合结果。你可以看到墙钟时间上的收益：三个各耗时 0.3 秒的工作者在约 0.35 秒内完成，而不是 0.9 秒。

## 实际运用

`outputs/skill-supervisor-designer.md` 接收一个用户查询，并产出一份监督者模式的设计：主导者系统提示词、工作者角色、子问题拆解规则，以及综合模板。在构建一个新的研究式智能体系统之前，请先使用它。

## 上线交付

部署监督者模式之前的检查清单：

- **模型搭配。** 主导者使用推理级别的模型（Opus 级、`o3` 级）。工作者使用更快、更便宜的模型（Sonnet、`o4-mini`）。
- **工作者超时。** 任何运行时间超过中位数 2 倍的工作者都会被终止；主导者要么以更窄的范围重新派生它，要么在没有它的情况下继续。
- **每个工作者的 token 上限。** 设置硬上限（比如预期综合输入的 10 倍）可以防止失控的工作者撑爆预算。
- **可观测性。** 追踪主导者的计划、每个工作者的工具调用，以及综合环节。这是任何事后调试的基础。
- **彩虹式上线。** 有状态的长时运行智能体需要渐进的版本切换，而不是热替换。

## 练习

1. 运行 `code/main.py`，然后修改主导者，让它派生 5 个工作者而不是 3 个。观察墙钟时间的效果。在这个 demo 中，工作者数量达到多少时，派生开销会超过并行节省？
2. 实现一个工作者超时机制：终止任何运行超过 0.5 秒的工作者，并让主导者综合剩余的结果。你需要什么样的可观测性才能知道某个工作者被切断了？
3. 给主导者的综合环节增加一个冲突检测步骤：如果两个工作者返回相互矛盾的答案，主导者要指出这一分歧，而不是选取其一。在不调用 LLM 的情况下，你如何检测矛盾？
4. 阅读 Anthropic 关于研究系统的工程博客。列出三项这个玩具 demo 若要在生产中运行需要采纳的实践。
5. 对比 LangGraph 的 `create_supervisor`（遗留方案）与新的工具调用推荐方案。哪一个能让你更好地控制监督者看到什么？为什么 Anthropic 明确地只把子答案、而非工作者的原始上下文传入综合环节？

## 关键术语

| 术语 | 人们怎么说 | 它实际指什么 |
|------|----------------|------------------------|
| 监督者（Supervisor） | 「主导智能体」 | 一个负责规划、委派和综合的编排者智能体。它本身不做具体工作。 |
| 工作者（Worker） | 「子智能体」 | 由监督者调用的、范围狭窄并拥有自己上下文窗口的专注型智能体。 |
| 编排者-工作者（Orchestrator-worker） | 「监督者模式」 | 同一回事，不同叫法。2026 年的文献两种说法都在用。 |
| 全新上下文（Fresh context） | 「干净的窗口」 | 工作者的上下文从它的系统提示词和被分配的问题开始，而不是从主导者的历史开始。 |
| 彩虹部署（Rainbow deployment） | 「渐进式上线」 | 长时运行的有状态智能体需要带版本的「排空并替换」，而非蓝绿部署。 |
| token 主导性（Token dominance） | 「上下文才是变量」 | 据 Anthropic，研究评测 80% 的差异来自所用的总 token 量，而非模型选择。 |
| 按投入扩展（Scale effort） | 「让智能体数量匹配复杂度」 | 主导者估算查询难度，据此派生 1 个还是 10+ 个工作者。 |
| 综合冲突（Synthesis conflict） | 「工作者意见不一」 | 两个工作者返回相互矛盾的事实；主导者必须暴露这一分歧，而不是悄悄选一个。 |
