# 09 · 并行 / 蜂群 / 网络化架构

> 与监督者（supervisor）模式形成对比：没有中央决策者。各智能体读取共享的「事件总线（event bus）」，异步领取工作，再把结果写回。LangGraph 明确支持面向去中心化、动态环境的「蜂群架构（Swarm Architecture）」。Matrix（arXiv:2511.21686）把控制流与数据流都表示为经由分布式队列传递的序列化消息，从而消除编排器（orchestrator）这一瓶颈。权衡很明确：用确定性与可追溯性换取可扩展性。蜂群适合具有大量相互独立子问题的任务；不适合需要单一连贯计划的任务。

**类型：** 学习 + 构建
**语言：** Python（标准库、`threading`、`queue`）
**前置：** 第 16 阶段 · 05（监督者模式），第 16 阶段 · 04（原语模型）
**时长：** 约 75 分钟

## 问题

监督者模式能扩展到几个工作者（worker）。那几百个呢？此时监督者自身成为瓶颈：关于谁做什么的每一个决策都要汇集到这一个智能体。一个缓慢的计划步骤就会拖住整个系统。

蜂群架构反转了这一设计。不再由中央规划者分派工作，而是由工作者从共享队列上领取工作。「协调」被内建进事件总线的语义中。没有编排器；系统的可扩展性取决于队列能扩展到多大。

## 概念

### 形态

```
                ┌──── shared queue ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            results pool
```

没有编排器。每个工作者重复以下动作：拉取一个任务、处理、写回结果（并可选择把后续任务入队）。

### 何时适合用蜂群

- **大量相互独立的任务。** 抓取、转换、分类。任务之间互不依赖。
- **耗时不一的工作。** 如果有些任务耗时 100ms、有些耗时 10s，蜂群会自动均衡负载——空闲快的工作者会去拉取下一个作业。监督者则必须预先估计耗时。
- **吞吐优先于确定性。** 你关心的是总完成时间，而非严格的顺序。

### 何时蜂群会失效

- **有序工作流。** 如果第 3 步需要第 2 步的输出，蜂群就有可能在第 2 步完成前就触发第 3 步。
- **需要全局计划的任务。** 复杂的研究类问题受益于规划者。一群研究员式工作者只会产出彼此独立的事实，而不是一份连贯的报告。
- **调试。** 在没有中央日志且工作异步进行的情况下，复现一个 bug 的代价很高。

### Matrix（arXiv:2511.21686）

Matrix 是 2025 年的一篇论文，它把蜂群推向其自然的极致：控制流与数据流都是分布式队列上的序列化消息。没有中央协调者。容错来自消息的持久性（durability）。可扩展性是消息代理（message broker）需要解决的问题，而不是系统本身的问题。

贡献：一种编程模型，其中多智能体协调被表述为「这个智能体订阅哪个消息主题（topic）？」，而非「监督者接下来挑选哪个智能体？」。这让系统看起来像一张「发布/订阅（pub/sub）」事件网格。

### LangGraph 的蜂群架构

LangGraph 2025 年的文档明确把「蜂群架构」描述为多智能体模式之一：智能体是节点，但边构成带环的有向图，任何节点都可以从池中被激活。工作者按条件从可用工作中挑选，而不是由监督者分配。

### 失效模式：饥饿与热点

如果所有工作者都拉取「当下最快可完成」的任务，那些长耗时任务就永远轮不到，直到它们成为仅剩的任务为止。这是典型的队列「饥饿（starvation）」。

缓解措施：

- 带显式「老化（aging）」的优先级队列（priority queue）（随等待时间提升优先级）。
- 工作者专门化：某些工作者只接「长」任务。
- 「反压（back-pressure）」：限制有多少快任务进入队列。

### 与基于内容路由的关联

蜂群天然地与「基于内容的路由（content-based routing）」（第 22 课）配对。不用一条通用队列，而是为每种消息类型设一条队列。专职工作者只订阅自己负责的类型。这正是可扩展到数千个智能体的消息总线（message-bus）架构的基础。

## 动手构建

`code/main.py` 实现了一个由 4 个工作者线程组成的蜂群，它们从一个共享的 `queue.Queue` 中拉取任务。任务的耗时各不相同（有快有慢）。该演示对比了：

- **顺序基线（Sequential baseline）：** 一个工作者串行处理所有任务。
- **固定分配（Fixed assignment）：** 每个任务预先分配给特定工作者（监督者风格）。
- **蜂群（Swarm）：** 工作者从共享队列中拉取。

蜂群会自动均衡负载；而固定分配会在某个工作者被分到的任务很慢时，让其它快工作者空闲。

运行：

```
python3 code/main.py
```

输出展示了每个工作者的任务数（蜂群的分配不均匀但最优）以及挂钟时间（wall-clock time）。

## 用起来

`outputs/skill-swarm-fit.md` 评估某个任务应当采用蜂群还是监督者。输入项包括：任务独立性、耗时方差、顺序要求、可调试性需求。

## 上线交付

清单：

- **带老化的优先级队列。** 防止长任务饥饿。
- **工作者幂等性（idempotency）。** 如果某个工作者在运行中途崩溃，一个任务可能被拉取多次。工作者必须是幂等的。
- **持久化队列。** 生产环境请使用 Kafka、Redis Streams 或以数据库为后端的队列。`queue.Queue` 仅存于内存中。
- **逐任务可观测性。** 每个任务都有一个追踪 ID（trace ID）；每个工作者都用它记录开始/结束。
- **反压。** 如果队列增长速度超过工作者的消费速度，就放慢生产者。

## 练习

1. 运行 `code/main.py`。在耗时不一的工作负载下，蜂群比顺序方式快多少？比固定分配快多少？
2. 增加一个优先级队列变体（使用 `queue.PriorityQueue`）。按任务的 "importance" 字段分配优先级。观察在持续负载下，低优先级任务是否会出现饥饿。
3. 实现一个热点检测器：当任何工作者处理的任务数达到最慢工作者的 3 倍时记录日志。这说明了任务耗时分布的什么特征？
4. 阅读 Matrix 论文（arXiv:2511.21686）的摘要与第 3 节。指出 Matrix 接受的一个具体权衡（可扩展性收益）以及它放弃的一个权衡（可追溯性、确定性）。
5. 把蜂群演示改造为使用 (task_type, payload) 元组的 `queue.Queue`，工作者只订阅特定类型。当任务异构（heterogeneous）时，哪些路由规则是合理的？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 蜂群架构（Swarm architecture） | “去中心化的智能体” | 工作者从共享队列中拉取；没有中央编排器。 |
| 事件总线（Event bus） | “智能体订阅主题” | 按类型或内容把任务路由给工作者的消息代理。 |
| 饥饿（Starvation） | “任务永远不运行” | 由于更高优先级的工作持续到来，低优先级任务永远轮不上。 |
| 热点（Hot-spotting） | “一个工作者被淹没” | 负载失衡，某一个工作者拿到了大多数任务。 |
| 反压（Back-pressure） | “放慢生产者” | 在队列被填满时向上游发出信号、令其停止生产的机制。 |
| 幂等工作者（Idempotent worker） | “重跑也安全” | 同一任务处理两次会产生相同结果。因为工作者可能在中途崩溃，所以这是必需的。 |
| 持久化队列（Durable queue） | “能扛住崩溃” | 以磁盘或复制存储为后端的队列；工作者崩溃时任务不会丢失。 |
| Matrix 框架 | “完全消息传递的蜂群” | 数据流与控制流都是分布式队列上的序列化消息。 |
