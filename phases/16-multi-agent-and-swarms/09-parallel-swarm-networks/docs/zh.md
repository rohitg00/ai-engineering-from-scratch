# 并行/群体/网络化架构

> 与监督者对比：没有中心决策者。智能体读取共享事件总线，异步拾取工作，写回结果。LangGraph 明确支持"群体架构（Swarm Architecture）"用于去中心化、动态环境。Matrix（arXiv:2511.21686）将控制流和数据流都表示为通过分布式队列传递的序列化消息，以消除编排器瓶颈。权衡是显式的：可 determinism 和 traceability 换取可扩展性。群体适合具有许多独立子问题的任务；它不适合需要单一连贯计划的任务。

**类型：** 学习 + 构建
**语言：** Python（标准库、`threading`、`queue`）
**前置条件：** 阶段 16.05（监督者模式）、阶段 16.04（原语模型）
**时长：** 约 75 分钟

## 问题背景

监督者扩展到少量工作者。那成百上千呢？监督者本身成为瓶颈：关于谁做什么的每个决策都 funnel 通过一个智能体。一个缓慢的计划步骤会停滞整个系统。

群体架构翻转了设计。不是中心规划器分发工作，而是工作者从共享队列拾取工作。"协调"被烘焙到事件总线语义中。没有编排器；系统扩展到队列能够承受的程度。

## 概念讲解

### 形状

```
                ┌──── 共享队列 ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     工作者    工作者    工作者     工作者
      A        B        C         D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            结果池
```

没有编排器。每个工作者重复：拉取任务、处理、写结果（并可选择入队后续任务）。

### 何时适合群体

- **许多独立任务。** 爬取、转换、分类。任务不相互依赖。
- **可变持续时间工作。** 如果某些任务需要 100ms 而其他需要 10s，群体自动平衡负载——快速工作者拉取下一个作业。监督者必须预测持续时间。
- **吞吐量优于确定性。** 你关心总完成时间，而不是严格的排序。

### 何时群体失败

- **有序工作流。** 如果步骤 3 需要步骤 2 的输出，群体冒着在步骤 2 完成之前触发步骤 3 的风险。
- **全局计划任务。** 复杂的研究问题受益于规划器。研究者群体产生独立事实，而不是连贯的报告。
- **调试。** 没有中心日志和异步工作，重现错误是昂贵的。

### Matrix（arXiv:2511.21686）

Matrix 是 2025 年的论文，将群体带到其自然结论：控制流和数据流都是分布式队列上的序列化消息。没有中心协调器。容错来自消息持久性。可扩展性是关于消息代理的问题，而不是系统的问题。

贡献：一个编程模型，其中多智能体协调是"这个智能体订阅什么消息主题？"而不是"监督者接下来选择哪个智能体？" 这使系统看起来像发布/订阅事件网格。

### LangGraph 的群体架构

LangGraph 2025 文档明确将"群体架构"描述为多智能体模式之一：智能体是节点，但边形成具有循环的 directed 图，并且任何节点都可以从池激活。工作者按条件拾取可用工作，而不是通过监督者分配。

### 失败模式：饥饿（starvation）和热点（hot-spotting）

如果所有工作者都拉取最快可用任务，长时间运行的任务永远不会被拾取，直到它们是唯一剩下。经典的队列饥饿。

缓解措施：
- 具有显式老化（aging）的优先队列（随等待时间增加优先级）。
- 工作者专业化：某些工作者只接受"长"任务。
- 背压（Back-pressure）：限制多少快速任务进入队列。

### 基于内容的路由链接

群体自然与基于内容的路由（第 22 课）配对。不是通用队列，而是每个消息类型一个队列。专家工作者只订阅其类型。这是扩展到数千个智能体的消息总线架构的基础。

## 构建实现

`code/main.py` 实现了一个从共享 `queue.Queue` 拉取的 4 工作者线程的群体。任务具有可变持续时间（一些快，一些慢）。演示对比：

- **顺序基线：** 一个工作者串行处理所有任务。
- **固定分配：** 每个任务预分配给特定工作者（监督者风格）。
- **群体：** 工作者从共享队列拉取。

群体自动平衡负载；当分配的任务慢时，固定分配使快速工作者闲置。

运行：

```
python3 code/main.py
```

输出显示每个工作者的任务计数（群体不均匀但最优地分发）和挂钟时间。

## 实际应用

`outputs/skill-swarm-fit.md` 评估任务是否应该使用群体 vs 监督者。输入：任务独立性、持续时间方差、排序要求、可调试性需求。

## 部署实现

检查清单：

- **具有老化的优先队列。** 防止长任务饥饿。
- **工作者幂等性（Idempotency）。** 如果工作者在运行中崩溃，任务可能会被拉取超过一次。工作者必须是幂等的。
- **持久队列。** 对生产使用 Kafka、Redis Streams 或数据库支持的队列。`queue.Queue` 仅在内存中。
- **每个任务的可观测性。** 每个任务都有一个追踪 ID；每个工作者用其记录开始/结束。
- **背压（Back-pressure）。** 如果队列增长快于工作者排干它，减慢生产者。

## 练习

1. 运行 `code/main.py`。在可变持续时间工作负载上，群体比顺序快多少？比固定分配快多少？
2. 添加优先队列变体（使用 `queue.PriorityQueue`）。按任务"重要性"字段分配优先级。观察在低优先级任务下是否曾经饥饿连续负载。
3. 实现热点检测器：记录当任何工作者处理比最慢工作者多 3 倍的任务时。关于任务持续时间分布，这表明什么？
4. 阅读 Matrix 论文（arXiv:2511.21686）摘要和第 3 节。确定 Matrix 接受的一个特定权衡（可扩展性增益）和它放弃的一个（可追溯性、确定性）。
5. 将群体演示转换为使用（`task_type`, `payload`）元组的 `queue.Queue`，工作者只订阅特定类型。当任务异构时，什么路由规则有意义？

## 关键术语

| 术语 | 人们说的 | 它实际意味着什么 |
|------|----------------|------------------------|
| Swarm architecture（群体架构） | "去中心化智能体" | 工作者从共享队列拉取；没有中心编排器。 |
| Event bus（事件总线） | "智能体订阅主题" | 按类型或内容将任务路由到工作者的消息代理。 |
| Starvation（饥饿） | "任务永不运行" | 低优先级任务永远不会被拾取，因为更高优先级的工持续到达。 |
| Hot-spotting（热点） | "一个工作者淹没" | 一个工作者获得大部分任务的负载不平衡。 |
| Back-pressure（背压） | "减慢生产者" | 当队列填满时向上游发信号停止生产的机制。 |
| Idempotent worker（幂等工作者） | "安全重新运行" | 处理两次的任务产生相同结果。因为工作者可能在运行中崩溃而需要。 |
| Durable queue（持久队列） | "在崩溃中存活" | 由磁盘或复制存储支持的队列；工作者崩溃时任务不会丢失。 |
| Matrix framework（矩阵框架） | "全消息传递群体" | 数据和控制流都是分布式队列上的序列化消息。 |
