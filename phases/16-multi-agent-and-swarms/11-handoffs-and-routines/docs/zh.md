# 11 · 交接与例程——无状态编排

> OpenAI 的 Swarm（2024 年 10 月）把多智能体编排提炼为两个原语：**例程（routines）**（作为系统提示词的指令 + 工具）和**交接（handoffs）**（一个返回另一个 Agent 的工具）。没有状态机，没有分支 DSL——由大语言模型（LLM）通过调用正确的交接工具来完成路由。OpenAI Agents SDK（2025 年 3 月）是其生产级的继任者。Swarm 本身仍是最简洁的概念参考——其全部源码只占几百行。这套模式之所以广为流行，是因为其 API 表面大致就是「agent = 提示词 + 工具；handoff = 返回 agent 的函数」。局限在于：它是无状态的，因此记忆是调用方自己的问题。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置：** 阶段 16 · 04（原语模型）
**时长：** 约 60 分钟

## 问题

每个多智能体框架都想让你学习它自己的 DSL：LangGraph 的节点与边、CrewAI 的 crew 与 task、AutoGen 的 GroupChat 与 manager。这些 DSL 都是真实有用的抽象，但它们让整件事显得比实际需要的更沉重。

Swarm 朝相反方向用力：直接使用模型已经具备的工具调用能力。交接变成了工具调用。编排器就是当前持有对话的那个智能体。状态机隐含在各个智能体的系统提示词之中。

## 概念

### 两个原语

**例程（Routine）。** 一个定义智能体角色及可用工具的系统提示词。可以把它想成一组限定范围的指令：「你是一个分诊智能体；如果用户问到退款，就交接给退款智能体。」

**交接（Handoff）。** 智能体可以调用的一个工具，它返回一个新的 Agent 对象。Swarm 运行时检测到返回值是 Agent 时，就在下一轮切换活跃智能体。

这就是整个抽象的全部。

```
def transfer_to_refunds():
    return refund_agent  # Swarm 看到返回值是 Agent → 切换活跃智能体

triage_agent = Agent(
    name="triage",
    instructions="Route the user to the right specialist.",
    functions=[transfer_to_refunds, transfer_to_sales, transfer_to_support],
)
```

分诊智能体的系统提示词使它根据用户消息选择正确的交接。路由由 LLM 的工具调用完成。

### 为什么它广为流行

- **API 很小。** 只有两个概念要学。
- **复用模型本就会做的事。** 工具调用在各家厂商那里早已是生产级能力。
- **没有状态机负担。** 你无需描述图结构；智能体的提示词描述了它们要交接给谁。

### 无状态的取舍

Swarm 在各次运行之间明确是无状态的。框架在一次运行期间会保留消息历史，但不会持久化任何东西。记忆、连续性、长时运行任务——全都是调用方的问题。

在生产环境中（OpenAI Agents SDK，2025 年 3 月），这正是发生改变的主要内容之一：SDK 在保留交接原语的同时，加入了内置的会话管理、护栏（guardrails）和追踪（tracing）。

### Swarm / 交接适合的场景

- **分诊模式。** 前线智能体把用户路由给某个专家。
- **基于技能的交接。** 「如果任务需要写代码，就调用编码智能体；如果需要做研究，就调用研究智能体。」
- **简短、有界的对话。** 客户支持、从 FAQ 到工单、简单工作流。

### Swarm 吃力的场景

- **带共享记忆的长会话。** 交接会把对话状态重置为新智能体的提示词加历史。在没有调用方管理记忆的情况下，跨智能体没有持久状态。
- **并行执行。** 交接是一次一个的——活跃智能体被切换。并行需要调用方来编排多个 Swarm 运行。
- **审计与重放。** 无状态运行很难精确重放；LLM 的交接选择并非确定性的。

### OpenAI Agents SDK（2025 年 3 月）

这个生产级继任者增加了：

- **会话状态。** 跨多次运行的持久化线程。
- **护栏。** 输入/输出校验钩子。
- **追踪。** 每次工具调用和交接都被记录。
- **交接过滤器。** 控制在交接时传递哪些上下文。

交接原语得以保留；围绕它增加了生产可用的工程化能力。

### Swarm 对比 GroupChat

两者都使用 LLM 驱动的路由，但它们在**由谁选择下一个**上有所不同：

- GroupChat：由一个选择器（函数或 LLM）从外部挑选下一个发言者。
- Swarm：当前智能体通过调用交接工具来挑选自己的继任者。

Swarm 是「智能体决定接下来是谁」；GroupChat 是「管理器决定接下来是谁」。Swarm 的决策存在于活跃智能体的工具调用里；GroupChat 的决策存在于 `GroupChatManager` 里。

## 动手构建

`code/main.py` 从零实现了 Swarm：一个 Agent 数据类、一套交接机制（工具返回 Agent），以及一个能检测智能体切换的运行循环。

演示：一个分诊智能体路由到退款、销售或支持专家。每个专家有自己的工具。运行循环会打印出每一次交接。

运行：

```
python3 code/main.py
```

## 实际运用

`outputs/skill-handoff-designer.md` 为给定任务设计一套交接拓扑：存在哪些智能体、它们可以调用哪些交接、传递哪些上下文。

## 上线交付

检查清单：

- **交接日志。** 每次交接都写入一条追踪事件，包含来源智能体、目标智能体、上下文快照。
- **上下文传递规则。** 决定在交接时传递什么：完整历史（昂贵）、最近 N 条消息，还是一份摘要。
- **交接上的护栏。** 交接到一个拥有不同工具权限的专家时必须经过身份验证——否则提示注入（prompt injection）可能强制触发不期望的交接。
- **环路检测。** 两个智能体来回交接是常见的失败模式；用一个简单的最近 K 项环形检查来发现它。
- **兜底智能体。** 如果交接目标不存在，回退到一个安全的默认值。

## 练习

1. 运行 `code/main.py`，分诊到退款智能体。确认第二轮的活跃智能体是退款智能体。
2. 添加一条环路检测规则：如果同样的两个智能体连续交接了 3 次，就强制退出。设计好兜底方案。
3. 阅读 OpenAI Agents SDK 关于交接过滤器的文档。实现一个「交接时摘要」的版本：发出方智能体在接收方智能体接管之前，把上下文压缩成要点摘要。
4. 比较 Swarm 的交接与 GroupChatManager 的选择器。哪种模式让提示注入更糟糕，为什么？
5. 阅读 Swarm cookbook（https://developers.openai.com/cookbook/examples/orchestrating_agents）。找出 Swarm 做出的一个明确设计决策，看看 OpenAI Agents SDK 是改变了它还是保留了它。

## 关键术语

| 术语 | 人们怎么说 | 它实际的含义 |
|------|----------------|------------------------|
| 例程（Routine） | 「智能体提示词」 | 系统提示词 + 工具列表。定义角色和可用的交接。 |
| 交接（Handoff） | 「转交给另一个智能体」 | 活跃智能体可调用的一个工具，它返回一个新的 Agent。运行时切换活跃智能体。 |
| 无状态（Stateless） | 「各次运行之间没有记忆」 | Swarm 不持久化任何东西；记忆是调用方的责任。 |
| 活跃智能体（Active agent） | 「现在是谁在说话」 | 当前持有对话的智能体。交接会改变它。 |
| 上下文传递（Context transfer） | 「交接时传什么」 | 关于接收方智能体看到哪些历史的策略：完整、最近 N 条，还是已摘要。 |
| 交接环路（Handoff loop） | 「智能体来回乒乓」 | 一种失败模式，两个智能体不停地把对话交还给对方。 |
| OpenAI Agents SDK | 「生产级 Swarm」 | 2025 年 3 月的继任者；在交接原语之上加入会话、护栏、追踪。 |
| 交接过滤器（Handoff filter） | 「在转交处设关卡」 | SDK 的一项功能，可在交接边界处检查并修改上下文。 |
