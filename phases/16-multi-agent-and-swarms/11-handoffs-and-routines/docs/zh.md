# 交接与例程——无状态编排

> OpenAI的Swarm（2024年10月）将多智能体编排提炼为两个原语：**例程**（指令 + 工具作为系统提示）和**交接**（返回另一个智能体的工具）。没有状态机，没有分支DSL——LLM通过调用正确的交接工具来路由。OpenAI Agents SDK（2025年3月）是生产继任者。Swarm本身仍然是最干净的概念参考——其整个源码只有几百行。该模式是病毒式的，因为API表面大致是"智能体 = 提示 + 工具；交接 = 返回智能体的函数"。限制：无状态，所以内存是调用者的问题。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置知识：** 第16阶段 · 04（原语模型）
**时间：** 约60分钟

## 问题

每个多智能体框架都想让你学习它的DSL：LangGraph节点和边、CrewAI团队和任务、AutoGen GroupChat和管理者。DSL是真实的抽象，但它们使事情感觉比需要的更重。

Swarm推向相反方向：使用模型已有的工具调用能力。交接变成工具调用。编排器是当前持有对话的任何智能体。状态机隐含在智能体的系统提示中。

## 核心概念

### 两个原语

**例程。** 定义智能体角色和可用工具的系统提示。将其视为一组有范围的指令："你是分诊智能体；如果用户询问退款，交接给退款智能体。"

**交接。** 智能体可以调用的返回新智能体对象的工具。Swarm运行时检测到智能体返回值，并为下一轮切换活动智能体。

这就是整个抽象。

```
def transfer_to_refunds():
    return refund_agent  # Swarm看到智能体返回 → 切换活动智能体

triage_agent = Agent(
    name="triage",
    instructions="Route the user to the right specialist.",
    functions=[transfer_to_refunds, transfer_to_sales, transfer_to_support],
)
```

分诊智能体的系统提示使其基于用户消息选择正确的交接。LLM的工具调用执行路由。

### 为什么它是病毒式的

- **小API。** 两个概念需要学习。
- **使用模型已有的东西。** 工具调用在各提供商之间已经是生产级的。
- **没有状态机负担。** 你不描述图；智能体的提示描述它们交接给谁。

### 无状态权衡

Swarm在运行之间明确无状态。框架在运行期间保持消息历史，但不持久化任何东西。内存、连续性、长期任务——所有调用者的问题。

在生产中（OpenAI Agents SDK，2025年3月）这是主要变化之一：SDK添加内置会话管理、防护栏和追踪，同时保持交接原语。

### Swarm/交接何时适合

- **分诊模式。** 前线智能体将用户路由到专家。
- **基于技能的交接。** "如果任务需要代码，调用编码器；如果需要研究，调用研究者。"
- **短、有界对话。** 客户支持、FAQ到工单、简单工作流。

### Swarm何时挣扎

- **带有共享内存的长期会话。** 交接将对话状态重置为新智能体的提示加历史。没有调用者管理的内存，跨智能体没有持久状态。
- **并行执行。** 交接是一次一个——活动智能体切换。并行性需要调用者编排多个Swarm运行。
- **审计和重放。** 无状态运行难以精确重放；LLM的交接选择不是确定性的。

### OpenAI Agents SDK（2025年3月）

生产继任者添加：

- **会话状态。** 跨运行的持久线程。
- **防护栏。** 输入/输出验证钩子。
- **追踪。** 每个工具调用和交接都被记录。
- **交接过滤器。** 控制交接时什么上下文转移。

交接原语存活；生产人体工程学围绕它添加。

### Swarm vs 群聊

两者都使用LLM驱动路由，但它们在**谁选择下一个**方面不同：

- 群聊：选择器（函数或LLM）从外部选择下一个说话者。
- Swarm：当前智能体通过调用交接工具选择其继任者。

Swarm是"智能体决定下一个是什么"；群聊是"管理者决定下一个是什么"。Swarm的决策存在于活动智能体的工具调用中；群聊的决策存在于`GroupChatManager`中。

## 构建它

`code/main.py`从头实现Swarm：一个智能体dataclass、交接机制（工具返回智能体），以及检测智能体切换的运行循环。

演示：分诊智能体路由到退款、销售或支持专家。每个专家有自己的工具。运行循环打印每次交接。

运行：

```
python3 code/main.py
```

## 使用它

`outputs/skill-handoff-designer.md`为给定任务设计交接拓扑：哪些智能体存在，它们可以调用哪些交接，什么上下文转移。

## 交付它

检查清单：

- **交接日志。** 每次交接写入追踪事件，包含from-agent、to-agent、上下文快照。
- **上下文转移规则。** 决定交接时什么移动：完整历史（昂贵）、最后N条消息或摘要。
- **交接防护栏。** 交接到具有不同工具权限的专家必须经过身份验证——否则提示注入可以强制不想要的交接。
- **循环检测。** 两个智能体来回交接是常见失败；用简单的最后K环检查检测。
- **回退智能体。** 如果交接目标不存在，回退到安全默认值。

## 练习

1. 运行`code/main.py`，分诊到退款智能体。确认第二轮的活动智能体是退款。
2. 添加循环检测规则：如果相同的两个智能体连续交接3次，强制退出。设计回退。
3. 阅读OpenAI Agents SDK文档关于交接过滤器。实现"交接时摘要"版本：传出智能体在传入智能体接管之前将上下文压缩为项目符号摘要。
4. 比较Swarm交接与GroupChatManager选择器。哪个模式使提示注入更糟，为什么？
5. 阅读Swarm cookbook（https://developers.openai.com/cookbook/examples/orchestrating_agents）。识别Swarm做出的一个明确设计决策，OpenAI Agents SDK改变或保留了它。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 例程 | "智能体提示" | 系统提示 + 工具列表。定义角色和可用交接。 |
| 交接 | "转移到另一个智能体" | 活动智能体可以调用的返回新智能体的工具。运行时切换活动智能体。 |
| 无状态 | "运行之间没有内存" | Swarm不持久化任何东西；内存是调用者的责任。 |
| 活动智能体 | "现在谁在说话" | 当前持有对话的智能体。交接改变这个。 |
| 上下文转移 | "交接时什么移动" | 传入智能体看到什么历史的策略：完整、最后N或摘要。 |
| 交接循环 | "智能体乒乓" | 两个智能体持续交接回彼此的失败模式。 |
| OpenAI Agents SDK | "生产Swarm" | 2025年3月继任者；在交接原语之上添加会话、防护栏、追踪。 |
| 交接过滤器 | "转移门控" | SDK功能，在交接边界检查和修改上下文。 |
