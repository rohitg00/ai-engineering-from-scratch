# 交接与例程 —— 无状态编排

> OpenAI 的 Swarm（2024 年 10 月）将多智能体编排提炼为两个原语：**例程（routines）**（指令 + 工具作为系统提示词）和**交接（handoffs）**（返回另一个智能体的工具）。没有状态机，没有分支 DSL——LLM 通过调用正确的交接工具进行路由。OpenAI Agents SDK（2025 年 3 月）是生产后继者。Swarm 本身仍然是最清晰的概念参考——其整个源代码适合几百行。这个模式具有病毒式传播性，因为 API 表面大致是"智能体 = 提示词 + 工具；交接 = 返回智能体的函数。" 限制：无状态，所以内存是调用者的问题。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置条件：** 阶段 16.04（原语模型）
**时长：** 约 60 分钟

## 问题背景

每个多智能体框架都想让你学习它的 DSL：LangGraph 节点和边，CrewAI 团队和任务，AutoGen GroupChat 和管理器。DSL 是真实的抽象，但它们让事情感觉比需要的更重。

Swarm 向相反方向推进：使用模型已经拥有的工具调用能力。交接成为工具调用。编排器是当前的智能体持有对话。状态机隐含在智能体的系统提示词中。

## 概念讲解

### 两个原语

**例程（Routine）。** 定义智能体角色和可用工具的系统提示词。把它想象成一组范围化指令："你是一个分诊智能体；如果用户询问退款，交接给退款智能体。"

**交接（Handoff）。** 智能体可以调用的返回新智能体对象的工具。Swarm 运行时检测智能体返回值并切换下一轮的活跃智能体。

这就是整个抽象。

```
def transfer_to_refunds():
    return refund_agent  # Swarm 看到智能体返回 → 切换活跃智能体

triage_agent = Agent(
    name="triage",
    instructions="Route the user to the right specialist.",
    functions=[transfer_to_refunds, transfer_to_sales, transfer_to_support],
)
```

分诊智能体的系统提示词使其基于用户消息选择正确的交接。LLM 的工具调用进行路由。

### 为什么它具有病毒式传播性

- **小型 API。** 两个概念要学习。
- **使用模型已经做的事情。** 工具调用已经跨提供商生产级别。
- **无状态机负担。** 你不描述图；智能体的提示词描述它们交接给谁。

### 无状态权衡

Swarm 在运行之间明确无状态。框架在运行期间保持消息历史，但它不持久化任何东西。内存、连续性、长时间运行的任务——都是调用者的问题。

在生产中（OpenAI Agents SDK，2025 年 3 月），这是改变的主要事情之一：SDK 添加了内置会话管理、护栏和追踪，同时保持交接原语。

### 何时适合 Swarm/交接

- **分诊模式。** 前线智能体将用户路由到专家。
- **基于技能的交接。** "如果任务需要代码，调用编码员；如果需要研究，调用研究员。"
- **简短、有边界的对话。** 客户支持、FAQ 到工单、简单工作流。

### 何时 Swarm 挣扎

- **具有共享内存的长会话。** 交接将对话状态重置为新智能体的提示词加上历史。没有调用者管理的内存，就无法跨智能体持久化状态。
- **并行执行。** 交接是一次一个——活跃智能体切换。并行性需要调用者编排多个 Swarm 运行。
- **审计和重放。** 无状态运行很难精确重放；LLM 的交接选择不是确定性的。

### OpenAI Agents SDK（2025 年 3 月）

生产后继者添加了：

- **会话状态。** 跨运行持久化线程。
- **护栏。** 输入/输出验证钩子。
- **追踪。** 每个工具调用和交接都被记录。
- **交接过滤器。** 控制什么上下文在交接时转移。

交接原语保留；生产人体工程学围绕它添加。

### Swarm vs GroupChat

两者都使用 LLM 驱动的路由，但它们在**谁选择下一个**上不同：

- GroupChat：选择器（函数或 LLM）从外部选择下一个发言人。
- Swarm：当前智能体通过调用交接工具选择其继任者。

Swarm 是"智能体决定下一个是什么"；GroupChat 是"管理者决定下一个是什么。" Swarm 的决策存在于活跃智能体的工具调用中；GroupChat 的决策存在于 `GroupChatManager` 中。

## 构建实现

`code/main.py` 从头实现 Swarm：一个智能体数据类、一个交接机制（工具返回智能体）和一个检测智能体切换的运行循环。

演示：分诊智能体路由到退款、销售或支持专家。每个专家都有自己的工具。运行循环打印每个交接。

运行：

```
python3 code/main.py
```

## 实际应用

`outputs/skill-handoff-designer.md` 为给定任务设计交接拓扑：哪些智能体存在，它们可以调用什么交接，什么上下文转移。

## 部署实现

检查清单：

- **交接日志记录。** 每个交接都写一个带有从智能体、到智能体、上下文快照的追踪事件。
- **上下文转移规则。** 决定在交接时移动什么：完整历史（昂贵）、最后 N 条消息或摘要。
- **交接护栏。** 具有不同工具权限的交接必须被认证——否则提示词注入可以强制不需要的交接。
- **循环检测。** 两个智能体来回交接是常见失败；用简单的最后 K 环检查检测。
- **回退智能体。** 如果交接目标不存在，回退到安全的默认。

## 练习

1. 运行 `code/main.py`，分诊到退款智能体。确认第二轮次的活跃智能体是退款。
2. 添加循环检测规则：如果相同的两个智能体连续交接 3 次，强制退出。设计回退。
3. 阅读 OpenAI Agents SDK 文档关于交接过滤器。实现一个"在交接时总结"版本：传出智能体在传入智能体接管之前将上下文压缩为项目符号摘要。
4. 将 Swarm 交接与 GroupChatManager 选择器进行比较。哪种模式使提示词注入更糟，为什么？
5. 阅读 Swarm 食谱（https://developers.openai.com/cookbook/examples/orchestrating_agents）。确定 Swarm 做出的一个显式设计决策，OpenAI Agents SDK 改变或保留了它。

## 关键术语

| 术语 | 人们说的 | 它实际意味着什么 |
|------|----------------|------------------------|
| Routine（例程） | "智能体提示词" | 系统提示词 + 工具列表。定义角色和可用交接。 |
| Handoff（交接） | "转移到另一个智能体" | 活跃智能体可以调用的返回新智能体的工具。运行时切换活跃智能体。 |
| Stateless（无状态） | "运行之间没有内存" | Swarm 不持久化任何东西；内存是调用者的责任。 |
| Active agent（活跃智能体） | "谁现在说话" | 当前持有对话的智能体。交接改变这个。 |
| Context transfer（上下文转移） | "在交接时移动什么" | 传入智能体看到什么历史的策略：完整、最后 N 个或摘要。 |
| Handoff loop（交接循环） | "智能体乒乓球" | 两个智能体不断交接回彼此的失败模式。 |
| OpenAI Agents SDK | "生产 Swarm" | 2025 年 3 月后继者；在交接原语之上添加会话、护栏、追踪。 |
| Handoff filter（交接过滤器） | "转移门控" | SDK 特征，用于在交接边界检查和修改上下文。 |
