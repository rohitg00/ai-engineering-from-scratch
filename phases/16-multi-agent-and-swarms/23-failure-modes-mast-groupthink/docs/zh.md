# 23 · 失败模式 —— MAST、群体思维、单一文化与级联错误

> 2026 年的参考分类法是 **MAST**（Cemri 等人，NeurIPS 2025，arXiv:2503.13657），它源自对 7 个最先进的开源多智能体系统（Multi-Agent System, MAS）的 1642 条执行轨迹的分析，显示出 **41%–86.7% 的失败率**。三大根本类别为：**规格问题（Specification Problems）**（41.77%）—— 角色含糊、任务定义不清；**协调失败（Coordination Failures）**（36.94%）—— 通信中断、状态失同步；**验证缺口（Verification Gaps）**（21.30%）—— 缺少校验、缺失质量检查。**群体思维（Groupthink）** 家族（arXiv:2508.05687）又补充了：单一文化崩溃（同一基础模型 → 关联性失败）、从众偏差（智能体相互强化彼此的错误）、心智理论缺失、动机混合动态、级联式可靠性失败。级联示例：重试风暴中，一次支付失败触发订单重试，订单重试又触发库存重试，最终压垮库存服务（数秒内负载达到 10 倍 —— 需要断路器）。记忆投毒：某个智能体的幻觉进入共享记忆，下游智能体把它当作事实；准确率逐渐衰减，使得根因诊断异常痛苦。**STRATUS**（NeurIPS 2025）报告称，通过专门的检测 / 诊断 / 验证智能体，可将缓解成功率提升 1.5 倍。本课把失败模式当作一等的工程目标来对待。

**类型：** 学习
**语言：** Python（标准库）
**前置：** 阶段 16 · 13（共享记忆）、阶段 16 · 14（共识与 BFT）、阶段 16 · 15（投票与辩论拓扑）
**时长：** 约 75 分钟

## 问题

多智能体系统在真实任务上有 41%–86.7% 的概率失败（Cemri 等人于 2025 年在 7 个开源 MAS 上实测得出）。这种问题无法靠"多加几个智能体"来调试。这些失败有结构性的成因。MAST 分类法给了你类别框架。本课把每个类别都对应到一套具体的检测、诊断与缓解模式，让这些数字不再显得任意。

2026 年的生产实践是把失败模式当作设计输入。在你能够指着每一个 MAST 类别、并说出你部署了什么缓解措施之前，你的架构都不算"足够好"。

## 概念

### MAST 类别

**规格问题（占失败的 41.77%）。** 智能体的任务定义得不够严密。示例：

- 角色含糊：两个智能体都认为自己是评审者。
- 任务欠规约：用户想要某个特定角度，却只说了"总结一下这个"。
- 成功标准隐含：智能体无法判断自己是否成功。

缓解措施：
- 编写明确的角色契约。每个智能体的提示词要写清它做什么*以及不做什么*。
- 为每个任务设置验收测试。在智能体开始之前，定义"完成的样子是 X"。
- 飞行前规格检查：派发任务之前，由一个独立的智能体审阅任务定义。

**协调失败（36.94%）。** 通信或状态层面的崩溃。

示例：
- 两个智能体在没有同步的情况下更新共享状态。
- 消息在智能体之间丢失（队列故障、超时）。
- 状态漂移：智能体 A 认为任务已完成；智能体 B 还在执行。

缓解措施：
- 带版本的共享状态，配合乐观并发控制。
- 对关键消息进行显式确认（重试直至收到确认）。
- 周期性状态同步检查点；尽早检测漂移。

**验证缺口（21.30%）。** 对输出没有独立的检查。

示例：
- 一个智能体声称成功；无人验证。
- 智能体链条中，每一个都信任前一个的输出。
- 在涌现出的组合行为上缺少测试覆盖。

缓解措施：
- 独立的验证者智能体（第 13 课）。只读、独立的数据源访问权限。
- 显式的交接契约："A 的输出必须先通过检查器 C，B 才能开始。"
- 记录结果，便于事后分析。

### 群体思维家族（arXiv:2508.05687）

当智能体趋同或相互模仿时，会出现五种相关的失败：

**单一文化崩溃（Monoculture collapse）。** 相同的基础模型或训练数据 → 关联性错误。当三个智能体共用一个 LLM 时，它们就共享了它的幻觉。

**从众偏差（Conformity bias）。** 智能体会朝着声音最大或最自信的同伴调整，即便对方是错的。

**心智理论缺失（Deficient ToM）。** 智能体无法对彼此的信念建模；协调随之崩溃（第 18 课）。

**动机混合动态（Mixed-motive dynamics）。** 激励只是部分对齐的智能体会漂向折中的中间地带，结果谁都不满意。

**级联式可靠性失败（Cascading reliability failures）。** 一个组件的错误模式会触发其依赖组件中的错误模式。

### 级联示例 —— 重试风暴

一个经典的 2026 年事故模式：

```
payment service fails 10% of requests
   ↓
order agent retries payment (exponential backoff but naive)
   ↓
each retry is a new order-inventory check
   ↓
inventory service sees 2x normal load
   ↓
inventory service starts timing out
   ↓
every order retries inventory check
   ↓
inventory service sees 10x normal load
   ↓
cluster goes down
```

修复方式是经典的：**断路器（circuit breakers）**。当下游错误率超过阈值时，短路返回缓存结果或默认结果。再加上每个请求设上限的重试预算。

断路器是少数几个你可以原封不动地从分布式系统直接借用的多智能体失败缓解措施之一。

### 记忆投毒（再探）

源自第 13 课：某个智能体的幻觉成为共享记忆中的"事实"；下游智能体基于这个被投毒的事实进行推理。用 MAST 的术语来说，这是共享记忆层的一个验证缺口。

逐渐的准确率衰减是其症状。你不会遭遇崩溃；你得到的是难以追溯根因的缓慢漂移。

缓解措施：仅追加（append-only）日志、来源溯源（provenance）、不可写的验证者。已在第 13 课覆盖。

### STRATUS —— 用于失败检测的专门化智能体

STRATUS（NeurIPS 2025）报告称，当你部署以下三类智能体时，缓解成功率可提升 1.5 倍：

- **检测智能体（Detection agent）。** 监视症状模式（高分歧、重试激增、准确率漂移）。
- **诊断智能体（Diagnosis agent）。** 给定症状，从 MAST 分类法推断可能的根因。
- **验证智能体（Validation agent）。** 在应用缓解措施之后，检查症状是否消除。

这就是 SRE 风格的事故响应，应用到智能体系统上。这三个角色都可以是配有专门提示词的 LLM 智能体。

### 失败模式审计

2026 年的一个最佳实践是开展年度（或每个大版本一次）的失败模式审计：

1. **采样轨迹。** 收集约 1000 条真实执行轨迹。
2. **分类。** 对每条轨迹中的失败，映射到 MAST + 群体思维类别。
3. **计算按类别的失败率。** 哪些类别在你的系统中占主导？
4. **缓解措施排序。** 哪个修复能消除最多的失败？
5. **挑选 2–3 个缓解措施。** 实施；下个季度重新审计。

这套纪律比具体的选择更重要。没有审计，失败就会混入噪声，永远得不到系统性的处理。

### 当系统静默失败时

最危险的失败类别是静默的正确性失败。会大声失败（崩溃、异常、告警）的系统是可以被监控的。而产出"看似合理但实际错误"输出的系统，无法靠异常日志检测出来。这就是为什么验证缺口虽然按数量只占 21.30%，却是单次失败成本最高的类别。

应投入于：
- 基于抽样的人工审查。
- 黄金数据集（golden-dataset）回归测试。
- 对重要输出进行跨智能体交叉核验。

### 失败 vs 慢失败

有些失败是即时的；有些是缓慢的。即时失败（超时、模式不匹配、鉴权错误）检测成本低。慢失败（记忆投毒、单一文化漂移、角色含糊）则检测和预防成本高。

2026 年的工程做法：为慢失败设置代理指标（proxy）并加以监测，以便在它演变为可见错误之前捕捉到漂移。一致率、重试率、输出长度分布，以及相邻智能体版本之间的编辑距离，都是有用的代理指标。

`outputs/skill-mast-auditor.md` 对一个多智能体系统执行 MAST 风格的失败模式审计。轨迹 → 分类 → 缓解措施排序。

## 上线交付

生产环境中的失败模式纪律：

- **每季度一次 MAST 审计。** 不是每年。随着系统成长，类别构成会发生变化。
- **处处部署断路器。** 对任何依赖服务的每一次出站调用都要设。默认打开阈值设在 5%–10% 错误率。
- **黄金数据集。** 小而高质量、人工审核过。每周对其做回归测试。
- **STRATUS 三件套。** 检测 + 诊断 + 验证智能体监控生产。先只上检测智能体；当症状嘈杂时再加入诊断。
- **失败预算。** 为每个类别的失败率设定明确的 SLO。超出预算就触发一次"停止发布"的对话。

## 练习

1. 运行 `code/main.py`。确认断路器封住了重试风暴。改变失败阈值，观察其中的权衡。
2. 实现一个**慢失败代理指标**：3 个并行智能体之间的一致率。当它急剧下降时触发告警。通过逐步让智能体输出相互关联，模拟一次单一文化漂移。
3. 阅读 Cemri 等人的论文（arXiv:2503.13657）。从他们的 7 个 MAS 系统中挑一个，映射出它排名前 3 的失败类别。这些与 MAST 的预测相比如何？
4. 阅读群体思维论文（arXiv:2508.05687）。指出五种模式中哪一种在生产环境里最难检测。提出一个代理指标。
5. 为你熟悉的某个具体多智能体系统设计一套 STRATUS 风格的检测-诊断-验证三件套。检测要监视哪些症状？诊断会推荐哪些缓解措施？验证如何确认它们生效？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| MAST | "2026 年的分类法" | Cemri 2025；3 个根本类别 + 14 个失败子类型。 |
| 规格问题（Specification Problem） | "角色含糊" | 任务或角色定义不足；智能体不知道该做什么。 |
| 协调失败（Coordination Failure） | "状态漂移" | 智能体之间的通信或同步崩溃。 |
| 验证缺口（Verification Gap） | "没人检查" | 输出未经独立校验就被接受。 |
| 群体思维家族（Groupthink family） | "同质化失败" | 单一文化、从众、心智理论缺失、动机混合、级联。 |
| 单一文化崩溃（Monoculture collapse） | "同一个模型，同样的幻觉" | 来自共享基础模型或训练数据的关联性错误。 |
| 重试风暴（Retry storm） | "级联错误放大" | 一次失败触发重试，重试又放大下游负载。 |
| 断路器（Circuit breaker） | "按错误率快速失败" | 错误率超过阈值时打开；用默认值短路返回。 |
| STRATUS | "事故响应三件套" | 检测 + 诊断 + 验证智能体。缓解成功率提升 1.5 倍。 |
| 记忆投毒（Memory poisoning） | "幻觉传播" | 共享记忆中的事实被污染；下游智能体基于毒源推理。 |
