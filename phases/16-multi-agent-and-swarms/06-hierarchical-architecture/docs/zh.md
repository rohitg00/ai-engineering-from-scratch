# 06 · 分层架构及其失效模式

> 分层（hierarchical）就是监督者（supervisor）的嵌套：经理智能体之上还有经理，经理之下管着子经理，子经理之下再管工人。CrewAI 的 `Process.hierarchical` 是教科书式的版本：一个 `manager_llm` 动态地分派任务并校验产出。LangGraph 中的对应物是 `create_supervisor(create_supervisor(...))`。当任务本身就是一张真实的组织架构图时，这是最自然的模式；但它也是最容易塌缩为「经理式循环」的模式——经理智能体糟糕地分派工作、误读子产出，或者迟迟无法达成共识。顺序式（sequential）往往反而更胜一筹。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置：** 阶段 16 · 05（监督者模式）
**时长：** 约 60 分钟

## 问题

一旦监督者模式想通了，下一个自然的念头就是「如果工人本身也是监督者会怎样？」团队下面有子团队，公司里有部门套着部门。分层架构正是对这种结构的映射。

问题在于：LLM 经理和人类经理并不一样。人类经理对下属知道什么有着稳定的先验认知；而 LLM 经理在每一轮里都会根据上下文中现有的任何内容，从头重新推理整个组织。该上下文哪怕只有微小的漂移，整棵树都会错误地分配工作。

## 概念

### 形态

```
                 Manager
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       Sub-Mgr A         Sub-Mgr B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点都负责规划、分派和综合。只有叶子节点才真正干活。

### 它擅长的场景

- **清晰的组织映射。** 如果真实任务是部门性的（「法务审查这份文档，财务审查这份文档，工程审查这份文档，然后为高管做总结」），层级结构就是显式的。
- **局部综合。** 每个子经理在顶层经理看到之前，先对本团队的产出做综合。顶层经理看到的是三份子经理摘要，而不是十五份工人产出。

### 它失效的场景

2026 年的多份事后复盘反复发现三种失效模式：

1. **任务分派错误（task assignment error）。** 经理读取目标，幻觉出一种任务分解，然后把它分派给错误的子经理。由于子经理会乖乖地处理分给它的内容，错误只有到顶层综合时才会暴露——这比人类本可以捕捉到错误的位置晚了一层。
2. **产出误读（output misinterpretation）。** 子经理返回「无法验证主张 X」。顶层经理将其总结为「主张 X 未被确认」。含义在每一层都发生漂移。
3. **共识循环（consensus loop）。** 两个子经理意见相左；顶层经理要求它们协调；它们又向下重新分派；工人重新运行；子经理返回略有不同的答案；如此循环。CrewAI 的 `Process.hierarchical` 用步数上限来防范这一点，但这个上限本身如今又成了一个超参数。

### 决定性的问题

顺序式（线性流水线）对比分层式：你的任务是否真的存在彼此独立的子团队，还是说它只是一条假装成树状结构的线性流程？如果是后者，请用顺序式；如果是前者，请用分层式，但要为显式的协调规则预留预算。

### CrewAI 的实现

`Process.hierarchical` 在多个专家团队（crew）之上接入一个经理 LLM。这个经理会：

- 接收顶层任务，
- 把子任务分派给各团队，
- 评估各团队的产出，
- 决定是接受、重新分派还是迭代。

文档：https://docs.crewai.com/en/introduction （在 Core Concepts 下查找「Hierarchical Process」）。

### LangGraph 的实现

LangGraph 使用嵌套的 `create_supervisor` 调用。内层监督者拥有自己的图（graph），外层监督者把内层图当作一个不透明节点来对待。这在调试时比 CrewAI 更清晰（你可以分别单步调试每张图），但更难表达对树结构的动态重塑。

参考：https://reference.langchain.com/python/langgraph-supervisor 。

## 动手构建

`code/main.py` 运行一个 3 层级的层级结构：

- 顶层经理：把任务拆分为「工程」和「法务」两个分支，
- 工程子经理：拆分为「前端」和「后端」两个工人，
- 法务子经理：一个工人。

演示对比了顺利路径（所有人意见一致）与一条**被扰动的路径（perturbed path）**：顶层经理的分解把「法务」误标为「财务」，然后观察错误如何级联——子经理乖乖地做起财务工作，顶层综合者报告财务方面的发现，而原本的法务问题始终没有得到回答。

运行：

```
python3 code/main.py
```

输出会展示两条路径，并清晰地并列对比「被要求做什么」与「实际交付了什么」。

## 应用

`outputs/skill-hierarchy-fitness.md` 评估一个给定任务应当采用分层式、顺序式，还是扁平监督者（flat supervisor）。输入：任务描述、组织结构、协调预算。输出：模式推荐，并附上需要防范的具体失效模式。

## 上线交付

如果你要上线分层式：

- **将树深度限制在 2 层。** 三层级就已经会让大多数错误对可观测性（observability）不可见。
- **显式的协调预算。** 设定顶层经理必须做出最终决定之前的最大轮数，通常是 2。
- **每次综合都要带来源（provenance）。** 每个节点的摘要都必须注明是哪些叶子产出生成了它。
- **对分解漂移（decomposition drift）告警。** 逐步记录经理的分解结果；与用户查询做差异比对。如果分解结果不再覆盖该查询，就触发告警。

## 练习

1. 运行 `code/main.py`，对比顺利路径与被扰动路径。需要经过多少层经理交接，顶层输出才会与用户的问题完全偏离？
2. 增加第三层（顶层 → 子 → 子子 → 工人）。随着深度增长，测量被扰动路径自我纠正与完全偏离各自发生的频率。
3. 在每个子经理处实现一个「金丝雀（canary）」工人，它始终被原封不动地问以原始用户问题。用金丝雀的答案来检测分解漂移。当金丝雀与综合后的答案不一致时，经理应当如何反应？
4. 阅读 CrewAI 的 `Process.hierarchical` 文档。找出 CrewAI 应用的一个具体护栏（步数上限、`manager_llm` 约束），并描述它针对的是哪种失效模式。
5. 对比嵌套的 LangGraph 监督者与 CrewAI 分层式。哪一种能让协调循环更廉价地被检测出来？

## 关键术语

| 术语 | 人们口中的说法 | 实际含义 |
|------|----------------|------------------------|
| 分层（Hierarchical） | 「组织架构图模式」 | 监督者套着监督者；只有叶子节点干活。 |
| 经理 LLM（Manager LLM） | 「老板」 | 在内部节点负责分解、分派和校验的 LLM。 |
| 分解漂移（Decomposition drift） | 「老板跑题了」 | 顶层经理的拆分不再覆盖原始问题。 |
| 协调循环（Reconciliation loop） | 「没完没了的会」 | 子经理意见相左；顶层重新分派；工人重新运行；循环直到预算耗尽。 |
| 深度 2 上限（Depth-2 ceiling） | 「别超过 2 层」 | 经验性护栏：3 层及以上会让可观测性崩塌。 |
| 金丝雀问题（Canary question） | 「每一层的基准事实」 | 一个始终被原封不动问以原始查询的工人，用于检测漂移。 |
| 来源链（Provenance chain） | 「谁说了什么」 | 从每次综合追溯回生成它的那些叶子产出。 |
