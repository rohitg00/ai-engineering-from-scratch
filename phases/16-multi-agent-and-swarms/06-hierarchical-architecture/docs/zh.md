# 层级架构及其失败模式

> 层级是嵌套的监督者。管理智能体在子管理智能体之上，子管理智能体在工作者之上。CrewAI的`Process.hierarchical`是教科书版本：一个`manager_llm`动态委派任务并验证输出。LangGraph的等价物是`create_supervisor(create_supervisor(...))`。当任务是真正的组织结构图时，这是自然的模式。它也是最容易崩溃为管理循环的模式——管理智能体分配工作不当，误解子输出，或无法达成共识。顺序模式通常胜过它。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置知识：** 第16阶段 · 05（监督者模式）
**时间：** 约60分钟

## 问题

一旦监督者模式被理解，自然的下一步是"如果工作者本身就是监督者呢？"团队有子团队；公司有部门的部门。层级架构反映了这一点。

问题：LLM管理者与人类管理者不同。人类管理者对其下属知道什么有稳定的先验。LLM管理者每轮从其上下文中的任何内容重新推理组织。上下文中的微小漂移，整个树就会错误分配工作。

## 核心概念

### 形状

```
                 管理者
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       子管理A           子管理B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点计划、委派和综合。只有叶子节点做工作。

### 它闪耀的地方

- **清晰的组织映射。** 如果真实任务是部门性的（"法律审查文档，财务审查文档，工程审查文档，然后为高管总结"），层级是显式的。
- **本地综合。** 每个子管理者在其团队输出到达顶级管理者之前进行综合。顶级管理者看到三个子管理者摘要，不是十五个工作者输出。

### 它崩溃的地方

2026年事后分析不断发现的三种失败模式：

1. **任务分配错误。** 管理者读取目标，幻觉分解，并委派给错误的子管理者。因为子管理者顺从地处理给它的内容，错误只在顶级综合时浮出水面——比人类能够捕获它的地方高一级。
2. **输出误解。** 子管理者返回"无法验证声明X"。顶级管理者总结为"声明X未确认。"含义在每层漂移。
3. **共识循环。** 两个子管理者不同意；顶级管理者要求他们协调；他们向下重新委派；工作者重新运行；子管理者返回 slightly 不同的答案；循环。CrewAI的`Process.hierarchical`用步骤限制防范这一点，但限制本身现在是超参数。

### 决定性问题

顺序（线性流水线）vs 层级：你的任务实际上有独立的子团队，还是一个假装成树的线性流？如果是后者，使用顺序。如果是前者，使用层级但预算显式协调规则。

### CrewAI的实现

`Process.hierarchical`在专业团队之上连接一个管理LLM。管理者：

- 接收顶级任务，
- 将子任务分配给团队，
- 评估团队输出，
- 决定接受、重新委派还是迭代。

文档：https://docs.crewai.com/en/introduction（在核心概念下查找"层级流程"）。

### LangGraph的实现

LangGraph使用嵌套的`create_supervisor`调用。内部监督者有自己的图；外部监督者将内部图视为不透明节点。这比CrewAI更干净用于调试（你可以分别逐步通过每个图），但更难表达树的动态重塑。

参考：https://reference.langchain.com/python/langgraph-supervisor。

## 构建它

`code/main.py`运行一个3级层级：

- 顶级管理者：将任务拆分为"工程"和"法律"分支，
- 工程子管理者：拆分为"前端"和"后端"工作者，
- 法律子管理者：一个工作者。

演示对比快乐路径（每个人都同意）与**扰动路径**，其中顶级管理者的分解将"法律"误标为"财务"，并观察错误级联——子管理者顺从地做财务工作，顶级综合器报告财务发现，原始法律问题未得到回答。

运行：

```
python3 code/main.py
```

输出显示两条路径，并清晰并排"要求了什么" vs "交付了什么"。

## 使用它

`outputs/skill-hierarchy-fitness.md`评估给定任务是否应该使用层级、顺序或扁平监督者。输入：任务描述、组织结构、协调预算。输出：模式建议以及需要防范的特定失败模式。

## 交付它

如果你交付层级：

- **将树深度限制为2。** 三个级别已经将大多数错误隐藏在可观察性之外。
- **显式协调预算。** 在顶级管理者必须提交之前设置最大轮数。通常是2。
- **每个综合的来源。** 每个节点的摘要必须引用哪些叶子输出产生了它。
- **分解漂移警报。** 记录管理者每步的分解；与用户查询进行差异比较。如果分解不再覆盖查询，触发警报。

## 练习

1. 运行`code/main.py`并比较快乐路径与扰动路径。在顶级输出完全偏离用户问题之前，需要多少级管理者交接？
2. 添加第三级（顶级 → 子 → 子子 → 工作者）。测量扰动路径自我纠正与完全偏离的频率随深度增长。
3. 在每个子管理者处实现一个"金丝雀"工作者，始终被问到未改变原始用户问题。使用金丝雀答案检测分解漂移。当金丝雀与综合答案不一致时，管理者应该如何反应？
4. 阅读CrewAI的`Process.hierarchical`文档。识别CrewAI应用的一个具体防护栏（步骤限制、manager_llm约束）并描述它针对什么失败模式。
5. 比较嵌套LangGraph监督者与CrewAI层级。哪个使协调循环更便宜地检测？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 层级 | "组织结构图模式" | 监督者之上的监督者；只有叶子节点做工作。 |
| 管理LLM | "老板" | 在内部节点分解、分配和验证的LLM。 |
| 分解漂移 | "老板失去了情节" | 顶级管理者的拆分不再覆盖原始问题。 |
| 协调循环 | "无尽会议" | 子管理者不同意；顶级重新委派；工作者重新运行；循环直到预算耗尽。 |
| 深度-2上限 | "不要深入超过2级" | 经验防护栏：3+级崩溃可观察性。 |
| 金丝雀问题 | "每级的基本事实" | 始终被问到未改变原始查询的工作者，用于检测漂移。 |
| 来源链 | "谁说了什么" | 从每个综合追溯到产生它的叶子输出的追踪。 |
