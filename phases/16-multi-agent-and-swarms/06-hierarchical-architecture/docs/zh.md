# 分层架构及其失败模式

> 分层是监督者的嵌套。管理者智能体在工作者之上还有子管理者。CrewAI 的 `Process.hierarchical` 是教科书版本：一个 `manager_llm` 动态委派任务并验证输出。LangGraph 的等价物是 `create_supervisor(create_supervisor(...))`。当任务是真实的组织架构时，这是自然的模式。这也是最容易崩溃为管理循环的模式——管理者智能体分配工作不力、误解子输出或无法达成共识。顺序模式通常胜过它。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置条件：** 阶段 16.05（监督者模式）
**时长：** 约 60 分钟

## 问题背景

一旦监督者模式被理解，自然的下一步是"如果工作者本身也是监督者呢？" 团队有子团队；公司部门有子部门。分层架构反映了这一点。

问题：LLM 管理者与人类管理者不同。人类管理者对其下属知道什么有稳定的先验。LLM 管理者从它上下文中的任何内容每一轮重新推理组织。该上下文中的微小漂移，整个树就会错误分配工作。

## 概念讲解

### 形状

```
                 Manager
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       Sub-Mgr A         Sub-Mgr B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点规划、委派和综合。只有叶子节点做工作。

### 它的亮点

- **清晰的组织映射。** 如果真实任务是部门性的（"法务审查文档，财务审查文档，工程审查文档，然后为高管总结"），层级结构是显式的。
- **本地汇总。** 在顶层管理者看到之前，每个子管理者综合其团队的输出。顶层管理者看到三个子管理者摘要，而不是十五个工作者输出。

### 它的崩溃点

2026 年事后分析持续发现的三种失败模式：

1. **任务分配错误。** 管理者读取目标，幻觉分解，并委派给错误的子管理者。因为子管理者顺从地工作在分配给它的内容上，错误只在顶层综合时浮出水面——离人类本可以发现问题的一级之遥。
2. **输出误解。** 子管理者返回"无法验证声明 X。" 顶层管理者总结为"声明 X 未确认。" 含义在每个层级漂移。
3. **共识循环。** 两个子管理者不同意；顶层管理者要求他们调和；他们重新向下委派；工作者重新运行；子管理者返回略有不同的答案；循环。CrewAI 的 `Process.hierarchical` 用步骤限制防范这一点，但限制本身现在是一个超参数。

### 决定性问题

顺序（线性流水线）vs 分层：你的任务是否真的有独立的子团队，还是它是一个假装是树的线性流？如果是后者，使用顺序。如果是前者，使用分层但预算明确的调和规则。

### CrewAI 的实现

`Process.hierarchical` 连接管理者 LLM 在专家团队之上。管理者：

- 接收顶层任务，
- 将子任务分配给团队，
- 评估团队输出，
- 决定是否接受、重新委派或迭代。

文档：https://docs.crewai.com/en/introduction（在"核心概念"下查找"Hierarchical Process"）。

### LangGraph 的实现

LangGraph 使用嵌套的 `create_supervisor` 调用。内部监督者有自己的图；外部监督者将内部图视为不透明节点。这比 CrewAI 更清晰便于调试（你可以分别单步执行每个图），但更难表达树的动态重塑。

参考：https://reference.langchain.com/python/langgraph-supervisor。

## 构建实现

`code/main.py` 运行一个 3 级分层：

- 顶层管理者：将任务拆分为"工程"和"法务"分支，
- 工程子管理者：拆分为"前端"和"后端"工作者，
- 法务子管理者：一个工作者。

演示对比快乐路径（每个人都同意）与**扰动路径**，其中顶层管理者的分解将"法务"错误标记为"财务"并观察错误级联——子管理者顺从地做财务工作，顶层综合器报告财务发现，原始法务问题没有得到回答。

运行：

```
python3 code/main.py
```

输出显示两条路径，清晰并排展示"被问到什么"与"交付了什么"。

## 实际应用

`outputs/skill-hierarchy-fitness.md` 评估给定任务是否应该使用分层、顺序或平面监督者。输入：任务描述、组织结构、调和预算。输出：模式建议及要防范的特定失败模式。

## 部署实现

如果你部署分层：

- **将树深度上限设为 2。** 三级已经隐藏了大部分可观测性错误。
- **明确的调和预算。** 在顶层管理者必须提交之前设置最大轮数。通常是 2。
- **每个综合上的来源。** 每个节点的摘要必须引用产生它的叶子输出。
- **分解漂移警报。** 记录管理者每步的分解；与用户查询对比。如果分解不再覆盖查询，触发警报。

## 练习

1. 运行 `code/main.py` 并比较快乐与扰动。在顶层输出完全偏离用户问题之前，需要多少级管理者交接？
2. 添加第三级（顶层 → 子 → 子子 → 工作者）。测量随着深度增长，扰动路径自我修正 vs 完全偏离的频率。
3. 在每个子管理者处实现一个"金丝雀"工作者，它总是被问到原始用户问题不变。使用金丝雀答案检测分解漂移。当金丝雀与综合答案不同意时，管理者应该如何反应？
4. 阅读 CrewAI 的 `Process.hierarchical` 文档。识别 CrewAI 应用的一个具体护栏（步骤限制、manager_llm 约束）并描述它针对的失败模式。
5. 比较嵌套 LangGraph 监督者与 CrewAI 分层。哪个使调和循环更便宜检测？

## 关键术语

| 术语 | 人们说的 | 它实际意味着什么 |
|------|----------------|------------------------|
| Hierarchical（分层） | "组织架构模式" | 监督者之上的监督者；只有叶子做工作。 |
| Manager LLM（管理者 LLM） | "老板" | 在内部节点分解、分配和验证的 LLM。 |
| Decomposition drift（分解漂移） | "老板迷失了情节" | 顶层管理者的拆分不再覆盖原始问题。 |
| Reconciliation loop（调和循环） | "无休止的会议" | 子管理者不同意；顶层重新委派；工作者重新运行；循环直到预算耗尽。 |
| Depth-2 ceiling（深度 2 上限） | "不要深入超过 2 级" | 经验护栏：3+ 级崩溃可观测性。 |
| Canary question（金丝雀问题） | "每一级的真实情况" | 一个总是被问到原始查询不变的工作者，用于检测漂移。 |
| Provenance chain（来源链） | "谁说了什么" | 从每个综合回溯到产生它的叶子输出的追踪。 |
