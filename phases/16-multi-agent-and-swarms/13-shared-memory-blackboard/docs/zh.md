# 13 · 共享内存与黑板模式

> 2026 年的多智能体系统中并存着两种方法：**消息池（message pool）**（人人都能看到所有人的消息，如 AutoGen GroupChat 或 MetaGPT）和**带订阅的黑板（blackboard with subscription）**（智能体订阅自己关心的事件，如 Context-Aware MCP 或 Matrix 框架）。两者都是多智能体系统中唯一有状态的部分——这也意味着两者都是有趣 bug 的滋生地。教科书式的失效模式是**内存投毒（memory poisoning）**：某个智能体幻觉出一个"事实"，其他智能体将其当作已核实的内容接受，准确率随之逐步衰减，这种衰减比立即崩溃要难调试得多。本课用标准库构建这两种结构，注入一次投毒攻击，并展示在生产中真正有效的三种缓解手段。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**前置：** 阶段 16 · 04（原语模型）、阶段 16 · 09（并行蜂群网络）
**时长：** 约 75 分钟

## 问题

多智能体系统需要一个让智能体共享事实的场所。一种直白的做法是"把一切都放进消息里传递"——但这等于在重新发明共享状态，还附带额外的拷贝开销。另一种是"给每个智能体一份全局日志"——但全局日志会无限制地膨胀，而且极易被投毒。第三种是"为每个智能体投影一份视图"——可扩展，但模式（schema）负担重。

当其中一个智能体产生幻觉并把幻觉写入共享状态时，下游每一个读取该状态的智能体都会把这个幻觉当作事实接受下来。等到人类注意到问题时，推理链条已经深达五步，而根因却是历史上写入的第三条消息。调试多智能体的准确率衰减，比调试一次崩溃要难得多。

这就是内存投毒。它是 MAST 分类法（Cemri 等人，arXiv:2503.13657）中记录数量第二多的失效家族，而且它是结构性的：任何没有溯源（provenance）和不可写校验者的共享内存设计，最终都会表现出这种问题。

## 概念

### 两种主要拓扑

**全量消息池（full message pool）。** 每个智能体都读取每一条消息。AutoGen GroupChat 和 MetaGPT 采用这种方式。它简单、透明、可检视，但当智能体超过约 10 个时就无法扩展，因为每个智能体的上下文都会被其他智能体的工作填满。

```
agent-A ──write──▶ ┌────────────────┐ ◀──read── agent-D
                   │ message pool   │
agent-B ──write──▶ │                │ ◀──read── agent-E
                   │ (global log)   │
agent-C ──write──▶ └────────────────┘ ◀──read── agent-F
```

**带订阅的黑板（blackboard with subscription）。** 智能体声明对某些主题（topic）感兴趣；底层基础设施只路由相关的消息。CA-MCP（arXiv:2601.11595）和去中心化的 Matrix 框架（arXiv:2511.21686）采用这种方式。它扩展性更强，但需要在前期设计好模式，才能让订阅有意义。

```
                   ┌─ topic: prices ──┐
agent-A ──pub────▶ │                  │ ──▶ agent-D (subscribed)
                   ├─ topic: orders ──┤
agent-B ──pub────▶ │                  │ ──▶ agent-E (subscribed)
                   ├─ topic: alerts ──┤
agent-C ──pub────▶ │                  │ ──▶ agent-F (subscribed)
                   └──────────────────┘
```

### 各自的适用场景

- **全量池**在智能体数量少（< 10）、异构、且对话是短时程的场景下胜出。当人人都能看到一切时，推断"谁说了什么"是轻而易举的。
- **黑板**在智能体数量多、角色同质但实例众多（蜂群）、且对话长时间运行的场景下胜出。路由能节省 token 成本并减少上下文污染。

生产系统常常混用：顶层是一个小的全量池（规划层），下面是若干黑板（工作层）。

### 一个场景中的内存投毒

三个智能体协作完成一项研究任务。智能体 A 是检索智能体，智能体 B 是摘要智能体，智能体 C 是分析智能体。

1. A 抓取了一个页面，并向共享状态写入一条消息："该研究报告了 42% 的准确率提升。"
2. 被抓取的页面实际上写的是"4.2% 的提升"。A 幻觉出了一个小数点。
3. B 读取共享状态后写道："报告了高达 42% 的准确率提升（来源：A）。"
4. C 读取共享状态后写道："建议采用——42% 的提升具有变革性意义。"
5. 最终报告引用了一个从未存在过的 42% 数字。

没有智能体崩溃，没有测试失败。系统"正常工作"。幻觉通过共享状态，从一个智能体的上下文跨越到了下游每一个智能体的推理之中。

### 为什么这是结构性问题

如果没有共享状态，智能体 A 的幻觉就停留在 A 的上下文里。下游智能体会重新抓取或重新推导，从而可能发现错误。而有了朴素的共享状态后，A 的上下文变成了所有人的上下文，幻觉被"洗白"成了事实。

问题不在于共享状态本身——而在于**没有溯源、也没有独立校验者**的共享状态。有三种缓解手段可以应对：

1. **在每次写入时附加溯源信息。** 共享状态中的每一条记录都记录由谁写入、何时写入、在什么提示词下写入、以及（如适用）智能体引用了什么来源。下游智能体在读取时带着基于溯源的怀疑态度。
2. **对写入进行版本化；把它们当作只追加（append-only）。** 一次更正是一条取代旧记录的新记录，而不是原地更新。审计轨迹得以保留。
3. **保留至少一个无法向共享状态写入的智能体。** 一个只读的校验者（verifier）智能体对记录进行抽样、重新抓取来源、并标记不一致之处。因为它无法向池中写入，所以它不会被池投毒。

### 黑板的先例（Hayes-Roth，1985）

黑板模式比 LLM 智能体早了四十年。Hayes-Roth（1985，《A Blackboard Architecture for Control》）描述了一些专家型知识源（Knowledge Source），它们观察一块全局黑板、贡献部分解、并触发其他知识源。2026 年的黑板（CA-MCP、Matrix）正是同一种模式，只不过把 LLM 智能体作为知识源、把 JSON 数据块作为部分解。旧文献已经记录了针对写入争用、机会性控制和一致性的解决方案，而现代系统正在重新发现它们。

### 投影 vs 全量视图

纯粹的黑板给每个订阅者相同的投影（按主题划分范围）。一种更激进的设计是**按智能体投影（per-agent projection）**：每个智能体获得一份为其角色定制的视图。LangGraph 的状态归约器（state reducer）是 2026 年的经典实现——归约器函数把全局状态折叠成一个角色专属的切片。

按智能体投影扩展性更强，但需要一份模式。没有模式，你就得在每个智能体的提示词里临时重建投影。

### 写入争用模式

多个智能体同时写入是一个并发问题，而不仅仅是 LLM 问题。有三种模式可行：

- **顺序写入者（单生产者）。** 所有写入都经过一个协调者智能体进行串行化。简单，但会成为瓶颈。
- **带版本化的乐观并发。** 每条记录都有一个版本号；写入者在版本不匹配时失败并重试。这是经典的数据库技术。
- **主题分区。** 不同的智能体拥有不同的主题。没有跨主题的争用。需要设计好的分区边界。

大多数 2026 年的框架默认采用顺序写入者，因为 LLM 调用足够慢，使得争用很少发生，瓶颈也不会造成伤害。

### 不可写的校验者

最关键的缓解手段是只读校验者。实现规则：

- 校验者与团队共享状态（读取黑板或池）。
- 校验者没有向共享状态写入的句柄——只有向一个独立校验通道写入的句柄。
- 校验者独立抓取写入中引用的来源，标记出分歧。
- 校验者自身的输出被路由给人类或一个独立的决策智能体，绝不回灌进池中。

没有这种隔离，校验者的输出就会成为池中的新记录，这意味着被投毒的池会投毒校验者，进而投毒它的校验结果。

## 动手构建

`code/main.py` 用标准库 Python 实现了这两种拓扑，外加一个玩具级的投毒攻击和三种缓解手段。

- `MessagePool`——线程安全的只追加日志，可全量读出。
- `Blackboard`——按主题键控的发布/订阅，带按智能体的订阅。
- `ProvenanceEntry`——每次写入都记录 (writer, timestamp, prompt_hash, source_uri)。
- `PoisoningScenario`——运行一个三智能体研究任务，其中智能体 A 幻觉出一个小数点。打印最终报告。
- `Verifier`——一个只读智能体，重新抓取来源并标记不一致之处。在校验者在场的情况下运行同一个场景。

运行：

```
python3 code/main.py
```

预期输出：
- 运行 1（无校验者）：幻觉出的 42% 传播到了最终报告。
- 运行 2（有校验者）：校验者标记了不一致，池被标注为"flagged"，最终报告包含一条撤回声明。

## 应用它

`outputs/skill-memory-auditor.md` 是一个技能（skill），它会审计任何多智能体系统的共享内存设计，检查溯源、版本化和校验者隔离。在新的多智能体架构投入生产之前，对其运行该技能。

## 交付它

对于任何共享内存设计：

- 在每次写入时记录溯源：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
- 让日志只追加。更正是引用被取代记录的新记录。
- 部署至少一个具备独立来源访问能力的只读校验者智能体。
- 把校验者输出路由到一个独立通道，而不是回灌进共享池。
- 记录"取代型写入"占全部写入的比例——这个比例上升是幻觉模式的早期证据。

## 练习

1. 运行 `code/main.py`。确认运行 1 传播了幻觉，而运行 2 捕获了它。
2. 添加第二个幻觉：智能体 B 编造一个数据集大小。校验者应当能同时捕获两者，而无需为任一项手工调优。
3. 把全量池切换为带主题分区（`prices`、`summaries`、`analyses`）的黑板。主题分区让哪些投毒场景更难得手，又对哪些场景没有帮助？
4. 阅读 Hayes-Roth（1985，《A Blackboard Architecture for Control》）。从论文中找出两个本课未讨论、但 2026 年系统能从中受益的控制模式。
5. 阅读 CA-MCP（arXiv:2601.11595）。把它的共享上下文存储（Shared Context Store）映射到 `code/main.py` 中的 `MessagePool` 或 `Blackboard` 类。CA-MCP 在其之上额外增加了哪些原语？

## 关键术语

| 术语 | 人们怎么说 | 它实际指什么 |
|------|----------------|------------------------|
| 消息池（Message pool） | "共享聊天历史" | 每个智能体都读取的只追加日志。完全透明，扩展性差。 |
| 黑板（Blackboard） | "共享工作区" | 按主题键控的发布/订阅。智能体订阅相关主题。扩展性更强。 |
| 溯源（Provenance） | "谁写了什么" | 每次写入上的元数据：写入者、时间戳、提示词、来源。 |
| 内存投毒（Memory poisoning） | "幻觉在扩散" | 一个智能体的错误进入共享状态，下游智能体将其当作事实接受。 |
| 只追加（Append-only） | "不做原地更新" | 更正是取代旧记录的新记录。保留审计轨迹。 |
| 不可写校验者（Unwritable verifier） | "独立审计者" | 只读智能体，重新抓取来源并标记不一致之处。 |
| 投影（Projection） | "范围受限的视图" | 由全局状态计算出的按智能体视图。LangGraph 归约器是经典案例。 |
| 知识源（Knowledge Source） | "专家型智能体" | Hayes-Roth 1985 年对黑板参与者的称呼。 |
