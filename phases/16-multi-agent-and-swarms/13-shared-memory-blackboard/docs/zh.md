# 共享内存与黑板模式#

> 两种方在 2026 年多智能体系统中共存：**消息池**（每个人看到每个人的消息，如 AutoGen GroupChat 或 MetaGPT）和**带订阅的黑板**（智能体订阅相关事件，如 Context-Aware MCP 或 Matrix 框架）。两者都是多智能体系统中唯一有状态的部分 —— 这意味着两者都是有趣错误存在的地方。参考失败模式是**内存中毒**：一个智能体产生幻觉的"事实"，其他智能体将其视为已验证，并且准确性逐渐衰减，这比立即崩溃更难调试。本课用标准库构建这两种结构，注入中毒攻击，并展示在生产中实际有效的三种缓解措施。

**类型：** 学习 + 构建
**语言：** Python (标准库, `threading`)
**前置条件：** 第 16 阶段 · 04 (基元模型), 第 16 阶段 · 09 (并行集群网络)
**时间：** ~75 分钟

## 问题#

多智能体系统需要一个地方让智能体共享事实。一个天真的选项是"在消息中传递所有内容" —— 但这用额外的复制重新发明了共享状态。另一个选项是"给每个人一个全局日志" —— 但全局日志增长无界并且容易中毒。第三个选项是"每个智能体投影一个视图" —— 可扩展但模式很重。

当其中一个智能体产生幻觉并将幻觉写入共享状态时，每个读取该状态的下游智能体都将幻觉采纳为事实。当人类注意到时，推理链已经深达五步，而根本原因是曾经写入的第三条消息。调试多智能体准确性衰减比调试崩溃更难。

这就是内存中毒。它是 MAST 分类法（Cemri et al., arXiv:2503.13657）中记录第二多的失败家族，并且它是结构性的：任何没有出处和不可写验证器的共享内存设计最终都会表现出它。

## 概念#

### 两种主要拓扑#

**全消息池。** 每个智能体读取每条消息。AutoGen GroupChat 和 MetaGPT 使用这个。简单、透明、可检查，但扩展到约 10 个智能体以外因为每个智能体的上下文充满了其他智能体的工作。

```
agent-A ──写入──▶ ┌────────────────┐ ◀──读取── agent-D
                   │ 消息池   │
agent-B ──写入──▶ │                │ ◀──读取── agent-E
                   │ (全局日志)   │
agent-C ──写入──▶ └────────────────┘ ◀──读取── agent-F
```

**带订阅的黑板。** 智能体声明对主题的兴趣；底层路由仅相关消息。CA-MCP (arXiv:2601.11595) 和 Matrix 去中心化框架 (arXiv:2511.21686) 使用这个。进一步扩展，但需要预先模式设计以使订阅有意义。

```
                   ┌─ 主题：价格 ──┐
agent-A ──发布────▶ │                  │ ──▶ agent-D (已订阅)
                   ├─ 主题：订单 ──┤
agent-B ──发布────▶ │                  │ ──▶ agent-E (已订阅)
                   ├─ 主题：警报 ──┤
agent-C ──发布────▶ │                  │ ──▶ agent-F (已订阅)
                   └──────────────────┘
```

### 每种获胜的场景#

- **全池** 在智能体很少 (< 10)、异构并且对话是短 Horizon 时获胜。当每个人看到所有内容时，推理谁说了什么很简单。
- **黑板** 在智能体很多、角色同质但实例众多（集群）、并且对话是长时间运行时获胜。路由节省令牌成本和上下文污染。

生产系统通常混合：顶部的小全池（规划层），下面的黑板（工作者层）。

### 内存中毒，在一个场景中#

三个智能体处理研究任务。智能体 A 是检索智能体。智能体 B 是总结器。智能体 C 是分析员。

1. A 获取一个页面并写入一条消息到共享状态："研究报告了 42% 的准确性改进。"
2. 获取的页面实际上说的是"4.2% 改进。" A 产生幻觉了一个小数。
3. B 读取共享状态，写入："报告了大型 42% 准确性收益（来源：A）。"
4. C 读取共享状态，写入："建议采用 —— 42% 提升是变革性的。"
5. 最终报告引用了一个从未存在过的 42% 数字。

没有智能体崩溃。没有测试失败。系统"工作了"。幻觉通过共享状态从一个智能体的上下文进入了每个下游智能体的推理。

### 为什么这是结构性的#

没有共享状态，智能体 A 的幻觉停留在 A 的上下文中。下游智能体将重新获取或重新推导，并可能捕获错误。使用天真的共享状态，A 的上下文成为每个人的上下文，并且幻觉被洗白为事实。

问题不是共享状态本身 —— 而是共享状态**没有出处并且没有独立的验证器**。三种缓解措施解决这个问题：

1. **在每次写入时属性出处。** 共享状态中的每个条目记录谁写入了它、何时、在什么提示下，以及（如果适用）智能体引用了什么来源。下游智能体以基于出处的怀疑态度读取。
2. **版本化写入；将它们视为仅追加。** 纠正是优于旧条目的新条目，而不是就地更新。审计跟踪被保留。
3. **保持至少一个不能写入共享状态的智能体。** 只读验证器智能体采样条目、重新获取来源并标记不一致。因为它不能写入池，它不能被池中毒。

### 黑板先例 (Hayes-Roth, 1985)#

黑板模式比 LLM 智能体早四十年。Hayes-Roth (1985, "A Blackboard Architecture for Control") 描述了观察全局黑板、贡献部分解决方案并触发其他来源的专家知识来源。2026 年的黑板 (CA-MCP, Matrix) 是相同的模式，LLM 智能体作为知识来源，JSON 斑点作为部分解决方案。旧文献已经记录了写入争用、机会控制和一致性的解决方案，现代系统正在重新发现它们。

### 投影 vs 全视图#

纯黑板给每个订阅者相同的投影（主题范围的）。一个更激进的设计是**每个智能体投影**：每个智能体获得为其角色定制的视图。LangGraph 的状态归约器是规范的 2026 年实现 —— 归约器函数将全局状态折叠为特定角色的切片。

每个智能体投影进一步扩展，但需要模式。没有一个，你在每个智能体的提示中重建临时投影。

### 写入争用模式#

多个智能体同时写入是一个并发问题，不仅仅是 LLM 问题。三种模式有效：

- **顺序写入器（单生产者）。** 所有写入都通过一个序列化它们的协调器智能体。简单，但是瓶颈。
- **带版本化的乐观并发。** 每个条目有一个版本；写入器在版本不匹配时失败并重试。经典的数据库技术。
- **主题分区。** 不同的智能体拥有不同的主题。没有跨主题争用。需要设计的分区边界。

大多数 2026 框架默认为顺序写入器，因为 LLM 调用足够慢，争用很少，并且瓶颈不会造成伤害。

### 不可写验证器#

最重要的负载承载缓解措施是不可写验证器。实现规则：

- 验证器与团队共享状态（读取黑板或池）。
- 验证器对共享状态没有写入句柄 —— 只有单独的验证通道。
- 验证器独立获取写入中引用的来源。标记分歧。
- 验证器的自己的输出被路由到人类或单独的决定智能体，永远不会反馈到池中。

没有这种分离，验证器的输出成为池中的新条目，这意味着中毒的池会毒害验证器，而验证器会毒害其验证。

## 构建它#

`code/main.py` 在标准库 Python 中实现两种拓扑以及一个玩具中毒攻击和三种缓解措施。

- `MessagePool` —— 线程安全的仅追加日志，具有完全读出。
- `Blackboard` —— 主题键控的发布/订阅，具有每个智能体的订阅。
- `ProvenanceEntry` —— 每次写入记录（写入器、时间戳、prompt_hash、source_uri）。
- `PoisoningScenario` —— 运行一个三个智能体的研究任务，其中智能体 A 产生幻觉一个小数。打印最终报告。
- `Verifier` —— 一个只读智能体，重新获取来源并标记不一致。使用存在的验证器运行相同场景。

运行：

```
python3 code/main.py
```

预期输出：
- 运行 1（无验证器）：幻觉的 42% 传播到最终报告。
- 运行 2（带验证器）：验证器标记不一致，池被标记为"已标记"，最终报告包括一个撤回。

## 使用它#

`outputs/skill-memory-auditor.md` 是一个技能，用于审计任何多智能体系统的共享内存设计是否包含出处、版本控制和验证器分离。在生产之前对新的多智能体架构运行它。

## 发布它#

对于任何共享内存设计：

- 在每次写入时记录出处：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
- 使日志仅追加。纠正是引用被取代条目的新条目。
- 部署至少一个具有独立来源访问权限的只读验证器智能体。
- 将验证器输出路由到单独的通道，而不是回到共享池。
- 记录作为超越的写入比率 —— 上升的比率是幻觉模式的早期证据。

## 练习#

1. 运行 `code/main.py`。确认运行 1 传播幻觉，运行 2 捕获它。
2. 添加第二个幻觉：智能体 B 发明一个数据集大小。验证器应该捕获两者，而无需为任一进行手调。
3. 将全池切换到具有主题分区的黑板（`prices`, `summaries`, `analyses`）。主题分区使哪些中毒场景更难执行，哪些它没有帮助？
4. 阅读 Hayes-Roth (1985, "A Blackboard Architecture for Control")。确定本文中未在本课中讨论的两种控制模式，2026 年的系统将从中受益。
5. 阅读 CA-MCP (arXiv:2601.11595)。将其共享上下文存储映射到 `code/main.py` 中的 MessagePool 或 Blackboard 类。CA-MCP 在顶部添加了哪些基元？

## 关键术语#

| 术语 | 人们的说法 | 实际含义 |
|------|----------------|------------------------|
| 消息池 | "共享聊天历史" | 每个智能体读取的仅追加日志。完全透明，扩展性差。 |
| 黑板 | "共享工作空间" | 主题键控的发布/订阅。智能体订阅相关主题。扩展得更远。 |
| 出处 | "谁写了什么" | 每次写入上的元数据：写入器、时间戳、提示、来源。 |
| 内存中毒 | "幻觉传播" | 一个智能体的错误进入共享状态，下游智能体将其采纳为事实。 |
| 仅追加 | "无就地更新" | 纠正是超越的旧条目。保留审计跟踪。 |
| 不可写验证器 | "独立审计器" | 只读智能体，重新获取来源并标记不一致。 |
| 投影 | "范围化视图" | 从全局状态计算的每个智能体视图。LangGraph 归约器是规范案例。 |
| 知识来源 | "专家智能体" | Hayes-Roth 1985 年对黑板参与者的术语。 |
