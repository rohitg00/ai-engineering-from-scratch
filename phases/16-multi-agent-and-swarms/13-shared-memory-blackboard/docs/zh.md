# 共享内存与黑板模式

> 2026年多智能体系统中两种方法共存：**消息池**（每个人都看到每个人的消息，如AutoGen GroupChat或MetaGPT）和**带订阅的黑板**（智能体订阅相关事件，如Context-Aware MCP或Matrix框架）。两者都是多智能体系统中唯一有状态的部分——这意味着两者都是有趣bug存在的地方。参考失败模式是**内存中毒**：一个智能体幻觉一个"事实"，其他智能体将其视为已验证，准确性逐渐衰减，这比立即崩溃更难调试。本课程从标准库构建两种结构，注入中毒攻击，并展示在生产中实际有效的三种缓解措施。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**前置知识：** 第16阶段 · 04（原语模型），第16阶段 · 09（并行群体网络）
**时间：** 约75分钟

## 问题

多智能体系统需要一个地方让智能体共享事实。一个字面选项是"将所有东西通过消息传递"——但这用额外复制重新发明了共享状态。另一个是"给每个人一个全局日志"——但全局日志无界增长且容易中毒。第三个是"为每个智能体投影一个视图"——可扩展但模式繁重。

当其中一个智能体产生幻觉并将幻觉写入共享状态时，每个读取该状态的下游智能体都将幻觉采纳为事实。当人类注意到时，推理链已经深达五步，根本原因是第三条消息。调试多智能体准确性衰减比调试崩溃更难。

这就是内存中毒。它是MAST分类法（Cemri等人，arXiv:2503.13657）中第二多记录的失败家族，它是结构性的：任何没有来源和不可写验证者的共享内存设计最终都会表现出它。

## 核心概念

### 两种主要拓扑

**完整消息池。** 每个智能体读取每条消息。AutoGen GroupChat和MetaGPT使用这个。简单、透明、可检查，但在约10个智能体之后无法扩展，因为每个智能体的上下文被其他智能体的工作填满。

```
智能体-A ──写入──▶ ┌────────────────┐ ◀──读取── 智能体-D
                   │ 消息池         │
智能体-B ──写入──▶ │                │ ◀──读取── 智能体-E
                   │ (全局日志)     │
智能体-C ──写入──▶ └────────────────┘ ◀──读取── 智能体-F
```

**带订阅的黑板。** 智能体声明对主题的兴趣；基板只路由相关消息。CA-MCP（arXiv:2601.11595）和Matrix去中心化框架（arXiv:2511.21686）使用这个。扩展更远，但需要预先的模式设计来使订阅有意义。

```
                   ┌─ 主题: prices ──┐
智能体-A ──发布────▶ │                  │ ──▶ 智能体-D (已订阅)
                   ├─ 主题: orders ──┤
智能体-B ──发布────▶ │                  │ ──▶ 智能体-E (已订阅)
                   ├─ 主题: alerts ──┤
智能体-C ──发布────▶ │                  │ ──▶ 智能体-F (已订阅)
                   └──────────────────┘
```

### 每种何时获胜

- **完整池**在智能体少（< 10）、异构且对话短期时获胜。当每个人看到一切时，推理谁说了什么是微不足道的。
- **黑板**在智能体多、角色同质但实例众多（群体）且对话长期运行时获胜。路由节省token成本和上下文污染。

生产系统经常混合：顶部小完整池（规划层），下方黑板（工作者层）。

### 内存中毒，一个场景

三个智能体在研究任务上工作。智能体A是检索智能体。智能体B是摘要器。智能体C是分析师。

1. A获取一个页面并写入共享状态："研究报告42%的准确性改进。"
2. 获取的页面实际说"4.2%的改进。"A幻觉了一个小数点。
3. B读取共享状态，写入："报告了大的42%准确性增益（来源：A）。"
4. C读取共享状态，写入："建议采用——42%的提升是变革性的。"
5. 最终报告引用了一个从未存在的42%数字。

没有智能体崩溃。没有测试失败。系统"工作"了。幻觉从一个智能体的上下文通过共享状态进入每个下游智能体的推理。

### 为什么这是结构性的

没有共享状态，智能体A的幻觉留在A的上下文中。下游智能体会重新获取或重新推导，可能捕获错误。有朴素共享状态，A的上下文变成每个人的上下文，幻觉被洗白为事实。

问题不是共享状态本身——它是**没有来源和没有独立验证者**的共享状态。三种缓解措施解决这个问题：

1. **每次写入属性来源。** 共享状态中的每个条目记录谁写的、何时、在什么提示下，以及（如果适用）智能体引用的什么来源。下游智能体带着基于来源的怀疑读取。
2. **版本写入；将它们视为仅追加。** 修正是取代旧条目的新条目，不是原地更新。审计追踪被保留。
3. **保持至少一个不能写入共享状态的智能体。** 只读验证者智能体采样条目、重新获取来源并标记不一致。因为它不能写入池，它不能被池中毒。

### 黑板先例（Hayes-Roth，1985）

黑板模式比LLM智能体早四十年。Hayes-Roth（1985，"控制的黑板架构"）描述了观察全局黑板、贡献部分解决方案并触发其他来源的专家知识源。2026年的黑板（CA-MCP、Matrix）是相同的模式，LLM智能体作为知识源，JSON blob作为部分解决方案。旧文献有文档记录的解决方案，用于写争用、机会控制和一致性，现代系统重新发现这些。

### 投影 vs 完整视图

纯黑板给每个订阅者相同的投影（主题范围的）。更激进的设计是**每智能体投影**：每个智能体获得为其角色定制的视图。LangGraph的状态归约器是2026年的规范实现——归约器函数将全局状态折叠成角色特定的切片。

每智能体投影扩展更远但需要一个模式。没有模式，你在每个智能体的提示中重建临时投影。

### 写争用模式

多个智能体同时写入是一个并发问题，不只是LLM问题。三种模式有效：

- **顺序写入者（单生产者）。** 所有写入通过一个协调智能体进行，该智能体序列化。简单，但是瓶颈。
- **乐观并发与版本控制。** 每个条目有一个版本；写入者在版本不匹配时失败并重试。经典数据库技术。
- **主题分区。** 不同智能体拥有不同主题。没有跨主题争用。需要设计的分区边界。

大多数2026框架默认使用顺序写入者，因为LLM调用足够慢，争用很少且瓶颈不影响。

### 不可写验证者

最承重的缓解措施是只读验证者。实现规则：

- 验证者与团队共享状态（读取黑板或池）。
- 验证者没有写入共享状态的句柄——只有单独的验证通道。
- 验证者独立获取写入中引用的来源。标记分歧。
- 验证者自己的输出被路由到人类或单独的决策智能体，从不反馈回池。

没有这种分离，验证者的输出变成池中的新条目，这意味着中毒的池中毒验证者，中毒其验证。

## 构建它

`code/main.py`在标准库Python中实现两种拓扑加上玩具中毒攻击和三种缓解措施。

- `MessagePool` —— 线程安全的仅追加日志，完整读出。
- `Blackboard` —— 主题键控发布/订阅，带每智能体订阅。
- `ProvenanceEntry` —— 每次写入记录（写入者、时间戳、提示哈希、来源URI）。
- `PoisoningScenario` —— 运行三个智能体的研究任务，其中智能体A幻觉一个小数点。打印最终报告。
- `Verifier` —— 重新获取来源并标记不一致的只读智能体。在有验证者的情况下运行相同场景。

运行：

```
python3 code/main.py
```

预期输出：
- 运行1（无验证者）：幻觉的42%传播到最终报告。
- 运行2（有验证者）：验证者标记不一致，池被标记为"已标记"，最终报告包含撤回。

## 使用它

`outputs/skill-memory-auditor.md`是一个技能，审计任何多智能体系统的共享内存设计，检查来源、版本控制和验证者分离。在生产之前在新多智能体架构上运行它。

## 交付它

对于任何共享内存设计：

- 每次写入记录来源：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
- 使日志仅追加。修正是引用被取代条目的新条目。
- 部署至少一个具有独立来源访问的只读验证者智能体。
- 将验证者输出路由到单独的通道，不是回共享池。
- 记录写入中是取代的比例——上升的比例是幻觉模式的早期证据。

## 练习

1. 运行`code/main.py`。确认运行1传播幻觉，运行2抓住它。
2. 添加第二个幻觉：智能体B发明一个数据集大小。验证者应该在不针对两者手工调整的情况下抓住两者。
3. 将完整池切换为带主题分区（`prices`、`summaries`、`analyses`）的黑板。主题分区使哪些中毒场景更难实施，哪些它没有帮助？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 消息池 | "共享聊天历史" | 每个智能体读取的仅追加日志。完全透明，扩展性差。 |
| 黑板 | "共享工作空间" | 主题键控发布/订阅。智能体订阅相关主题。扩展更远。 |
| 来源 | "谁写了什么" | 每次写入的元数据：写入者、时间戳、提示、来源。 |
| 内存中毒 | "幻觉传播" | 一个智能体的错误进入共享状态，下游智能体将其采纳为事实。 |
| 仅追加 | "无原地更新" | 修正是取代的新条目。保留审计追踪。 |
| 不可写验证者 | "独立审计员" | 重新获取来源并标记不一致的只读智能体。 |
| 投影 | "范围视图" | 从全局状态计算的每智能体视图。LangGraph归约器是规范案例。 |
| 知识源 | "专家智能体" | Hayes-Roth 1985年黑板参与者的术语。 |
