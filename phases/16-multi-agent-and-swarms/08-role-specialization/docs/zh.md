# 角色专业化——规划者、批判者、执行者、验证者

> 2026年最常见的多智能体分解：一个智能体规划，一个执行，一个批判或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码到角色提示中的SOP——产品经理、架构师、项目经理、工程师、QA工程师——遵循`Code = SOP(Team)`。ChatDev（arXiv:2307.07924）通过"聊天链"将设计师、程序员、审查者、测试员串联起来，具有"交流性去幻觉化"（智能体明确请求缺失的细节）。验证者是承重的：Cemri等人（MAST，arXiv:2503.13657）表明每个多智能体失败都可以追溯到缺失或损坏的验证。PwC报告CrewAI中结构化验证循环带来7倍准确性提升（10% → 70%）。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置知识：** 第16阶段 · 04（原语模型），第16阶段 · 05（监督者）
**时间：** 约60分钟

## 问题

通用多智能体系统产生通用输出。群聊中的三个编码器编写三种相同平庸代码的变体。你可以添加更多智能体，添加更多轮次，仍然无法跨越质量门槛。

修复不是更多智能体——而是*不同*的智能体。分配不同角色。给批判者规划者没有的工具。给验证者一个客观的测试套件。现在系统有内部分歧与基于事实的纠正，不只是并行猜测。

## 核心概念

### 四个经典角色

**规划者。** 读取目标，产生步骤列表或规范。工具：知识检索、文档。输出：结构化计划。

**执行者。** 一次读取一个计划步骤，产生工件。工具：实际工作工具（代码编译器、shell、API客户端）。输出：工件。

**批判者。** 根据规划者的意图读取执行者的输出。工具：对工件的只读访问、静态分析。输出：接受/拒绝及原因。

**验证者。** 读取工件并运行确定性检查。工具：测试运行器、类型检查器、模式验证器。输出：通过/失败及证据。

批判者是主观的、有意见的，通常是基于LLM的。验证者是客观的、确定性的，通常是基于代码的。它们不是同一个角色。

### MetaGPT的SOP模式

MetaGPT（arXiv:2308.00352）将软件工程SOP编码为角色提示：

- **产品经理**编写PRD。
- **架构师**产生系统设计。
- **项目经理**拆分任务。
- **工程师**实现。
- **QA工程师**运行测试。

每个角色有严格的输入/输出模式。角色提示说明角色*是什么*以及它*必须产生什么*。`Code = SOP(Team)`公式——确定性SOP将LLM团队变成可预测的流水线。

### ChatDev的交流性去幻觉化

ChatDev添加了一个关键举措：当执行者需要计划中未包含的特定细节时，它在继续之前明确询问设计师。这防止了LLM的经典失败：合理地发明细节。

实现：角色提示包括"当你需要未被给予的特定信息时，在产生输出之前按名称询问相关角色。"

### 为什么验证者最重要

Cemri等人（MAST）追踪了1642个多智能体执行失败。21.3%是验证差距——系统交付了一个无人检查的答案。剩余79%通常追溯到"有一个检查静默失败或从未运行。"验证者是承重的角色。

PwC报告（CrewAI部署，2025）添加结构化验证循环将准确性从10%提升到70%。一个角色带来7倍增益。

### 批判者 vs 验证者

- 批判者是审查工件质量的LLM。主观的。可能被合理的散文愚弄。
- 验证者是在工件上运行的确定性程序。客观的。给出通过/失败及证据。

两者都用。批判者抓住验证者无法表达的味道问题。验证者抓住批判者看不到的bug，因为它们只在运行时出现。

### 反模式

系统中的每个角色都是LLM，每个角色的输出都是"看起来不错。"经典的MAST失败模式。添加至少一个验证者，其通过/失败由代码决定，不是由LLM决定。

### 框架映射

- **CrewAI** — `Agent(role, goal, backstory)`是教科书专业化表面。
- **LangGraph** — 节点可以有专门的提示；边强制执行流水线。
- **AutoGen** — GroupChat中具有单字名称的角色特定ConversableAgents。
- **OpenAI Agents SDK** — 角色专业化智能体之间的交接工具。

## 构建它

`code/main.py`实现了一个构建简单Python函数的4角色流水线：

- **规划者**产生规范。
- **执行者**生成代码字符串。
- **批判者**（LLM模拟的）标记明显问题。
- **验证者**在沙箱（`exec`）中运行生成的代码，针对测试用例。

演示运行两次：一次执行者产生正确代码（批判者 + 验证者都通过），一次执行者产生离规范代码（批判者因为看起来合理而错过bug，验证者因为测试失败而抓住它）。

运行：

```
python3 code/main.py
```

## 使用它

`outputs/skill-role-designer.md`接受任务并产生角色名册（3-5个角色）、每个角色的输入/输出模式，以及验证者检查。在将智能体接入框架之前使用此功能。

## 交付它

检查清单：

- **至少一个确定性验证者。** 永远不要全LLM。
- **每个角色的显式I/O模式。** 规划者返回规范，不是散文；执行者读取该模式。
- **交流性去幻觉化。** 执行者在信息缺失时必须询问规划者；永远不要发明它。
- **批判者/验证者排序。** 先运行批判者（便宜，抓住设计问题），再运行验证者（慢，抓住bug）。
- **循环预算。** 在升级到人类之前，最多2轮批判者-执行者修订轮次。

## 练习

1. 运行`code/main.py`并观察验证者如何抓住批判者错过的bug。添加静态分析检查（计算`return`出现次数）作为额外验证者。它抓住什么运行时测试遗漏的？
2. 添加第5个角色："需求分析师"，将用户愿望翻译成规划者就绪的规范。什么交流性去幻觉化请求应该流向它？
3. 阅读MetaGPT第3节（"智能体"）。列出MetaGPT的5个角色中每个的输入/输出模式。
4. 阅读ChatDev的聊天链图（arXiv:2307.07924图3）。识别交流性去幻觉化在哪里打破一个否则会无限的循环。
5. PwC的7倍准确性增益来自验证循环。假设三个添加验证者不会帮助的任务——确定性正确性检查不可能或过于昂贵的任务。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 角色专业化 | "不同智能体，不同工作" | 为规划者/执行者/批判者/验证者角色调整的不同系统提示。 |
| SOP模式 | "编码的标准操作程序" | MetaGPT的框架：每个角色的严格I/O模式将团队变成流水线。 |
| 交流性去幻觉化 | "发明前询问" | ChatDev模式：执行者在细节缺失时询问规划者，而不是编造一个。 |
| 批判者 | "LLM审查者" | 主观的、有意见的审查者。抓住味道问题。可能被合理的散文愚弄。 |
| 验证者 | "确定性检查" | 基于代码的通过/失败。测试运行器、类型检查器、模式验证器。无法被愚弄。 |
| 验证差距 | "无人检查" | 21.3%的MAST失败。答案交付了，没有会抓住bug的检查。 |
| 修订循环 | "批判者发回" | 批判者拒绝触发执行者用反馈重新运行。需要预算。 |
| 全LLM反模式 | "看起来不错" | 每个角色都是LLM，没有确定性检查。经典的MAST失败。 |
