# 角色专业化 —— 规划器、批评者、执行器、验证器

> 2026 年最常见的多智能体分解：一个智能体规划，一个执行，一个批评或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码到角色提示词中的标准操作程序（SOP）—— 产品经理、架构师、项目经理、工程师、QA 工程师 —— 遵循 `Code = SOP(Team)`。ChatDev（arXiv:2307.07924）通过带有"通信去幻觉"（智能体显式请求缺失细节）的"聊天链"连接设计师、程序员、审查员、测试员。验证器是承重结构：Cemri 等人（MAST，arXiv:2503.13657）显示每个多智能体失败都可以追溯到缺失或损坏的验证。PwC 报告称，CrewAI 中结构化验证循环带来了 7 倍准确率提升（10% → 70%）。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置条件：** 阶段 16.04（原语模型）、阶段 16.05（监督者）
**时长：** 约 60 分钟

## 问题背景

通用多智能体系统产生通用输出。群聊中的三个编码员写出三种风格的同样平庸的代码。你可以添加更多智能体、添加更多轮次，仍然无法跨越质量阈值。

解决方法不是更多智能体——而是*不同的*智能体。分配不同的角色。给批评者规划器没有的工具。给验证器一个客观测试套件。现在系统有了基于事实修正的内部分歧，而不仅仅是并行猜测。

## 概念讲解

### 四个规范角色

**规划器（Planner）。** 读取目标，产生步骤列表或规范。工具：知识检索、文档。输出：结构化计划。

**执行器（Executor）。** 一次读取一个计划步骤，产生产物。工具：实际工作工具（代码编译器、shell、API 客户端）。输出：产物。

**批评者（Critic）。** 对照规划器的意图读取执行器的输出。工具：对产物的只读访问、静态分析。输出：带理由的接受/拒绝。

**验证器（Verifier）。** 读取产物并运行确定性检查。工具：测试运行器、类型检查器、模式验证器。输出：带证据的通过/失败。

批评者是主观的、有意见的，通常基于 LLM。验证器是客观的、确定性的，通常基于代码。它们不是同一个角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）将软件工程 SOP 编码为角色提示词：

- **产品经理**撰写 PRD（产品需求文档）。
- **架构师**产生系统设计。
- **项目经理**拆分任务。
- **工程师**实施。
- **QA 工程师**运行测试。

每个角色都有严格的输入/输出模式。角色提示词说明角色*是*什么以及它*必须产生*什么。`Code = SOP(Team)` 公式——确定性 SOP 将 LLM 团队转变为可预测的流水线。

### ChatDev 的通信去幻觉

ChatDev 增加了一个关键动作：当执行器需要计划中未包含的 specific 细节时，它在继续之前显式询问设计师。这防止了经典 LLM 失败——看似合理地编造细节。

实现：角色提示词包括"当你需要未给出的 specific 信息时，在产生输出之前按名称询问相关角色。"

### 为什么验证器最重要

Cemri 等人（MAST）追踪了 1642 个多智能体执行失败。21.3% 是验证缺口——系统发布了一个没有人检查过的答案。其余 79% 通常追溯到"有一个检查静默失败或从未运行。" 验证器是承重角色。

PwC 报告称（CrewAI 部署，2025），添加结构化验证循环将准确率从 10% 提升到 70%。一个角色带来 7 倍增益。

### 批评者 vs 验证器

- 批评者是审查产物质量的 LLM。主观的。可能被看似合理的散文欺骗。
- 验证器是在产物上运行的确定性程序。客观的。给出带证据的通通过/失败。

两者都用。批评者捕获验证器无法表达的品味问题。验证器捕获批评者无法看到的错误，因为它们只在运行时出现。

### 反模式

你系统中的每个角色都是 LLM，每个角色的输出都是"在我看来不错"。经典的 MAST 失败模式。至少添加一个通过代码而不是 LLM 决定通过/失败的验证器。

### 框架映射

- **CrewAI** — `Agent(role, goal, backstory)` 是教科书式的专业化表面。
- **LangGraph** — 节点可以有专门的提示词；边强制执行流水线。
- **AutoGen** — 群聊中具有单字名称的特定于角色的可交谈智能体。
- **OpenAI Agents SDK** — 角色专门化智能体之间的交接工具。

## 构建实现

`code/main.py` 实现了一个构建简单 Python 函数的 4 角色流水线：

- **规划器**产生规范。
- **执行器**生成代码字符串。
- **批评者**（LLM 模拟）标记明显问题。
- **验证器**在沙箱（`exec`）中针对测试用例运行生成的代码。

演示运行两次：一次执行器产生正确代码（批评者 + 验证器都通过），一次执行器产生不符合规范的代码（批评者因为看起来合理而错过错误，验证器因为测试失败而捕获它）。

运行：

```
python3 code/main.py
```

## 实际应用

`outputs/skill-role-designer.md` 接受一个任务并产生角色名单（3-5 个角色）、每个角色的输入/输出模式以及验证器检查。在将智能体连接到框架之前使用此功能。

## 部署实现

检查清单：

- **至少一个确定性验证器。** 绝不全是 LLM。
- **每个角色的显式 I/O 模式。** 规划器返回规范，不是散文；执行器读取该模式。
- **通信去幻觉。** 执行器必须在信息缺失时询问规划器；绝不编造。
- **批评者/验证器排序。** 首先运行批评者（便宜，捕获设计问题），其次运行验证器（慢，捕获错误）。
- **循环预算。** 在升级到人类之前最多 2 次批评者-执行器修订轮次。

## 练习

1. 运行 `code/main.py` 并观察验证器如何捕获批评者错过的错误。添加静态分析检查（计算 `return` 的出现次数）作为附加验证器。它捕获了运行时测试错过的什么？
2. 添加第 5 个角色："需求分析师"，将用户愿望转换为规划器就绪的规范。什么通信去幻觉请求应该向上流向它？
3. 阅读 MetaGPT 第 3 节（"智能体"）。列出 MetaGPT 5 个角色中每个的输入/输出模式。
4. 阅读 ChatDev 的聊天链图（arXiv:2307.07924 图 3）。确定通信去幻觉在哪里打破了否则会是无限的循环。
5. PwC 的 7 倍准确率增益来自验证循环。假设三个添加验证器没有帮助的任务——确定性正确性检查不可能或成本高昂得令人望而却步。

## 关键术语

| 术语 | 人们说的 | 它实际意味着什么 |
|------|----------------|------------------------|
| Role specialization（角色专业化） | "不同的智能体，不同的工作" | 针对规划器/执行器/批评者/验证器角色调整的不同系统提示词。 |
| SOP pattern（SOP 模式） | "编码的标准操作程序" | MetaGPT 的框架：每个角色的严格 I/O 模式将团队转变为流水线。 |
| Communicative dehallucination（通信去幻觉） | "在编造之前询问" | ChatDev 模式：执行器在细节缺失时询问规划器，而不是编造一个。 |
| Critic（批评者） | "LLM 审查员" | 主观、有意见的审查员。捕获品味问题。可能被看似合理的散文欺骗。 |
| Verifier（验证器） | "确定性检查" | 基于代码的通过/失败。测试运行器、类型检查器、模式验证器。无法被欺骗。 |
| Verification gap（验证缺口） | "没有人检查" | 21.3% 的 MAST 失败。答案在会捕获错误的检查之外发布了。 |
| Revision loop（修订循环） | "批评者将其发回" | 批评者拒绝触发带有反馈的执行器重新运行。需要预算。 |
| All-LLM anti-pattern（全 LLM 反模式） | "在我看来不错" | 每个角色都是 LLM，没有确定性检查。经典的 MAST 失败。 |
