# 08 · 角色专业化——规划者、批评者、执行者、验证者

> 这是 2026 年最常见的多智能体分解方式：一个智能体负责规划，一个负责执行，一个负责批评或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码进角色提示词的标准作业程序「SOP（Standard Operating Procedure）」——产品经理、架构师、项目经理、工程师、QA 工程师——遵循 `Code = SOP(Team)`。ChatDev（arXiv:2307.07924）通过一条「聊天链（chat chain）」把设计师、程序员、审查者、测试员串联起来，并引入「沟通式去幻觉（communicative dehallucination）」（智能体会显式请求缺失的细节）。验证者是承重角色：Cemri 等人（MAST，arXiv:2503.13657）证明，每一次多智能体失败都能追溯到缺失或失效的验证。普华永道（PwC）报告称，在 CrewAI 中引入结构化校验循环带来了 7 倍的准确率提升（10% → 70%）。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置：** 阶段 16 · 04（基础模型原语）、阶段 16 · 05（监督者）
**时长：** 约 60 分钟

## 问题

通用的多智能体系统只会产出通用的结果。群聊里的三个程序员，写出的是同一份平庸代码的三种口味。你可以增加更多智能体、增加更多轮次，却依然跨不过质量门槛。

解决之道不是增加智能体，而是引入*不同*的智能体。分配截然不同的角色。给批评者配备规划者没有的工具。给验证者一套客观的测试集。如此一来，系统内部便有了带有据可依的纠错机制的分歧，而不只是并行的瞎猜。

## 概念

### 四种典型角色

**规划者（Planner）。** 读取目标，产出步骤清单或一份规格说明。工具：知识检索、文档。输出：结构化的计划。

**执行者（Executor）。** 一次读取计划中的一步，产出产物。工具：实际干活的工具（代码编译器、shell、API 客户端）。输出：产物。

**批评者（Critic）。** 对照规划者的意图来审读执行者的输出。工具：对产物的只读访问、静态分析。输出：接受/拒绝并附理由。

**验证者（Verifier）。** 读取产物并运行确定性检查。工具：测试运行器、类型检查器、模式校验器。输出：通过/失败并附证据。

批评者是主观的、带个人观点的，通常基于 LLM。验证者是客观的、确定性的，通常基于代码。两者不是同一个角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）将软件工程的标准作业程序编码为角色提示词：

- **产品经理（Product Manager）** 撰写 PRD。
- **架构师（Architect）** 产出系统设计。
- **项目经理（Project Manager）** 拆分任务。
- **工程师（Engineer）** 实现。
- **QA 工程师（QA Engineer）** 运行测试。

每个角色都有严格的输入/输出模式。角色提示词规定了该角色*是什么*以及它*必须产出什么*。`Code = SOP(Team)` 这一表述意味着：确定性的 SOP 把一支 LLM 团队变成一条可预测的流水线。

### ChatDev 的沟通式去幻觉

ChatDev 增加了一个关键动作：当执行者需要某个计划中没有的具体细节时，它会在继续之前显式地向设计师询问。这避免了 LLM 那种经典的失败——煞有介事地凭空编造细节。

实现方式：角色提示词中包含「当你需要未被告知的具体信息时，在产出输出之前先按名字向相关角色询问」。

### 为什么验证者最重要

Cemri 等人（MAST）追踪了 1642 次多智能体执行失败。其中 21.3% 是验证缺口——系统交付了一个无人核查过的答案。其余 79% 往往也能追溯到「有一项检查静默地失败了，或者根本没运行」。验证是那个承重角色。

普华永道报告称（CrewAI 部署案例，2025）：加入一个结构化校验循环后，准确率从 10% 提升到 70%。仅凭一个角色就带来 7 倍提升。

### 批评者 vs 验证者

- 批评者是一个 LLM，从质量角度审查产物。主观。可能被看似合理的文字蒙骗。
- 验证者是一个在产物上运行的确定性程序。客观。给出带证据的通过/失败。

两者都要用。批评者能抓住验证者无法言明的品味问题。验证者能抓住批评者看不见的 bug——因为那些 bug 只在运行时才暴露。

### 反模式

你系统里的每个角色都是 LLM，而每个角色的输出都是「我看着挺好」。这是经典的 MAST 失败模式。至少加入一个验证者，让它的通过/失败由代码决定，而不是由 LLM 决定。

### 框架映射

- **CrewAI**——`Agent(role, goal, backstory)` 就是教科书式的专业化接口面。
- **LangGraph**——节点可以拥有专门的提示词；边来强制执行流水线。
- **AutoGen**——在 GroupChat 中使用单词命名、角色特定的 ConversableAgents。
- **OpenAI Agents SDK**——在角色专业化的 Agents 之间使用 handoff 工具。

## 动手构建

`code/main.py` 实现了一条构建简单 Python 函数的四角色流水线：

- **规划者** 产出一份规格说明。
- **执行者** 生成一段代码字符串。
- **批评者**（LLM 模拟）标记出明显的问题。
- **验证者** 在沙箱（`exec`）中针对一个测试用例运行所生成的代码。

演示运行两次：一次执行者产出正确代码（批评者与验证者都通过），一次执行者产出偏离规格的代码（批评者因代码看似合理而漏掉 bug，验证者因测试失败而抓住它）。

运行：

```
python3 code/main.py
```

## 实战运用

`outputs/skill-role-designer.md` 接收一个任务，产出角色名册（3-5 个角色）、每个角色的输入/输出模式，以及验证者的检查项。在把智能体接入某个框架之前先用它。

## 上线交付

清单：

- **至少一个确定性验证者。** 切勿全员都是 LLM。
- **每个角色都有显式的 I/O 模式。** 规划者返回的是规格说明而非散文；执行者读取该模式。
- **沟通式去幻觉。** 执行者在信息缺失时必须向规划者询问；切勿凭空编造。
- **批评者/验证者的先后顺序。** 先跑批评者（便宜，抓设计问题），后跑验证者（慢，抓 bug）。
- **循环预算。** 在上报给人类之前，最多进行 2 轮批评者-执行者的修订。

## 练习

1. 运行 `code/main.py`，观察验证者如何抓住批评者漏掉的 bug。再加一个静态分析检查（统计 `return` 出现的次数）作为额外的验证者。它能抓到哪些运行时测试漏掉的东西？
2. 加入第 5 个角色：「需求分析师」，把用户的愿望翻译成规划者可直接使用的规格说明。哪些沟通式去幻觉的请求应当向上流转到它那里？
3. 阅读 MetaGPT 第 3 节（「Agents」）。列出 MetaGPT 五个角色各自的输入/输出模式。
4. 阅读 ChatDev 的聊天链示意图（arXiv:2307.07924 图 3）。指出沟通式去幻觉在哪里打破了一个本来会无限循环的环路。
5. 普华永道 7 倍的准确率提升来自验证循环。假设三类任务，在其中加入验证者也无济于事——也就是对正确性进行确定性检查不可能或代价过高的场景。

## 关键术语

| 术语 | 人们常说 | 它实际的含义 |
|------|----------------|------------------------|
| 角色专业化（Role specialization） | 「不同的智能体，不同的工作」 | 为规划者/执行者/批评者/验证者角色分别调校的、各不相同的系统提示词。 |
| SOP 模式（SOP pattern） | 「编码的标准作业程序」 | MetaGPT 的框架：每个角色的严格 I/O 模式，把一支团队变成一条流水线。 |
| 沟通式去幻觉（Communicative dehallucination） | 「编造之前先问」 | ChatDev 模式：执行者在细节缺失时向规划者询问，而非自行编造。 |
| 批评者（Critic） | 「LLM 审查者」 | 主观的、带个人观点的审查者。抓品味问题。可能被看似合理的文字蒙骗。 |
| 验证者（Verifier） | 「确定性检查」 | 基于代码的通过/失败。测试运行器、类型检查器、模式校验器。无法被蒙骗。 |
| 验证缺口（Verification gap） | 「没人查过」 | 占 MAST 失败的 21.3%。在没有那个本可抓住 bug 的检查的情况下就交付了答案。 |
| 修订循环（Revision loop） | 「批评者把它打回去」 | 批评者拒绝会触发执行者带反馈重跑。需要一个预算。 |
| 全员 LLM 反模式（All-LLM anti-pattern） | 「我看着挺好」 | 每个角色都是 LLM，没有确定性检查。经典的 MAST 失败。 |
