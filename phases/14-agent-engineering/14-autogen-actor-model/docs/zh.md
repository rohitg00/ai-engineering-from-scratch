# AutoGen v0.4：Actor 模型 Agent

> AutoGen v0.4 将 agent 重新架构为 actor。每个 agent 是一个 actor：一个具有状态、消息收件箱和并发执行的独立计算单元。这与共享内存并发不同，因为 agent 不共享状态——它们通过不可变消息传递进行通信。

**类型：** 构建
**语言：** Python（标准库）
**前置条件：** 第 14 阶段 · 01（Agent Loop），第 14 阶段 · 12（Anthropic 工作流模式）
**时间：** ~75 分钟

## 学习目标

- 解释 actor 模型：actor、消息、邮箱、无共享状态。
- 对比 actor 并发与共享内存并发（线程、锁）。
- 实现一个标准库 actor 系统，包含消息传递、调度和错误隔离。
- 识别 actor 模型何时适合 agent 系统。

## 问题

大多数 agent 框架使用共享状态：多个 agent 读取和写入同一个状态对象。这导致：

1. **竞争条件。** Agent A 读取状态，agent B 写入状态，agent A 基于陈旧数据行动。
2. **死锁。** Agent A 持有锁 X 等待锁 Y；agent B 持有锁 Y 等待锁 X。
3. **级联失败。** Agent A 崩溃时持有锁；agent B、C、D 永远等待。

AutoGen v0.4 的答案：actor 模型。每个 agent 是独立的。状态是私有的。通信通过不可变消息。没有锁，没有竞争条件，崩溃被隔离。

## 概念

### Actor 模型基础

Actor 模型（Hewitt 1973）有三个原语：

1. **Actor。** 具有私有状态和行为的计算单元。
2. **消息。** 不可变数据包，actor 之间发送。
3. **邮箱。** 每个 actor 的消息队列；异步、FIFO。

Actor 只能：
- 创建更多 actor。
- 发送消息。
- 定义接收消息时的行为。

没有共享状态。没有锁。没有直接方法调用。

### AutoGen v0.4 的 Actor 架构

AutoGen v0.4（Microsoft Research，2024）将每个 agent 实现为 actor：

- **AgentActor。** 包装 LLM + 工具 + 记忆。状态是私有的。
- **消息。** 包含发送者、接收者、内容、元数据（时间戳、ID）。
- **运行时。** 管理 actor 生命周期、消息路由和调度。
- **组聊天。** 特殊 actor 作为消息代理，广播到组成员。

### 与共享内存并发的对比

| 方面 | 共享内存（线程） | Actor 模型 |
|------|----------------|-----------|
| 状态 | 共享，需要锁 | 私有，无锁 |
| 通信 | 直接方法调用 | 异步消息传递 |
| 故障隔离 | 差（级联崩溃） | 强（actor 崩溃不影响其他） |
| 可扩展性 | 受锁竞争限制 | 受消息传递带宽限制 |
| 调试 | 困难（非确定性） | 更容易（消息日志是轨迹） |
| 复杂度 | 低（熟悉） | 中（新心智模型） |

### 消息传递模式

- **请求-响应。** Actor A 发送请求，等待 actor B 的响应。超时处理。
- **发布-订阅。** Actor 发布到主题；多个订阅者接收。
- **管道。** Actor A -> B -> C，每个处理并转发。
- **路由器。** 特殊 actor 根据内容将消息分派给工作者。

### 错误处理

Actor 模型中的错误隔离：

- **监督。** 父 actor 监控子 actor 生命周期。子崩溃时，父决定重启、升级或终止。
- **邮箱溢出。** 当 actor 处理速度慢于接收速度时。策略：丢弃、背压、死信队列。
- **消息丢失。** 网络或 actor 崩溃时。策略：至少一次传递、恰好一次（更难）。

### 何时使用 Actor 模型

- **高并发 agent。** 数十到数千个 agent 同时运行。
- **长运行 agent。** 需要数天运行的 agent；崩溃隔离防止级联故障。
- **分布式 agent。** 跨机器运行的 agent；消息传递自然映射到网络。
- **容错需求。** 单个 agent 失败不应使整个系统停止。

### 何时不使用 Actor 模型

- **简单顺序 agent。** 单个 ReAct 循环不需要 actor。
- **低延迟同步需求。** 消息传递增加异步开销。
- **小团队。** Actor 模型增加认知负担；先验证需求。

### 此模式出错的地方

- **消息膨胀。** 消息变得比状态还大。定义紧凑的消息模式。
- **分布式幻觉。** 团队假设 actor = 分布式。标准库 actor 是单机的；分布式需要网络层。
- **过度并发。** 为每个 LLM 调用创建 actor。Actor 是重量级抽象；复用它们。

## 练习

1. 实现监督树：父 actor 创建 3 个子 actor。一个子崩溃时，父重启它并记录故障。
2. 添加背压：当 actor 邮箱超过 100 条消息时，向发送者发送"减速"信号。
3. 实现恰好一次消息传递：消息 ID + 接收者确认 + 发送者重试超时。
4. 测量 actor 开销：创建 1000 个 actor，发送 10000 条消息。与线程池替代方案对比。
5. 将第 12 课的编排器-工作者模式实现为 actor 系统。编排器和工作者是 actor 时，什么变干净，什么变复杂？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| Actor | "并发单元" | 具有私有状态和邮箱的独立计算实体 |
| Mailbox | "消息队列" | 每个 actor 的 FIFO 消息缓冲区 |
| Message | "不可变数据包" | actor 之间传递的数据；发送后不可变 |
| No shared state | "隔离" | actor 不直接访问彼此的状态 |
| Supervision | "故障恢复" | 父 actor 监控并重启崩溃的子 actor |
| Backpressure | "流量控制" | 接收者告诉发送者减速 |
| At-least-once | "可靠传递" | 消息可能重复，但不会丢失 |
| Exactly-once | "完美传递" | 消息既不丢失也不重复；更难实现 |
