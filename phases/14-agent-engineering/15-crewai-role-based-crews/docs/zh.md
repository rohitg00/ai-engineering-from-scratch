# CrewAI：基于角色的 Crew 和 Flow

> CrewAI 将 agent 组织为角色（研究员、写手、分析师），通过结构化流程编排。Crew 是并行或顺序执行的任务集合。Flow 是跨 crew 的条件工作流。这与无角色 agent 不同，因为角色约束了 agent 的行为和工具访问。

**类型：** 构建
**语言：** Python（标准库）
**前置条件：** 第 14 阶段 · 01（Agent Loop），第 14 阶段 · 12（Anthropic 工作流模式）
**时间：** ~75 分钟

## 学习目标

- 解释 CrewAI 的核心抽象：Agent、Task、Crew、Process 和 Flow。
- 对比基于角色的 crew 与无角色 agent 集合。
- 实现一个标准库 crew，包含角色定义、任务分配和流程编排。
- 识别何时角色约束值得其设计成本。

## 问题

无约束的 agent 集合有两个问题：

1. **责任模糊。** 当多个 agent 可以执行相同动作时，不清楚谁应该做什么。
2. **工具冲突。** Agent A 使用工具 X 时，agent B 可能同时需要工具 X，导致竞争或错误。

CrewAI 的答案：为每个 agent 分配角色。角色定义目标、背景故事、工具和约束。Crew 按结构化流程（顺序或层次）编排角色。

## 概念

### 角色（Agent）

CrewAI 的 `Agent` 不只是 LLM + 工具。它有：

- **角色。** 身份标签（"研究员"、"写手"）。
- **目标。** 角色要实现的单一目标。
- **背景故事。** 塑造 agent 语气和方法的叙事上下文。
- **工具。** 角色专属的工具子集。
- **约束。** 角色不能做什么（"不写代码"、"不访问互联网"）。

角色约束行为。研究员 agent 不会尝试写手的工作，因为它的工具和目标不同。

### 任务（Task）

任务是分配给 agent 的工作单元：

- **描述。** 要做什么。
- **预期输出。** 成功标准。
- **Agent。** 分配给谁。
- **上下文。** 来自其他任务的输入。
- **回调。** 完成后运行的函数。

任务可以依赖其他任务的输出。这创建了数据流，不是控制流。

### Crew 和 Process

Crew 是 agent + 任务 + 流程的集合：

- **顺序流程。** 任务一个接一个。简单，可预测。
- **层次流程。** 管理者 agent 将任务委派给工作者 agent。管理者审查和批准。
- **并行流程。** 任务并发运行。需要无依赖。

流程定义执行语义，不是图结构。

### Flow

Flow（CrewAI 的新特性）是跨 crew 的条件工作流：

- **状态。** 跨 crew 共享的数据。
- **条件。** 基于状态决定下一个 crew。
- **路由。** 动态选择执行路径。

Flow 将 crew 组合成更大的系统。

### 与无角色 Agent 的对比

| 方面 | 无角色 Agent | 基于角色的 Crew |
|------|-------------|----------------|
| 身份 | 无 | 角色 + 目标 + 背景故事 |
| 工具访问 | 全部 | 按角色限制 |
| 责任 | 模糊 | 清晰（角色拥有任务） |
| 编排 | 动态（LLM 决定） | 结构化（流程定义） |
| 可预测性 | 低 | 高 |
| 设计成本 | 低 | 中（需要角色设计） |

### 何时使用基于角色的 Crew

- **团队模拟。** 当任务自然映射到人类角色（研究 -> 写作 -> 编辑）。
- **合规需求。** 需要审计谁做了什么；角色提供责任。
- **工具隔离。** 某些工具昂贵或危险；限制访问。
- **多语言团队。** 不同角色使用不同语言或领域术语。

### 何时不使用基于角色的 Crew

- **简单任务。** 单个 agent 可以完成。
- **探索阶段。** 角色约束过早；先自由探索。
- **动态团队。** 角色频繁变化；维护成本高。

### 此模式出错的地方

- **角色过细。** 10 个 agent 每个做一件事。开销超过收益。
- **角色冲突。** 两个角色的目标矛盾。定义清晰的边界。
- **流程僵化。** 顺序流程无法适应运行时变化。使用层次或条件流程。

## 构建

`code/main.py` 实现标准库 crew：

- `Role` —— 名称、目标、工具集、约束。
- `Agent` —— 包装角色 + LLM + 记忆。
- `Task` —— 描述、预期输出、分配的角色、依赖。
- `Crew` —— 角色集合、任务集合、流程（顺序/层次/并行）。
- `Flow` —— 条件路由，基于状态选择下一个 crew。

运行：

```
python3 code/main.py
```

跟踪显示角色创建、任务分配、顺序执行、层次委派和条件流路由。

## 使用

- **CrewAI（Python）** —— 生产框架；与 LangChain 集成。
- **自定义 crew** —— 标准库实现（如本课）覆盖核心概念；CrewAI 添加工具集成、记忆、流式。
- **替代模式** —— AutoGen 的 group chat（第 14 课）是动态角色；CrewAI 是静态角色。

## 练习

1. 添加角色冲突检测器：当两个角色的目标可能矛盾时（"节省成本" vs "最大化质量"），发出警告。
2. 实现动态角色分配：基于任务描述，自动选择最合适的角色（从角色库中）。
3. 为层次流程添加"审查门"：管理者批准前，任务输出不可传递给下游。
4. 测量角色开销：对比 1 个无角色 agent vs 3 个角色 agent 执行相同任务。token 成本和延迟差异？
5. 将 CrewAI 角色映射到人类团队。你的组织中的哪些角色可以自动化？哪些需要保留人类判断？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| Role | "Agent 身份" | 约束行为的目标 + 工具 + 背景故事 |
| Crew | "Agent 团队" | 角色 + 任务 + 流程的集合 |
| Process | "执行模式" | 顺序、层次或并行 |
| Flow | "跨 crew 工作流" | 条件路由和状态共享 |
| Task context | "输入依赖" | 一个任务的输出作为另一个任务的输入 |
| Callback | "后处理" | 任务完成后运行的函数 |
| Role boundary | "职责分离" | 角色不能跨越的工具/任务限制 |
| Manager agent | "委派者" | 层次流程中分配任务给工作者的角色 |
