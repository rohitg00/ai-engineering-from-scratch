# 26 · 失败模式：为什么智能体会崩溃

> MASFT（Berkeley，2025）将多智能体的 14 种失败模式归入 3 个类别。Microsoft 的分类法记录了既有 AI 失败如何在智能体（agentic）场景下被放大。来自业界的现场数据收敛到五种反复出现的模式：幻觉动作、范围蔓延、级联错误、上下文丢失、工具误用。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置：** 阶段 14 · 05（Self-Refine 与 CRITIC）、阶段 14 · 24（可观测性）
**时长：** 约 60 分钟

## 学习目标

- 说出 MASFT 的三大失败类别，并在每个类别中说出至少四种具体模式。
- 解释为什么智能体失败会放大既有的 AI 失败模式（偏见、幻觉）。
- 描述业界反复出现的五种模式及其缓解手段。
- 实现一个基于标准库的检测器，为智能体轨迹（trace）打上失败模式标签。

## 问题所在

团队上线的智能体在 90% 的轨迹上能正常工作。那 10% 的失败并非随机噪声——它们落在为数不多的几个反复出现的类别里。一旦你能为它们命名，你就能监控它们并修复它们。

## 概念

### MASFT（Berkeley，arXiv:2503.13657）

多智能体系统失败分类法（Multi-Agent System Failure Taxonomy）。14 种失败模式聚类到 3 个类别。标注者间一致性 Cohen's Kappa 为 0.88——这些类别可以被可靠地区分。

核心主张：失败是多智能体系统中根本性的设计缺陷，而不是用更好的基础模型就能修复的 LLM 局限。

### Microsoft《智能体 AI 系统失败模式分类法》

- 既有的 AI 失败（偏见、幻觉、数据泄露）在智能体场景下被放大。
- 自主性催生出新的失败：规模化的非预期动作、工具误用、任务漂移（mission drift）。
- 这份白皮书就是智能体产品的风险登记册（risk register）。

### 《刻画智能体 AI 中的故障》（arXiv:2603.06847）

- 失败源于编排（orchestration）、内部状态演化，以及与环境的交互。
- 不仅仅是「糟糕的代码」或「糟糕的模型输出」。

### 《LLM 智能体幻觉综述》（arXiv:2509.18970）

两种主要表现形式：

1. **指令遵循偏离（Instruction-following Deviation）**——智能体不遵守系统提示词。
2. **长程上下文误用（Long-range Contextual Misuse）**——智能体遗忘或误用了来自更早轮次的上下文。

子意图错误（sub-intention errors）：遗漏（Omission，漏掉某步骤）、冗余（Redundancy，重复某步骤）、错序（Disorder，步骤顺序错乱）。

### 业界反复出现的五种模式

Arize、Galileo、NimbleBrain 在 2024–2026 年的现场分析收敛到：

1. **幻觉动作（Hallucinated actions）。** 智能体调用了不存在的工具，或捏造了参数。
2. **范围蔓延（Scope creep）。** 智能体把任务扩展到用户请求之外（创建多余的 PR、发出多余的邮件）。
3. **级联错误（Cascading errors）。** 一次错误调用触发下游连锁效应。一个虚构 SKU 的幻觉触发了四次 API 调用——演变成跨多系统的事故。
4. **上下文丢失（Context loss）。** 长周期任务遗忘了早期轮次的约束。
5. **工具误用（Tool misuse）。** 用错误参数调用了正确的工具，或干脆调用了错误的工具。

级联是最致命的。智能体无法区分「我失败了」和「这个任务根本不可能完成」，并且在遇到 400 错误时常常会幻觉出一条成功消息来收尾闭环。

### 缓解：在每一步设置关卡

在推理链的每一步设置自动化验证关卡（gate），对照环境状态检查事实是否有据可依。具体而言：

- 每步的安全分类器（第 21 课）。
- 工具调用的参数校验（第 06 课）。
- 将检索到的内容与已知事实交叉核对（第 05 课，CRITIC）。
- 通过重新探测状态来检测成功幻觉（文件真的被创建了吗？）。

### 失败监控容易出错的地方

- **只标记崩溃。** 大多数智能体失败会产生看起来合法的输出。需要内容层面的检查。
- **没有基线。** 漂移检测需要一个「最近已知良好（last-known-good）」状态；没有它，你无法判断「情况正在变糟」。
- **过度告警。** 每个失败都触发一次寻呼。要做聚类和限流。

## 动手构建

`code/main.py` 实现了一个基于标准库的失败模式打标器：

- 一份覆盖五种模式的合成轨迹数据集。
- 每种模式对应的检测函数（针对工具调用、输出、重复动作的特征模式）。
- 一个打标器，为每条轨迹打标签并报告模式分布。

运行它：

```
python3 code/main.py
```

输出：逐条轨迹的标签 + 聚合分布，廉价地复现了 Phoenix 的轨迹聚类所能揭示的内容。

## 实际运用

- **Phoenix** 用于生产环境的漂移聚类（第 24 课）。
- **Langfuse** 用于会话回放 + 标注。
- **自定义方案** 用于你的可观测性平台无法检测的领域特定特征。

## 交付落地

`outputs/skill-failure-detector.md` 会生成针对你所在领域定制、并接入轨迹存储的失败模式检测器。

## 练习

1. 为「成功幻觉」添加一个检测器：智能体返回成功，但目标状态毫无变化。
2. 从你构建过的某个产品中给 100 条真实轨迹打标签。哪种模式占主导？修复它的成本是多少？
3. 实现一个「级联半径（cascade radius）」指标：给定第 N 步的一个失败，它影响了多少个下游步骤？
4. 阅读 MASFT 的 14 种失败模式。挑选三种适用于你产品的模式，编写检测器。
5. 将一个检测器接入 CI 作业：如果有 >=5% 的轨迹被打上某种模式标签，就让构建失败。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| MASFT | “多智能体失败分类法” | Berkeley 的 14 模式分类体系 |
| 级联错误（Cascading error） | “涟漪式失败” | 一个早期错误在 N 个步骤中传播 |
| 上下文丢失（Context loss） | “忘了约束” | 长周期轮次丢掉了早期轮次的事实 |
| 工具误用（Tool misuse） | “错工具 / 错参数” | 调用合法，但调用方式错误 |
| 成功幻觉（Success hallucination） | “伪造完成” | 智能体在 400 上声称成功；状态毫无变化 |
| 范围蔓延（Scope creep） | “越权” | 智能体做了超出请求的事 |
| 指令遵循偏离（Instruction-following deviation） | “不服从” | 忽略系统提示词或用户约束 |
| 子意图错误（Sub-intention errors） | “计划 bug” | 计划执行中的遗漏、冗余、错序 |
