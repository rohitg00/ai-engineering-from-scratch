# 28 · 编排模式：监督者、蜂群、分层

> 在 2026 年的各类框架中反复出现四种编排模式：监督者-工作者（supervisor-worker）、蜂群/点对点（swarm / peer-to-peer）、分层（hierarchical）、辩论（debate）。Anthropic 的建议是：「关键在于构建适合你需求的系统。」从简单做起；只有当单个智能体加上五种工作流模式仍不够用时，才引入拓扑结构。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置：** 阶段 14 · 12（工作流模式）、阶段 14 · 25（多智能体辩论）
**时长：** 约 60 分钟

## 学习目标

- 说出四种反复出现的编排模式，以及各自适用的场景。
- 描述 2026 年 LangChain 的推荐做法：基于工具调用的监督，对比监督者库（supervisor libraries）。
- 解释 Anthropic 的「构建适合的系统」原则，以及它如何约束拓扑选择。
- 用标准库针对一个共同的脚本化 LLM 实现全部四种模式。

## 问题所在

团队在真正需要之前就急于上「多智能体」。四种模式在各框架中反复出现；一旦你能叫出它们的名字，就能挑出正确的那一种——或者干脆跳过拓扑结构。

## 核心概念

### 监督者-工作者（Supervisor-worker）

- 一个中心化的路由 LLM 把任务分派给专家智能体。
- 它做出决策：回到自身循环、移交给专家、或终止。
- 专家之间互不通信；所有路由都经过监督者。

框架：LangGraph `create_supervisor`、Anthropic 的编排者-工作者（orchestrator-workers）、CrewAI 的 Hierarchical Process。

**2026 年 LangChain 推荐：** 通过直接的工具调用来做监督，而不是用 `create_supervisor`。这能带来更细粒度的「上下文工程（context engineering）」控制——你可以精确决定每个专家看到什么。

### 蜂群/点对点（Swarm / peer-to-peer）

- 智能体通过共享的工具表面（shared tool surface）直接相互移交。
- 没有中心路由器。
- 比监督者模式延迟更低（跳数更少）。
- 更难推理（没有单一的控制点）。

框架：LangGraph 的蜂群拓扑、OpenAI Agents SDK 的移交（handoffs）（当所有智能体都能向其他所有智能体移交时）。

### 分层（Hierarchical）

- 监督者管理子监督者，子监督者再管理工作者。
- 在 LangGraph 中以嵌套子图（nested subgraphs）实现；在 CrewAI 中以嵌套 crew 实现。
- 能扩展到大规模的智能体群体，代价是运维复杂度。

何时需要它：当单个监督者的上下文预算无法容纳所有专家的描述时。

### 辩论（Debate）

- 并行的提议者 + 迭代式的交叉批评（见第 25 课）。
- 它其实不算编排——更像验证——但在框架中它会作为一种拓扑选择出现。

### CrewAI 的 Crew 对比 Flow

CrewAI 将两种部署模式正式化：

- **Flow** 用于确定性的、事件驱动的自动化（生产环境推荐的起点）。
- **Crew** 用于自主的、基于角色的协作。

这与上面四种模式是正交的，但可以映射到拓扑：Flow 通常是监督者或分层；Crew 通常是带 LLM 路由器的监督者。

### Anthropic 的建议

「在 LLM 领域，成功并非在于构建最复杂的系统，而在于构建适合你需求的系统。」

决策顺序：

1. 单智能体 + 工作流模式（第 12 课）——从这里开始。
2. 监督者-工作者——当你有 2-4 个专家时。
3. 蜂群——当延迟比推理清晰度更重要时。
4. 分层——仅当监督者的上下文预算撑不住时。
5. 辩论——当准确性比成本更重要时。

### 这个模式会在哪里出错

- **拓扑优先思维。** 在弄清楚多智能体到底解决什么问题之前，就说「我们需要多智能体」。
- **蜂群中的来回弹跳移交。** A -> B -> A -> B。请使用跳数计数器。
- **假分层。** 因为「企业级」搞了三层；实际上只有两个团队。请压平。

## 动手构建

`code/main.py` 用标准库针对一个脚本化 LLM 实现了全部四种模式：

- `Supervisor`——中心路由器。
- `Swarm`——带直接移交的点对点。
- `Hierarchical`——监督者的监督者。
- `Debate`——并行提议者 + 批评。

每种模式都处理相同的三意图任务（退款 / bug / 销售）。轨迹的形状各不相同。

运行它：

```
python3 code/main.py
```

输出：每种模式的轨迹 + 操作计数。监督者最清晰；蜂群最短；分层最深；辩论最昂贵。

## 实战运用

- **LangGraph** 用于监督者和分层（嵌套子图）。
- **OpenAI Agents SDK** 用于「移交即工具」（handoffs-as-tools，监督者形态）。
- **CrewAI Flow** 用于生产环境的确定性场景。
- **自定义实现** 用于辩论，或当你想要精确控制时。

## 练习

1. 通过移除路由器，把一个监督者-工作者改造成蜂群。什么坏了？什么变好了？
2. 给蜂群加一个跳数计数器：超过 3 次移交后拒绝。它能抓住 A->B->A 这样的来回弹跳吗？
3. 为一个有 12 个专家的领域构建一个两级分层系统。在不使用嵌套的情况下，上下文预算会在哪里撑不住？
4. 在一个生产形态的工作负载上对这四种模式进行性能剖析。在哪个指标（延迟、成本、准确性、可调试性）上谁胜出？
5. 阅读 Anthropic 的《Building Effective Agents》一文。把你生产中的每个流程映射到这四种模式之一。有没有哪个无法干净地映射上？

## 关键术语

| 术语 | 人们怎么说 | 它的真正含义 |
|------|----------------|------------------------|
| 监督者-工作者（Supervisor-worker） | 「路由器 + 专家」 | 中心 LLM 分派给专家；专家之间互不通信 |
| 蜂群（Swarm） | 「点对点」 | 通过共享工具直接移交；没有中心路由器 |
| 分层（Hierarchical） | 「监督者的监督者」 | 面向大规模群体的嵌套子图 |
| 辩论（Debate） | 「提议者 + 批评」 | 并行提议者，交叉批评（第 25 课） |
| 基于工具调用的监督（Tool-call-based supervision） | 「不用库的监督者」 | 把监督者实现为直接的工具调用，以获得上下文控制 |
| Crew | 「自主团队」 | CrewAI 基于角色的协作模式 |
| Flow | 「确定性工作流」 | CrewAI 事件驱动的生产模式 |
