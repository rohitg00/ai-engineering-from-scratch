# 06 · 工具使用与函数调用

> Toolformer（Schick 等，2023）开创了自监督式工具标注。伯克利函数调用排行榜 V4（Berkeley Function Calling Leaderboard V4，Patil 等，2025）确立了 2026 年的评测标杆：40% 智能体（agentic）、30% 多轮、10% 实时（live）、10% 非实时（non-live）、10% 幻觉检测。单轮调用已被攻克，而记忆、动态决策与长程工具链尚未解决。

**类型：** 实践构建
**语言：** Python（标准库）
**前置：** 阶段 14 · 01（智能体循环）、阶段 13 · 01（函数调用深入剖析）
**时长：** 约 60 分钟

## 学习目标

- 解释 Toolformer 的自监督训练信号：仅当执行工具能降低下一个 token 的损失时，才保留该工具标注。
- 说出 BFCL V4 的五个评测类别，以及各自衡量什么。
- 用标准库实现一个工具注册表（tool registry），具备模式校验、参数强制转换（argument coercion）与执行沙箱（execution sandboxing）能力。
- 诊断 2026 年的三大未解难题：长程工具链、动态决策与记忆。

## 问题所在

早期的工具使用追问的是：模型能否预测出一次正确的函数调用？而现代工具使用追问的是：模型能否跨越 40 个步骤串联工具——在有记忆、部分可观测（partial observability）、能从工具失败中恢复、且不会臆造出根本不存在的工具的前提下？

Toolformer 奠定了基线：模型可以通过自监督学会何时调用工具。BFCL V4 定义了 2026 年的评测目标。二者之间的鸿沟，正是生产级智能体所栖身的空间。

## 核心概念

### Toolformer（Schick 等，NeurIPS 2023）

思路：让模型用候选的 API 调用来标注它自己的预训练语料。对每个候选项执行调用。仅当把工具结果纳入后能降低下一个 token 的损失时，才保留该标注。最后在过滤后的语料上做微调。

覆盖的工具：计算器、问答系统、搜索引擎、翻译器、日历。其自监督信号纯粹关乎工具是否有助于预测文本——没有任何人工标注。

规模化结果：工具使用能力随规模涌现。较小的模型会被工具标注拖累，较大的模型则从中获益。这正是为何 2026 年的前沿模型天生就具备强大的工具使用能力，而多数 7B 模型则需要专门的工具使用微调才能保证可靠。

### 伯克利函数调用排行榜 V4（Patil 等，ICML 2025）

BFCL 是 2026 年事实上的评测标准。V4 的构成：

- **智能体（Agentic，40%）**——完整的智能体轨迹：记忆、多轮、动态决策。
- **多轮（Multi-Turn，30%）**——带工具链的交互式对话。
- **实时（Live，10%）**——用户提交的真实提示（分布更难）。
- **非实时（Non-Live，10%）**——合成测试用例。
- **幻觉（Hallucination，10%）**——检测何时不应调用任何工具。

V3 引入了基于状态的评测：在一段工具序列执行后，检查 API 的实际状态（例如"文件是否已创建？"），而非去匹配工具调用的抽象语法树（AST）。V4 新增了网页搜索、记忆与格式敏感性等类别。

2026 年的关键发现：单轮函数调用已近乎攻克。失败集中在记忆（跨轮携带上下文）、动态决策（基于先前结果选择工具）、长程链（20 步以上后产生漂移）以及幻觉检测（在没有合适工具时拒绝调用）。

### 工具模式（Tool schema）

每个供应商都有自己的模式。它们在细节上有别，但共享同一种形态：

```
name: string
description: string (what it does, when to use it)
input_schema: JSON Schema (properties, required, types, enums)
```

Anthropic 直接使用 `input_schema`。OpenAI 使用 `function.parameters`。两者都接受 JSON Schema。描述是承重的——模型靠阅读描述来挑选正确的工具。糟糕的工具描述是"选错工具"类故障的头号根因。

### 参数校验

不要信任任何工具调用。务必校验：

1. **类型强制转换。** 模式声明为 int，但模型可能返回字符串 "5"。若无歧义则强制转换；否则拒绝。
2. **枚举校验。** 若模式声明 `status in {"open", "closed"}`，而模型给出 `"in_progress"`，则以一条描述性错误拒绝。
3. **必填字段。** 缺失必填字段时——立即向模型返回一条错误观测，而不是直接崩溃。
4. **格式校验。** 日期、邮箱、URL——用具体的解析器校验，而非正则表达式。

每一次校验失败都应返回一条结构化观测，以便模型用正确的形态重试。

### 并行工具调用

现代供应商支持在一个 assistant 轮次内并行调用工具。其循环为：

1. 模型发出 3 个工具调用，各自带有不同的 `tool_use_id`。
2. 运行时执行它们（若彼此独立则并行执行）。
3. 每个结果作为一个 `tool_result` 块返回，并通过 `tool_use_id` 关联。

工程准则：把关联 ID 当作承重项对待。一旦把它们调换，你就会得到"错误工具对应错误结果"的路由错乱。

### 沙箱化

工具执行就是沙箱边界。详见第 09 课。简而言之：每个工具都应明确其读写面、网络访问、超时与内存上限。通用的 `run_shell(cmd)` 是一个危险信号；具体的 `git_status()` 则更安全。

## 动手构建

`code/main.py` 实现了一个生产级形态的工具注册表：

- JSON Schema 子集校验器（仅用标准库）。
- 带描述、输入模式、超时与执行器的工具注册。
- 参数强制转换与枚举校验。
- 带关联 ID 的并行工具分发。
- 以结构化字符串形式呈现的错误观测。

运行它：

```
python3 code/main.py
```

该轨迹展示了一个迷你智能体在一个轮次内调用三个工具，其中一个被故意构造成畸形调用，并以一条模型可据以行动的描述性错误被拒绝。

## 实际应用

每个供应商都有自己的工具模式——Anthropic、OpenAI、Gemini、Bedrock。若需要多供应商支持，可使用一个转换层（OpenAI Agents SDK、Vercel AI SDK、LangChain 工具适配器）。BFCL 是参考基准——如果工具使用是产品的核心，请在发布前用它来评测你的智能体。

## 发布上线

`outputs/skill-tool-registry.md` 会为给定的任务领域生成工具目录、模式与注册表。其中包含描述质量检查（每个工具的描述是否告诉了模型何时使用它？）。

## 练习

1. 添加一个"空操作（no-op）"工具，让模型可以显式地拒绝使用任何其他工具。在一个类 BFCL 的幻觉测试上进行衡量。
2. 为"int 形式的字符串"和"float 形式的字符串"实现参数强制转换。强制转换从哪里开始会掩盖真正的 bug？
3. 添加按工具的超时与熔断器（连续失败 3 次后，拒绝该工具 60 秒）。这对模型的恢复方式带来了什么改变？
4. 阅读 BFCL V4 的说明。挑选一个类别（例如"多轮"），用你的智能体跑 10 个示例提示。报告通过率。
5. 把标准库版的校验器移植到 Pydantic 或 Zod。Pydantic/Zod 捕获了哪些玩具版漏掉的问题？

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|----------------|------------------------|
| Function calling | "工具使用" | 带校验模式的结构化输出工具调用 |
| Toolformer | "自监督式工具标注" | Schick 2023——保留那些结果能降低下一个 token 损失的工具调用 |
| BFCL | "伯克利函数调用排行榜" | 2026 基准：40% 智能体、30% 多轮、10% 实时、10% 非实时、10% 幻觉 |
| Tool schema | "面向模型的函数签名" | name、description、参数的 JSON Schema |
| tool_use_id | "关联 ID" | 把一次工具调用与其结果绑定；对并行分发至关重要 |
| Hallucination detection | "知道何时不该调用" | V4 类别：在没有合适工具时拒绝调用 |
| Argument coercion | "字符串转 int 的修复" | 针对可预测的模式不匹配做窄修复；有歧义则拒绝 |
| Sandboxing | "工具执行边界" | 按工具划定读写面、网络、超时与内存上限 |
