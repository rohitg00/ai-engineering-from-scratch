# 11 · 用 HTN 与进化搜索做规划

> 符号化规划处理那些「计划可被证明正确」的场景；进化式代码搜索处理那些「适应度函数可被机器校验」的场景。ChatHTN（2025）与 AlphaEvolve（2025）展示了二者各自在与 LLM 配对后能解锁的能力。

**类型：** 构建
**语言：** Python（标准库）
**前置：** 阶段 14 · 02（ReWOO 与 Plan-and-Execute）
**时长：** 约 75 分钟

## 学习目标

- 解释「层次任务网络（Hierarchical Task Networks，HTN）」：任务、方法、算子、前置条件、效果。
- 描述 ChatHTN 的混合循环——以符号搜索为主、以 LLM 回退分解为辅。
- 解释 AlphaEvolve 的进化循环，以及它为何只在拥有程序化评估器时才奏效。
- 用标准库实现一个玩具级 HTN 规划器，外加一个玩具级进化搜索。

## 问题所在

ReWOO（第 02 课）、Plan-and-Execute 与 ReAct 覆盖了智能体规划的绝大多数场景。但有两类场景它们处理得并不好：

1. **可证明正确的计划。** 排程、航线规划、合规工作流——这些计划必须在构造上就是可靠的。一个流畅的 LLM 计划如果偶尔幻觉出某个步骤，便是不可接受的。
2. **带有机器可校验适应度函数的优化。** 矩阵乘法、调度启发式、编译器 pass——目标不是「一个正确的计划」，而是「最优的计划」。

HTN 规划与 AlphaEvolve 解决的是这两个不同的问题。二者都把 LLM 当作放大器，而非替代品。

## 核心概念

### 层次任务网络

一个 HTN 由以下部分组成：

- **任务（Tasks）**——复合任务（需要被分解）与原子任务（可直接执行）。
- **方法（Methods）**——把一个复合任务分解为若干子任务的方式，附带前置条件。
- **算子（Operators）**——带有前置条件与效果的原子动作。
- **状态（State）**——一组事实的集合。

规划过程：给定一个目标任务和一个初始状态，找到一种分解，使其落到若干原子算子上，且这些算子的前置条件能按顺序得到满足。

HTN 比 LLM 出现得更早，至今仍是「可证明正确的计划」领域的参照标准。

### ChatHTN（Gopalakrishnan 等，2025）

ChatHTN（arXiv:2505.11814）将符号化 HTN 与 LLM 查询交织在一起：

1. 尝试用现有方法分解当前的复合任务。
2. 若没有方法适用，则询问 LLM：「在状态 `s` 下，你会如何分解 `task`？」
3. 把 LLM 的回应翻译成候选子任务。
4. 对照算子模式（schema）进行校验；拒绝非法的分解。
5. 递归处理。

该论文的核心论断是：所产生的每一个计划都是可证明可靠的，因为 LLM 的建议只能作为候选分解进入流程，绝不会作为对计划的直接修改。符号层掌控正确性，LLM 则负责扩充方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 年的后续工作）增加了一个学习器，通过回归对 LLM 产出的分解进行泛化——可将 LLM 查询频率削减最多 75%。

### AlphaEvolve（Novikov 等，2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025 年 6 月）则是另一种路数：由 Gemini 2.0 Flash/Pro 集成（ensemble）所编排的进化式代码搜索。

循环如下：

1. 从一个种子程序 + 一个程序化评估器（返回适应度分数）开始。
2. 由一组 LLM 集成提出变异（mutations）。
3. 让变异通过评估器跑一遍。
4. 保留最优者；再次变异。

已发表的成果：

- 在 56 年来首次改进了 4x4 复数矩阵乘法的 Strassen 算法（48 次标量乘法）。
- 通过一项 Borg 调度启发式，为 Google 收回了 0.7% 的算力。
- 在一项前沿工作负载上将 FlashAttention 提速 32%。

硬性约束：适应度函数必须可被机器校验。对散文式答案做进化搜索是不会收敛的。

### 何时选用哪一种

| 问题类别 | 选用 | 原因 |
|---------------|-----|-----|
| 带硬约束的排程 | HTN + ChatHTN | 可证明的可靠性 |
| 编译器优化 | AlphaEvolve | 适应度可被机器校验 |
| 多步任务执行 | ReAct / ReWOO | LLM 处于循环中，无形式化保证 |
| 带测试的代码改进 | AlphaEvolve | 测试即评估器 |
| 受策略约束的自动化 | HTN | 前置条件可编码策略 |

### 这一模式容易出错的地方

- **没有算子的 HTN。** 若缺少前置条件/效果模式，可靠性论断便会崩塌。ChatHTN 的「LLM 建议分解」依赖该模式来拒绝非法动作。
- **没有真实评估器的 AlphaEvolve。** 「问 LLM 这段代码是不是更好」不是适应度函数。评估器必须是确定性的且快速的。
- **过度工程化。** 大多数智能体任务两者都不需要。请先伸手去拿 ReAct 或 ReWOO。

## 动手构建

`code/main.py` 实现了两个玩具：

- 一个标准库 HTN 规划器，包含算子、方法、前置条件、效果，以及一个在没有方法匹配某复合任务时启动的 `LLMFallback`。这里的「LLM」是一个脚本化的分解器，因此规划器可离线运行。
- 一个针对算术程序的标准库进化搜索：生长出一些表达式，使其输出在测试集上最小化 `|f(x) - target|`。评估器是确定性的。

运行它：

```
python3 code/main.py
```

跟踪输出会展示 HTN 规划器分解一个复合任务（其间触发一次计划中段的 LLM 回退），以及进化循环收敛到某个目标表达式的过程。

## 实际应用

- **HTN 规划器**——`pyhop`、`SHOP3`，或者为面向特定领域的策略执行自行构建一个。
- **ChatHTN**——研究代码；其模式（符号 + LLM 回退）可干净地移植到任何 HTN 规划器上。
- **AlphaEvolve**——DeepMind 论文；其模式（集成 + 评估器）是可复现的。OpenEvolve 及类似的开源分支正在涌现。
- **智能体框架**——目前还没有任何框架内置一流的 HTN 或 AlphaEvolve 支持。请把它构建为一个子智能体或一个后台工作者。

## 交付落地

`outputs/skill-hybrid-planner.md` 会生成一个混合规划器脚手架（HTN 或进化式），其中明确界定了 LLM 的角色范围。

## 练习

1. 为 HTN 规划器加上回溯：当某个算子的后置条件在运行时失败时，回滚并尝试下一个方法。
2. 给 ChatHTN 加一个 LLM-方法缓存：当 LLM 在状态模式 `P` 下分解任务 `T` 时，存下结果。下次调用时先重新检查方法库。
3. 把进化搜索的评估器换成一套真实测试套件。进化出一个能通过 20 个测试用例的排序函数；报告收敛所需的代数。
4. 阅读 AlphaEvolve 关于评估器设计的笔记。为你所关心的某个领域（SQL 查询优化、测试套件最小化、部署 YAML）设计一个评估器。
5. 组合：用 HTN 把一个复合任务分解为若干子任务，再对每个子任务的原子算子做进化搜索。它在哪里大放异彩，又在哪里过度工程化？

## 关键术语

| 术语 | 人们怎么说 | 它实际指什么 |
|------|----------------|------------------------|
| HTN | 「层次规划器」 | 带算子、前置条件、效果的任务分解 |
| 方法（Method） | 「分解规则」 | 把复合任务拆成子任务的方式 |
| 算子（Operator） | 「原子动作」 | 带前置条件与效果的具体步骤 |
| ChatHTN | 「LLM + HTN」 | 当无方法匹配时，符号规划器询问 LLM |
| AlphaEvolve | 「进化式代码搜索」 | 集成 LLM 变异代码；确定性评估器进行选择 |
| 适应度函数（Fitness function） | 「评估器」 | 对输出给出确定性、可机器校验的分数 |
| 在线方法学习 | 「缓存的 LLM 分解」 | 存储并泛化 LLM 计划以削减查询成本 |
