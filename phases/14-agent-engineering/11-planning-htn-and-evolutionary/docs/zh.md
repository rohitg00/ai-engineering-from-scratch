# 用 HTN 和进化搜索规划

> 符号规划处理计划可证明正确的案例。进化代码搜索处理适应度函数可机器检查的案例。ChatHTN（2025）和 AlphaEvolve（2025）展示了当与 LLM 配对时，各自解锁了什么。

**类型：** 构建
**语言：** Python（标准库）
**前置条件：** 第 14 阶段 · 02（ReWOO 和 Plan-and-Execute）
**时间：** ~75 分钟

## 学习目标

- 解释层次任务网络：任务、方法、操作符、前提条件、效果。
- 描述 ChatHTN 的混合循环——带 LLM 回退分解的符号搜索。
- 解释 AlphaEvolve 的进化循环，以及为什么它只适用于程序化评估器。
- 用标准库实现玩具 HTN 规划器加玩具进化搜索。

## 问题

ReWOO（第 02 课）、Plan-and-Execute 和 ReAct 覆盖了大多数 agent 规划。有两种情况它们覆盖不好：

1. **可证明正确性的计划。** 调度、飞行路径、合规工作流——计划必须按构造正确。流畅的 LLM 计划有时幻觉步骤是不可接受的。
2. **适应度函数可机器检查的优化。** 矩阵乘法、调度启发式、编译器通道——目标不是"正确计划"而是"最佳计划"。

HTN 规划和 AlphaEvolve 解决两个不同问题。两者都使用 LLM 作为放大器，而非替代。

## 概念

### 层次任务网络

HTN 是：

- **任务** —— 复合（待分解）和原语（直接可执行）。
- **方法** —— 将复合任务分解为子任务的方式，带前提条件。
- **操作符** —— 带前提条件和效果的原语动作。
- **状态** —— 事实集。

规划：给定目标任务和初始状态，找到分解为原语操作符的序列，其前提条件按顺序满足。

HTN 比 LLM 更古老，仍然是可证明正确计划的参考。

### ChatHTN（Gopalakrishnan 等，2025）

ChatHTN（arXiv:2505.11814）交错符号 HTN 与 LLM 查询：

1. 尝试用现有方法分解当前复合任务。
2. 如果没有方法适用，问 LLM："在状态 `s` 中你会如何分解 `task`？"
3. 将 LLM 响应翻译为候选子任务。
4. 针对操作符模式验证；拒绝无效分解。
5. 递归。

论文的核心主张：每个产生的计划都是可证明正确的，因为 LLM 建议只作为候选分解进入，从不作为直接计划编辑。符号层拥有正确性；LLM 扩展方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 后续）添加了一个学习者，通过回归泛化 LLM 产生的分解——将 LLM 查询频率降低多达 75%。

### AlphaEvolve（Novikov 等，2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025年6月）是另一种野兽：由 Gemini 2.0 Flash/Pro 集成编排的进化代码搜索。

循环：

1. 从种子程序 + 程序化评估器（返回适应度分数）开始。
2. LLM 集成提议突变。
3. 通过评估器运行突变。
4. 保留最佳；再次突变。

已发表的胜利：

- 56 年来首次超越 Strassen 的 4x4 复矩阵乘法改进（48 次标量乘法）。
- 通过 Borg 调度启发式恢复 0.7% Google 计算。
- 在前沿工作负载上 FlashAttention 加速 32%。

硬约束：适应度函数必须可机器检查。对散文答案的进化搜索不会收敛。

### 何时使用哪个

| 问题类别 | 使用 | 原因 |
|----------|------|------|
| 带硬约束的调度 | HTN + ChatHTN | 可证明正确性 |
| 编译器优化 | AlphaEvolve | 可机器检查的适应度 |
| 多步任务执行 | ReAct / ReWOO | LLM 在循环中，无形式保证 |
| 带测试的代码改进 | AlphaEvolve | 测试就是评估器 |
| 策略约束自动化 | HTN | 前提条件编码策略 |

### 此模式出错的地方

- **没有操作符的 HTN。** 没有前提条件/效果模式，正确性主张崩溃。ChatHTN 的"LLM 建议分解"需要模式来拒绝无效移动。
- **没有真实评估器的 AlphaEvolve。** "问 LLM 代码是否更好"不是适应度函数。评估器必须是确定性和快速的。
- **过度工程。** 大多数 agent 任务不需要任何一个。先使用 ReAct 或 ReWOO。

## 构建

`code/main.py` 实现两个玩具：

- 标准库 HTN 规划器，带操作符、方法、前提条件、效果和 `LLMFallback`，当没有方法匹配复合任务时触发。"LLM"是脚本化分解器，因此规划器可离线运行。
- 标准库进化搜索，覆盖算术程序：增长表达式，其输出在测试集上最小化 `|f(x) - target|`。评估器是确定性的。

运行：

```
python3 code/main.py
```

跟踪显示 HTN 规划器分解复合任务（带中间 LLM 回退）和进化循环收敛到目标表达式。

## 使用

- **HTN 规划器** —— `pyhop`、`SHOP3`，或为你自己的领域特定策略执行构建。
- **ChatHTN** —— 研究代码；模式（符号 + LLM 回退）可干净地移植到任何 HTN 规划器。
- **AlphaEvolve** —— DeepMind 论文；模式（集成 + 评估器）可复现。OpenEvolve 和类似开源分支正在出现。
- **Agent 框架** —— 尚无框架提供一等 HTN 或 AlphaEvolve。将其构建为子 agent 或后台工作者。

## 交付

`outputs/skill-hybrid-planner.md` 生成混合规划器脚手架（HTN 或进化），显式限定 LLM 角色。

## 练习

1. 用回溯扩展 HTN 规划器：当操作符的后置条件在运行时失败，回滚并尝试下一个方法。
2. 为 ChatHTN 添加 LLM 方法缓存：当 LLM 在状态模式 `P` 中分解任务 `T` 时，存储结果。下次调用时先重新检查方法库。
3. 将进化搜索评估器换成真实测试套件。进化一个通过 20 个测试用例的排序函数；报告收敛代数。
4. 阅读 AlphaEvolve 的评估器设计笔记。为你关心的领域设计评估器（SQL 查询优化、测试套件最小化、部署 YAML）。
5. 组合：用 HTN 将复合任务分解为子任务，然后对每个子任务的原语操作符使用进化搜索。哪里闪耀，哪里过度工程？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| HTN | "层次规划器" | 带操作符、前提条件、效果的任务分解 |
| Method | "分解规则" | 将复合任务拆分为子任务的方式 |
| Operator | "原语动作" | 带前提条件和效果的具体步骤 |
| ChatHTN | "LLM + HTN" | 当没有方法匹配时，符号规划器询问 LLM |
| AlphaEvolve | "进化代码搜索" | 集成 LLM 突变代码；确定性评估器选择 |
| Fitness function | "评估器" | 对输出的确定性、可机器检查分数 |
| Online method learning | "缓存 LLM 分解" | 存储 + 泛化 LLM 计划以降低查询成本 |
