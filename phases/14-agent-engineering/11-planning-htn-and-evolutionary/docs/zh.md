# 使用 HTN 和进化搜索进行规划

> 符号规划处理计划可证明正确的情况。进化代码搜索处理适合度函数可由机器检查的情况。ChatHTN（2025）和 AlphaEvolve（2025）展示了与 LLM 配对时各自解锁的内容。

**类型：** 构建（Build）
**语言：** Python（标准库）
**前置要求：** 阶段 14 · 02（ReWOO and Plan-and-Execute）
**时长：** 约 75 分钟

## 学习目标

- 解释层次任务网络（Hierarchical Task Networks）：任务、方法、操作符、前置条件、效果。
- 描述 ChatHTN 的混合循环——带有 LLM 后备分解的符号搜索。
- 解释 AlphaEvolve 的进化循环，以及为什么它只适用于程序化评估器。
- 在标准库中实现玩具 HTN 规划器以及玩具进化搜索。

## 问题背景

ReWOO（第 02 课）、Plan-and-Execute 和 ReAct 涵盖了大多数 Agent 规划。它们没有很好覆盖的两个情况：

1. **具有可证明正确性的计划。** 调度、飞行路径规划、合规工作流——计划必须在构造上可靠。有时会产生幻觉步骤的流利 LLM 计划是不可接受的。
2. **具有机器可检查适应度函数的优化。** 矩阵乘法、调度启发式、编译器通道——目标不是"正确的计划"，而是"最佳计划"。

HTN 规划和 AlphaEvolve 解决了这两个不同的问题。两者都使用 LLM 作为放大器，而不是替代品。

## 核心概念

### 层次任务网络（Hierarchical Task Networks）

HTN 是：

- **任务（Tasks）**——复合的（待分解）和原始的（可直接执行的）。
- **方法（Methods）**——将复合任务分解为子任务的方式，带有前置条件。
- **操作符（Operators）**——带有前置条件和效果的原始动作。
- **状态（State）**——一组事实。

规划：给定目标任务和初始状态，找到分解为前置条件按顺序满足的原始操作符。

HTN 比 LLM 更古老，仍然可证明正确计划的参考。

### ChatHTN（Gopalakrishnan 等人，2025）

ChatHTN（arXiv:2505.11814）将符号 HTN 与 LLM 查询交错：

1. 尝试用现有方法分解当前复合任务。
2. 如果没有方法适用，问 LLM："你如何在状态 `s` 中分解 `task`？"
3. 将 LLM 响应转换为候选子任务。
4. 根据操作符模式验证；拒绝无效分解。
5. 递归。

论文的核心主张：产生的每个计划都是可证明可靠的，因为 LLM 建议仅作为候选分解进入，绝不作为直接计划编辑。符号层拥有正确性；LLM 扩展方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 年后续）添加了一个通过学习泛化 LLM 产生的分解的学习器——将 LLM 查询频率降低高达 75%。

### AlphaEvolve（Novikov 等人，2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025 年 6 月）是一个不同的野兽：由 Gemini 2.0 Flash/Pro 集成编排的进化代码搜索。

循环：

1. 从种子程序 + 程序化评估器（返回适应度分数）开始。
2. LLM 集成提出变异。
3. 通过评估器运行变异。
4. 保留最好的；再次变异。

已发布的胜利：

- 56 年来首次改进 4x4 复矩阵乘法的 Strassen（48 次标量乘法）。
- 通过 Borg 调度启发式恢复 0.7% 的 Google 计算。
- 前沿工作负载上的 FlashAttention 加速 32%。

硬约束：适应度函数必须可由机器检查。对散文答案的进化搜索不会收敛。

### 何时使用哪个

| 问题类别 | 使用 | 为什么 |
|---------------|-----|-----|
| 带有硬约束的调度 | HTN + ChatHTN | 可证明的可靠性 |
| 编译器优化 | AlphaEvolve | 机器可检查的适应度 |
| 多步骤任务执行 | ReAct / ReWOO | LLM 在循环中，无正式保证 |
| 带测试的代码改进 | AlphaEvolve | 测试就是评估器 |
| 策略绑定自动化 | HTN | 前置条件编码策略 |

### 这种模式哪里会出错

- **没有操作符的 HTN。** 没有前置条件/效果模式，可靠性主张就会崩溃。ChatHTN 的"LLM 建议分解"需要模式来拒绝无效移动。
- **没有真实评估器的 AlphaEvolve。** "询问 LLM 代码是否更好"不是适应度函数。评估器必须是确定性和快速的。
- **过度工程。** 大多数 Agent 任务不需要两者。首先使用 ReAct 或 ReWOO。

## 构建它

`code/main.py` 实现两个玩具：

- 一个带有操作符、方法、前置条件、效果的标准库 HTN 规划器，以及在没有方法匹配复合任务时启动的 `LLMFallback`。 "LLM"是一个脚本化分解器，因此规划器可以离线运行。
- 对算术程序的标准库进化搜索：增长其输出在测试集上最小化 `|f(x) - target|` 的表达式。评估器是确定性的。

运行它：

```
python3 code/main.py
```

轨迹显示 HTN 规划器分解复合任务（带有计划中期的 LLM 后备）和进化循环收敛到目标表达式。

## 使用它

- **HTN 规划器**——`pyhop`、`SHOP3`，或为自己构建特定于域的策略执行。
- **ChatHTN**——研究代码；模式（符号 + LLM 后备）干净地移植到任何 HTN 规划器。
- **AlphaEvolve**——DeepMind 论文；模式（集成 + 评估器）是可重现的。OpenEvolve 和类似的开源分支正在出现。
- **Agent 框架**——还没有任何一个提供一等的 HTN 或 AlphaEvolve。将其构建为子 Agent 或后台工作器。

## 部署它

`outputs/skill-hybrid-planner.md` 生成一个带有显式范围 LLM 角色的混合规划器脚手架（HTN 或进化）。

## 练习

1. 用回溯扩展 HTN 规划器：当操作符的后置条件在运行时失败时，回滚并尝试下一个方法。
2. 向 ChatHTN 添加 LLM 方法缓存：当 LLM 在状态模式 `P` 中分解任务 `T` 时，存储结果。在下次调用时首先检查方法库。
3. 将进化搜索评估器换成一个真实的测试套件。进化一个通过 20 个测试用例的排序函数；报告到收敛的代数。
4. 阅读 AlphaEvolve 的评估器设计说明。为你关心的域（SQL 查询优化、测试套件最小化、部署 YAML）设计评估器。
5. 组合：使用 HTN 将复合任务分解为子任务，然后在每个子任务的原始操作符上使用进化搜索。它在哪里闪耀，在哪里过度工程？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|---------|
| HTN | "层次规划器" | 带有操作符、前置条件、效果的任分解 |
| Method | "分解规则" | 将复合任务分解为子任务的方式 |
| Operator | "原始动作" | 带有前置条件和效果的具步骤 |
| ChatHTN | "LLM + HTN" | 当没有方法匹配时，符号规划器询问 LLM |
| AlphaEvolve | "进化代码搜索" | 集成 LLM 变异代码；确定性评估器选择 |
| Fitness function | "评估器" | 对输出的确定性、机器可检查的分数 |
| Online method learning | "缓存的 LLM 分解" | 存储 + 泛化 LLM 计划以降低查询成本 |
