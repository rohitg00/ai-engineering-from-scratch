# LLM 生产的混沌工程

> 2026 年，LLM 的混沌工程是其自身的学科。在生产中运行实验的先决条件：定义的 SLI/SLO、跟踪+指标+日志可观测性、自动回滚、运行手册、待命。架构有四个平面：控制（实验调度器）、目标（服务、基础设施、数据存储）、安全（防护 + 中止 + 流量过滤器）、可观测性（指标 + 跟踪 + 日志）、反馈（进入 SLO 调整）。防护栏是强制性的：如果每日错误预算消耗 > 2 倍预期，燃烧率警报会暂停实验；抑制窗口 + 跟踪 ID 关联去重警报噪声。节奏：每周小型金丝雀 + SLO 审查；每月游戏日和事后剖析；每季度跨团队弹性审计 + 依赖关系映射。LLM 特定实验：内存过载、网络故障、提供商中断、格式错误的提示、KV 缓存驱逐风暴。工具：Harness Chaos Engineering（LLM 衍生的建议、爆炸半径缩小、MCP 工具集成）；LitmusChaos（CNCF）；Chaos Mesh（CNCF Kubernetes 原生）。

**类型：** 学习
**语言：** Python（标准库，简单的混沌实验运行器）
**先决条件：** 阶段 17 · 23（AI 的 SRE）、阶段 17 · 13（可观测性）
**时间：** 约 60 分钟

## 学习目标

- 说出五个混沌工程先决条件（SLI/SLO、可观测性、回滚、运行手册、待命）并解释为什么跳过任何一个都会破坏实践。
- 绘制四个平面（控制、目标、安全、可观测性）和进入 SLO 的反馈循环。
- 列举五个 LLM 特定实验（内存过载、网络故障、提供商中断、格式错误的提示、KV 驱逐风暴）。
- 根据技术栈选择工具——Harness、LitmusChaos、Chaos Mesh。

## 问题

传统技术栈中的混沌测试已经确立。LLM 技术栈增加了新的故障模式。带有毒字符的 4K token 提示会使 tokenizer 停滞 12 秒。上游提供商 429；你的网关重试；你的服务在重试放大的并发上 OOM。突发负载下的 KV 缓存驱逐风暴导致重新预填充级联，使计算饱和。

这些都不会在单元测试中出现。混沌工程是你如何在用户之前发现它们。

## 概念

### 先决条件

没有以下条件不要在生产中运行混沌：

1. **SLI/SLO**——定义的服务级指标和目标。
2. **可观测性**——跟踪、指标、日志，连接到仪表板。
3. **自动回滚**——阶段 17 · 20 策略标志回滚。
4. **运行手册**——结构化，阶段 17 · 23。
5. **待命**——有人响应。

缺少任何一个都意味着混沌成为真实事件。

### 四个平面 + 反馈

**控制平面**——实验调度器（Litmus 工作流、Chaos Mesh 调度、Harness UI）。

**目标平面**——服务、pod、节点、负载均衡器、数据存储。

**安全平面**——终止开关、抑制窗口、爆炸半径限制、错误预算门。

**可观测性平面**——正常指标 + 跟踪 ID 关联以区分混沌引起的故障和自然故障。

**反馈循环**——发现反馈回 SLO 调整、运行手册更新、代码修复。

### 防护栏是强制性的

- **燃烧率警报**：如果每日错误预算消耗超过预期的 2 倍，则暂停实验。
- **抑制窗口**：在实验期间静默爆炸半径内的非实验警报。
- **跟踪 ID 关联**：所有实验引起的错误都带有一个标签，以便待命可以去重。

### 五个 LLM 特定实验

1. **内存过载**——通过发送具有高并发性的长文本请求来强制 KV 缓存抢占风暴。观察：服务是正常丢弃还是崩溃？

2. **网络故障**——切断推理网关和提供商之间的连接。观察：回退是否在 SLA 内启动？（阶段 17 · 19）

3. **提供商中断模拟**——来自 OpenAI 的 100% 429。观察：路由是否故障转移到 Anthropic？（阶段 17 · 16、19）

4. **格式错误的提示**——注入 tokenizer 停滞的有效负载（例如，深度嵌套的 unicode、巨大的 UTF-8 代码点）。观察：单个请求是否会锁定工作器？

5. **KV 驱逐风暴**——通过使 vLLM 块预算饱和来强制驱逐。观察：LMCache 是恢复还是服务降级？

### 节奏

- **每周**——在预发布中进行小型金丝雀实验，可能是 5% 的生产。
- **每月**——针对特定场景的计划游戏日；跨团队出勤；事后剖析。
- **每季度**——跨团队弹性审计；依赖关系图更新。

### 工具

- **Harness Chaos Engineering**——商业；AI 衍生的实验建议；爆炸半径缩小；MCP 工具集成。
- **LitmusChaos**——CNCF 毕业；基于 Kubernetes 工作流。
- **Chaos Mesh**——CNCF 沙箱；Kubernetes 原生 CRD 风格。
- **Gremlin**——商业；广泛支持。
- **AWS FIS** / **Azure Chaos Studio**——托管云产品。

### 从小处着手

第一个实验：在稳定流量下 pod-kill 一个 decode 副本。观察重新路由和恢复。如果这有效且看起来安全，则升级到网络混沌。

第一个 LLM 特定实验：注入一个提供商 429 持续 5 分钟。观察回退。大多数团队发现他们的回退没有经过充分测试。

### 你应该记住的数字

- 四个平面：控制、目标、安全、可观测性。
- 燃烧率暂停：2 倍预期的每日预算消耗。
- 节奏：每周金丝雀，每月游戏日，每季度审计。
- 五个 LLM 实验：内存、网络、提供商、格式错误的提示、KV 风暴。
