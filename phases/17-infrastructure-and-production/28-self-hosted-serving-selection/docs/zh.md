# Self-Hosted Serving Selection（自托管推理服务选择）

## 1. 简介：为何要自托管 LLM？

虽然托管 API（例如，OpenAI、Anthropic）为许多用例提供了便捷的入门方式，但随着规模扩大，它们面临着几个限制：
- **成本**：对于高流量应用程序，API 成本可能会变得高昂。
- **延迟**：共享端点可能会受到"嘈杂邻居"效应的影响，导致延迟增加。
- **数据隐私**：将敏感数据发送到第三方 API 可能会引发合规性问题。
- **定制化**：您无法控制模型权重、量化方案或推理优化。

**自托管**通过在您自己的基础设施（本地或云）上部署 LLM 来缓解这些问题。然而，它也带来了新的挑战：
- **运营开销**：您需要管理 GPU、扩展、监控、故障恢复。
- **技术复杂性**：您需要选择正确的推理服务器、优化技术和硬件。
- **前期成本**：购买或租赁 GPU 需要资本支出。

本指南将帮助您应对这些挑战，并选择适合您需求的自托管解决方案。

## 2. 决策框架：我应该自托管吗？

在深入研究技术细节之前，请问自己以下几个问题：

### 2.1 自托管清单

| 问题 | 是 | 否 |
|------|----|----|
| 您每月的 API 成本是否超过 $10,000？ | 考虑自托管 | 可能不经济 |
| 您是否有严格的延迟 SLA（例如，< 200ms TTFF）？ | 自托管可以提供更多控制 | 托管 API 可能就足够了 |
| 您是否处理敏感数据（例如，医疗、金融）？ | 自托管可以更好地控制数据 | 托管 API 可能没问题 |
| 您是否需要自定义模型（例如，微调、合并）？ | 自托管是必需的 | 托管 API 可能支持微调 |
| 您是否有工程师可以管理 GPU 基础设施？ | 自托管是可行的 | 考虑托管端点 |

### 2.2 成本盈亏平衡分析

将自托管与托管 API 进行成本比较：

**示例**：
- **托管 API**：GPT-4 Turbo，输入 $0.01/1K Token，输出 $0.03/1K Token。平均每天 1 亿个 Token（输入：输出 = 1:1）→ 每月成本 ~$45,000。
- **自托管**：在 8 个 A100 GPU 上部署 Llama 3.1 70B（成本：$262.16/小时）→ 每月成本 ~$189,000（云）或 ~$36,000（本地）。然而，自托管可以处理更高的流量（例如，每秒 6,000 个 Token），从而降低每个 Token 的成本。

**经验法则**：如果您的每月 Token 使用量超过 ~100 亿个，自托管通常会变得更具成本效益。

## 3. 自托管技术栈

自托管 LLM 涉及多个组件。以下是典型的技术栈：

```
┌─────────────────────────────────────────────────┐
│            Load Balancer / API Gateway          │
│         (LiteLLM, Kong, Envoy)                 │
└───────────────────┬─────────────────────────────┘
                    │
┌───────────────────▼─────────────────────────────┐
│          Inference Server Cluster                │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐          │
│  │ vLLM     │ │ TensorRT│ │ Text    │          │
│  │ Node 1   │ │ Node 2 │ │ Generation│         │
│  └─────────┘ └─────────┘ └─────────┘          │
└───────────────────┬─────────────────────────────┘
                    │
┌───────────────────▼─────────────────────────────┐
│          Model Repository / Cache                │
│    (S3, Hugging Face Hub, local NFS)           │
└───────────────────┬─────────────────────────────┘
                    │
┌───────────────────▼─────────────────────────────┐
│          GPU Infrastructure                     │
│    (Kubernetes + GPU Operators)                │
└─────────────────────────────────────────────────┘
```

### 3.1 推理服务器

选择合适的推理服务器是一个关键决策。以下是主要选项的比较：

| 特性 | vLLM | TensorRT-LLM | Text Generation Inference (TGI) | SGLang |
|------|------|---------------|----------------------------------|--------|
| **性能** | 优秀（PagedAttention） | 最佳（NVIDIA 优化） | 良好 | 优秀（RadixAttention） |
| **易用性** | 高 | 中（需要编译） | 高 | 中 |
| **模型支持** | 广泛 | NVIDIA GPU 专属 | 广泛 | 广泛 |
| **量化** | AWQ, GPTQ, INT4/8 | FP8, INT8, NVFP4 | AWQ, GPTQ | AWQ, INT4/8 |
| **动态批处理** | 是 | 是 | 是 | 是 |
| **KV 缓存** | PagedAttention | 分页 KV 缓存 | 是 | RadixAttention |
| **部署** | Python, Docker | Docker, Triton | Docker, Kubernetes | Python, Docker |
| **开源** | 是（Apache 2.0） | 是（部分） | 是（Apache 2.0） | 是（Apache 2.0） |

**建议**：
- **通用用例**：从 **vLLM** 开始（良好的性能、易用性、社区支持）。
- **最大性能（NVIDIA 专属）**：使用 **TensorRT-LLM**。
- **复杂工作流（-chain-of-thought、agent）**：考虑 **SGLang**。

### 3.2 模型格式和优化

为了获得最佳性能，您需要将模型转换为优化的格式：

| 格式 | 描述 | 工具 |
|------|------|------|
| **Hugging Face（PyTorch）** | 原始格式，未优化 | 无需转换 |
| **GGUF**（以前是 GGML） | 量化格式，CPU/GPU 推理 | `llama.cpp`, `convert.py` |
| **Safetensors** | 安全、快速的序列化格式 | `transformers` 库 |
| **TensorRT-LLM** | NVIDIA 优化的格式 | `trtllm-build` |
| **ONNX** | 跨平台推理格式 | `torch.onnx.export` |

**建议**：
- 对于 **vLLM**：使用 Hugging Face 格式（自动处理优化）。
- 对于 **TensorRT-LLM**：将模型转换为 TensorRT 格式（需要编译步骤）。
- 对于 **CPU 推理**：考虑 GGUF（量化）。

### 3.3 量化和精度

量化可以显著减少显存使用并提高吞吐量。

| 量化方案 | 精度 | 显存减少 | 质量下降 | 工具 |
|----------|------|----------|----------|------|
| FP16 | 16 位浮点 | 基线 | 无 | 原生支持 |
| FP8 | 8 位浮点 | ~15% | 可忽略不计 | TensorRT-LLM, vLLM |
| INT8 | 8 位整数 | ~40% | 微小（< 1%） | AWQ, TensorRT-LLM |
| INT4 | 4 位整数 | ~60% | 中等（1-3%） | GPTQ, AWQ |

**建议**：
- 从 **FP8** 或 **INT8** 开始（良好的平衡）。
- 如果质量下降可以接受，则使用 **INT4**（最大成本节省）。

## 4. 部署模式

### 4.1 裸机（Bare Metal）

在专用 GPU 服务器上部署 LLM。

**优点**：
- 最佳性能（无虚拟化开销）。
- 完全控制硬件和软件。

**缺点**：
- 高前期成本（资本支出）。
- 运营开销（维护、冷却、电力）。

**用例**：大规模、持续使用（例如，> 10,000 个 GPU）。

### 4.2 Kubernetes（K8s）

在 Kubernetes 集群上部署 LLM，使用 GPU 运营商（例如，NVIDIA GPU Operator、Kubecost）。

**优点**：
- 弹性扩缩容。
- 资源管理和调度。
- 与现有 CI/CD 管道集成。

**缺点**：
- 运营复杂性（需要 K8s 专业知识）。
- 虚拟化开销（轻微）。

**工具**：
- **NVIDIA GPU Operator**：在 K8s 上管理 GPU。
- **Karpenter**：自动节点扩缩容。
- **vLLM + K8s**：使用 Helm charts 部署 vLLM。

**用例**：云部署、混合云、需要弹性的工作负载。

### 4.3 无服务器（Serverless）

使用无服务器平台（例如，Modal、Replicate、Banana）部署 LLM。

**优点**：
- 零运营开销。
- 按使用量付费（无空闲成本）。
- 自动扩展。

**缺点**：
- 冷启动延迟。
- 对推理环境的控制较少。
- 对于持续的高流量可能更昂贵。

**用例**：流量波动的工作负载、概念验证、小型团队。

### 4.4 混合

结合多种模式（例如，用于稳态流量的本地部署 + 用于突发的云部署）。

**用例**：需要成本控制和弹性的大型企业。

## 5. 性能优化

### 5.1 连续批处理（Continuous Batching）

动态地对请求进行批处理，而不是等待固定的批处理大小。

**工具**：
- **vLLM**：本机支持连续批处理。
- **TensorRT-LLM**：支持 Inflight Batching（连续批处理的一种形式）。

**影响**：可以将吞吐量提高 2-5 倍。

### 5.2 KV 缓存优化

高效地管理 KV 缓存可以显著减少显存使用并提高吞吐量。

**技术**：
- **PagedAttention**（vLLM）：将 KV 缓存划分为"页面"，允许非连续存储。
- **RadixAttention**（SGLang）：跨请求共享公共前缀的 KV 缓存。
- **KV 缓存量化**：将 KV 缓存量化为 INT8 或 FP8。

### 5.3 推测解码（Speculative Decoding）

使用较小的"草稿"模型来提议 Token，然后由较大的模型进行验证。

**工具**：
- **EAGLE-3**：最先进的推测解码方法。
- **Medusa**：一种替代方法（使用额外的解码头）。

**影响**：可以将延迟降低 2-3 倍（对于延迟敏感型工作负载）。

### 5.4 预填充/解码分离（Disaggregated Prefill/Decode）

将 prefill 和 decode 阶段分离到不同的 GPU 上。

**工具**：
- **NVIDIA Dynamo**：用于分离部署的框架。
- **llm-d**：基于 Kubernetes 的分离部署。

**影响**：可以提高整体吞吐量 30-50%（通过专门优化每个阶段）。

## 6. 监控和可观测性

自托管需要强大的监控来确保性能和可靠性。

### 6.1 关键指标

| 类别 | 指标 | 描述 |
|------|------|------|
| **性能** | TTFT（首 Token 时间） | 第一个 Token 的延迟 |
| | TPOT（每输出 Token 时间） | 每个输出 Token 的延迟 |
| | ITL（Token 间延迟） | Token 之间的延迟 |
| | Goodput | 成功请求数/总请求数 |
| **资源** | GPU 利用率 | GPU 计算使用率 |
| | 显存使用率 | GPU 显存使用率 |
| | 网络吞吐量 | 数据传输速率 |
| **成本** | 每 Token 成本 | 云计算成本 / Token 数 |
| | 每请求成本 | 总成本 / 请求数 |

### 6.2 工具栈

| 工具 | 功能 |
|------|------|
| **Prometheus** | 指标收集 |
| **Grafana** | 指标可视化 |
| **Langfuse** | LLM 特定监控（追踪、评估） |
| **Arize AX** | AI 可观测性（漂移检测、根因分析） |
| **vLLM Prometheus Metrics** | vLLM 内置指标 |

**示例 vLLM 指标**：
- `vllm:num_requests_running`
- `vllm:gpu_cache_usage`
- `vllm:time_to_first_token`

## 7. 选择指南：逐步决策树

以下是一个决策树，可帮助您选择合适的自托管解决方案。

```
开始
│
├─ 流量是否稳定（> 100 个请求/秒）？
│  ├ 是 → 考虑本地部署或 Kubernetes
│  └ 否 → 考虑无服务器或云部署
│
├─ 延迟 SLA 是否 < 500ms TTFF？
│  ├ 是 → 使用 vLLM 或 TensorRT-LLM（连续批处理）
│  └ 否 → 可以使用 TGI 或无服务器
│
├─ 是否需要自定义模型？
│  ├ 是 → 自托管（vLLM、TensorRT-LLM）
│  └ 否 → 可以考虑托管 API
│
├─ 数据隐私是否关键？
│  ├ 是 → 自托管（本地或 VPC）
│  └ 否 → 托管 API 可能没问题
│
├─ 是否有 GPU 专业知识？
│  ├ 是 → 考虑 TensorRT-LLM（最大性能）
│  └ 否 → 使用 vLLM（易用性）
│
└─ 预算是否有限？
   ├ 是 → 使用量化（INT8/INT4）+  spot 实例
   └ 否 → 可以使用 FP16 + 按需实例
```

## 8. 推荐的技术栈

### 8.1 通用（Good Balance）

- **推理服务器**：vLLM
- **部署**：Kubernetes（使用 NVIDIA GPU Operator）
- **量化**：INT8（使用 AWQ）
- **监控**：Prometheus + Grafana + Langfuse

### 8.2 最大性能（NVIDIA 专属）

- **推理服务器**：TensorRT-LLM
- **部署**：Kubernetes 或裸机
- **量化**：FP8 或 INT8
- **监控**：Prometheus + NVIDIA DCGM

### 8.3 易用性（快速入门）

- **推理服务器**：vLLM（Docker 部署）
- **部署**：单个 GPU 服务器（Docker Compose）
- **量化**：FP16（无量化）
- **监控**：vLLM 内置指标

### 8.4 成本优化

- **推理服务器**：vLLM
- **部署**：Kubernetes + spot 实例（使用 Karpenter）
- **量化**：INT4（使用 GPTQ）
- **缓存**：提示缓存 + 语义缓存

## 9. 常见陷阱

1. **低估运营复杂性**：自托管需要持续的维护、监控和故障恢复。
2. **过度优化早期**：首先确保功能正常，然后再优化性能。
3. **忽略冷启动**：对于无服务器部署，冷启动可能会显著影响用户体验。
4. **不监控成本**：自托管仍然会产生成本（电力、冷却、人员）。跟踪它们。
5. **不考虑弹性**：流量可能会波动。确保您的部署可以扩展（或缩小）。

---

## 10. 总结

自托管 LLM 是一个强大的选择，适用于具有大规模、低延迟或严格数据隐私需求的应用程序。然而，它需要仔细的规划、技术专业知识和持续的运营。

**关键要点**：
1. **评估您的需求**：在承诺自托管之前，使用决策框架（第 2 节）。
2. **从小处着手**：从简单的部署（例如，单个 GPU 服务器上的 vLLM）开始，然后逐步扩展。
3. **优化迭代**：首先关注架构决策（路由、量化），然后关注推理优化（批处理、缓存）。
4. **监控一切**：建立强大的可观测性堆栈，以捕获性能、成本和可靠性。
5. **保持敏捷**：LLM 领域在不断发展。准备好根据新技术（例如，新推理服务器、量化方案）调整您的堆栈。

通过遵循本指南，您将能够很好地选择和实施满足您需求的自托管 LLM 解决方案。
