# SGLang 和 RadixAttention 用于前缀重工作负载

> SGLang 将 KV 缓存视为存储在基数树中的一等、可重用资源。在 vLLM 调度请求 FCFS（先到先服务）的地方，SGLang 的缓存感知调度器优先处理具有更长共享前缀的请求——有效地进行深度优先基数遍历，以便热分支保持在 HBM 中。在具有 ShareGPT 类 1K 提示的 Llama3.1 8B 上，SGLang 达到约 16,200 tok/s，而 vLLM 约 12,500，约 29% 的优势。在前缀重 RAG 工作负载上，优势达到 6.4 倍。在语音克隆类工作负载上，缓存命中率超过 86%。2026 年在 xAI、LinkedIn、Cursor、Oracle、GCP、Azure、AWS 的 400,000+ GPU 上部署。陷阱是当前缀排序不一致时，6.4 倍的数字会消失——排序是工程师的杠杆。

**类型：** 学习
**语言：** Python（标准库，简单的基数树缓存 + 缓存感知调度器）
**先修要求：** 阶段 17 · 04（vLLM 服务内部原理）、阶段 14（Agentic RAG）
**时间：** 约 75 分钟

## 学习目标

- 绘制 RadixAttention 图：前缀如何存储在基数树中，以及 KV 块如何在扎根于同一分支的序列之间共享。
- 解释缓存感知调度，以及为什么 FCFS 对于前缀重流量是错误的。
- 给定前缀缓存命中率和提示长度分布，计算工作负载的预期加速。
- 说出使 6.4 倍数字真实而不是失去优势的前缀排序规则。

## 问题

经典服务将每个请求的提示视为不透明的。即使 5,000 个 RAG 请求都以相同的 2,000 token 系统提示加上相同的检索前言开始，vLLM 也会将那 2,000 token 前缀 prefill 5,000 次。GPU 一遍又一遍地做同样的工作。

观察：agentic 和 RAG 工作负载中的提示几乎总是共享长前缀。系统提示、工具模式、少样本示例、检索头、对话历史——所有这些都跨请求重复。如果你为该前缀存储 KV 缓存一次并重用它，你就不会再次 prefill 它。

RadixAttention 正是这样做的。Token 被索引在基数树中；每个节点拥有其从根的路径上 token 序列的 KV 块。新请求遍历树：任何 token 匹配的节点都重用该节点的 KV 块。Prefill 成本与"新"后缀成比例，而不是完整提示。

挑战在于调度。如果两个请求共享 2,000 token 前缀，而第三个仅共享相同前缀的 200 个 token，你希望一起服务两个长共享请求，以便长前缀保持在 HBM 中。FCFS 做相反的事情——它服务先到达的任何人，可能在下一个长前缀请求到达之前驱逐热分支。

## 概念

### 作为 KV 索引的基数树

基数树（紧凑 trie）存储 token 序列。每个节点拥有一个 token 范围和为该范围计算的 KV 块。子节点扩展序列一个或更多 token。

```
root
 |- "你是一个有用的助手..."  (2,000 个 token，124 个 KV 块)
      |- "上下文：<doc A>..."        (500 个 token，31 个块)
           |- "问题：Alice..."    (80 个 token，5 个块)
           |- "问题：Bob..."      (95 个 token，6 个块)
      |- "上下文：<doc B>..."        (520 个 token，33 个块)
```

新请求带着系统提示 + "上下文：<doc A>" + "问题：Carol" 进入。调度器遍历：系统前缀匹配（124 个块重用），doc-A 分支匹配（31 个块重用），然后仅为 "问题：Carol" 分配新块（4 个块）。Prefill 成本：4 个块的新 token。没有树：160 个块。Prefill 节省约 40 倍。

### 缓存感知调度

如果缓存流失，基数树支持的重用是没有意义的。两个关键策略：

1. **深度优先分派**。从队列中选择下一个请求时，优先处理与当前运行集扎根于同一分支的请求。这保持热分支固定。
2. **分支级别的 LRU，而不是块级别**。驱逐整个分支（从最少使用的叶子开始），而不是单个块，因此缓存形状与基数形状匹配。

FCFS 违反两者。共享 2,000 个 token 的请求坐在共享 50 个的请求后面，然后 2,000 token 分支被驱逐以接纳 50 token 的那个。

### 你应该记住的基准数字

- Llama 3.1 8B、H100、ShareGPT 1K 提示：SGLang ~16,200 tok/s vs vLLM ~12,500（约 29% 优势）。
- 前缀重 RAG（相同系统 + 相同文档，变化的问题）：SGLang 上高达 6.4 倍。
- 语音克隆工作负载：86.4% 前缀缓存命中率。
- SGLang 客户的生产命中率：50-99%，取决于提示规则。
- 2026 年部署在 400,000+ GPU 上。

### 排序陷阱

6.4 倍数字依赖于一致的提示模板排序。如果你的客户端在某些请求中构造提示为 `[system, tools, context, history, question]`，在其他请求中构造为 `[system, context, tools, history, question]`，树无法找到共享前缀。对人类来说看起来像共享前缀的东西，对基数树来说是两个不同的序列。

工程师的杠杆：你的提示模板是缓存键。修复顺序。将一切不可变的（系统、工具、模式）放在第一位。接下来放检索上下文。最后放用户问题。不要将动态内容交织到前缀中。

研究中的真实案例：将动态内容移出可缓存前缀，在一次更改中将一个部署的缓存命中率从 7% 提高到 74%。

### RadixAttention 胜出和失败的地方

胜出：
- RAG（相同的检索前言，变化的问题）。
- Agent（相同的工具模式，变化的查询）。
- 带有长系统提示的聊天。
- 带有重复前言的语音 / 视觉工作负载。

失败（返回到 vLLM 级吞吐量）：
- 具有唯一提示的一次性生成（代码完成、没有系统提示的开放式聊天）。
- 每个请求将唯一内容交织到前缀中的动态提示。

### 为什么这是调度器问题，而不仅仅是内核问题

你可以将 KV 重用实现为内核技巧。SGLang 的洞察是，只有当调度器保持热分支驻留时，重用才有回报。朴素的"如果可用则重用"策略将在混合负载下流失缓存。基数树索引的调度器就是将内核技巧转化为 29% 生产优势的东西。

### 与 vLLM 的相互作用

这两个系统不是严格的竞争者。在 2026 年，vLLM 添加了前缀缓存（`--enable-prefix-caching`）和缓存感知路由器（Rust 中的 vLLM Router）。差距缩小但没有完全消失——SGLang 的整个栈是基数优先的；vLLM 嫁接了它。对于由前缀重用主导的工作负载，SGLang 仍然是默认选择。对于没有强前缀模式的通用服务，vLLM 仍然相等或更好。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| RadixAttention | "SGLang 的东西" | KV 缓存索引为基数树，以便共享前缀重用块 |
| Radix tree | "紧凑 trie" | 每个节点拥有一个 token 范围及其 KV 块的树 |
| Cache-aware scheduler | "热分支优先" | 优先处理共享驻留分支的请求调度器 |
| Prefix-cache hit rate | "你的提示有多少是免费的" | 从重用的 KV 块提供服务的提示 token 分数 |
| FCFS | "先到先服务" | 破坏前缀局部性的默认调度 |
| Branch-level LRU | "驱逐叶子" | 匹配基数形状的驱逐策略 |
| Prompt template ordering | "缓存键" | 提示的组件顺序决定树可以共享什么 |
| System prompt pinning | "驻留前缀" | 保持不可变系统部分固定以避免驱逐抖动 |
