# 06 · 面向前缀密集型负载的 SGLang 与 RadixAttention

> SGLang 把 KV 缓存当作一等的、可复用的资源，存储在一棵基数树（radix tree）中。vLLM 采用 FCFS（先到先服务，first-come, first-served）调度请求，而 SGLang 的缓存感知（cache-aware）调度器则优先处理共享前缀更长的请求——本质上是一次深度优先的基数树遍历，从而让热点分支常驻于 HBM。在 Llama 3.1 8B 上、使用类 ShareGPT 的 1K 提示词时，SGLang 达到约 16,200 tok/s，而 vLLM 约为 12,500，领先约 29%。在前缀密集型的 RAG 负载上，优势可达 6.4 倍。在语音克隆（voice-cloning）形态的负载上，缓存命中率超过 86%。2026 年已部署于 xAI、LinkedIn、Cursor、Oracle、GCP、Azure、AWS 等处合计 400,000+ 张 GPU。需要注意的坑在于：一旦前缀顺序不一致，6.4 倍的数字就会蒸发——顺序正是工程师手中的杠杆。

**类型：** 学习
**语言：** Python（标准库，玩具级基数树缓存 + 缓存感知调度器）
**前置：** 阶段 17 · 04（vLLM 服务内部机制）、阶段 14（智能体式 RAG）
**时长：** 约 75 分钟

## 学习目标

- 画出 RadixAttention 的结构：前缀如何存储在基数树中，以及 KV 块（KV blocks）如何在同一分支下的多个序列间共享。
- 解释缓存感知调度，以及为什么 FCFS 不适合前缀密集型流量。
- 在给定前缀缓存命中率与提示词长度分布的情况下，计算某负载的预期加速比。
- 说出能让 6.4 倍数字成真（而非沦为损失掉的潜在收益）的提示词排序规范。

## 问题所在

经典的服务方式把每个请求的提示词当作不透明的整体。即使 5,000 个 RAG 请求全都以同一段 2,000 个 token 的系统提示词加同一段检索前言开头，vLLM 也会把这段 2,000 token 的前缀预填充（prefill）5,000 次。GPU 一遍又一遍地做同样的工作。

关键观察是：智能体式负载与 RAG 负载中的提示词几乎总是共享很长的前缀。系统提示词、工具 schema、少样本示例、检索头部、对话历史——这些都会在请求之间重复。如果你只为该前缀存储一次 KV 缓存并复用它，就无需再次预填充它。

RadixAttention 做的正是这件事。token 被索引在一棵基数树中；每个节点拥有其从根到该节点路径上的 token 序列所对应的 KV 块。新请求会遍历这棵树：任何 token 能匹配上的节点都会复用该节点的 KV 块。预填充成本因此只与「新增」后缀成正比，而非整段提示词。

挑战在于调度。如果两个请求共享一段 2,000 token 的前缀，而第三个请求只共享该前缀中的 200 个 token，你会希望把那两个共享长前缀的请求放在一起服务，好让长前缀常驻 HBM。FCFS 的做法恰好相反——它服务最先到达者，可能会在下一个长前缀请求到来之前就把热点分支驱逐出去。

## 核心概念

### 把基数树当作 KV 索引

基数树（紧凑前缀树，compact trie）存储 token 序列。每个节点拥有一段 token 范围，以及为该范围计算出的 KV 块。子节点把序列向后延伸一个或多个 token。

```
root
 |- "You are a helpful assistant..."  (2,000 tokens, 124 KV blocks)
      |- "Context: <doc A>..."        (500 tokens, 31 blocks)
           |- "Question: Alice..."    (80 tokens, 5 blocks)
           |- "Question: Bob..."      (95 tokens, 6 blocks)
      |- "Context: <doc B>..."        (520 tokens, 33 blocks)
```

一个新请求带着 系统提示词 + "Context: <doc A>" + "Question: Carol" 进来。调度器开始遍历：系统前缀匹配（复用 124 块），doc-A 分支匹配（复用 31 块），然后只为 "Question: Carol" 分配全新的块（4 块）。预填充成本：4 块新 token。若没有这棵树：160 块。预填充上约 40 倍的节省。

### 缓存感知调度

如果缓存频繁翻搅，基于基数树的复用就毫无意义。两条关键策略：

1. **深度优先派发（Depth-first dispatch）**。从队列中挑选下一个请求时，优先选择与当前运行集挂在同一分支下的请求。这能把热点分支钉住。
2. **分支级 LRU，而非块级 LRU**。以整条分支为单位驱逐（从使用时间最短的叶子开始），而不是驱逐单个块，从而让缓存形态与基数树形态保持一致。

FCFS 同时违反这两条。一个共享 2,000 token 的请求排在一个只共享 50 token 的请求后面，随后那条 2,000 token 的分支为了接纳那个 50 token 的请求而被驱逐。

### 你应当记住的基准数字

- Llama 3.1 8B、H100、ShareGPT 1K 提示词：SGLang 约 16,200 tok/s vs vLLM 约 12,500（领先约 29%）。
- 前缀密集型 RAG（相同系统提示词 + 相同文档，仅问题不同）：SGLang 最高可达 6.4 倍。
- 语音克隆负载：86.4% 的前缀缓存命中率。
- SGLang 客户的生产环境命中率：视提示词规范程度而定，介于 50%-99% 之间。
- 2026 年已部署于 400,000+ 张 GPU。

### 顺序之坑

6.4 倍这个数字依赖于一致的提示词模板顺序。如果你的客户端在某些请求中把提示词构造为 `[system, tools, context, history, question]`，在另一些请求中又构造为 `[system, context, tools, history, question]`，那么这棵树就找不到共享前缀。在人眼看来是共享前缀的东西，对基数树而言却是两条不同的序列。

工程师手中的杠杆：你的提示词模板就是一个缓存键（cache key）。固定好顺序。把所有不变的内容（系统提示词、工具、schema）放在最前。把检索上下文放在其后。把用户问题放在最后。不要把动态内容穿插进前缀里。

来自研究的真实案例：把动态内容移出可缓存前缀，仅此一项改动就让某次部署的缓存命中率从 7% 提升到 74%。

### RadixAttention 在哪里取胜、在哪里失利

取胜：

- RAG（相同的检索前言，仅问题不同）。
- 智能体（相同的工具 schema，仅查询不同）。
- 带长系统提示词的对话。
- 带重复前言的语音 / 视觉负载。

失利（回落到 vLLM 级别的吞吐）：

- 提示词各不相同的单次生成（代码补全、无系统提示词的开放式对话）。
- 每个请求都把独特内容穿插进前缀的动态提示词。

### 为什么这是调度问题，而不仅仅是 kernel 问题

你可以把 KV 复用实现为一个 kernel 技巧。SGLang 的洞见在于：只有当调度器让热点分支保持常驻时，复用才划算。在混合负载下，一个朴素的「有就复用」策略会把缓存翻搅一空。正是这个以基数树为索引的调度器，把 kernel 技巧变成了 29% 的生产环境优势。

### 与 vLLM 的相互作用

这两个系统并非严格的竞争对手。2026 年，vLLM 加入了前缀缓存（`--enable-prefix-caching`）以及一个缓存感知路由器（用 Rust 写的 vLLM Router）。差距缩小了，但并未完全消失——SGLang 的整个技术栈都是基数树优先的；而 vLLM 是后来嫁接上去的。对于以前缀复用为主的负载，SGLang 仍是默认选择。对于没有强前缀模式的通用服务，vLLM 仍然相当甚至更优。

## 动手用

`code/main.py` 实现了一个玩具级的基数树 KV 缓存，外加一个带两种策略的调度器：FCFS 与缓存感知。它让同一份负载分别跑过两种策略，报告前缀缓存命中率与吞吐差值。然后再跑一份「打乱顺序」的负载，展示 6.4 倍是如何崩塌的。

## 交付物

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| RadixAttention | 「SGLang 那玩意儿」 | 把 KV 缓存索引为一棵基数树，从而让共享前缀复用块 |
| 基数树（Radix tree） | 「紧凑前缀树」 | 每个节点拥有一段 token 范围及其 KV 块的树 |
| 缓存感知调度器 | 「热点分支优先」 | 优先选择共享常驻分支的请求的调度器 |
| 前缀缓存命中率 | 「你的提示词有多少是免费的」 | 由复用的 KV 块所服务的提示词 token 占比 |
| FCFS | 「先到先服务」 | 破坏前缀局部性的默认调度方式 |
| 分支级 LRU | 「驱逐叶子」 | 与基数树形态相匹配的驱逐策略 |
| 提示词模板顺序 | 「缓存键」 | 提示词各组成部分的顺序决定了树能共享什么 |
| 系统提示词钉固 | 「常驻前缀」 | 把不变的系统部分钉住，以避免驱逐翻搅 |
