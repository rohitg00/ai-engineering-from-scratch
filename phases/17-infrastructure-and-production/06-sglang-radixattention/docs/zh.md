# SGLang和RadixAttention用于前缀重工作负载

> SGLang将KV缓存视为存储在基数树中的头等可重用资源。vLLM按FCFS（先来先服务）调度请求，SGLang的缓存感知调度器优先处理具有更长共享前缀的请求 —— 实际上是深度优先基数遍历，使热分支驻留在HBM中。在Llama 3.1 8B上，使用类似ShareGPT的1K提示，SGLang达到约16,200 tok/s，vLLM约12,500，约29%优势。在前缀重RAG工作负载上，优势达到6.4倍。在语音克隆形状工作负载上，缓存命中率超过86%。2026年在xAI、LinkedIn、Cursor、Oracle、GCP、Azure、AWS的400,000+ GPU上部署。陷阱是，当前缀排序不一致时，6.4倍数字会消失 —— 排序是工程师的杠杆。

## 学习目标

- 绘制RadixAttention：前缀如何存储在基数树中，以及KV块如何跨根植于同一分支的序列共享。
- 解释缓存感知调度，以及为什么FCFS对前缀重流量是错误的。
- 给定前缀缓存命中率和提示长度分布，计算工作负载的预期加速。
- 命名使6.4倍数字真实 vs 失去优势的提示排序规则。

## 问题

经典服务将每个请求的提示视为不透明。即使5,000个RAG请求都以相同的2,000-token系统提示加相同检索前言开头，vLLM预填充该2,000-token前缀5,000次。GPU一遍又一遍地做相同工作。

观察：智能体和RAG工作负载中的提示几乎总是共享长前缀。系统提示、工具模式、少样本示例、检索头、对话历史 —— 全部跨请求重复。如果你存储该前缀的KV缓存一次并重用，你不会再预填充它。

RadixAttention正是这样做的。Token在基数树中索引；每个节点拥有其从根路径上token序列的KV块。新请求遍历树：任何token匹配的节点重用该节点的KV块。预填充成本与"新"后缀成比例，不是完整提示。

挑战是调度。如果两个请求共享2,000-token前缀，第三个仅共享同一前缀的200 token，你希望一起服务两个长共享请求，使长前缀留在HBM中。FCFS做相反的事 —— 它服务先到者，可能在下一个长前缀请求命中前驱逐热分支。

## 概念

### 基数树作为KV索引

基数树（紧凑trie）存储token序列。每个节点拥有一个token范围和为该范围计算的KV块。子节点扩展序列一个或多个token。

```
root
 |- "You are a helpful assistant..."  (2,000 token, 124 KV块)
      |- "Context: <doc A>..."        (500 token, 31块)
           |- "Question: Alice..."    (80 token, 5块)
           |- "Question: Bob..."      (95 token, 6块)
      |- "Context: <doc B>..."        (520 token, 33块)
```

新请求以系统提示 + "Context: <doc A>" + "Question: Carol"进入。调度器遍历：系统前缀匹配（124块重用），doc-A分支匹配（31块重用），然后仅为"Question: Carol"分配新块（4块）。预填充成本：4块新token。没有树：160块。预填充上约40倍节省。

### 缓存感知调度

基数树支持的复用如果缓存搅动就无意义。两个关键策略：

1. **深度优先分派**。从队列中选择下一个请求时，优先选择与当前运行集根植于同一分支的请求。这使热分支固定。
2. **分支级LRU，不是块级**。驱逐整个分支（从最短使用的叶子开始），而不是单个块，因此缓存形状匹配基数形状。

FCFS违反两者。共享2,000 token的请求坐在共享50的请求后面，然后2,000-token分支被驱逐以接纳50-token分支。

### 你应该记住的基准数字

- Llama 3.1 8B、H100、ShareGPT 1K提示：SGLang约16,200 tok/s vs vLLM约12,500（约29%优势）。
- 前缀重RAG（相同系统 + 相同文档，变化问题）：SGLang上高达6.4倍。
- 语音克隆工作负载：86.4%前缀缓存命中率。
- SGLang客户间的生产命中率：50-99%，取决于提示规则。
- 2026年在400,000+ GPU上部署。

### 排序陷阱

6.4倍数字依赖于一致的提示模板排序。如果你的客户端在一些请求中将提示构造为`[system, tools, context, history, question]`，在其他请求中为`[system, context, tools, history, question]`，树无法找到共享前缀。对人类看起来像共享前缀的东西，对基数树是两个不同序列。

工程师的杠杆：你的提示模板是缓存键。固定顺序。将所有不可变内容（系统、工具、模式）放在前面。将检索上下文放在其次。将用户问题放在最后。不要将动态内容交错到前缀中。

研究中的真实案例：将动态内容移出可缓存前缀使一次部署的缓存命中率从7%到74%，一次更改。

### RadixAttention在哪里赢和输

赢：
- RAG（相同检索前言，变化问题）。
- 智能体（相同工具模式，变化查询）。
- 长系统提示的聊天。
- 重复前言的语音/视觉工作负载。

输（返回vLLM级吞吐量）：
- 独特提示的单次生成（代码补全，无系统提示的开放式聊天）。
- 每个请求将独特内容交错到前缀中的动态提示。

### 为什么这是调度器问题，不仅仅是内核问题

你可以将KV复用实现为内核技巧。SGLang的洞察是，复用只有在调度器保持热分支驻留时才回报。在混合负载下，朴素的"如果可用则复用"策略会搅动缓存。基数树索引调度器是将内核技巧转化为29%生产优势的要素。

### 与vLLM的交互

两个系统不是严格竞争者。2026年vLLM添加了前缀缓存（`--enable-prefix-caching`）和缓存感知路由器（Rust中的vLLM Router）。差距缩小但未完全消失 —— SGLang的整个栈是基数优先；vLLM嫁接它。对于以前缀复用为主导的工作负载，SGLang保持默认。对于没有强前缀模式的通用服务，vLLM保持相等或更好。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| RadixAttention | "SGLang的东西" | KV缓存索引为基数树，使共享前缀重用块 |
| 基数树 | "紧凑trie" | 每个节点拥有token范围及其KV块的树 |
| 缓存感知调度器 | "热分支优先" | 优先处理共享驻留分支的请求的调度器 |
| 前缀缓存命中率 | "多少提示是免费的" | 从重用KV块服务的提示token比例 |
| FCFS | "先来先服务" | 破坏前缀局部性的默认调度 |
| 分支级LRU | "驱逐叶子" | 匹配基数形状的驱逐策略 |
| 提示模板排序 | "缓存键" | 提示的组件顺序决定树可以共享什么 |
| 系统提示固定 | "驻留前缀" | 保持不可变系统部分固定以避免驱逐搅动 |
