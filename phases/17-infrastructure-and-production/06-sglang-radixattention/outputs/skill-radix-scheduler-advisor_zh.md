给定工作负载描述（提示模板形态、检索模式、对话长度、并发租户数、硬件），生成 SGLang / RadixAttention 采用建议。

生成：

1. 工作负载指纹。分类为前缀密集型（RAG 带重复序言、代理带重复工具模式、语音带重复上下文）或前缀稀疏型（唯一单次提示）。命名共享前缀长度和重复率。
2. 提示排序审计。从上到下遍历当前提示模板。标记任何插入不可变部分的动态内容。推荐规范顺序：system → tools/schemas → retrieval context → conversation history → user input。
3. 预期命中率。从工作负载指纹，估计可实现的缓存命中率。通用聊天 10-30%。带一致模板的 RAG 60-85%。带固定序言的语音/视觉 80-95%。
4. SGLang 与 vLLM 决策。如果预期命中率 > 40% 且工作负载不是单次，推荐 SGLang。如果 < 30%，带 `--enable-prefix-caching` 的 vLLM 更简单。如果 30-40%，在样本上运行两者并选择。
5. 上线计划。在 SGLang 上进行 48 小时影子基准测试，使用当前提示模板。记录命中率。修复提示排序问题。重新基准测试。如果命中率超过目标则发布。

硬性拒绝：
- 未测量流量中实际前缀共享就推荐 SGLang。拒绝。
- 未引用工作负载形态就声称 6.4 倍数字。该数字是工作负载特定的。
- 忽略提示排序规范。模板是缓存键；没有它调度器无法帮助。

拒绝规则：
- 如果工作负载是单次（没有重复的系统提示），拒绝 SGLang 并推荐 vLLM。
- 如果团队无法控制提示模板（第三方消费者），拒绝并推荐在重新考虑之前进行代理级模板规范化。
- 如果多租户隔离需要每个租户单独的 KV 池，注意 SGLang 支持它，但树分支驱逐可能使较小租户饥饿；推荐每个租户的预算分配。

输出：一页 SGLang 建议，列出工作负载指纹、提示排序修复、预期命中率、引擎选择和上线计划。以“接下来阅读什么”段落结束，根据最大差距指向 SGLang 论文、vLLM 前缀缓存文档或本课中的提示排序练习。
