# 影子流量、金丝雀发布和 LLM 渐进式部署

> LLM 发布结合了软件部署中最困难的部分：无单元测试、分散的失败模式、延迟的信号。序列是 (1) 影子模式——复制生产请求到候选模型，记录，与零用户影响进行比较；捕获明显的分发问题，但不是质量保证；(2) 金丝雀发布——渐进式流量转移 10% → 25% → 50% → 75% → 100%，每个步骤都有门控；跟踪延迟百分位数、成本/请求、错误/拒绝率、输出长度分布、用户反馈率；(3) 在稳定性确认后进行 A/B 测试以进行明确的替代方案。非确定性是不可简化的——由于 GPU FP 非结合性加上批量大小方差，在相同输入下运行时，准确率变化高达 15%。成本是一个变量，不是常数——一个好 20% 的模型每次调用可能贵 3 倍。回滚速度具有决定性意义：如果回滚需要重新部署，你就太慢了。策略存在于配置/标志中；模型存在于带有固定摘要的注册表中；回滚 = 翻转策略 + 还原阈值 + 在几秒钟内固定旧模型。

**类型：** 学习
**语言：** Python（标准库，简单的金丝雀进度模拟器）
**先修要求：** 阶段 17 · 13（可观测性）、阶段 17 · 21（A/B 测试）
**时间：** 约 60 分钟

## 学习目标：

- 区分影子模式（零影响比较）、金丝雀（实时流量渐进式）和 A/B（稳定性确认后的比较）。
- 列举五个 LLM 特定的金丝雀指标（延迟、成本/请求、错误/拒绝、输出长度分布、用户反馈）。
- 解释 LLM 非确定性（高达 15%）如何改变发布中"稳定"的含义。
- 设计一种需要几秒钟（策略翻转）而不是几小时（重新部署）的回滚路径。

## 问题：

你发布了一个新模型。离线评估显示准确率提升 3%。你在生产中启用它。24 小时内，成本上升 40%，用户点踩率上升 8%，三张客户罚单报告"奇怪的答案。" 你回滚。重新部署需要 3 小时。你的周末被毁了。

这一切都是可以避免的。影子模式会在任何用户看到之前捕获 40% 的成本激增。金丝雀会在点踩移动时停在 10%。策略延迟回滚需要 30 秒。纪律填补了"离线评估看起来不错"和"真实用户满意"之间的空白。

## 概念：

### 影子模式：

候选者接收与生产相同的请求；输出被记录，不返回给用户。零用户影响。日志：
- 输出内容（与生产的差异）。
- Token 计数（成本增量）。
- 延迟。
- 拒绝和错误。

捕获：成本激增、长度回归、明显的拒绝变化、硬错误。不捕获：用户会感知到的质量增量。影子是一个冒烟测试，不是质量测试。

### 金丝雀发布：

带有门控的渐进式流量转移。典型进度：1% → 10% → 25% → 50% → 75% → 100%。每个步骤的 5 个指标门控：
1. **延迟百分位数**——P50、P95、P99。违反：金丝雀的 P99 > 基线的 1.5 倍。
2. **每次请求的成本**——混合美元。违反：> 基线 20%。
3. **错误 / 拒绝率**——5xx 加上明确的拒绝。违反：基线的 2 倍。
4. **输出长度分布**——平均值 + P99。违反：分布转移。
5. **用户反馈率**——点踩 / 罚单提交。违反：基线的 1.5 倍。

### 非确定性是新的方差：

相同输入产生非相同输出。原因：
- GPU FP 非结合性（浮点归约顺序因批次而异）。
- 批量大小方差（在 128 的批次 vs 16 的批次中相同的提示）。
- 采样（temperature > 0）。

测量：在相同评估集上，运行到运行的准确率变化高达 15%。发布中的"稳定"意味着指标在预期方差范围内，不是与基线相同。在噪声底线之上设置门控。

### 成本是一个变量：

好 20% 的模型每次调用可能贵 3 倍。每次请求的成本是五个门控之一。发布一个"更好"但打破单位经济学的模型是一个回滚案例。

### 回滚是武器：

- 策略标志（功能标志系统）：在配置中翻转百分比；需要几秒钟。
- 模型固定（注册表摘要）：固定模型不会自动升级。
- 回滚 = 还原标志 + 将固定摘要设置为之前的状态。几秒钟，不是几小时。

如果你的栈需要重新部署来回滚，请在推出之前修复它。

### 工具：

**Argo Rollouts** / **Flagger**——Kubernetes 渐进式交付控制器。与 Istio/Linkerd 加权路由集成。

**Istio 加权路由**——服务网格级流量分割。

**KServe / Seldon Core**——具有内置金丝雀的模型服务。

**功能标志**——LaunchDarkly、Flagsmith、Unleash。策略级翻转，无需重新部署。

### 指标节奏：

金丝雀门控根据流量量每 5-15 分钟检查一次。1% 流量，10 个请求/分钟，每个窗口给出 50-150 个数据点——足以进行延迟，但用户反馈嘈杂。10% 给出约 10 倍更多。进度应在每个步骤停留足够长的时间以积累足够的样本。

### A/B 步骤是可选的：

如果新模型明显不同（不同的行为、不同的成本曲线、不同的基调），在金丝雀通过后在 50% 下进行 A/B 测试。如果它只是一个改进版本，在金丝雀门控通过时跳到 100%。

### 你应该记住的数字：

- 金丝雀进度：1% → 10% → 25% → 50% → 75% → 100%。
- 非确定性上限：在相同输入上，运行到运行的方差高达 15%。
- 五个金丝雀指标：延迟、成本、错误/拒绝、输出长度、用户反馈。
- 成本门控：> 基线 20% 是违反。
- 回滚：几秒钟，不是几小时。

## 关键术语：

| 术语 | 人们的说法 | 实际含义 |
|------|----------------|------------------------|
| Shadow mode | "复制到新" | 零影响发送到候选者以进行日志记录 |
| Canary | "渐进式流量" | 带有门控的渐进式用户暴露发布 |
| Gates | "发布检查" | 阻止进度的指标阈值 |
| Non-determinism | "LLM 方差" | 运行到运行的不可约差异 |
| Policy flag | "标志翻转回滚" | 配置级回滚，几秒钟而不是几小时 |
| Model pin | "注册表摘要" | 对模型版本的不可变引用 |
| Argo Rollouts | "K8s 渐进式" | Kubernetes 原生金丝雀/回滚控制器 |
| KServe | "推理 K8s" | 具有金丝雀原语的模型服务 |
| Istio weighted | "网格分割" | 服务网格流量分割器 |
