# 20 · LLM 的影子流量、金丝雀发布与渐进式部署

> LLM 上线集成了软件部署中最棘手的几个难点：没有单元测试、失败模式弥散、信号延迟。其推进顺序为：（1）影子模式（shadow mode）——把生产请求复制一份发给候选模型，记录日志、做对比，对用户零影响；它能抓出明显的分布问题，但不构成质量保证；（2）金丝雀发布（canary rollout）——以 10% → 25% → 50% → 75% → 100% 渐进切换流量，每一步都设有闸门；跟踪延迟分位数、单次请求成本、错误率/拒答率、输出长度分布、用户反馈率；（3）在稳定性确认之后，对显著不同的备选方案做 A/B 测试。非确定性（non-determinism）不可消除——由于 GPU 浮点运算不满足结合律，再叠加批大小（batch size）方差，相同输入在多次运行间的准确率波动可达 15%。成本是变量而非常量——一个好 20% 的模型，单次调用成本可能贵 3 倍。回滚速度是决定性因素：如果回滚需要重新部署，那就太慢了。策略（policy）放在配置/开关里；模型放在带固定摘要（digest）的注册表里；回滚 = 翻转策略 + 还原阈值 + 把旧模型钉住，几秒内完成。

**类型：** 学习
**语言：** Python（标准库，玩具级金丝雀推进模拟器）
**前置：** 阶段 17 · 13（可观测性）、阶段 17 · 21（A/B 测试）
**时长：** 约 60 分钟

## 学习目标

- 区分影子模式（零影响对比）、金丝雀（线上流量渐进）与 A/B（稳定性确认后的对比）。
- 列举五个 LLM 专属的金丝雀指标（延迟、单次请求成本、错误率/拒答率、输出长度分布、用户反馈）。
- 解释为什么 LLM 的非确定性（最高 15%）会改变发布中「稳定」的定义。
- 设计一条以秒计（翻转策略）而非以小时计（重新部署）的回滚路径。

## 问题所在

你上线了一个新模型。离线评测显示准确率提升 3%。你在生产环境把它打开。24 小时内，成本上涨 40%，用户点踩上升 8%，三张客户工单报告「答案很怪」。你回滚。重新部署耗时 3 小时。你的周末被毁了。

这一切本都可以避免。影子模式本能在任何用户看到之前就抓出那 40% 的成本飙升。金丝雀本能在点踩异动时停在 10%。基于策略开关（policy flag）的回滚本只需 30 秒。这套纪律，填补的正是「离线评测看起来不错」与「真实用户满意」之间的鸿沟。

## 核心概念

### 影子模式

候选模型收到与生产环境相同的请求；其输出被记录而非返回给用户。对用户零影响。记录：

- 输出内容（与生产环境做差异对比）。
- token 计数（成本差值）。
- 延迟。
- 拒答与错误。

能抓住：成本爆炸、长度回退、明显的拒答变化、硬性错误。抓不住：用户能感知的质量差异。影子模式是冒烟测试（smoke test），不是质量测试。

### 金丝雀发布

带闸门的渐进式流量切换。典型推进：1% → 10% → 25% → 50% → 75% → 100%。每一步对 5 个指标设闸门：

1. **延迟分位数** —— P50、P95、P99。越界：金丝雀的 P99 > 基线的 1.5 倍。
2. **单次请求成本** —— 综合美元成本。越界：高于基线 20% 以上。
3. **错误率 / 拒答率** —— 5xx 加上显式拒答。越界：基线的 2 倍。
4. **输出长度分布** —— 均值 + P99。越界：分布发生偏移。
5. **用户反馈率** —— 点踩 / 工单提交。越界：基线的 1.5 倍。

### 非确定性就是新的方差

相同的输入会产生不相同的输出。原因：

- GPU 浮点运算不满足结合律（浮点归约的顺序随批次而变）。
- 批大小方差（同一个提示词处在 128 的批中还是 16 的批中）。
- 采样（temperature > 0）。

实测：在相同的评测集上，多次运行间准确率波动可达 15%。发布中的「稳定」意味着指标落在预期方差之内，而非与基线完全一致。把闸门设在噪声底（noise floor）之上。

### 成本是个变量

一个好 20% 的模型，单次调用成本可能贵 3 倍。单次请求成本是五个闸门之一。上线一个「更好」却破坏单位经济性的模型，属于应当回滚的情形。

### 回滚是武器

- 策略开关（功能开关系统）：在配置中翻转流量百分比；以秒计。
- 模型钉固（注册表摘要）：被钉住的模型不会自动升级。
- 回滚 = 还原开关 + 把固定摘要设回上一版本。是几秒，不是几小时。

如果你的技术栈必须重新部署才能回滚，那就在发布之前先修好这一点。

### 工具

**Argo Rollouts** / **Flagger** —— Kubernetes 渐进式交付控制器。与 Istio/Linkerd 的加权路由集成。

**Istio 加权路由** —— 服务网格（service mesh）层面的流量切分。

**KServe / Seldon Core** —— 内置金丝雀能力的模型服务。

**功能开关** —— LaunchDarkly、Flagsmith、Unleash。策略级翻转，无需重新部署。

### 指标采集节奏

金丝雀闸门视流量规模每 5–15 分钟检查一次。1% 的流量、每分钟 10 个请求，每个窗口给出 50–150 个数据点——对延迟够用，但对用户反馈而言噪声偏大。10% 的流量则多出约 10 倍。推进应在每一步停留足够久，以累积足够多的样本。

### A/B 这一步是可选的

如果新模型显著不同（行为不同、成本曲线不同、语气不同），就在金丝雀通过后于 50% 处做 A/B 测试。如果它只是一个改进版，那就在金丝雀闸门通过后直接跳到 100%。

### 你该记住的数字

- 金丝雀推进：1% → 10% → 25% → 50% → 75% → 100%。
- 非确定性上限：相同输入下多次运行间方差最高 15%。
- 五个金丝雀指标：延迟、成本、错误率/拒答率、输出长度、用户反馈。
- 成本闸门：高于基线 20% 以上即为越界。
- 回滚：以秒计，而非以小时计。

## 关键术语

| 术语 | 人们怎么说 | 它实际指什么 |
|------|----------------|------------------------|
| 影子模式（Shadow mode） | 「复制给新模型」 | 零影响地把请求发给候选模型以供记录 |
| 金丝雀（Canary） | 「渐进流量」 | 带闸门、逐步向用户暴露的发布 |
| 闸门（Gates） | 「发布检查」 | 阻止推进的指标阈值 |
| 非确定性（Non-determinism） | 「LLM 方差」 | 不可消除的多次运行间差异 |
| 策略开关（Policy flag） | 「翻开关回滚」 | 配置级回滚，以秒计而非以小时计 |
| 模型钉固（Model pin） | 「注册表摘要」 | 对某个模型版本的不可变引用 |
| Argo Rollouts | 「K8s 渐进」 | Kubernetes 原生的金丝雀/回滚控制器 |
| KServe | 「推理 K8s」 | 带金丝雀原语的模型服务 |
| Istio 加权（Istio weighted） | 「网格切分」 | 服务网格流量切分器 |
