给定模型大小、SLA（TTFT P99）、流量形态（稳定与突发）和预算态势，生成冷启动缓解计划。

生成：

1. 冷启动预算。分解原始冷启动路径（节点供应、镜像拉取、权重到 HBM、引擎初始化、首次前向）。使用 2026 年所述模型大小的标称秒数。
2. 层选择。选择将总数降至 SLA 以下的最少数量的层：预置镜像（L1）、模型流式加载器（L2）、GPU 快照（L3）、温池（L4）、分层加载（L5）。针对每个层攻击的具体阶段证明其合理性。
3. 温池大小。为主路径声明 `min_workers`。如果 SLA 是 TTFT P99 < 60s 的 70B+ 模型，无论成本如何都强制使用温池。
4. 成本估算。所选温池的月度 GPU 成本和每天预期的冷启动次数。
5. 尾部策略。新鲜副本上的第一个用户会发生什么——他们是排队到温副本，还是支付冷启动税？命名具体策略（例如，“在 10 秒内将第一个请求路由到任何温副本；回退到冷副本”）。
6. 失败模式。如果温副本在会话中死亡会发生什么。恢复是自动的（实时迁移），还是下一个请求是冷启动？

硬性拒绝：
- 未计算月度成本就提出“只需添加温池”。
- 声称缓解措施而不说明其攻击的具体阶段（例如，“使用 Bottlerocket”而不说明它消除了 180 秒镜像拉取）。
- 忽略 GPU 快照的每 GPU 拓扑约束——如果平台迁移 SKU，快照无效。

拒绝规则：
- 如果 SLA 是 TTFT P99 < 5s，在 2026 年基础设施速度下没有温池的新鲜 70B 冷启动，拒绝——数学上不可能。
- 如果预算禁止温池但 SLA 需要 sub-30s 冷启动，命名平台特定修复（Modal GPU 快照、Baseten 预热）并拒绝在没有它的情况下在不同平台上承诺 SLA。
- 如果运维人员要求对突发流量的 70B 模型进行缩放到零，拒绝承诺 SLA——没有快照或温池数学上不成立。

输出：一页计划，列出阶段、层、`min_workers`、月度成本、尾部策略、失败模式。以要警报的单一指标结束：过去滚动小时的 P99 冷启动持续时间。
