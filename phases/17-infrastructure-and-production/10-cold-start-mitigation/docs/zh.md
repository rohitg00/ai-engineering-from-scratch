# 无服务器LLM的冷启动缓解

> 20 GB模型镜像从冷到服务需要5-10分钟（7B）到20+分钟（70B）。在真正的无服务器世界中，这不是预热 —— 这是中断。缓解措施在五层操作：预置节点镜像（AWS上的Bottlerocket、双卷架构）、模型流式传输（NVIDIA Run:ai Model Streamer、vLLM原生）、GPU内存快照（Modal检查点、快达10倍重启）、温池（`min_workers=1`）、分层加载（ServerlessLLM的NVMe→DRAM→HBM管道、延迟降低10-200倍），以及移动输入token（KB）而非KV缓存（GB）的实时迁移。Modal发布2-4秒冷启动作为底线；Baseten默认5-10秒，预预热下亚秒级。本课程教你如何测量、预算和堆叠五层。

**类型：** 学习
**语言：** Python（标准库，玩具冷启动路径模拟器）
**前置知识：** 第17阶段 · 02（推理平台经济学），第17阶段 · 03（GPU自动扩展）
**时间：** 约60分钟

## 学习目标

- 枚举冷启动缓解的五层并命名每层的一个工具或模式。
- 计算70B模型的总冷启动时间为（节点配置）+（权重下载）+（权重加载到HBM）+（引擎初始化）的总和。
- 解释为什么实时迁移传输输入token（KB）而非KV缓存（GB），以及惩罚是什么（重新计算）。
- 命名温池权衡（为空闲GPU付费或接受冷启动尾部）以及`min_workers > 0`成为强制性的SLA阈值。

## 问题

你的无服务器LLM端点夜间扩展到零。早上8点流量飙升。第一个请求等待：

1. Karpenter配置GPU节点：45-60秒。
2. 容器拉取30 GB带权重的镜像：120-300秒。
3. 引擎将权重加载到HBM：45-120秒，取决于模型大小和存储速度。
4. vLLM或TRT-LLM初始化CUDA图、KV缓存池、分词器：10-30秒。

总计：220-510秒（约3-8分钟），然后一个token返回。你的SLA是2秒。你部署温池（`min_workers=1`），问题似乎消失 —— 但现在你24x7为一个空闲GPU付费。如果你的服务有5个产品，每个有一个温副本，那就是5 × 24 × 30 = 3,600 GPU小时/月，无论是否有一个用户调用。

冷启动缓解是如何在保持无服务器经济学的同时近似始终在线的延迟。

## 概念

### 第1层 —— 预置节点镜像（Bottlerocket）

在AWS上，Bottlerocket的双卷架构将OS与数据分离。快照数据卷，预拉取容器镜像；在`EC2NodeClass`中引用快照ID。新节点启动时权重已在本地NVMe上 —— 步骤2和部分3消失。与Karpenter原生配合工作。典型节省：大模型每次冷启动2-4分钟。

GCP等效：预烘焙容器层的自定义VM镜像。Azure：具有相同模式的托管磁盘快照。

### 第2层 —— 模型流式传输（Run:ai Model Streamer）

不是在回答第一个请求之前加载完整文件，而是将权重逐层流式传输到GPU内存，并在第一个transformer块驻留后立即开始处理。NVIDIA Run:ai Model Streamer在vLLM 2026中原生发布。与S3、GCS和本地NVMe配合工作。通过将I/O与计算设置重叠，将大模型的权重加载时间大致减半。

### 第3层 —— GPU内存快照（Modal）

Modal在首次加载后检查GPU状态（权重、CUDA图、KV缓存区域）。后续重启直接反序列化到HBM —— 比重新初始化快10倍。这是最接近"2秒内启动温GPU"的东西。权衡：快照是按GPU拓扑的，因此如果Karpenter将你迁移到不同SKU，你重新检查点。

### 第4层 —— 温池（min_workers=1）

最简单的缓解：保持一个副本始终就绪。成本是一个GPU的每小时费率24x7。算术对小模型是残酷的（你支付$0.85-$1.50/小时以避免30秒冷启动），对大模型是仁慈的（支付$4/小时以避免5分钟冷启动）。温池成为强制性的SLA阈值：通常70B+模型上TTFT P99 < 60秒。

### 第5层 —— 分层加载（ServerlessLLM）

ServerlessLLM将存储视为层次结构：NVMe（快但大）、DRAM（中等但分层）、HBM（小但即时）。权重预加载到DRAM；按需加载到HBM。论文报告与朴素磁盘到HBM相比，冷加载延迟降低10-200倍。生产采用尚早，但与vLLM的集成存在。

### 第6层 —— 实时迁移（奖励模式）

当节点不可用（点驱逐、节点耗尽）时，传统模式是冷启动另一个副本并排空请求队列。实时迁移将输入token（千字节）移动到已加载模型的目标，并在目标上重新计算KV缓存。重新计算比通过网络传输GB级KV缓存更便宜。适用于解耦部署。

### 温池数学

对于P99 TTFT SLA为2秒的服务，问题不是"温池是/否"，而是"多少个温副本，以及哪些路径获得它们。"

- 高价值交互路径（实时聊天、语音代理）：`min_workers=1-2`。
- 后台批处理路径（夜间分类）：接受扩展到零，容忍5-10分钟冷启动。
- 高级层：每个租户`min_workers`带专用容量。

### 优化前测量

新节点上70B模型的冷启动解剖（说明性）：

| 阶段 | 时间 | 缓解 |
|------|------|------|
| 节点配置 | 50秒 | Bottlerocket + 预置镜像、温池 |
| 镜像拉取 | 180秒 | 预置数据卷（消除） |
| 权重到HBM | 75秒 | 模型流式传输（减半）；GPU快照（消除） |
| 引擎初始化 | 20秒 | 持久CUDA图缓存 |
| 首次前向 | 3秒 | 最小固有延迟 |
| **冷总计** | **328秒** | |
| **带缓解总计** | **约15秒** | 22倍降低 |

### 你应该记住的数字

- Modal冷启动：2-4秒（带GPU快照）。
- Baseten默认冷启动：5-10秒；预预热下亚秒级。
- 原始70B冷启动：3-8分钟。
- Run:ai Model Streamer：约2倍权重加载加速。
- ServerlessLLM分层加载：10-200倍延迟降低（论文数字）。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 冷启动 | "大暂停" | 新副本上从请求到首token的时间 |
| 温池 | "始终在线最小值" | `min_workers >= 1`以保持至少一个副本就绪 |
| 预置镜像 | "烘焙AMI" | 容器权重预驻留的节点镜像 |
| Bottlerocket | "AWS节点OS" | 具有双卷快照支持的AWS容器优化OS |
| 模型流式传输 | "流式加载" | 将权重I/O与计算设置重叠 |
| GPU快照 | "检查点到HBM" | 序列化加载后GPU状态；重启时反序列化 |
| 分层加载 | "NVMe + DRAM + HBM" | 存储层层次结构；按需加载 |
| 实时迁移 | "移动token" | 传输输入（KB），在目标上重新计算KV |
| `min_workers` | "温副本" | 无服务器最小保持活动计数 |
| 扩展到零 | "完全无服务器" | 空闲时无成本；接受全额冷启动税 |
