# 27 · LLM 的 FinOps —— 单元经济学与多租户费用归属

> 传统 FinOps 在面对 LLM 支出时彻底失效。成本是 Token 级别的事务,而非资源占用时长。标签不适用于此——一次 API 调用是一次事务,不是一项资产。工程决策（提示词设计、上下文窗口、输出长度）实际上就是财务决策。2026 年的路线图包含三个费用归属维度,必须在第一天就做好埋点：按用户（`user_id`）——用于席位定价和增长分析;按任务（`task_id` + `route`）——用于产品面的成本与优先级排序;按租户（`tenant_id`）——用于单元经济学和续约定价。四层 Token（提示词、工具调用、记忆、响应）若装在一个桶里,成本就被隐藏了。面向多租户产品应实施三级执行阶梯：按租户的限流（2–3 倍预期峰值,返回 429 + Retry-After）;每日消费上限（签约上限的 1.5–3 倍;触发后收紧限流 + 发送告警）;以及在消费 z-score > 4 时触发熔断开关（自动暂停 + 呼叫值班）。费用归属模式：标签聚合、遥测关联（Trace-ID → 计费;精度最高）、采样外推、基于模型的分配、事件溯源、实时流式。单元指标：单次查询解决成本、单次生成物成本——而非 $/M tokens。事后标签永远会有遗漏;务必在请求创建时完成埋点。

**类型：** 学习
**语言：** Python
**前置：** Phase 17 · 13（可观测性）,Phase 17 · 14（缓存）
**时长：** 约 60 分钟

## 学习目标

- 解释为什么传统 FinOps（标签 + 层级）在 LLM 支出上失效,并说出三个新的费用归属维度。
- 列举四层 Token（提示词、工具调用、记忆、响应）,并解释单一桶计费如何掩盖成本。
- 为多租户产品设计三级执行阶梯（限流 → 消费上限 → 熔断开关）。
- 选择单元指标（单次解决成本 / 单次生成物成本）,而非 $/M tokens。

## 问题所在

你的账单上写着 $40,000。你不知道：
- 哪个租户花掉了这笔钱。
- 哪个产品功能驱动了它。
- 是否有单个用户存在滥用行为。
- 提示词膨胀、工具调用还是记忆放大是元凶。

在服务商侧按标签聚合,适用于标签能传播到行项目的云资源（EC2、S3）。但 LLM API 调用不会自动打标签——你必须在调用点把用户/任务/租户信息打上去并持续携带。事后费用归属总会有遗漏。

## 核心概念

### 三个费用归属维度

**按用户**（`user_id`）：谁花了多少钱。驱动席位定价、增长对话、识别重度用户。

**按任务**（`task_id` + `route`）：哪个产品面花了多少钱。驱动功能优先级排序、停用昂贵功能的决策。

**按租户**（`tenant_id`）：哪个客户是盈利的。驱动单元经济学、续约定价、分层阈值。

上述三个维度必须在第一天就嵌入调用点。事后打标永远是次优解。

### 四层 Token

| 层 | 示例 | 占总量的典型比例 |
|-|-|-|
| 提示词 | 系统提示 + 用户输入 | 40–60% |
| 工具调用 | 回传的工具调用结果 | 20–40%（智能体工作负载） |
| 记忆 | 之前的对话 / 检索到的文档 | 10–30% |
| 响应 | 模型输出 | 10–30% |

把所有四层混在一起,优化就无从下手。在你的费用归属 schema 中把它们拆分开。

### 三级执行阶梯

1. **限流**：按租户。2–3 倍预期峰值。返回 429 及 `Retry-After`。租户感知到摩擦;但不会有账单惊喜。

2. **每日消费上限**：按租户。签约上限的 1.5–3 倍。触发：收紧限流 + 告警客户成功团队。

3. **熔断开关**：当消费 z-score 相对租户基线 > 4 时,自动暂停该租户,呼叫值班,升级到运营与 CS。

### 费用归属模式

- **标签聚合**：在 metadata header 上打标,后续聚合。简单,但粗糙。
- **遥测关联**：通过 Trace-ID 将追踪关联到计费。精度最高。成熟团队的做法。
- **采样外推**：采样 5–10%,再乘以系数。粗略成本控制中有性价比;但会遗漏尾部。
- **基于模型的分配**：回归推导成本驱动因素。适用于没有标签的历史数据。
- **事件溯源**：将成本作为流（Kafka / Kinesis）中的事件。实时。
- **实时流式**：仪表盘亚秒级更新。

### 成本 per X 是真正的单元指标

$/M tokens 是服务商口径。产品指标才是关键：

- 每解决一张工单的成本。
- 每生成一篇文章的成本。
- 每完成一次智能体任务的成本。
- 每用户-会话-分钟的成本。

必须将成本绑定到产品产出上,否则优化将失去锚点。

### 费用归属 Trace 形态

```
trace_id: abc123
  user_id: u_42
  tenant_id: t_7
  task_id: task_classify_doc
  route: model_haiku
  layers:
    prompt_tokens: 1800
    tool_tokens: 600
    memory_tokens: 400
    response_tokens: 150
  cost_usd: 0.0135
  cached_input: true
  batch: false
```

每次调用都要发送上述信息。存储到数据湖。按维度聚合。Phase 17 · 13 的可观测性栈就是用来承载这些的。

### 组合降本栈

将缓存 + 批量 + 路由 + 网关全部组合起来：
- L2 缓存（Phase 17 · 14）：输入成本约 1/10。
- 批量 API（Phase 17 · 15）：5 折。
- 路由到廉价模型（Phase 17 · 16）：成本降低 60%。
- 网关效率（Phase 17 · 19）：冗余 + 重试保障。

最佳情况叠加后可降至朴素基准的 ~5–10%。多数团队仅握住了 2–3 个杠杆;极少有团队四者全部叠满。

### 你应该记住的数字

- 费用归属维度：按用户、按任务、按租户。
- 四层 Token：提示词、工具调用、记忆、响应。
- 熔断开关：消费 z-score > 4。
- 单元指标：单次查询解决成本,而非 $/M tokens。
- 组合优化：可降至基准的 ~5–10%。

## 关键术语

| 术语 | 人们常这么说 | 实际含义 |
|-|-|-|
| 按用户归属 | 「用户级成本」 | 每次调用嵌入 `user_id` |
| 按任务归属 | 「功能成本」 | `task_id` + `route` 标识产品面 |
| 按租户归属 | 「客户成本」 | `tenant_id`;驱动单元经济学 |
| 四层 Token | 「成本层级」 | 提示词 + 工具调用 + 记忆 + 响应 |
| 限流 | 「429 防护」 | 在网关层强制执行的按租户上限 |
| 每日消费上限 | 「日度天花板」 | 带告警提醒的租户范围预算 |
| 熔断开关 | 「自动暂停」 | 消费 z-score > 4 触发自动暂停 |
| 单次解决成本 | 「产品单元指标」 | 绑定到产品产出的成本,而非 Token |
| 遥测关联 | 「Trace → 计费」 | 最高精度的费用归属模式 |
| 组合优化 | 「缓存+批量+路由+网关」 | 叠加降本至 ~5–10% 基准 |
