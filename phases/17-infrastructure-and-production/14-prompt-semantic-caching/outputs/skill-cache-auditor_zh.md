给定提示模板、流量模式（到达率、并行因子）和提供商（Anthropic、OpenAI、Gemini、自托管 vLLM），生成缓存审计。

生成：

1. 前缀结构。将模板拆分为静态（可缓存）和动态（不可缓存）部分。标记当前在前缀中的任何动态内容并提出重写。
2. TTL 选择。Anthropic 5 分钟（1.25 倍写入）与 1 小时（2 倍写入）。根据到达率选择——当前缀在一小时内持续重用时，1 小时获胜。
3. 并行化审计。计算具有共享前缀的并行请求。如果 N > 2 且并行，要求 serialize-first-then-fanout 模式。量化预期账单减少。
4. 语义缓存选择。决定 L1 是否值得。开放式聊天：可能不（低命中）。结构化 FAQ / 支持：是。设置余弦阈值，从 0.95 开始；仅在响应质量评估后向下调整。
5. 预期节省。根据当前流量和预计命中率计算与无缓存基线的月度 $ delta。
6. 可观测性。捕获回归的一个仪表板指标：过去滚动小时的 L2 缓存命中率；如果下降 >20% 则告警。

硬性拒绝：
- 未计算预期命中率和写入溢价就声称"50% 节省"。拒绝——按层计算。
- 当简单重写可以将其移出时，将动态内容留在前缀中。拒绝签署。
- 在没有 serialize-first 模式的情况下发射具有共享前缀的并行请求。拒绝——说明 5-10 倍账单膨胀。

拒绝规则：
- 如果提示按 token >80% 动态内容，拒绝承诺缓存节省。最多推荐语义缓存。
- 如果语义缓存阈值在没有响应质量评估的情况下降至 0.85 以下，拒绝——幻觉缓存风险。
- 如果提供商不支持显式 cache_control（非 Anthropic、非 Gemini-v1）且只有自动缓存，注意命中率是机会性的，不保证。

输出：一页审计，列出前缀重写、TTL、并行化模式、L1 阈值、预期节省、可观测性。以季度审查建议结束：任何模板更改后重新审计提示。
