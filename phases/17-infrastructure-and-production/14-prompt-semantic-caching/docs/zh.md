# 提示缓存和语义缓存经济学

> **定价快照日期 2026-04。** 以下数字声明反映的是本课发布时捕获的供应商费率卡；在下游引用它们之前，请对照链接的文档进行验证。

> 缓存在两个层发生。L2（提供商级）提示/前缀缓存对重复前缀重用注意力 KV——Anthropic 的提示缓存文档宣传在长提示上高达 90% 的成本降低和 85% 的延迟降低；对于 Claude 3.5 Sonnet，缓存读取是 0.30 美元/百万 vs 3.00 美元/百万新写入，具有 5 分钟 TTL 和 1 小时 TTL 选项的 2 倍写入溢价（docs.anthropic.com，2026-04）。OpenAI 提示缓存在提示 ≥1024 token 时自动应用，并以大约 90% 的折扣定价缓存输入 vs 新输入（platform.openai.com，2026-04）；确切的每模型缓存费率取决于实时费率卡。L1（应用级）语义缓存在嵌入相似性命中时完全跳过 LLM。供应商"95% 准确性"指的是匹配正确性，而不是命中率——报告的生产命中率范围从 10%（开放式聊天）到 70%（结构化 FAQ）；两家提供商都没有发布官方基线，因此将这些视为社区遥测而不是保证。生产陷阱：并行化会破坏缓存（在第一次缓存写入之前发出的 N 个并行请求可能会使支出膨胀数倍），前缀中的动态内容完全阻止缓存命中。ProjectDiscovery 报告通过将动态文本移出可缓存前缀，从 7% 到 74% 的命中率（2025-11）。

**类型：** 学习
**语言：** Python（标准库，简单的两层缓存模拟器）
**先修要求：** 阶段 17 · 04（vLLM 服务内部原理）、阶段 17 · 06（SGLang RadixAttention）
**时间：** 约 60 分钟

## 学习目标

- 区分 L2 提示/前缀缓存（提供商处的 KV 重用）与 L1 语义缓存（在相似提示时绕过 LLM）。
- 解释 Anthropic 的 `cache_control` 显式标记和两种 TTL 选项（5 分钟 vs 1 小时）及其价格乘数。
- 给定命中率、提示/响应组合和 token 价格，计算预期的每月节省。
- 说出使账单膨胀 5-10 倍的并行化反模式和使命中率崩溃的动态内容反模式。

## 问题

你为 RAG 服务添加了提示缓存。账单保持平稳。你测量命中率；它是 7%。你的提示看起来是静态的，但它们不是——系统提示包括格式化为分钟的当前日期、请求 ID 和用于多样性的随机示例重新排序。每个请求写入一个新的缓存条目，读取零。

另外，你的代理每个用户问题运行十个并行工具调用。所有十个都在第一次缓存写入完成之前到达提供商。十次写入，零次读取。你的账单是"使用缓存"应该花费的 5-10 倍。

缓存是一个协议，而不是一个标志。两层，两种不同的失败模式。

## 概念

### L2——提供商提示/前缀缓存

提供商存储可缓存前缀的注意力 KV，并在下一个匹配前缀的请求上重用它。你支付一次写入成本，读取几乎免费。

**Anthropic (Claude 3.5 / 3.7 / 4 系列)**：请求中的显式 `cache_control` 标记。你标记哪些块是可缓存的。TTL：5 分钟（写入成本 1.25 倍基础）或 1 小时（写入成本 2 倍基础）。缓存读取：Claude 3.5 Sonnet 上 0.30 美元/百万 vs 3.00 美元/百万新写入——便宜 10 倍（docs.anthropic.com，截至 2026-04）。费率因模型（Opus/Haiku 单独发布）而异；始终交叉检查实时定价页面。

**OpenAI**：对提示 ≥1024 token 自动缓存（platform.openai.com，2026-04）。无显式标志。缓存输入在当前 gpt-4o/gpt-5 费率卡上比新输入便宜大约 10 倍。文档和发行说明都没有发布官方命中率基线；社区报告在仔细的提示设计下围绕 30-60% 聚集。监控 `usage.cached_tokens` 以测量你自己的。

**Google (Gemini)**：通过显式 API 的上下文缓存；1M token 上下文意味着缓存支付更多。

**自托管 (vLLM, SGLang)**：阶段 17 · 06 涵盖 RadixAttention——在你自己的计算上具有相同模式。

### L1——应用级语义缓存

在完全调用 LLM 之前，哈希提示，嵌入它，并寻找相似的缓存请求（余弦相似度高于阈值，通常为 0.95+）。命中时，返回缓存的响应。未命中时，调用 LLM 并缓存结果。

开源：Redis Vector Similarity、GPTCache、Qdrant。商业：Portkey Cache、Helicone Cache。

供应商准确性声明指的是返回的缓存响应在语义上适当的频率——而不是你命中的频率。生产命中率：

- 开放式聊天：10-15%。
- 结构化 FAQ / 支持：40-70%。
- 代码问题：20-30%（微小变体杀死命中）。
- 重复提示的语音代理：50-80%（语音规范化固定集）。

### 并行化反模式

你的代理并行进行 10 个工具调用。所有 10 个都有相同的 4K token 系统提示。Anthropic 缓存写入是每个请求的；第一次缓存写入在提供商看到提示后大约 300 毫秒完成。请求 2-10 在同一毫秒窗口中到达，每个都看到缓存未命中。你支付 10 次写入溢价，0 次读取折扣。

修复：先用顺序批处理——单独发出请求 1，然后在请求 1 的缓存已填充后激发 2-10。为第一个工具调用增加 300 毫秒；节省 5-10 倍的账单。

### 动态内容反模式

你的系统提示看起来像：

```
你是一个有用的助手。当前时间是 14:32:17。
用户 ID：abc123。今天是星期二...
```

每个请求都是唯一的。每个请求都写入。零命中。

修复：将一切真正静态的移动到可缓存前缀；在缓存边界后附加动态内容：

```
[cacheable]
你是一个有用的助手。[规则、示例、指令]
[/cacheable]
[dynamic, not cached]
当前时间：14:32:17。用户：abc123。
```

ProjectDiscovery 通过这种方式从 7% 移动到 74% 缓存命中率并发布了分析。

### 堆栈批处理 + 缓存用于隔夜工作负载

批处理 API（阶段 17 · 15）以 24 小时周转提供 50% 折扣。顶部的缓存输入让你在此基础上再获得约 10 倍。隔夜分类、标记和报告生成工作负载可以通过堆叠降至同步未缓存成本的大约 10%。

### 你应该记住的数字

定价点是从链接的供应商文档捕获的 2026-04，并且每隔几个月就会漂移——在依赖它们之前重新检查。

- Anthropic 缓存读取：Claude 3.5 Sonnet 上 0.30 美元/百万，比新输入便宜大约 10 倍（docs.anthropic.com）。
- Anthropic 缓存写入溢价：1.25 倍（5 分钟 TTL）或 2 倍（1 小时 TTL）。
- OpenAI 自动缓存：适用于提示 ≥1024 token；在当前费率卡上，缓存输入定价约为新输入的 10%（platform.openai.com）。
- 语义缓存命中率（社区报告）：约 10% 开放聊天；高达约 70% 结构化 FAQ。不是供应商记录的基线。
- ProjectDiscovery：通过将动态移出前缀，从 7% → 74% 命中率（项目博客，2025-11）。
- 并行化反模式：当 N 个并行请求错过第一次缓存写入时，典型的 5-10 倍账单膨胀报告。

## 练习

1. 运行 `code/main.py`。切换并行化标志。账单变化多少？
2. 你的系统提示有一个日期。将它移出。显示前后命中率数学。
3. 给定你的请求到达率，计算 1 小时 TTL（2 倍写入）vs 5 分钟 TTL（1.25 倍写入）的盈亏平衡。
4. 在 0.95 阈值下的语义缓存命中 20%。在 0.85 时它命中 50%，但你看到不正确的缓存响应。选择正确的阈值并证明。
5. 你每个用户问题批处理 10 个并行子查询。为缓存友好性重写而不增加端到端延迟。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| L2 prompt cache | "前缀缓存" | 提供商为重复前缀存储 KV |
| `cache_control` | "Anthropic 缓存标记" | 标记可缓存块的显式属性 |
| Cache write premium | "写入税" | 第一次未命中到缓存的额外成本（1.25 倍或 2 倍） |
| L1 semantic cache | "嵌入缓存" | 在调用 LLM 之前的应用级哈希和嵌入 |
| GPTCache | "LLM 缓存库" | 流行的 OSS L1 缓存库 |
| Cache hit rate | "命中 / 总计" | 从缓存提供服务的请求分数 |
| Parallelization anti-pattern | "N 写入陷阱" | N 个并行请求 N 次错过缓存 |
| Dynamic content trap | "提示中的时间陷阱" | 前缀中的动态字节杀死命中率 |
| RadixAttention | "副本内缓存" | SGLang 的前缀缓存实现 |
