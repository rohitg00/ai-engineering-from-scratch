# 提示缓存和语义缓存经济学

> **定价快照日期2026-04。** 以下数字声明反映本课程发布时捕获的供应商费率卡；在下游引用前对照链接文档验证。

> 缓存在两层发生。L2（提供商级）提示/前缀缓存重用重复前缀的注意力KV —— Anthropic的提示缓存文档宣传长提示上高达90%成本降低和85%延迟降低；对于Claude 3.5 Sonnet，缓存读取为$0.30/M vs $3.00/M新鲜，5分钟TTL和1小时TTL选项的2倍写入溢价（docs.anthropic.com，2026-04）。OpenAI提示缓存自动应用于提示≥1024 token，并将缓存输入定价为大致比新鲜便宜90%（platform.openai.com，2026-04）；确切每模型缓存率取决于实时费率卡。L1（应用级）语义缓存在嵌入相似性命中上完全跳过LLM。供应商"95%准确率"指匹配正确性，非命中率 —— 报告的生产命中率范围从10%（开放式聊天）到70%（结构化FAQ）；提供商均未发布官方基线，因此将这些视为社区遥测而非保证。生产陷阱：并行化杀死缓存（在首次缓存写入前发出的N个并行请求可能将支出膨胀数倍），前缀内的动态内容完全阻止缓存命中。ProjectDiscovery报告通过将动态文本移出可缓存前缀，命中率从7%移动到74%（2025-11）。

## 学习目标

- 区分L2提示/前缀缓存（提供商处的KV重用）与L1语义缓存（相似提示上绕过LLM）。
- 解释Anthropic的`cache_control`显式标记和两个TTL选项（5分钟 vs 1小时）及其价格乘数。
- 给定命中率、提示/响应组合和token价格，计算预期月度节省。
- 命名将账单膨胀5-10倍的并行化反模式和崩溃命中率的动态内容反模式。

## 问题

你将提示缓存添加到RAG服务。账单保持平稳。你测量命中率；它是7%。你的提示看起来是静态的，但它们不是 —— 系统提示包括格式化为分钟的当前日期、请求ID和用于多样性的随机化示例重新排序。每个请求写入新缓存条目，读取零。

另外，你的代理每个用户问题运行十个并行工具调用。所有十个在首次缓存写入完成前到达提供商。十次写入，零次读取。你的账单是"带缓存"应有成本的5-10倍。

缓存是协议，不是标志。两层，两种不同失败模式。

## 概念

### L2 —— 提供商提示/前缀缓存

提供商存储可缓存前缀的注意力KV，并在下一个匹配前缀的请求上重用它。你支付一次写入成本，读取几乎免费。

**Anthropic（Claude 3.5 / 3.7 / 4系列）**：请求中的显式`cache_control`标记。你标记哪些块是可缓存的。TTL：5分钟（写入成本1.25倍基础）或1小时（写入成本2倍基础）。缓存读取：Claude 3.5 Sonnet上$0.30/M vs $3.00/M新鲜 —— 便宜10倍（docs.anthropic.com，截至2026-04）。费率因模型而异（Opus/Haiku单独发布）；始终交叉检查实时定价页面。

**OpenAI**：提示≥1024 token的自动缓存（platform.openai.com，2026-04）。无显式标志。当前gpt-4o/gpt-5费率卡上，缓存输入大致比新鲜便宜10倍。文档和发布说明均未发布官方命中率基线；社区报告在仔细提示设计下聚集在30-60%。监控`usage.cached_tokens`以测量你自己的。

**Google（Gemini）**：通过显式API的上下文缓存；1M token上下文意味着缓存支付更多。

**自托管（vLLM、SGLang）**：第17阶段 · 06覆盖RadixAttention —— 在你自己的计算上相同模式。

### L1 —— 应用级语义缓存

在完全调用LLM之前，哈希提示，嵌入它，并查找相似的缓存请求（余弦相似性高于阈值，通常0.95+）。命中时，返回缓存响应。未命中时，调用LLM并缓存结果。

开源：Redis Vector Similarity、GPTCache、Qdrant。商业：Portkey Cache、Helicone Cache。

供应商准确率声明指返回的缓存响应在语义上适当的频率 —— 不是你命中的频率。生产命中率：

- 开放式聊天：10-15%。
- 结构化FAQ / 支持：40-70%。
- 代码问题：20-30%（小变体杀死命中）。
- 重复提示的语音代理：50-80%（语音规范化固定集）。

### 并行化反模式

你的代理进行10个并行工具调用。所有10个具有相同的4K token系统提示。Anthropic缓存写入是按请求的；首次缓存写入在提供商看到提示后约300毫秒完成。请求2-10在同一毫秒内到达，每个看到缓存未命中。你支付10次写入溢价，0次读取折扣。

修复：用顺序优先批处理 —— 单独进行请求1，然后在1的缓存填充后触发2-10。为首次工具调用增加300毫秒；节省5-10倍账单。

### 动态内容反模式

你的系统提示看起来像这样：

```
You are a helpful assistant. The current time is 14:32:17.
User ID: abc123. Today is Tuesday...
```

每个请求都是唯一的。每个请求都写入。零命中。

修复：将所有真正静态的内容移动到可缓存前缀；在缓存边界后附加动态内容：

```
[cacheable]
You are a helpful assistant. [rules, examples, instructions]
[/cacheable]
[dynamic, not cached]
Current time: 14:32:17. User: abc123.
```

ProjectDiscovery通过这种方式从7%移动到74%缓存命中率，并发布了解剖。

### 为夜间工作负载堆叠批处理 + 缓存

批处理API（第17阶段 · 15）在24小时周转时给予50%折扣。顶部的缓存输入在顶部再给你约10倍。夜间分类、标记和报告生成工作负载可以通过堆叠降至同步未缓存成本的约10%。

### 你应该记住的数字

定价点从链接的供应商文档捕获于2026-04，每隔几个月漂移 —— 在依赖前重新检查。

- Anthropic缓存读取：Claude 3.5 Sonnet上$0.30/M，大致比新鲜输入便宜10倍（docs.anthropic.com）。
- Anthropic缓存写入溢价：1.25倍（5分钟TTL）或2倍（1小时TTL）。
- OpenAI自动缓存：应用于提示≥1024 token；当前费率卡上缓存输入定价为新鲜输入的大致10%（platform.openai.com）。
- 语义缓存命中率（社区报告）：开放式聊天约10%；结构化FAQ高达约70%。非供应商文档基线。
- ProjectDiscovery：通过将动态内容移出前缀，7% → 74%命中率（项目博客，2025-11）。
- 并行化反模式：N个并行请求错过首次缓存写入时，典型报告5-10倍账单膨胀。

1. 运行`code/main.py`。切换并行化标志。账单变化多少？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| L2提示缓存 | "前缀缓存" | 提供商存储重复前缀的KV |
| `cache_control` | "Anthropic缓存标记" | 标记可缓存块的显式属性 |
| 缓存写入溢价 | "写入税" | 首次未命中到缓存的额外成本（1.25倍或2倍） |
| L1语义缓存 | "嵌入缓存" | 调用LLM之前的应用级哈希和嵌入 |
| GPTCache | "LLM缓存库" | 流行的OSS L1缓存库 |
| 缓存命中率 | "命中/总计" | 从缓存服务的请求比例 |
| 并行化反模式 | "N写入陷阱" | N个并行请求错过缓存N次 |
| 动态内容陷阱 | "提示中的时间陷阱" | 前缀中的动态字节杀死命中率 |
| RadixAttention | "副本内缓存" | SGLang的前缀缓存实现 |
