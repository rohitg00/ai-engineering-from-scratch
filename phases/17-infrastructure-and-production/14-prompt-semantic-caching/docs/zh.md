# 14 · 提示缓存与语义缓存的经济账

> **定价快照采集于 2026-04。** 下文的数值结论反映的是本课发布时采集到的厂商价目表;在向下游引用之前,请对照所链接的文档进行核实。

> 缓存发生在两个层级。L2(供应商级)提示/前缀缓存复用重复前缀的注意力 KV(attention KV)——Anthropic 的提示缓存文档宣称在长提示上最高可降低 90% 成本、降低 85% 延迟;就 Claude 3.5 Sonnet 而言,缓存读取为 $0.30/M,而全新输入为 $3.00/M,TTL 为 5 分钟,1 小时 TTL 选项的写入溢价为 2 倍(docs.anthropic.com,2026-04)。OpenAI 的提示缓存对 ≥1024 token 的提示自动生效,缓存输入相对全新输入约打 9 折(即约 10% 价格)(platform.openai.com,2026-04);每个模型确切的缓存费率取决于实时价目表。L1(应用级)语义缓存在「嵌入相似度(embedding similarity)」命中时完全跳过大语言模型(LLM)。厂商所称的「95% 准确率」指的是匹配的正确性,而非命中率——已报告的生产命中率从 10%(开放式聊天)到 70%(结构化 FAQ)不等;两家供应商都未发布官方基线,因此应将这些数字视为社区遥测数据而非保证。生产中的陷阱有:并行化会扼杀缓存(在首次缓存写入完成前发出的 N 个并行请求会使开销膨胀数倍),前缀内的动态内容会彻底阻止缓存命中。ProjectDiscovery 报告称(2025-11),通过将动态文本移出可缓存前缀,其命中率从 7% 提升到了 74%。

**类型：** 学习
**语言：** Python(标准库,玩具级双层缓存模拟器)
**前置：** 阶段 17 · 04(vLLM 服务内部原理),阶段 17 · 06(SGLang RadixAttention)
**时长：** 约 60 分钟

## 学习目标

- 区分 L2 提示/前缀缓存(供应商侧的 KV 复用)与 L1 语义缓存(对相似提示绕过 LLM)。
- 解释 Anthropic 的 `cache_control` 显式标记,以及两种 TTL 选项(5 分钟与 1 小时)及其价格倍率。
- 在给定命中率、提示/响应配比与 token 价格的情况下,计算预期的每月节省额。
- 说出会使账单膨胀 5-10 倍的并行化反模式,以及会令命中率崩塌的动态内容反模式。

## 问题所在

你给 RAG 服务加上了提示缓存。账单纹丝不动。你测量命中率,结果是 7%。你的提示看起来是静态的,其实不是——系统提示中包含了精确到分钟的当前日期、一个请求 ID,以及为追求多样性而随机重排的示例。每个请求都写入一个新的缓存条目,读取为零。

另一方面,你的智能体(agent)对每个用户问题会发起十个并行工具调用。这十个请求全都在第一次缓存写入完成之前就抵达了供应商。十次写入,零次读取。你的账单是「带缓存」本应花费的 5-10 倍。

缓存是一套协议,而不是一个开关。两个层级,两种不同的失败模式。

## 概念解析

### L2 —— 供应商提示/前缀缓存

供应商为可缓存的前缀存储注意力 KV,并在下一个匹配该前缀的请求上复用它。你只需支付一次写入成本,读取则几乎免费。

**Anthropic(Claude 3.5 / 3.7 / 4 系列)**:在请求中显式加 `cache_control` 标记。你来标注哪些块是可缓存的。TTL:5 分钟(写入成本为基础价的 1.25 倍)或 1 小时(写入成本为基础价的 2 倍)。缓存读取:Claude 3.5 Sonnet 上为 $0.30/M,而全新输入为 $3.00/M——便宜 10 倍(docs.anthropic.com,截至 2026-04)。各模型费率不同(Opus/Haiku 单独发布);务必交叉核对实时定价页面。

**OpenAI**:对 ≥1024 token 的提示自动缓存(platform.openai.com,2026-04)。无显式标志。在当前 gpt-4o/gpt-5 价目表上,缓存输入比全新输入便宜约 10 倍。无论文档还是发布说明都未公布官方命中率基线;社区报告在精心设计提示的情况下大致集中在 30–60%。监控 `usage.cached_tokens` 来度量你自己的命中率。

**Google(Gemini)**:通过显式 API 进行上下文缓存;100 万 token 的上下文意味着缓存带来的回报更可观。

**自托管(vLLM、SGLang)**:阶段 17 · 06 讲解了 RadixAttention——在你自己的算力上实现同样的模式。

### L1 —— 应用级语义缓存

在调用 LLM 之前,先对提示做哈希、做嵌入,然后查找一个相似的已缓存请求(余弦相似度高于阈值,通常 0.95+)。命中时,返回缓存的响应;未命中时,调用 LLM 并缓存结果。

开源方案:Redis Vector Similarity、GPTCache、Qdrant。商业方案:Portkey Cache、Helicone Cache。

厂商的准确率说法指的是所返回的缓存响应在语义上恰当的频率——而非你命中的频率。生产命中率:

- 开放式聊天:10-15%。
- 结构化 FAQ / 客服支持:40-70%。
- 代码问题:20-30%(细微变体会扼杀命中)。
- 重复提示的语音智能体:50-80%(语音归一化为固定集合)。

### 并行化反模式

你的智能体并行发起 10 个工具调用。这 10 个调用都带有相同的 4K-token 系统提示。Anthropic 的缓存写入是按请求进行的;在供应商看到提示后约 300 毫秒,第一次缓存写入才完成。请求 2-10 在同一毫秒窗口内到达,每个都遭遇缓存未命中。你支付了 10 次写入溢价,获得 0 次读取折扣。

修复方法:采用「顺序优先」的批处理——先单独发出请求 1,等到请求 1 的缓存填充完毕后,再发出 2-10。这会给第一个工具调用增加 300 毫秒,但节省 5-10 倍的账单。

### 动态内容反模式

你的系统提示长这样:

```
You are a helpful assistant. The current time is 14:32:17.
User ID: abc123. Today is Tuesday...
```

每个请求都是唯一的。每个请求都会写入。零命中。

修复方法:把真正静态的一切移入可缓存前缀;把动态内容追加到缓存边界之后:

```
[cacheable]
You are a helpful assistant. [rules, examples, instructions]
[/cacheable]
[dynamic, not cached]
Current time: 14:32:17. User: abc123.
```

ProjectDiscovery 正是用这种方式将缓存命中率从 7% 提升到了 74%,并公布了其结构剖析。

### 为夜间工作负载叠加批处理 + 缓存

批处理 API(阶段 17 · 15)在 24 小时周转时间下给出 50% 折扣。在此之上叠加缓存输入,再获得约 10 倍的优惠。夜间的分类、标注与报告生成类工作负载,通过叠加可以降至同步、无缓存成本的约 10%。

### 你应该记住的数字

定价数据采集于 2026-04,来自所链接的厂商文档,且每隔几个月就会漂移——在依赖它们之前请重新核对。

- Anthropic 缓存读取:Claude 3.5 Sonnet 上为 $0.30/M,约比全新输入便宜 10 倍(docs.anthropic.com)。
- Anthropic 缓存写入溢价:1.25 倍(5 分钟 TTL)或 2 倍(1 小时 TTL)。
- OpenAI 自动缓存:适用于 ≥1024 token 的提示;在当前价目表上缓存输入的定价约为全新输入的 10%(platform.openai.com)。
- 语义缓存命中率(社区报告):开放式聊天约 10%;结构化 FAQ 最高约 70%。这不是厂商文档化的基线。
- ProjectDiscovery:通过将动态内容移出前缀,命中率从 7% → 74%(项目博客,2025-11)。
- 并行化反模式:当 N 个并行请求错过首次缓存写入时,典型报告称账单膨胀 5–10 倍。

## 关键术语

| 术语 | 人们怎么说 | 它实际的含义 |
|------|----------------|------------------------|
| L2 提示缓存 | 「前缀缓存」 | 供应商为重复前缀存储 KV |
| `cache_control` | 「Anthropic 缓存标记」 | 标注可缓存块的显式属性 |
| 缓存写入溢价 | 「写入税」 | 首次「未命中到入缓存」的额外成本(1.25 倍或 2 倍) |
| L1 语义缓存 | 「嵌入缓存」 | 调用 LLM 之前的应用级哈希与嵌入 |
| GPTCache | 「LLM 缓存库」 | 流行的开源 L1 缓存库 |
| 缓存命中率 | 「命中 / 总数」 | 由缓存服务的请求占比 |
| 并行化反模式 | 「N 次写入陷阱」 | N 个并行请求 N 次错过缓存 |
| 动态内容陷阱 | 「提示中放时间的陷阱」 | 前缀中的动态字节扼杀命中率 |
| RadixAttention | 「副本内缓存」 | SGLang 的前缀缓存实现 |
