# 15 · 批处理 API——已成为行业标准的五折优惠

> 每一家主流厂商都提供异步「批处理 API（Batch API）」，享有五折优惠和约 24 小时的处理时限。OpenAI、Anthropic、Google 以及大多数推理平台（Fireworks 批处理层、Together 批处理）都实现了相同的模式。把批处理与「提示缓存（prompt caching）」叠加使用，隔夜流水线的成本可降至同步无缓存成本的约 10%。规则简单到近乎残酷：只要不是交互式的，就该放进批处理。内容生成流水线、文档分类、数据抽取、报告生成、批量标注、目录打标——任何能容忍 24 小时延迟的任务，在迁移到批处理之前都是白白扔掉的钱。2026 年的生产实践是把每一个新的 LLM 工作负载分流到三条车道：交互式（带缓存的同步调用）、半交互式（带回退的异步队列）、批处理（隔夜、叠加输入缓存）。那些假装自己是交互式、实则能容忍数分钟延迟的工作负载，浪费最为严重。

## 学习目标

- 说出三家厂商的批处理 API（OpenAI、Anthropic、Google），以及共有的五折优惠 + 24 小时处理时限保证。
- 计算在一个隔夜分类工作负载上叠加批处理 + 输入缓存的成本，并与同步无缓存的基线对比。
- 把一个工作负载分流为交互式 / 半交互式 / 批处理，并论证所选车道的合理性。
- 说出两个陷阱：部分交互性（用户期望比 24 小时更快）和输出 schema 漂移（批处理文件格式因厂商而异）。

## 问题所在

你的团队上线了一条每晚运行的报告生成流水线。50,000 篇文档，逐篇做摘要、对摘要聚类、起草一份高管简报。同步运行需要 4 小时，每晚花费 2,000 美元。你听说了批处理 API。

批处理给你打五折。你又对系统提示（在全部 5 万次调用间共享）启用了提示缓存。叠加之后，账单降到每晚 180 美元——约为基线的 9%。同一条流水线，三处配置改动。

批处理是 LLM 成本工具箱里最便宜、却没人去拉的那根杠杆。原因多半出在组织层面：团队一想就是「实时」，而实际的 SLA 不过是「明早之前」。本课要讲的就是别把账单的 90% 白白扔掉。

## 核心概念

### 三家批处理 API

**OpenAI Batch API**：上传一个包含请求列表的 JSONL 文件。承诺 24 小时处理时限（实践中通常约 2-8 小时）。输入和输出 token 享五折优惠。符合缓存条件的输入还可在此基础上叠加输入缓存定价。

**Anthropic Message Batches**：上传 JSONL。24 小时处理时限。五折优惠。支持 `cache_control`——缓存写入是显式的，读取则在批处理内自动发生。

**Google Vertex AI Batch Prediction**：以 BigQuery 或 GCS 作为输入。对 Gemini 提供类似的五折优惠。可与 Vertex 流水线集成。

### 语义：是异步，不是慢

批处理的含义是「我承诺在 24 小时内返回」——而非「这件事要花 24 小时」。典型 P50 是 2-6 小时。厂商会把你的批处理调度到 GPU 库存未被充分利用的非高峰时段。

### 与缓存叠加

一个使用相同 4K-token 系统提示的 5 万篇文档摘要任务：

- 同步无缓存：50000 × （$input × 4000 + $output × 200），全价。
- 同步带缓存：系统提示在首次写入后被缓存；其余 49999 次调用的输入便宜 10 倍。
- 批处理带缓存：在上述基础上，对读取和写入再叠加五折优惠。

叠加效果：批处理 + 缓存 = 同步无缓存账单的约 10%。任何隔夜运行且带有共享系统提示的工作负载都应这么做。

### 工作负载分流

**交互式**——用户在等待响应。「首 token 时延（TTFT）」很重要。用带提示缓存的同步调用。不能批处理。

**半交互式**——用户提交一个任务，几分钟后回来查看。用异步队列，在批处理不可用时回退到同步。可以联想中等吞吐量的 RAG 索引。

**批处理**——用户期望「明早」或「下个小时」拿到结果。内容流水线、大规模分类、离线分析。一律批处理，一律叠加缓存。

常见错误：因为流水线是生产环境，就把所有东西都归类为交互式。生产环境并不是一项延迟规格——SLA 才是。

### 部分交互性陷阱

有些功能看起来是交互式的，但其实能容忍 5-10 分钟。例如：一份带「刷新」按钮的每晚客户健康度报告。用户点击刷新；等 10 分钟也没问题。团队却把它做成了同步的。50 个并发刷新的花费，是「批处理 + 邮件投递」方案的 10 倍。

要问的问题是：「24 小时对这个用户意味着什么？」如果答案是「他们根本不会察觉」，那就批处理。

### 输出 schema 陷阱

批处理文件格式因厂商而异：

- OpenAI：JSONL，每行一个请求。
- Anthropic：JSONL，每行一条 message；响应格式内嵌其中。
- Vertex：BigQuery 表，或带 TFRecord 的 GCS 前缀。

想写出一个跨厂商的「统一批处理客户端」，意味着要为每家厂商编写适配器代码。那些标榜支持多厂商批处理的网关（Portkey、LiteLLM 的某些层级）仍然只是对原始格式做了薄封装。

### 你应该记住的数字

- 各厂商批处理优惠：输入 + 输出一律五折。
- 处理时限 SLA：保证 24 小时，典型 P50 为 2-6 小时。
- 叠加批处理 + 输入缓存：约为同步无缓存成本的 10%。
- 工作负载分流规则：只要能接受 24 小时延迟，就一律批处理。

## 关键术语

| 术语 | 大家怎么说 | 实际含义 |
|------|----------------|------------------------|
| Batch API | 「异步优惠」 | 五折优惠，配 24 小时处理时限 |
| JSONL | 「批处理格式」 | 每行一个 JSON 请求；OpenAI/Anthropic 标准 |
| Message Batches | 「Anthropic 批处理」 | Anthropic 批处理 API 的产品名 |
| Batch prediction | 「Vertex 批处理」 | Vertex AI 的批处理 API 产品 |
| Turnaround SLA | 「24 小时承诺」 | 是保证，而非典型值；典型为 2-6 小时 |
| Workload triage | 「交互性决策」 | 交互式 / 半交互式 / 批处理的路由决策 |
| Output schema | 「响应格式」 | 每家厂商各自的 JSONL 布局；不可移植 |
| Stacked discount | 「批处理 + 缓存」 | 两者同时生效时，约为无缓存同步账单的 10% |
