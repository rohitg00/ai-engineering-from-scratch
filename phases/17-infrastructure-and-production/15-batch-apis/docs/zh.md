# 批处理 API — 50% 折扣作为行业标准

> 每个主要提供商都提供了一个异步批处理 API，具有 50% 折扣和约 24 小时周转时间。OpenAI、Anthropic、Google 和大多数推理平台（Fireworks batch tier、Together batch）都实现了相同的模式。将批处理与提示缓存和隔夜管道堆叠，成本降至同步未缓存成本的约 10%。规则极其简单：如果它不是交互式的，它就应该属于批处理。内容生成管道、文档分类、数据提取、报告生成、批量标记、目录标记——任何容忍 24 小时延迟的工作都是放在桌面上未拿的钱，直到它转移到批处理。2026 年的生产模式是将每个新的 LLM 工作负载分流到三个通道：交互式（带缓存的同步）、半交互式（带回退的异步队列）、批处理（隔夜、堆叠缓存输入）。假装是交互式但容忍数分钟延迟的工作负载浪费最多。

**类型：** 学习
**语言：** Python（标准库，简单的批处理与同步成本模拟器）
**先修要求：** 阶段 17 · 14（提示和语义缓存）
**时间：** 约 45 分钟

## 学习目标

- 说出三个提供商批处理 API（OpenAI、Anthropic、Google）和常见的 50% 折扣 + 24 小时周转保证。
- 计算隔夜分类工作负载上堆叠批处理 + 缓存输入的成本，并与同步未缓存基线进行比较。
- 将工作负载分流到交互式 / 半交互式 / 批处理，并证明通道的合理性。
- 说出两个陷阱：部分交互性（用户期望快于 24 小时）和输出模式漂移（批处理文件格式因提供商而异）。

## 问题

你的团队发布了一个夜间报告生成管道。50,000 个文档，分别总结每个，聚类摘要，起草执行简报。同步运行需要 4 小时，成本 2,000 美元/晚。你听说了批处理 API。

批处理为你节省 50%。你还在系统提示（在所有 50k 次调用中共享）上启用了提示缓存。堆叠后，账单降至 180 美元/晚——约为基线的 9%。相同的管道，三个配置更改。

批处理是 LLM 成本工具包中最便宜的杠杆，但没有人拉动。原因主要是组织性的：团队认为"实时"，而 SLA 实际上是"明早之前"。本课是关于不在桌面上留下 90% 的账单。

## 概念

### 三个批处理 API

**OpenAI Batch API**：JSONL 文件上传，包含请求列表。承诺 24 小时周转（实际上通常约 2-8 小时）。输入和输出 token 50% 折扣。`/v1/batches` 端点。符合缓存条件的输入在顶部也获得缓存输入定价。

**Anthropic Message Batches**：JSONL 上传。24 小时周转。50% 折扣。支持 `cache_control`——缓存写入是显式的，读取在批处理内自动发生。

**Google Vertex AI Batch Prediction**：BigQuery 或 GCS 输入。Gemini 类似的 50% 折扣。与 Vertex 管道集成。

### 语义：异步，不慢

批处理是"我承诺在 24 小时内返回"——不是"这将需要 24 小时。"典型的 P50 是 2-6 小时。提供商在 GPU 库存未充分利用的离峰窗口期间调度你的批处理。

### 与缓存堆叠

一个具有相同的 4K token 系统提示的 50k 文档摘要：

- 同步未缓存：50000 ×（$input × 4000 + $output × 200）按全速率。
- 同步缓存：第一次写入后系统提示被缓存；剩余的 49999 次获得 10 倍更便宜的输入。
- 批处理缓存：以上所有加上读取和写入的 50% 折扣。

堆叠：批处理 + 缓存 = 同步未缓存账单的约 10%。任何在夜间运行并具有共享系统提示的工作负载都应使用此。

### 工作负载分流

**交互式**——用户等待响应。TTFT 很重要。带提示缓存的同步调用。不能批处理。

**半交互式**——用户提交任务，几分钟后回来查看。带回退到同步的异步队列（如果批处理不可用）。想想中等量的 RAG 索引。

**批处理**——用户期望"明早"或"下小时"的结果。内容管道、大规模分类、离线分析。始终批处理，始终堆叠缓存。

常见错误：因为管道是生产环境就将一切分类为交互式。生产环境不是延迟规范——SLA 才是。

### 部分交互性陷阱

某些功能看起来是交互式的，但容忍 5-10 分钟。示例：带有"刷新"按钮的夜间客户运行状况报告。用户点击刷新；等待 10 分钟没问题。团队将其作为同步交付。50 个并发刷新花费的成本是批处理并通过电子邮件交付的 10 倍。

要问的问题："24 小时对此用户意味着什么？"如果答案是"他们不会注意到"，就将其批处理。

### 输出模式陷阱

批处理文件格式因提供商而异：

- OpenAI：JSONL，每行一个请求。
- Anthropic：JSONL，每行一条消息；响应格式嵌入。
- Vertex：带有 TFRecord 的 BigQuery 表或 GCS 前缀。

跨提供商编写"一个批处理客户端"意味着每个提供商都需要适配器代码。宣传多提供商批处理的网关（Portkey、LiteLLM 某些层级）仍然精简包装原始格式。

### 你应该记住的数字

- 跨提供商的批处理折扣：输入 + 输出统一 50% 折扣。
- 周转 SLA：保证 24 小时，典型的 P50 是 2-6 小时。
- 堆叠的批处理 + 缓存输入：同步未缓存成本的约 10%。
- 工作负载分流规则：如果 24 小时延迟可接受，始终批处理。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| Batch API | "异步折扣" | 50% 折扣，24 小时周转 |
| JSONL | "批处理格式" | 每行一个 JSON 请求；OpenAI/Anthropic 标准 |
| Message Batches | "Anthropic 批处理" | Anthropic 的批处理 API 产品名称 |
| Batch prediction | "Vertex 批处理" | Vertex AI 的批处理 API 产品 |
| Turnaround SLA | "24 小时承诺" | 保证，不是典型；典型是 2-6 小时 |
| Workload triage | "交互性决策" | 交互式 / 半交互式 / 批处理路由决策 |
| Output schema | "响应格式" | 每提供商的 JSONL 布局；不可移植 |
| Stacked discount | "批处理 + 缓存" | 当两者都应用时，同步未缓存账单的约 10% |
