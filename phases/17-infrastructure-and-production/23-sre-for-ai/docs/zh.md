# 23 · 面向 AI 的 SRE —— 多智能体事件响应、Runbook 与预测性检测

> AI SRE 使用由基础设施数据（日志、Runbook、服务拓扑）通过 RAG 做知识增强的大语言模型,来实现调查、文档编写和协调阶段的自动化。2026 年的架构模式是多智能体编排（Multi-Agent Orchestration）——由一名监督者（Supervisor）协调专注于不同领域的智能体（日志、指标、Runbook）;AI 提出假设与查询,人类批准涉及判定的操作。Datadog Bits AI 与 Azure SRE Agent 以托管产品形式发布上述能力。Runbook 正在演进：NeuBird Hawkeye 采用对抗式评估（两个模型独立分析同一事件;一致即高置信,分歧即不确定）;运营记忆（Operational Memory）能够跨越团队成员变动持续保存。自动化修复保持谨慎：AI 建议,人类批准。全自主操作被限定在狭窄范围内（重启 Pod、回滚特定部署）,并配有严格护栏——任何声称"设置后便可不管"的人都在过度承诺。新兴前沿：事前预测。MIT 研究报告称,在历史日志+GPU 温度+API 错误模式上训练的大语言模型提前 10–15 分钟预测了 89% 的中断事件。预测：到 2026 年底,95% 的企业级大语言模型将具备自动故障转移能力。

**类型：** 学习
**语言：** Python
**前置：** Phase 17 · 13（可观测性）,Phase 17 · 24（混沌工程）
**时长：** 约 60 分钟

## 学习目标

- 画出多智能体 AI SRE 架构图：监督者+领域智能体（日志、指标、Runbook）+ 人类审批门。
- 解释为什么自动化修复被限定在狭窄范围（重启 Pod、回滚部署）而非广泛动作（重新设计服务架构）。
- 说出对抗式评估模式（NeuBird Hawkeye）：两个模型达成一致 = 高置信;不一致 = 升级到人工。
- 引用 MIT 89% 早期检测结果及运营约束：没有执行动作的预测仅仅是仪表盘。

## 问题所在

一名值班工程师在凌晨 3 点被呼叫。"结账服务错误率偏高。"他检查 Datadog、Loki、三份 Runbook、部署日志。30 分钟后,他意识到根因是 vLLM 的 KV 缓存飙升导致的 OOM。他重启 Pod;错误清除。

在 2026 年,上述调查的前 20 分钟已可自动化。按服务聚合日志、关联到最近的部署、与 Runbook 做匹配——这些全部是 RAG + 工具使用的范畴。一个受监督的智能体可以在人类打开 Datadog 之前完成第一轮分类分析并呈现假设。

完全自主修复则是另一回事。重启 Pod：安全。如果策略允许,扩容 GPU 池：安全。重新设计服务架构：绝对不行。这门学科的核心就是画清那条窄线。

## 核心概念

### 多智能体架构

```
          事件（Incident）
              │
              ▼
         监督者（Supervisor）
         /    |    \
        ▼     ▼     ▼
   日志智能体  指标智能体  Runbook 智能体
       │     │     │
       └─────┴─────┘
              │
              ▼
       假设 + 证据
              │
              ▼
         人类审批
              │
              ▼
       动作（限定范围）
```

监督者将事件拆分为子查询。各领域智能体拥有工具访问权限（日志搜索、PromQL、文档检索）。监督者综合信息,向人类呈现假设与证据。人类批准或重定向。

### 自动化修复的范围

**安全（窄）**：重启 Pod,回滚特定部署,在预批准范围内调整资源池,启用预批准的特性开关。

**不安全（广）**：变更服务拓扑,修改资源限制,部署新代码,变更 IAM,变更数据库。

任何声称"设置后便可不管"的人都在过度承诺。安全动作集随着 AI SRE 的成熟会不断扩大,但边界始终存在。

### 对抗式评估（NeuBird Hawkeye）

两个模型独立分析同一事件。如果它们在根因上达成一致,置信度就很高。如果出现分歧,则将两者假设同时展现给人类,升级处理。这是一个简单模式,却是对抗幻觉根因的有效过滤器。

### 运营记忆

团队成员流动是传统 SRE 的无形杀手——部落知识随之流失。AI SRE 将 Runbook + 事后复盘（Post-Mortem）存入向量数据库;智能体在每次新事件发生时都进行检索。当新工程师加入时,AI 掌握了完整历史。

### 事前预测

MIT 2025 年研究：在历史日志、GPU 温度与 API 错误模式上训练的 LLM 在测试集上提前 10–15 分钟预测了 89% 的中断事件。

现实检验：没有执行动作的预测仅仅是仪表盘。真正的运营问题是"当我们做出预测,我们要采取什么行动？"预先排水？呼叫？自动伸缩？答案是特定策略驱动的。

### 2026 年产品格局

- **Datadog Bits AI** —— DataDog 内的托管 SRE 副驾驶。
- **Azure SRE Agent** —— Azure 原生。
- **NeuBird Hawkeye** —— 对抗式评估 + 运营记忆。
- **PagerDuty AIOps** —— 分类与去重。
- **Incident.io Autopilot** —— 事件指挥官 + 协调。

### Runbook 代码化

Runbook 正从 Confluence 页面演进为带结构化章节的版本化 Markdown（症状、假设、验证、行动）。结构化 Runbook 能提供更好的 RAG 检索效果。任何 AI SRE 推行的第一步,都是将非结构化 Runbook 转为结构化形式。

### 你应该记住的数字

- MIT 早期检测：89% 的中断事件,10–15 分钟提前量。
- 多智能体分类：监督者 +（日志、指标、Runbook）+ 人类。
- 安全自动化修复集合：重启 Pod、回滚部署、在限定范围内伸缩。
- 对抗式评估：两个模型独立分析;达成一致 = 高置信。

## 练习

1. 运行 `code/main.py`。如果日志智能体与指标智能体存在分歧,监督者如何解决？
2. 为你的服务定义三个"安全"的自动修复动作。逐一论证。
3. 撰写一份结构化 Runbook 模板：包括所需章节、必填字段、验证命令。
4. 预测性检测以 12 分钟提前量触发报警。你的策略是什么——单独呼叫,预先排水,还是两者？

## 关键术语

| 术语 | 人们常这么说 | 实际含义 |
|-|-|-|
| AI SRE | 「值班智能体」 | 基于 LLM 的事件调查与协调 |

| 领域智能体 | 「专长智能体」 | 拥有工具访问权限的子智能体（日志、指标、Runbook） |

| 运营记忆 | 「向量化 Runbook」 | 存入向量数据库的事后复盘 + Runbook,供 RAG 检索 |
| 对抗式评估 | 「双模型交叉验证」 | 独立分析;一致 = 高置信 |
| NeuBird Hawkeye | 「对抗式那个」 | 采用对抗式评估 + 运营记忆的产品 |
