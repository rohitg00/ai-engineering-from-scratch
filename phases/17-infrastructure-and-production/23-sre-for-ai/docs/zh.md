# AI 的 SRE — 多智能体事件响应、运行手册、预测性检测

> AI SRE 使用通过 RAG 基于基础设施数据（日志、运行手册、服务拓扑）的 LLM 来自动化调查、文档和协调阶段。2026 年的架构模式是多智能体编排——由监督者协调的专用智能体（日志、指标、运行手册）；AI 提出假设和查询，人类批准判断调用。Datadog Bits AI 和 Azure SRE Agent 将其作为托管产品提供。运行手册正在演变：NeuBird Hawkeye 使用对抗性评估（两个模型分析同一事件；一致 = 置信度，不一致 = 不确定性）；操作记忆跨越团队变更持续存在。自动修复保持谨慎：AI 建议，人类批准。完全自主操作范围狭窄（重启 pod、回滚特定部署），带有严格的防护栏——任何销售"设置后即忘记"的人都在夸大其词。新兴前沿：事件前预测。MIT 研究报告称，在历史事件日志 + GPU 温度 + API 错误模式上训练的 LLM 提前 10-15 分钟预测了 89% 的中断。预测：到 2026 年底，95% 的企业 LLM 具有自动故障转移。

**类型：** 学习
**语言：** Python（标准库，简单的多智能体事件分类模拟器）
**先决条件：** 阶段 17 · 13（可观测性）、阶段 17 · 24（混沌工程）
**时间：** 约 60 分钟

## 学习目标

- 绘制多智能体 AI SRE 架构图：监督者 + 专用智能体（日志、指标、运行手册）+ 人类批准门。
- 解释为什么自动修复范围狭窄（重启 pod、恢复部署）而不是广泛（重新架构服务）。
- 说出对抗性评估模式（NeuBird Hawkeye）：两个模型一致 = 置信度；不一致 = 升级。
- 引用 MIT 89% 的早期检测结果和操作约束：没有驱动器的预测只是仪表板。

## 问题

待命工程师在凌晨 3 点收到页面。"结账错误率高。"他们检查 Datadog、Loki、三个运行手册、部署日志。30 分钟后，他们意识到根本原因是来自 KV 缓存峰值的 vLLM OOM。他们重启 pod；错误清除。

在 2026 年，调查的前 20 分钟是可自动化的。按服务对日志进行分组、与最近的部署相关联、与运行手册进行匹配——所有这些都是 RAG + 工具使用。监督智能体可以在人类打开 Datadog 之前进行首次分类并提出假设。

完全自主修复是一个不同的问题。重启 pod：安全。扩展 GPU 池：如果策略允许则安全。重新架构服务：绝对不行。纪律是划定狭窄的界限。

## 概念

### 多智能体架构

```
          Incident
             │
             ▼
        Supervisor
        /    |    \
       ▼     ▼     ▼
  Log agent  Metric agent  Runbook agent
       │     │     │
       └─────┴─────┘
             │
             ▼
        Hypothesis + evidence
             │
             ▼
        Human approval
             │
             ▼
        Action (narrow set)
```

监督者将事件分解为子查询。专用智能体具有工具访问权限（日志搜索、PromQL、文档检索）。监督者综合，向人类提出假设 + 证据。人类批准或重新定向。

### 自动修复范围

**安全（狭窄）**：重启 pod，恢复特定部署，在预先批准的范围内扩展池，启用预先批准的功能标志。

**不安全（广泛）**：更改服务拓扑，修改资源限制，部署新代码，更改 IAM，更改数据库。

任何销售"设置后即忘记"的人都在夸大其词。随着 AI SRE 的成熟，安全集不断扩大，但边界是真实的。

### 对抗性评估（NeuBird Hawkeye）

两个模型独立分析同一事件。如果他们就根本原因达成一致，则置信度高。如果他们不同意，则升级到有两条假设可见的人类。

简单的模式，针对幻觉根因的有效过滤器。

### 操作记忆

团队流失是传统 SRE 的静默杀手——部落知识离开。AI SRE 将运行手册 + 事后剖析存储在向量数据库中；智能体在每次新事件时检索。当新工程师加入时，AI 拥有完整的历史记录。

### 事件前预测

MIT 2025 研究：在测试集上，在历史事件日志、GPU 温度、API 错误模式上训练的 LLM 提前 10-15 分钟预测了 89% 的中断。

现实检查：没有驱动器的预测是仪表板。操作问题是"当我们预测时，我们做什么？"抢占式排空？寻呼机？自动扩展？答案是特定于策略的。

### 2026 年的产品

- **Datadog Bits AI**——Datadog 内的托管 SRE 副驾驶。
- **Azure SRE Agent**——Azure 原生。
- **NeuBird Hawkeye**——对抗性评估 + 操作记忆。
- **PagerDuty AIOps**——分类 + 重复数据删除。
- **Incident.io Autopilot**——事件指挥官 + 协调。

### 运行手册即代码

运行手册从 Confluence 页面演变为具有结构化部分（症状、假设、验证、操作）的版本化 markdown。结构化运行手册提供更好的 RAG 检索。通过将方程序化运行手册转换为结构化来启动任何 AI-SRE 推出。

### 你应该记住的数字

- MIT 早期检测：89% 的中断，提前 10-15 分钟。
- 多智能体分类：监督者 +（日志、指标、运行手册）+ 人类。
- 安全自动修复集：重启 pod、恢复部署、在范围内扩展。
- 对抗性评估：两个独立模型；一致 = 置信度。

## 使用它

`code/main.py` 模拟多智能体分类：日志智能体发现错误，指标智能体发现 CPU 峰值，运行手册智能体匹配到已知问题。监督者对假设进行排名。

1. 运行 `code/main.py`。如果日志和指标智能体不同意怎么办？监督者如何解决？

3. 编写结构化运行手册模板：部分、必填字段、验证命令。
4. 预测性检测在提前 12 分钟触发。你的策略是什么——寻呼机、抢占式排空还是两者兼而有之？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------------|------------------------|
| AI SRE | "待命智能体" | LLM 支持的事件调查 + 协调 |
| Supervisor agent | "编排器" | 将事件分解为子查询的顶层智能体 |
| Specialized agent | "领域智能体" | 具有工具访问权限的子智能体（日志、指标、运行手册） |
| Auto-remediation | "AI 修复它" | 狭窄的预先批准的操作；不是广泛的重新架构 |
| Operational memory | "向量运行手册" | 用于 RAG 的向量数据库中的事后剖析 + 运行手册 |
| Adversarial eval | "双模型检查" | 独立分析；一致 = 置信度 |
| NeuBird Hawkeye | "对抗性那个" | 具有对抗性评估 + 记忆模式的产品 |
| Bits AI | "Datadog 的 SRE 智能体" | Datadog 托管的 AI SRE |
| Pre-incident prediction | "早期检测" | 中断预测提前 10-15 分钟 |
