# 长时间运行的后台代理：持久化执行

> 生产环境中的长程代理不会在 `while True` 中运行。每次 LLM 调用都变成一个带有检查点、重试和回放的活动。Temporal 的 OpenAI Agents SDK 集成于 2026 年 3 月正式发布。Claude Code Routines（Anthropic）运行预定的 Claude Code 调用，无需持久的本地进程。会话在人类输入时暂停，在部署中存活，并从由 `thread_id` 键控的最新检查点恢复。在新的人机工程学背后是一个旧模式——工作流编排——加上一个新输入：LLM 调用作为非确定性活动，必须在恢复时确定性回放。

**类型：** 学习
**语言：** Python（标准库，最小持久化执行状态机）
**前置条件：** 第 15 阶段 · 10（权限模式），第 15 阶段 · 01（长程代理）
**时间：** ~60 分钟

## 问题

考虑一个运行四小时的代理。它调用三个工具，提示用户两次，并进行四十次 LLM 调用。中途，它运行的主机重新启动。会发生什么？

- 在简单的 `while True` 循环中：一切丢失。运行从头开始。三个工具调用（具有真实的副作用）再次执行。用户再次被提示他们已经批准的事情。四十次 LLM 调用被重新计费。
- 使用持久化执行：运行从最近的检查点恢复。已完成的活动不会重新执行；它们的结果从持久日志中回放。用户不会重新批准他们已经批准的事情。已经进行的 LLM 调用不会被重新计费。

这是工作流引擎已经发布十年的相同模式（Temporal、Cadence、Uber 的 Cherami）。新的是 LLM 调用现在是一种活动——非确定性、昂贵、有副作用——它们干净地适合这个模式。

本课的主题：长程可靠性会衰减（METR 观察到“35 分钟退化”——成功率随时间跨度大致呈二次下降）。持久化执行使运行能够超过可靠性配置文件的支撑，如果设计正确，这是一种新的安全失败方式；如果设计错误，则是一种新的不安全失败方式。

## 概念

### 活动、工作流和回放

- **工作流**：确定性编排代码。定义活动序列、分支、等待。必须是确定性的，以便可以从事件日志中回放而不会产生意外的分歧。
- **活动**：非确定性的、可能失败的单元工作。LLM 调用、工具调用、文件写入、HTTP 请求。每个活动都记录其输入和（完成后）输出。
- **事件日志**：持久化后端存储。记录每个活动的开始、完成、失败、重试，以及每个工作流决策。
- **回放**：恢复时，工作流代码从开始重新运行；每个已完成的活动返回其记录的结果而不重新执行。只有未完成的活动才会实际运行。

这与 React 针对虚拟 DOM 重新渲染，或 Git 从提交重建工作树的形状相同。编排器中的确定性是使持久化变得廉价的原因。

### 为什么 LLM 调用适合这个模式

LLM 调用是：
- 非确定性的（温度 > 0；即使温度为 0，跨模型版本也会漂移）。
- 昂贵的（金钱和延迟）。
- 可能失败的（速率限制、超时）。
- 有副作用的（如果它们调用工具）。

这正是活动配置文件。将每次 LLM 调用包装为活动，可以获得指数退避重试、跨重启的检查点，以及可回放的调试跟踪。

### 由 `thread_id` 键控的检查点

LangGraph、Microsoft Agent Framework、Cloudflare Durable Objects 和 Claude Code Routines 都收敛于相同的 API 形状：`thread_id`（或等效物）标识会话；每个状态转换持久化到后端（PostgreSQL 默认，SQLite 用于开发，Redis 用于缓存）；恢复读取最新的检查点。

后端选择很重要：

- **PostgreSQL**：持久化、可查询、在部署中存活。LangGraph 的默认选择。
- **SQLite**：仅本地开发；跨主机丢失数据。
- **Redis**：快速但短暂，除非配置 AOF/快照。
- **Cloudflare Durable Objects**：透明分布式；由唯一键限定范围；存活数小时到数周。

### 人类输入作为一等状态

提出-提交（第 15 课）需要一个持久的“等待人类”状态。工作流暂停，外部队列保存待处理请求，批准从该点精确恢复。没有持久化，这是尽力而为；有了持久化，隔夜批准到达，工作流在早上继续。

### 35 分钟退化

METR 观察到，每个测量的代理类别在超过约 35 分钟的连续操作后可靠性都会衰减。将任务持续时间翻倍大致会使失败率翻两番。持久化执行不能解决这个问题；它让你运行超过可靠性配置文件的支撑。安全模式是将持久化与需要重新进入时新鲜人工介入的检查点相结合，以及与预算终止开关（第 13 课）相结合，无论挂钟时间如何都限制总计算量。

### 持久化执行是错误答案的情况

- 运行时间短于几分钟，没有人类输入。开销 > 收益。
- 严格只读的信息检索。
- 正确性要求在一个上下文窗口内端到端的任务（某些推理任务；某些一次性生成）。

## 使用

`code/main.py` 在标准库 Python 中实现了一个最小持久化执行引擎。它支持：

- `@activity` 装饰器，将输入和输出记录到 JSON 事件日志。
- 一个编排活动的 workflow 函数。
- 一个 `run_or_replay(workflow, event_log)` 函数，回放已完成的活动而不重新执行它们。

驱动程序模拟一个三活动工作流，中途崩溃，并显示 (a) 简单重试重新执行所有内容与 (b) 回放仅运行缺失活动之间的区别。

## 交付

`outputs/skill-durable-execution-review.md` 审查拟议的长运行代理部署的正确持久化执行形状：活动、确定性、检查点后端、人类输入状态和恢复时人工介入策略。

## 练习

1. 运行 `code/main.py`。观察简单重试和回放之间的活动执行计数差异。更改崩溃点并显示回放计数相应变化。

2. 将玩具引擎转换为显式使用 `thread_id`。模拟两个共享引擎的并发会话，并确认它们的事件日志不会冲突。

3. 在玩具引擎中取一个活动。引入非确定性（工作流决策中的挂钟时间戳）。演示回放时的分歧。解释真实引擎如何处理这个问题（副作用注册、`Workflow.now()` API）。

4. 阅读 LangChain “生产深度代理背后的运行时”文章。列出运行时持久化的每个状态，并命名每个状态覆盖的失败模式。

5. 为一个 6 小时的自主编码任务设计检查点策略。你在哪里检查点？崩溃后恢复是什么样？什么需要新鲜的人工介入？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| Workflow | "代理的脚本" | 确定性编排代码；可从事件日志回放 |
| Activity | "一个步骤" | 非确定性单元（LLM 调用、工具调用）；前后记录 |
| Event log | "后端存储" | 每个状态转换的持久记录 |
| Replay | "恢复" | 重新运行工作流；已完成的活动返回记录的结果而不重新执行 |
| Checkpoint | "保存点" | 由 thread_id 键控的持久状态；恢复时最新胜出 |
| thread_id | "会话键" | 限定持久状态范围的标识符 |
| 35-minute degradation | "可靠性衰减" | METR：成功率随时间跨度大致呈二次下降 |
| Non-determinism | "回放时漂移" | 挂钟、随机、LLM 输出；必须注册为副作用 |
