# 长时间运行后台智能体：持久执行

> 生产长时域智能体不在 `while True` 中运行。每个 LLM 调用都成为具有检查点、重试和重放的活动。Temporal 的 OpenAI Agents SDK 集成在2026年3月 GA。Claude Code 例程（Anthropic）运行计划的 Claude Code 调用，而没有持久本地进程。会话在人工输入时暂停，在部署中存活，并从由 `thread_id` 键控的最新检查点恢复。在新的工效学背后坐着一种旧模式——工作流编排——带有一个新输入：LLM 调用作为非确定性活动，必须在恢复时确定性地重放。

**类型：** 学习
**语言：** Python（标准库，最小持久执行状态机）
**前置条件：** 第15阶段 · 10（权限模式），第15阶段 · 01（长时域智能体）
**时间：** 约60分钟

## 问题

考虑一个运行四小时的智能体。它调用三个工具，提示用户两次，并进行四十次 LLM 调用。中途，它运行的主机重启。会发生什么？

- 在朴素的 `while True` 循环中：一切都丢失了。运行从头开始重新启动。三个工具调用（具有真实副作用）再次执行。用户再次被提示他们已经批准的事情。四十次 LLM 调用被重新计费。
- 使用持久执行：运行从最近的检查点恢复。已经完成的活动不会重新执行；它们的结果从持久日志中重放。用户不会重新批准他们已经批准的事情。已经进行的 LLM 调用不会被重新计费。

这与工作流引擎已经出货十年的模式相同（Temporal、Cadence、Uber 的 Cherami）。新的是，LLM 调用现在是一种活动——非确定性的、昂贵的、具有副作用的——并且它们清晰地适应这种模式。

课程的running主题：长时域可靠性衰减（METR 观察到"35分钟退化"——成功率随时域大致二次方下降）。持久执行使得运行可以比可靠性配置文件支持的更长，如果设计正确，这是一种安全失败的新方式，如果设计错误，则是不安全地失败。

## 概念

### 活动、工作流和重放

- **工作流**：确定性编排代码。定义活动的顺序、分支、等待。必须是确定性的，以便可以从事件日志重放而不会有令人惊讶的分歧。
- **活动**：非确定性、可能失败的工作单元。LLM 调用、工具调用、文件写入、HTTP 请求。每个活动都以其输入和（一旦完成）输出记录。
- **事件日志**：持久后备存储。每个活动开始、完成、失败、重试和每个工作流决策都被记录。
- **重放**：在恢复时，工作流代码从开始重新运行；每个已经完成的活动返回其记录的结果而不重新执行。只有尚未完成的活动才实际运行。

这与针对虚拟 DOM 的 React 重新渲染或 Git 从提交重建工作树的形状相同。编排器中的确定性使得持久性便宜。

### 为什么 LLM 调用适应模式

LLM 调用是：

- 非确定性的（temperature > 0；甚至 temperature 0 跨模型版本漂移）。
- 昂贵的（金钱和延迟）。
- 可能失败的（速率限制、超时）。
- 具有副作用的（如果它们调用工具）。

这正是活动配置文件。将每个 LLM 调用包装为活动为你提供指数退避重试、跨重启的检查点以及用于调试的可重放跟踪。

### 由 `thread_id` 键控的检查点

LangGraph、Microsoft 智能体框架、Cloudflare Durable Objects 和 Claude Code 例程都收敛于相同的 API 形状：`thread_id`（或等效项）标识会话；每个状态转换持久化到后端（PostgreSQL 默认，开发用 SQLite，缓存用 Redis）；恢复读取最新检查点。

后端选择很重要：

- **PostgreSQL**：持久的、可查询的、在部署中存活。LangGraph 的默认设置。
- **SQLite**：仅本地开发；跨主机丢失数据。
- **Redis**：快速但是临时的，除非配置了 AOF/快照。
- **Cloudflare Durable Objects**：透明分布；由唯一键限定范围；存活数小时到数周。

### 人工输入作为一等状态

提议然后提交（第15课）需要持久的"等待人工"状态。工作流暂停，外部队列保存待处理请求，批准从确切点恢复。没有持久性，这是尽力而为的；有了它，一夜之间的批准到达，工作流在早晨拾取。

### 35分钟退化

METR 观察到，每个测量的智能体类别都显示超过约35分钟连续操作的可靠性衰减。将任务持续时间加倍大致使失败率增加四倍。持久执行不能修复这一点；它让你运行比可靠性配置文件支持的时间更长。安全模式是将持久性与在重新进入时需要新的 HITL 的检查点相结合，以及与无论挂钟时间如何都限制总计算的预算终止开关（第13课）相结合。

### 持久执行是错误的答案时

- 运行短于几分钟而没有人工输入。开销 > 收益。
- 严格只读信息检索。
- 正确性需要在单个上下文窗口内端到端的任务（一些推理任务；一些一次性生成）。

## 使用

`code/main.py` 在标准库 Python 中实现最小持久执行引擎。它支持：

- `@activity` 装饰器，将输入和输出记录到 JSON 事件日志。
- 排序活动的工作流函数。
- `run_or_replay(workflow, event_log)` 函数，重放已完成的活动而不重新执行它们。

驱动程序模拟一个三活动工作流，中途崩溃，并显示（a）朴素重试重新执行所有内容 vs（b）仅运行缺失活动的重放。

## 实战

`outputs/skill-durable-execution-review.md` 审查提议的长时间运行智能体部署的正确持久执行形状：活动、确定性、检查点后端、人工输入状态以及恢复时 HITL 策略。

## 练习

1. 运行 `code/main.py`。观察朴素重试与重放之间的活动执行计数差异。更改崩溃点并显示重放计数相应变化。

2. 将玩具引擎转换为显式使用 `thread_id`。模拟两个并发会话共享引擎，并确认它们的事件日志不会冲突。

3. 取玩具引擎中的一个活动。引入非确定性（工作流决策内的挂钟时间戳）。演示重放时的分歧。解释真实引擎如何处理这一点（副作用注册、`Workflow.now()` API）。

4. 阅读 LangChain"生产深度智能体背后的运行时"帖子。列出运行时持久化的每个状态，并命名每个覆盖的失败模式。

5. 为6小时自主编码任务设计检查点策略。你在哪里检查点？崩溃时恢复看起来像什么？什么需要新的 HITL？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| 工作流（Workflow） | "智能体的脚本" | 确定性编排代码；可从事件日志重放 |
| 活动（Activity） | "一个步骤" | 非确定性单元（LLM 调用、工具调用）；前后记录 |
| 事件日志（Event Log） | "后备存储" | 每个状态转换的持久记录 |
| 重放（Replay） | "恢复" | 重新运行工作流；已完成的活动返回记录结果而不重新执行 |
| 检查点（Checkpoint） | "保存点" | 由 thread_id 键控的持久化状态；恢复时最新获胜 |
| thread_id | "会话密钥" | 限定持久化状态的标识符 |
| 35分钟退化（35-Minute Degradation） | "可靠性衰减" | METR：成功率随时域大致二次方下降 |
| 非确定性（Non-Determinism） | "重放时漂移" | 挂钟、随机、LLM 输出；必须注册为副作用 |
