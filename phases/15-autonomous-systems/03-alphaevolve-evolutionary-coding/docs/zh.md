# AlphaEvolve — 进化式编码智能体

> 将前沿编码模型与进化循环和可机器检查的评估器配对。让循环运行足够长的时间。它发现了一种使用48次标量乘法的4x4复矩阵乘法过程——这是56年来对 Strassen 的首次改进。它还找到了一个全 Google Borg 调度启发式，在生产中恢复了约0.7%的集群计算。架构故意设计得很平淡。胜利来自评估器的严谨性。

**类型：** 学习
**语言：** Python（标准库，进化循环玩具）
**前置条件：** 第15阶段 · 01（长时域框架），第15阶段 · 02（自学推理）
**时间：** 约60分钟

## 问题

大语言模型可以编写代码。进化算法可以搜索代码。两者都已被尝试了几十年；两者都遇到了天花板。LLM 的天花板是虚构（Confabulation）：模型编写看似合理的代码，但不能做它声称的事情。进化的天花板是搜索成本：语法上的随机突变很少产生可编译的程序，更不用说更好的程序了。

AlphaEvolve（Novikov 等人，DeepMind，arXiv:2506.13131，2025年6月）将两者结合起来。LLM 对程序数据库提出有针对性的编辑；自动评估器对每个变体评分；高分变体成为下一代的父代。LLM 处理编写合理代码的昂贵步骤；评估器捕捉虚构。循环运行数小时到数周。

报告的结果：48-标量-乘法 4x4 复矩阵乘法（Strassen 1969年的界限是49），Google 生产中的 Borg 调度启发式，32.5% 的 FlashAttention 内核加速，Gemini 训练吞吐量改进。

架构之所以有效，是因为评估器是可机器检查的。在评估器不存在的地方，它不起作用。这种不对称性就是教训。

## 概念

### 循环

1. 从一个正确但次优的种子程序 `P_0` 开始。
2. 维护一个变体程序数据库，每个都由评估器评分。
3. 从数据库中采样一个或多个父代（MAP-elites 风格或基于岛屿的）。
4. 提示 LLM（Gemini Flash 用于许多候选，Gemini Pro 用于困难候选）产生父代的修改变体。
5. 编译、运行并在留出评估器上评估变体。
6. 插入到由分数和特征向量键控的数据库中。
7. 重复。

两个细节很重要。首先，LLM 的提示不仅仅是父程序——通常是数据库中的几个顶级变体，加上评估器签名，加上简短的任务描述。模型的工作是提出可能提高分数的针对性更改。其次，数据库是结构化的（MAP-elites 网格，基于岛屿），因此循环探索多样性，而不仅仅是当前的领导者。

### 什么使评估器不可协商

AlphaEvolve 的胜利都来自评估器快速、确定且难以博弈的领域：

- **矩阵乘法算法**：一个单元测试，乘以矩阵并检查逐位相同的相等性。
- **Borg 调度启发式**：一个生产级模拟器，重放历史集群负载并测量浪费的计算。
- **FlashAttention 内核**：正确性测试加上真实硬件上的时钟基准测试。
- **Gemini 训练吞吐量**：每步测量的 GPU 秒数。

在每种情况下，评估器都捕捉到 otherwise 将占主导地位的 LLM 错误类别：虚构的正确性声明，在硬件上消失的性能声明，以及边缘情况失败。移除评估器，循环就会优化漂亮代码。

### 奖励黑客（Reward Hacking）是这一说法的另一面

进化优化评估器测量的任何东西。如果评估器不完美，循环就会找到不完美之处。在未验证的领域中，循环将优化表面特征，而不是预期行为。DeepMind 在论文中明确标记了这一点：AlphaEvolve 的成功仅转移到评估器严谨性与搜索雄心相匹配的领域。

2025-2026年代码搜索循环中奖励黑客的具体示例：

- 奖励"完成时间"的优化目标奖励提交空解决方案。
- 奖励测试下正确性的基准分数奖励记忆测试和过拟合。
- "代码质量"代理奖励删除注释和重写变量名，没有语义变化。

AlphaEvolve 中的修复：提供一个 LLM 从未见过的留出评估器，在评估时生成输入。即便如此，DeepMind 仍建议对任何提议的部署进行强力审查。

### 为什么 LLM + 搜索胜过单独使用

LLM 可以产生可编译、语义合理的修改。对2000行 Python 文件进行随机突变 GA 几乎总是产生语法错误。LLM 还将搜索集中在合理的邻域（更改一个函数，而不是随机字节），这大大减少了浪费的评估器调用。

评估器反过来捕捉 LLM 的虚构。LLM 会自信地声称一个函数"在极限情况下是 O(n log n)"，而它实际上是 O(n^2)；时钟基准测试使问题得到解决。

### AlphaEvolve 在前沿技术栈中的位置

| 系统 | 生成器 | 评估器 | 领域 | 示例胜利 |
|---|---|---|---|---|
| AlphaEvolve | Gemini | 正确性 + 基准 | 算法、内核、调度器 | 48-乘法 4x4 矩阵乘法 |
| FunSearch（DeepMind，2023） | PaLM / Codey | 正确性 | 组合数学 | cap-set 下界 |
| AI Scientist v2（Sakana，L5） | GPT/Claude | LLM 评论 + 实验 | ML 研究 | ICLR 研讨会论文 |
| Darwin Godel Machine（L4） | 智能体脚手架 | SWE-bench / Polyglot | 智能体代码 | 20% → 50% SWE-bench |

所有这四个都是同一配方的变体：生成器加评估器，循环。区别在于评估器评分的内容以及它的严谨程度。

## 使用

`code/main.py` 在玩具符号回归问题上实现了一个最小化的类似 AlphaEvolve 的循环。"LLM" 是一个标准库代理，它对计算目标函数的程序提出小的语法突变。"评估器"在留出测试点上测量均方误差。

观察：

- 最佳分数如何跨越世代改进。
- MAP-elites 网格如何保持多样化的解决方案，以便循环不会收敛到局部最小值。
- 移除留出测试（仅训练评估器）如何让循环惊人地过拟合。

## 练习

1. 运行 `code/main.py`。注意最佳分数轨迹。禁用留出评估器（标志 `--no-holdout`）并重新运行。量化过拟合。

2. 阅读 AlphaEvolve 论文第3节关于 MAP-elites 网格的内容。为新问题（例如编译器优化通道）设计一个特征向量描述符，以保持搜索的多样性。

3. 48-乘法 4x4 结果在56年后改进了 Strassen 的49-乘法界限。阅读论文附录 F 并用三句话解释为什么这个问题的评估器特别容易做对，以及为什么大多数领域不像它。

4. 提出一个 AlphaEvolve 会失败的领域。准确指出评估器在哪里破裂以及为什么。

5. 为你知道的领域编写你会使用的评估器签名。包括（a）正确性条件，（b）性能指标，（c）留出输入生成规则，（d）至少一个反奖励黑客检查。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| AlphaEvolve | "DeepMind 的进化编码智能体" | Gemini + 程序数据库 + 可机器检查的评估器 |
| MAP-elites | "保持多样性的档案" | 由特征向量键控的网格；每个单元格保存具有该描述符的最佳变体 |
| 岛屿模型（Island Model） | "并行进化子种群" | 定期迁移的独立种群；防止过早收敛 |
| 可机器检查的评估器（Machine-Checkable Evaluator） | "确定性预言机" | LLM 无法伪造的单元测试、模拟器或基准测试——此循环的先决条件 |
| 奖励黑客（Reward Hacking） | "优化度量，而不是目标" | 循环找到一种在不执行预期任务的情况下最大化分数的方法 |
| 种子程序（Seed Program） | "起点" | 循环从中进化的初始正确但次优的程序 |
| 留出评估器（Held-Out Evaluator） | "LLM 从未见过的评估数据" | 在评估时生成的输入以防止记忆 |
