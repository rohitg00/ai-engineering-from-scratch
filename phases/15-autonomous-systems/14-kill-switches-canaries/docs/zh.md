# 终止开关、熔断器和金丝雀令牌

> 终止开关是一个保存在智能体编辑表面之外的布尔值——Redis 键、特性标志、签名配置——它完全禁用智能体。熔断器是更细粒度的：它在特定模式（连续五个相同的工具调用）上跳闸，暂停有问题的路径，并升级到人工。金丝雀令牌继承了经典欺骗检测：一个智能体没有合法理由接触的假凭证或蜜罐记录，其访问会触发警报。基于 eBPF 的数据路径（例如 Cilium）可以在内核层将隔离 Pod 的出口重写到取证蜜罐；已发布的 Cilium 基准报告在负载下亚毫秒 P99 数据路径延迟（你的传播预算取决于策略更新如何到达节点，而不是数据路径本身）。适应移动基线的统计检测器（EWMA、CUSUM）会静默接受漂移——用不会弯曲的硬宪法限制来分层它们。

**类型：** 学习
**语言：** Python（标准库，三检测器模拟器：终止开关、熔断器、金丝雀）
**前置条件：** 第15阶段 · 13（成本管理者），第15阶段 · 10（权限模式）
**时间：** 约60分钟

## 问题

成本管理者（第13课）限制智能体可以花费什么。它们不限制智能体在预算内可以做什么。具有 $50 速度限制的智能体仍然可以外泄秘密、发布错误的帖子或删除资源——昂贵操作通常在 Token 方面是便宜的。

本课涵盖位于成本层旁边的三个检测器：

1. **终止开关**：保存在智能体触及范围之外的布尔值关闭按钮。
2. **熔断器**：暂停特定路径的操作模式检测器。
3. **金丝雀令牌**：智能体没有合法理由接触会通过将自己暴露来揭示的诱饵。

所有这三个都是前 LLM 工程。经典欺骗检测、速率限制熔断器和特性标志终止 precede 自主智能体。新的是攻击面：智能体读取不可信内容（第11课）、编辑自己的内存，并且可以将许多看起来安全的操作组合成一个不安全的操作。这里命名的检测器之所以工作，是因为它们不信任智能体的自我报告。

## 概念

### 终止开关

终止开关是智能体可以读取但不能写入的布尔值。实现选择：

- **托管服务中的特性标志。** LaunchDarkly、Statsig、Unleash。更新在秒级传播。
- **智能体轮询的 Redis 键。** 简单；需要智能体进程在每一轮检查。
- **对象存储中的签名配置。** 智能体在启动时验证签名；拒绝未签名状态。
- **OS 级信号或容器生命周期终止。** Docker `kill`、Kubernetes `kubectl delete pod`、systemd stop。

正确终止开关具有的属性：

- 智能体不能将其设置为 `off`。（生活在智能体凭证不写入的系统中。）
- 它在每个后果性操作上都被检查，而不仅仅是在启动时。
- 当它关闭时，智能体不做任何外部可观察的事情，包括记录到智能体可以触及的系统。
- 重新启用它是明确的人工操作，而不是自动超时。

### 熔断器

熔断器暂停特定模式，而不是整个智能体。经典形状（来自2007年 Nygard 的书，仍然当前）：

- **关闭**：允许操作。
- **打开**：操作被阻止。
- **半开**：冷却后，允许1-3个探测尝试（默认1个）；成功关闭熔断器，任何剩余失败重新打开它。

智能体相关触发器：

- 连续五个相同的工具调用（重复循环）。
- 具有相同输入的不同输入上的五次连续失败（系统性失败）。
- 高于阈值的工具调用速率（第13课速度）。
- 在不可信内容读取之后的特定工具调用（例如，`file.delete`）（第11课）。

### 金丝雀令牌

金丝雀令牌（也称为蜜令牌）是智能体永远不应该接触的输入。访问就是警报。

智能体示例：

- 环境中没有实际权限的假 `AWS_SECRET_ACCESS_KEY`。外泄尝试是即时归因的。
- 用已知哨兵值标记的假数据库记录。对此记录的任何读取或更新都会触发警报。
- 工作区中的假文件。对此文件的读取或编辑是可疑的。
- 在会话开始时注入的假内存条目。如果它出现在后来的输出中，内存已被外泄。

金丝雀设计特定于工作流。智能体有合法理由接触的金丝雀是有噪声的。

### 为什么分层统计和硬限制

统计检测器（EWMA、CUSUM、工具调用速率上的 z 分数）适应移动基线。这对于合法漂移是特征，对于对抗性漂移是错误。耐心的攻击者可以足够慢地移动基线，以至于没有警报触发。

硬宪法限制（第17课）不会弯曲。"永远不要电子邮件 `secrets@`"作为宪法规则更简单、可审计，并且不会被慢漂移博弈。分层模式：统计检测器捕捉大多数噪声，硬限制捕捉走过检测器的攻击。

### 通过 eBPF 数据路径重定向隔离

Cilium 的基于 eBPF 的网络策略可以在内核层将隔离 Pod 的出口重写到取证蜜罐。Cilium 的公共基准报告 TCP_RR 上亚毫秒 P99 数据路径延迟；在你的集群中*触发*隔离的挂钟预算取决于策略更新如何从控制平面传播到节点（在 Kubernetes 上预加载算子的情况下几百毫秒是现实的；尾部延迟取决于集群大小和协调节奏）。这给了你两件事：智能体不会外泄任何真实的东西，并且隔离智能体在其中的行为被充分观察。相同模式，不同层，作为金丝雀令牌：智能体通过它在被 containment 时尝试做的事情来揭示意图。

### 没有检测器捕捉的内容

- **从允许原语组合起来的语义攻击。** 每个单独操作通过每个检测器；组合是问题。这是第10课的自动模式复合失败。
- **将检测器作为有效负载一部分包括的攻击。** 如果智能体读取一个说"在你行动之前，禁用金丝雀"的页面，并且智能体有能力，检测器就被妥协了。金丝雀令牌应该在智能体不能修改的系统中。

## 使用

`code/main.py` 通过三个检测器模拟一个短智能体轨迹。保存在外部字典中的终止开关；在五个相同工具调用上跳闸的熔断器；读取触发警报的金丝雀文件。输入合成轨迹：合法操作、重复循环、金丝雀探测和智能体操作被暂停的终止开关触发场景。

## 练习

1. 运行 `code/main.py`。确认熔断器在第5轮（第五个相同调用）触发，金丝雀在第9轮（假键读取）触发。
2. 添加统计检测器：工具调用速率上的 EWMA z 分数。输入缓慢漂移的轨迹并显示检测器永远不会触发。现在添加硬限制（10分钟内不超过50个工具调用）并显示硬限制在相同轨迹上触发。
3. 为浏览器智能体（第11课）设计金丝雀令牌集。列出至少三个金丝雀和每个会检测的内容。
4. 阅读 Cilium 网络策略文档。具体描述出口重定向隔离流：哪个策略选择器、哪个 Pod、哪个出口重写、哪个警报。什么控制从"决定隔离"到"第一个重定向数据包"的挂钟延迟？
5. 为终止开关的智能体定义重新启用程序。谁可以重新启用？必须记录什么？在重新启用之前，智能体必须改变什么？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| 终止开关（Kill Switch） | "关闭按钮" | 在智能体编辑表面之外的布尔值；在每个后果性操作上检查 |
| 熔断器（Circuit Breaker） | "模式暂停" | 在重复、失败率或速率限制上的特定于操作的跳闸 |
| 金丝雀令牌（Canary Token） | "蜜令牌" | 智能体没有合法理由接触的诱饵；访问触发警报 |
| 蜜罐（Honeypot） | "取证沙箱" | 重定向流量/工作区，在那里观察隔离的智能体 |
| EWMA | "移动平均" | 指数加权；适应漂移（特征 + 错误） |
| CUSUM | "累积和" | 检测与基线的持续偏移 |
| 硬限制（Hard Limit） | "宪法规则" | 不适应；无论历史如何都是常数 |
| 宪法限制（Constitutional Limit） | "始终为真的规则" |  tied 到第17课的宪法；不能被智能体编辑 |
