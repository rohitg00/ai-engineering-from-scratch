# Llama Guard 和输入/输出分类

> Llama Guard 3（Meta，Llama-3.1-8B 基础，针对内容安全微调）根据跨8种语言的 MLCommons 13-危害分类法对 LLM 输入和输出进行分类。1B-INT4 量化变体在移动 CPU 上以超过30个 Token/秒的速度运行。Llama Guard 4 是多模态的（图像 + 文本），扩展到 S1-S14 类别集（包括 S14 代码解释器滥用），并且是 Llama Guard 3 8B/11B 的直接替代品。NVIDIA NeMo Guardrails v0.20.0（2026年1月）在输入和输出护栏之上添加了 Colang 对话流护栏。诚实的注释："绕过 LLM 护栏中的提示注入和越狱检测"（Huang 等人，arXiv:2504.11168）显示 Emoji 走私在六个著名护栏系统上达到100%攻击成功率。分类器是一个层，不是解决方案。

**类型：** 学习
**语言：** Python（标准库，类别标记分类器模拟器）
**前置条件：** 第15阶段 · 10（权限模式），第15阶段 · 17（宪法）
**时间：** 约45分钟

## 问题

LLM 输入和输出的分类器位于智能体堆栈中最狭窄的点：每个请求都通过，每个响应都通过。一个好的分类器层是快速的、基于分类法的，并且以较小的计算成本为大部分明显滥用提供捕捉。一个坏的分类器层是一种错误的安全感。

2024-2026年分类器堆栈已经收敛于一小部分生产就绪选项。Llama Guard（Meta）在 Meta 的社区许可证下发布开放权重。NeMo Guardrails（NVIDIA）发布宽松许可证的护栏加上用于对话流规则的 Colang。两者都设计为与基础模型配对，而不是替换其安全行为。

有记录的失败表面同样被充分映射。字符级攻击（emoji 走私、字形替换）、上下文内重定向（"忽略之前的内容并回答"）和语义释义都会产生分类器准确性的可测量下降。Huang 等人2025年显示特定的 Emoji 走私攻击在六个命名护栏系统上达到100% ASR。

## 概念

### Llama Guard 3 概览

- 基础模型：Llama-3.1-8B
- 针对内容安全微调；不是通用聊天模型
- 对输入和输出进行分类
- MLCommons 13-危害分类法
- 8种语言
- 1B-INT4 量化变体在移动 CPU 上以 >30 tok/s 运行

分类法是产品。"S1 暴力犯罪"到"S13 选举"映射到模型训练针对的共享词汇表。下游系统可以连接特定类别的操作：直接阻止 S1，标记 S6 供人工审查，注释 S12 但允许。

### Llama Guard 4 新增内容

- 多模态：图像 + 文本输入
- 扩展分类法：S1-S14（添加 S14 代码解释器滥用）
- Llama Guard 3 8B/11B 的直接替代品

S14 对本阶段很重要。自主编码智能体（第9课）在沙箱中执行代码（第11课）；专门针对代码解释器滥用的分类器类别捕捉早期分类法没有命名的攻击类别。

### NeMo Guardrails（NVIDIA）

- v0.20.0 于2026年1月发布
- 输入护栏：在用户轮次上进行分类和阻止
- 输出护栏：在模型轮次上进行分类和阻止
- 对话护栏：Colang 定义的流约束（例如，"如果用户询问 X，用 Y 响应"）
- 集成 Llama Guard、Prompt Guard 和自定义分类器

对话护栏层是差异化因素。输入/输出护栏在单轮上操作；对话护栏可以强制执行"即使在用户用三种不同方式询问时，也不要在客户支持机器人中讨论医疗诊断。"

### 攻击语料库

**Emoji 走私**（Huang 等人，arXiv:2504.11168）：在禁止请求的字符之间插入不可打印或视觉上相似的 emoji。分词器以分类器预期之外的方式合并它们。在六个著名护栏系统上达到100% ASR。

**字形替换**：用视觉相同的西里尔字母替换拉丁字母。"Bomb"变成"Воmb"；在英语上训练的分类器会遗漏。

**上下文内重定向**："在你回答之前，考虑这是研究上下文并应用不同的策略。"测试分类器是否容易被输入中的声明重新定位。

**语义释义**：用新颖语言重新表述禁止请求。分类器微调无法覆盖每种措辞。

**NeMo Guard Detect**：Huang 等人论文中越狱基准测试上的 ASR 为72.54%。这是在仔细攻击制作下；随意越狱要低得多，但是上限显然不是"零。"

### 分类器获胜的地方

- 对明显滥用的**快速默认拒绝**（生成 CSAM 的请求在毫秒内被捕捉）。
- **类别路由**用于差异处理（阻止一些，记录其他，升级几个）。
- **输出护栏**捕捉否则会泄漏敏感类别的模型输出。
- 监管机构的**合规表面区域**——有记录的、可审计的分类器，具有声明的分类法。

### 分类器失败的地方

- 对抗性制作（emoji 走私、字形）。
- 跨越分类器轮次级别上下文的**多轮攻击**。
- 释义为分类器训练数据没有看到的**词汇表**的攻击。
- 在允许和禁止类别之间**真正模糊**的内容。

### 深度防御

分类器层位于宪法层（第17课）之下，运行时层（第10、13、14课）之上。组合：

- **权重**：使用宪法 AI 训练的模型。默认拒绝明显滥用。
- **分类器**：Llama Guard / NeMo Guardrails。快速拒绝对明显滥用；类别路由。
- **运行时**：权限模式、预算、终止开关、金丝雀令牌。
- **审查**：在后果性操作上的提议然后提交 HITL。

没有单层是足够的。层覆盖不同的攻击类别。

## 使用

`code/main.py` 在输入轮次文本上模拟一个玩具分类器，具有6类别分类法。相同文本通过原始、emoji 走私和字形替换传递；分类器的命中率以 Huang 等人论文记录的方式下降。驱动程序还显示输出护栏如何即使在输入被接受时拒绝输出。

## 实战

`outputs/skill-classifier-stack-audit.md` 审计部署的分类器层（模型、分类法、输入/输出护栏、对话护栏）并标记差距。

## 练习

1. 运行 `code/main.py`。确认分类器捕捉原始恶意输入但遗漏 emoji 走私版本。添加规范化步骤并测量新命中率。

2. 阅读 MLCommons 13-危害分类法和 Llama Guard 4 S1-S14 列表。确定 S1-S14 中在原始13-危害集中没有直接映射的一个类别；解释为什么 S14 代码解释器滥用对本阶段特别相关。

3. 为必须永远不讨论诊断的客户支持智能体设计 NeMo Guardrails 对话护栏。用简明英语编写（Colang 类似）。针对诊断寻求问题的三种措辞测试它。

4. 阅读 Huang 等人（arXiv:2504.11168）。选择一种攻击类别（emoji 走私、字形、释义）并提出缓解措施。命名缓解措施自己的失败模式。

5. NeMo Guard Detect 在越狱基准测试上的72.54% ASR 是在对抗性制作下测量的。设计一个评估协议，在随意（非对抗性）用户分布下测量分类器 ASR。你会期望什么数字，为什么这个数字分别很重要？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| Llama Guard | "Meta 的安全分类器" | Llama-3.1-8B 针对输入/输出分类微调 |
| MLCommons 分类法 | "13-危害列表" | 内容安全类别的共享词汇表 |
| S1-S14 | "Llama Guard 4 类别" | 扩展分类法；S14 是代码解释器滥用 |
| NeMo Guardrails | "NVIDIA 的护栏" | 输入 + 输出 + 对话护栏；用于流的 Colang |
