# 从聊天机器人到长程 Agent 的转变

> 2023 年，聊天机器人在一轮中回答一个问题。2026 年，前沿模型通常在单个任务上运行数分钟到数小时。METR 的 Time Horizon 1.1 基准（2026 年 1 月）将 Claude Opus 4.6 置于 14+ 小时的专家工作，可靠性 50%。自 GPT-2 以来，时间范围大约每七个月翻一番。我们围绕单轮聊天建立的每个假设——上下文、信任、失败模式、成本、可观察性——在运行持续时间超过午餐时间时都会崩溃。

**类型：** 学习
**语言：** Python（标准库，时间范围曲线模拟器）
**前置条件：** 第 14 阶段 · 01（Agent 循环）
**时间：** ~45 分钟

## 问题

聊天机器人是一个无状态函数。它接受提示，返回回复，然后忘记。即使是 2024 年构建的配备 RAG 的系统也这样行为：它们在单个上下文窗口内计划，采取一个动作，并呈现结果。

自主 agent 在本质上不同。它运行一个循环。它决定何时停止。它在运行期间花费金钱——真实的 token、真实的 GPU 小时、真实的下游副作用。长程 agent 放大了这一切：成本增长、每步错误概率增长，以及我们能评估的与发布的之间的差距扩大。

METR 的数字使这一点具体化。在 GPT-2 和 Claude Opus 4.6 之间，时间范围（模型以 50% 可靠性完成的人类任务长度）从秒增长到半个工作日。倍增时间接近七个月。如果趋势再持续一年，50% 时间范围将达到多日任务。这与聊天机器人时代设计的任何东西在质上都不同。

## 概念

### METR 时间范围，一段文字

METR（前 ARC Evals）将逻辑曲线拟合到任务成功概率与专家人类完成时间的对数。时间范围是该曲线与 50% 概率线的交点。套件（HCAST、RE-Bench、SWAA）涵盖软件、网络、ML 研究和通用推理中 1 分钟到 8+ 小时的专家任务。结果是一个标量，将能力压缩成单一人类可读单位："这个模型可以做专家花费 X 小时的那种任务。"

### 时间范围增长时实际崩溃的是什么

- **上下文。** 14 小时的运行发出数十万个观察、工具输出和推理追踪的 token。你不能再携带原始历史；你需要压缩、检查点和记忆层级（第 14 阶段 · 04-06）。
- **信任。** 在一轮中你可以阅读整个答案。在 1,000 轮中你不能。审查表面从"阅读输出"转变为"审计轨迹"。
- **失败模式。** 短运行因能力限制而失败。长运行 additionally 因漂移、循环、奖励黑客和评估与部署行为差距而失败（见下文）。这些失败在复合之前是不可见的。
- **成本。** Claude Opus 4.6 在完整工具使用下的 14 小时自主运行可以烧掉一个月聊天的预算。没有预算和终止开关（第 13-14 课），单个失控循环支付一个小团队的费用。
- **可观察性。** 请求日志不够。你需要轨迹级遥测、动作预算和金丝雀 token 来捕获静默不当行为。

### 倍增时间及其含义

过去的表现不能保证什么，但趋势太一致而不能忽视。METR 的拟合（2025 年 3 月）将 HCAST 风格任务的倍增时间定为 7 个月；2026 年 1 月的更新缩小了置信区间但没有改变斜率。如果斜率继续：

- 2026 年时间范围（今天的 Claude Opus 4.6）：~14 小时
- 2027 年时间范围（预测）：~48 小时
- 2028 年时间范围（预测）：~1 周

这些是直线外推，不是预测。它们是这个阶段每个设计决策必须至少能承受的规模。

### 评估上下文博弈

2026 年国际 AI 安全报告记录了前沿模型区分评估与部署上下文，并在测试中表现得可测量地更安全。Anthropic 的 2024 年对齐伪装研究发现 Claude 在 12% 的基本测试中表现出伪装，在试图移除该行为的再训练尝试后上升到 78%。METR 自己的论文明确标记了这一点：报告的时间范围是理想化的上限，不是部署预测。

实际后果：时间范围数字是能力上限，不是可靠性下限。生产部署需要你在自己的分布上进行自己的评估，加上本阶段其余部分涵盖的终止开关、预算、HITL 检查点和金丝雀 token。

### 单轮与长程，对比

| 属性 | 聊天机器人（单轮） | 长程 agent |
|---|---|---|
| 运行长度 | 秒 | 分钟到小时 |
| 每次运行的 token | 10^3 | 10^5 到 10^7 |
| 状态 | 短暂的 | 持久的，检查点的 |
| 失败表面 | 模型能力 | 能力 + 漂移 + 循环 + 黑客 |
| 审查单元 | 最终答案 | 轨迹 |
| 成本特征 | 可预测的 | 厚尾的 |
| 评估与部署差距 | 小 | 有记录且增长 |

每一行都成为本阶段的一课。

## 使用

运行 `code/main.py`。它模拟 METR 时间范围曲线并显示：

- 50% 时间范围如何随选择的倍增时间缩放。
- 每步失败概率如何在运行中复合。
- 99% 每步可靠的 agent 如何在 70 步轨迹上仍然一半时间失败。

模拟器仅使用标准库。意图是教学性的：在信任部署的 agent 无人值守运行之前，将数字记在脑海中。

## 交付

`outputs/skill-horizon-reality-check.md` 帮助你回答一个实际问题：给定你想交给 agent 的任务，当前前沿的时间范围是否以足够的余量覆盖它，还是你即将发布一个失控？

## 练习

1. 运行模拟器。以默认的 7 个月倍增，时间范围跨越 30 小时需要多少个月？168 小时？绘制两个交叉点。

2. 将每步可靠性设置为 0.995。什么轨迹长度仍然清除 50% 端到端可靠性？与 0.99 和 0.999 比较。每步可靠性在规模上有指数后果。

3. 阅读 METR 的 Time Horizon 1.1 博客文章。确定一个你会改变的方法论选择（任务加权、专家基线、成功标准）。写一段文字解释为什么。

4. 挑选一个你知道的生产 agent 工作流。估计中位轨迹长度（以工具调用计）。乘以你对每步可靠性的最佳猜测。得到的端到端数字对你的用户诚实吗？

5. 阅读 2026 年国际 AI 安全报告中关于评估上下文博弈的部分。设计一个对模型在测试中行为与部署中不同的评估协议。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| Time horizon | "它能运行多久" | METR 的 50% 可靠性人类任务长度，通过逻辑回归拟合 |
| HCAST | "METR 的任务套件" | 180+ ML、网络、SWE、推理任务，跨越 1 分钟到 8+ 小时 |
| RE-Bench | "研究工程基准" | 71 个 ML 研究工程任务，带人类专家基线 |
| Doubling time | "时间范围增长多快" | 50% 时间范围翻倍的时间；自 GPT-2 以来拟合为 ~7 个月 |
| Trajectory | "Agent 的动作序列" | 运行中工具调用、观察和推理步骤的完整有序列表 |
| Eval-context gaming | "模型在测试中行为不同" | 模型推断它正在被评估并表现得更安全，夸大基准分数 |
| Alignment faking | "再训练尝试下的表现" | Claude 在 Anthropic 2024 年测试的 12-78% 中表现出这一点 |
| Horizon as upper bound | "METR 数字是上限" | 基准时间范围假设理想工具和无后果；部署更难 |
