# 人工介入：提出-提交

> 2026 年关于 HITL 的共识是具体的。它不是"代理询问，用户点击批准"。它是提出-提交：提出的操作被持久化到带有幂等键的持久存储；向审查者展示意图、数据谱系、触及的权限、爆炸半径和回滚计划；仅在积极确认后提交；执行后验证以确认副作用实际发生。LangGraph 的 `interrupt()` 加 PostgreSQL 检查点、Microsoft Agent Framework 的 `RequestInfoEvent` 和 Cloudflare 的 `waitForApproval()` 都实现了相同的形状。

**类型：** 学习
**语言：** Python（标准库，提出-提交状态机与幂等性）
**前置条件：** 第 15 阶段 · 12（持久化执行），第 15 阶段 · 14（绊线）
**时间：** ~60 分钟

## 问题

代理执行操作。用户必须决定：批准还是不批准。如果决定是即时的，那可能不是审查。如果决定是结构化的，那会很慢但值得信赖。工程问题是如何让结构化审查成为阻力最小的路径。

2023 年代的 HITL 模式是一个同步提示："代理想要发送邮件给 X，内容为 Y —— 批准吗？"用户点击批准。每个人都感觉系统是安全的。实际上，这种表面被大量盖章：用户快速批准，批准预测性低，当代理出错时，审计跟踪显示用户无法回忆的长串批准历史。

2026 模式 —— 提出-提交 —— 将 HITL 移到持久基底上，附加结构化元数据，并要求积极提交。每个托管代理 SDK 都提供一个版本：LangGraph `interrupt()`、Microsoft Agent Framework `RequestInfoEvent`、Cloudflare `waitForApproval()`。API 名称不同；形状相同。

## 概念

### 提出-提交状态机

1. **提出。** 代理产生一个提出的操作。持久化到持久存储（PostgreSQL、Redis、Durable Object）。包括：
   - 意图（代理为什么这样做）
   - 数据谱系（什么来源导致此提议）
   - 触及的权限（哪些范围/文件/端点）
   - 爆炸半径（最坏情况是什么）
   - 回滚计划（如果提交，如何撤销）
   - 幂等键（每个提议唯一；重新提交返回相同记录）
2. **展示。** 审查者看到带有所有元数据的提议。审查者是人（不是代理自己审查）。
3. **提交。** 积极确认。操作执行。
4. **验证。** 执行后，副作用被读回并确认。如果验证步骤失败，系统处于已知的坏状态，警报启动。

### 幂等键

没有幂等键，瞬态失败后的重试可能会双重执行已批准的操作。具体示例：用户批准"从 A 转账 100 美元到 B"。网络闪烁。工作流重试。用户只批准了一次，但转账执行了两次。幂等键将批准绑定到单个、唯一的副作用；第二次执行是无操作。

这与 Stripe 和 AWS API 使用的幂等模式相同。将其重新用于代理批准在 Microsoft Agent Framework 文档中是明确的。

### 持久性：为什么批准比进程更持久

批准等待室是代理不拥有的状态片段。工作流暂停（第 12 课）。当批准到达时，工作流从该点精确恢复。这就是 LangGraph 将 `interrupt()` 与 PostgreSQL 检查点配对，而不仅仅是内存状态的原因 —— 两天后的批准仍然发现工作流完好无损。

### 盖章批准和挑战-响应缓解

HITL 的默认 UI（"批准"/"拒绝"按钮）产生没有真正审查的快速批准。记录的缓解措施：挑战-响应清单，在启用批准按钮之前需要对特定问题给出积极答案。具体形状：

- "你理解这触及什么资源吗？[ ]"
- "你验证了爆炸半径可接受吗？[ ]"
- "如果失败，你有回滚计划吗？[ ]"

不是为了官僚主义 —— 一个强制功能。无法勾选框的审查者要么要求澄清（升级），要么拒绝（安全默认）。Anthropic 的代理安全研究明确引用清单驱动的 HITL 作为盖章批准模式的缓解措施。

### 什么算重大

并非每个操作都需要提出-提交。2026 指南：

- **重大操作**（始终 HITL）：不可逆写入、金融交易、出站通信、生产数据库更改、破坏性文件系统操作。
- **可逆操作**（有时 HITL）：本地文件编辑、暂存环境更改、具有明确回滚的可逆写入。
- **读取和检查**（从不 HITL）：读取文件、列出资源、调用只读 API。

### 操作后验证

"提交运行了"与"副作用发生了"不同。网络分区和竞态条件可能产生认为成功但后端未持久化的工作流。验证步骤在提交后重新读取目标资源以确认。这与数据库事务的 `RETURNING` 子句或 AWS `GetObject` 在 `PutObject` 之后的模式相同。

### 欧盟 AI 法案第 14 条

第 14 条要求欧盟高风险 AI 系统的有效人工监督。"有效"不是装饰性的。监管语言明确排除盖章模式。带有挑战-响应的提出-提交是在 Microsoft Agent Governance Toolkit 合规文档中经受第 14 条审查的形状。

`code/main.py` 在标准库 Python 中实现提出-提交状态机。持久存储是 JSON 文件。幂等键是 (thread_id, action_signature) 的哈希。驱动程序模拟三种情况：干净的批准流、瞬态失败后的重试（不得双重执行），以及盖章默认与挑战-响应流。

`outputs/skill-hitl-design.md` 审查提出的 HITL 工作流，检查提出-提交形状，并标记缺失的元数据、幂等性、验证或挑战-响应层。

## 练习

1. 运行 `code/main.py`。确认已批准提议的重试使用持久记录，不会重新执行。现在将幂等键更改为包含时间戳，并显示重试双重执行。

2. 用 `rollback` 字段扩展提议记录。模拟验证步骤失败的执行。显示回滚自动触发。

3. 阅读 Microsoft Agent Framework 的 `RequestInfoEvent` 文档。识别 API 包含但玩具引擎缺失的一个元数据字段。添加它并解释它保护什么。

4. 为特定操作设计挑战-响应清单（例如，"发布到公共 Twitter 账户"）。审查者必须回答哪三个问题？为什么这三个？

5. 选择一个同步 "批准吗？"提示足够的情况（不需要持久存储）。解释为什么，并命名你接受的风险类别。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| 提出-提交 | "两阶段批准" | 持久化提议 + 积极提交 + 验证 |
| 幂等键 | "重试安全令牌" | 每个提议唯一；第二次执行无操作 |
| 数据谱系 | "它来自哪里" | 导致提议的特定源内容 |
| 爆炸半径 | "最坏情况" | 如果操作出错，影响范围 |
| 盖章 | "快速批准" | 未经真正审查点击"批准" |
| 挑战-响应 | "强制清单" | 审查者必须积极确认特定问题 |
| RequestInfoEvent | "MS Agent Framework 原语" | 带有结构化元数据的持久 HITL 请求 |
| `interrupt()` / `waitForApproval()` | "框架原语" | LangGraph / Cloudflare 的相同形状等价物 |
