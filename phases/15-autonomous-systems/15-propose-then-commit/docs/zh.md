# 人在回路：提议-然后-提交

> 2026 年关于 HITL（人在回路）的共识是具体的。它不是"智能体询问，用户点击批准。"而是提议-然后-提交：提议的操作被持久化到持久存储中，带有幂等性密钥；向审查者展示意图、数据血缘、触及的权限、爆炸半径和回滚计划；仅在积极确认后提交；执行后验证以确认副作用实际发生。LangGraph 的 `interrupt()` 加上 PostgreSQL 检查点、Microsoft Agent Framework 的 `RequestInfoEvent` 和 Cloudflare 的 `waitForApproval()` 都实现了相同的形态。典型的失败模式是橡皮图章批准："批准？"未经审查就被点击。文档化的缓解措施是带有明确清单的挑战-响应。

**类型：** 学习
**语言：** Python（标准库，带幂等性的提议-然后-提交状态机）
**前置条件：** 阶段 15 · 12（持久执行），阶段 15 · 14（绊网）
**时间：** ~60 分钟

## 问题

智能体执行一个操作。用户必须决定：批准还是不批准。如果决策是即时的，它可能不是审查。如果决策是结构化的，它很慢但可信。工程问题是如何使结构化审查成为阻力最小的路径。

2023 年时代的 HITL 模式是同步提示："智能体想要发送邮件给 X，正文为 Y——批准？"用户点击批准。每个人都觉得系统是安全的。实际上，这个表面 heavily rubber-stamped（橡皮图章化）：用户快速批准，批准预测性很低，当智能体出错时，审计跟踪显示用户无法回忆的长串批准历史。

2026 年的模式——提议-然后-提交——将 HITL 移动到持久底层，附加结构化元数据，并要求积极提交。每个托管智能体 SDK 都提供了一个版本：LangGraph `interrupt()`、Microsoft Agent Framework `RequestInfoEvent`、Cloudflare `waitForApproval()`。API 名称不同；形态相同。

## 概念

### 提议-然后-提交状态机

1. **提议。** 智能体产生提议的操作。持久化到持久存储（PostgreSQL、Redis、Durable Object）。包括：
   - 意图（智能体为什么要这样做）
   - 数据血缘（什么来源导致了这个提议）
   - 触及的权限（哪些范围/文件/端点）
   - 爆炸半径（最坏情况是什么）
   - 回滚计划（如果提交，我们如何撤销）
   - 幂等性密钥（每个提议唯一；重新提交返回相同记录）

2. **展示。** 审查者看到带有所有元数据的提议。审查者是人（不是智能体自己审查自己）。

3. **提交。** 积极确认。操作执行。

4. **验证。** 执行后，读回副作用并确认。如果验证步骤失败，系统处于已知错误状态并触发警报。

### 幂等性密钥

没有幂等性密钥，瞬态故障后的重试可能双重执行已批准的操作。具体示例：用户批准"从 A 向 B 转账 100 美元。"网络闪断。工作流重试。用户批准了一次，但转账执行了两次。幂等性密钥将批准绑定到单个、唯一的副作用；第二次执行是无操作。

这与 Stripe 和 AWS API 使用的幂等性模式相同。Microsoft Agent Framework 文档中明确将其重用于智能体批准。

### 持久性：为什么批准比进程更持久

批准等候室是智能体不拥有的一段状态。工作流暂停（课程 12）。当批准到达时，工作流从确切那个点恢复。这就是为什么 LangGraph 将 `interrupt()` 与 PostgreSQL 检查点配对，而不仅仅是内存状态——两天后的批准仍然找到完整的工作流。

### 橡皮图章批准和挑战-响应缓解

HITL 的默认 UI（"批准"/"拒绝"按钮）产生没有真正审查的快速批准。文档化的缓解措施：挑战-响应清单，在启用批准按钮之前需要积极回答特定问题。具体形态：

- "您了解这会触及什么资源吗？[ ]"
- "您是否已验证爆炸半径是可接受的？[ ]"
- "如果失败，您有回滚计划吗？[ ]"

不是为了官僚主义而官僚主义——而是一个强制函数。无法勾选框的审查者要么要求澄清（升级），要么拒绝（安全默认）。Anthropic 智能体安全研究明确引用清单驱动的 HITL 作为橡皮图章批准模式的缓解措施。

### 什么算作重大操作

不是每个操作都需要提议-然后-提交。2026 年指南：

- **重大操作**（总是 HITL）：不可逆写入、金融交易、出站通信、生产数据库更改、破坏性文件系统操作。
- **可逆操作**（有时 HITL）：对本地文件的编辑、暂存环境更改、带有明确回滚的可逆写入。
- **读取和检查**（从不 HITL）：读取文件、列出资源、调用只读 API。

### 操作后验证

"提交运行了"与"副作用发生了"不同。网络分区和竞争条件可能产生一个认为它成功而后端未持久化的工作流。验证步骤在提交后重新读取目标资源以确认。这与带有 `RETURNING` 子句的数据库事务或 `PutObject` 后的 AWS `GetObject` 是相同的模式。

### 欧盟 AI 法案第 14 条

第 14 条强制要求对欧盟的高风险 AI 系统进行有效的人为监督。"有效"不是装饰性的。监管语言明确排除橡皮图章模式。带有挑战-响应的提议-然后-提交是在 Microsoft Agent Governance Toolkit 合规文档中经受住第 14 条审查的形态。

## 使用

`code/main.py` 在标准库 Python 中实现提议-然后-提交状态机。持久存储是 JSON 文件。幂等性密钥是（thread_id, action_signature）的哈希。驱动程序模拟三种情况：干净批准流程、瞬态故障后的重试（不得双重执行）和橡皮图章默认与挑战-响应流程。

## 交付

`outputs/skill-hitl-design.md` 审查提议的 HITL 工作流的提议-然后-提交形态，并标记缺失的元数据、幂等性、验证或挑战-响应层。

## 练习

1. 运行 `code/main.py`。确认已批准提议的重试使用持久记录并且不重新执行。现在将幂等性密钥更改为包含时间戳，并显示重试双重执行。

2. 使用 `rollback` 字段扩展提议记录。模拟验证步骤失败的执行。显示回滚自动触发。

3. 阅读 Microsoft Agent Framework 的 `RequestInfoEvent` 文档。识别 API 包含的一个玩具引擎缺失的元数据字段。添加它并解释它防范什么。

4. 为特定操作（例如，"发布到公共 Twitter 账户"）设计挑战-响应清单。审查者必须回答哪三个问题？为什么是这三个？

5. 选择一个同步"批准？"提示就足够的情况（不需要持久存储）。解释为什么，并命名您接受的风险类别。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| 提议-然后-提交 | "两阶段批准" | 持久提议 + 积极提交 + 验证 |
| 幂等性密钥 | "重试安全令牌" | 每个提议唯一；第二次执行无操作 |
| 数据血缘 | "它来自哪里" | 导致提议的特定来源内容 |
| 爆炸半径 | "最坏情况" | 如果操作出错的影响范围 |
| 橡皮图章 | "快速批准" | 未经真正审查就点击"批准" |
| 挑战-响应 | "强制清单" | 审查者必须积极确认特定问题 |
| RequestInfoEvent | "MS Agent Framework 原语" | 带结构化元数据的持久 HITL 请求 |
| `interrupt()` / `waitForApproval()` | "框架原语" | LangGraph / Cloudflare 相同形态的等价物 |
