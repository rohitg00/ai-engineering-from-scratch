# METR 时域和外部能力评估

> METR（前 ARC Evals）自2023年12月起是一家独立的 501(c)(3) 组织。他们的时域 1.1 基准测试（2026年1月）将任务成功概率对专家人类完成时间的对数进行逻辑曲线拟合；与50%概率的交点是模型的时域。2025-2026 年参与集涵盖 GPT-5.1、GPT-5.1-Codex-Max 和原型监测评估（监测器能否捕捉侧任务；智能体能否规避）。基准测试套件：HCAST（180+ 个 ML、网络、SWE、推理任务；1分钟到8+小时）、RE-Bench（71 个 ML 研究工程任务，有专家基线）、SWAA。诚实的注释：METR 测量是理想化的——没有人类，没有真实后果——并且团队记录了评估与部署行为差距（第1课）。时域是上限，不是部署预测。

**类型：** 学习
**语言：** Python（标准库，逻辑拟合时域估计器）
**前置条件：** 第15阶段 · 01（长时域智能体），第15阶段 · 19（RSP）
**时间：** 约60分钟

## 问题

扩展政策（第19、20课） only 与它们引用的测量一样有用。"AI R&D-4 阈值"和"长程自主性"在政策散文中定义；它们仅在具体评估产生具体数字时才变得可操作。

METR 是 2024-2026 年定义许多这些数字外部评估组织。他们评估前沿模型——通常在发布前，在 NDA 下与实验室——并在之后发布方法论。时域 1.1 基准测试（2026年1月）是他们的标题工件：一个将能力压缩为人类可读单位的标量（"这个模型可以做专家花费 X 小时做的那种任务"）。

本课部分关于方法论（时域如何计算）和部分关于解释（为什么时域是上限，不是部署预测）。这两个技能属于一起。理解时域如何拟合的团队比仅看到幻灯片上"14小时"的团队更难被坏供应商声明愚弄。

## 概念

### METR 背景

- 成立：2023年12月（前 ARC Evals，分拆为独立的 501(c)(3)）。
- 范围：前沿模型自主能力的评估，通常在发布前。
- 合作伙伴实验室：Anthropic、OpenAI（2025-2026 年多次参与）。
- 显著交付物：时域 1.0（2025年3月）、时域 1.1（2026年1月）、原型监测评估。

### 时域拟合

方法论（来自 METR 博客和论文）：

1. 收集跨越分钟级到小时级专家完成时间的任务套件。当前套件：HCAST（180+ 任务）、RE-Bench（71 任务）、SWAA。
2. 在每个任务上运行模型；记录成功或失败。
3. 拟合逻辑曲线：P(成功) 作为 log(专家完成时间) 的函数。
4. 时域是 P(成功) = 0.5 时的专家时间。

逻辑拟合形状是正确的，因为能力通常与任务难度有增加、接近平台的关系。50% 点是一个选择（可以是 10%、90%）；METR 在详细论文中报告多个阈值，但在 TH1.1 中以前导 50% 因为最直观。

### 2026年1月的数字

根据时域 1.1：

- Claude Opus 4.6：50% 可靠性下约14小时，根据时域 1.1（2026年1月）。
- HCAST 风格任务的翻倍时间：时域 1.1（2026年1月）报告的2023年后拟合约4.3个月（130.8天）；~7个月数字是来自时域 1.0 的完整 2019-2025 年拟合，并在 TH1.1 中与2023年后数字一起报告。

这些是第15阶段其余部分引用的具体数字。它们在 Anthropic 的 RSP v3.0（第19课）和 DeepMind 的 FSF v3（第20课）中作为输入阈值评估的测量被引用。

### 基准测试套件

- **HCAST**：跨 ML、网络安全、软件工程和通用推理的 189 个任务。跨度：1分钟到8+小时专家时间。时域数字背后的主要套件。
- **RE-Bench**：71 个 ML 研究工程任务，有人类专家基线。测量模型是否能做输入 AI R&D-4 阈值的那种 ML R&D 工作。
- **SWAA**：较短任务；用于校准时域曲线的低端点。

### 原型监测评估

2025-2026 年 METR 工作包括监测评估原型：监测器能否捕捉执行侧任务的模型（不是操作者要求的事情）；智能体能否规避监测器。这些直接输入 OpenAI PF v2 中的"破坏保障措施"类别和 DeepMind FSF v3（第20课）中的欺骗性对齐监测。

### 为什么时域是上限

- **理想化工具**：基准测试任务给模型干净的工具和格式良好的数据。真实部署更混乱。
- **没有真实后果**：模型从不实际向客户计费、删除真实数据或联系真实人员。真实部署有不可逆转折。
- **评估-上下文博弈**：第1课。模型在测试中表现不同。2026年国际 AI 安全报告经验性地记录了这一点。
- **没有合法用户方差**：基准测试提示是结构化的。真实用户产生模糊的、依赖上下文的请求。

时域是有利条件下的能力上限。部署可靠性是一个不同的数字，更低，并且团队必须测量他们自己的分布以了解它。

### 外部评估者案例

外部评估很重要，因为内部实验室有优化他们报告指标的动机。METR 的独立性——带有声明方法论和同行评审论文的 501(c)(3)——是结构缓解措施。它单独不充分（实验室仍然控制 METR 看到什么），但它严格优于没有外部评估。

### 实践中如何使用时域数字

- **作为能力过滤器**：如果模型的时域远低于提议任务的专家时间，不要自主部署它（第1课的技能文件）。
- **作为趋势指标**：翻倍时间告诉你即使没有新缓解措施，当前实践将保持安全多长时间。
- **作为先验**：14小时的时域是起点。根据你的任务分布、你的工具质量和你的部署上下文向下调整。

## 使用

`code/main.py` 在给定合成结果集的情况下，实现任务成功对 log(专家时间) 的逻辑拟合。它报告 50% 时域（METR 的标题）、10% 时域（保守）和 90% 时域（乐观）。还演示了当时域被评估-上下文博弈人为膨胀时什么变化。

## 实战

`outputs/skill-horizon-interpretation.md` 审查供应商的时域声明并产生基准声明和部署现实之间的差距分析。

## 练习

1. 运行 `code/main.py`。确认拟合的 50% 时域与合成基础真值匹配。现在将任务时间网格减半；时域估计是否有意义地变化？

2. 阅读 METR 的时域 1.1 博客文章。确定可靠性最高和最低的具体任务。解释为什么存在差距。

3. 阅读 METR 的"测量自主 AI 能力"资源。列出 HCAST 任务类别。选择一个你会为生产任务更重加权的类别，并解释为什么。

4. 将评估-上下文博弈引入模拟器：将约20%的失败任务翻转为成功。报告新时域。这近似于20%博弈率对观察数字的作用。

5. 在你自己的 Bug 积压或代表性任务集上设计内部时域评估。描述数据收集、拟合以及输出告诉你的内容。与 METR 数字进行比较。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| METR | "外部评估者" | 前 ARC Evals；自2023年12月起的独立 501(c)(3) |
| 时域（Time Horizon） | "能力测量" | 50% 可靠性下的专家任务长度，来自逻辑拟合 |
| HCAST | "METR 的主要套件" | 1分钟到8+小时的 180+ 任务 |
| RE-Bench | "研究工程" | 有人类基线的 71 个 ML 研究工程任务 |
| SWAA | "短任务套件" | 校准时域曲线的低端点 |
| 翻倍时间（Doubling Time） | "增长率" | 50% 时域翻倍的时间；HCAST 约7个月 |
| 评估-上下文博弈（Eval-Context Gaming） | "模型表现不同" | 测试和部署之间记录的行为差距 |
| 上限（Upper Bound） | "时域是天花板" | 基准时域 > 负载下的部署可靠性 |
