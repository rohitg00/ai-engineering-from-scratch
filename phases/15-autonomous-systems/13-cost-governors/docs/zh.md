# 13 · 行动预算、迭代上限与成本调控器

> 某中型电商智能体（agent）的月度 LLM 成本在团队启用「订单跟踪」技能后，从 1,200 美元飙升到 4,800 美元。这不是定价 bug，而是一个智能体发现了新的循环（loop），并在其中不停烧钱。微软的智能体治理工具包（Agent Governance Toolkit，2026 年 4 月 2 日）将针对这一类问题的防御手段系统化：每请求的 `max_tokens`、每任务的 token 与美元预算、每日/每月上限、迭代上限、分层模型路由（tiered model routing）、提示缓存（prompt caching）、上下文窗口化（context windowing）、对昂贵操作设置人在回路（HITL）检查点、预算突破时的终止开关（kill switch）。Anthropic 的 Claude Code Agent SDK 以不同的名字提供了同一套原语。金融速度限制（financial velocity limit）——例如「10 分钟内消费超过 50 美元即切断访问」——比月度上限更快地捕获循环。

**类型：** 学习
**语言：** Python（标准库，分层成本调控器模拟器）
**前置：** 第 15 阶段 · 10（权限模式）、第 15 阶段 · 12（持久化执行）
**时长：** 约 60 分钟

## 问题所在

自主智能体在每一轮（turn）都在花真金白银。聊天机器人的糟糕输出只是一条糟糕的回复；智能体的糟糕循环却是一张账单。业界对这种失败模式有一个公认的术语——「钱包拒绝服务（Denial of Wallet）」：智能体不停推理、不停调用工具、不停计费，而没有任何东西能阻止它，因为压根没有为此做过设计。

修复之道不是某一个数字，而是一整套不同时间尺度与粒度上的限制叠加：每请求、每任务、每小时、每天、每月。一套设计良好的叠加体系能在几分钟内捕获失控循环、在几小时内捕获缓慢泄漏、在一天内捕获糟糕的发布。同样这套体系，能在智能体长周期且自主运行时让预算始终存在。

这是一节工程课：数学是平凡的，真正让团队栽跟头的是纪律。下文列出的限制清单，要么出现在微软智能体治理工具包中，要么出现在 Anthropic Claude Code Agent SDK 文档中。

## 核心概念

### 成本调控器叠加体系

1. **每请求 `max_tokens`。** 简单。防止任何单次调用产出无界的补全（completion）。
2. **每任务 token 预算。** 整个运行过程中不得超过 N 个 token。到达上限即硬性停止。
3. **每任务美元预算。** 与 token 同理，只是以货币计。即 Claude Code 中的 `max_budget_usd`。
4. **每工具调用上限。** `WebFetch` 调用不超过 N 次、`shell_exec` 调用不超过 N 次，等等。
5. **迭代上限（`max_turns`）。** 智能体循环的总迭代次数；防止无限推理循环。
6. **每分钟 / 每小时 / 每天 / 每月上限。** 滚动窗口（rolling window）。捕获不同时间尺度上的泄漏。
7. **金融速度限制。** 例如「若 10 分钟内消费超过 50 美元，则切断访问」。在月度上限触发之前就拦住基于循环的烧钱。
8. **分层模型路由。** 默认使用较小的模型；只有当分类器（classifier）判断任务确有必要时才升级到更大的模型。
9. **提示缓存。** 系统提示和稳定上下文存储在提供商缓存中；重新发送的 token 成本近乎为零。
10. **上下文窗口化。** 通过压缩（compaction）/ 摘要将活动上下文保持在阈值以下；直接降低 token 成本。
11. **对昂贵操作设置 HITL 检查点。** 在执行已知昂贵的操作（耗时的工具调用、大体积下载、代价高昂的模型升级）之前，要求人工点头确认。
12. **预算突破时的终止开关。** 任一上限触发时会话即中止。上限记录在案；需要一条独立的重新启用路径。

### 为什么要叠加体系，而不是单一上限

单一的月度上限只有在钱包已被掏空之后才能捕获失控的智能体。单一的每请求上限在会话层面什么也捕获不了。不同的失败模式需要不同的时间尺度：

- **失控循环**（智能体卡在 5 秒一次的重试中）：由速度限制捕获。
- **缓慢泄漏**（智能体每个任务做了约 2 倍于预期的工作）：由每日上限捕获。
- **糟糕发布**（新版本消耗 5 倍 token）：由每周 / 每月上限捕获。
- **正常激增**（真实需求，并非 bug）：由小时 / 每日上限捕获，并附带清晰日志。

### Claude Code 的预算接口

Claude Code Agent SDK 暴露了（据公开文档）：

- `max_turns` —— 迭代上限。
- `max_budget_usd` —— 美元上限；突破时会话中止。
- `allowed_tools` / `disallowed_tools` —— 工具白名单与黑名单。
- 工具使用前的钩子点（hook point），用于自定义成本核算。

将其与权限模式阶梯（第 10 课）结合使用。一个没有 `max_budget_usd` 的 `autoMode` 会话就是不受治理的自主性。Anthropic 明确将 Auto Mode 定位为需要预算控制；分类器与成本是正交（orthogonal）的两件事。

### 欧盟 AI 法案、OWASP 智能体 Top 10

微软的智能体治理工具包覆盖了 OWASP 智能体 Top 10（OWASP Agentic Top 10）以及欧盟 AI 法案第 14 条（人类监督）的要求。对于在欧盟的生产部署，日志记录与上限强制执行并非可选项。

### 观测到的 1,200 → 4,800 美元案例

微软文档中的真实案例：一个电商智能体在新增一个工具后月度成本翻了三倍。该工具允许智能体在每个会话期间轮询订单状态。没有循环检测，没有每工具上限，没有针对周环比增长的告警。修复方案是一个每工具上限外加一个日增长告警。这是一个模板：每个新的工具界面都是一个潜在的新循环；每个新工具都需要它自己的上限和自己的告警。

## 动手实践

`code/main.py` 模拟了一次智能体运行——分别在有和没有分层成本调控器叠加体系的情况下。模拟智能体在若干轮之后漂移进入一个轮询循环；分层叠加体系会在速度窗口内捕获它，而单一的月度上限要到数天之后才会触发。

## 练习

1. 运行 `code/main.py`。确认在轮询循环轨迹上，速度限制会先于迭代上限触发。现在禁用速度限制，测量在迭代上限捕获它之前，智能体「花掉」了多少。

2. 为一个浏览器智能体（第 11 课）设计一套每工具上限。哪个工具需要最紧的上限？哪个工具可以无界运行而无风险？

3. 阅读微软智能体治理工具包文档。列出该工具包命名的每一种上限类型。将每一种映射到某一失败模式（失控循环、缓慢泄漏、糟糕发布、激增）。

4. 为一个现实任务（例如「对一个仓库中的 50 个 issue 做分诊」）的一次无人值守通宵运行定价。将 `max_budget_usd` 设为你点估计值的 2 倍。论证这个 2 倍的理由。

5. Claude Code 的 `max_budget_usd` 基于会话累计成本触发。设计一个你将在外部强制执行的互补速度限制。是什么触发了切断？重新启用又是什么样子？

## 关键术语

| 术语 | 人们怎么说 | 它实际指什么 |
|---|---|---|
| Denial of Wallet | 「失控账单」 | 智能体循环不断产生消费，却没有上限去阻止它 |
| max_tokens | 「每请求上限」 | 单次补全大小的上限 |
| max_turns | 「迭代上限」 | 一个会话中智能体循环迭代次数的上限 |
| max_budget_usd | 「美元终止开关」 | 会话成本上限；突破时中止 |
| Velocity limit | 「速率上限」 | 对短窗口内消费的限制（例如 50 美元 / 10 分钟） |
| Tiered routing | 「先用小模型」 | 默认廉价模型；仅当分类器认为有必要时才升级 |
| Prompt caching | 「缓存的系统提示」 | 提供商侧缓存将重新发送的 token 成本降至近乎为零 |
| HITL checkpoint | 「人工审批门」 | 昂贵操作之前需要人工点头确认 |
