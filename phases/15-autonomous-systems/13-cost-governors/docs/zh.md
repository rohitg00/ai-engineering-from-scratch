# 行动预算、迭代上限和成本治理器

> 一个中型电商代理的月度 LLM 成本在其团队启用"订单追踪"技能后从 1,200 美元跃升至 4,800 美元。这不是定价错误。这是一个发现了新循环并在其中持续消费的代理。微软的 Agent Governance Toolkit（2026 年 4 月 2 日）将针对此类问题的防御措施编纂为：每次请求的 `max_tokens`、每项任务的令牌和美元预算、每日/每月上限、迭代上限、分层模型路由、提示缓存、上下文窗口管理、昂贵行动上的人工介入检查点、预算突破时的终止开关。Anthropic 的 Claude Code Agent SDK 以不同名称提供了相同的原语。财务速度限制——例如，10 分钟内超过 50 美元即切断访问——比月度上限更快地捕捉循环。

**类型：** 学习
**语言：** Python（标准库，分层成本治理器模拟器）
**前置条件：** 第 15 阶段 · 10（权限模式），第 15 阶段 · 12（持久化执行）
**时间：** ~60 分钟

## 问题

自主代理每次轮次都会花费真金白银。聊天机器人的不良输出是一条糟糕的回复；代理的不良循环是一张账单。业界对此失败模式的记录术语是"拒绝钱包"——代理持续推理、持续工具调用、持续计费，而没有任何东西阻止它，因为没有任何东西被设计来阻止。

解决方案不是一个数字。而是在不同时间尺度和粒度上的一堆限制：每次请求、每项任务、每小时、每天、每月。设计良好的堆栈能在几分钟内捕捉失控循环，在几小时内捕捉缓慢泄漏，在一天内捕捉糟糕的发布。当代理是长程且自主时，同一堆栈始终保持预算。

这是一堂工程课：数学是微不足道的，纪律是团队失败的地方。下面的限制列表全部以微软 Agent Governance Toolkit 或 Anthropic Claude Code Agent SDK 文档中的名称命名。

## 概念

### 成本治理器堆栈

1. **每次请求的 `max_tokens`。** 简单。防止任何一次调用发出无限制的补全。
2. **每项任务的令牌预算。** 整个运行过程中，不要超过 N 个令牌。达到上限时硬停止。
3. **每项任务的美元预算。** 与令牌相同，但以货币计。Claude Code 中的 `max_budget_usd`。
4. **每次工具调用上限。** `WebFetch` 调用不超过 N 次，`shell_exec` 调用不超过 N 次，等等。
5. **迭代上限（`max_turns`）。** 代理循环总迭代次数；防止无限推理循环。
6. **每分钟/每小时/每天/每月上限。** 滚动窗口。在不同时间尺度上捕捉泄漏。
7. **财务速度限制。** 例如，"如果 10 分钟内花费超过 50 美元，切断访问。"在月度上限触发之前捕捉基于循环的消耗。
8. **分层模型路由。** 默认使用较小的模型；仅当分类器判断任务需要时才升级到较大的模型。
9. **提示缓存。** 系统提示和稳定上下文存储在提供商缓存中；重新发送的令牌成本接近零。
10. **上下文窗口管理。** 压缩/摘要以保持活动上下文低于阈值；直接减少令牌成本。
11. **昂贵行动上的人工介入检查点。** 在已知昂贵的操作（长工具调用、大下载、昂贵的模型升级）之前，需要人工点击。
12. **预算突破时的终止开关。** 任何上限触发时会话中止。记录上限；需要单独的重新启用路径。

### 为什么是堆栈，而不是一个上限

单一的月度上限只有在钱包花光后才能捕捉失控代理。单一的每次请求上限在会话层面什么都捕捉不到。不同的失败模式需要不同的时间尺度：

- **失控循环**（代理卡在 5 秒重试中）：由速度限制捕捉。
- **缓慢泄漏**（代理每项任务做约 2 倍预期工作）：由每日上限捕捉。
- **糟糕发布**（新版本使用 5 倍令牌）：由每周/每月上限捕捉。
- **合法激增**（真实需求，不是错误）：由小时/天上限捕捉，并带有清晰日志。

### Claude Code 的预算面

Claude Code Agent SDK 公开了（公开文档）：

- `max_turns` —— 迭代上限。
- `max_budget_usd` —— 美元上限；突破时会话中止。
- `allowed_tools` / `disallowed_tools` —— 工具允许列表和拒绝列表。
- 工具使用前的钩子点用于自定义成本核算。

与权限模式阶梯（第 10 课）结合。没有 `max_budget_usd` 的 `autoMode` 会话是不受约束的自主性。Anthropic 明确将 Auto Mode 框架为需要预算控制；分类器与成本正交。

### 欧盟 AI 法案，OWASP Agentic Top 10

微软的 Agent Governance Toolkit 涵盖了 OWASP Agentic Top 10 和欧盟 AI 法案第 14 条（人类监督）要求。对于欧盟的生产环境，日志记录和上限执行不是可选的。

### 观察到的 1,200 美元 → 4,800 美元案例

微软文档中的真实案例：一个电商代理在添加新工具后月度成本翻了三倍。该工具允许代理在每次会话期间轮询订单状态。没有循环检测。没有每次工具上限。没有对周环比增长发出警报。解决方案是每次工具上限加上每日增长警报。这是一个模板：每个新工具表面都是新的潜在循环；每个新工具都需要自己的上限和自己的警报。

## 使用

`code/main.py` 模拟了有和没有分层成本治理器堆栈的代理运行。模拟代理在某些轮次后漂移进轮询循环；分层堆栈在速度窗口内捕捉它，而单一月度上限要到几天后才触发。

## 交付

`outputs/skill-agent-budget-audit.md` 审计拟议代理部署的成本治理器堆栈，并标记缺失的层。

## 练习

1. 运行 `code/main.py`。确认在轮询循环轨迹上，速度限制在迭代上限之前触发。现在禁用速度限制，测量代理在迭代上限捕捉它之前"花费"了多少。

2. 为浏览器代理（第 11 课）设计一组每次工具上限。哪个工具需要最紧的上限？哪个工具可以无风险地无限制运行？

3. 阅读微软 Agent Governance Toolkit 文档。列出工具包命名的每种上限类型。将每种映射到一个失败模式（失控循环、缓慢泄漏、糟糕发布、激增）。

4. 为一个现实任务（例如，"分类仓库中的 50 个问题"）的通宵无人值守运行定价。将 `max_budget_usd` 设置为你点估计的 2 倍。证明 2 倍的合理性。

5. Claude Code 的 `max_budget_usd` 在会话总成本上触发。设计一个你会外部执行的补充速度限制。什么触发切断，重新启用是什么样？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| Denial of Wallet | "失控账单" | 代理循环在没有上限阻止的情况下产生花费 |
| max_tokens | "每次请求上限" | 单次补全大小的上限 |
| max_turns | "迭代上限" | 会话中代理循环迭代次数的上限 |
| max_budget_usd | "美元终止开关" | 会话成本上限；突破时中止 |
| Velocity limit | "速率上限" | 短窗口内花费的限制（例如，50 美元/10 分钟） |
| Tiered routing | "小模型优先" | 默认使用廉价模型；仅当分类器需要时升级 |
| Prompt caching | "缓存的系统提示" | 提供商端缓存将重新发送的令牌成本降至接近零 |
| HITL checkpoint | "人工批准门" | 昂贵操作前需要人工点击 |
