# 操作预算、迭代上限和成本管理者

> 一个中型电商智能体的月度 LLM 成本在团队启用"订单跟踪"技能后从 $1,200 跃升至 $4,800。这不是定价错误。而是一个发现了新循环并持续在其中花费的智能体。Microsoft 的智能体治理工具包（2026年4月2日）将针对此类失败的防御规范化：每个请求的 `max_tokens`、每个任务的 Token 和美元预算、每日/每月上限、迭代上限、分层模型路由、提示缓存、上下文窗口化、在昂贵操作上的 HITL 检查点、预算违反时的终止开关。Anthropic 的 Claude Code Agent SDK 以不同名称提供相同的原语。财务速度限制——例如10分钟内花费 >$50 就切断访问——比月度上限更快地捕捉循环。

**类型：** 学习
**语言：** Python（标准库，分层成本管理者模拟器）
**前置条件：** 第15阶段 · 10（权限模式），第15阶段 · 12（持久执行）
**时间：** 约60分钟

## 问题

自主智能体在每一轮都花费真金白银。聊天机器人的坏输出是一个坏回复；智能体的坏循环是一张账单。行业记录的失败模式术语是"拒绝钱包（Denial of Wallet）"——智能体持续推理、持续调用工具、持续计费，并且没有东西被设计来阻止它。

修复不是一个数字。它是不同时间尺度和粒度的限制堆栈：每个请求、每个任务、每小时、每天、每月。一个设计良好的堆栈在数分钟内捕捉失控循环，在数小时内捕捉慢泄漏，在一天内捕捉坏发布。当智能体是长时域和自主时，相同的堆栈始终保持预算。

这是一门工程课：数学是平凡的，纪律是团队失败的地方。下面列出的限制都在 Microsoft 智能体治理工具包或 Anthropic Claude Code Agent SDK 文档中命名。

## 概念

### 成本管理者堆栈

1. **每个请求的 `max_tokens`。** 简单。防止任何一次调用发出不界限的补全。
2. **每个任务的 Token 预算。** 整个运行不超过 N 个 Token。达到上限时硬停止。
3. **每个任务的美元预算。** 与 Token 相同但是用货币。`max_budget_usd` 在 Claude Code 中。
4. **每个工具调用上限。** 不超过 N 个 `WebFetch` 调用、N 个 `shell_exec` 调用等。
5. **迭代上限（`max_turns`）。** 智能体循环迭代总数；防止无限推理循环。
6. **每分钟/每小时/每天/每月上限。** 滚动窗口。在不同时间尺度上捕捉泄漏。
7. **财务速度限制。** 例如，"如果10分钟内支出超过 $50，切断访问。"在月度上限触发之前捕捉基于循环的消耗。
8. **分层模型路由。** 默认使用较小模型；仅当分类器判断任务保证时才升级到较大模型。
9. **提示缓存。** 系统提示和稳定上下文存储在提供者缓存中；重新发送的 Token 成本接近零。
10. **上下文窗口化。** 压缩/摘要以保持活动上下文低于阈值；直接减少 Token 成本。
11. **在昂贵操作上的 HITL 检查点。** 在已知昂贵的操作（长工具调用、大型下载、昂贵的模型升级）之前，需要人工点击。
12. **预算违反时的终止开关。** 任何上限触发时会话中止。记录上限；需要单独的重新启用路径。

### 为什么是堆栈，而不是一个上限

单个月度上限仅在钱包消失后才捕捉失控智能体。单个每个请求上限在会话级别捕捉不到任何东西。不同的失败模式需要不同的时间尺度：

- **失控循环**（智能体卡在5秒重试中）：被速度限制捕捉。
- **慢泄漏**（智能体每个任务做 ~2倍预期工作）：被每日上限捕捉。
- **坏发布**（新版本使用5倍 Token）：被每周/每月上限捕捉。
- **合法激增**（真实需求，不是 Bug）：被带有清晰日志的小时/天上限捕捉。

### Claude Code 的预算表面

Claude Code Agent SDK 公开（公共文档）：

- `max_turns` — 迭代上限。
- `max_budget_usd` — 美元上限；违反时中止。
- `allowed_tools` / `disallowed_tools` — 工具允许列表和拒绝列表。
- 工具使用前的钩子点，用于自定义成本记账。

与权限模式阶梯结合使用（第10课）。没有 `max_budget_usd` 的 `autoMode` 会话是无管理的自主性。Anthropic 明确将自动模式框定为需要预算控制；分类器与成本正交。

### EU AI 法案，OWASP 智能体 Top 10

Microsoft 的智能体治理工具包涵盖 OWASP Agentic Top 10 和 EU AI 法案第14条（人类监督）要求。对于 EU 中的生产，日志记录和上限执行不是可选的。

### 观察到的 $1,200 → $4,800 案例

Microsoft 文档中的真实案例：一个电商智能体在添加新工具后月度成本翻了三倍。该工具允许智能体在每个会话期间轮询订单状态。没有循环检测。没有每工具上限。没有周环比增长警报。修复是每工具上限加上每日增长警报。这是一个模板：每个新工具表面都是一个潜在的新循环；每个新工具都需要自己的上限和自己的警报。

## 使用

`code/main.py` 模拟带和不带分层成本管理者堆栈的智能体运行。模拟智能体在若干轮后漂移到轮询循环中；分层堆栈在速度窗口内捕捉它，而单个月度上限要到数天后才会触发。

## 练习

1. 运行 `code/main.py`。确认速度限制在轮询循环轨迹上在迭代上限之前触发。现在禁用速度限制并测量在迭代上限捕捉之前智能体"花费"多少。
2. 为浏览器智能体（第11课）设计一套每工具上限。哪个工具需要最严格的上限？哪个工具可以在没有风险的情况下无界限运行？
3. 阅读 Microsoft 智能体治理工具包文档。列出工具包命名的每种上限类型。将每种映射到一种失败模式（失控循环、慢泄漏、坏发布、激增）。
4. 为一个现实任务（例如，"分类仓库中的50个问题"）的夜间无人值守运行定价。将 `max_budget_usd` 设为你的点估计的2倍。证明 2 倍的合理性。
5. Claude Code 的 `max_budget_usd` 在会话聚合成本上触发。设计一个你将在外部强制的补充速度限制。什么触发切断，重新启用看起来像什么？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| 拒绝钱包（Denial of Wallet） | "失控账单" | 智能体循环产生支出而没有上限来阻止它 |
| max_tokens | "每个请求上限" | 单个补全大小的上限 |
| max_turns | "迭代上限" | 会话中智能体循环迭代的上限 |
| max_budget_usd | "美元终止开关" | 会话成本上限；违反时中止 |
| 速度限制（Velocity Limit） | "速率上限" | 短窗口内的支出限制（例如，$50 / 10分钟） |
| 分层路由（Tiered Routing） | "先小模型" | 便宜模型默认；仅当分类器保证时才升级 |
| 提示缓存（Prompt Caching） | "缓存的系统提示" | 提供者端缓存将重新发送的 Token 成本降低到接近零 |
| HITL 检查点（HITL Checkpoint） | "人工批准门控" | 在昂贵操作之前需要人工点击 |
