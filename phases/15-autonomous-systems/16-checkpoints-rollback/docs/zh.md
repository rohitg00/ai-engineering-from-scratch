# 检查点与回滚

> 每个图状态转换都持久化。当工作器崩溃时，其租约到期，另一个工作器在最新检查点接管。Cloudflare Durable Objects 在数小时或数周内保持状态。提议-然后-提交（课程 15）为每个操作定义回滚计划。操作后验证闭合循环。欧盟 AI 法案第 14 条强制要求对高风险系统进行有效人为监督——实践中这意味着检查点必须可查询、回滚必须演练，并且审计跟踪必须经受住部署。尖锐的失败模式：没有幂等性密钥和前置条件检查，瞬态故障后的重试可能双重执行已批准的操作。操作后验证是捕获它的方法。

**类型：** 学习
**语言：** Python（标准库，检查点和回滚状态机）
**前置条件：** 阶段 15 · 12（持久执行），阶段 15 · 15（提议-然后-提交）
**时间：** ~60 分钟

## 问题

持久执行（课程 12）使崩溃的智能体可恢复。提议-然后-提交（课程 15）使已批准的操作可审计。本课连接它们：当已批准的操作部分执行、崩溃和恢复时会发生什么？回滚何时运行，针对什么状态？

实际系统以不同方式连接：

- **LangGraph** 将每个图状态转换检查点到 PostgreSQL。工作器崩溃时，租约释放，另一个工作器在最新检查点恢复。工作流在 `interrupt()` 上暂停，它本身也持久化。
- **Cloudflare Durable Objects** 在数小时或数周内保持每键状态。将计算与已批准操作的存储共置。
- **Microsoft Agent Framework** 在工作流 API 中公开 `Checkpoint` 原语；重放加幂等性覆盖重试。

在每种情况下，实际有效的组合是：幂等性密钥（防止双重执行）+ 前置条件检查（状态仍与我们批准的相符）+ 操作后验证（副作用实际发生）+ 验证失败时回滚。

## 概念

### 每个转换都持久化

图状态转换是将工作流从一个命名状态移动到另一个的任何步骤。天真的实现仅在特定提交点持久化；生产实现持久化每个转换。成本（一些额外的写入）相对于可靠性增益（重放到达任何地方，租约恢复是精确的）很小。

### 租约恢复

当工作器崩溃时，工作流不会丢失；租约（此工作器正在执行此运行的短期声明）简单地到期。另一个工作器接管最新检查点并恢复。租约机制使生产系统能够在滚动部署中幸存而不会丢失进行中的工作。

### 幂等性加前置条件

仅有幂等性是不够的。考虑：工作流被批准为"当余额 > $1000 时从 A 向 B 转账 $100。"工作流已提交，在中途执行时崩溃并恢复。如果仅检查幂等性密钥，并且执行恢复，转账运行一次（正确）。但考虑在崩溃和恢复之间，A 的余额通过不同的工作流降至 $500。幂等性检查仍通过；前置条件不通过。没有前置条件检查，我们发送了透支。

每个重大操作都需要两者：

- **幂等性密钥**：防止双重执行。
- **前置条件检查**：确认状态仍与批准的一致。

### 操作后验证

"工具返回 200"不是验证。真正的验证重新读取目标状态并确认副作用实际发生。模式：

- 数据库更新：`UPDATE ... RETURNING *` 然后断言返回的行与预期状态匹配。
- 邮件发送：提交后检查已发送文件夹中的消息 ID。
- 文件写入：读回文件并哈希它。
- API 调用：在目标资源上进行后续 `GET`。

如果验证失败，工作流处于已知错误状态。回滚启动。

### 回滚计划

提议-然后-提交（课程 15）中的每个重大操作都带有回滚计划。类型：

- **带内回滚**：直接反转副作用（`INSERT` 后的 `DELETE`，发送后的 `发送更正邮件`）。
- **补偿事务**：中和原始操作的新操作（标准 SAGA 模式）。
- **带外回滚**：警告人类，暂停工作流，留下错误状态以供调查。

无操作回滚（"我们无法撤销此操作"）必须在提议中命名。没有回滚的操作在提交时需要更强的 HITL（课程 15 挑战-响应）。

### 欧盟 AI 法案第 14 条操作解读

第 14 条要求对高风险系统进行"有效人为监督"。在操作术语中，实施者将其解读为：

- 检查点可由审计员查询。
- 回滚已演练（至少端到端测试一次）。
- 审计跟踪经受住部署（检查点后端不是临时的）。
- 验证失败会触发警报，而不是静默记录。

在提交中途崩溃、恢复并在没有验证 + 回滚路径的情况下完成副作用的工作流无法通过第 14 条测试。

### 尖锐的失败模式：双重执行

此领域中最常见的生产事件：

1. 操作批准，幂等性密钥 k。
2. 提交开始，执行，返回 200。
3. 在持久化"已提交"状态之前工作流崩溃。
4. 工作流恢复；看到"已批准但未提交"；重新执行。
5. 副作用触发两次。

缓解措施：在执行之前持久化"进行中"意图，使用幂等性密钥执行，然后仅在操作后验证成功后将标记"已提交"。如果操作触发且状态写入失败，您知道要验证并（如有必要）重新触发。如果状态写入成功且操作失败，您通过恢复路径精确触发一次。

## 使用

`code/main.py` 实现了带幂等性、前置条件、验证和回滚的检查点工作流。驱动程序模拟四种场景：干净运行、崩溃后重试（幂等性捕获）、前置条件失败（工作流在未触发的情况下中止）、验证失败（回滚触发）。

## 交付

`outputs/skill-rollback-rehearsal.md` 为提议的工作流设计回滚演练测试，并审计检查点后端的审计跟踪持久性。

## 练习

1. 运行 `code/main.py`。验证四种场景。对于提交期间崩溃的情况，确认操作在重试中精确触发一次。

2. 修改"先标记为完成，然后执行它"模式，使状态写入在操作后触发。重新运行崩溃场景。测量触发了多少重复操作。

3. 为特定生产操作（例如，"发布到 Slack 频道"）设计回滚计划。分类为带内、补偿或带外。证明选择的合理性。

4. 取一个您知道的工作流。识别每个状态转换。用持久性要求（持久化/不持久化）标记每个。计算您当前未持久化的那些。

5. 演练回滚测试：设计一个端到端测试，运行真实工作流，使其崩溃，并确认回滚路径触发。测试断言什么？
