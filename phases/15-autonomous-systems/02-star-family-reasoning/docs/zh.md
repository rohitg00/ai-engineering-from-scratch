# STaR、V-STaR、Quiet-STaR — 自学推理（Self-Taught Reasoning）

> 最小可能的自我改进循环位于推理链内部。模型生成思维链（Chain of Thought），保留那些得出正确答案的，并在这些基础上进行微调。这就是 STaR。V-STaR 增加了一个验证器，以便在推理时进行更好的选择。Quiet-STaR 将推理链下推到每个 Token。这三种方法都有效。但它们都不是魔法——循环会保留任何碰巧得出正确答案的捷径。

**类型：** 学习
**语言：** Python（标准库，自举循环模拟器）
**前置条件：** 第13阶段 · 01-03（推理和 CoT），第15阶段 · 01（长时域框架）
**时间：** 约60分钟

## 问题

教模型推理的直接方法是收集人类编写的推理轨迹。这既昂贵又缓慢，并且受限于人类愿意编写高质量思维链（Chain-of-Thought）的数量。

STaR（Self-Taught Reasoner，Zelikman 等人，2022）提出了一个问题：如果模型自己编写推理依据（Rationale）并根据已知答案对其进行评分呢？循环如下：

1. 采样一个推理轨迹加答案。
2. 如果最终答案正确，保留该轨迹。
3. 在保留的轨迹上进行微调。
4. 重复。

它有效。GSM8K 和 CommonsenseQA 都在没有新人类标注的情况下得到了改进。但循环有一个内置偏差：任何产生正确答案的推理依据都会被保留，无论推理本身是否合理。V-STaR（Hosseini 等人，2024）用学习的验证器修补了这个问题；Quiet-STaR（Zelikman 等人，2024）将这一思想推广到每个 Token 的内部推理依据。

## 概念

### STaR：对有效内容进行自举

从具有一定弱推理能力的基础模型开始。在每个训练问题上，采样一个推理依据加答案。如果答案与标签匹配，保留（问题、推理依据、答案）三元组。在保留集上微调模型。重复。

有一个转折很重要。如果模型永远无法正确回答某个问题，循环就无法在其上学习。STaR 增加了**合理化（Rationalization）**：对于模型失败的问题，注入正确答案作为提示，并重新提示模型生成通向该答案的推理依据。合理化后的推理依据被添加到训练集中。

原始论文（Zelikman 等人，2022）中的结果：GPT-J 基础模型通过带有合理化的重复 STaR 轮次，在 GSM8K 上从 5.8% 提高到 10.7%——大约提高了5个百分点。在 CommonsenseQA 上，经过 STaR 训练的 GPT-J 6B 达到了 72.5%，与经过手工标注推理依据微调的 GPT-3 175B（约73%）相当——后者是一个大约30倍大的模型。

### V-STaR：用 DPO 训练验证器

STaR 丢弃不正确的推理依据。Hosseini 等人（2024）观察到这些也是数据：每个（推理依据，"这是否正确"）对都可以训练一个验证器。他们使用正确和不正确解决方案上的直接偏好优化（DPO）来构建排序器。在推理时，采样 N 个推理依据并选择验证器的首选。

报告的增量：在 GSM8K 和 MATH 上，比先前的自我改进基线高出4到17个百分点，大部分增益来自使用验证器进行推理时选择，而不是用于额外的生成器微调。

### Quiet-STaR：每 Token 内部推理依据

Zelikman 等人（2024）问道：如果模型学会在每个 Token 位置生成简短的内部推理依据，而不仅仅是在问题和答案之间呢？Quiet-STaR 训练模型在每个预测 Token 之前发出一个隐藏的"思考"，然后通过学习权重将感知思考的预测与基线预测混合。

结果：Mistral 7B 在 GSM8K 上的绝对零样本改进从 5.9% 提高到 10.9%，在 CommonsenseQA 上从 36.3% 提高到 47.2%，无需特定任务的微调。模型学会了"何时思考"——困难 Token 获得更长的内部推理依据；简单 Token 几乎不获得。

### 为什么这三种方法都有安全顾虑

所有三种方法都使用最终答案作为梯度信号。通过有缺陷推理——利用捷径、猜测或使用非泛化模式——得出正确答案的推理依据会得到正强化。在分布内（In-Distribution）问题上，捷径有效。在分布外（Out-of-Distribution）问题上，它会静默失效。

V-STaR 的验证器通过学习对推理依据进行排序来缓解，但验证器是在同一标签集上训练的。它可以学会偏爱格式良好的错误推理而非诚实的不确定性。更安全的设计是将 STaR 风格的数据与（a）过程监督奖励模型（奖励中间步骤，而不仅仅是答案）和（b）打破简单捷径的留出 OOD 评估相结合。

### 对比

| 方法 | 训练信号 | 推理成本 | 数据浪费 | 已知失败模式 |
|---|---|---|---|---|
| STaR | 如果正确则保留（推理依据，答案） | 1x | 丢弃所有不正确的推理依据 | 捷径推理依据 |
| STaR + 合理化 | 上述 + 正确答案提示重试 | 1x | 较少 | 合理化的推理依据可能不可信 |
| V-STaR | STaR + 来自两类的 DPO 验证器 | Nx（Best-of-N） | 最小 | 验证器可能强化自信的错误 |
| Quiet-STaR | 每 Token 推理依据 + 混合权重 | 1.5-3x | 最小 | 仍然是答案条件梯度 |

### 这在2026年技术栈中的位置

STaR 已经过时。但这一模式在2025-2026年无处不在。可验证数学问题上的 RL（DeepSeek-R1、Kimi-k1.5、o1）是 STaR 的答案条件梯度信号，进行了扩展。过程奖励模型（Lightman 等人，2023；OpenAI 的"Let's verify step by step"）是过程监督的替代方案。AlphaEvolve（第3课）是代码版的 STaR，用程序评估器代替标签。Darwin Godel Machine（第4课）是智能体脚手架本身的 STaR。

理解 STaR 可以让所有这些变得清晰。它是最小可行的自我改进循环。

## 使用

`code/main.py` 在玩具算术任务上运行模拟的 STaR 循环。你可以观察：

- 准确率如何在自举轮次中攀升。
- 捷径如何潜入：模拟器包括一个"懒惰"推理依据类，它在40%的时间内得出正确答案，但泛化性差。观察 STaR 是否保留它们。
- 验证器（V-STaR 风格）如何在推理时提供帮助，但无法完全修剪训练期间引入的捷径。

## 实战

`outputs/skill-star-loop-reviewer.md` 帮助你在训练之前审计提议的自学推理管道。

## 练习

1. 运行模拟器。将捷径频率设为零，然后设为 0.4。即使两者在训练分布上都达到 >90%，两次运行的最终准确率有多大差异？

2. 向模拟器添加一个留出的 OOD 测试。从不同的分布中抽取问题，并在分布内和 OOD 集上评估自举模型。量化差距。

3. 阅读 Quiet-STaR 论文（arXiv:2403.09629）第3节。用三句话分别解释"思考结束（End-of-Thought）"Token 和混合权重头（Mixing-Weight Head）。

4. 将 STaR 的如果正确则保留过滤器与独立奖励每个推理依据步骤的过程监督替代方案进行比较。确定标注成本差异和合理的质量差异。

5. 设计一个能够捕捉已部署模型中捷径推理依据的评估。它不必完美——它必须打破 STaR 循环会强化的简单捷径。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| STaR | "自学推理器（Self-Taught Reasoner）" | 在模型生成的得出正确答案的推理依据上进行微调；重复 |
| 合理化（Rationalization） | "提示重试" | 注入正确答案并为基模型失败的问题重新提示推理依据 |
| V-STaR | "验证器 STaR" | 在正确和不正确的推理依据上用 DPO 训练验证器，用于推理时选择 |
| Quiet-STaR | "每 Token 推理依据" | 在每个 Token 位置生成隐藏思考；与基线预测混合 |
| 答案条件梯度（Answer-Conditioned Gradient） | "基于结果的信号" | 训练循环奖励最终答案，而不是推理步骤 |
| 过程奖励模型（Process Reward Model） | "步骤级验证器" | 在每步正确性而非结果上训练的奖励模型——与 STaR 形成对比 |
| 捷径推理依据（Shortcut Rationale） | "答案正确，推理错误" | 通过非泛化模式达到标签的推理依据；STaR 保留这些 |
