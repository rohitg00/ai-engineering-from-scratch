# LLM 路由层 — LiteLLM、OpenRouter、Portkey

> 提供商锁定成本高昂。不同的工具调用工作负载适合不同的模型。路由网关提供一个 API 表面、重试、故障转移、成本跟踪和防护措施。2026 年主导三种原型：LiteLLM（开源自托管）、OpenRouter（托管 SaaS）、Portkey（生产级，2026 年 3 月开源）。本课命名了决策标准并演练了 stdlib 路由网关。

**类型：** 学习
**语言：** Python (stdlib, 路由 + 故障转移 + 成本跟踪器)
**前置条件：** 阶段 13 · 02 (函数调用), 阶段 13 · 17 (网关)
**时间：** ~45 分钟

## 学习目标

- 区分自托管、托管和生产级路由选项。
- 实现以定义的优先级顺序在提供商失败时重试的回退链。
- 跨提供商跟踪每请求成本和令牌使用。
- 根据给定的生产约束在 LiteLLM、OpenRouter 和 Portkey 之间做出选择。

## 问题背景

提供商路由重要的场景：

1. **成本。** Claude Sonnet 成本是 Haiku 的 3 倍。对于分流任务，Haiku 足够；对于综合任务，Sonnet 值得。每请求路由。

2. **故障转移。** OpenAI 有一个糟糕的时刻。每个请求都失败。你想要自动回退到 Anthropic 而无需重新部署。

3. **延迟。** 实时聊天 UI 需要快速的首令牌时间。批量摘要器不需要。按延迟 SLA 路由。

4. **合规性。** 欧盟用户必须留在欧盟区域。按区域路由。

5. **实验。** 在同一工作负载上 A/B 两个模型。按测试桶路由。

手动编码所有这些每个集成都是重复的。路由网关提供一个 OpenAI 兼容的 API 并处理其余部分。

## 概念详解

### OpenAI 兼容代理形态

每个人都讲 OpenAI 形态。路由网关暴露 `/v1/chat/completions`，接受 OpenAI 模式，并在内部代理到 Anthropic / Gemini / Cohere / Ollama / 任何。客户端不在乎。

### 模型别名

而不是 `claude-3-5-sonnet-20251022`，你的代码说 `our_smart_model`。网关将别名映射到真实模型。当 Anthropic 发布 Claude 4 时，你在服务器端更改别名；你的代码不触碰任何东西。

### 回退链

```
primary: openai/gpt-4o
on 5xx: anthropic/claude-3-5-sonnet
on 5xx: google/gemini-1.5-pro
on 5xx: refuse
```

网关在配置中定义这个。重试按预算计数，以便回退级联不会爆炸成本。

### 语义缓存

相同或近乎相同的提示词命中缓存而不是提供商。重复智能体循环的节省可以是 30% 到 60%。密钥是基于嵌入的；近乎相同的提示词共享一个缓存槽。

### 防护措施

网关级别：

- **PII 编辑。** 在发送提示词之前进行基于正则表达式或 ML 的传递。
- **策略违规。** 拒绝带有禁止内容的提示词。
- **输出过滤器。** 擦洗补全中的泄漏。

Portkey 和 Kong 都发布观点化的防护措施。LiteLLM 让它们可选。

### 每密钥速率限制

一个 API 密钥 = 一个团队。每密钥预算防止一个团队消耗共享配额。大多数网关支持这一点。

### 自托管 vs 托管权衡

| 因素 | LiteLLM (自托管) | OpenRouter (托管) | Portkey (生产) |
|--------|----------------------|----------------------|----------------------|
| 代码 | 开源, Python | 托管 SaaS | 开源 (2026 年 3 月) + 托管 |
| 设置 | 部署代理 | 注册 | 任一 |
| 提供商 | 100+ | 300+ | 100+ |
| 计费 | 你自己的密钥 | OpenRouter 信用 | 你自己的密钥 |
| 可观测性 | OpenTelemetry | 仪表板 | 完整 OTel + PII 编辑 |
| 最适合 | 想要完全控制的团队 | 快速原型 | 带合规性的生产 |

当你有 SRE 团队并想要数据主权时，LiteLLM 获胜。当你想要单个订阅且没有基础设施时，OpenRouter 获胜。当你需要开箱即用的防护措施和合规性时，Portkey 获胜。

### 成本跟踪

每个请求携带 `provider`、`model`、`input_tokens`、`output_tokens`。乘以每模型每令牌价格（从网关维护的定价表中提取）。每用户 / 每团队 / 每项目聚合。

### MCP 加路由

网关可以路由 LLM 调用和 MCP 采样请求。当采样请求的 modelPreferences 偏好特定模型时，网关转换到正确的后端。这是阶段 13 · 17 (MCP 网关) 和本课路由网关有时合并为一个服务的地方。

### 路由策略

- **静态优先级。** 列表中的第一个；错误时回退。
- **负载均衡。** 轮询或加权。
- **成本感知。** 选择满足延迟 / 质量的最便宜模型。
- **延迟感知。** 在过去 N 分钟内选择最快的模型。
- **任务感知。** 提示词分类器将编码路由到一个模型，将摘要路由到另一个。

## 练习

1. 运行 `code/main.py`。触发中断场景；确认回退落在第二个提供商上并且成本被正确归因。

2. 添加语义缓存：提示词的 SHA256 是查找密钥；缓存命中立即返回。测量重复调用上的成本节省。

3. 添加一个提示词分类器，将"code ..."提示词路由到偏好智能的别名，将"summarize ..."提示词路由到偏好速度的别名。

4. 设计每团队预算：每个团队有一个月度支出上限；一旦达到上限，网关拒绝请求。选择执行粒度（每请求或窗口化）。

5. 并排阅读 LiteLLM、OpenRouter 和 Portkey 文档。命名每个发布的其他两个没有的一个功能。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|---------|
| 路由网关 | "LLM 代理" | 许多提供商前面的单 API 表面层 |
| OpenAI 兼容 | "讲 OpenAI 模式" | 接受 `/v1/chat/completions` 形态，转换到任何后端 |
| 模型别名 | "our_smart_model" | 网关映射到具体模型的代码中的名称 |
| 回退链 | "重试列表" | 失败时尝试的提供商排序列表 |
| 语义缓存 | "提示词嵌入缓存" | 密钥是提示词的嵌入；近乎重复共享缓存命中 |
| 防护措施 | "输入/输出过滤器" | 编辑 PII，拒绝策略违规 |
| 每密钥速率限制 | "团队预算" | 限定到 API 密钥的配额 |
| 成本跟踪 | "每请求支出" | 聚合令牌使用 x 每模型价格 |
| LiteLLM | "开源代理" | 可自托管 OSS 路由网关 |
| OpenRouter | "托管 SaaS" | 带基于信用的计费的托管网关 |
| Portkey | "生产选项" | 内置防护措施的开源 + 托管 |
