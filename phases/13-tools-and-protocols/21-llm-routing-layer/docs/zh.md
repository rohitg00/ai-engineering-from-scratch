# LLM 路由层 —— LiteLLM、OpenRouter、Portkey

> 提供商锁定成本高昂。不同的工具调用工作负载适合不同的模型。路由网关提供单一 API 表面、重试、故障转移、成本跟踪和护栏。2026 年三种原型占据主导：LiteLLM（开源自托管）、OpenRouter（托管 SaaS）、Portkey（生产级，2026 年 3 月开源）。本课介绍决策标准并演练 stdlib 路由网关。

**类型：** Learn
**语言：** Python（stdlib，路由 + 故障转移 + 成本跟踪器）
**前置知识：** Phase 13 · 02（函数调用），Phase 13 · 17（网关）
**时间：** ~45 分钟

## 学习目标

- 区分自托管、托管和生产级路由选项。
- 实现一个在提供商故障时按定义优先级顺序重试的备用链。
- 跟踪跨提供商的每次请求成本和令牌使用。
- 针对给定生产约束在 LiteLLM、OpenRouter 和 Portkey 之间做出选择。

## 问题所在

提供商路由重要的场景：

1. **成本。** Claude Sonnet 的成本是 Haiku 的 3 倍。对于分类任务，Haiku 足够；对于综合任务，Sonnet 值得。按请求路由。

2. **故障转移。** OpenAI 有一个糟糕的小时。每次请求都失败。你希望自动回退到 Anthropic 而无需重新部署。

3. **延迟。** 实时聊天 UI 需要快速的首令牌时间。批处理摘要器不需要。按延迟 SLA 路由。

4. **合规。** 欧盟用户必须留在欧盟区域。按区域路由。

5. **实验。** 对相同工作负载进行两个模型的 A/B 测试。按测试桶路由。

为每个集成手动编码所有这些是重复的。路由网关提供单一 OpenAI 兼容 API 并处理其余部分。

## 核心概念

### OpenAI 兼容代理形式

每个人都使用 OpenAI 形式。路由网关暴露 `/v1/chat/completions`，接受 OpenAI 模式，并在内部代理到 Anthropic / Gemini / Cohere / Ollama / 任何。客户端不关心。

### 模型别名

你的代码不说 `claude-3-5-sonnet-20251022`，而是说 `our_smart_model`。网关将别名映射到真实模型。当 Anthropic 发布 Claude 4 时，你在服务端更改别名；你的代码无需改动。

### 备用链

```
primary: openai/gpt-4o
on 5xx: anthropic/claude-3-5-sonnet
on 5xx: google/gemini-1.5-pro
on 5xx: refuse
```

网关在配置中定义此内容。重试计入预算，以便备用级联不会爆炸成本。

### 语义缓存

相同或近相同的提示命中缓存而非提供商。重复智能体循环的节省可达 30% 到 60%。键基于嵌入；近相同提示共享缓存槽。

### 护栏

网关级：

- **PII 脱敏。** 发送提示前的正则表达式或基于 ML 的传递。
- **策略违规。** 拒绝包含禁止内容的提示。
- **输出过滤器。** 清理完成中的泄漏。

Portkey 和 Kong 都提供有主见的护栏。LiteLLM 将它们设为可选。

### 每密钥速率限制

一个 API 密钥 = 一个团队。每密钥预算防止一个团队消耗共享配额。大多数网关支持此功能。

### 自托管与托管的权衡

| 因素 | LiteLLM（自托管） | OpenRouter（托管） | Portkey（生产级） |
|--------|----------------------|----------------------|----------------------|
| 代码 | 开源，Python | 托管 SaaS | 开源（2026 年 3 月）+ 托管 |
| 设置 | 部署代理 | 注册 | 任选 |
| 提供商 | 100+ | 300+ | 100+ |
| 计费 | 你自己的密钥 | OpenRouter 积分 | 你自己的密钥 |
| 可观察性 | OpenTelemetry | 仪表板 | 完整 OTel + PII 脱敏 |
| 最适合 | 想要完全控制的团队 | 快速原型设计 | 需要合规的生产环境 |

当你有 SRE 团队并想要数据主权时，LiteLLM 获胜。当你想要单一订阅且无需基础设施时，OpenRouter 获胜。当你需要开箱即用的护栏和合规时，Portkey 获胜。

### 成本跟踪

每个请求携带 `provider`、`model`、`input_tokens`、`output_tokens`。乘以每个模型每令牌价格（从网关维护的定价表中提取）。每用户 / 每团队 / 每项目聚合。

### MCP 加路由

网关可以路由 LLM 调用和 MCP 采样请求。当采样请求的 modelPreferences 偏好特定模型时，网关转换到正确的后端。这是 Phase 13 · 17（MCP 网关）和本课路由网关有时合并为一个服务的地方。

### 路由策略

- **静态优先级。** 列表中的第一个；错误时回退。
- **负载均衡。** 轮询或加权。
- **成本感知。** 选择满足延迟 / 质量的最便宜模型。
- **延迟感知。** 选择最近 N 分钟中最快的模型。
- **任务感知。** 提示分类器将编码路由到一个模型，将摘要路由到另一个模型。

## 使用它

`code/main.py` 在约 150 行中实现路由网关：接受 OpenAI 形式的请求，转换到每提供商存根，运行优先级备用链，跟踪每次请求成本，并在输入上应用 PII 脱敏传递。用三个场景运行它：正常请求、触发回退的主提供商中断、被脱敏捕获的 PII 泄漏。

需要查看的内容：

- `ROUTES` 字典：别名 -> 按优先级排序的具体提供商列表。
- 备用循环在 5xx 时重试。
- 成本跟踪器将令牌使用乘以每模型费率。
- PII 脱敏器在转发前清理 SSN 形状的模式。

## 交付它

本课产出 `outputs/skill-routing-config-designer.md`。给定工作负载配置文件（延迟、成本、合规），该技能选择 LiteLLM / OpenRouter / Portkey 并产出路由配置。

## 练习

1. 运行 `code/main.py`。触发中断场景；确认回退落在第二个提供商上，并且成本正确归属。

2. 添加语义缓存：提示的 SHA256 是查找键；缓存命中立即返回。测量重复调用的成本节省。

3. 添加一个提示分类器，将"code ..."提示路由到偏向智能的别名，将"summarize ..."提示路由到偏向速度的别名。

4. 设计每团队预算：每个团队有每月支出上限；一旦达到上限，网关拒绝请求。选择执行粒度（每次请求或窗口化）。

5. 并排阅读 LiteLLM、OpenRouter 和 Portkey 文档。命名每个提供而其他两个没有的一个功能。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 路由网关 | "LLM 代理" | 位于多个提供商前面的单一 API 表面层 |
| OpenAI 兼容 | "使用 OpenAI 模式" | 接受 `/v1/chat/completions` 形式，转换到任何后端 |
| 模型别名 | "our_smart_model" | 你的代码中的名称，网关将其映射到具体模型 |
| 备用链 | "重试列表" | 故障时尝试的按顺序排列的提供商列表 |
| 语义缓存 | "提示嵌入缓存" | 键是提示的嵌入；近重复共享缓存命中 |
| 护栏 | "输入/输出过滤器" | 脱敏 PII，拒绝策略违规 |
| 每密钥速率限制 | "团队预算" | 限定到 API 密钥的配额 |
| 成本跟踪 | "每次请求支出" | 聚合令牌使用 x 每模型价格 |
| LiteLLM | "开放代理" | 可自托管的 OSS 路由网关 |
| OpenRouter | "托管 SaaS" | 基于信用计费的托管网关 |
| Portkey | "生产选项" | 内置护栏的开源 + 托管 |
