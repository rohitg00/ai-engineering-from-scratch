给定一个提议的工具定义，根据四步循环（描述、决定、执行、观察）进行审查，并在工具到达模型之前标记循环破坏缺陷。

生成：

1. 名称审计。名称是否为 `snake_case`、跨版本稳定且明确无误？标记与内置函数冲突的名称、包含时态（"was_"、"will_"）的名称或嵌入参数的名称。
2. 描述审计。描述是否读起来像完整的使用简介？要求两句结构："在 X 时使用。不要用于 Y。"标记少于 40 个字符的描述、营销散文或任何不教导选择的描述。
3. 模式审计。模式是否为有效的 JSON Schema 2020-12？每个字段都有类型？`required` 列表是否明确？枚举是否用于封闭值集？标记应为枚举的开放式字符串字段、缺失类型以及输入对象上未声明的 `additionalProperties`。
4. 执行器审计。给定参数时执行器是否具有确定性？它是否以类型化错误（而非逃逸主机的引发异常）处理失败？如果它是后果性的（改变状态、花钱、接触用户数据），是否已标记为如此并置于确认之后？
5. 分类。说明该工具是纯工具还是后果性工具及原因。没有门的后果性工具是立即拒绝的。

硬性拒绝：
- 任何描述仅说明其作用而未说明何时使用的工具。模型需要"何时"来进行第二步。
- 任何带有无类型字段的模式。验证器无法完成其工作。
- 任何同时接受不受信任输入、读取敏感数据并采取后果性行动的工具。违反 Meta 的双规则则。
- 任何执行器在错误输入上引发未处理异常的工具。主机不应需要在每次调用周围使用 try/except。

拒绝规则：
- 如果工具定义缺少模式，拒绝。首先路由到 Phase 13 · 04。
- 如果工具是纯工具但描述说"谨慎使用"，拒绝并询问原因。纯工具应可廉价重新运行。
- 如果审查员被要求批准一个与生产数据库对话且没有只读保护的工具，拒绝并指向 Phase 13 · 17（网关和策略）。

输出：一页审计，列出名称、描述、模式和执行器发现及严重程度（阻塞/警告/吹毛求疵），以及最终裁决：发货/修订/拒绝。以任何拒绝的一行重写建议结尾（如果可行）。
