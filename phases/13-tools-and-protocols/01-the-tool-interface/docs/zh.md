# 01 · 工具接口——为什么智能体需要结构化输入输出

> 语言模型产出 token，而程序执行动作。横亘在两者之间的鸿沟就是工具接口（tool interface）：一份让模型能够请求某个动作、宿主能够执行该动作的契约。2026 年的每一套技术栈——OpenAI、Anthropic、Gemini 上的函数调用（function calling）；MCP 的 `tools/call`；A2A 的任务分片（task parts）——都是同一个四步循环的不同编码方式。本课为这个循环命名，并展示运行它所需的最小机制。

**类型：** 学习
**语言：** Python（标准库，不调用 LLM）
**前置：** 第 11 阶段（LLM 补全 API）
**时长：** 约 45 分钟

## 学习目标

- 解释为什么一个只能生成文本的 LLM，单凭自身无法对真实世界采取行动。
- 画出四步工具调用循环（描述 → 决策 → 执行 → 观察），并指出每一步由谁负责。
- 用三部分写出一份工具描述：名称、JSON Schema 输入，以及一个确定性的执行器函数。
- 区分纯工具（pure）与有副作用工具（side-effecting），并阐明这种划分对安全为何重要。

## 问题所在

LLM 输出的是下一个 token 上的概率分布。这就是它全部的输出界面。如果你问聊天模型「班加罗尔现在天气怎么样」，它能写出一句看似合理的话，但它无法拨通某个天气 API。那句话可能恰好蒙对，也可能已经过时三天。

弥合这道鸿沟正是工具接口的目的。宿主程序（host program）——你的智能体运行时、Claude Desktop、ChatGPT、Cursor，或一段自定义脚本——向模型广告（advertise）一份可调用的工具清单。当模型判断需要采取某个动作时，它输出一个结构化载荷（payload），其中指明工具名及其参数。宿主解析该载荷，真正运行该工具，再把结果回喂给模型。循环持续进行，直到模型判断不再需要调用为止。

这份契约的第一个版本于 2023 年 6 月随 OpenAI 的 "functions" 参数发布。Anthropic 随后在 Claude 2.1 中以 `tool_use` 块跟进。几个月后 Gemini 加入了 `functionDeclarations`。如今每家供应商都暴露出相同的形态：输入一份由 JSON Schema 类型化的工具清单，输出一个 JSON 载荷形式的工具调用。模型上下文协议（Model Context Protocol，2024 年 11 月）将这份契约通用化，使一个工具注册表（registry）能服务于每一种模型。A2A（2026 年 4 月，v1.0）则把同一原语叠加到智能体之间的委派（agent-to-agent delegation）上。

四步循环是所有这些之下的不变量（invariant）。第 13 阶段的其余一切都是它的展开。

## 核心概念

### 第一步：描述

宿主用三个字段声明每个工具。

- **名称（Name）。** 一个稳定、机器可读的标识符。是 `get_weather`，而不是「weather thing」。
- **描述（Description）。** 一段自然语言的简介。「当用户询问某座城市的当前状况时使用。不要用于历史数据。」
- **输入 schema（Input schema）。** 一个描述工具参数的 JSON Schema 对象（draft 2020-12）。

模型接收到这份清单。现代供应商会用各自特定的模板把这些声明序列化进系统提示词（system prompt）里，因此你作为调用方只需处理这套结构化形式。

### 第二步：决策

在给定用户消息和可用工具后，模型从三种行为中择其一。

1. **直接用文本回答（Answer directly）。** 不发起工具调用。
2. **调用一个或多个工具（Call one or more tools）。** 输出结构化的调用对象。在 `parallel_tool_calls: true` 时（OpenAI 和 Gemini 默认开启，Anthropic 需手动启用），模型可以在一轮中输出多个调用。
3. **拒绝（Refuse）。** 严格模式（strict mode）的结构化输出可以产出一个类型化的 `refusal` 块，而非一次调用。

一个工具调用载荷有三个稳定字段：调用 `id`、工具 `name`，以及一个 JSON 形式的 `arguments` 对象。`id` 的存在使宿主能够把后续结果与具体那次调用对应起来——这在并行调用乱序返回时尤为重要。

### 第三步：执行

宿主收到调用，对照声明的 schema 校验参数，然后运行执行器。参数无效意味着模型臆造了某个字段或用错了类型——这在弱模型上是非常常见的失败模式。生产级宿主在参数无效时会采取以下三种做法之一：快速失败并把错误暴露给模型，用受约束的解析器修复 JSON，或将校验错误一并写入提示词后重试模型。

执行器本身就是普通代码。Python、TypeScript、一条 shell 命令、一次数据库查询。它产出一个结果，通常是字符串，但也可以是任意 JSON 值，或一个结构化内容块（在 MCP 中可为文本、图像或资源引用）。结果必须是可序列化的。

### 第四步：观察

宿主把工具结果追加进会话（作为一条带匹配 `id` 的 `tool` 角色消息），然后重新调用模型。此时模型的上下文里已有工具输出，可以产出最终答案，或请求更多调用。如此持续，直到模型停止发起调用，或宿主触及对迭代次数的安全上限。

### 信任划分

工具有两种风味，对安全而言意义重大。

- **纯工具（Pure）。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可以放心地投机性调用。
- **有后果工具（Consequential）。** 会变更状态、花钱、触碰用户数据。`send_email`、`delete_file`、`execute_trade`。必须设闸把关。

Meta 在 2026 年提出的智能体安全「二选二规则」（Rule of Two）指出：单一轮次至多只能组合以下三项中的两项——不可信输入、敏感数据、有后果动作。工具接口正是你强制执行这条规则的地方——通过拒绝调用、要求用户确认或升级权限范围。完整的安全章节见第 13 阶段 · 15，智能体级别的权限策略见第 14 阶段 · 09。

### 循环存在于何处

| 场景 | 谁负责描述 | 谁负责决策 | 谁负责执行 |
|---------|---------------|-------------|--------------|
| 单轮函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP 服务器 | 经由 MCP 客户端的 LLM | MCP 服务器 |
| A2A | Agent Card 发布方 | 调用方智能体 | 被调用方智能体 |
| Web 浏览器（函数调用智能体） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

无论在哪里，都是同样的四步。列名会变；结构不变。

### 为什么不干脆提示模型直接输出 JSON？

「让模型用 JSON 回复」是函数调用出现之前的做法。它在前沿模型上约有 5% 到 15% 的概率失败，在较小的模型上失败率高得多。失败模式包括缺少花括号、多余的尾逗号、臆造字段、类型错误。于是你又需要一道 JSON 修复流程、一次重试，或一个受约束的解码器。

原生函数调用更优，理由有三。其一，供应商用确切的调用形态对模型做了端到端训练，因此在严格模式下，合法 JSON 率攀升至 98% 到 99%。其二，调用载荷位于它自己的协议槽位里，而非夹在自由文本中——这样一次工具调用绝不会泄漏进用户可见的回复里。其三，供应商通过受约束解码（constrained decoding）强制 schema 合规（OpenAI 的 strict mode、Anthropic 的 `tool_use`、Gemini 的 `responseSchema`）。其输出保证能通过校验。

### 断路器

循环会在模型停止发起调用、或宿主触及最大轮数时终止。生产级宿主把这个值设在 5 到 20 轮之间。超出这个范围，你几乎肯定陷入了一个模型无法自行退出的循环。Claude Code 默认 20；OpenAI Assistants 为 10；Cursor 的智能体模式为 25。

另一种选择——无界循环——每隔半年就以「智能体一夜之间花掉 400 美元 API 调用费」这类事故复盘的形式出现一次。没有上限就不要上线。

## 动手用

`code/main.py` 在不调用 LLM 的情况下运行这个四步循环。一个假的「决策器（decider）」函数通过对用户消息做模式匹配来模拟模型；而执行器、schema 校验器以及观察步骤的脚手架（harness）都是真实的。运行它，你能看到完整的请求/响应编排，并可打印中间状态；待到后续课程，再用任意真实供应商替换掉这个假决策器。

要重点看的地方：

- 工具注册表为每个工具保存几个字段：名称、描述、schema，以及一个执行器引用。
- 校验器是 JSON Schema 的一个最小子集（类型、required、enum、min/max），纯用标准库写成。第 13 阶段 · 04 会给出一个更完整的版本。
- 循环把迭代次数上限设为五。生产级智能体正需要这种断路器。

## 交付物

本课产出 `outputs/skill-tool-interface-reviewer.md`。给定一份工具定义草稿（名称 + 描述 + schema + 执行器轮廓），该 skill 会从循环适配度（loop fitness）的角度审查它：名称是否机器稳定，描述是否是一份完整的使用简介，schema 是否正确使用了 JSON Schema 2020-12，以及纯工具与有后果工具的分类是否明确。

## 练习

1. 给 `code/main.py` 添加第四个工具，名为 `get_stock_price(ticker)`。把它的描述写成「当用户按股票代码询问当前股价时使用。不要用于历史价格或市场综述。」运行脚手架，确认假决策器会把提到股票代码的查询路由到这个新工具。

2. 把 schema 校验器搞坏。传入一个其 `arguments` 对象缺少某个必填字段的调用，确认宿主在执行前就拒绝了它。然后再传入一个带有额外未知字段的调用。请决定：宿主应当拒绝还是忽略？用一个安全方面的论证来为你的选择辩护。

3. 把脚手架里的每个工具分类为纯工具或有后果工具。给需要的注册表条目加上 `consequential: true` 标志，并修改循环，使其在每次选中一个有后果工具时打印一行「将与用户确认」。这正是每个生产级宿主都需要的确认闸门的雏形。

4. 在纸上画出这个四步循环，并把上面那张供应商列表按你最爱的客户端（Claude Desktop、Cursor、ChatGPT 或自定义栈）填好。与第 13 阶段 · 06 中 MCP 专属的变体相互对照。

5. 把 OpenAI 的函数调用指南从头到尾读一遍。找出那个出现在请求中、却不在本课所呈现的四步循环里的字段。解释它增添了什么，以及为什么它是方便之物而非必需之物。

## 关键术语

| 术语 | 人们怎么说 | 它实际指什么 |
|------|----------------|------------------------|
| 工具（Tool） | 「模型能调用的东西」 | 名称 + JSON Schema 类型化输入 + 执行器函数 三者构成的三元组 |
| 函数调用（Function calling） | 「原生工具使用」 | 供应商层面的 API 支持，用于输出结构化工具调用而非散文 |
| 工具调用（Tool call） | 「模型发出的行动请求」 | 由模型输出的、带 `id`、`name`、`arguments` 的 JSON 载荷 |
| 工具结果（Tool result） | 「工具返回的东西」 | 执行器的输出，包裹在一条带匹配 id 的 `tool` 角色消息里 |
| 并行工具调用（Parallel tool calls） | 「一次发起多个调用」 | 一轮模型输出中的多个调用对象，彼此独立，可按 id 排序 |
| 严格模式（Strict mode） | 「保证是 JSON」 | 受约束解码，强制模型输出通过声明的 schema 校验 |
| 纯工具（Pure tool） | 「只读工具」 | 无副作用；可安全重跑 |
| 有后果工具（Consequential tool） | 「动作工具」 | 变更外部状态；需要设闸、审计或用户确认 |
| 四步循环（Four-step loop） | 「工具调用周期」 | 描述 → 决策 → 执行 → 观察 |
| 宿主（Host） | 「智能体运行时」 | 持有工具注册表、调用模型并运行执行器的那个程序 |
