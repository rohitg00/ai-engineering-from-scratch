# 工具接口——为什么代理需要结构化 I/O

> 语言模型产生 token。程序执行动作。两者之间的鸿沟就是工具接口：一个契约，让模型请求动作，让宿主执行它。每个 2026 年技术栈——OpenAI、Anthropic 和 Gemini 上的函数调用；MCP 的 `tools/call`；A2A 的任务部件——都是同一个四步循环的不同编码。本课命名这个循环并展示运行它的最小机制。

**类型：** Learn
**语言：** Python（stdlib，无 LLM）
**前置知识：** Phase 11（LLM 补全 API）
**时间：** ~45 分钟

## 学习目标

- 解释为什么一个只能生成文本的 LLM 无法独自对真实世界采取行动。
- 画出四步工具调用循环（描述 → 决定 → 执行 → 观察）并指明每一步由谁负责。
- 将工具描述写为三部分：名称、JSON Schema 输入和确定性执行器函数。
- 区分纯工具和带副作用的工具，并说明这种划分对安全为何重要。

## 问题所在

LLM 发出下一个 token 的概率分布。这就是整个输出面。如果你问聊天模型"班加罗尔现在的天气如何"，它可以写出一个看似合理的句子，但它无法拨入天气 API。这个句子可能碰巧正确，也可能是三天前的旧数据。

弥合这个鸿沟就是工具接口的目的。宿主程序——你的代理运行时、Claude Desktop、ChatGPT、Cursor 或自定义脚本——向模型广告一系列可调用工具。模型在决定需要动作时，发出一个结构化载荷，指明工具及其参数。宿主解析该载荷，实际运行工具，并将结果反馈回去。循环持续，直到模型决定不再需要调用。

这个契约的第一个版本于 2023 年 6 月作为 OpenAI 的 "functions" 参数发布。Anthropic 随后在 Claude 2.1 中推出了 `tool_use` 块。Gemini 几个月后添加了 `functionDeclarations`。每个提供商现在都暴露相同的形状：JSON-Schema 类型的工具列表输入，JSON 载荷工具调用输出。模型上下文协议（2024 年 11 月）将该契约泛化，使一个工具注册表可以服务每个模型。A2A（2026 年 4 月，v1.0）将相同原语分层用于代理间委托。

四步循环是所有这些之下的不变量。Phase 13 的其他一切都是对它的 elaboration。

## 核心概念

### 第一步：描述

宿主用三个字段声明每个工具。

- **名称。** 稳定、机器可读的标识符。`get_weather`，而非"天气东西"。
- **描述。** 一段自然语言简介。"当用户询问特定城市的当前状况时使用。不要用于历史数据。"
- **输入模式。** 一个 JSON Schema 对象（draft 2020-12），描述工具的参数。

模型接收列表。现代提供商使用提供商特定的模板将这些声明序列化到系统提示中，因此作为调用者你只处理结构化形式。

### 第二步：决定

给定用户消息和可用工具，模型选择三种行为之一。

1. **直接以文本回答。** 无工具调用。
2. **调用一个或多个工具。** 发出结构化调用对象。在 `parallel_tool_calls: true` 下（OpenAI 和 Gemini 默认，Anthropic 可选），模型可以在一个回合中发出多个调用。
3. **拒绝。** 严格模式结构化输出可以产生类型化的 `refusal` 块而非调用。

工具调用载荷有三个稳定字段：调用 `id`、工具 `name` 和 JSON `arguments` 对象。id 存在以便宿主可以将后续结果与特定调用关联，这在并行调用无序返回时很重要。

### 第三步：执行

宿主接收调用，根据声明的模式验证参数，并运行执行器。无效参数意味着模型幻觉了一个字段或使用了错误类型——弱模型上非常常见的失败模式。生产宿主对无效参数做三件事之一：快速失败并将错误暴露给模型、用约束解析器修复 JSON、或将验证错误包含在提示中重试模型。

执行器本身是普通代码。Python、TypeScript、shell 命令、数据库查询。它产生一个结果，通常是字符串但可以是任何 JSON 值或结构化内容块（MCP 中的文本、图像或资源引用）。结果必须是可序列化的。

### 第四步：观察

宿主将工具结果附加到对话（作为带匹配 `id` 的 `tool` 角色消息）并重新调用模型。模型现在将工具输出纳入上下文，可以产生最终答案或请求更多调用。这持续，直到模型停止发出调用或宿主达到迭代次数的安全限制。

### 信任划分

工具有两种对安全重要的风味。

- **纯。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。安全地推测性调用。
- ** consequential。** 改变状态、花钱、触碰用户数据。`send_email`、`delete_file`、`execute_trade`。必须设门。

Meta 2026 年代理安全的"双规则"说，单个回合最多只能组合以下两项中的两项：不可信输入、敏感数据、consequential 动作。工具接口是你执行该规则的地方——通过拒绝调用、要求用户确认或提升范围。完整安全章节见 Phase 13 · 15，代理级权限策略见 Phase 14 · 09。

### 循环在哪里运行

| 上下文 | 谁描述 | 谁决定 | 谁执行 |
|---------|---------------|-------------|--------------|
| 单回合函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP 服务器 | 通过 MCP 客户端的 LLM | MCP 服务器 |
| A2A | 代理卡片发布者 | 调用代理 | 被调用代理 |
| Web 浏览器（函数调用代理） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

到处，相同的四步。列名改变；结构不变。

### 为什么不直接提示模型发出 JSON？

"让模型以 JSON 回复"是函数调用前的模式。在前沿模型上失败约 5% 到 15%，在较小模型上失败更多。失败模式包括缺少大括号、尾随逗号、幻觉字段和错误类型。然后你需要 JSON 修复通道、重试或约束解码器。

原生函数调用更好，原因有三。首先，提供商在确切调用形状上端到端训练模型，因此严格模式下有效 JSON 率攀升到 98% 到 99%。其次，调用载荷位于自己的协议槽中，不在自由文本内——因此工具调用永远不会泄漏到用户可见的回复中。第三，提供商用约束解码强制执行模式合规（OpenAI 的严格模式、Anthropic 的 `tool_use`、Gemini 的 `responseSchema`）。输出保证验证。

### 熔断器

循环在模型停止发出调用或宿主达到最大回合数时终止。生产宿主将此设置为 5 到 20 回合之间。超过此数，你几乎肯定处于模型无法退出的循环中。Claude Code 默认 20；OpenAI Assistants 默认 10；Cursor 的代理模式默认 25。

替代方案——无界循环——每六个月以"代理一夜之间花了 400 美元 API 调用"的事后分析出现。不要在没有边界的情况下发布。

Phase 14 · 12 深入覆盖错误恢复和自愈；Phase 17 覆盖生产速率限制。

## 使用它

`code/main.py` 在没有 LLM 的情况下运行四步循环。一个伪造的"决定者"函数通过模式匹配用户消息来模拟模型；执行器、模式验证器和观察步骤框架是真实的。运行它以查看可打印中间状态的完整请求/响应编排，然后在后续课程中用任何真实提供商替换伪造决定者。

看点：

- 工具注册表为每个工具保存三个字段：名称、描述、模式和执行器引用。
- 验证器是最小的 JSON Schema 子集（类型、必需、枚举、最小/最大），仅用 stdlib 编写。Phase 13 · 04 发布更完整的版本。
- 循环将迭代次数限制为五。生产代理需要正是这种熔断器。

## 交付它

本课产出 `outputs/skill-tool-interface-reviewer.md`。给定草稿工具定义（名称 + 描述 + 模式 + 执行器大纲），该技能审核其循环适配性：名称是否机器稳定、描述是否完整使用简介、模式是否正确使用 JSON Schema 2020-12、纯 vs consequential 分类是否明确。

## 练习

1. 向 `code/main.py` 添加第四个工具 `get_stock_price(ticker)`。将其描述写为"当用户按股票代码询问当前股价时使用。不要用于历史价格或市场摘要。"运行框架并确认伪造决定者将提及股票代码的查询路由到新工具。

2. 破坏模式验证器。传递一个 `arguments` 对象缺少必需字段的调用，并确认宿主在执行前拒绝它。然后传递一个带额外未知字段的调用。决定：宿主应该拒绝还是忽略？用安全论证证明你的选择。

3. 将框架中的每个工具分类为纯或 consequential。向需要它的注册表条目添加 `consequential: true` 标志，并更改循环以在选择 consequential 工具时打印"将与用户确认"行。这是每个生产宿主需要的确认门的形状。

4. 在纸上画出四步循环，并为你的 favorite 客户端（Claude Desktop、Cursor、ChatGPT 或自定义栈）填写上面的提供商列表格。与 Phase 13 · 06 中的 MCP 特定变体交叉引用。

5. 从头到尾阅读 OpenAI 的函数调用指南。识别一个存在于请求中但不在此处呈现的四步循环中的字段。解释它添加了什么以及为什么它是便利而非本质。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 工具 | "模型可以调用的东西" | 名称 + JSON-Schema 类型输入 + 执行器函数的三元组 |
| 函数调用 | "原生工具使用" | 提供商级 API 支持，用于发出结构化工具调用而非散文 |
| 工具调用 | "模型的动作请求" | 模型发出的带 `id`、`name`、`arguments` 的 JSON 载荷 |
| 工具结果 | "工具返回的内容" | 执行器的输出，包装在带匹配 id 的 `tool` 角色消息中 |
| 并行工具调用 | "一次多个调用" | 一个模型回合中的多个调用对象，独立且可按 id 排序 |
| 严格模式 | "保证 JSON" | 约束解码，强制模型输出针对声明的模式验证 |
| 纯工具 | "只读工具" | 无副作用；安全地重新运行 |
| Consequential 工具 | "动作工具" | 改变外部状态；需要门、审计或用户确认 |
| 四步循环 | "工具调用周期" | 描述 → 决定 → 执行 → 观察 |
| 宿主 | "代理运行时" | 持有工具注册表、调用模型并运行执行器的程序 |
