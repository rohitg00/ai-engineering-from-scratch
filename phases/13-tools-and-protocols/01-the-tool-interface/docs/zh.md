# 工具接口 — 为什么代理需要结构化I/O

> 语言模型生成token（token）。程序执行操作。这两者之间的差距就是工具接口：一个允许模型请求操作并由主机执行的契约。2026年的所有技术栈——OpenAI、Anthropic和Gemini上的函数调用；MCP的`tools/call`；A2A的任务部分——都是同一个四步循环的不同编码方式。本课程命名了这个循环并展示了运行它的最小机制。

**类型：** 学习
**语言：** Python（标准库，无LLM）
**先决条件：** 第11阶段（LLM完成API）
**时间：** 约45分钟

## 学习目标

- 解释为什么只能生成文本的LLM无法自行对现实世界采取行动。
- 绘制四步工具调用循环（描述→决定→执行→观察）并命名每个步骤的所有者。
- 将工具描述编写为三个部分：名称、JSON Schema输入和确定性执行器函数。
- 区分纯工具和有副作用的工具，并说明这种区分对安全的重要性。

## 问题

LLM对下一个token发出概率分布。这就是全部的输出表面。如果你问聊天模型"班加罗尔现在的天气如何"，它可以写一个合理的句子，但它无法接入天气API。这个句子可能是巧合正确，也可能是三天前的旧信息。

弥合这一差距就是工具接口的目的。主机程序——你的代理运行时、Claude Desktop、ChatGPT、Cursor或自定义脚本——向模型展示可调用工具列表。当模型决定需要执行操作时，它会发出一个结构化负载，命名一个工具及其参数。主机解析该负载，真实运行工具，并将结果反馈回去。循环持续进行，直到模型决定不再需要调用。

该契约的第一个版本于2023年6月以OpenAI的"functions"参数形式发布。Anthropic随后在Claude 2.1中加入了`tool_use`块。Gemini在几个月后添加了`functionDeclarations`。现在每个提供商都暴露相同的形状：输入JSON-Schema类型的工具列表，输出JSON负载的工具调用。模型上下文协议（2024年11月）将契约泛化，使一个工具注册表服务所有模型。A2A（2026年4月，v1.0）为代理到代理的委托添加了相同的原语。

四步循环是所有这些的基础不变量。第13阶段的其他内容都是对此的详细阐述。

## 概念

### 第一步：描述

主机使用三个字段声明每个工具。

- **名称。** 一个稳定的、机器可读的标识符。`get_weather`，而不是"天气工具"。
- **描述。** 一段自然语言的简短说明。"当用户询问特定城市的当前状况时使用。不要用于历史数据。"
- **输入模式。** 一个JSON Schema对象（草案2020-12），描述工具的参数。

模型接收列表。现代提供商使用提供商特定的模板将这些声明序列化到系统提示中，因此作为调用者，你只需处理结构化形式。

### 第二步：决定

根据用户的消息和可用工具，模型选择三种行为之一。

1. **直接以文本回答。** 不调用工具。
2. **调用一个或多个工具。** 发出结构化调用对象。在`parallel_tool_calls: true`下（OpenAI和Gemini默认，Anthropic选择加入），模型可以在一轮中发出多个调用。
3. **拒绝。** 严格模式的结构化输出可以生成类型化的`refusal`块而不是调用。

工具调用负载有三个稳定字段：调用`id`、工具`name`和JSON `arguments`对象。id的存在使主机能够将后续结果与特定调用关联起来，这在并行调用乱序返回时很重要。

### 第三步：执行

主机接收调用，根据声明的模式验证参数，并运行执行器。无效参数意味着模型产生了幻觉字段或使用了错误类型——这是弱模型上非常常见的故障模式。生产主机对无效参数做三种处理之一：快速失败并向模型展示错误，使用受限解析器修复JSON，或在提示中包含验证错误后重试模型。

执行器本身是普通代码。Python、TypeScript、shell命令、数据库查询。它产生一个结果，通常是字符串，但可以是任何JSON值或结构化内容块（文本、图像或MCP中的资源引用）。结果必须是可序列化的。

### 第四步：观察

主机将工具结果附加到对话中（作为带有匹配`id`的`tool`角色消息），并重新调用模型。现在模型在上下文中有了工具输出，可以产生最终答案或请求更多调用。这持续进行，直到模型停止发出调用或主机达到迭代次数的安全限制。

### 信任分离

工具有两种与安全相关的类型。

- **纯工具。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可以推测性调用。
- **有后果的工具。** 修改状态、花费金钱、接触用户数据。`send_email`、`delete_file`、`execute_trade`。必须被门控。

Meta的2026年"二元法则"（Rule of Two）用于代理安全，表示单个回合最多可以结合以下两种：不可信输入、敏感数据、有后果的操作。工具接口是你执行该规则的地方——通过拒绝调用、要求用户确认或扩大范围。参见第13阶段·15获取完整的安全章节，以及第14阶段·09获取代理级别的权限策略。

| 上下文 | 谁描述 | 谁决定 | 谁执行 |
|---------|---------------|-------------|--------------|
| 单轮函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |

### 为什么不只是提示模型发出JSON？

"让模型以JSON回复"是函数调用前的模式。它在前沿模型上失败约5-15%的时间，在较小模型上失败率更高。故障模式包括缺少大括号、尾随逗号、幻觉字段和错误类型。然后你需要一个JSON修复步骤、重试或受限解码器。

原生函数调用更好有三个原因。首先，提供商使用确切的调用形状对模型进行端到端训练，因此在严格模式下有效JSON率上升到98-99%。其次，调用负载位于自己的协议槽中，而不是在自由文本内部——因此工具调用永远不会泄漏到用户可见的回复中。第三，提供商使用受限解码（OpenAI的严格模式、Anthropic的`tool_use`、Gemini的`responseSchema`）强制模式合规。输出保证可以验证。

### 断路器

当模型停止发出调用或主机达到最大轮次计数时，循环终止。生产主机将其设置为5到20轮之间。超过这个值，你几乎肯定处于一个模型无法退出的循环中。Claude Code默认为20；OpenAI Assistants为10；Cursor的代理模式为25。

替代方案——无界循环——每六个月就会出现一次"代理一晚上花费400美元API调用"的事后分析。不要在没有边界的情况下发布。

## 使用它

`code/main.py`在没有LLM的情况下运行四步循环。一个假的"决定器"函数通过模式匹配用户消息来模拟模型；执行器、模式验证器和观察步骤工具是真实的。运行它以查看带有可打印中间状态的完整请求/响应编排，然后在后续课程中用任何真实提供商替换假决定器。

查看内容：

- 工具注册表每个工具保存三个字段：名称、描述、模式和执行器引用。
- 验证器是最小的JSON Schema子集（类型、必需、枚举、最小/最大），仅使用标准库编写。第13阶段·04提供了一个更完整的版本。
- 循环将迭代次数限制为五。生产代理需要这种类型的断路器。

## 练习

1. 向`code/main.py`添加第四个工具`get_stock_price(ticker)`。将其描述编写为"当用户要求按股票代码获取当前股价时使用。不要用于历史价格或市场摘要。"运行工具并确认假决定器将提及股票代码的查询路由到新工具。

2. 破坏模式验证器。传递一个调用，其`arguments`对象缺少必需字段，并确认主机在执行前拒绝它。然后传递一个带有额外未知字段的调用。决定：主机应该拒绝还是忽略？用安全论据证明你的选择。

3. 将工具中的每个工具分类为纯工具或有后果的工具。为需要的注册表条目添加`consequential: true`标志，并修改循环以在有后果工具被选择时打印"将向用户确认"行。这是每个生产主机需要的确认门控的形状。

4. 在纸上绘制四步循环，并填写上面提供商列表格，填入你最喜欢的客户端（Claude Desktop、Cursor、ChatGPT或自定义堆栈）。与第13阶段·06中的MCP特定变体交叉引用。

5. 从头到尾阅读OpenAI的函数调用指南。识别请求中存在但在此处四步循环中不存在的字段。解释它添加了什么以及为什么它是便利而非必需的。

## 关键术语

| 术语 | 人们所说的 | 实际含义 |
|------|----------------|------------------------|
| 工具 | "模型可以调用的东西" | 名称+JSON Schema类型输入+执行器函数的三元组 |
| 函数调用 | "原生工具使用" | 提供商级别的API支持，用于发出结构化工具调用而非散文 |
| 工具调用 | "模型请求行动" | 模型发出的带有`id`、`name`、`arguments`的JSON负载 |
| 工具结果 | "工具返回的内容" | 执行器的输出，包装在带有匹配id的`tool`角色消息中 |
| 并行工具调用 | "同时多次调用" | 在一个模型回合中的多个调用对象，独立且可通过id排序 |
| 严格模式 | "保证的JSON" | 受限解码，强制模型输出验证声明的模式 |
| 纯工具 | "只读工具" | 无副作用；可安全重运行 |
| 有后果的工具 | "行动工具" | 修改外部状态；需要门控、审计或用户确认 |
| 四步循环 | "工具调用周期" | 描述→决定→执行→观察 |
| 主机 | "代理运行时" | 持有工具注册表、调用模型并运行执行器的程序 |
