给定提议的 MCP 服务器能力（以纯英文或草案工具列表形式），将每个能力分类为工具、资源或提示词并附一句理由。

生成：

1. 每个能力分类。对于每个项目，返回 `{name, primitive: tool | resource | prompt, rationale}`。
2. 资源 URI 方案。如果任何能力成为资源，提议 URI 方案（`notes://`、`gh://`、`db://`）和模板模式。
3. 提示词参数骨架。如果任何能力成为提示词，提议参数列表和必需/可选标志。
4. 订阅候选。标记经常变化且会从 `resources/subscribe` 受益的资源。
5. 反模式标志。指出旧设计将读取包装在工具中的情况（例如 `notes_read(id)`），当资源服务更好时。

硬性拒绝：
- 任何未拆分就分类为"既是工具又是资源"的能力。选择一个或搭建一对。
- 任何未识别必需参数的提示词。在斜杠命令 UI 中暴露需要参数模式。
- 任何不可寻址的资源 URI 方案（自由格式字符串，非 URI）。

拒绝规则：
- 如果所有能力都归类为工具，拒绝并询问服务器是否有可作为资源的只读数据。
- 如果没有能力适合提示词，没问题；提示词是可选的。不要发明它们。
- 如果服务器的域更适合 A2A（agent 间协作、不透明状态），拒绝并重定向到 Phase 13 · 19。

输出：一页决策报告，包含分类表、URI 方案提议、提示词骨架和订阅标志。以该服务器最具影响力的工具 -> 资源转换结尾。
